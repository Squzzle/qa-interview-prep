# Что такое Awaitility и для каких целей его используют при тестировании асинхронных операций (в т.ч. API)?

**Коротко:** Awaitility — Java-библиотека для тестирования асинхронного и многопоточного кода, которая заменяет хрупкие `Thread.sleep()` на декларативное ожидание условия ("подожди, пока условие X не станет истинным, но не дольше Y секунд").

## Развёрнутый ответ

При тестировании асинхронных операций (обработка сообщений из очереди, фоновые задачи, асинхронные API-вызовы, вебхуки, отложенная обработка данных) часто возникает проблема: результат операции появляется не сразу, а через непредсказуемый промежуток времени. Наивное решение — вставить `Thread.sleep(5000)` — плохо, потому что:

- если операция завершится быстрее — тест будет неоправданно медленным;
- если операция завершится дольше (например, из-за нагрузки на CI) — тест упадёт, хотя система работает корректно (это называется "флакающий" тест, flaky test).

**Awaitility** решает эту проблему, предоставляя fluent DSL для polling-ожидания условия:

```java
await().atMost(10, SECONDS)
       .pollInterval(500, MILLISECONDS)
       .until(() -> orderService.getStatus(orderId).equals("COMPLETED"));
```

Здесь Awaitility будет периодически (каждые 500 мс) проверять условие, пока оно не станет истинным, но не дольше 10 секунд, после чего либо тест продолжится, либо упадёт с понятным сообщением о таймауте.

**Ключевые возможности:**

- **`atMost(duration)`** — максимальное время ожидания.
- **`pollInterval(duration)`** — интервал между проверками условия.
- **`pollDelay(duration)`** — задержка перед первой проверкой.
- **`until(Callable<Boolean>)` / `untilAsserted(ThrowingRunnable)`** — условие ожидания; `untilAsserted` позволяет использовать полноценные ассерты (например, AssertJ) внутри, а не просто boolean-условие.
- **`ignoreExceptions()`** — игнорировать исключения во время проверки условия (полезно, если ресурс ещё не создан и запрос временно возвращает 404).
- Возможность задавать стратегию по умолчанию глобально через `Awaitility.setDefaultTimeout()`/`setDefaultPollInterval()`.

**Применение в API-тестировании:**

- проверка, что асинхронно созданный ресурс (например, отчёт, генерируемый в фоне) в итоге появится и станет доступным через `GET`;
- проверка вебхуков — что после определённого действия внешний сервис получит и обработает callback;
- проверка обработки сообщений из очереди (Kafka, RabbitMQ), когда между отправкой сообщения и результатом обработки есть задержка;
- проверка eventual consistency в распределённых системах, где данные могут появиться не мгновенно во всех узлах/кэшах.

## Пример / когда применяется

Проверка асинхронного создания заказа, статус которого меняется через некоторое время:

```java
String orderId = given()
    .body(orderRequest)
.when()
    .post("/orders")
.then()
    .statusCode(202) // Accepted — обработка асинхронная
    .extract().path("id");

await()
    .atMost(15, TimeUnit.SECONDS)
    .pollInterval(1, TimeUnit.SECONDS)
    .untilAsserted(() -> {
        Response response = given().get("/orders/" + orderId);
        assertThat(response.jsonPath().getString("status")).isEqualTo("COMPLETED");
    });
```

## На что смотрит интервьюер

- Понимание проблемы `Thread.sleep()` и почему он делает тесты флакающими или медленными.
- Знание основных методов Awaitility (`atMost`, `pollInterval`, `until`/`untilAsserted`).
- Умение привести реальный кейс из практики: асинхронная обработка, очереди сообщений, вебхуки, отложенная генерация отчётов.
- Красный флаг: если кандидат предлагает решать проблему асинхронности только через увеличение `Thread.sleep()`, не зная про инструменты polling-ожидания.
