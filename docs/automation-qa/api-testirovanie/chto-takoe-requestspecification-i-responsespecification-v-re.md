# Что такое `RequestSpecification` и `ResponseSpecification` в RestAssured? Для чего они нужны?

**Коротко:** Это объекты-спецификации, которые позволяют один раз задать общие настройки запроса (`RequestSpecification`) или ожидаемые проверки ответа (`ResponseSpecification`) и переиспользовать их во всех тестах, избегая дублирования кода.

## Развёрнутый ответ

**`RequestSpecification`** — объект, инкапсулирующий общие параметры запроса: базовый URL, заголовки, аутентификацию, content-type, таймауты, параметры и т.д. Создаётся через `RequestSpecBuilder` или напрямую в `given().spec(...)`. Используется для того, чтобы не повторять одни и те же настройки (например, base URI и заголовок авторизации) в каждом тесте.

**`ResponseSpecification`** — аналогичный объект, но для описания ожидаемых характеристик ответа: ожидаемый статус-код, content-type, время ответа, соответствие JSON Schema и т.д. Создаётся через `ResponseSpecBuilder` и применяется в `then().spec(...)`.

Зачем это нужно:

- **Устранение дублирования (DRY)** — если у большинства тестов одинаковый базовый URL, заголовки авторизации, ожидаемый content-type — вместо копирования этих настроек в каждый тест они выносятся в единую спецификацию.
- **Централизованное изменение конфигурации** — если, например, поменялся базовый URL окружения или формат авторизации, достаточно поправить один класс/метод, а не каждый тестовый метод.
- **Читаемость тестов** — тест становится короче и фокусируется на специфичной для него логике, а не на общей настройке.
- **Поддержка нескольких окружений** — можно создавать разные спецификации для dev/stage/prod и передавать нужную в зависимости от конфигурации запуска.

Также существует глобальная настройка через `RestAssured.requestSpecification` и `RestAssured.responseSpecification`, которая применяется ко всем запросам автоматически, если не переопределена локально.

## Пример / когда применяется

Создание `RequestSpecification`:

```java
RequestSpecification requestSpec = new RequestSpecBuilder()
    .setBaseUri("https://api.example.com")
    .setContentType(ContentType.JSON)
    .addHeader("Authorization", "Bearer " + token)
    .build();
```

Создание `ResponseSpecification`:

```java
ResponseSpecification responseSpec = new ResponseSpecBuilder()
    .expectStatusCode(200)
    .expectContentType(ContentType.JSON)
    .expectResponseTime(lessThan(2000L))
    .build();
```

Использование в тесте:

```java
given()
    .spec(requestSpec)
.when()
    .get("/users/123")
.then()
    .spec(responseSpec)
    .body("name", equalTo("Иван"));
```

Такой подход особенно полезен в больших проектах с десятками тестовых классов, где единая точка конфигурации критична для поддержки.

## На что смотрит интервьюер

- Понимание назначения спецификаций именно как способа переиспользования и централизации конфигурации.
- Знание, как создать спецификацию через билдеры (`RequestSpecBuilder`/`ResponseSpecBuilder`).
- Умение объяснить разницу между локальной спецификацией (для конкретного теста/класса) и глобальной (`RestAssured.requestSpecification`).
- Красный флаг: если кандидат не видит проблемы дублирования кода и не понимает, зачем нужна абстракция вместо повторения `given()` с одинаковыми параметрами в каждом тесте.
- Follow-up: "Как совмещать несколько спецификаций в одном запросе?", "Как переопределить часть настроек из глобальной спецификации в конкретном тесте?".
