# Что такое REST и SOAP? В чем их ключевые отличия?

**Коротко:** REST — архитектурный стиль поверх HTTP, использующий стандартные методы и обычно JSON; SOAP — строгий протокол обмена сообщениями на базе XML со своей спецификацией (WSDL, конверты сообщений).

## Развёрнутый ответ

**REST (Representational State Transfer)** — это архитектурный стиль, предложенный Роем Филдингом, а не протокол. REST-API строится на принципах:

- используются стандартные HTTP-методы: `GET`, `POST`, `PUT`, `PATCH`, `DELETE`;
- ресурсы идентифицируются через URI (например, `/users/123`);
- клиент-серверная модель без хранения состояния сессии на сервере (stateless) — каждый запрос содержит всю необходимую информацию;
- поддерживается кэширование ответов;
- формат данных обычно JSON (реже XML, YAML);
- используются стандартные HTTP-коды ответа (`200`, `404`, `500` и т.д.).

**SOAP (Simple Object Access Protocol)** — это протокол обмена структурированными сообщениями, строго регламентированный спецификацией W3C:

- сообщения оформляются в виде XML-конвертов с обязательной структурой: `Envelope`, `Header`, `Body`;
- описание сервиса задаётся файлом WSDL (Web Services Description Language), который формально описывает доступные операции, типы данных, эндпоинты;
- поддерживает встроенные стандарты безопасности (WS-Security), транзакции (WS-AtomicTransaction), надёжную доставку сообщений;
- может работать не только поверх HTTP, но и поверх SMTP, TCP и других протоколов;
- обычно тяжелее и медленнее из-за строгой XML-структуры и валидации по схеме (XSD).

**Ключевые отличия:**

| Критерий | REST | SOAP |
|---|---|---|
| Тип | Архитектурный стиль | Протокол |
| Формат данных | JSON, XML, plain text | Только XML |
| Транспорт | Только HTTP/HTTPS | HTTP, SMTP, TCP и др. |
| Контракт | Опционален (OpenAPI/Swagger) | Обязателен (WSDL) |
| Состояние | Stateless | Может быть stateful |
| Безопасность | HTTPS, OAuth, JWT | WS-Security (встроенная) |
| Вес сообщений | Лёгкий | Тяжёлый (много служебной разметки) |
| Типичное применение | Веб/мобильные API, микросервисы | Банковские, корпоративные системы, legacy-интеграции |

## Пример / когда применяется

REST используется в большинстве современных веб- и мобильных приложений (например, API соцсетей, интернет-магазинов). Пример REST-запроса:

```text
GET /api/v1/users/123 HTTP/1.1
Host: example.com
Accept: application/json
```

SOAP чаще встречается в банковском секторе, госсистемах, legacy-интеграциях, где важны строгая типизация и формальный контракт. Пример тела SOAP-запроса:

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
  <soapenv:Header/>
  <soapenv:Body>
    <getUserRequest>
      <userId>123</userId>
    </getUserRequest>
  </soapenv:Body>
</soapenv:Envelope>
```

## На что смотрит интервьюер

- Понимание, что REST — это стиль, а не протокол (частая ошибка кандидатов — называть REST протоколом).
- Знание структуры SOAP-сообщения и роли WSDL.
- Умение объяснить, когда стоит выбрать SOAP вместо REST (строгие контракты, безопасность на уровне сообщений, транзакции).
- Красный флаг: если кандидат путает SOAP с XML или не может назвать HTTP-методы REST.
