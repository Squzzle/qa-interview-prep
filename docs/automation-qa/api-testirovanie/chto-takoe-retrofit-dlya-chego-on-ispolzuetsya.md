# Что такое Retrofit? Для чего он используется?

**Коротко:** Retrofit — типобезопасный HTTP-клиент для Java/Kotlin (изначально для Android), который превращает REST API в Java-интерфейс с аннотациями, автоматически выполняя сериализацию/десериализацию и сетевые вызовы.

## Развёрнутый ответ

Retrofit — библиотека от Square, которая упрощает работу с REST API, описывая HTTP-эндпоинты через Java/Kotlin-интерфейсы с аннотациями (`@GET`, `@POST`, `@PUT`, `@DELETE`, `@Path`, `@Query`, `@Body` и т.д.). Под капотом Retrofit использует OkHttp как транспортный слой и конвертеры (обычно Gson или Jackson) для (де)сериализации.

Ключевые особенности:

- **Декларативный подход** — вместо ручного построения URL и выполнения запроса разработчик описывает интерфейс, а Retrofit генерирует реализацию во время выполнения (через динамические прокси).
- **Типобезопасность** — методы интерфейса возвращают строго типизированные объекты (`Call<User>`, `Response<User>`), что снижает вероятность ошибок при работе с ответами.
- **Поддержка синхронных и асинхронных вызовов** — через `execute()` (синхронно) и `enqueue()` (асинхронно с callback), а также интеграция с RxJava/Coroutines.
- **Гибкая настройка** — можно подключать интерцепторы OkHttp для логирования, добавления заголовков, повторных попыток, таймаутов.
- **Изначально создан для Android**, но активно используется и в серверных Java/Kotlin-приложениях, а также в автотестах API, где нужен типизированный клиент к внешнему сервису.

В контексте автоматизации тестирования Retrofit применяют, когда:

- нужен переиспользуемый типизированный API-клиент (например, общий для тестов и production-кода, либо отдельный тестовый клиент);
- проект уже использует Retrofit на стороне production-кода (мобильные приложения), и тесты пишутся в том же стиле;
- важна строгая типизация запросов/ответов, а не работа со "свободным" JSON, как в RestAssured.

Отличие от RestAssured: RestAssured больше заточен под BDD-стиль (`given()/when()/then()`) и удобен для быстрого написания API-тестов "с нуля", а Retrofit требует описания интерфейса заранее, зато даёт более строгую типизацию и переиспользуемость кода.

## Пример / когда применяется

Описание интерфейса API:

```java
public interface UserApi {
    @GET("users/{id}")
    Call<User> getUser(@Path("id") int id);

    @POST("users")
    Call<User> createUser(@Body User user);
}
```

Создание клиента и вызов:

```java
Retrofit retrofit = new Retrofit.Builder()
    .baseUrl("https://api.example.com/")
    .addConverterFactory(GsonConverterFactory.create())
    .build();

UserApi api = retrofit.create(UserApi.class);
Response<User> response = api.getUser(1).execute();

assertThat(response.code()).isEqualTo(200);
assertThat(response.body().getName()).isEqualTo("Иван");
```

## На что смотрит интервьюер

- Понимание, что Retrofit — это клиент, генерирующий реализацию интерфейса, а не просто "ещё одна HTTP-библиотека".
- Знание, что Retrofit использует OkHttp и конвертеры для JSON.
- Умение сравнить Retrofit с RestAssured и обосновать выбор в зависимости от контекста.
- Красный флаг: если кандидат путает Retrofit с Retrofit-Mock или не может объяснить роль аннотаций.
- Follow-up: "Как в Retrofit сделать асинхронный вызов?", "Как добавить логирование запросов через интерцептор OkHttp?".
