# Что такое сериализация и десериализация данных? Какие библиотеки для этого используются в Java (Jackson, Gson)?

**Коротко:** Сериализация — преобразование объекта в поток байт или текстовый формат (например, JSON) для передачи или хранения; десериализация — обратный процесс восстановления объекта из этого формата. В Java для этого чаще всего используют Jackson и Gson.

## Развёрнутый ответ

**Сериализация (serialization)** — процесс преобразования объекта языка программирования (например, POJO в Java) в формат, пригодный для передачи по сети или сохранения (JSON, XML, бинарный формат). Например, объект `User` с полями `id`, `name`, `email` превращается в JSON-строку.

**Десериализация (deserialization)** — обратный процесс: из строки JSON/XML восстанавливается объект языка программирования со всеми полями.

В контексте API-тестирования сериализация/десериализация критически важна, потому что:

- при отправке запроса (`POST`/`PUT`) тестовые данные — Java-объекты — сериализуются в JSON-тело запроса;
- при получении ответа JSON-тело десериализуется обратно в Java-объект (POJO/DTO), чтобы удобно проверять поля через геттеры, а не парсить строки вручную;
- это упрощает написание тестов и делает их более читаемыми и типобезопасными.

**Основные библиотеки в Java:**

- **Jackson** — самая распространённая библиотека, входит по умолчанию во многие фреймворки (Spring, RestAssured). Поддерживает богатый набор аннотаций (`@JsonProperty`, `@JsonIgnore`, `@JsonCreator` и др.), работу с датами через модуль `jackson-datatype-jsr310`, кастомные (де)сериализаторы, стриминговый API (`JsonParser`/`JsonGenerator`) для больших данных.
- **Gson** — библиотека от Google, более простая в использовании, с минимальной настройкой "из коробки" сериализует/десериализует объекты без обязательных аннотаций, но менее гибкая по сравнению с Jackson (меньше готовых модулей и расширений).
- **Genson**, **Moshi** — менее популярные альтернативы, используются реже.

В RestAssured сериализация/десериализация происходит "под капотом" через ObjectMapper (по умолчанию Jackson, если он есть в classpath), что позволяет передавать/получать объекты напрямую, не работая с сырым JSON.

## Пример / когда применяется

Java-класс:

```java
public class User {
    private int id;
    private String name;
}
```

Сериализация в JSON (через Jackson):

```json
{
  "id": 1,
  "name": "Иван"
}
```

Пример теста с RestAssured, где сериализация/десериализация происходит автоматически:

```java
User newUser = new User(1, "Иван");

User response = given()
    .contentType("application/json")
    .body(newUser)          // сериализация объекта в JSON
.when()
    .post("/users")
.then()
    .statusCode(201)
    .extract().as(User.class); // десериализация JSON в объект

assertThat(response.getName()).isEqualTo("Иван");
```

## На что смотрит интервьюер

- Понимание разницы между сериализацией и десериализацией на уровне терминологии (кандидаты часто путают направления).
- Знание конкретных библиотек и их отличий (Jackson vs Gson).
- Умение объяснить, зачем это нужно именно в тестировании (упрощение проверок, типобезопасность).
- Красный флаг: если кандидат не может объяснить, что такое POJO/DTO, или путает сериализацию с валидацией JSON-схемы.
- Follow-up: "Как настроить кастомный сериализатор для дат?", "Что произойдёт, если в JSON есть поле, которого нет в классе?" (зависит от настройки `FAIL_ON_UNKNOWN_PROPERTIES`).
