# Для чего нужно тестировать API?

**Коротко:** API-тестирование проверяет бизнес-логику, контракты и стабильность серверной части независимо от UI — это быстрее, надёжнее и позволяет находить дефекты раньше, до появления интерфейса.

## Развёрнутый ответ

API (Application Programming Interface) — это слой, через который взаимодействуют компоненты системы: фронтенд и бэкенд, микросервисы между собой, внешние интеграции. Тестирование API нужно по нескольким причинам:

- **Проверка бизнес-логики на уровне сервера.** UI может скрывать или маскировать ошибки (например, валидацией на клиенте), а API-тесты проверяют, как система ведёт себя "напрямую", без посредников.
- **Скорость и стабильность.** API-тесты не зависят от рендеринга страниц, анимаций, задержек загрузки DOM — они выполняются в разы быстрее UI-тестов и меньше подвержены флакам (нестабильным падениям).
- **Раннее обнаружение дефектов.** API обычно готово раньше, чем UI, поэтому тестирование можно начинать на более ранних этапах разработки (shift-left подход), не дожидаясь готового интерфейса.
- **Проверка контрактов между сервисами.** В микросервисной архитектуре важно убедиться, что сервисы обмениваются данными в ожидаемом формате (контрактное тестирование), что коды ответов, структура JSON/XML и заголовки соответствуют спецификации.
- **Безопасность.** Через API можно проверить авторизацию, аутентификацию, обработку некорректных данных, SQL-инъекции, доступ к чужим данным (IDOR) и другие уязвимости.
- **Покрытие edge-кейсов.** Через API проще и быстрее подготовить нестандартные данные, граничные значения, ошибочные сценарии (некорректные типы, отсутствующие поля, большие payload), чем через UI.
- **Экономическая эффективность.** По пирамиде тестирования, API-тесты (уровень сервиса) должны составлять значительную долю автоматизации — они дешевле в поддержке, чем UI-тесты, и дороже/точнее unit-тестов в плане покрытия интеграций.

## Пример / когда применяется

Типичный сценарий: перед релизом фичи "оформление заказа" QA пишет API-тесты, которые проверяют:

- создание заказа (`POST /orders`) — корректный статус-код, структура ответа;
- получение заказа (`GET /orders/{id}`) — соответствие данных;
- отмену заказа (`DELETE /orders/{id}`) — изменение статуса;
- обработку ошибок — попытка получить несуществующий заказ должна возвращать `404`, а не `500`.

Эти тесты можно запускать в CI на каждый коммит, не дожидаясь развёртывания UI, что ускоряет обратную связь команде разработки.

## На что смотрит интервьюер

- Понимает ли кандидат, что API-тесты не заменяют UI-тесты, а дополняют их (пирамида тестирования).
- Может ли объяснить разницу между уровнями: unit → API/service → UI.
- Упоминает ли скорость выполнения, стабильность, возможность тестировать раньше готовности UI.
- Красный флаг: сведение ответа только к "чтобы проверить, что API работает" без раскрытия конкретных выгод (скорость, стабильность, безопасность, контракты).
- Частый follow-up: "Чем API-тестирование отличается от UI-тестирования?" или "Какую долю тестов стоит выносить на уровень API?".
