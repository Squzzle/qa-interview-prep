# Как с помощью Awaitility дождаться выполнения определенного условия?

**Коротко:** Библиотека Awaitility предоставляя DSL `await().atMost(...).until(...)`, позволяет опрашивать (poll) условие через заданные интервалы до тех пор, пока оно не станет истинным или не истечёт таймаут, заменяя ненадёжные `Thread.sleep`.

## Развёрнутый ответ

Awaitility — это Java/Kotlin библиотека для тестирования асинхронного и распределённого кода. Проблема, которую она решает: при проверке асинхронных операций (обработка сообщения из очереди, срабатывание вебхука, обновление статуса в БД после отправки запроса, завершение фоновой задачи) нельзя точно знать, когда результат будет готов. Использование фиксированного `Thread.sleep(5000)` либо делает тест избыточно медленным, либо приводит к нестабильным (flaky) падениям, если операция заняла больше времени, чем заложено.

Awaitility вместо статичной паузы реализует polling: она периодически (с заданным интервалом, по умолчанию 100 мс) вызывает переданное условие и завершает ожидание сразу, как только условие выполнится, либо кидает исключение `ConditionTimeoutException`, если истёк максимальный таймаут.

Базовый синтаксис:

- `await()` — точка входа в DSL.
- `.atMost(Duration.ofSeconds(10))` — максимальное время ожидания.
- `.pollInterval(Duration.ofMillis(200))` — как часто проверять условие (опционально).
- `.pollDelay(Duration.ofMillis(500))` — задержка перед первой проверкой (опционально).
- `.until(...)` или `.untilAsserted(...)` — само условие.

Есть два основных варианта задания условия:

1. `until(Callable<Boolean> conditionEvaluator)` — передаётся лямбда, возвращающая `boolean`. Ожидание заканчивается, когда лямбда вернёт `true`.
2. `untilAsserted(ThrowingRunnable assertion)` — передаётся блок с обычными assert-ами (например, AssertJ или JUnit). Ожидание заканчивается, когда блок перестаёт кидать `AssertionError`. Это удобнее, так как позволяет переиспользовать обычные проверки без превращения их в boolean-выражение.

Также можно ждать значение через `.until(Callable<T> supplier, Matcher<T> matcher)` с использованием Hamcrest-матчеров.

Важные нюансы:

- По умолчанию Awaitility ловит исключения при опросе условия (кроме нескольких системных, таких как `OutOfMemoryError`) и просто повторяет попытку, пока не истечёт `atMost`, а не проваливает тест сразу.
- Если условие не выполнилось за отведённое время, кидается `ConditionTimeoutException` с понятным сообщением о том, что и сколько ожидалось.
- Можно настраивать глобальные дефолты через `Awaitility.setDefaultTimeout(...)`, `setDefaultPollInterval(...)` и т.д., чтобы не дублировать конфигурацию в каждом тесте.
- Библиотека потокобезопасна и предназначена именно для тестового кода, а не для использования в проде как замена реактивным/асинхронным механизмам.
- Есть поддержка `ignoreExceptions()`, `and().with()`, `alias("описание")` для более читаемых сообщений об ошибках.

## Пример / когда применяется

Типичный сценарий: после отправки POST-запроса, инициирующего асинхронную обработку, сервис через некоторое время должен изменить статус ресурса на "COMPLETED". Тест не может проверить это сразу же после ответа API — нужно подождать.

```java
import static org.awaitility.Awaitility.await;
import static org.assertj.core.api.Assertions.assertThat;
import java.time.Duration;

@Test
void shouldProcessOrderAsynchronously() {
    // given: отправили запрос, инициирующий асинхронную обработку
    String orderId = orderApiClient.createOrder(payload);

    // when / then: ждём, пока статус заказа не станет COMPLETED
    await()
        .atMost(Duration.ofSeconds(15))
        .pollInterval(Duration.ofMillis(500))
        .untilAsserted(() -> {
            OrderStatus status = orderApiClient.getOrderStatus(orderId);
            assertThat(status).isEqualTo(OrderStatus.COMPLETED);
        });
}
```

Аналогичный вариант с boolean-условием:

```java
await()
    .atMost(Duration.ofSeconds(15))
    .until(() -> orderApiClient.getOrderStatus(orderId) == OrderStatus.COMPLETED);
```

Такой подход применяется в API-тестировании при проверке: обработки сообщений из Kafka/RabbitMQ, срабатывания вебхуков, завершения фоновых джобов (batch, cron), синхронизации данных между микросервисами, готовности контейнера или сервиса в интеграционных тестах (совместно с Testcontainers).

## На что смотрит интервьюер

- Понимает ли кандидат, зачем нужен Awaitility, и может объяснить проблему `Thread.sleep` (нестабильность и потеря времени) как мотивацию для polling-подхода.
- Знает ли разницу между `until` (boolean-условие) и `untilAsserted` (готовые assert-выражения) и когда что удобнее использовать.
- Понимает роль `atMost`, `pollInterval`, `pollDelay` и то, что происходит при истечении таймаута (`ConditionTimeoutException`).
- Красный флаг: если кандидат утверждает, что Awaitility просто "ускоряет" тест или заменяет асинхронность синхронностью — это неверно, библиотека лишь избавляет от жёстких пауз и делает ожидание детерминированным по условию, а не по времени.
- Частый follow-up: как настроить глобальные дефолты таймаутов для всего проекта, как обрабатывать исключения внутри условия ожидания, как использовать Awaitility вместе с Hamcrest-матчерами или в связке с Testcontainers/RestAssured.
- Красный флаг: путаница между Awaitility и обычными ретраями HTTP-запросов на уровне клиента — это разные механизмы (Awaitility про ожидание состояния, а не про повтор запроса при сетевой ошибке).
