# Какие методы AssertJ для проверки коллекций (`hasSize`, `contains`, `extracting`) вы знаете?

**Коротко:** AssertJ предоставляет широкий набор методов для проверки коллекций: размер (`hasSize`), содержимое (`contains`, `containsExactly`, `containsExactlyInAnyOrder`), извлечение полей (`extracting`), фильтрацию (`filteredOn`) и проверку пустоты (`isEmpty`, `isNotEmpty`).

## Развёрнутый ответ

Ключевые методы AssertJ для работы с коллекциями (`List`, `Set` и т.д.):

- **`hasSize(int)`** — проверяет точное количество элементов в коллекции.
- **`isEmpty()` / `isNotEmpty()`** — проверка на пустоту коллекции.
- **`contains(T...)`** — проверяет, что коллекция содержит указанные элементы (порядок и наличие других элементов не важны).
- **`containsExactly(T...)`** — проверяет, что коллекция содержит ровно указанные элементы в точно таком же порядке.
- **`containsExactlyInAnyOrder(T...)`** — проверяет, что коллекция содержит ровно указанные элементы, но порядок не важен (часто применяется при проверке ответов API, где порядок элементов не гарантирован).
- **`containsOnly(T...)`** — проверяет, что коллекция содержит только указанные элементы (без учёта порядка и количества повторений).
- **`doesNotContain(T...)`** — проверяет отсутствие указанных элементов.
- **`extracting(String/Function)`** — извлекает конкретное поле или результат функции из каждого элемента коллекции и возвращает новый список для дальнейших проверок; удобно для проверки, например, только имён пользователей из списка объектов `User`.
- **`filteredOn(condition)`** — фильтрует коллекцию по условию перед дальнейшими проверками.
- **`allMatch(Predicate)` / `anyMatch(Predicate)` / `noneMatch(Predicate)`** — проверка условия для всех/хотя бы одного/ни одного элемента.
- **`first()` / `last()` / `element(index)`** — доступ к конкретному элементу коллекции для отдельной проверки.

Эти методы особенно полезны при тестировании API, когда ответ содержит массив объектов (например, список заказов), и нужно проверить не весь объект целиком, а конкретные поля или подмножество данных.

## Пример / когда применяется

Допустим, ответ API возвращает список пользователей:

```json
[
  { "id": 1, "name": "Иван", "age": 25 },
  { "id": 2, "name": "Пётр", "age": 30 }
]
```

Десериализация и проверки:

```java
List<User> users = given()
.when()
    .get("/api/users")
.then()
    .extract()
    .jsonPath().getList(".", User.class);

assertThat(users)
    .hasSize(2)
    .extracting(User::getName)
    .containsExactlyInAnyOrder("Иван", "Пётр");

assertThat(users)
    .extracting(User::getAge)
    .allMatch(age -> age > 18);

assertThat(users)
    .filteredOn(u -> u.getAge() > 26)
    .extracting(User::getName)
    .containsExactly("Пётр");
```

## На что смотрит интервьюер

- Знание разницы между `contains`, `containsExactly` и `containsExactlyInAnyOrder` — частый практический вопрос, так как это влияет на строгость проверки порядка.
- Умение использовать `extracting()` для проверки конкретных полей вместо сравнения целых объектов.
- Понимание, когда уместно использовать `filteredOn()` вместо ручного стриминга `.stream().filter()`.
- Красный флаг: если кандидат не знает разницы между `contains` и `containsExactly`, что может привести к ложноположительным тестам (тест "зелёный", хотя порядок или состав данных на самом деле неверный).
- Follow-up: "Как проверить, что список отсортирован?", "Как сравнить два списка объектов, игнорируя порядок и одно из полей?" (через `usingElementComparatorIgnoringFields` или `usingRecursiveFieldByFieldElementComparatorIgnoringFields`).
