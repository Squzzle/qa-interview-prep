# Что такое Akita? Какие основные компоненты в него входят (`akitaScenario`, `AkitaPage`, `AkitaStepDefinitions`)?

**Коротко:** Akita — open-source BDD-фреймворк для автоматизации UI-тестирования на Java, построенный поверх Selenide и Cucumber, который упрощает создание Page Object'ов через аннотации и предоставляет готовую инфраструктуру (хранилище состояния сценария, базовые шаги, конфигурацию окружений) для быстрого старта BDD-проекта.

## Развёрнутый ответ

Akita (разработан компанией alfa-laboratory/AlfaBank, распространяется как open-source) — фреймворк-надстройка, которая объединяет Selenide (обёртка над Selenium с более удобным и лаконичным API и встроенными ожиданиями) и Cucumber (BDD-движок для Gherkin) в единую готовую инфраструктуру, избавляя команду от необходимости писать боилерплейт-код для связки этих инструментов с нуля.

Основные компоненты Akita:

- **`akitaScenario`** — центральный объект-хранилище состояния текущего сценария (по сути, обёртка над `ScenarioContext`/thread-local хранилищем). Позволяет передавать переменные между шагами одного сценария (например, сохранить ID созданного пользователя в одном шаге и использовать его в другом), не прибегая к статическим полям. Реализует паттерн "Scenario Context" — данные живут в рамках одного прогона сценария и очищаются после его завершения, что обеспечивает изоляцию между сценариями при параллельном запуске.

- **`AkitaPage`** — базовый класс для Page Object'ов в Akita. Вместо ручного объявления `WebElement`/`SelenideElement` полей и написания `PageFactory.initElements`, `AkitaPage` использует аннотацию `@Name` для человекочитаемого описания элемента (используется в логах и отчётах при ошибках) в связке с `@FindBy` (стандартная Selenium-аннотация для локатора). Наследники `AkitaPage` автоматически инициализируются фреймворком, и к найденным элементам применяется рефлективный механизм для получения читаемых сообщений об ошибках ("Элемент 'Кнопка Войти' не найден", а не голый XPath).

- **`AkitaStepDefinitions`** (условное имя базовых/готовых степ-дефинишенов, поставляемых с фреймворком) — набор готовых, часто используемых шагов (например, "Открываю страницу", "Ожидаю появления элемента", "Кликаю на элемент с текстом") уже реализованных внутри фреймворка, чтобы командам не приходилось писать типовые шаги заново в каждом проекте. Команда переиспользует эти шаги в своих `.feature`-файлах и дополняет только специфичной для проекта бизнес-логикой.

Дополнительно Akita предоставляет:
- Механизм управления страницами через реестр (`Page` enum/аннотацию), позволяющий обращаться к нужной странице по имени без явного создания объектов.
- Интеграцию с конфигурацией через `.properties`-файлы (управление URL окружений, таймаутами, браузером).
- Встроенную поддержку скриншотов при падении теста и интеграцию с Allure-отчётами.

Ключевая ценность Akita — сокращение времени старта нового BDD/UI-автоматизационного проекта: команда получает готовую архитектуру (Selenide + Cucumber + Page Object на аннотациях + управление состоянием сценария) вместо написания этой инфраструктуры с нуля.

## Пример / когда применяется

Пример использования `akitaScenario` в step definition:

```java
@When("пользователь создаёт заявку")
public void createOrder() {
    String orderId = orderService.create();
    akitaScenario.getEnvironment().setVar("orderId", orderId);
}

@Then("заявка отображается в списке")
public void checkOrderInList() {
    String orderId = akitaScenario.getEnvironment().getVar("orderId").toString();
    orderListPage.checkOrderPresent(orderId);
}
```

Akita применяется в проектах на Java, где команда уже выбрала связку Selenide + Cucumber для BDD-тестирования UI и хочет получить готовую инфраструктуру вместо написания собственного каркаса, особенно в компаниях финансового сектора СНГ, где фреймворк исторически широко распространён.

## На что смотрит интервьюер

- Понимание, что Akita — не самостоятельный движок, а надстройка над Selenide + Cucumber, упрощающая интеграцию.
- Знает ли кандидат назначение `akitaScenario` как контейнера состояния сценария и зачем нужна изоляция данных между сценариями (особенно при параллельном запуске).
- Понимание разницы между `@Name` (человекочитаемое описание для логов) и `@FindBy` (технический локатор) в `AkitaPage`.
- Follow-up: как решается проблема "утечки" данных между параллельными сценариями, если используется общий контекст.
- Готовность честно сказать, если фреймворк не использовался на практике, но продемонстрировать понимание общей идеи (Page Object + BDD + shared context) — это ценится больше, чем выдуманные детали.
