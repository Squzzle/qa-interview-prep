# Что такое BDD и Gherkin? Для чего нужны `.feature` файлы и ключевые слова (Given, When, Then)?

**Коротко:** BDD (Behavior-Driven Development) — подход к разработке через описание поведения системы на понятном для бизнеса языке, а Gherkin — язык разметки для написания таких сценариев в файлах `.feature`, использующий ключевые слова Given/When/Then для описания предусловий, действий и ожидаемых результатов.

## Развёрнутый ответ

BDD (разработка через поведение) — это методология, развивающая идеи TDD, но смещающая фокус с "что тестировать технически" на "как система должна себя вести с точки зрения бизнеса и пользователя". Цель BDD — создать единый язык общения между бизнес-аналитиками, разработчиками и тестировщиками, чтобы требования, тесты и документация были синхронизированы и понятны всем участникам, включая нетехнических стейкхолдеров.

Gherkin — это специальный язык (DSL, structured natural language), на котором пишутся сценарии поведения. Он строится по строгому синтаксису, распознаваемому инструментами (Cucumber, SpecFlow, Behave), которые парсят текст сценария и сопоставляют шаги с реализующим их кодом (step definitions).

Основные ключевые слова Gherkin:

- **Feature** — описание функциональности верхнего уровня, обычно в начале файла, с кратким описанием "в чём ценность для пользователя".
- **Scenario** (или **Scenario Outline** для параметризованных сценариев) — конкретный тестовый случай/пример поведения.
- **Given** — описывает исходное состояние системы (предусловие) до начала действия. Например: "Given пользователь авторизован в системе".
- **When** — описывает действие, которое выполняет пользователь или триггерит система. Например: "When пользователь нажимает кнопку Купить".
- **Then** — описывает ожидаемый результат/проверку после выполнения действия. Например: "Then товар добавлен в корзину".
- **And / But** — используются для добавления дополнительных условий/действий/проверок без повторения Given/When/Then, для читаемости.
- **Background** — общие предусловия для всех сценариев в файле, чтобы не дублировать одинаковые Given-шаги.
- **Scenario Outline + Examples** — позволяет параметризовать один сценарий множеством наборов данных (таблица примеров).

`.feature`-файлы — это файлы с расширением `.feature`, содержащие сценарии, написанные на Gherkin. Они выполняют двойную роль: (1) живая документация — понятная бизнесу спецификация поведения системы, которая не расходится с реальным тестовым покрытием, потому что сама является исполняемым тестом; (2) исходный код для автотестов — каждый шаг сценария связывается через регулярное выражение или аннотацию с методом (step definition) на языке программирования (Java, C#, Python и т.д.), который реализует техническую логику выполнения шага.

Важный нюанс: Gherkin-сценарий должен описывать поведение с точки зрения пользователя/бизнеса ("что" происходит), а не технические детали реализации ("как" это сделано, например, "кликнуть на элемент с id=submit-button") — иначе теряется главное преимущество BDD, читаемость для нетехнических людей.

## Пример / когда применяется

Пример `.feature`-файла:

```gherkin
Feature: Авторизация пользователя

  Background:
    Given пользователь находится на странице логина

  Scenario: Успешный вход с корректными данными
    When пользователь вводит логин "user@test.com" и пароль "Pass123"
    And нажимает кнопку "Войти"
    Then пользователь перенаправлен на главную страницу
    And отображается приветствие "Добро пожаловать"

  Scenario Outline: Вход с некорректными данными
    When пользователь вводит логин "<login>" и пароль "<password>"
    And нажимает кнопку "Войти"
    Then отображается ошибка "<error>"

    Examples:
      | login          | password  | error                  |
      | wrong@test.com | Pass123   | Пользователь не найден |
      | user@test.com  | wrongpass | Неверный пароль        |
```

BDD применяется в проектах, где важна коммуникация между бизнесом и разработкой (Agile-команды, продукты с частыми изменениями требований), а также как способ построить регрессионное покрытие, понятное не только автоматизаторам.

## На что смотрит интервьюер

- Понимание, что BDD — это в первую очередь методология коммуникации, а не просто "ещё один способ писать тесты".
- Знание разницы между Scenario и Scenario Outline, умение объяснить, когда что использовать.
- Понимание антипаттерна — написание технических, императивных шагов вместо декларативных, ориентированных на пользователя.
- Follow-up: как избежать дублирования шагов между разными feature-файлами, как организовать степ-дефинишены (общие шаги, переиспользование).
- Знание, что `.feature`-файлы не содержат исполняемого кода напрямую — реализацию логики берут на себя step definitions.
