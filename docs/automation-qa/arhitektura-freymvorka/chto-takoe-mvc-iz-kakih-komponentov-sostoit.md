# Что такое MVC? Из каких компонентов состоит?

**Коротко:** MVC (Model-View-Controller) — архитектурный паттерн, разделяющий приложение на три компонента: Model (данные и бизнес-логика), View (представление/UI) и Controller (обработка пользовательского ввода и связь между Model и View), что снижает связанность кода и упрощает тестирование и поддержку.

## Развёрнутый ответ

MVC — один из самых распространённых архитектурных паттернов для построения приложений с пользовательским интерфейсом, а также применяется как концептуальная основа для организации фреймворков автоматизации тестирования.

Три компонента:

- **Model (модель).** Отвечает за данные и бизнес-логику приложения: получение, хранение, валидацию и обработку данных. Модель не знает ничего о том, как данные отображаются пользователю. В контексте автоматизации тестирования это может быть слой работы с тестовыми данными или доменными объектами (DTO).

- **View (представление).** Отвечает за отображение данных пользователю — UI, HTML-шаблоны, компоненты интерфейса. View получает данные от Model (напрямую или через Controller) и отображает их, но не содержит бизнес-логики.

- **Controller (контроллер).** Принимает пользовательский ввод (клики, HTTP-запросы), обрабатывает его, обращается к Model для получения/изменения данных и выбирает, какое View показать пользователю. Контроллер — это связующее звено между Model и View.

Принцип работы: пользователь взаимодействует с View → событие обрабатывается Controller → Controller обновляет Model → Model уведомляет View об изменении данных (напрямую или через Controller) → View перерисовывается с новыми данными.

Преимущества MVC:

1. **Разделение ответственности (Separation of Concerns)** — изменения в UI не затрагивают бизнес-логику и наоборот.
2. **Тестируемость** — Model и Controller можно тестировать независимо от UI (unit-тесты бизнес-логики без поднятия интерфейса).
3. **Параллельная разработка** — фронтенд и бэкенд команды могут работать независимо, если контракт (API/интерфейс) между слоями зафиксирован.
4. **Переиспользование** — одна и та же Model может обслуживать разные View (веб, мобильное приложение, API).

Для QA-инженера знание MVC важно по двум причинам: во-первых, тестируемое приложение часто построено на этом паттерне (Spring MVC, ASP.NET MVC, Ruby on Rails), и понимание архитектуры помогает точнее локализовать баги (это ошибка в данных/логике или в отображении?). Во-вторых, сам фреймворк автоматизации иногда концептуально проектируется по аналогии с MVC: Page Object как View, доменные сервисы как Model, Step Definitions/тестовый раннер как Controller.

## Пример / когда применяется

Пример на Spring MVC (Java):

```java
@Controller
public class UserController {

    @Autowired
    private UserService userService; // работа с Model

    @GetMapping("/users/{id}")
    public String getUser(@PathVariable Long id, Model model) {
        User user = userService.findById(id); // Model
        model.addAttribute("user", user);
        return "userView"; // имя View (шаблона)
    }
}
```

Здесь `UserService`/`User` — Model, `userView.html` (Thymeleaf-шаблон) — View, `UserController` — Controller.

Применяется в веб-разработке (Spring MVC, ASP.NET MVC, Django, Ruby on Rails) и как общая концепция при проектировании тестовых фреймворков.

## На что смотрит интервьюер

- Понимание разницы между MVC и MVP/MVVM (более продвинутые паттерны, где связь View и Model организована иначе — через Presenter или ViewModel).
- Может ли кандидат объяснить, зачем QA-инженеру знать MVC — не просто теоретический вопрос, а связь с локализацией дефектов (баг на уровне данных, логики или отображения).
- Умение провести аналогию между MVC и структурой фреймворка автоматизации (Page Object/Model/Controller-подобные компоненты).
- Частая ошибка — путаница, кто кому "сообщает": в классическом MVC Model не должна напрямую зависеть от View.
