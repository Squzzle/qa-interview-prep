# Что такое паттерн Page Object? Какие проблемы он решает?

**Коротко:** Page Object — это паттерн проектирования автотестов, при котором каждая страница (или значимый компонент UI) представляется отдельным классом, инкапсулирующим локаторы элементов и методы взаимодействия с ними; тесты работают через методы этого класса, а не напрямую с локаторами.

## Развёрнутый ответ

Page Object решает главную проблему автоматизации UI-тестов — дублирование и хрупкость кода. Без этого паттерна локаторы и логика взаимодействия со страницей разбросаны по множеству тестов. Если верстка меняется (меняется `id`, `class` или структура DOM), приходится править десятки мест. С Page Object локаторы и низкоуровневые действия хранятся в одном классе, поэтому изменение верстки требует правки только в одном месте.

Основные принципы паттерна:

- Каждая страница или логически обособленный блок UI (например, шапка сайта, форма логина, модальное окно) описывается отдельным классом.
- Класс содержит локаторы элементов страницы (как поля) и публичные методы, отражающие действия пользователя на этой странице (например, `login(username, password)`, `submitForm()`).
- Методы Page Object обычно возвращают либо `this` (для цепочек вызовов), либо другой Page Object — если действие приводит к переходу на другую страницу.
- Тестовый код (сценарии) не содержит прямых обращений к локаторам и низкоуровневым вызовам `driver.findElement`/`click`/`sendKeys` — вся эта логика скрыта внутри Page Object. Тесты работают на уровне бизнес-действий: "залогиниться", "добавить товар в корзину", "открыть настройки".
- Ассерты (проверки) обычно выносят из Page Object в сам тест (либо в отдельный слой ассертов), чтобы Page Object отвечал только за взаимодействие, а не за проверку результата — это разделение ответственности иногда оформляют отдельным паттерном (Page Object + Page Assert / Page Factory).

Проблемы, которые решает паттерн:

1. **Дублирование кода.** Один и тот же локатор или последовательность действий не копируется в каждый тест, а переиспользуется.
2. **Хрупкость тестов при изменении UI.** Изменения в верстке требуют правки в одном месте (в Page Object), а не во всех тестах, где элемент использовался.
3. **Читаемость и поддерживаемость.** Тесты становятся декларативными и читаются как описание бизнес-сценария, а не как набор технических вызовов.
4. **Разделение ответственности.** Логика взаимодействия со страницей отделена от логики самого теста (проверки условий, подготовки данных).
5. **Переиспользование между тестами.** Один и тот же Page Object используют разные тестовые сценарии.

## Пример / когда применяется

Пример на Java с Selenium:

```java
public class LoginPage {
    private WebDriver driver;

    @FindBy(id = "username")
    private WebElement usernameInput;

    @FindBy(id = "password")
    private WebElement passwordInput;

    @FindBy(css = "button[type=submit]")
    private WebElement loginButton;

    public LoginPage(WebDriver driver) {
        this.driver = driver;
        PageFactory.initElements(driver, this);
    }

    public HomePage login(String username, String password) {
        usernameInput.sendKeys(username);
        passwordInput.sendKeys(password);
        loginButton.click();
        return new HomePage(driver);
    }
}
```

В тесте:

```java
LoginPage loginPage = new LoginPage(driver);
HomePage homePage = loginPage.login("user", "pass");
assertTrue(homePage.isUserLoggedIn());
```

Паттерн применяется практически в любом UI-фреймворке автоматизации (Selenium, Playwright, Cypress, Appium) и является стандартом де-факто для больших проектов с большим количеством регрессионных тестов.

## На что смотрит интервьюер

- Понимает ли кандидат разницу между Page Object и Page Factory (Page Factory — это способ инициализации элементов через аннотации, частный случай реализации).
- Знает ли, что ассерты лучше не размещать внутри Page Object (частая ошибка — смешение проверок и взаимодействия).
- Может ли объяснить, почему методы должны возвращать Page Object следующей страницы (для построения цепочек и явного отражения навигации).
- Понимание проблемы избыточного дублирования кода без паттерна и как Page Object решает поддержку тестов при частых изменениях UI.
- Follow-up: чем Page Object отличается от Screenplay pattern или App Actions (Cypress) — более продвинутые альтернативы для сложных фреймворков.
