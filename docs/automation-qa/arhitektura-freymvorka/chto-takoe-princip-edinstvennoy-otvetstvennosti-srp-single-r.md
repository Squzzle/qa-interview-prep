# Что такое принцип единственной ответственности (SRP, Single Responsibility Principle)? Приведите пример в автотестах.

**Коротко:** SRP гласит, что у класса (модуля) должна быть только одна причина для изменения — то есть он должен решать одну чётко определённую задачу и не смешивать разные слои ответственности.

## Развёрнутый ответ

Принцип единственной ответственности — первый из принципов SOLID. Формулировка Роберта Мартина: "класс должен иметь только одну причину для изменения". Важно понимать, что речь не про "один метод в классе", а именно про причину изменения — то есть про то, ради какого требования бизнеса или технического слоя класс придётся модифицировать.

В контексте фреймворка автотестов ответственности принято разделять на несколько слоёв:

- **Взаимодействие с UI/API** — Page Object или API-клиент знает, как найти элемент, кликнуть по нему, отправить запрос.
- **Бизнес-логика теста** — сценарий (test case) описывает, что нужно проверить, не зная деталей реализации взаимодействия.
- **Генерация тестовых данных** — билдеры/фабрики данных отвечают только за создание валидных или невалидных сущностей.
- **Отчётность и логирование** — отдельный компонент отвечает за формирование шагов отчёта.
- **Конфигурация окружения** — отдельный модуль отвечает за чтение параметров запуска (URL, браузер, таймауты).

Если смешать эти ответственности в одном классе, например Page Object будет одновременно формировать тестовые данные, логировать шаги и обращаться к API для подготовки предусловий — при изменении логики логирования разработчику придётся трогать файл, который "должен" отвечать только за работу со страницей. Это увеличивает риск регрессий, усложняет ревью и делает класс невозможным для переиспользования в других контекстах (например, для мобильных тестов).

Нарушение SRP легко распознать по признакам: класс содержит десятки несвязанных методов, имя класса не отражает его реальной ответственности (например `Utils`, `Helper`, `TestBase` "обо всём"), при изменении одной фичи приходится редактировать файл, отвечающий за другую.

## Пример / когда применяется

```ts
// Плохо: страница отвечает за навигацию, валидацию данных и подготовку тестовых данных через API
class OrderPage {
  async createOrder(userId: string): Promise<void> {
    const payload = { userId, items: [{ sku: 'A1', qty: 1 }] }; // формирование данных
    await fetch('/api/orders', { method: 'POST', body: JSON.stringify(payload) }); // работа с API
    console.log(`Order created for ${userId}`); // логирование
  }
}

// Хорошо: каждая сущность отвечает за одну задачу
class OrderDataFactory {
  build(userId: string) {
    return { userId, items: [{ sku: 'A1', qty: 1 }] };
  }
}

class OrderApiClient {
  async createOrder(payload: unknown): Promise<void> {
    await fetch('/api/orders', { method: 'POST', body: JSON.stringify(payload) });
  }
}

class OrderPage {
  constructor(
    private readonly dataFactory: OrderDataFactory,
    private readonly apiClient: OrderApiClient,
  ) {}

  async prepareOrder(userId: string): Promise<void> {
    const payload = this.dataFactory.build(userId);
    await this.apiClient.createOrder(payload);
  }
}
```

SRP применяется буквально везде во фреймворке: при проектировании Page Object'ов, шагов (Step Definitions в BDD), утилит работы с датами/файлами, слоя логирования, драйвер-менеджера.

## На что смотрит интервьюер

- Кандидат должен объяснить SRP через "причину для изменения", а не просто сказать "один класс — одна функция".
- Способность привести реальный пример из Page Object Model, где смешаны UI-логика и подготовка данных.
- Follow-up: "Как SRP связан с поддерживаемостью тестов при частых изменениях UI?"
- Красный флаг — если кандидат считает, что SRP означает "класс должен содержать один метод" — это неверная и слишком буквальная трактовка.
