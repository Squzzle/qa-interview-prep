# Что такое принцип инверсии зависимостей (DIP, Dependency Inversion Principle) и как он помогает в автотестах?

**Коротко:** DIP гласит, что модули верхнего уровня не должны зависеть от модулей нижнего уровня — оба должны зависеть от абстракций, а абстракции не должны зависеть от деталей реализации; на практике это достигается через внедрение зависимостей (Dependency Injection) и программирование против интерфейсов.

## Развёрнутый ответ

DIP — последний принцип SOLID и во многом обобщающий: он завязан на использовании абстракций (интерфейсов, абстрактных классов) вместо конкретных реализаций, что напрямую поддерживает OCP и LSP. Формулировка Роберта Мартина состоит из двух частей:

1. Модули верхнего уровня (бизнес-логика, оркестрация) не должны напрямую зависеть от модулей нижнего уровня (детали реализации: конкретный драйвер, конкретная БД, конкретный HTTP-клиент). Оба должны зависеть от абстракций.
2. Абстракции не должны зависеть от деталей. Детали (конкретные реализации) должны зависеть от абстракций.

Важно не путать DIP с Dependency Injection (DI). DI — это техника (способ передачи зависимостей извне, через конструктор/сеттер/DI-контейнер), а DIP — это принцип проектирования (о направлении зависимостей между слоями). DI часто используется как инструмент для реализации DIP, но это не одно и то же.

Без DIP тесты и Page Object'ы напрямую создают и используют конкретные реализации — например, тест напрямую инстанциирует `ChromeDriver`, а не абстракцию `WebDriver`. Это приводит к тому, что:

- Тесты становится невозможно (или очень трудно) запускать на другом браузере/платформе без правки самого теста.
- Юнит-тестирование логики фреймворка усложняется, так как нельзя подставить мок/стаб вместо реального драйвера или реального HTTP-клиента.
- Изменение конкретной технологии (например, миграция с Selenium на Playwright) требует правок во всех местах, где эта технология напрямую используется, а не в одном адаптере.

DIP решает эту проблему: тесты и Page Object'ы работают с абстракцией (`interface Driver`, `interface HttpClient`), а конкретная реализация подставляется извне — через конфигурацию, DI-контейнер или фабрику. Это позволяет легко подменять реализацию (реальный API-клиент на моковый в компонентных тестах, Chrome на Firefox, продовое окружение на стейджинг) без изменения кода бизнес-логики тестов.

## Пример / когда применяется

```ts
// Плохо: тест напрямую зависит от конкретной реализации
class LoginTest {
  private driver = new ChromeDriverImpl(); // жёсткая зависимость от деталей

  async run(): Promise<void> {
    await this.driver.navigate('/login');
  }
}

// Хорошо: зависимость от абстракции, реализация внедряется извне (DIP + DI)
interface Driver {
  navigate(url: string): Promise<void>;
  click(selector: string): Promise<void>;
}

class ChromeDriverImpl implements Driver {
  async navigate(url: string): Promise<void> {}
  async click(selector: string): Promise<void> {}
}

class PlaywrightDriverImpl implements Driver {
  async navigate(url: string): Promise<void> {}
  async click(selector: string): Promise<void> {}
}

class LoginTest {
  constructor(private readonly driver: Driver) {} // зависимость от абстракции

  async run(): Promise<void> {
    await this.driver.navigate('/login');
  }
}

// Конкретная реализация подставляется на этапе конфигурации/запуска
const test = new LoginTest(new PlaywrightDriverImpl());
```

DIP применяется практически в любом фреймворке автоматизации: абстрагирование драйвера браузера, абстрагирование HTTP-клиента для API-тестов, абстрагирование источника тестовых данных (файл/БД/API), внедрение конфигурации через DI-контейнер (например, `InversifyJS`, `tsyringe` в TypeScript-проектах).

## На что смотрит интервьюер

- Чёткое разграничение DIP (принцип) и Dependency Injection (техника реализации).
- Умение объяснить, зачем зависеть от интерфейса, а не от конкретного класса, на примере смены драйвера или окружения.
- Follow-up: "Как DIP облегчает мокирование в тестах фреймворка?" — правильный ответ через возможность подставить фейковую реализацию интерфейса.
- Красный флаг — если кандидат утверждает, что DIP и DI — синонимы, или не может объяснить направление зависимостей между абстракцией и деталями.
