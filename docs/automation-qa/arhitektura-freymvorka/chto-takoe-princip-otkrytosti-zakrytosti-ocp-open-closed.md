# Что такое принцип открытости/закрытости (OCP, Open/Closed Principle) и как применять его во фреймворке автотестов?

**Коротко:** OCP означает, что программные сущности (классы, модули, функции) должны быть открыты для расширения, но закрыты для модификации — новую функциональность добавляют через расширение (наследование, композицию, реализацию интерфейсов), а не через изменение уже отлаженного и протестированного кода.

## Развёрнутый ответ

Принцип открытости/закрытости сформулирован Бертраном Мейером и переосмыслен в контексте ООП Робертом Мартином. Идея: если для добавления новой функциональности приходится редактировать существующий рабочий класс, это повышает риск сломать то, что уже работало и было проверено. Правильный подход — спроектировать код так, чтобы новое поведение добавлялось через новые классы/реализации, подключаемые к системе через уже существующие абстракции (интерфейсы, абстрактные классы), без изменения кода, который эти абстракции использует.

Технически OCP достигается за счёт:

- **Полиморфизма** — базовый класс или интерфейс описывает контракт, а конкретное поведение реализуется в наследниках.
- **Composition over inheritance** — вместо ветвления `if/switch` по типу используется внедрение стратегии (Strategy pattern).
- **Абстракций и точек расширения** — фреймворк заранее продумывает "хуки" для расширения (например, кастомные условия ожидания, кастомные репортеры).

Во фреймворке автотестов OCP особенно актуален для мест, которые точно будут расти: добавление новых браузеров/платформ, новых форматов отчётов, новых стратегий сравнения (assertions), новых источников тестовых данных. Если такие точки роста спроектированы без OCP, разработчик вынужден раз за разом дописывать `if (browserType === 'chrome') {...} else if (browserType === 'firefox') {...}` в одном и том же методе — при добавлении Safari нужно снова редактировать проверенный, стабильный код, что повышает риск регрессии.

Важный нюанс: OCP не означает "нельзя менять код никогда". Багфиксы и рефакторинг существующей логики — это нормально. Принцип касается именно добавления новой функциональности: новый вариант поведения не должен требовать правки уже отработавшего модуля, если архитектура заранее предусмотрела точку расширения.

## Пример / когда применяется

```ts
// Плохо: добавление нового типа отчёта требует правки существующего метода
class TestReporter {
  report(type: 'allure' | 'html', message: string): void {
    if (type === 'allure') {
      // логика Allure
    } else if (type === 'html') {
      // логика HTML-отчёта
    }
    // для добавления JSON-отчёта придётся снова редактировать этот метод
  }
}

// Хорошо: расширение через интерфейс, без изменения существующего кода
interface ReportStrategy {
  report(message: string): void;
}

class AllureReportStrategy implements ReportStrategy {
  report(message: string): void {
    /* логика Allure */
  }
}

class HtmlReportStrategy implements ReportStrategy {
  report(message: string): void {
    /* логика HTML-отчёта */
  }
}

// Новый формат добавляется новым классом, старый код не трогаем
class JsonReportStrategy implements ReportStrategy {
  report(message: string): void {
    /* логика JSON-отчёта */
  }
}

class TestReporter {
  constructor(private readonly strategy: ReportStrategy) {}

  report(message: string): void {
    this.strategy.report(message);
  }
}
```

Такой подход применяется при проектировании драйвер-менеджеров (поддержка новых браузеров через фабрику), систем валидации (новые кастомные matcher'ы для assertion-библиотеки), генераторов тестовых данных (новые провайдеры данных без изменения ядра).

## На что смотрит интервьюер

- Понимание, что OCP реализуется через абстракции, а не "просто не трогать код".
- Умение отличить рефакторинг/багфикс (это нормально) от добавления фичи через изменение существующего класса (нарушение OCP).
- Follow-up: "Как OCP связан со Strategy и Factory Method паттернами?"
- Красный флаг — расплывчатый ответ "код не должен меняться вообще" без понимания, о какой именно модификации идёт речь.
