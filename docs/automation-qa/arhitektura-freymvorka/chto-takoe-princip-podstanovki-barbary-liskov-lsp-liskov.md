# Что такое принцип подстановки Барбары Лисков (LSP, Liskov Substitution Principle)?

**Коротко:** LSP требует, чтобы объекты подкласса можно было использовать вместо объектов базового класса без изменения корректности программы — наследник должен полностью соответствовать контракту родителя, не сужать его гарантии и не менять ожидаемое поведение.

## Развёрнутый ответ

Принцип сформулирован Барбарой Лисков в 1987 году: "пусть `q(x)` — свойство, доказуемое для объектов `x` типа `T`. Тогда `q(y)` должно быть верно для объектов `y` типа `S`, где `S` — подтип `T`". Проще: если код работает с объектом базового типа, он должен продолжать корректно работать, если этот объект заменить экземпляром любого подкласса, — вызывающий код не должен знать и не должен ничего "доламывать" из-за подмены.

Нарушения LSP типично проявляются так:

- Подкласс выбрасывает исключение в методе, который базовый класс гарантированно выполняет без ошибок (например, `NotImplementedError` в переопределённом методе).
- Подкласс ослабляет предусловия или усиливает постусловия, меняя контракт (например, родитель принимает любую строку, наследник — только непустую, и это не задокументировано в базовом контракте).
- Подкласс меняет поведение метода так, что код, рассчитывающий на поведение базового класса, начинает работать неправильно (классический пример — "Квадрат не является Прямоугольником" в реализации, где `setWidth`/`setHeight` меняют оба измерения).

В контексте фреймворка автотестов LSP чаще всего всплывает в иерархиях Page Object'ов, драйверов и адаптеров под разные платформы (Web/Mobile/Desktop). Если базовый класс `BasePage` гарантирует, что метод `waitForLoad()` дожидается загрузки страницы и не выбрасывает исключение, а наследник `MobilePage` вдруг требует дополнительного вызова перед этим методом или бросает исключение в обычной ситуации — любой тест, написанный против интерфейса `BasePage`, может неожиданно упасть при работе с `MobilePage`. Это разрушает саму идею полиморфизма, ради которой строилась иерархия.

## Пример / когда применяется

```ts
// Плохо: наследник нарушает контракт базового класса
abstract class BaseDriverWrapper {
  abstract click(selector: string): Promise<void>;
}

class WebDriverWrapper extends BaseDriverWrapper {
  async click(selector: string): Promise<void> {
    // клик по элементу
  }
}

class ReadOnlyDriverWrapper extends BaseDriverWrapper {
  async click(selector: string): Promise<void> {
    // нарушение LSP: наследник не может выполнить контракт базового класса
    throw new Error('Click is not supported in read-only mode');
  }
}

// Хорошо: контракт разделён так, чтобы каждый наследник мог его честно выполнить
interface Clickable {
  click(selector: string): Promise<void>;
}

interface Readable {
  getText(selector: string): Promise<string>;
}

class WebDriverWrapper implements Clickable, Readable {
  async click(selector: string): Promise<void> {
    /* клик по элементу */
  }
  async getText(selector: string): Promise<string> {
    return '';
  }
}

class ReadOnlyDriverWrapper implements Readable {
  async getText(selector: string): Promise<string> {
    return '';
  }
  // ReadOnlyDriverWrapper просто не реализует Clickable — не обещает того, чего не может выполнить
}
```

LSP важен при проектировании иерархий `BasePage -> WebPage/MobilePage`, базовых классов для API-клиентов разных версий API, а также при создании кастомных `WebElement`-обёрток поверх Selenium/Playwright.

## На что смотрит интервьюер

- Умение объяснить LSP не просто как "наследник должен переопределять методы родителя", а через сохранение контракта (пред- и постусловий).
- Способность привести пример нарушения контракта именно во фреймворке автотестов (иерархия Page Object, драйверов).
- Follow-up: "Что делать, если поведение наследника объективно должно отличаться?" — правильный ответ: пересмотреть иерархию, использовать композицию или разделить интерфейсы (связь с ISP).
- Красный флаг — путаница LSP с простым наследованием без понимания, что нарушается именно ожидание вызывающего кода.
