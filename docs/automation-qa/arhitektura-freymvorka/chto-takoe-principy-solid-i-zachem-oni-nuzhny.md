# Что такое принципы SOLID и зачем они нужны в автоматизации тестирования?

**Коротко:** SOLID — это пять принципов объектно-ориентированного проектирования (SRP, OCP, LSP, ISP, DIP), которые помогают строить гибкую, расширяемую и поддерживаемую архитектуру фреймворка автотестов, снижая связанность кода и упрощая внесение изменений.

## Развёрнутый ответ

SOLID — аббревиатура из пяти принципов, сформулированных Робертом Мартином (Uncle Bob):

- **S — Single Responsibility Principle** (принцип единственной ответственности): у класса должна быть только одна причина для изменения.
- **O — Open/Closed Principle** (принцип открытости/закрытости): сущности должны быть открыты для расширения, но закрыты для модификации.
- **L — Liskov Substitution Principle** (принцип подстановки Барбары Лисков): объекты наследников должны быть взаимозаменяемы с объектами базового класса без нарушения корректности программы.
- **I — Interface Segregation Principle** (принцип разделения интерфейсов): много узкоспециализированных интерфейсов лучше, чем один универсальный.
- **D — Dependency Inversion Principle** (принцип инверсии зависимостей): модули верхнего уровня не должны зависеть от модулей нижнего уровня, оба должны зависеть от абстракций.

В автоматизации тестирования фреймворк — это такой же программный продукт, как и продукт под тестом, и он живёт долго: растёт число тестов, страниц, API-клиентов, меняются требования. Без соблюдения SOLID фреймворк быстро превращается в "спагетти-код": Page Object'ы содержат сотни методов, тесты дублируют логику, изменение одного локатора ломает десяток тестов, а добавление нового драйвера или окружения требует переписывания половины кодовой базы.

Применение SOLID даёт конкретные практические выгоды:

- Снижение стоимости поддержки — изменения в одном месте не требуют правок во всём проекте.
- Возможность безопасно расширять фреймворк (новые типы проверок, новые окружения, новые типы отчётов) без риска сломать существующие тесты.
- Упрощённое тестирование самого фреймворка (юнит-тесты для утилит, хелперов, билдеров данных).
- Более простое ревью кода и онбординг новых членов команды, потому что структура предсказуема.

## Пример / когда применяется

Пример нарушения и исправления на уровне архитектуры фреймворка:

```ts
// Плохо: один класс отвечает и за UI-действия, и за отправку данных в отчёт, и за логирование
class LoginPage {
  login(user: string, pass: string): void {
    // работа с элементами страницы
    // формирование отчёта Allure
    // запись в лог-файл
  }
}

// Хорошо: ответственности разделены (SRP), зависимости абстрагированы (DIP)
interface Reporter {
  step(message: string): void;
}

class LoginPage {
  constructor(private readonly reporter: Reporter) {}

  login(user: string, pass: string): void {
    this.reporter.step(`Логин пользователем ${user}`);
    // только работа с элементами страницы
  }
}
```

Такой подход применяется при проектировании базовых классов Page Object, фабрик драйверов, слоя работы с API, генераторов тестовых данных и модулей интеграции с системами отчётности.

## На что смотрит интервьюер

- Может ли кандидат объяснить каждый принцип своими словами, а не просто расшифровать аббревиатуру.
- Умеет ли кандидат привести пример именно из автоматизации тестирования, а не абстрактный пример из учебника.
- Понимание, что SOLID — не догма, а инструмент: избыточное применение (например, чрезмерная абстракция ради DIP) может усложнить простой фреймворк.
- Частый follow-up: "Приведите пример нарушения конкретного принципа в Page Object Model" — кандидат должен уметь показать антипаттерн и его исправление.
- Красный флаг — если кандидат путает принципы SOLID с паттернами проектирования (Singleton, Factory и т.д.) или не может связать теорию с практикой автотестов.
