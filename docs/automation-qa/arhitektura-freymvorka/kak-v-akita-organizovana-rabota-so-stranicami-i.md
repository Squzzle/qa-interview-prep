# Как в Akita организована работа со страницами и элементами через аннотации (`@Name`, `@FindBy`)?

**Коротко:** В Akita каждая страница описывается классом-наследником `AkitaPage`, где элементы объявляются как поля с двумя аннотациями — `@FindBy` (стандартная Selenium-аннотация, задаёт технический локатор элемента) и `@Name` (специфичная для Akita аннотация, задаёт человекочитаемое имя элемента для логов и отчётов об ошибках), а сама страница регистрируется и получается через единый реестр (`@Name`-аннотация на классе страницы + доступ через `akitaScenario.getPage(...)`).

## Развёрнутый ответ

Akita строит работу со страницами по расширенному паттерну Page Object, добавляя слой удобства поверх стандартного Selenium/Selenide подхода.

**Уровень страницы:**
Каждая страница описывается Java-классом, наследующим `AkitaPage`. Класс аннотируется `@Name("Название страницы")` — это имя используется фреймворком для регистрации страницы в едином реестре и для отображения в логах/отчётах, какая именно страница участвовала в шаге. Благодаря такой регистрации получить нужную страницу можно не через прямое создание объекта (`new LoginPage()`), а через универсальный метод доступа у `akitaScenario`, что снижает связанность кода и позволяет переиспользовать один и тот же экземпляр страницы в разных шагах сценария.

**Уровень элементов:**
Внутри класса страницы элементы объявляются как поля типа `SelenideElement` (тип Selenide, обёртка над `WebElement` с автоматическими умными ожиданиями). У каждого поля-элемента ставятся две аннотации:

- **`@FindBy`** — стандартная Selenium-аннотация (`@FindBy(id = "...")`, `@FindBy(css = "...")`, `@FindBy(xpath = "...")`), которая задаёт технический локатор, по которому Selenium/Selenide найдёт элемент в DOM.
- **`@Name("Человекочитаемое имя")`** — Akita-специфичная аннотация, задающая описательное имя элемента на естественном языке. Это имя не участвует в поиске элемента, а используется исключительно для читаемости: когда тест падает (элемент не найден, не кликабелен и т.д.), в логе/отчёте появляется сообщение с этим понятным именем ("Элемент 'Кнопка отправки формы' не найден"), а не техническим селектором, который сложно интерпретировать нетехническому человеку, читающему отчёт.

Инициализация полей происходит автоматически при создании страницы — фреймворк через рефлексию проходит по полям класса, инициализирует `SelenideElement` на основе `@FindBy`, и связывает с ним метаданные из `@Name` для последующего логирования.

**Дополнительные механизмы:**
- Akita поддерживает также аннотацию для блоков (composite-элементов/виджетов), позволяя описывать переиспользуемые части UI (например, "шапка сайта", "модальное окно") как вложенные объекты внутри Page-класса, также с `@Name` для читаемости.
- Ошибки, возникающие при работе с элементами, автоматически оборачиваются в исключения с понятным сообщением, содержащим имя из `@Name`, что существенно ускоряет диагностику падений в CI без необходимости лезть в исходный код Page Object.
- Такая связка (`@FindBy` + `@Name`) фактически реализует разделение "технического" и "смыслового" описания одного и того же элемента, что удобно как при отладке, так и при передаче отчётов нетехническим стейкхолдерам (в BDD-отчётах видно понятное описание шага и понятное описание элемента, участвовавшего в падении).

## Пример / когда применяется

Пример Page Object в Akita:

```java
@Name("Страница логина")
public class LoginPage extends AkitaPage {

    @Name("Поле логина")
    @FindBy(id = "username")
    public SelenideElement usernameInput;

    @Name("Поле пароля")
    @FindBy(id = "password")
    public SelenideElement passwordInput;

    @Name("Кнопка входа")
    @FindBy(css = "button[type=submit]")
    public SelenideElement loginButton;
}
```

Использование в step definition:

```java
@When("пользователь вводит логин {string} и пароль {string}")
public void enterCredentials(String login, String password) {
    LoginPage loginPage = akitaScenario.getPage(LoginPage.class);
    loginPage.usernameInput.setValue(login);
    loginPage.passwordInput.setValue(password);
}
```

Если `loginButton` не найдётся на странице, в отчёте появится ошибка вида "Элемент 'Кнопка входа' не найден по локатору button[type=submit]" — что сразу понятно и разработчику, и QA, читающему отчёт CI.

## На что смотрит интервьюер

- Понимание разницы ролей: `@FindBy` — для поиска элемента (техническая часть), `@Name` — для читаемости и диагностики (человеческая часть), а не дублирование одной и той же информации.
- Знает ли кандидат, что `AkitaPage` берёт на себя автоматическую инициализацию полей через рефлексию, избавляя от ручного `PageFactory.initElements`.
- Понимание, как обеспечивается переиспользование объекта страницы между шагами сценария (через реестр/`akitaScenario`, а не через `new` в каждом шаге).
- Follow-up: как организовать переиспользуемые составные компоненты (виджеты, повторяющиеся блоки) внутри Page Object в Akita.
- Красный флаг — если кандидат путает `@Name` с локатором или не понимает, что `@Name` не влияет на поиск элемента, а только на удобство логирования и отчётности.
