# Что такое артефакты (artifacts) в контексте CI/CD и для чего они используются?

**Коротко:** Артефакты — это файлы, созданные во время выполнения одного этапа пайплайна (собранное приложение, отчеты о тестах, логи, скриншоты, покрытие кода), которые сохраняются и могут быть переданы на следующий этап или скачаны позже для анализа.

## Развёрнутый ответ

Артефакт — любой результат работы job, который нужно сохранить после завершения этого job, потому что он либо используется на следующем этапе пайплайна, либо нужен человеку для анализа результата. Без артефактов каждый job пришлось бы пересобирать все с нуля, что медленно и не гарантирует, что деплой и тесты работают с одной и той же версией кода.

Типичные примеры артефактов:

- Скомпилированное приложение или Docker-образ, собранный на этапе Build и используемый на этапах Test и Deploy.
- Отчеты о прохождении тестов (JUnit XML, Allure-отчеты, HTML-репорты) — используются для публикации результатов в CI-системе и в системах трекинга качества.
- Скриншоты и видео падений UI-тестов (например, из Playwright или Cypress) — критичны для быстрой диагностики причины падения без необходимости локально воспроизводить тест.
- Логи выполнения (логи приложения, логи сервера, логи самого теста).
- Отчеты о покрытии кода (code coverage), которые могут агрегироваться и отправляться в системы вроде SonarQube или Codecov.

Артефакты обычно хранятся ограниченное время (retention policy, например 7-30 дней) из-за ограничений на дисковое пространство CI-системы, и настраиваются через ключевые слова в конфиге пайплайна (`artifacts` в GitLab CI, `actions/upload-artifact` в GitHub Actions, `archiveArtifacts` в Jenkins).

Важно отличать артефакты от кэша (cache): кэш используется для ускорения повторяющихся операций (например, кэш `node_modules` или зависимостей Maven) и не гарантированно сохраняется между разными ветками/пайплайнами, тогда как артефакт — это осознанно сохраняемый результат конкретного запуска пайплайна.

## Пример / когда применяется

Пример в GitLab CI — сохранение отчета Allure и скриншотов падений после прогона UI-тестов:

```yaml
e2e_tests:
  stage: test
  script:
    - npm run test:e2e
  artifacts:
    when: always
    paths:
      - allure-results/
      - test-results/screenshots/
    expire_in: 7 days
```

Пример в GitHub Actions:

```yaml
- name: Run tests
  run: npm run test:e2e
- name: Upload test report
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: test-report
    path: playwright-report/
```

Флаг `when: always` / `if: always()` важен: артефакты с логами и скриншотами падений нужно сохранять даже если тесты упали, а не только при успехе.

## На что смотрит интервьюер

Интервьюер проверяет понимание разницы между артефактом и кэшем, а также практическое умение настроить сохранение отчетов и скриншотов даже при падении тестов. Follow-up вопросы: "Как вы настраиваете сохранение скриншотов только для упавших тестов?", "Как передать артефакт сборки между job build и deploy?", "Какой retention policy вы используете и почему?". Красный флаг — если кандидат не знает, что по умолчанию артефакты падающего job могут не сохраниться без явного указания `when: always`/`if: always()`.
