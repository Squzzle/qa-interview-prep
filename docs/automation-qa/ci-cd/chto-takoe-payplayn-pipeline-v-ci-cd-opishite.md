# Что такое пайплайн (Pipeline) в CI/CD? Опишите его типовую структуру (Source, Build, Test, Deploy).

**Коротко:** Пайплайн — это описанная в конфигурационном файле последовательность автоматизированных этапов (stages), через которые проходит изменение кода от коммита в репозиторий до выпуска в production; типовые этапы — Source, Build, Test, Deploy.

## Развёрнутый ответ

Пайплайн CI/CD — это конвейер обработки изменений кода, состоящий из набора последовательных или параллельных этапов (stages), каждый из которых содержит одно или несколько заданий (jobs). Пайплайн запускается по триггеру: пуш в ветку, открытие пул-реквеста, по расписанию (cron) или вручную. Если любой этап падает, пайплайн обычно останавливается (fail fast), и команда получает уведомление.

Типовая структура:

- **Source (источник)** — этап получения кода: checkout репозитория, определение ветки/коммита, иногда кэширование зависимостей. Здесь же может выполняться валидация коммита (например, проверка формата commit message, линтер).
- **Build (сборка)** — компиляция или сборка приложения: установка зависимостей, транспиляция, сборка Docker-образа, статический анализ кода (linting, SAST). Результат этапа — артефакт (jar, бинарник, docker-image, dist-папка), который передается дальше по пайплайну без пересборки.
- **Test (тестирование)** — запуск автотестов разных уровней: unit-тесты, интеграционные тесты, статические проверки безопасности, а на более поздних стадиях — API- и UI-автотесты на развернутом окружении. Часто разбивается на несколько параллельных jobs для ускорения (parallel test execution, sharding).
- **Deploy (развертывание)** — доставка собранного и протестированного артефакта на целевое окружение: dev → staging → production. Может включать миграции БД, smoke-тесты после деплоя, canary-раскатку или blue-green деплой.

Между этапами часто добавляют gate-проверки (approval, quality gate по покрытию тестами), а результаты каждого этапа (логи, отчеты, артефакты) сохраняются для последующего анализа и трассируемости.

## Пример / когда применяется

Пример структуры в терминах GitLab CI (`.gitlab-ci.yml`):

```yaml
stages:
  - build
  - test
  - deploy

build_job:
  stage: build
  script:
    - docker build -t myapp:$CI_COMMIT_SHA .

unit_test_job:
  stage: test
  script:
    - npm run test:unit

deploy_staging:
  stage: deploy
  script:
    - ./deploy.sh staging
  environment: staging
```

QA-инженер обычно отвечает за содержимое этапа Test (написание и поддержка автотестов) и часто за smoke-проверки сразу после Deploy на staging/production.

## На что смотрит интервьюер

Интервьюер ждет, что кандидат назовет реальные этапы своего пайплайна, а не просто перечислит абстрактные слова. Уточняющие вопросы: "Какие тесты у вас запускаются на каком этапе?", "Что происходит, если тесты упали — блокируется ли деплой?", "Как ускоряете прогон тестов в пайплайне (параллелизация, шардинг)?". Красный флаг — если кандидат не понимает, что артефакт должен собираться один раз и переиспользоваться на всех последующих этапах, а не пересобираться заново.
