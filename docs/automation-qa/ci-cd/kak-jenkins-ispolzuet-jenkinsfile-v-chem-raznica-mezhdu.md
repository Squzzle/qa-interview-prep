# Как Jenkins использует `Jenkinsfile`? В чем разница между декларативным и скриптовым синтаксисом Pipeline?

**Коротко:** `Jenkinsfile` — это файл в репозитории проекта, описывающий пайплайн как код (Pipeline as Code) на языке Groovy; декларативный синтаксис имеет строгую фиксированную структуру (блоки `pipeline`, `stages`, `steps`) и проще в освоении, а скриптовый синтаксис — это полноценный Groovy-скрипт с произвольной логикой, более гибкий, но и более сложный в поддержке.

## Развёрнутый ответ

`Jenkinsfile` хранится в корне репозитория проекта (Pipeline as Code) и описывает весь процесс сборки, тестирования и деплоя. Jenkins считывает этот файл при запуске job типа "Pipeline" (или Multibranch Pipeline, который автоматически создает job для каждой ветки/PR репозитория) и выполняет описанные в нем стадии на одном или нескольких агентах (build agents/nodes).

**Декларативный синтаксис (Declarative Pipeline)** — более новый и рекомендуемый Jenkins способ описания пайплайна. Имеет строго заданную структуру верхнего уровня: обязательный блок `pipeline`, внутри которого объявляются `agent` (где выполнять), `stages` (список этапов), а внутри каждого stage — `steps` (команды). Дополнительно поддерживаются `post` (действия после завершения — например, отправка уведомлений), `environment` (переменные окружения), `options`, `parameters`, `when` (условия запуска stage). Синтаксис проще читать, легче валидировать и он лучше визуализируется в UI Jenkins (Blue Ocean).

**Скриптовый синтаксис (Scripted Pipeline)** — это исторически первый способ, представляющий собой произвольный Groovy-скрипт, обернутый в блок `node`. Дает полную свободу: циклы, условия, функции, обработку исключений (`try/catch`) в произвольных местах пайплайна. Такая гибкость мощнее, но сложнее в поддержке и чтении, требует более глубокого знания Groovy и чаще приводит к трудноотлаживаемым пайплайнам.

Практическая рекомендация в индустрии — использовать декларативный синтаксис как основной, а при необходимости выполнить нестандартную логику внутри одного шага — вставлять блок `script { ... }` внутри декларативного pipeline, комбинируя простоту декларативного подхода с гибкостью Groovy там, где это действительно нужно.

## Пример / когда применяется

Декларативный Pipeline:

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'npm ci'
            }
        }
        stage('Test') {
            steps {
                sh 'npm run test:e2e'
            }
        }
    }
    post {
        always {
            junit 'test-results/*.xml'
        }
    }
}
```

Скриптовый Pipeline (эквивалентная логика с произвольным Groovy):

```groovy
node {
    stage('Build') {
        sh 'npm ci'
    }
    stage('Test') {
        try {
            sh 'npm run test:e2e'
        } finally {
            junit 'test-results/*.xml'
        }
    }
}
```

## На что смотрит интервьюер

Интервьюер проверяет, что кандидат реально писал `Jenkinsfile`, а не только слышал термин. Follow-up вопросы: "Когда вы бы выбрали скриптовый синтаксис вместо декларативного?", "Как в декларативном pipeline вставить кастомную Groovy-логику?", "Чем Multibranch Pipeline отличается от обычного Pipeline job?". Красный флаг — если кандидат не может объяснить структуру декларативного пайплайна (`pipeline` → `stages` → `steps`) или считает, что разницы между синтаксисами нет.
