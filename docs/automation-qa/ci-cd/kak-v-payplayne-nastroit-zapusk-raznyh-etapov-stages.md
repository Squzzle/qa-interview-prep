# Как в пайплайне настроить запуск разных этапов (stages) и заданий (jobs)?

**Коротко:** Этапы (stages) задают порядок выполнения группы заданий и выполняются последовательно, а задания (jobs) внутри одного этапа по умолчанию выполняются параллельно; порядок и зависимости управляются ключевыми словами вроде `stage`, `needs`/`depends_on`, а условия запуска — через `rules`/`only`/`if`/`when`.

## Развёрнутый ответ

Stage — логическая группа заданий, определяющая порядок выполнения на верхнем уровне: пока не завершатся (успешно) все job текущего stage, следующий stage не начнется. Внутри одного stage все job по умолчанию запускаются параллельно, что ускоряет пайплайн (например, разные группы автотестов можно гонять одновременно на разных раннерах).

Job — конкретное задание с набором шагов (`steps`/`script`), которое привязывается к определенному этапу через ключевое слово `stage` (GitLab) или через `jobs:` верхнего уровня с зависимостями `needs` (GitHub Actions, где стадийность эмулируется через граф зависимостей, а не строгие "stages").

Для управления порядком и условиями запуска используются:

- **Явные зависимости** — `needs` в GitHub Actions или `needs`/порядок stages в GitLab CI, позволяющие job из одного stage не ждать все job предыдущего, а зависеть только от конкретных.
- **Условия запуска** — `rules`/`only`/`except` в GitLab CI, `if` в GitHub Actions: позволяют запускать job только для определенной ветки, тега, события (push/merge request), или пропускать при изменении только определенных файлов.
- **Ручные подтверждения** — `when: manual` в GitLab CI или `environment` с required reviewers в GitHub Actions, чтобы задание (например, деплой в prod) ждало нажатия кнопки человеком.
- **Матрицы (matrix)** — позволяют размножить один job на несколько параллельных вариантов (например, разные версии Node.js или разные браузеры для UI-тестов).

Для тестовых пайплайнов частая практика — разбить тесты на несколько job внутри одного stage `test` (например, `unit_tests`, `api_tests`, `ui_tests`), чтобы они шли параллельно и не увеличивали общее время пайплайна.

## Пример / когда применяется

GitLab CI: несколько stage, внутри stage `test` — параллельные job:

```yaml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script: [ "docker build -t app:$CI_COMMIT_SHA ." ]

unit_tests:
  stage: test
  script: [ "npm run test:unit" ]

api_tests:
  stage: test
  script: [ "npm run test:api" ]

deploy_prod:
  stage: deploy
  script: [ "./deploy.sh prod" ]
  when: manual
  only:
    - main
```

GitHub Actions: зависимости через `needs`, матрица для параллельных браузеров:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps: [ { uses: actions/checkout@v4 } ]

  test:
    needs: build
    runs-on: ubuntu-latest
    strategy:
      matrix:
        browser: [chromium, firefox, webkit]
    steps:
      - run: npx playwright test --project=${{ matrix.browser }}
```

## На что смотрит интервьюер

Интервьюер проверяет понимание разницы между последовательностью stages и параллельностью job внутри stage, а также умение настраивать условный запуск и матрицы. Follow-up вопросы: "Как ускорить пайплайн, если тесты занимают 40 минут?", "Как настроить, чтобы деплой в prod требовал ручного подтверждения?", "Как сделать так, чтобы job зависел не от всего предыдущего stage, а от конкретного job?". Красный флаг — если кандидат считает, что все job всегда выполняются строго последовательно, не понимая параллелизм внутри stage.
