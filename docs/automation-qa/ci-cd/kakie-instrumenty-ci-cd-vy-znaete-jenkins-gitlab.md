# Какие инструменты CI/CD вы знаете? (Jenkins, GitLab CI, GitHub Actions, TeamCity)

**Коротко:** Основные инструменты CI/CD — Jenkins (self-hosted, максимально гибкий через плагины и `Jenkinsfile`), GitLab CI (встроен в GitLab, конфигурация в `.gitlab-ci.yml`), GitHub Actions (встроен в GitHub, конфигурация в `.github/workflows`) и TeamCity (продукт JetBrains с удобным UI и мощной поддержкой .NET/Java-проектов).

## Развёрнутый ответ

**Jenkins** — старейший и один из самых распространенных open-source серверов автоматизации. Устанавливается самостоятельно (self-hosted), обладает огромной экосистемой плагинов (более 1800), поддерживает как декларативный, так и скриптовый (Groovy-based) синтаксис `Jenkinsfile`. Плюсы: максимальная гибкость, богатая настройка агентов и распределенных сборок (master/agent архитектура). Минусы: требует администрирования сервера, обновления плагинов иногда ломают совместимость, UI считается устаревшим.

**GitLab CI/CD** — встроен непосредственно в GitLab, конфигурация описывается в файле `.gitlab-ci.yml` в корне репозитория. Плюсы: единая экосистема (код, issues, CI, реестр контейнеров, security-сканирование — все в одном продукте), удобные Merge Request pipelines, встроенный Docker registry и Auto DevOps. Минусы: сильная привязка к GitLab, self-hosted вариант требует ресурсов на runners.

**GitHub Actions** — нативная CI/CD-система GitHub, workflows описываются YAML-файлами в `.github/workflows/`. Плюсы: огромный маркетплейс готовых actions, простая интеграция с GitHub-событиями (issues, PR, релизы), бесплатные минуты для публичных репозиториев. Минусы: привязка к GitHub, сложные сценарии требуют написания собственных composite/reusable actions.

**TeamCity** — продукт JetBrains, может быть self-hosted или облачным. Отличается удобным визуальным UI для настройки пайплайнов без написания YAML (хотя поддерживается и Kotlin DSL), хорошей поддержкой Java/.NET экосистем, встроенным механизмом build chains и тестовой аналитикой (flaky test detection из коробки). Минусы: коммерческая лицензия для крупных команд, менее популярен в open-source сообществе.

Также стоит упомянуть CircleCI, Bamboo, Azure DevOps Pipelines, Travis CI — как альтернативы с похожими принципами, но разным набором интеграций и облачных особенностей.

## Пример / когда применяется

| Инструмент | Конфигурация | Типичный кейс использования |
|---|---|---|
| Jenkins | `Jenkinsfile` (Groovy) | Крупные enterprise-проекты, кастомные пайплайны, self-hosted инфраструктура |
| GitLab CI | `.gitlab-ci.yml` | Команды, уже использующие GitLab как основную платформу |
| GitHub Actions | `.github/workflows/*.yml` | Open-source проекты, команды на GitHub |
| TeamCity | UI / Kotlin DSL | Java/.NET-проекты, компании с JetBrains-стеком |

Пример выбора: если тестовый фреймворк на Java и команда уже использует IntelliJ/JetBrains-инструменты — логичен TeamCity; если весь код и issue-tracking в GitLab — используют GitLab CI, чтобы не плодить лишние системы.

## На что смотрит интервьюер

Интервьюер хочет услышать не просто список названий, а понимание отличий и опыт реальной работы хотя бы с одним инструментом: как выглядит конфиг, как настроены раннеры/агенты, как хранятся секреты. Follow-up вопросы: "Чем декларативный Jenkinsfile отличается от скриптового?", "Как в GitHub Actions переиспользовать шаги между workflows (reusable workflows, composite actions)?", "Как масштабируете раннеры под нагрузку?". Красный флаг — если кандидат называет инструменты, но не может описать структуру простого конфиг-файла ни в одном из них.
