# Какие плюсы использования Docker для запуска автотестов?

**Коротко:** Docker дает воспроизводимое, изолированное и одинаковое для всех окружение запуска тестов ("работает одинаково у всех"), быстрый и дешевый запуск множества параллельных изолированных сред, легкую версионируемость окружения через Dockerfile и простую очистку состояния между прогонами.

## Развёрнутый ответ

**Воспроизводимость окружения.** Dockerfile описывает точную версию ОС, языка выполнения, браузеров, драйверов и зависимостей. Это устраняет классическую проблему "у меня локально тесты падают/проходят, а на CI — наоборот" — тесты выполняются в одинаковом контейнере и на локальной машине разработчика, и на CI-раннере.

**Изоляция.** Каждый прогон тестов может выполняться в чистом, только что созданном контейнере, без остатков состояния от предыдущих запусков (кэш браузера, временные файлы, испорченные данные БД). Это снижает флакейность (flakiness) тестов, вызванную "грязным" окружением.

**Параллелизация и масштабируемость.** Контейнеры поднимаются за секунды и занимают немного ресурсов, поэтому можно легко запускать десятки контейнеров параллельно (например, шардинг UI-тестов по браузерам или по группам тестов), сокращая общее время прогона регрессии.

**Простая интеграция зависимых сервисов.** С помощью Docker Compose можно поднять полное тестовое окружение — приложение, базу данных, очередь сообщений, мок-сервисы — одной командой, что особенно важно для интеграционных и end-to-end тестов, не требуя ручной настройки инфраструктуры.

**Версионирование окружения.** Dockerfile/docker-compose.yml хранятся в репозитории вместе с кодом тестов, поэтому изменения окружения (например, обновление версии Selenium/браузера) проходят через тот же процесс ревью и версионирования, что и код.

**Простота очистки.** После завершения тестов контейнер просто уничтожается — не остается "мусора" в системе (в отличие от установки браузеров/зависимостей напрямую на раннер), что упрощает поддержку CI-инфраструктуры.

## Пример / когда применяется

Пример `docker-compose.yml` для e2e-тестов с приложением, базой и Selenium Grid/Playwright-контейнером:

```yaml
version: "3.9"
services:
  app:
    build: .
    ports:
      - "3000:3000"
    depends_on:
      - db
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: test
  tests:
    image: mcr.microsoft.com/playwright:v1.45.0-jammy
    volumes:
      - ./tests:/tests
    working_dir: /tests
    command: npx playwright test
    depends_on:
      - app
```

В CI-пайплайне job запускает `docker compose up --abort-on-container-exit`, что поднимает все зависимости, прогоняет тесты и завершает работу с корректным exit-кодом для отчета в пайплайне.

## На что смотрит интервьюер

Интервьюер ждет конкретных примеров, а не общих слов "Docker — это удобно". Follow-up вопросы: "Как вы поднимаете зависимые сервисы (БД, моки) для интеграционных тестов через Docker?", "Как ускоряете сборку образа для тестов (кэширование слоев, multi-stage build)?", "Как решаете проблему флакейности UI-тестов внутри контейнера (например, headless-режим браузера)?". Красный флаг — если кандидат не может объяснить, зачем изолировать тестовое окружение, или никогда не работал с Docker Compose для поднятия зависимостей теста.
