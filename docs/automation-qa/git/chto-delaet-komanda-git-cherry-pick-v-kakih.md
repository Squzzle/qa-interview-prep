# Что делает команда `git cherry-pick`? В каких сценариях она может быть полезна?

**Коротко:** `git cherry-pick` применяет изменения одного конкретного коммита из другой ветки к текущей ветке, создавая новый коммит с этими изменениями, без слияния всей ветки целиком.

## Развёрнутый ответ

`git cherry-pick <commit-hash>` берёт diff указанного коммита и применяет его поверх текущего HEAD, создавая новый коммит с теми же изменениями (сообщением, содержимым), но с новым хешем и новым родителем. В отличие от `merge` или `rebase`, cherry-pick переносит не всю историю ветки, а только выбранный(е) коммит(ы) — точечная операция.

Основные сценарии применения:

- **Перенос критического исправления (hotfix) в другую ветку**: например, баг исправлен в ветке `hotfix/1.2.1` и слит в `main`, но нужно также применить именно это исправление в `develop` или в ветке предыдущего релиза, не сливая всю ветку целиком.
- **Выборочный перенос фичи**: если в feature-ветке сделано несколько коммитов, но в релиз нужно взять только один конкретный (например, только исправление автотеста, а не незавершённую фичу).
- **Восстановление случайно удалённого коммита**: если коммит остался в другой ветке или в reflog, его можно cherry-pick-нуть обратно.
- **Перенос изменений между несвязанными или давно разошедшимися ветками**, где полноценный merge/rebase невозможен или нежелателен из-за большого количества конфликтов.

При конфликтах cherry-pick останавливается так же, как merge/rebase: нужно вручную разрешить конфликт, сделать `git add`, затем `git cherry-pick --continue` (или `git cherry-pick --abort`, чтобы отменить операцию).

Важный нюанс: cherry-pick создаёт **новый** коммит с новым хешем, поэтому при последующем merge между ветками Git может не распознать, что изменения уже были перенесены, и в редких случаях привести к дублированию/конфликтам — это стоит держать в уме при частом использовании cherry-pick в связке с merge.

## Пример / когда применяется

```bash
# Посмотреть хеш нужного коммита в hotfix-ветке
git log hotfix/1.2.1 --oneline

# Переключиться в ветку develop и перенести туда конкретный коммит
git checkout develop
git cherry-pick a1b2c3d

# Перенос нескольких коммитов подряд
git cherry-pick a1b2c3d d4e5f6a

# При конфликте
git status               # посмотреть, какие файлы конфликтуют
# ... вручную разрешить конфликт ...
git add <resolved-file>
git cherry-pick --continue
```

Практический пример для QA: тестировщик обнаружил, что исправление бага в автотесте (один коммит) находится в ветке `release/2.0`, но такое же исправление нужно и в `develop`, где ещё не завершён следующий релиз. Вместо слияния всей `release/2.0` в `develop` (что может притащить незавершённые изменения), делается `git cherry-pick` только нужного коммита.

## На что смотрит интервьюер

- Понимание, что cherry-pick создаёт новый коммит с новым хешем (в отличие от простого переноса того же коммита).
- Знание типичных сценариев: перенос hotfix, выборочный перенос отдельных изменений между ветками.
- Умение объяснить, как разрешать конфликты при cherry-pick.
- Follow-up: чем cherry-pick отличается от rebase; риски дублирования при последующем merge.
- Red flag: непонимание, что cherry-pick — точечная операция для одного/нескольких коммитов, а не способ слияния целых веток.
