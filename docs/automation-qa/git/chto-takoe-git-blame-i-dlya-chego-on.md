# Что такое `git blame` и для чего он используется?

**Коротко:** `git blame` показывает построчную историю файла — кто, каким коммитом и когда последний раз изменял каждую конкретную строку кода. Используется для поиска автора и причины появления бага или конкретного изменения.

## Развёрнутый ответ

`git blame <файл>` выводит содержимое файла построчно, добавляя перед каждой строкой:

- сокращённый хэш коммита, в котором эта строка была последний раз изменена;
- имя автора этого коммита;
- дату изменения;
- саму строку кода.

Это удобно, когда нужно понять, кто и почему добавил определённый кусок кода — например, странную проверку, магическое число или закомментированный участок. Зная коммит, можно перейти к его полному описанию (`git show <хэш>`) и увидеть контекст: сообщение коммита, связанную задачу, другие изменённые файлы.

Важные нюансы и полезные флаги:

- `-L <начало>,<конец>` — ограничить вывод конкретным диапазоном строк, что удобно для больших файлов.
- `-w` — игнорировать изменения в пробелах (whitespace), чтобы не "спотыкаться" о форматирование.
- `-C` / `-M` — учитывать перемещение и копирование кода между файлами, чтобы находить "настоящего" автора строки, даже если она была перенесена из другого файла.
- `--ignore-rev <хэш>` (начиная с Git 2.23) — игнорировать конкретный коммит при поиске (полезно, если был массовый рефакторинг/переформатирование, который "испортил" blame, указывая на него, а не на реального автора логики).
- Ограничение: `git blame` показывает только последнее изменение строки — если строка менялась 10 раз, будет виден только самый последний коммит. Для полной истории строки используют связку с `git log -p` или `git log --follow -L`.

## Пример / когда применяется

Использование в реальной работе:

```bash
# посмотреть, кто и когда менял каждую строку файла
git blame src/utils/validate.js

# посмотреть только строки 40-60
git blame -L 40,60 src/utils/validate.js

# посмотреть полную информацию о коммите, найденном через blame
git show a1b2c3d
```

Типичный сценарий: в продакшене обнаружен баг в функции расчёта скидки. QA-инженер или разработчик запускает `git blame` по файлу с этой функцией, находит коммит, который последний раз менял подозрительную строку, читает сообщение коммита и связанную задачу в трекере, чтобы понять первоначальный замысел изменения и было ли оно намеренным.

## На что смотрит интервьюер

- Понимает ли кандидат, что `blame` показывает только последнее изменение строки, а не полную историю правок.
- Знает ли о проблеме "испорченного" blame после массового форматирования кода и как её обойти (`-w`, `--ignore-rev`, файл `.git-blame-ignore-revs`).
- Может ли объяснить практический сценарий использования (поиск причины бага, поиск ответственного за код для code review или вопроса).
- Путают ли `git blame` с `git log` — интервьюер проверяет, различает ли кандидат построчный анализ файла и просмотр истории коммитов репозитория в целом.
- Красный флаг: если кандидат описывает `git blame` как инструмент "обвинения" коллег в негативном смысле, а не как техническое средство анализа истории кода.
