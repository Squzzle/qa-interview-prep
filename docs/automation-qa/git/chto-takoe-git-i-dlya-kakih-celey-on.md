# Что такое Git и для каких целей он используется в разработке и тестировании?

**Коротко:** Git — распределённая система контроля версий (VCS), которая отслеживает историю изменений файлов, позволяет нескольким людям параллельно работать над кодом и тест-артефактами, а также откатываться к любой версии.

## Развёрнутый ответ

Git — это инструмент управления версиями исходного кода, созданный Линусом Торвальдсом в 2005 году. Он хранит полную историю изменений проекта в виде снапшотов (снимков состояния файлов), а не дельт-патчей, что делает большинство операций быстрыми и локальными, поскольку у каждого разработчика есть полная копия репозитория, включая всю историю.

Основные задачи, которые решает Git в разработке и тестировании:

- Версионирование исходного кода приложения, тестовых фреймворков, автотестов, конфигураций CI/CD.
- Совместная работа команды: несколько человек могут параллельно вносить изменения в разные ветки без конфликта друг с другом.
- История изменений: можно посмотреть, кто, когда и почему внёс правку (полезно при поиске регрессий — `git bisect`, `git blame`).
- Откат изменений: если автотест или код сломался, можно быстро вернуться к рабочей версии.
- Ветвление (branching): позволяет разрабатывать новую функциональность или писать тесты для неё изолированно, не затрагивая стабильный код.
- Резервное копирование: удалённый репозиторий (GitHub, GitLab, Bitbucket) служит бэкапом и точкой синхронизации команды.

Для QA-инженера и автоматизатора Git особенно важен, потому что автотесты, page objects, конфигурации и тестовые данные — это тоже код, который нужно версионировать, ревьюить через pull request и интегрировать в CI/CD пайплайны.

## Пример / когда применяется

Типичный рабочий процесс автоматизатора:

1. Клонировать репозиторий с автотестами: `git clone`.
2. Создать ветку под новую задачу: `git checkout -b feature/login-tests`.
3. Написать/поправить тесты, закоммитить: `git add . && git commit -m "add login negative tests"`.
4. Отправить изменения на сервер и создать pull request: `git push`.
5. После код-ревью изменения вливаются в основную ветку и автоматически запускаются в CI.

## На что смотрит интервьюер

- Понимает ли кандидат разницу между VCS вообще и именно распределённой моделью Git.
- Может ли объяснить, зачем QA-инженеру Git, а не только разработчикам (совместная работа над тест-кодом, CI/CD, ревью тестов).
- Красный флаг: если кандидат говорит про Git только как про "хранилище кода на GitHub", не понимая локальной природы репозитория и офлайн-работы.
- Follow-up: чем Git отличается от систем типа SVN, что такое staging area.
