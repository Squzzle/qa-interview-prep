# Что такое `git squash` и как его можно выполнить?

**Коротко:** Squash — это объединение нескольких коммитов в один, обычно применяемое перед слиянием feature-ветки, чтобы получить чистую и понятную историю без промежуточных "WIP"-коммитов; выполняется через интерактивный rebase (`git rebase -i`) или через squash-merge при слиянии pull request.

## Развёрнутый ответ

Строго говоря, отдельной команды `git squash` не существует — это операция, выполняемая либо через интерактивный rebase, либо через специальный режим merge. Squash полезен, когда в процессе разработки фичи накопилось много мелких, "технических" коммитов (например, "fix typo", "wip", "fix test", "address review comments"), которые не несут самостоятельной ценности для истории проекта. Объединение их в один осмысленный коммит делает историю `main`/`develop` чище и облегчает последующий анализ (`git log`, `git bisect`, ревью изменений).

Способы выполнения squash:

1. **Интерактивный rebase** (`git rebase -i HEAD~N` или `git rebase -i <base-commit>`): открывается список последних N коммитов, где для всех коммитов, кроме первого, нужно заменить `pick` на `squash` (или короткое `s`). Git объединит их в один коммит и предложит отредактировать итоговое сообщение коммита.
2. **Squash-merge при слиянии Pull Request** (доступно в GitHub/GitLab/Bitbucket как опция "Squash and merge"): все коммиты feature-ветки объединяются в один коммит поверх целевой ветки при слиянии, без создания отдельного merge-коммита с несколькими родителями.
3. **`git merge --squash <branch>`**: применяет все изменения указанной ветки к текущей рабочей директории и staging area одним пакетом, но не создаёт коммит автоматически — пользователь сам делает `git commit` после, формируя один итоговый коммит.

Важно: как и rebase, squash переписывает историю (создаёт новые коммиты с новыми хешами), поэтому его следует выполнять **до** публикации коммитов в общей ветке или сразу использовать через безопасный squash-merge на уровне pull request, а не вручную переписывать уже опубликованную историю.

## Пример / когда применяется

```bash
# Вариант 1: интерактивный rebase для последних 4 коммитов
git rebase -i HEAD~4
# в открывшемся редакторе:
# pick a1b2c3d add login test skeleton
# squash d4e5f6a fix typo
# squash b2c3d4e address review comments
# squash e5f6a7b fix flaky assertion
# -> сохранить, затем отредактировать итоговое commit message

# Вариант 2: squash через merge --squash
git checkout develop
git merge --squash feature/login-tests
git commit -m "add comprehensive login test suite"

# Вариант 3: squash-merge через интерфейс GitHub при мерже pull request
# (кнопка "Squash and merge" вместо "Merge pull request")
```

Практический сценарий: автоматизатор за неделю работы над тестами накопил 15 коммитов вида "wip", "fix", "typo". Перед созданием pull request он делает `git rebase -i` и объединяет их в 1-2 логических коммита: "add login test suite" и "add checkout regression tests", что упрощает код-ревью.

## На что смотрит интервьюер

- Знание, что squash — не отдельная команда, а техника, реализуемая через rebase или специальный merge-режим.
- Умение выполнить squash практически (интерактивный rebase, замена `pick` на `squash`).
- Понимание, что squash переписывает историю и небезопасен для уже опубликованных общих веток.
- Follow-up: разница между squash-merge и обычным merge с точки зрения истории; когда squash нежелателен (если важно сохранить гранулярную историю коммитов, например для аудита).
- Red flag: попытка объяснить squash как встроенную git-команду или незнание, что при squash меняются хеши коммитов.
