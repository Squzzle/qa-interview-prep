# Как с помощью Git найти коммит, который внес ошибку (`git bisect`)?

**Коротко:** `git bisect` выполняет бинарный поиск по истории коммитов между заведомо "хорошим" и "плохим" состоянием репозитория, позволяя за логарифмическое число шагов найти конкретный коммит, который внёс баг.

## Развёрнутый ответ

Идея `git bisect` — вместо того, чтобы проверять коммиты один за другим последовательно, Git берёт диапазон коммитов между известной рабочей версией (`good`) и версией с багом (`bad`), и на каждом шаге переключается на коммит из середины этого диапазона. Тестировщик или разработчик проверяет, воспроизводится ли баг на этом коммите, и сообщает Git результат — после чего диапазон поиска сужается вдвое. Такой подход требует всего O(log n) проверок вместо O(n) при полном переборе истории.

Процесс работы:

1. `git bisect start` — запустить сессию поиска.
2. `git bisect bad` — пометить текущий коммит (обычно HEAD) как содержащий баг.
3. `git bisect good <хэш-или-тег>` — указать заведомо рабочий коммит в прошлом (например, последний релизный тег).
4. Git автоматически переключается (checkout) на коммит из середины диапазона.
5. Тестировщик собирает/запускает проект на этом коммите и проверяет наличие бага.
6. По результату вводит `git bisect good` или `git bisect bad`.
7. Шаги 4-6 повторяются, пока Git не сузит диапазон до одного коммита и не сообщит: "first bad commit is ...".
8. `git bisect reset` — завершить сессию и вернуться на исходную ветку/коммит.

Дополнительно есть **`git bisect run <скрипт>`** — автоматический режим, в котором Git сам выполняет указанный скрипт (например, тест или проверку сборки) на каждом шаге и интерпретирует его код возврата: `0` означает "good", ненулевой (кроме 125) — "bad", а код `125` означает "пропустить этот коммит" (например, если он вообще не собирается и не может быть корректно протестирован). Это позволяет полностью автоматизировать поиск, если баг можно детектировать программно (unit-тест, e2e-тест, проверка exit code скрипта).

Также можно вручную пропускать коммиты командой `git bisect skip`, если конкретный коммит невозможно протестировать (например, он ломает сборку по несвязанной причине).

## Пример / когда применяется

Ручной режим:

```bash
git bisect start
git bisect bad HEAD
git bisect good v1.4.0

# Git переключает на коммит из середины диапазона
# запускаем приложение/тесты, проверяем баг вручную
git bisect good   # если баг отсутствует
# или
git bisect bad    # если баг воспроизводится

# после нахождения проблемного коммита
git bisect reset
```

Автоматический режим с тестовым скриптом:

```bash
git bisect start
git bisect bad HEAD
git bisect good v1.4.0
git bisect run npm test -- --testPathPattern=checkout.spec.js
```

Типичный сценарий для QA: автоматизатор обнаружил, что регрессионный e2e-тест начал падать, но неизвестно, в каком коммите за последние две недели появилась регрессия. Вместо того чтобы вручную откатываться по коммитам, запускается `git bisect run`, который прогоняет этот конкретный тест на каждом промежуточном коммите и находит точный коммит, сломавший функциональность — это существенно ускоряет root cause analysis.

## На что смотрит интервьюер

- Понимает ли кандидат принцип бинарного поиска (а не линейного перебора) и почему это эффективно на большой истории.
- Знает ли команды `bisect start / good / bad / reset` и не забывает ли про обязательный `git bisect reset` в конце.
- Знает ли про `git bisect run` и автоматизацию через exit code скрипта (0 / non-zero / 125 для skip) — это отличает опытного кандидата от того, кто знает только теорию.
- Может ли привести практический пример использования в контексте автоматизации тестирования (например, привязка к CI, автоматический тестовый скрипт).
- Красный флаг: если кандидат путает `git bisect` с `git blame` (blame ищет автора строки, bisect ищет коммит по воспроизводимому поведению/тесту) или не может объяснить, что нужны заведомо good и bad точки для старта.
