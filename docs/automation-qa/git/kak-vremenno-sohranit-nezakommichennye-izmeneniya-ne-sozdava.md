# Как временно сохранить незакоммиченные изменения, не создавая коммит? Какая команда для этого используется (`git stash`)?

**Коротко:** Команда `git stash` временно откладывает незакоммиченные изменения (staged и unstaged) в отдельный стек, возвращая рабочую директорию к чистому состоянию последнего коммита, а затем позволяет вернуть их обратно командой `git stash pop`/`apply`.

## Развёрнутый ответ

`git stash` полезен, когда нужно быстро переключиться на другую ветку (например, для срочного хотфикса) или обновить текущую (`git pull`), но текущие изменения ещё не готовы к коммиту и не хочется делать "мусорный" промежуточный коммит.

Основные команды:

- **`git stash`** (или `git stash push`) — сохраняет изменения из рабочей директории и staging area в новую запись стека (stash), возвращая файлы к состоянию последнего коммита. По умолчанию untracked-файлы не включаются.
- **`git stash -u`** (`--include-untracked`) — дополнительно сохраняет untracked-файлы.
- **`git stash list`** — показывает список сохранённых stash-записей (стек, каждая с индексом `stash@{0}`, `stash@{1}`...).
- **`git stash pop`** — применяет последнюю (или указанную) stash-запись к текущей рабочей директории и удаляет её из стека.
- **`git stash apply`** — применяет stash, но не удаляет его из стека (можно применить один и тот же stash несколько раз, например к разным веткам).
- **`git stash drop stash@{n}`** — удаляет конкретную запись из стека без применения.
- **`git stash show -p stash@{0}`** — показывает diff содержимого stash.
- **`git stash push -m "сообщение"`** — сохраняет stash с описательным комментарием, что удобно при нескольких активных stash-записях.

Технически stash реализован как специальные коммиты (не привязанные ни к одной ветке), хранящиеся в `refs/stash`, поэтому его содержимое надёжно, но не является частью обычной истории репозитория, пока не будет применено.

Важный нюанс: при `pop`/`apply` могут возникнуть конфликты, если файлы в рабочей директории с момента stash изменились несовместимым образом — тогда нужно разрешить конфликт вручную, как при merge.

## Пример / когда применяется

```bash
# Есть незакоммиченные изменения, но пришла срочная задача — нужно переключиться на другую ветку
git status
git stash push -m "WIP: login validation tests"

# Переключение и работа над срочной задачей
git checkout hotfix/critical-bug
# ... исправление и коммит ...

# Возврат к прерванной работе
git checkout feature/login-tests
git stash list
git stash pop
```

Практический сценарий для QA-автоматизатора: пишет автотест, но нужно срочно проверить баг на другой ветке. Вместо коммита незавершённого теста ("заглушки") используется `git stash`, что сохраняет чистоту истории коммитов.

## На что смотрит интервьюер

- Знание разницы между `pop` и `apply` (удаляет из стека или нет).
- Понимание, что stash — это стек (LIFO), и умение работать с несколькими записями по индексу.
- Знание, что untracked-файлы по умолчанию не сохраняются без флага `-u`.
- Follow-up: что делать при конфликте после `stash pop`; чем stash отличается от создания временной ветки/коммита.
- Red flag: путаница stash с обычным коммитом или незнание, что стек стирается после `pop`.
