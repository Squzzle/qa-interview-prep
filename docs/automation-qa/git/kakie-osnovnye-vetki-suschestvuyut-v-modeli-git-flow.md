# Какие основные ветки существуют в модели Git Flow? Объясните их назначение (main, develop, feature, release, hotfix).

**Коротко:** Git Flow использует две долгоживущие ветки — `main` (стабильные релизы) и `develop` (интеграция разработки) — и три типа временных веток: `feature` (новая функциональность), `release` (подготовка релиза) и `hotfix` (срочные исправления в продакшене).

## Развёрнутый ответ

Git Flow — модель ветвления, предложенная Винсентом Дриссеном, ориентированная на проекты с чёткими релизными циклами.

- **main (или master)** — содержит только production-ready код. Каждый коммит в `main` обычно помечается тегом версии (`v1.2.0`) и соответствует реальному релизу, развёрнутому у пользователей.
- **develop** — основная интеграционная ветка, куда сливаются все завершённые фичи. Отражает "следующий релиз" в разработке; это база для создания feature-веток.
- **feature/\*** — создаются от `develop` для разработки отдельной функциональности или, применительно к QA, для написания автотестов на конкретный функционал. По завершении сливаются обратно в `develop` через pull request. Никогда не взаимодействуют напрямую с `main`.
- **release/\*** — создаются от `develop`, когда накопленный функционал готов к выпуску. В этой ветке происходит финальное тестирование, стабилизация, исправление мелких багов, обновление номера версии и документации — но не добавление новых фич. После готовности `release` сливается и в `main` (с тегом), и обратно в `develop` (чтобы исправления попали в основную линию разработки).
- **hotfix/\*** — создаются напрямую от `main`, когда в продакшене обнаружен критический баг, который нужно исправить немедленно, не дожидаясь следующего релизного цикла. После исправления `hotfix` сливается и в `main` (с новым патч-тегом), и в `develop` (чтобы исправление не потерялось при следующем релизе).

Такая модель даёт чёткое разделение: `main` всегда стабилен, `develop` — всегда актуален с последними принятыми фичами, а временные ветки (`feature`, `release`, `hotfix`) изолируют работу и удаляются после слияния.

Для QA важно понимать, где и что тестировать: `feature`-ветки тестируются на уровне модульных/компонентных автотестов при каждом PR, `release`-ветки — на предмет регрессии перед выпуском, `hotfix` — точечно, чтобы быстро подтвердить исправление критичного бага.

## Пример / когда применяется

```bash
# Начало новой фичи от develop
git checkout develop
git checkout -b feature/payment-validation

# Завершение фичи — слияние обратно в develop
git checkout develop
git merge --no-ff feature/payment-validation
git branch -d feature/payment-validation

# Подготовка релиза
git checkout -b release/1.2.0 develop
# ... финальное тестирование, багфиксы ...
git checkout main
git merge --no-ff release/1.2.0
git tag -a v1.2.0 -m "release 1.2.0"
git checkout develop
git merge --no-ff release/1.2.0

# Срочный хотфикс в продакшене
git checkout -b hotfix/1.2.1 main
# ... исправление критического бага ...
git checkout main
git merge --no-ff hotfix/1.2.1
git tag v1.2.1
git checkout develop
git merge --no-ff hotfix/1.2.1
```

## На что смотрит интервьюер

- Понимание назначения именно каждой ветки, а не только общего перечисления названий.
- Знание, что `hotfix` ответвляется от `main`, а не от `develop`, и вливается в обе ветки.
- Знание альтернатив: GitHub Flow (проще, одна `main` + `feature`-ветки, непрерывный деплой) и Trunk-Based Development — умение сравнить, когда Git Flow оправдан (проекты с версионированными релизами), а когда избыточен (непрерывная поставка/CD).
- Red flag: путаница, откуда ответвляется `hotfix`, или незнание, зачем `release`/`hotfix` сливаются обратно в `develop`.
