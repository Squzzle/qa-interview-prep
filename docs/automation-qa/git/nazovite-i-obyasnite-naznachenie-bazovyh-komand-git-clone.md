# Назовите и объясните назначение базовых команд Git: `clone`, `add`, `commit`, `push`, `pull`, `status`, `log`.

**Коротко:** `clone` копирует удалённый репозиторий локально, `add` помещает изменения в staging area, `commit` фиксирует снимок в истории, `push`/`pull` синхронизируют локальный репозиторий с удалённым, а `status`/`log` показывают текущее состояние и историю изменений.

## Развёрнутый ответ

- **`git clone <url>`** — создаёт полную локальную копию удалённого репозитория, включая всю историю коммитов и ветки. После клонирования локальный репозиторий автоматически получает remote с именем `origin`, указывающий на источник.
- **`git add <файл|.>`** — добавляет изменения из рабочей директории в индекс (staging area). Это промежуточный шаг перед коммитом, позволяющий выбрать, какие именно изменения войдут в следующий коммит (частичный коммит).
- **`git commit -m "сообщение"`** — фиксирует всё, что находится в staging area, как новый неизменяемый снимок (снапшот) в истории репозитория, с автором, датой и сообщением. Без `add` изменения не попадут в коммит.
- **`git push [remote] [branch]`** — отправляет локальные коммиты текущей (или указанной) ветки на удалённый репозиторий, обновляя соответствующую удалённую ветку. Если удалённая ветка ушла вперёд (кто-то ещё запушил), push будет отклонён, пока не сделан `pull`/`fetch`+merge.
- **`git pull [remote] [branch]`** — забирает изменения из удалённого репозитория и сразу интегрирует их в текущую локальную ветку. По сути это `git fetch` + `git merge` (или `git rebase`, если настроено `pull.rebase`).
- **`git status`** — показывает текущее состояние рабочей директории и staging area: какие файлы изменены, добавлены в индекс, не отслеживаются (untracked), а также на какой ветке находится пользователь и есть ли расхождения с удалённой веткой.
- **`git log`** — выводит историю коммитов текущей ветки: хеш, автора, дату, сообщение. С флагами (`--oneline`, `--graph`, `--all`, `-p`) можно получить компактное или визуальное представление истории, включая ветвление и слияния.

## Пример / когда применяется

```bash
# Клонировать репозиторий с автотестами
git clone https://github.com/company/qa-autotests.git
cd qa-autotests

# Посмотреть текущее состояние
git status

# Внести изменения, добавить в staging и закоммитить
git add tests/login_test.py
git commit -m "add negative login test cases"

# Проверить историю перед пушем
git log --oneline -5

# Отправить изменения на сервер
git push origin feature/login-tests

# Получить свежие изменения от коллег
git pull origin develop
```

## На что смотрит интервьюер

- Понимание разницы между рабочей директорией, staging area и историей коммитов — то есть зачем нужен `add` перед `commit`.
- Знание, что `pull` = `fetch` + `merge`/`rebase`, а не отдельная самостоятельная операция.
- Умение объяснить, что покажет `status` и `log` в конкретной ситуации (например, staged/unstaged файлы, расхождение веток).
- Red flag: если кандидат не может объяснить назначение staging area или путает `push`/`pull` по направлению передачи данных.
