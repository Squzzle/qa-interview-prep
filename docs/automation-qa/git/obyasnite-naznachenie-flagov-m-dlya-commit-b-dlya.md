# Объясните назначение флагов: `-m` (для `commit`), `-b` (для `checkout`), `--amend` (для `commit`), `-d` и `-D` (для `branch`), `-f` (для `push`).

**Коротко:** это ключевые флаги для базовых операций Git — задать сообщение коммита, создать и переключиться на ветку, исправить последний коммит, удалить ветку (безопасно или принудительно) и принудительно перезаписать удалённую историю.

## Развёрнутый ответ

Рассмотрим каждый флаг подробно.

**`git commit -m "сообщение"`** — позволяет указать сообщение коммита прямо в командной строке, не открывая текстовый редактор (который по умолчанию настроен через `core.editor`). Без этого флага Git откроет редактор, чтобы ввести сообщение. Можно указать флаг несколько раз (`-m "заголовок" -m "тело"`), тогда каждое значение станет отдельным абзацем сообщения.

**`git checkout -b <имя-ветки>`** — комбинированная команда, которая создаёт новую ветку от текущего HEAD и сразу переключается на неё. Это эквивалент двух команд: `git branch <имя-ветки>` + `git checkout <имя-ветки>`. В современных версиях Git это же делает `git switch -c <имя-ветки>`.

**`git commit --amend`** — изменяет последний коммит вместо создания нового: можно поправить сообщение коммита и/или добавить в него новые изменения из staging area (`git add` + `git commit --amend`). Важный нюанс: `--amend` переписывает историю (меняется хэш коммита), поэтому если коммит уже был запушен и другие разработчики могли его получить, после `--amend` потребуется force push, что опасно для общих веток.

**`git branch -d <имя-ветки>`** — удаляет локальную ветку, но только если она уже полностью слита (merged) в текущую ветку или в upstream. Это защитная мера от потери коммитов.

**`git branch -D <имя-ветки>`** — принудительное удаление ветки, эквивалент `--delete --force`. Удаляет ветку независимо от того, слита она или нет. Используется, когда ветка точно больше не нужна (например, эксперимент, фича была отменена), но требует осторожности — несмерженные коммиты могут быть потеряны безвозвратно (хотя какое-то время их можно восстановить через `git reflog`).

**`git push -f`** (или `--force`) — принудительно перезаписывает удалённую ветку локальной историей, даже если это приведёт к потере коммитов на удалённом репозитории (например, после `rebase` или `amend`). Более безопасная альтернатива — `git push --force-with-lease`, которая откажется от push, если на сервере появились новые коммиты, о которых локальный репозиторий не знает (т.е. кто-то другой запушил изменения после последнего fetch).

## Пример / когда применяется

Типичный рабочий сценарий:

```bash
# создать и переключиться на новую ветку фичи
git checkout -b feature/login-form

# сделать коммит с сообщением
git commit -m "Добавлена форма логина"

# заметили опечатку в последнем коммите - исправляем без нового коммита
git add .
git commit --amend -m "Добавлена форма логина с валидацией"

# ветка была экспериментальной и не нужна
git branch -D feature/old-experiment

# после amend/rebase истории, запушенной ранее, нужно перезаписать удалённую ветку
git push --force-with-lease origin feature/login-form
```

## На что смотрит интервьюер

- Понимает ли кандидат разницу между `-d` и `-D` для `branch` (защищённое vs принудительное удаление) — частая ошибка: путают их местами.
- Знает ли, почему `--amend` и `push -f` опасны на общих ветках (main/develop) и как коллеги могут потерять свои изменения при force push.
- Упоминает ли `--force-with-lease` как более безопасную альтернативу `-f` — это признак опытного специалиста.
- Знает ли, что `-b` в `checkout` — это именно "создать и переключиться", а не просто переключиться (для этого нужен просто `git checkout <branch>` без `-b`).
- Красный флаг: если кандидат говорит, что `git branch -d` удаляет ветку "в любом случае" — не различает поведение с несмерженными коммитами.
