# Объясните разницу между `git fetch` и `git pull`.

**Коротко:** `git fetch` скачивает изменения из удалённого репозитория, обновляя локальные удалённые ссылки (`origin/main`), но не трогает текущую рабочую ветку; `git pull` делает то же самое и сразу автоматически сливает (или перебазирует) полученные изменения в текущую локальную ветку.

## Развёрнутый ответ

**`git fetch`** обращается к удалённому репозиторию и скачивает все новые коммиты, ветки и теги, которых ещё нет локально, обновляя ссылки вида `origin/main`, `origin/develop` и т.д. При этом текущая рабочая ветка и рабочая директория никак не изменяются — пользователь просто получает актуальную информацию о состоянии удалённого репозитория и может сам решить, что с ней делать: посмотреть diff (`git diff main origin/main`), просмотреть лог (`git log origin/main`) или слить вручную.

**`git pull`** — это фактически комбинация двух операций: `git fetch` + `git merge origin/<branch>` (или `git rebase`, если настроена опция `pull.rebase=true` или передан флаг `--rebase`). То есть `pull` сразу интегрирует полученные изменения в текущую ветку, автоматически создавая merge-коммит (или перебазируя, в зависимости от настройки) — без промежуточного шага проверки, что именно было получено.

Практическое следствие разницы:

- `fetch` безопаснее — не изменяет текущую работу, позволяет сначала посмотреть, что изменилось, прежде чем сливать.
- `pull` быстрее и удобнее для повседневного использования, но может неожиданно создать merge-коммит или конфликт прямо посреди работы, если не ожидалось расхождение веток.
- Рекомендуемая практика в командах с активной разработкой: использовать `fetch` + ручной `rebase`/`merge`, чтобы контролировать момент и способ интеграции чужих изменений, особенно перед пушем собственных коммитов.

## Пример / когда применяется

```bash
# Безопасный способ: сначала посмотреть, что изменилось
git fetch origin
git log HEAD..origin/main --oneline   # какие коммиты появились на удалённой main
git diff HEAD origin/main             # что именно изменилось

# Затем осознанно слить или перебазировать
git merge origin/main
# или
git rebase origin/main

# Быстрый способ (fetch + merge/rebase одной командой)
git pull origin main

# Настроить pull так, чтобы он всегда делал rebase, а не merge
git config --global pull.rebase true
```

Пример ситуации: перед тем как запушить свою feature-ветку и открыть pull request, разработчик делает `git fetch origin` и `git rebase origin/develop`, чтобы убедиться, что его изменения применяются поверх актуального `develop` без сюрпризов, вместо бездумного `git pull`, который мог бы создать лишний merge-коммит.

## На что смотрит интервьюер

- Точное понимание: `pull` = `fetch` + `merge`/`rebase`, а не отдельная независимая команда.
- Знание, что `fetch` не изменяет рабочую директорию/текущую ветку.
- Понимание рисков бездумного `pull` (неожиданные merge-коммиты, конфликты) и того, почему многие команды предпочитают `fetch` + осознанный `rebase`.
- Follow-up: как настроить `git pull --rebase` по умолчанию; что произойдёт при конфликте во время `pull`.
- Red flag: утверждение, что `fetch` и `pull` делают одно и то же, либо что `fetch` изменяет рабочие файлы.
