# Опишите основные концепции Git: репозиторий, коммит, ветка.

**Коротко:** Репозиторий — хранилище всей истории проекта; коммит — зафиксированный снимок изменений с уникальным хешем и метаданными; ветка — именованный указатель на цепочку коммитов, позволяющий вести параллельную разработку.

## Развёрнутый ответ

**Репозиторий (repository, repo)** — это хранилище проекта, содержащее все файлы, а также полную историю их изменений (объекты Git: коммиты, деревья, blob-объекты, теги). Репозиторий бывает локальным (папка `.git` на компьютере разработчика) и удалённым (remote), например на GitHub/GitLab. Локальный репозиторий включает три логические области: рабочую директорию (working directory) — файлы, с которыми работает пользователь; индекс/область подготовки (staging area / index) — промежуточное место, куда добавляются изменения командой `git add` перед коммитом; и саму историю коммитов в `.git`.

**Коммит (commit)** — это зафиксированный снимок (snapshot) состояния всех отслеживаемых файлов на определённый момент времени. Каждый коммит имеет уникальный SHA-1 (или SHA-256 в новых версиях) хеш-идентификатор, ссылку на родительский коммит (или несколько родителей для merge-коммита), автора, дату и сообщение коммита (commit message). Коммиты образуют направленный ациклический граф (DAG) — историю проекта. Коммит — неизменяемый объект: изменение уже сделанного коммита (`amend`, `rebase`) технически создаёт новый коммит с новым хешем.

**Ветка (branch)** — это просто именованный, изменяемый указатель на конкретный коммит (по сути файл с хешем внутри `.git/refs/heads/`). Когда делается новый коммит на ветке, указатель ветки автоматически сдвигается на этот новый коммит. Ветки позволяют вести параллельную разработку: например, `main` содержит стабильный код, а `feature/xyz` — изменения для новой функциональности, которые не влияют на `main` до момента слияния (`merge`). Специальный указатель `HEAD` показывает, на какой ветке (или коммите) сейчас находится пользователь.

Дополнительно стоит упомянуть тег (tag) — статичную метку на конкретном коммите (например, версия релиза `v1.0.0`), которая, в отличие от ветки, не двигается при новых коммитах.

## Пример / когда применяется

```bash
# Инициализация репозитория
git init

# Создание файла и коммита
echo "test" > test.txt
git add test.txt
git commit -m "add test file"

# Создание и переключение на новую ветку
git checkout -b feature/new-tests

# Просмотр графа коммитов и веток
git log --oneline --graph --all
```

Практическая ситуация: QA-автоматизатор создаёт ветку `feature/checkout-autotests` от `develop`, пишет автотесты отдельными коммитами, затем открывает pull request для слияния обратно в `develop` после ревью.

## На что смотрит интервьюер

- Чёткое разграничение понятий: ветка — это указатель, а не копия файлов.
- Понимание staging area и трёх состояний файла (untracked/modified, staged, committed).
- Знание, что коммит неизменяем, а любое "изменение истории" — это создание новых коммитов.
- Red flag: если кандидат путает ветку с отдельной физической копией репозитория или не может объяснить, что такое HEAD.
- Follow-up: что такое detached HEAD, чем тег отличается от ветки.
