# В чем отличие `git revert` от `git reset`? Какой из них безопаснее для использования в общей ветке?

**Коротко:** `git revert` создаёт новый коммит, отменяющий изменения указанного коммита, сохраняя всю историю нетронутой; `git reset` перемещает указатель текущей ветки на другой коммит, фактически удаляя (или откатывая) коммиты из истории ветки. Для общих (публичных) веток безопасен только `revert`.

## Развёрнутый ответ

**`git revert <commit>`** создаёт новый коммит, содержимое которого — это "антидиф" (обратные изменения) указанного коммита. Исходный коммит остаётся в истории нетронутым, просто добавляется новый коммит, отменяющий его эффект. Поскольку история не переписывается, а только дополняется, `revert` полностью безопасен для использования на общих/публичных ветках (`main`, `develop`), которыми пользуются другие участники команды — никто не потеряет данные, и никому не придётся синхронизировать локальную историю силой.

**`git reset <commit>`** перемещает указатель текущей ветки (и, в зависимости от режима, индекс и рабочую директорию) на указанный коммит, эффективно "забывая" все коммиты, сделанные после него на этой ветке. У `reset` есть три режима:

- **`--soft`** — перемещает только указатель ветки/HEAD; все изменения из "отменённых" коммитов остаются в staging area (готовы к повторному коммиту).
- **`--mixed`** (по умолчанию) — перемещает указатель ветки и очищает staging area, но изменения остаются в рабочей директории как unstaged.
- **`--hard`** — перемещает указатель ветки и полностью сбрасывает и staging area, и рабочую директорию до состояния целевого коммита; все незакоммиченные и "отменённые" изменения теряются безвозвратно (если не restore-ить через reflog).

Поскольку `reset` физически меняет, на какой коммит указывает ветка, использование его на уже опубликованной (запушенной, используемой другими) ветке требует последующего `git push --force`, что перезаписывает удалённую историю и может привести к потере коммитов у коллег, если они не синхронизируются правильно. Поэтому `reset` считается **опасным для общих веток** и подходит в первую очередь для локальной, ещё не опубликованной истории (например, отменить свой последний локальный коммит перед пушем).

Итог: если нужно отменить изменения в ветке, которой уже пользуются другие (production, `main`, `develop`), — используется `revert`. Если нужно исправить свою собственную ещё не запушенную историю (например, убрать лишний локальный коммит или изменить staged-файлы) — можно использовать `reset`.

## Пример / когда применяется

```bash
# Отменить проблемный коммит в общей ветке main безопасным способом
git checkout main
git revert a1b2c3d
# создаётся новый коммит "Revert 'add broken feature'"
git push origin main

# Локально отменить последний коммит, но сохранить изменения в staging area
git reset --soft HEAD~1

# Локально отменить последний коммит и изменения, но оставить их в рабочей директории
git reset --mixed HEAD~1   # (эквивалент git reset HEAD~1)

# Полностью откатить локальные изменения к состоянию 2 коммита назад (данные теряются!)
git reset --hard HEAD~2
```

Практический пример: в `main` попал коммит, который сломал автотесты в CI. Поскольку ветка уже подтянута другими членами команды, безопасный способ исправления — `git revert <hash>` с последующим пушем, а не `reset --hard` + `force push`, который может стереть чужие коммиты и сломать локальные копии коллег.

## На что смотрит интервьюер

- Чёткое понимание: `revert` добавляет новый коммит, `reset` перемещает указатель и может удалять коммиты из истории ветки.
- Знание трёх режимов `reset` (`--soft`, `--mixed`, `--hard`) и разницы между ними.
- Понимание, почему `reset` на опубликованной ветке требует `force push` и почему это опасно для команды.
- Follow-up: как восстановить "потерянный" после `reset --hard` коммит через `git reflog`; когда допустим `push --force-with-lease` вместо обычного `--force`.
- Red flag: рекомендация использовать `reset --hard` + `force push` для отмены изменений в общей/production ветке без понимания рисков.
