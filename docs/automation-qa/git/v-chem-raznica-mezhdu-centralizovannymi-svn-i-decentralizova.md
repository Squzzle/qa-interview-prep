# В чем разница между централизованными (SVN) и децентрализованными (Git) системами контроля версий?

**Коротко:** В централизованных VCS (SVN) существует один центральный сервер с полной историей, а клиенты хранят только рабочую копию; в распределённых (Git) каждый клиент имеет полную копию репозитория со всей историей, что даёт офлайн-работу и большую отказоустойчивость.

## Развёрнутый ответ

Централизованные системы контроля версий (Centralized VCS, CVCS), например SVN (Subversion) или CVS, устроены по модели клиент-сервер: единственный центральный репозиторий хранит всю историю проекта, а разработчики получают только текущую рабочую копию файлов. Чтобы посмотреть историю, сделать коммит или переключить ветку, клиенту нужно постоянное соединение с сервером. Если сервер недоступен или повреждён, работа команды останавливается, а восстановление истории зависит от бэкапов сервера.

Распределённые системы контроля версий (Distributed VCS, DVCS), в первую очередь Git (а также Mercurial), устроены иначе: при клонировании репозитория каждый разработчик получает не только текущее состояние файлов, но и полную историю всех коммитов, веток и тегов. Это даёт ряд практических преимуществ:

- Работа офлайн: можно коммитить, смотреть историю, создавать ветки без доступа к сети; синхронизация с удалённым репозиторием (push/pull) происходит отдельно.
- Отказоустойчивость: если центральный сервер (GitHub/GitLab) выйдет из строя, любая локальная копия репозитория может стать новым источником восстановления.
- Быстрые локальные операции: коммит, diff, log, branch выполняются мгновенно, так как не требуют сетевого запроса.
- Гибкое ветвление и слияние: создание и переключение веток в Git — дешёвая операция (указатель на коммит), тогда как в SVN ветки исторически были "дорогими" (копирование каталогов).
- Множественные рабочие процессы: можно синхронизироваться не только с одним центральным сервером, но и напрямую с репозиторием коллеги.

Минус децентрализованной модели — больший размер локального клона (вся история) и более сложная для новичка ментальная модель (staging area, локальные и удалённые ветки, разные состояния коммитов).

## Пример / когда применяется

Сравнение на практике:

| Критерий | SVN (централизованная) | Git (распределённая) |
|---|---|---|
| Хранение истории | Только на сервере | На сервере и у каждого клиента |
| Работа офлайн | Невозможна (кроме просмотра рабочей копии) | Полностью возможна |
| Создание ветки | Дорогая операция | Дешёвая, почти мгновенная |
| Точка отказа | Единая (сервер) | Множество копий-реплик |
| Типичный workflow | Trunk-based с блокировками файлов | Feature branches, pull request/merge request |

## На что смотрит интервьюер

- Понимание, что "распределённость" — это про хранение полной истории у каждого клиента, а не просто про наличие GitHub.
- Знание практических следствий: офлайн-работа, дешёвое ветвление, отказоустойчивость.
- Red flag: путаница между "Git" и "GitHub" (GitHub — это просто хостинг для Git-репозиториев с дополнительными функциями типа PR и issue-трекера).
- Follow-up: почему в Git ветки "дешевле", чем в SVN; что произойдёт, если пропадёт GitHub — можно ли восстановить проект.
