# В чем разница между командами `git merge` и `git rebase`?

**Коротко:** `git merge` объединяет две ветки, создавая новый merge-коммит с двумя родителями и сохраняя исходную историю как есть; `git rebase` переписывает историю, перенося коммиты одной ветки поверх другой, создавая линейную историю без merge-коммитов.

## Развёрнутый ответ

**`git merge`** объединяет изменения из одной ветки в другую. Если объединяемая ветка не была изменена относительно точки ветвления (fast-forward), Git просто передвигает указатель ветки вперёд без создания нового коммита. Если же обе ветки имели независимые коммиты, Git создаёт специальный merge-коммит с двумя (или более) родителями, объединяющий изменения. Преимущество: история сохраняется в точности так, как она реально происходила — видно, когда и что было слито, легко откатить весь merge одной командой. Недостаток: при активной параллельной разработке история может стать "запутанной", с множеством пересекающихся линий и merge-коммитов.

**`git rebase`** переносит серию коммитов текущей ветки так, будто они были сделаны поверх последнего коммита целевой ветки (обычно `main`/`develop`). Технически Git берёт каждый коммит вашей ветки, применяет его diff поверх нового основания и создаёт новый коммит с новым хешем (старые коммиты не удаляются сразу, а становятся недостижимыми и попадают в garbage collection). Результат — линейная, чистая история без merge-коммитов, как будто разработка велась последовательно. Недостаток: коммиты получают новые хеши, что делает rebase **опасным для уже опубликованных (запушенных) веток**, которыми пользуются другие люди — их локальная история разойдётся с переписанной. Золотое правило: не rebase-ить публичные/общие ветки, только свои локальные/личные feature-ветки перед пушем.

Итог сравнения:

| | merge | rebase |
|---|---|---|
| История | Сохраняется как есть, с merge-коммитами | Переписывается, становится линейной |
| Хеши коммитов | Не меняются | Меняются (новые коммиты) |
| Безопасность для общих веток | Безопасен | Опасен (переписывает опубликованную историю) |
| Читаемость истории | Может быть "ветвистой" | Линейная, легче читать `git log` |
| Откат | Один merge-коммит легко отменить | Сложнее откатить группу переписанных коммитов |

## Пример / когда применяется

```bash
# Merge: слить feature-ветку в develop, сохранив историю
git checkout develop
git merge feature/login-tests
# при конфликтах — разрешить и git merge --continue (или commit после add)

# Rebase: подтянуть свежий develop в свою локальную feature-ветку перед PR
git checkout feature/login-tests
git rebase develop
# при конфликтах — разрешить, git add ., git rebase --continue

# Интерактивный rebase для "причёсывания" своих коммитов перед пушем
git rebase -i HEAD~3
```

Типичный сценарий использования: разработчик держит feature-ветку неделю, за это время `develop` ушёл вперёд. Перед созданием pull request он делает `git rebase develop` в своей ветке (ещё не запушенной или запушенной только им одним), чтобы получить чистую линейную историю без лишнего merge-коммита. Финальное слияние в `develop`/`main` обычно всё равно делается через `merge` (часто squash-merge на уровне pull request).

## На что смотрит интервьюер

- Чёткое понимание, что rebase переписывает историю (новые хеши коммитов), а merge — нет.
- Знание правила "никогда не rebase-ить публичную/общую ветку".
- Умение объяснить, когда предпочтителен каждый подход (чистая линейная история vs сохранение реальной хронологии).
- Follow-up: что такое `git rebase -i`, как разрешать конфликты при rebase, что такое force push (`git push --force-with-lease`) и почему он нужен после rebase уже запушенной личной ветки.
- Red flag: утверждение, что rebase "просто быстрее" или "то же самое, что merge", без понимания разницы в истории.
