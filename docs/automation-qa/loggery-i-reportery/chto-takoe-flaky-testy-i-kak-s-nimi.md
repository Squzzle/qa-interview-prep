# Что такое flaky-тесты и как с ними бороться?

**Коротко:** Flaky-тест — это тест, который даёт непостоянный результат (то падает, то проходит) при неизменном коде приложения и тестов; бороться с ними нужно устранением первопричины нестабильности (гонки состояний, ожидания, изоляция данных), а не простым перезапуском.

## Развёрнутый ответ

Flaky-тест (нестабильный, "мерцающий" тест) — это автотест, результат которого не детерминирован: он может упасть в одном прогоне и пройти в другом без изменений в тестируемом коде, самом тесте или окружении. Flaky-тесты — одна из главных проблем автоматизации, потому что они подрывают доверие команды к тестам: если тест падает "просто так", разработчики начинают игнорировать реальные падения, считая их шумом.

**Основные причины flakiness:**

1. **Проблемы синхронизации/ожидания** — тест не дожидается загрузки элемента, ответа сервера или завершения асинхронной операции. Самая частая причина в UI- и API-автоматизации. Использование фиксированных `Thread.sleep()` вместо явных/динамических ожиданий (explicit wait, polling) усугубляет проблему.
2. **Гонки состояний (race conditions)** — параллельное выполнение тестов обращается к общим данным/ресурсам одновременно.
3. **Зависимость от порядка выполнения тестов** — тест полагается на состояние, оставленное предыдущим тестом, и падает, если порядок меняется или тесты запускаются параллельно/изолированно.
4. **Общие тестовые данные** — несколько тестов используют один и тот же аккаунт/запись в БД и конфликтуют друг с другом.
5. **Нестабильность окружения** — сетевые задержки, недоступность стороннего сервиса, нестабильная тестовая среда.
6. **Недетерминированные данные** — использование текущего времени, случайных чисел, ID без контроля, что иногда попадает в edge case.
7. **Проблемы с локаторами в UI-тестах** — анимации, динамически подгружаемый контент, изменение DOM после рендеринга.

**Способы борьбы:**

- **Устранение фиксированных sleep** — замена на explicit wait / polling с таймаутом (например, `WebDriverWait` с `ExpectedConditions` в Selenium, или retry-механизмы для API-ответов).
- **Изоляция тестовых данных** — каждый тест создаёт свои уникальные данные (через API/фикстуры) и не зависит от глобального состояния или других тестов.
- **Идемпотентность и независимость тестов** — тесты не должны зависеть от порядка выполнения; каждый тест должен уметь запускаться в изоляции.
- **Стабилизация окружения** — использование контейнеризации (Docker), моков внешних зависимостей (WireMock, Testcontainers) вместо реальных нестабильных сервисов.
- **Retry-механизм как временная мера** — авто-перезапуск упавшего теста (например, через `RetryAnalyzer` в TestNG) допустим как временное решение и способ отличить настоящий баг от нестабильности, но не должен маскировать первопричину навсегда.
- **Мониторинг и категоризация** — ведение реестра известных flaky-тестов (quarantine list), сбор статистики flakiness rate, регулярный анализ и приоритизация починки, а не просто игнорирование.
- **Анализ логов и скриншотов** падений именно нестабильных тестов, сравнение успешных и неуспешных прогонов для поиска паттерна.

Важно подчеркнуть: перезапуск (retry) — это способ сгладить симптом, а не решение проблемы. Если тест помечен как flaky и просто ретраится бесконечно, реальная причина нестабильности никогда не будет найдена, а доверие к автотестам продолжит падать.

## Пример / когда применяется

Плохой пример (причина flakiness):

```java
driver.findElement(By.id("submit")).click();
Thread.sleep(2000); // фиксированная задержка — источник нестабильности
assertThat(driver.findElement(By.id("result")).getText()).isEqualTo("Success");
```

Исправленный вариант с явным ожиданием:

```java
driver.findElement(By.id("submit")).click();
WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
WebElement result = wait.until(ExpectedConditions.visibilityOfElementLocated(By.id("result")));
assertThat(result.getText()).isEqualTo("Success");
```

Временный retry-механизм в TestNG (используется как страховка, а не как решение):

```java
public class RetryAnalyzer implements IRetryAnalyzer {
    private int count = 0;
    private static final int MAX_RETRY = 2;

    @Override
    public boolean retry(ITestResult result) {
        if (count < MAX_RETRY) {
            count++;
            return true;
        }
        return false;
    }
}
```

## На что смотрит интервьюер

- Может ли кандидат назвать конкретные причины flakiness, а не только общее определение.
- Понимает ли разницу между устранением причины (правильный подход) и retry (временная мера/симптоматическое лечение).
- Знает ли, как измерять flakiness (flakiness rate, quarantine list) и управлять этим на уровне процесса, а не только кода.
- Красный флаг — если кандидат говорит "мы просто ставим retry на все тесты и не разбираемся дальше" — это показывает непонимание проблемы и её последствий для качества автоматизации.
