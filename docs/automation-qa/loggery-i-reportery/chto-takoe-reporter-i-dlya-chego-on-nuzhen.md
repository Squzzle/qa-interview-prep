# Что такое репортер и для чего он нужен? Какие репортеры вы знаете (Allure, ExtentReports)?

**Коротко:** Репортер — это инструмент, который формирует наглядный, читаемый отчёт о результатах прогона автотестов (пройдено/упало/пропущено, шаги, вложения, тренды), понятный не только разработчикам, но и QA-лидам, менеджерам и заказчикам.

## Развёрнутый ответ

Репортер решает задачу представления результатов тестирования в удобном для восприятия виде. Сырой вывод тест-раннера (JUnit/TestNG/pytest) в консоли содержит минимум контекста: список тестов и статус pass/fail. Репортер добавляет:

- визуализацию структуры тестов (по фичам, эпикам, историям);
- шаги выполнения теста (step-by-step) с параметрами;
- вложения — скриншоты, логи, HTTP-запросы/ответы, видео;
- статистику: процент прохождения, длительность, распределение по severity;
- историю трендов между прогонами (flaky-тесты, стабильность во времени);
- группировку по окружениям, тегам, ответственным.

Репортер нужен, чтобы:

1. Быстро понять причину падения без чтения полного лога.
2. Показать результаты нетехническим стейкхолдерам (менеджерам, заказчику) в презентабельном виде.
3. Отслеживать динамику качества продукта — рост/падение количества failed-тестов, появление flaky-тестов.
4. Обеспечить трассируемость требований (Epic → Feature → Story → Test).

Наиболее распространённые репортеры:

- **Allure Report** — один из самых популярных инструментов, framework-agnostic (поддерживает JUnit, TestNG, pytest, Cucumber и др.). Генерирует HTML-отчёт с иерархией Epic/Feature/Story, шагами, вложениями, графиками, историей запусков. Работает через сбор JSON/XML-файлов результатов и последующую генерацию статического сайта командой `allure generate`.
- **ExtentReports** — Java-библиотека для генерации самодостаточных HTML-отчётов, часто используется в связке с Selenium/TestNG. Поддерживает логирование шагов через `ExtentTest.log()`, скриншоты, категории (author, device, category), параллельные отчёты через `ExtentReports` + `ExtentTest` ThreadLocal.
- **ReportPortal** — сервер-ориентированное решение с real-time агрегацией результатов, ML-анализом flaky-тестов и авто-категоризацией failure.
- **Cucumber HTML Reports / Serenity BDD** — отчёты, ориентированные на BDD-сценарии, показывают Gherkin-шаги.
- Встроенные отчёты CI (например, встроенный HTML-репорт pytest через `pytest-html`, JUnit XML для Jenkins/TeamCity).

## Пример / когда применяется

Подключение Allure к проекту на TestNG/Maven:

```xml
<dependency>
    <groupId>io.qameta.allure</groupId>
    <artifactId>allure-testng</artifactId>
    <version>2.24.0</version>
</dependency>
```

Генерация и открытие отчёта:

```bash
mvn clean test
allure serve target/allure-results
```

Пример использования ExtentReports:

```java
ExtentReports extent = new ExtentReports();
ExtentSparkReporter spark = new ExtentSparkReporter("report.html");
extent.attachReporter(spark);

ExtentTest test = extent.createTest("Login Test", "Проверка успешного логина");
test.log(Status.INFO, "Открытие страницы логина");
test.log(Status.PASS, "Логин выполнен успешно");

extent.flush();
```

Типичный сценарий применения: в CI-пайплайне после прогона тестов автоматически генерируется Allure-отчёт и публикуется как artifact/страница (например, через allure-jenkins-plugin или GitHub Pages), чтобы команда могла сразу видеть результаты без доступа к консоли CI.

## На что смотрит интервьюер

- Понимает ли кандидат разницу между тест-раннером (JUnit/TestNG/pytest) и репортером — это разные слои, репортер работает поверх результатов раннера.
- Знает ли, как Allure собирает данные (через аннотации и `allure-results` JSON-файлы) и что для генерации нужен отдельный шаг (`allure generate`/`allure serve`).
- Может ли объяснить практическую пользу репортера для нетехнической аудитории, а не только "красивые графики".
- Красный флаг — путаница репортера с логгером (это разные, хотя и дополняющие друг друга инструменты) или неспособность назвать ни одного реального инструмента, кроме общих слов.
- Follow-up вопросы: как настроить Allure history/trend, как прикрепить видео к отчёту, как интегрировать отчёт с CI (Jenkins/GitLab/GitHub Actions), как обрабатывать flaky-тесты в отчётности (retry + аннотация).
