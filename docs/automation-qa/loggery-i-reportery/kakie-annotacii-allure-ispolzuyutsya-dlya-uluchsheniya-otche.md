# Какие аннотации Allure используются для улучшения отчетов (`@Epic`, `@Feature`, `@Story`, `@Step`, `@Severity`, `@Attachment`)?

**Коротко:** Allure предоставляет набор аннотаций для структурирования отчёта по иерархии требований (`@Epic`/`@Feature`/`@Story`), детализации шагов теста (`@Step`), приоритизации дефектов (`@Severity`) и прикрепления артефактов (`@Attachment`).

## Развёрнутый ответ

Аннотации Allure делятся на несколько групп по назначению:

**Иерархия и группировка (BDD-структура):**

- `@Epic` — верхнеуровневая категория, обычно крупная бизнес-область или модуль продукта (например, "Управление заказами").
- `@Feature` — конкретная функциональность внутри эпика (например, "Оформление заказа").
- `@Story` — пользовательский сценарий или требование внутри фичи (например, "Оформление заказа с промокодом").

Эти три аннотации формируют дерево в разделе Behaviors отчёта Allure, что позволяет фильтровать и анализировать результаты по бизнес-логике, а не только по классам/пакетам тестов.

**Детализация выполнения:**

- `@Step` — помечает метод как шаг теста; текст шага может содержать плейсхолдеры `{параметр}`, которые подставляются значениями аргументов метода. Вложенные вызовы методов с `@Step` формируют иерархию шагов в отчёте.
- `@Attachment` — помечает метод, возвращаемое значение которого (обычно `byte[]` или `String`) прикрепляется к отчёту как вложение (скриншот, текст ответа, лог). Тип вложения указывается через `type` (например, `image/png`, `text/plain`).

**Приоритизация и метаданные:**

- `@Severity` — указывает критичность теста/дефекта: `BLOCKER`, `CRITICAL`, `NORMAL`, `MINOR`, `TRIVIAL`. Используется для приоритизации анализа падений — сначала смотрят BLOCKER/CRITICAL.
- `@Description` — текстовое описание теста, дополняющее его название.
- `@Link`, `@Issue`, `@TmsLink` — ссылки на внешние системы (Jira, TestRail) для трассируемости между тестом и требованием/дефектом.
- `@Owner` — указывает ответственного за тест.
- `@DisplayName`(или `title` в `@Test` для TestNG/JUnit5) — задаёт человекочитаемое имя теста в отчёте вместо имени метода.

Совместно эти аннотации превращают технический прогон тестов в структурированный, читаемый отчёт с бизнес-контекстом, что критично при работе с большим количеством тестов (сотни/тысячи), где без группировки по Epic/Feature/Story невозможно быстро ориентироваться.

## Пример / когда применяется

```java
@Epic("Управление заказами")
@Feature("Оформление заказа")
public class CheckoutTest {

    @Story("Оформление заказа с промокодом")
    @Severity(SeverityLevel.CRITICAL)
    @Test
    void shouldApplyPromoCodeSuccessfully() {
        openCheckoutPage();
        enterPromoCode("SALE10");
        submitOrder();
        assertDiscountApplied();
    }

    @Step("Открытие страницы оформления заказа")
    void openCheckoutPage() {
        driver.get("/checkout");
    }

    @Step("Ввод промокода {promoCode}")
    void enterPromoCode(String promoCode) {
        driver.findElement(By.id("promo")).sendKeys(promoCode);
    }

    @Attachment(value = "Скриншот страницы", type = "image/png")
    byte[] takeScreenshot() {
        return ((TakesScreenshot) driver).getScreenshotAs(OutputType.BYTES);
    }
}
```

В отчёте Allure это отобразится как дерево: Epic "Управление заказами" → Feature "Оформление заказа" → Story "Оформление заказа с промокодом" → шаги "Открытие страницы оформления заказа", "Ввод промокода SALE10" с прикреплённым скриншотом.

## На что смотрит интервьюер

- Знает ли кандидат разницу между `@Epic`/`@Feature`/`@Story` и может привести реальный пример иерархии, а не просто перечислить названия.
- Понимает ли механизм `@Attachment` — что прикрепляется именно возвращаемое значение метода, а не сам вызов.
- Знает ли значения `@Severity` и как они влияют на приоритизацию анализа результатов.
- Красный флаг — путаница `@Step` с обычным логированием (это разные вещи: `@Step` формирует визуальную структуру в отчёте, а не текстовый лог).
- Follow-up: как автоматически прикреплять скриншот при падении любого теста через `TestListener`/`ITestListener`, как настроить кастомные категории дефектов (`categories.json`) в Allure, как интегрировать `@TmsLink` с TestRail/Jira для трассируемости.
