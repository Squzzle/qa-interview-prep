# Сравните CSS-селекторы и XPath. Когда что лучше использовать?

**Коротко:** CSS-селекторы обычно быстрее, компактнее и лучше читаются, но не умеют искать по тексту и не могут перемещаться вверх по дереву к родителю. XPath медленнее и многословнее, но гибче: поддерживает поиск по тексту, оси в обе стороны и логические функции.

## Развёрнутый ответ

Сравнение по ключевым критериям:

| Критерий | CSS-селектор | XPath |
|---|---|---|
| Производительность | Обычно выше (нативный `querySelector`) | Обычно ниже, особенно в старых браузерах |
| Поиск по тексту элемента | Не поддерживается стандартом CSS3 | Поддерживается (`text()`, `contains()`) |
| Переход к родителю | Невозможен в чистом CSS | Возможен (`parent::`, `ancestor::`) |
| Переход к соседям | Только «вперёд» (`+`, `~`) | В обе стороны (`following-sibling`, `preceding-sibling`) |
| Читаемость | Обычно короче и понятнее | Более многословный синтаксис |
| Поддержка в кросс-браузерных движках | Везде одинаково (стандарт CSS) | В IE могут быть отличия от Chrome/Firefox по XPath-движку |
| Логические условия | Ограничены (`:not()`, атрибутные селекторы) | Богатый набор (`and`, `or`, `not()`, произвольные функции) |

**Когда использовать CSS-селектор:**
- Элемент можно однозначно идентифицировать по id, классу или атрибуту.
- Важна максимальная скорость выполнения теста (например, много проверок в цикле).
- Команда предпочитает единый стиль локаторов, максимально приближенный к тому, что используют фронтенд-разработчики в стилях.

**Когда использовать XPath:**
- Нужно найти элемент по его видимому тексту (кнопка, ссылка, ячейка таблицы) при отсутствии стабильных атрибутов.
- Нужно подняться от дочернего элемента к родителю или сложному предку.
- Нужна сложная логика фильтрации (комбинация условий, поиск n-го элемента с конца и т.д.).
- Работа с динамическими id/классами, где комбинация `starts-with`/`contains` даёт более читаемый результат, чем эквивалентный CSS.

На практике многие команды принимают правило: «где возможно — CSS, где нужен текст или движение вверх по дереву — XPath», без религиозного противопоставления одного другому.

## Пример / когда применяется

Одна и та же задача — найти кнопку "Купить" внутри карточки товара:

```css
/* CSS — если есть стабильный класс */
.product-card .buy-button
```

```text
// XPath — если нужно искать именно по видимому тексту
//div[@class='product-card']//button[text()='Купить']
```

Найти родительский блок ошибки по тексту сообщения (доступно только через XPath):

```text
//span[contains(text(), 'Неверный пароль')]/ancestor::div[@class='form-error']
```

## На что смотрит интервьюер

- Способность аргументированно, а не догматично, выбрать инструмент под конкретную задачу.
- Понимание конкретных ограничений CSS (нет текста, нет родителя) — это ключевой технический факт, который должен прозвучать.
- Красный флаг — категоричное «XPath всегда плохо/CSS всегда лучше» без объяснения контекста.
