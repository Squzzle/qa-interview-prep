# Дайте определения понятиям: идентификация, аутентификация, авторизация. Приведите пример последовательности этих процессов.

**Коротко:** Идентификация — предъявление системе своего идентификатора (кто ты); аутентификация — проверка подлинности этого идентификатора (доказательство, что ты действительно тот, за кого себя выдаёшь); авторизация — предоставление прав доступа к конкретным ресурсам после успешной аутентификации.

## Развёрнутый ответ

Идентификация (identification) — процесс, при котором пользователь или система заявляет о своей личности, обычно вводя уникальный идентификатор: логин, email, номер учётной записи, токен устройства. На этом этапе система ещё не проверяет подлинность заявленных данных, а лишь фиксирует, за кого себя выдаёт субъект.

Аутентификация (authentication) — процесс проверки подлинности предъявленного идентификатора. Система удостоверяется, что пользователь действительно тот, за кого себя выдаёт. Для этого используются факторы аутентификации:

- Знание (пароль, PIN-код, секретный вопрос).
- Владение (SMS-код, токен, приложение-аутентификатор, смарт-карта).
- Biometrия (отпечаток пальца, распознавание лица, голос).

Если используется больше одного фактора одновременно, это называется многофакторной аутентификацией (MFA/2FA), что повышает безопасность.

Авторизация (authorization) — процесс определения и предоставления прав доступа уже аутентифицированному пользователю к конкретным ресурсам, функциям или данным. Авторизация отвечает на вопрос "что тебе разрешено делать", в то время как аутентификация отвечает на вопрос "точно ли ты тот, кем представился". Авторизация обычно реализуется через роли (RBAC), права (permissions), политики доступа (ACL, ABAC).

Последовательность процессов на практике:

1. Идентификация — пользователь вводит логин (или email).
2. Аутентификация — пользователь вводит пароль (и, возможно, одноразовый код из SMS), система сверяет их с хранимыми данными.
3. Авторизация — после успешной проверки система определяет роль пользователя (например, "администратор" или "обычный пользователь") и открывает доступ только к тем разделам и действиям, которые разрешены этой роли.

## Пример / когда применяется

Пример из повседневной практики тестирования: при входе в систему пользователь вводит `login: ivanov@example.com` (идентификация), затем пароль `Passw0rd!` и код из SMS (аутентификация, в данном случае двухфакторная). После успешного входа система проверяет, что роль пользователя — "менеджер", и предоставляет доступ к разделу отчётов, но скрывает раздел администрирования пользователей (авторизация).

При тестировании это разделение важно проверять отдельно:
- негативный тест на аутентификацию — неверный пароль должен блокировать вход независимо от корректного логина;
- негативный тест на авторизацию — пользователь с ролью "гость" не должен получить доступ к API администратора, даже если он успешно аутентифицирован.

## На что смотрит интервьюер

Интервьюер проверяет, что кандидат чётко разделяет три термина и не путает аутентификацию с авторизацией (частая ошибка). Хороший ответ содержит бытовую аналогию (например, "предъявить паспорт — идентификация, сверить фото с лицом — аутентификация, пустить в VIP-зал по статусу билета — авторизация") и упоминает MFA. Красный флаг — использование терминов как синонимов. Follow-up вопросы: "Как бы вы тестировали разграничение прав доступа между ролями?", "В чём разница между RBAC и ABAC?", "Как проверить, что после логаута авторизация действительно аннулирована (сессия/токен инвалидирован)?".
