# Какие существуют типы тестов, подходящие для автоматизации (регрессионные, smoke, критический путь)?

**Коротко:** Для автоматизации лучше всего подходят регрессионные, smoke, критический путь (critical path), sanity-тесты, а также нагрузочные, API и модульные тесты — все они стабильны, повторяемы и имеют чёткий ожидаемый результат.

## Развёрнутый ответ

Наиболее подходящие для автоматизации типы тестов:

- Регрессионные тесты (regression testing) — проверяют, что новые изменения не сломали существующую функциональность. Выполняются многократно при каждом релизе/коммите, поэтому автоматизация даёт максимальную выгоду.
- Smoke-тесты (smoke testing) — быстрая проверка базовой работоспособности системы после сборки/деплоя ("не развалилось ли всё"). Обычно небольшой набор, запускается очень часто, идеально подходит для CI.
- Sanity-тесты — узконаправленная проверка конкретной новой функциональности или исправленного бага после сборки, тоже хорошо поддаётся автоматизации при повторных прогонках.
- Тесты критического пути (critical path testing) — проверяют ключевые бизнес-сценарии, от которых зависит основная ценность продукта (например, покупка товара, оформление кредита). Их поломка критична, поэтому важно проверять при каждом изменении.
- Модульные (unit) тесты — проверяют отдельные функции/классы в изоляции, пишутся разработчиками, по своей природе созданы для автоматического запуска.
- Интеграционные тесты и API-тесты — проверяют взаимодействие между компонентами/сервисами; стабильны, так как не зависят от UI, легко автоматизируются и быстро выполняются.
- Нагрузочные и производительные тесты (performance/load testing) — физически невозможно провести вручную с нужным количеством виртуальных пользователей, требуют инструментов автоматизации (JMeter, Gatling, k6).
- Тесты совместимости (cross-browser/cross-device) — большое количество комбинаций браузеров/устройств выгодно проверять автоматически (Selenium Grid, BrowserStack).
- Тесты с большим объёмом данных (data-driven testing) — например, проверка валидации формы с десятками наборов входных данных.

Менее подходящие или неподходящие для автоматизации:
- Исследовательское тестирование (exploratory testing).
- Usability и UX-тестирование.
- Тесты, требующие субъективной визуальной оценки (без специализированных инструментов визуального сравнения).
- Разовые или редко выполняемые проверки.

## Пример / когда применяется

Пример пирамиды тестирования в реальном проекте: большая база unit-тестов (быстрые, дешёвые, запускаются на каждый коммит), меньший слой API/интеграционных тестов (проверка бизнес-логики без UI), и небольшой набор E2E UI-тестов, покрывающий только критический путь (регистрация, оплата, оформление заказа) и smoke-проверки после деплоя в продакшн.

## На что смотрит интервьюер

Интервьюер ожидает, что кандидат перечислит конкретные типы тестов и объяснит, почему именно они подходят (стабильность, частота, детерминированность), а также упомянет пирамиду тестирования (много unit, меньше интеграционных, минимум UI E2E). Красный флаг — попытка автоматизировать всё через UI без учёта пирамиды ("ice cream cone" антипаттерн).
