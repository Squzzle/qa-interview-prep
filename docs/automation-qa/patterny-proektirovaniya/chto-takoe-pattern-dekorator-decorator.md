# Что такое паттерн Декоратор (Decorator)?

**Коротко:** Декоратор — структурный паттерн, который позволяет динамически добавлять объекту новое поведение, оборачивая его в один или несколько объектов-декораторов с тем же интерфейсом, не меняя код исходного класса.

## Развёрнутый ответ

Декоратор используется, когда нужно расширить функциональность объекта без наследования и без изменения его исходного класса — особенно если комбинаций дополнительных возможностей много и создавать под каждую комбинацию подкласс нерационально (взрыв числа подклассов). Идея: и базовый объект, и декораторы реализуют общий интерфейс `Component`. Декоратор хранит ссылку на «обёрнутый» объект `Component` и, реализуя тот же интерфейс, делегирует ему вызов, добавляя код до и/или после делегирования.

Структура паттерна:

- **Component** — общий интерфейс для оборачиваемых объектов и декораторов.
- **ConcreteComponent** — базовая реализация, которую можно декорировать.
- **Decorator** — абстрактный класс/интерфейс, хранящий ссылку на `Component` и реализующий тот же интерфейс (часто просто делегируя вызовы).
- **ConcreteDecorator** — добавляет конкретное поведение до/после вызова обёрнутого объекта.

Ключевые особенности:

- Декораторы можно комбинировать и накладывать друг на друга в любом порядке (обёртка над обёрткой), получая гибкие сочетания поведения во время выполнения программы, а не на этапе компиляции.
- В отличие от наследования, декоратор не меняет тип объекта на уровне класса — он оборачивает объект в объект того же интерфейса, поэтому поведение добавляется «снаружи», без модификации исходного кода (соблюдается принцип открытости/закрытости, OCP).
- Похож на Proxy по структуре (оба оборачивают объект тем же интерфейсом), но разное назначение: Proxy контролирует доступ к объекту (ленивая инициализация, кэширование, права доступа), Decorator — добавляет новую функциональность.
- Минус: много мелких классов-обёрток и порой сложно понять итоговое поведение, если декораторов много и порядок их применения важен.

## Пример / когда применяется

В автотестах Decorator часто применяют для добавления сквозной функциональности к шагам или к API-клиенту: логирование, ретраи, замер времени, — без изменения самой логики теста.

```ts
interface Action {
  run(): Promise<void>;
}

class ClickButtonAction implements Action {
  async run(): Promise<void> {
    // реальный клик по кнопке в UI
    console.log("click button");
  }
}

abstract class ActionDecorator implements Action {
  constructor(protected wrappee: Action) {}
  async run(): Promise<void> {
    await this.wrappee.run();
  }
}

class LoggingDecorator extends ActionDecorator {
  async run(): Promise<void> {
    console.log("Начало действия");
    await super.run();
    console.log("Действие завершено");
  }
}

class RetryDecorator extends ActionDecorator {
  constructor(wrappee: Action, private attempts: number = 3) {
    super(wrappee);
  }

  async run(): Promise<void> {
    let lastError: unknown;
    for (let i = 0; i < this.attempts; i++) {
      try {
        await super.run();
        return;
      } catch (err) {
        lastError = err;
      }
    }
    throw lastError;
  }
}

const action: Action = new LoggingDecorator(new RetryDecorator(new ClickButtonAction()));
await action.run();
```

## На что смотрит интервьюер

Важно чётко отделять Decorator от наследования и от Proxy. Хороший ответ показывает понимание, что декоратор реализует тот же интерфейс, что и оборачиваемый объект, и его можно применять цепочкой. Частые вопросы: «чем Decorator отличается от Proxy и от Adapter» (Adapter меняет интерфейс, Proxy контролирует доступ, Decorator расширяет поведение сохраняя интерфейс), «как соблюдается принцип единственной ответственности» (каждый декоратор отвечает за одну добавленную возможность), «какие проблемы возникают при большом количестве декораторов» (сложность отладки, важность порядка оборачивания). Красный флаг — если кандидат путает Decorator с обычным наследованием или не может объяснить, зачем нужен общий интерфейс.
