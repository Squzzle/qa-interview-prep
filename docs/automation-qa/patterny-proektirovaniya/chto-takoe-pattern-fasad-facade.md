# Что такое паттерн Фасад (Facade)?

**Коротко:** Фасад — структурный паттерн, который предоставляет простой унифицированный интерфейс к сложной подсистеме из множества классов, скрывая её внутреннюю сложность от клиента.

## Развёрнутый ответ

Фасад применяется, когда система состоит из множества взаимосвязанных классов со сложными зависимостями и последовательностью вызовов, а клиенту нужно решать конкретную задачу, не вникая в детали подсистемы. Класс-фасад инкапсулирует знание о том, какие объекты подсистемы создать, в каком порядке вызвать их методы и как обработать промежуточные результаты, предоставляя клиенту один или несколько простых методов верхнего уровня.

Важные характеристики паттерна:

- Фасад не запрещает прямой доступ к подсистеме — классы подсистемы остаются доступными напрямую для тех, кому нужна тонкая настройка; фасад лишь даёт удобную точку входа для типовых сценариев.
- Фасад снижает связанность (coupling) между клиентским кодом и подсистемой: клиент зависит от одного простого интерфейса, а не от десятков классов и их деталей.
- В отличие от Adapter, который приводит существующий интерфейс к ожидаемому клиентом виду (совместимость интерфейсов), Facade создаёт новый, более простой интерфейс поверх сложной подсистемы (упрощение использования), а не адаптацию.
- Может выступать в роли единственной точки входа в подсистему для конкретного слоя приложения, что упрощает последующий рефакторинг подсистемы — меняется реализация фасада, а не весь клиентский код.
- Иногда вводят несколько фасадов под разные сценарии использования одной и той же сложной подсистемы.

## Пример / когда применяется

В автотестах Facade — один из самых используемых паттернов: он скрывает низкоуровневые детали работы с браузером/API/базой данных за простым доменным методом, например в Page Object или в тестовом API-клиенте.

```ts
class HttpClient {
  async post(url: string, body: unknown): Promise<{ status: number; data: any }> {
    // низкоуровневый HTTP-запрос
    return { status: 200, data: { token: "abc123" } };
  }
}

class TokenStorage {
  save(token: string): void {
    // сохранение токена, например в localStorage через драйвер браузера
  }
}

class SessionValidator {
  validate(token: string): boolean {
    return token.length > 0;
  }
}

// Фасад скрывает три взаимодействующих класса за одним простым методом
class AuthFacade {
  private http = new HttpClient();
  private storage = new TokenStorage();
  private validator = new SessionValidator();

  async login(username: string, password: string): Promise<void> {
    const response = await this.http.post("/api/login", { username, password });
    if (!this.validator.validate(response.data.token)) {
      throw new Error("Некорректный токен авторизации");
    }
    this.storage.save(response.data.token);
  }
}

// В тесте:
const auth = new AuthFacade();
await auth.login("qa_user", "P@ssw0rd");
```

## На что смотрит интервьюер

Интервьюер ждёт чёткого разграничения Facade и Adapter (разные цели: упрощение против совместимости) и Facade и Mediator (Facade — однонаправленное упрощение доступа к подсистеме, Mediator — двунаправленная координация взаимодействий между компонентами, которые не должны знать друг о друге). Также могут спросить, как Facade помогает в тестовой автоматизации — обычно ожидают связку с Page Object Model, где страница выступает фасадом над низкоуровневыми локаторами и действиями драйвера. Красный флаг — если кандидат считает, что Facade полностью скрывает подсистему и делает прямой доступ к ней невозможным, либо путает Facade с абстрактным классом или интерфейсом подсистемы.
