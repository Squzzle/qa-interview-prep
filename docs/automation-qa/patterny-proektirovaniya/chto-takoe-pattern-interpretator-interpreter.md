# Что такое паттерн Интерпретатор (Interpreter)?

**Коротко:** Interpreter — поведенческий паттерн, который определяет грамматику простого языка и представляет её в виде иерархии классов, каждый узел которой умеет "интерпретировать" (вычислять) себя относительно некоторого контекста.

## Развёрнутый ответ

Паттерн применяется, когда нужно многократно обрабатывать выражения некоторого простого предметно-ориентированного языка (DSL) — например, математические выражения, условия фильтрации, правила маршрутизации. Вместо написания парсера "с нуля" вручную, каждое правило грамматики представляется отдельным классом, а разбор выражения превращается в построение дерева таких объектов (абстрактное синтаксическое дерево, AST). Вычисление выражения — это рекурсивный обход дерева, где каждый узел реализует метод `interpret(context)`.

Участники:

- **AbstractExpression** — общий интерфейс с методом `interpret(context)`.
- **TerminalExpression** — узел, соответствующий терминальному символу грамматики (например, число или переменная) — не содержит вложенных выражений.
- **NonterminalExpression** — узел, соответствующий правилу грамматики, комбинирующему другие выражения (например, сложение из двух под-выражений) — рекурсивно вызывает `interpret` у дочерних узлов.
- **Context** — содержит глобальную информацию, нужную для интерпретации (например, значения переменных).
- **Client** — строит дерево выражений (обычно с помощью отдельного парсера) и запускает интерпретацию.

Плюсы:
- Легко расширять грамматику новыми правилами — просто добавить новый класс выражения.
- Каждое правило грамматики изолировано в своём классе, что упрощает понимание отдельных частей языка.

Минусы:
- Для сложных грамматик количество классов растёт линейно с числом правил, а дерево становится громоздким и медленным при интерпретации (в отличие от компиляции).
- На практике для сложных языков используют генераторы парсеров (ANTLR, PEG.js) или готовые библиотеки выражений, а не ручную реализацию Interpreter — паттерн оправдан только для простых, стабильных и редко меняющихся грамматик.
- Считается одним из наименее используемых паттернов GoF в промышленной разработке именно из-за существования специализированных инструментов парсинга.

## Пример / когда применяется

В автоматизации тестирования Interpreter иногда применяют для реализации собственного мини-языка фильтрации тестовых данных или условий запуска тестов (например, DSL для выбора тестов по тегам: `"smoke AND NOT flaky"`).

```ts
interface Expression {
  interpret(tags: Set<string>): boolean;
}

class TagExpression implements Expression {
  constructor(private readonly tag: string) {}
  interpret(tags: Set<string>): boolean {
    return tags.has(this.tag);
  }
}

class AndExpression implements Expression {
  constructor(private left: Expression, private right: Expression) {}
  interpret(tags: Set<string>): boolean {
    return this.left.interpret(tags) && this.right.interpret(tags);
  }
}

class NotExpression implements Expression {
  constructor(private expr: Expression) {}
  interpret(tags: Set<string>): boolean {
    return !this.expr.interpret(tags);
  }
}

// Дерево для правила: "smoke AND NOT flaky"
const rule: Expression = new AndExpression(
  new TagExpression("smoke"),
  new NotExpression(new TagExpression("flaky"))
);

const testCaseTags = new Set(["smoke", "regression"]);
console.log(rule.interpret(testCaseTags)); // true — тест подходит под правило запуска
```

## На что смотрит интервьюер

- Понимание, что Interpreter — это про построение и обход AST для простого DSL, а не про парсинг вообще.
- Осознание границы применимости: паттерн подходит только для простых и стабильных грамматик; для сложных языков (парсинг JSON, XPath, полноценных языков программирования) на практике используют готовые парсеры или парсер-комбинаторы, а не ручную реализацию через Interpreter.
- Follow-up: "Почему Interpreter редко используется в реальных проектах?" — ожидается ответ про экспоненциальный рост классов и наличие специализированных инструментов (ANTLR, PEG.js, готовые движки выражений).
- Умение привести пример из автоматизации — DSL для тегов/фильтров тестов, кастомные условия для feature-флагов, простые правила валидации.
- Красный флаг: если кандидат путает Interpreter с Visitor — они действительно часто применяются вместе (Visitor для обхода AST с разными операциями над ним), но Interpreter — про сам процесс построения смысла из грамматики, а Visitor — про добавление операций к уже существующей структуре.
