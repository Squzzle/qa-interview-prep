# Что такое паттерн Итератор (Iterator)?

**Коротко:** Итератор — поведенческий паттерн, который предоставляет способ последовательного доступа к элементам коллекции, не раскрывая её внутреннее устройство (массив, связный список, дерево и т.д.).

## Развёрнутый ответ

Паттерн решает две связанные задачи: во-первых, дать единый способ обхода разных структур данных без знания их внутренней реализации; во-вторых, позволить иметь несколько независимых обходов одной и той же коллекции одновременно (каждый итератор хранит собственное состояние обхода — текущую позицию — отдельно от самой коллекции).

Основные участники:

- **Iterator** — интерфейс с методами обхода, обычно `hasNext()`/`next()` (или, в стриминговом/итеративном стиле, единая абстракция вроде протокола итераторов языка).
- **ConcreteIterator** — конкретная реализация обхода для конкретной структуры данных, хранит текущую позицию.
- **Aggregate (Iterable)** — интерфейс коллекции с методом создания итератора, например `createIterator()`.
- **ConcreteAggregate** — конкретная коллекция (массив, дерево, связный список), которая знает, как создать подходящий для себя итератор.

Важные нюансы:

- Итератор инкапсулирует алгоритм обхода — это позволяет менять способ обхода (например, в прямом или обратном порядке, в глубину или в ширину для дерева) без изменения кода самой коллекции и без изменения клиентского кода, работающего через общий интерфейс.
- Современные языки (TypeScript/JavaScript через `Symbol.iterator`, C# через `IEnumerable`, Java через `Iterable`/`Iterator`) имеют встроенную поддержку протокола итераторов на уровне языка — паттерн Iterator фактически стандартизирован в синтаксисе (`for...of`, foreach-циклы).
- Клиентский код, использующий итератор, не зависит от конкретного типа коллекции — это позволяет писать обобщённые алгоритмы, работающие с любой `Iterable`-коллекцией.
- Итератор можно комбинировать с Composite для обхода древовидных структур единым способом (см. паттерн Компоновщик).

## Пример / когда применяется

В автотестах Iterator полезен, когда нужно единообразно обходить разнородные наборы тестовых данных или шаги сценария, скрывая, откуда именно берутся данные (массив в памяти, постраничный ответ API, файл).

```ts
interface TestDataIterator<T> {
  hasNext(): boolean;
  next(): T;
}

class ArrayTestDataIterator<T> implements TestDataIterator<T> {
  private index = 0;

  constructor(private data: T[]) {}

  hasNext(): boolean {
    return this.index < this.data.length;
  }

  next(): T {
    return this.data[this.index++];
  }
}

interface TestDataSet<T> {
  createIterator(): TestDataIterator<T>;
}

interface UserFixture {
  login: string;
  role: "admin" | "user";
}

class UserFixtureCollection implements TestDataSet<UserFixture> {
  constructor(private users: UserFixture[]) {}

  createIterator(): TestDataIterator<UserFixture> {
    return new ArrayTestDataIterator(this.users);
  }
}

const collection = new UserFixtureCollection([
  { login: "admin1", role: "admin" },
  { login: "user1", role: "user" },
]);

const iterator = collection.createIterator();
while (iterator.hasNext()) {
  const user = iterator.next();
  console.log(`Запуск теста под пользователем: ${user.login} (${user.role})`);
}

// TypeScript также поддерживает встроенный протокол итерации через Symbol.iterator,
// что позволяет использовать for...of напрямую с классом коллекции.
```

## На что смотрит интервьюер

Интервьюер обычно проверяет понимание, что главная ценность Iterator — отделение алгоритма обхода от структуры данных и возможность иметь несколько параллельных независимых обходов одной коллекции. Часто спрашивают, как паттерн реализован «из коробки» в языке, который использует кандидат (протокол `Symbol.iterator`/`for...of` в TypeScript, `Iterable`/`Iterator` в Java, генераторы `yield`) — важно показать, что паттерн не абстрактная теория, а часть повседневной работы с языком. Могут спросить о разнице между внешним итератором (клиент сам вызывает `next()`, управляя ходом обхода) и внутренним (коллекция сама передаёт элементы в callback, как `forEach`) — и о плюсах/минусах каждого подхода. Красный флаг — если кандидат не может объяснить, зачем нужен отдельный объект-итератор, если можно просто написать цикл `for` по индексам коллекции напрямую (ответ: для инкапсуляции внутренней структуры и поддержки нескольких типов коллекций через единый интерфейс).
