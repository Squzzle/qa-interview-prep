# Что такое паттерн Компоновщик (Composite)?

**Коротко:** Компоновщик — структурный паттерн, который позволяет объединять объекты в древовидные структуры и работать с отдельным объектом и с группой объектов единообразно, через общий интерфейс.

## Развёрнутый ответ

Паттерн Компоновщик (Composite) решает задачу построения иерархии «часть-целое» так, чтобы клиентский код не различал, работает он с листовым элементом или с составным объектом (контейнером, который содержит другие элементы, в том числе другие контейнеры). Для этого вводится общий интерфейс `Component` с операциями, актуальными и для листьев, и для составных узлов. Листовые классы (`Leaf`) реализуют этот интерфейс напрямую, а составные классы (`Composite`) хранят коллекцию дочерних `Component` и делегируют им вызовы, агрегируя результат (например, суммируя, объединяя или просто вызывая операцию у каждого потомка).

Ключевые элементы паттерна:

- **Component** — общий интерфейс (или абстрактный класс) для листьев и контейнеров.
- **Leaf** — конечный элемент дерева, не имеющий потомков.
- **Composite** — узел, который хранит список дочерних `Component` и реализует операции через рекурсивный обход потомков.
- **Client** — работает только с интерфейсом `Component`, не зная, лист перед ним или составной объект.

Важные нюансы:

- Паттерн опирается на рекурсию: операция у `Composite` вызывает ту же операцию у всех детей, включая вложенные `Composite`.
- Есть два подхода: «прозрачный» (в интерфейсе `Component` объявлены и методы управления детьми — `add`/`remove`, что упрощает клиента, но листья вынуждены как-то реагировать на неприменимые к ним вызовы) и «безопасный» (методы управления детьми только в `Composite`, что типобезопаснее, но клиент должен различать типы через приведение).
- Часто применяется вместе с Итератором (для обхода дерева) и Посетителем (для добавления новых операций без изменения классов дерева).

## Пример / когда применяется

Классический пример — файловая система (файлы и папки) или дерево UI-компонентов. В контексте автотестов Composite удобен для построения иерархии проверок или составных степов сценария, где отдельный шаг и группа шагов выполняются одинаково через `execute()`.

```ts
interface TestComponent {
  execute(): Promise<void>;
  getName(): string;
}

class TestStep implements TestComponent {
  constructor(private name: string, private action: () => Promise<void>) {}

  async execute(): Promise<void> {
    await this.action();
  }

  getName(): string {
    return this.name;
  }
}

class TestSuite implements TestComponent {
  private children: TestComponent[] = [];

  constructor(private name: string) {}

  add(component: TestComponent): void {
    this.children.push(component);
  }

  async execute(): Promise<void> {
    for (const child of this.children) {
      await child.execute();
    }
  }

  getName(): string {
    return this.name;
  }
}

const login = new TestStep("Открыть страницу логина", async () => { /* ... */ });
const fillForm = new TestStep("Заполнить форму", async () => { /* ... */ });

const smokeSuite = new TestSuite("Smoke: авторизация");
smokeSuite.add(login);
smokeSuite.add(fillForm);

const regressionSuite = new TestSuite("Regression");
regressionSuite.add(smokeSuite); // вложенный composite

await regressionSuite.execute();
```

## На что смотрит интервьюер

Интервьюер проверяет, понимает ли кандидат разницу между Компоновщиком и обычной агрегацией объектов: ключевой признак паттерна — единый интерфейс для листа и контейнера, а не просто «список объектов». Частая ошибка — путать Composite с Decorator (в Decorator один компонент оборачивает ровно один другой компонент и добавляет поведение, а в Composite узел хранит множество детей и агрегирует их). Могут спросить про плюсы (упрощение клиентского кода, легко добавлять новые типы узлов) и минусы (сложно ограничить состав дерева типами — например, запретить определённому Leaf быть в определённом Composite), а также про поддержку рекурсивных операций и обхода через Iterator/Visitor.
