# Что такое паттерн Одиночка (Singleton)? Приведите пример и подводные камни.

**Коротко:** Singleton — порождающий паттерн, гарантирующий, что у класса есть только один экземпляр, и предоставляющий глобальную точку доступа к нему; в тестировании часто применяется для драйвера браузера или конфигурации, но злоупотребление им ведёт к скрытым зависимостям и проблемам с параллельным запуском тестов.

## Развёрнутый ответ

Singleton решает две задачи одновременно: контролирует, что класс создаёт ровно один экземпляр, и предоставляет глобальную точку доступа к этому экземпляру — обычно через статический метод `getInstance()`. Реализация типично включает приватный конструктор (чтобы нельзя было создать объект через `new` снаружи), статическое приватное поле для хранения единственного экземпляра и публичный статический метод, который либо создаёт экземпляр при первом обращении (ленивая инициализация), либо создаёт его сразу при загрузке класса (жадная инициализация).

Подводные камни:

- **Скрытые зависимости.** Классы, использующие Singleton, обращаются к нему напрямую через статический метод, а не через внедрение зависимости (Dependency Injection). Это делает код менее тестируемым — сложно подменить Singleton моком в юнит-тестах.
- **Проблемы в многопоточной/параллельной среде.** Наивная ленивая инициализация без синхронизации может привести к созданию двух экземпляров при параллельном доступе (race condition). В контексте автотестов это критично: если параллельно запускаются несколько тестов в разных потоках/воркерах, а Singleton хранит, например, WebDriver, тесты начнут "делить" один и тот же браузер и мешать друг другу.
- **Глобальное состояние.** Singleton — это, по сути, замаскированная глобальная переменная. Она усложняет отслеживание того, кто и когда изменил состояние, и делает тесты друг от друга зависимыми (нарушается изоляция тестов).
- **Сложности с уничтожением/сбросом состояния.** Между тестами нужно явно чистить состояние синглтона, иначе один тест "утекает" в другой.
- **Антипаттерн в SOLID-смысле.** Singleton нарушает Single Responsibility (класс отвечает и за бизнес-логику, и за управление своим жизненным циклом) и усложняет соблюдение Dependency Inversion.

Правильная практика в автоматизации — использовать Singleton точечно (например, для конфигурации, логгера) и предпочитать ThreadLocal-обёртки или DI-контейнеры для объектов вроде WebDriver, чтобы каждый поток/воркер имел свой изолированный экземпляр.

## Пример / когда применяется

```ts
class ConfigManager {
  private static instance: ConfigManager;
  private settings: Record<string, string> = {};

  private constructor() {
    this.settings = { baseUrl: 'https://example.com', timeout: '30000' };
  }

  static getInstance(): ConfigManager {
    if (!ConfigManager.instance) {
      ConfigManager.instance = new ConfigManager();
    }
    return ConfigManager.instance;
  }

  get(key: string): string {
    return this.settings[key];
  }
}

// Использование в тестовом фреймворке
const baseUrl = ConfigManager.getInstance().get('baseUrl');

// Подводный камень: WebDriver как Singleton в параллельном запуске
class DriverManager {
  private static driver: unknown;

  // При параллельном запуске (несколько workers) такой Singleton
  // приведёт к тому, что все потоки будут делить один драйвер.
  // Правильнее использовать ThreadLocal-подобный подход на воркер/поток.
  static getDriver(): unknown {
    if (!DriverManager.driver) {
      DriverManager.driver = {}; // здесь создаётся реальный WebDriver
    }
    return DriverManager.driver;
  }
}
```

## На что смотрит интервьюер

Интервьюер ожидает, что кандидат не просто опишет реализацию, а назовёт проблемы: тестируемость, потокобезопасность, глобальное состояние. Частый follow-up — "как сделать Singleton потокобезопасным?" (double-checked locking, статическая инициализация на этапе загрузки класса) и "как вы организуете драйвер в параллельных тестах, если не через голый Singleton?" (ответ: ThreadLocal / контекст на воркер). Красный флаг — кандидат считает Singleton универсально хорошим паттерном и не видит, что это форма глобального состояния.
