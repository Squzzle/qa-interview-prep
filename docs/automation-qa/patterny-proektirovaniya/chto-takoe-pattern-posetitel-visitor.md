# Что такое паттерн Посетитель (Visitor)?

**Коротко:** Visitor — поведенческий паттерн, позволяющий добавлять новые операции над объектами структуры классов, не изменяя сами эти классы, вынося логику операций во внешний объект-посетитель.

## Развёрнутый ответ

Проблема: есть иерархия классов (например, разные типы элементов документа или узлов дерева), и периодически нужно добавлять новые операции над всеми этими типами (сериализация, валидация, подсчёт статистики). Если добавлять метод для каждой новой операции прямо в каждый класс иерархии, это нарушает принцип открытости/закрытости и раздувает классы логикой, не относящейся к их основной ответственности.

Visitor решает это через приём, называемый "двойная диспетчеризация" (double dispatch): каждый элемент структуры реализует метод `accept(visitor)`, который вызывает соответствующий метод посетителя, передавая себя (`visitor.visitConcreteElementA(this)`). Так как `accept` вызывается полиморфно (первая диспетчеризация — выбор класса элемента), а внутри него посетитель тоже выбирает нужный метод по типу элемента (вторая диспетчеризация — через перегрузку или явную проверку типа), в результате выполняется правильная комбинация "операция x тип элемента" без единого `if/switch` по типам.

Участники:

- **Visitor** — интерфейс с методом `visit` для каждого конкретного типа элемента.
- **ConcreteVisitor** — реализует конкретную операцию для каждого типа.
- **Element** — интерфейс с методом `accept(visitor)`.
- **ConcreteElement** — конкретный класс структуры, реализующий `accept`.

Плюсы:
- Новую операцию можно добавить, просто создав новый класс-посетитель, не трогая существующие классы элементов (OCP).
- Логика операции, специфичная для типов, собрана в одном месте (в посетителе), а не разбросана по классам элементов.

Минусы:
- Добавление нового типа элемента требует изменения интерфейса Visitor и всех его реализаций — паттерн эффективен только тогда, когда иерархия классов элементов стабильна, а новые операции добавляются часто (обратная ситуация Strategy/Template Method).
- Нарушает инкапсуляцию: посетителю часто нужен доступ к внутреннему состоянию элемента, из-за чего приходится расширять его публичный интерфейс.
- Считается одним из сложнее воспринимаемых паттернов GoF из-за двойной диспетчеризации, особенно в языках без явной перегрузки методов (в TypeScript эмулируется через проверку типа или систему методов `visitX`).

## Пример / когда применяется

В автоматизации Visitor применяется при обходе сложных структур данных ответа API или дерева DOM-элементов, когда нужно выполнять разные операции (валидация схемы, логирование, подсчёт метрик) над разнородными узлами без встраивания этой логики в сами классы узлов.

```ts
interface NodeVisitor {
  visitTextField(node: TextFieldNode): void;
  visitCheckbox(node: CheckboxNode): void;
}

interface FormNode {
  accept(visitor: NodeVisitor): void;
}

class TextFieldNode implements FormNode {
  constructor(public readonly value: string) {}
  accept(visitor: NodeVisitor): void {
    visitor.visitTextField(this);
  }
}

class CheckboxNode implements FormNode {
  constructor(public readonly checked: boolean) {}
  accept(visitor: NodeVisitor): void {
    visitor.visitCheckbox(this);
  }
}

// Операция 1: валидация — добавляется без изменения классов узлов
class ValidationVisitor implements NodeVisitor {
  errors: string[] = [];
  visitTextField(node: TextFieldNode): void {
    if (node.value.trim() === "") this.errors.push("Пустое текстовое поле");
  }
  visitCheckbox(_node: CheckboxNode): void {
    // валидация чекбокса не требуется
  }
}

// Операция 2: сбор снапшота состояния формы для отчёта
class SnapshotVisitor implements NodeVisitor {
  snapshot: Record<string, unknown> = {};
  visitTextField(node: TextFieldNode): void {
    this.snapshot.text = node.value;
  }
  visitCheckbox(node: CheckboxNode): void {
    this.snapshot.checked = node.checked;
  }
}

const nodes: FormNode[] = [new TextFieldNode(""), new CheckboxNode(true)];
const validator = new ValidationVisitor();
nodes.forEach((n) => n.accept(validator));
console.log(validator.errors); // ["Пустое текстовое поле"]
```

## На что смотрит интервьюер

- Понимание двойной диспетчеризации — глубокий вопрос, отличающий уверенное знание от поверхностного.
- Осознание компромисса: Visitor удобен при стабильной иерархии типов и часто меняющихся операциях — это противоположность ситуации, где легче добавлять новые типы (там уместнее полиморфизм внутри самих классов).
- Follow-up: "Что произойдёт, если добавить новый тип элемента в иерархию?" — правильный ответ: придётся менять интерфейс Visitor и все его реализации, что является главным минусом паттерна.
- Практический пример обхода AST, DOM-дерева или сложной JSON-структуры ответа API — работодатели ценят, если кандидат может провести параллель с реальными задачами (например, обход схемы OpenAPI для генерации тестов).
- Красный флаг: если кандидат путает Visitor с простым перебором коллекции (`forEach` с колбэком) без понимания сути двойной диспетчеризации и разделения ответственности между структурой данных и операциями над ней.
