# Что такое паттерн Посредник (Mediator)?

**Коротко:** Mediator — поведенческий паттерн, который убирает прямые связи между объектами, заставляя их взаимодействовать только через объект-посредник, что снижает связанность (coupling) системы.

## Развёрнутый ответ

Без Mediator объекты общаются напрямую друг с другом — при большом количестве компонентов это порождает "паутину" связей "каждый с каждым", когда изменение одного класса тянет за собой изменения во всех, кто с ним взаимодействует. Mediator решает эту проблему: все компоненты знают только об объекте-посреднике и сообщают ему о своих событиях, а посредник сам решает, кого и как оповестить.

Ключевые участники:

- **Mediator** — интерфейс с методом уведомления (например, `notify(sender, event)`).
- **ConcreteMediator** — реализация, которая хранит ссылки на компоненты и содержит логику их координации.
- **Colleague (компонент)** — объект, который вместо прямого вызова других объектов вызывает метод посредника.

Плюсы:
- Уменьшает связанность между компонентами (принцип низкого зацепления).
- Логика взаимодействия сосредоточена в одном месте, а не размазана по классам — легче поддерживать и тестировать.
- Упрощает повторное использование отдельных компонентов, так как они не зависят друг от друга напрямую.

Минусы:
- Посредник может превратиться в "God Object" — перегруженный класс, который знает слишком много о системе и сложен в сопровождении.

Отличие от Facade: Facade даёт упрощённый интерфейс к подсистеме (односторонняя связь клиент → подсистема), а Mediator организует двустороннее взаимодействие между равноправными компонентами.

Отличие от Observer: Observer — это подписка на события конкретного субъекта (один ко многим), а Mediator координирует взаимодействие многих объектов между собой через центральный узел.

## Пример / когда применяется

В автотестах Mediator часто применяют в оркестрации шагов теста или в UI-моделях, где несколько виджетов страницы должны реагировать друг на друга (например, выбор в одном селекте должен обновить список в другом), не будучи напрямую связанными.

```ts
interface Mediator {
  notify(sender: Component, event: string): void;
}

abstract class Component {
  protected mediator?: Mediator;

  setMediator(mediator: Mediator): void {
    this.mediator = mediator;
  }
}

class LoginForm extends Component {
  submit(): void {
    this.mediator?.notify(this, "submit");
  }
}

class ResultsPanel extends Component {
  showSpinner(): void {
    console.log("Показан спиннер загрузки");
  }
}

class LoginPageMediator implements Mediator {
  constructor(
    private form: LoginForm,
    private results: ResultsPanel
  ) {
    form.setMediator(this);
    results.setMediator(this);
  }

  notify(sender: Component, event: string): void {
    if (sender === this.form && event === "submit") {
      this.results.showSpinner();
    }
  }
}

// Использование в тесте page-object'а
const form = new LoginForm();
const results = new ResultsPanel();
new LoginPageMediator(form, results);
form.submit(); // "Показан спиннер загрузки"
```

## На что смотрит интервьюер

- Понимание, что Mediator решает проблему связанности "многие-ко-многим", а не просто "обёртка над вызовами".
- Умение отличить Mediator от похожих паттернов: Facade, Observer, Chain of Responsibility.
- Осознание риска God Object — если кандидат не упоминает минусы, это красный флаг поверхностного знания.
- Частый follow-up: "Как Mediator помогает в тестировании UI с множеством взаимозависимых компонентов?" — ожидается пример из практики автоматизации, а не только теория.
- Ошибка новичков: путают Mediator с паттерном "Наблюдатель", считая их взаимозаменяемыми.
