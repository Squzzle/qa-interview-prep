# Что такое паттерн Приспособленец (Flyweight)?

**Коротко:** Flyweight — структурный паттерн для экономии памяти при работе с большим количеством похожих объектов за счёт разделения общего (внутреннего) состояния между объектами и хранения уникального (внешнего) состояния отдельно.

## Развёрнутый ответ

Паттерн решает проблему, когда приложению нужно создать огромное число мелких объектов, и накладные расходы на память становятся критичными. Идея — разделить состояние объекта на две части:

- **Внутреннее состояние (intrinsic state)** — неизменяемые данные, общие для многих объектов; хранятся внутри разделяемого объекта-flyweight и не зависят от контекста использования.
- **Внешнее состояние (extrinsic state)** — данные, специфичные для конкретного контекста использования; не хранятся во flyweight, а передаются ему извне (например, как аргументы метода) в момент выполнения операции.

Для управления разделяемыми объектами обычно вводится **FlyweightFactory** — фабрика, которая хранит пул уже созданных flyweight-объектов (обычно в `Map` по ключу внутреннего состояния) и возвращает существующий экземпляр вместо создания нового, если объект с таким внутренним состоянием уже существует.

Важные нюансы:

- Flyweight-объекты должны быть неизменяемыми (immutable) в части внутреннего состояния — иначе разделение между разными «пользователями» приведёт к взаимному искажению данных.
- Паттерн выгоден только при действительно большом количестве объектов и заметной доле общих данных; при малом числе объектов усложнение кода не окупается экономией памяти.
- Классический пример из GoF — символы текстового редактора (все буквы «А» с одним шрифтом и размером хранятся как один разделяемый объект, а позиция каждого символа на странице — внешнее состояние).
- Flyweight часто комбинируют с Factory Method/Abstract Factory (для создания и кэширования) и иногда с Singleton (сама фабрика может быть единственной).

## Пример / когда применяется

В автотестах Flyweight полезен, когда генерируется большой объём тестовых данных с повторяющимися общими параметрами — например, конфигурации браузера/окружения для параллельного запуска тысяч тестов, где сами конфигурации переиспользуются, а параметризуются только конкретные данные теста.

```ts
// Внутреннее состояние — разделяемое между многими тестами
class BrowserProfile {
  constructor(
    public readonly browserName: string,
    public readonly viewport: { width: number; height: number },
    public readonly userAgent: string
  ) {}
}

class BrowserProfileFactory {
  private static cache = new Map<string, BrowserProfile>();

  static getProfile(browserName: string, width: number, height: number): BrowserProfile {
    const key = `${browserName}_${width}x${height}`;
    if (!this.cache.has(key)) {
      this.cache.set(
        key,
        new BrowserProfile(browserName, { width, height }, `UA/${browserName}`)
      );
    }
    return this.cache.get(key)!;
  }
}

// Внешнее состояние — уникально для каждого тестового запуска
interface TestRun {
  profile: BrowserProfile;
  testId: string;
  userCredentials: { login: string; password: string };
}

function createTestRun(testId: string, login: string): TestRun {
  const profile = BrowserProfileFactory.getProfile("chromium", 1920, 1080);
  return { profile, testId, userCredentials: { login, password: "secret" } };
}

const run1 = createTestRun("TC-101", "user1");
const run2 = createTestRun("TC-102", "user2");
// run1.profile === run2.profile — один и тот же разделяемый объект
```

## На что смотрит интервьюер

Интервьюер обычно проверяет понимание разницы между внутренним и внешним состоянием — это ядро паттерна, и без этого разграничения ответ считается неполным. Могут спросить, чем Flyweight отличается от простого кэширования/пула объектов (Object Pool переиспользует изменяемые объекты по одному владельцу за раз для дорогих операций создания, а Flyweight разделяет один неизменяемый объект между множеством одновременных «владельцев» через внешний параметр). Красный флаг — если кандидат предлагает делать flyweight-объекты изменяемыми или не может привести реалистичный сценарий, где действительно есть проблема с памятью на большом количестве объектов.
