# Что такое паттерн Шаблонный метод (Template Method)?

**Коротко:** Template Method — поведенческий паттерн, который определяет скелет алгоритма в базовом классе, оставляя реализацию отдельных шагов подклассам, при этом сама последовательность шагов остаётся неизменной.

## Развёрнутый ответ

Паттерн применяется, когда есть несколько классов с похожим алгоритмом, отличающимся только в некоторых шагах. Базовый класс объявляет метод-шаблон (обычно `final`/непереопределяемый в языках, где это возможно), который вызывает набор шагов в фиксированном порядке. Часть шагов реализована в базовом классе (общая логика), часть объявлена как абстрактные методы, которые обязаны реализовать подклассы, а часть — необязательные "хуки" (hook methods) с реализацией по умолчанию, которые подклассы могут переопределить при необходимости.

Структура:

- **AbstractClass** — содержит шаблонный метод (`templateMethod()`), задающий последовательность вызовов; содержит как минимум один абстрактный шаг.
- **ConcreteClass** — реализует абстрактные шаги, определяя специфику алгоритма.

Ключевая идея — "принцип голливуда" (Hollywood Principle): "не звоните нам, мы сами позвоним вам" — базовый класс управляет потоком выполнения и сам вызывает методы подклассов в нужный момент, а не наоборот.

Плюсы:
- Устраняет дублирование общего кода алгоритма между похожими классами.
- Гарантирует одинаковую последовательность шагов во всех вариантах.
- Позволяет расширять алгоритм новыми шагами через переопределение хуков без изменения самого шаблона.

Минусы:
- Нарушает принцип подстановки Барбары Лисков в узком смысле, если подклассы должны точно следовать контракту родителя, но реализуют шаги по-разному — в целом при аккуратном проектировании это не проблема.
- Ограничивает гибкость: изменить порядок шагов можно только через изменение самого шаблонного метода, что требует правки базового класса.
- В языках без наследования интерфейсов или с приоритетом композиции над наследованием (что часто предпочтительнее) вместо Template Method иногда лучше использовать Strategy с внедрением зависимостей.

Отличие от Strategy: Template Method использует наследование (шаги переопределяются в подклассах), Strategy использует композицию (алгоритм целиком передаётся объектом-стратегией). Template Method фиксирует общий скелет и варьирует отдельные шаги, Strategy варьирует алгоритм целиком.

## Пример / когда применяется

Template Method — фундаментальный паттерн для фреймворков автотестов: базовый класс теста задаёт общий жизненный цикл (`setUp → execute → tearDown`), а конкретные тесты переопределяют только шаг выполнения.

```ts
abstract class ApiTestCase {
  // Шаблонный метод — фиксирует последовательность шагов
  async run(): Promise<void> {
    await this.setUp();
    try {
      await this.execute();
    } finally {
      await this.tearDown();
    }
  }

  protected async setUp(): Promise<void> {
    console.log("Общая инициализация: авторизация тестового пользователя");
  }

  protected abstract execute(): Promise<void>;

  protected async tearDown(): Promise<void> {
    console.log("Общая очистка: удаление тестовых данных");
  }
}

class GetUserProfileTest extends ApiTestCase {
  protected async execute(): Promise<void> {
    console.log("GET /users/me — проверяем код 200 и структуру ответа");
  }
}

class DeleteUserTest extends ApiTestCase {
  // Переопределяем хук setUp для специфичной подготовки
  protected async setUp(): Promise<void> {
    await super.setUp();
    console.log("Дополнительно: создаём временного пользователя для удаления");
  }

  protected async execute(): Promise<void> {
    console.log("DELETE /users/:id — проверяем код 204");
  }
}

const tests = [new GetUserProfileTest(), new DeleteUserTest()];
for (const test of tests) {
  await test.run();
}
```

## На что смотрит интервьюер

- Знание "принципа голливуда" и умение объяснить, чем он отличается от обычного вызова метода клиентом.
- Чёткое разграничение Template Method (наследование) и Strategy (композиция) — классический follow-up вопрос.
- Понимание разницы между обязательными абстрактными шагами и необязательными хуками.
- Практический пример из тест-фреймворков — большинство test runner'ов (JUnit, xUnit-подобные, Cypress hooks, Playwright fixtures) реализуют этот паттерн в своём жизненном цикле `beforeEach/test/afterEach`.
- Красный флаг: если кандидат не видит связи между Template Method и `setUp/tearDown` в тест-фреймворках — это говорит о слабом понимании архитектуры инструментов, которыми он пользуется каждый день.
