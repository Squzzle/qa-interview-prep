# Что такое паттерн Состояние (State)?

**Коротко:** State — поведенческий паттерн, при котором объект меняет своё поведение в зависимости от внутреннего состояния, делегируя логику отдельным классам-состояниям вместо громоздких условных конструкций.

## Развёрнутый ответ

Классическая проблема без State: объект (например, заказ, соединение, документ) имеет несколько состояний, и в каждом методе есть большой `switch`/`if-else` по текущему состоянию, определяющий поведение. С ростом числа состояний и методов такой код превращается в нечитаемую и хрупкую конструкцию, нарушающую принцип открытости/закрытости (добавление нового состояния требует правки всех методов-свитчей).

State решает это, выделяя каждое состояние в отдельный класс, реализующий общий интерфейс `State`. Объект-контекст (Context) хранит ссылку на текущий объект-состояние и делегирует ему вызовы. Переход между состояниями реализуется либо самим состоянием (оно само переключает контекст на следующее состояние), либо контекстом.

Участники:

- **Context** — объект с изменяемым поведением; хранит ссылку на текущее состояние и делегирует ему вызовы.
- **State** — интерфейс с методами, специфичными для поведения (например, `handle()`).
- **ConcreteState** — конкретная реализация поведения для одного состояния, а также логика перехода к следующему состоянию.

Сравнение с паттерном Strategy: структурно State и Strategy почти идентичны (оба используют делегирование через интерфейс). Различие — в намерении: Strategy — это взаимозаменяемые алгоритмы, выбираемые клиентом извне и обычно не переключающиеся сами по себе; State — это конечный автомат, где переходы между состояниями — часть логики паттерна, и состояния часто сами инициируют переход на следующее.

Плюсы:
- Устраняет громоздкие условные операторы.
- Каждое состояние — отдельный класс, легко тестируется изолированно.
- Упрощает добавление новых состояний (OCP).

Минусы:
- Увеличивает число классов.
- Излишен для объектов с 2-3 простыми состояниями — там `if/else` может быть проще и нагляднее.

## Пример / когда применяется

В автоматизации тестирования State хорошо подходит для моделирования жизненного цикла тестового объекта (заказ, сессия, задача), поведение которого меняется по этапам, — например, при тестировании workflow-системы через API, где нужно проверять допустимые/недопустимые переходы состояний.

```ts
interface OrderState {
  pay(context: OrderContext): void;
  ship(context: OrderContext): void;
  cancel(context: OrderContext): void;
}

class NewOrderState implements OrderState {
  pay(context: OrderContext): void {
    console.log("Оплата принята");
    context.setState(new PaidOrderState());
  }
  ship(): void {
    throw new Error("Нельзя отправить неоплаченный заказ");
  }
  cancel(context: OrderContext): void {
    console.log("Заказ отменён");
    context.setState(new CancelledOrderState());
  }
}

class PaidOrderState implements OrderState {
  pay(): void {
    throw new Error("Заказ уже оплачен");
  }
  ship(context: OrderContext): void {
    console.log("Заказ отправлен");
    context.setState(new ShippedOrderState());
  }
  cancel(context: OrderContext): void {
    console.log("Оплаченный заказ отменён, инициирован возврат");
    context.setState(new CancelledOrderState());
  }
}

class ShippedOrderState implements OrderState {
  pay(): void { throw new Error("Заказ уже отправлен"); }
  ship(): void { throw new Error("Заказ уже отправлен"); }
  cancel(): void { throw new Error("Отправленный заказ отменить нельзя"); }
}

class CancelledOrderState implements OrderState {
  pay(): void { throw new Error("Заказ отменён"); }
  ship(): void { throw new Error("Заказ отменён"); }
  cancel(): void { throw new Error("Заказ уже отменён"); }
}

class OrderContext {
  private state: OrderState = new NewOrderState();

  setState(state: OrderState): void {
    this.state = state;
  }

  pay(): void { this.state.pay(this); }
  ship(): void { this.state.ship(this); }
  cancel(): void { this.state.cancel(this); }
}

// В тесте: проверяем допустимые переходы конечного автомата
const order = new OrderContext();
order.pay();   // "Оплата принята"
order.ship();  // "Заказ отправлен"
```

## На что смотрит интервьюер

- Умение чётко провести границу между State и Strategy — самый частый follow-up вопрос на эту тему.
- Понимание, кто отвечает за переход между состояниями (само состояние или контекст) — оба варианта валидны, важно объяснить компромисс.
- Пример конечного автомата (state machine) из практики — тестирование workflow, статусов заявки, состояний соединения.
- Красный флаг: если кандидат описывает State просто как "замену switch на классы" без упоминания переходов между состояниями — это неполный ответ.
- Возможный follow-up: "Когда State избыточен?" — ожидается ответ про малое число состояний без сложной логики переходов.
