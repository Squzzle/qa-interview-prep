# Что такое паттерн Стратегия (Strategy)?

**Коротко:** Strategy — поведенческий паттерн, который выносит семейство взаимозаменяемых алгоритмов в отдельные классы с общим интерфейсом, позволяя выбирать и менять алгоритм в рантайме независимо от клиента, который им пользуется.

## Развёрнутый ответ

Проблема без Strategy: класс реализует несколько вариантов одного и того же алгоритма (например, разные способы валидации, сортировки, расчёта цены) внутри себя через условные операторы. Это нарушает принцип единственной ответственности (SRP) и принцип открытости/закрытости (OCP) — добавление нового алгоритма требует изменения существующего класса.

Strategy решает это разделением: каждый алгоритм выносится в отдельный класс, реализующий общий интерфейс `Strategy` с методом (например, `execute()`). Класс-контекст хранит ссылку на текущую стратегию и делегирует ей выполнение, не зная деталей реализации. Стратегию можно передать через конструктор, сеттер или параметр метода, и менять её в рантайме.

Участники:

- **Strategy** — общий интерфейс алгоритма.
- **ConcreteStrategy** — конкретная реализация алгоритма.
- **Context** — использует объект Strategy через интерфейс, не завися от конкретной реализации.

Плюсы:
- Устраняет разветвлённые условные конструкции.
- Алгоритмы легко тестировать изолированно и добавлять новые без изменения существующего кода (OCP).
- Инверсия зависимостей: контекст зависит от абстракции, а не от конкретных алгоритмов (DIP).

Минусы:
- Клиент должен знать о различиях между стратегиями, чтобы выбрать нужную.
- Увеличивает число классов при большом количестве вариантов алгоритма.

Отличие от State: см. вопрос про State — структурно паттерны похожи, но Strategy не подразумевает самостоятельных переходов между вариантами (выбор делает клиент/контекст извне), тогда как State моделирует конечный автомат с переходами.

## Пример / когда применяется

Strategy — один из самых востребованных паттернов в автоматизации тестирования: выбор стратегии ожидания элемента, стратегии авторизации (UI-логин vs API-логин), стратегии генерации тестовых данных или стратегии сравнения ответов API.

```ts
interface WaitStrategy {
  waitFor(selector: string): Promise<void>;
}

class PollingWaitStrategy implements WaitStrategy {
  async waitFor(selector: string): Promise<void> {
    console.log(`Поллинг DOM в поиске ${selector}`);
  }
}

class MutationObserverWaitStrategy implements WaitStrategy {
  async waitFor(selector: string): Promise<void> {
    console.log(`Подписка на MutationObserver для ${selector}`);
  }
}

class ElementLocator {
  constructor(private strategy: WaitStrategy) {}

  setStrategy(strategy: WaitStrategy): void {
    this.strategy = strategy;
  }

  async find(selector: string): Promise<void> {
    await this.strategy.waitFor(selector);
  }
}

// В тесте выбираем стратегию в зависимости от окружения
const locator = new ElementLocator(new PollingWaitStrategy());
await locator.find("#submit-button");

locator.setStrategy(new MutationObserverWaitStrategy());
await locator.find("#dynamic-widget");
```

## На что смотрит интервьюер

- Чёткое разграничение Strategy и State (частый вопрос "в чём разница").
- Понимание, что Strategy — реализация принципа инверсии зависимостей (DIP) и открытости/закрытости (OCP).
- Практический пример из автоматизации — умение показать, где в реальном фреймворке используется Strategy (стратегии ожидания, авторизации, репортинга, генерации данных).
- Follow-up: "Как передать стратегию — через конструктор или через параметр метода?" — ожидается обсуждение компромиссов (постоянная стратегия для объекта vs разовый выбор на вызов).
- Красный флаг: путаница между Strategy и простым использованием функций высшего порядка/колбэков — важно понимать, что в TS/JS Strategy часто реализуется именно через функции, а не только через классы, и уметь это озвучить.
