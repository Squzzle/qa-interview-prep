# Что такое паттерн Заместитель (Proxy)?

**Коротко:** Proxy — структурный паттерн, при котором объект-заместитель реализует тот же интерфейс, что и реальный объект, и контролирует доступ к нему (ленивая инициализация, кэширование, проверка прав, логирование), делегируя вызовы реальному объекту.

## Развёрнутый ответ

Proxy подставляется вместо реального объекта (`RealSubject`) и реализует тот же интерфейс (`Subject`), поэтому клиент работает с прокси так же, как работал бы с настоящим объектом, не подозревая о подмене. Прокси хранит ссылку на реальный объект (создавая его сам, лениво, либо получая извне) и добавляет собственную логику до и/или после делегирования вызова.

Выделяют несколько распространённых видов Proxy:

- **Виртуальный прокси (Virtual Proxy)** — откладывает создание «тяжёлого» объекта до первого реального обращения (lazy initialization).
- **Защищающий прокси (Protection Proxy)** — проверяет права доступа перед тем, как разрешить вызов реального объекта.
- **Удалённый прокси (Remote Proxy)** — представляет объект, находящийся в другом адресном пространстве/на другом сервере, скрывая сетевое взаимодействие.
- **Кэширующий прокси (Caching Proxy)** — сохраняет результаты дорогих операций и возвращает их повторно без обращения к реальному объекту.
- **Логирующий/умный прокси (Smart Reference)** — добавляет учёт обращений, логирование, подсчёт ссылок и т.п.

Ключевое отличие от похожих паттернов:

- От **Decorator** Proxy отличается назначением: Decorator добавляет новое поведение объекту, а Proxy контролирует доступ к объекту, зачастую вообще не добавляя новой функциональности «по сути задачи», а лишь управляя тем, когда и как вызывается реальный объект. Структурно оба паттерна похожи (обёртка с тем же интерфейсом), различие — в намерении.
- От **Facade** Proxy отличается тем, что работает с одним объектом через тот же интерфейс, тогда как Facade упрощает доступ к нескольким объектам подсистемы через новый, более простой интерфейс.
- От **Adapter** Proxy отличается тем, что не меняет интерфейс — интерфейс прокси идентичен интерфейсу реального объекта.

## Пример / когда применяется

В автотестах Proxy часто используют для оборачивания клиентов внешних сервисов или драйверов — например, добавить прозрачное кэширование ответов тестового окружения или проверку прав перед выполнением действия.

```ts
interface UserService {
  getUser(id: string): Promise<{ id: string; name: string }>;
}

class RealUserService implements UserService {
  async getUser(id: string): Promise<{ id: string; name: string }> {
    console.log(`Запрос к реальному API за пользователем ${id}`);
    // реальный сетевой вызов
    return { id, name: "Ivan" };
  }
}

// Кэширующий proxy — реализует тот же интерфейс UserService
class CachingUserServiceProxy implements UserService {
  private cache = new Map<string, { id: string; name: string }>();
  private realService = new RealUserService();

  async getUser(id: string): Promise<{ id: string; name: string }> {
    if (this.cache.has(id)) {
      console.log(`Пользователь ${id} взят из кэша`);
      return this.cache.get(id)!;
    }
    const user = await this.realService.getUser(id);
    this.cache.set(id, user);
    return user;
  }
}

const userService: UserService = new CachingUserServiceProxy();
await userService.getUser("42"); // реальный запрос
await userService.getUser("42"); // из кэша, без повторного запроса
```

## На что смотрит интервьюер

Интервьюер обычно проверяет умение отличить Proxy от Decorator по намерению, а не по структуре кода (структура почти идентична — обёртка с тем же интерфейсом). Хороший ответ приводит хотя бы один вид прокси (виртуальный, защищающий, кэширующий, удалённый) с конкретным примером. Возможные follow-up вопросы: «где Proxy встречается в стандартных библиотеках или фреймворках» (например, ORM с ленивой загрузкой связанных сущностей, `Proxy` объект в JavaScript), «как Proxy помогает в тестировании» (можно подставить прокси вместо реального сетевого сервиса для мокирования при сохранении того же интерфейса). Красный флаг — путаница Proxy с Mock/Stub из тестирования: концептуально они близки (оба подменяют реальный объект), но Proxy — паттерн проектирования продакшен-кода с определённым назначением (контроль доступа), а не техника тестового дублирования.
