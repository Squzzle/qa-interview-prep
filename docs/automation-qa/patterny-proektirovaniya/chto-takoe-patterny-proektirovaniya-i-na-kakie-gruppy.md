# Что такое паттерны проектирования и на какие группы они делятся?

**Коротко:** Паттерн проектирования — это типовое, проверенное временем решение часто встречающейся задачи проектирования кода; классические паттерны GoF (Gang of Four) делятся на три группы: порождающие, структурные и поведенческие.

## Развёрнутый ответ

Паттерн проектирования (design pattern) — это не готовый кусок кода, а описание общей схемы решения проблемы, которая многократно возникает в разработке ПО в определённом контексте. Паттерны фиксируют опыт множества разработчиков, дают единый словарь для обсуждения архитектуры («давай сделаем через Strategy») и позволяют не изобретать велосипед. Важно понимать: паттерн — это шаблон мышления, а не библиотека, которую можно подключить; его нужно адаптировать под конкретный язык и задачу.

Классическая книга "Design Patterns: Elements of Reusable Object-Oriented Software" (авторы Gamma, Helm, Johnson, Vlissides — отсюда название GoF, "Gang of Four") описывает 23 паттерна, разделённых на три категории:

1. **Порождающие (Creational)** — отвечают за процесс создания объектов, скрывая логику инстанцирования и делая систему независимой от того, как объекты создаются, компонуются и представляются. Примеры: Singleton, Factory Method, Abstract Factory, Builder, Prototype.
2. **Структурные (Structural)** — описывают, как классы и объекты компонуются в более крупные структуры, сохраняя гибкость и эффективность. Примеры: Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy.
3. **Поведенческие (Behavioral)** — касаются алгоритмов и распределения ответственности между объектами, того, как объекты взаимодействуют и обмениваются сообщениями. Примеры: Strategy, Observer, Command, Iterator, State, Template Method, Chain of Responsibility, Mediator, Memento, Visitor, Interpreter.

Также паттерны классифицируют по области действия — классовые (отношения фиксируются на этапе компиляции через наследование) и объектные (отношения устанавливаются во время выполнения через композицию).

Для QA-инженера-автоматизатора знание паттернов важно не только для собеседования: паттерны Page Object, Factory, Builder, Singleton, Strategy, Facade постоянно применяются при построении фреймворков автоматизации — они делают тесты поддерживаемыми, читаемыми и переиспользуемыми.

## Пример / когда применяется

В автоматизации тестирования группы паттернов проявляются так:

```ts
// Порождающий: Factory Method — создание нужного драйвера/страницы
interface Page {
  open(): void;
}

class LoginPage implements Page {
  open(): void {
    console.log('Открыта страница логина');
  }
}

class PageFactory {
  static create(pageName: string): Page {
    switch (pageName) {
      case 'login':
        return new LoginPage();
      default:
        throw new Error(`Unknown page: ${pageName}`);
    }
  }
}

// Структурный: Facade — упрощённый доступ к сложному API тестового фреймворка
class TestFacade {
  private page = PageFactory.create('login');

  loginAsUser(): void {
    this.page.open();
    // ...дальнейшие шаги скрыты за простым методом
  }
}

// Поведенческий: Strategy — разные стратегии ожидания элементов
interface WaitStrategy {
  wait(): Promise<void>;
}

class ExplicitWait implements WaitStrategy {
  async wait(): Promise<void> {
    /* ждать конкретное условие */
  }
}
```

## На что смотрит интервьюер

Частая ошибка — путать паттерн с готовой библиотекой или конкретной реализацией, а не с абстрактной схемой. Также кандидаты часто не могут назвать группы и путают, к какой группе относится тот или иной паттерн (например, путают Adapter со Strategy). Интервьюер может попросить привести пример паттерна из каждой группы применительно к фреймворку автотестов — важно уметь связать теорию с практикой (Page Object как вариация Facade/Proxy, Builder для тестовых данных, Singleton для драйвера). Красный флаг — попытка вспомнить все 23 паттерна наизусть без понимания принципов SOLID, на которых они базируются.
