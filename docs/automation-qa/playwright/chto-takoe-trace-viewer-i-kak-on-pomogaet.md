# Что такое Trace Viewer и как он помогает в анализе причин падения тестов?

**Коротко:** Trace Viewer — графический инструмент Playwright для просмотра записанной трассировки теста: он показывает пошаговый таймлайн действий, снимки DOM до/после каждого шага, сетевые запросы, консоль и исходный код, что позволяет точно локализовать причину падения без повторного запуска теста.

## Развёрнутый ответ

Trace Viewer открывает файл трассировки (`trace.zip`), сформированный модулем `context.tracing` или автоматически настройкой `trace` в конфигурации Playwright Test. Внутри интерфейса представлены несколько синхронизированных панелей:

- **Таймлайн действий** — список всех вызовов Playwright API (`click`, `fill`, `goto` и т. д.) с указанием времени выполнения каждого; можно кликнуть на любое действие и увидеть состояние страницы именно в этот момент.
- **Снимки DOM (snapshots)** — интерактивные копии страницы до и после действия, по которым можно "походить" мышью, как по живой странице, включая hover-состояния.
- **Сетевые запросы (Network)** — список всех запросов с телами, заголовками, статусами ответов, что позволяет понять, не был ли причиной падения ответ сервера.
- **Консоль (Console)** — логи и ошибки JavaScript, возникшие на странице.
- **Исходный код теста (Source)** — строка кода теста, соответствующая выбранному действию, с подсветкой.
- **Call log** — детали конкретного вызова API: параметры, время ожидания, причина таймаута (например, "элемент не видим" или "элемент задизейблен").

Главное преимущество Trace Viewer — возможность полностью восстановить контекст падения постфактум, без необходимости воспроизводить баг локально или добавлять `page.pause()`/логирование и перезапускать тест. Это особенно ценно для нестабильных (flaky) тестов, которые падают только в CI и не воспроизводятся на локальной машине.

Trace Viewer можно открыть локально командой `npx playwright show-trace trace.zip`, а также через веб-версию `trace.playwright.dev`, куда можно перетащить файл трассировки без установки инструментов.

## Пример / когда применяется

```bash
# Открыть локально сохранённый файл трассировки
npx playwright show-trace test-results/trace.zip

# Открыть трассировку прямо из CI-артефакта по URL
npx playwright show-trace https://example.com/artifacts/trace.zip
```

Типичный сценарий: тест упал в CI с ошибкой таймаута ожидания элемента. Вместо того чтобы гадать, что произошло, инженер скачивает `trace.zip` из артефактов CI, открывает его в Trace Viewer и видит, что на шаге клика элемент был перекрыт всплывающим баннером cookie-согласия — причина падения найдена без повторного запуска теста.

## На что смотрит интервьюер

Интервьюер оценивает, понимает ли кандидат, что Trace Viewer — это не просто "видео теста", а интерактивный инструмент с доступом к DOM, сети и консоли на каждом шаге. Частый вопрос — как включить трассировку только для упавших тестов (`trace: 'retain-on-failure'`) и как её открыть без установки Playwright локально (веб-версия trace.playwright.dev). Ошибкой считается путаница между Trace Viewer и обычным видео/скриншотами, а также незнание, что трассировку можно и нужно использовать в первую очередь для диагностики flaky-тестов в CI.
