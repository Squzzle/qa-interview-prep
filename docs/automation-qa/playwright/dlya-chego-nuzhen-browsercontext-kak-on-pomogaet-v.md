# Для чего нужен `BrowserContext`? Как он помогает в изоляции тестов?

**Коротко:** `BrowserContext` — это изолированная сессия внутри одного и того же процесса браузера (аналог отдельного incognito-профиля), которая даёт каждому тесту собственные куки, storage и настройки, не требуя перезапуска браузера целиком, что обеспечивает быструю и надёжную изоляцию тестов друг от друга.

## Развёрнутый ответ

Проблема, которую решает `BrowserContext`: если бы все тесты работали в рамках одного и того же "окна" браузера с общими куки и localStorage, состояние одного теста (например, авторизация, данные в localStorage, установленные cookies) могло бы влиять на результаты следующего теста, приводя к нестабильным (flaky) и трудно воспроизводимым падениям.

Полное решение — перезапускать браузер перед каждым тестом — работает, но дорого по времени (запуск процесса браузера занимает существенно больше времени, чем создание нового контекста) и по ресурсам (память, CPU), особенно при параллельном запуске большого количества тестов.

`BrowserContext` даёт компромисс: он создаётся практически мгновенно (`browser.newContext()`) внутри уже работающего процесса `Browser`, но при этом полностью изолирует:

- **Cookies** — свои для каждого контекста;
- **localStorage / sessionStorage** — не пересекаются между контекстами;
- **Кэш** — независимый кэш ресурсов;
- **Permissions** (геолокация, уведомления, камера/микрофон) — настраиваются индивидуально;
- **Viewport, locale, timezone, User-Agent** — можно задать отдельно для каждого контекста;
- **Аутентификационное состояние** — можно сохранить сессию через `storageState` и переиспользовать в других контекстах, минуя повторный логин.

Благодаря этому типичный паттерн в Playwright Test — создавать новый `BrowserContext` (и, соответственно, новую `Page`) перед каждым тестом (это делает встроенная фикстура `page` автоматически), а `Browser` запускать один раз на весь прогон тестов или на воркер. Это даёт полную изоляцию состояния между тестами при минимальных накладных расходах.

Дополнительно `BrowserContext` полезен для параллельного тестирования разных ролей пользователей в рамках одного теста — например, тестирование чата между двумя пользователями, где каждому нужен свой независимый "браузер" с собственной сессией, но без запуска двух реальных браузерных процессов.

## Пример / когда применяется

Переиспользование аутентификации между тестами через `storageState`, что ускоряет прогон, не жертвуя изоляцией:

```javascript
// Сохраняем состояние один раз после логина
const context = await browser.newContext();
const page = await context.newPage();
await page.goto('https://example.com/login');
await page.fill('#username', 'user');
await page.fill('#password', 'pass');
await page.click('#submit');
await context.storageState({ path: 'auth.json' });
await context.close();

// В других тестах используем сохранённую сессию без повторного логина
const authContext = await browser.newContext({ storageState: 'auth.json' });
```

Пример теста с двумя независимыми пользователями в одном браузере (чат):

```javascript
const browser = await chromium.launch();
const userA = await (await browser.newContext()).newPage();
const userB = await (await browser.newContext()).newPage();
// userA и userB полностью изолированы друг от друга, хотя работают в одном Browser
```

## На что смотрит интервьюер

- Понимание, что `BrowserContext` — это компромисс между полной изоляцией (новый браузер) и скоростью (общий процесс браузера).
- Знание конкретных вещей, которые изолируются на уровне контекста: cookies, storage, permissions, viewport/locale.
- Знание механизма `storageState` для переиспользования аутентификации между тестами/контекстами.
- Follow-up: "Как ускорить тесты, требующие логина, не теряя изоляцию?" (ответ через `storageState`).
- Красный флаг — если кандидат не может объяснить, почему `BrowserContext` быстрее, чем перезапуск браузера.
