# Как работает auto-waiting в Playwright? Для каких действий оно применяется?

**Коротко:** Перед выполнением действия над элементом (клик, ввод текста и т.д.) Playwright автоматически повторяет серию проверок готовности элемента ("actionability checks") с поллингом до истечения таймаута и только после успешного прохождения всех проверок выполняет само действие — это избавляет от необходимости писать явные ожидания вручную.

## Развёрнутый ответ

Auto-waiting — это встроенный механизм, при котором каждое действие Playwright, взаимодействующее с элементом, перед выполнением само дожидается, что элемент находится в состоянии, пригодном для этого действия. Вместо немедленной попытки выполнить действие и падения с ошибкой, Playwright запускает цикл проверок с коротким интервалом опроса (обычно раз в несколько десятков миллисекунд) вплоть до истечения таймаута действия (по умолчанию 30 секунд, настраивается через `timeout` или глобально в конфиге).

Набор проверок готовности (actionability checks) зависит от типа действия, но обычно включает:

- **Attached** — элемент присоединён к DOM (существует в дереве документа);
- **Visible** — элемент видим: имеет ненулевой размер (bounding box) и не имеет `visibility: hidden`/`display: none`;
- **Stable** — элемент не находится в процессе анимации/перемещения (координаты не меняются между двумя последовательными кадрами);
- **Receives Events** — элемент не перекрыт другим элементом в точке клика (например, модальным окном или всплывающей подсказкой поверх него);
- **Enabled** — элемент не имеет атрибута `disabled`;
- **Editable** (для полей ввода) — элемент не `readonly` и доступен для редактирования.

Действия, к которым применяется полный набор этих проверок:

- `click()` / `dblclick()` — проверяются Attached, Visible, Stable, Receives Events, Enabled;
- `fill()` — Attached, Visible, Enabled, Editable;
- `check()` / `uncheck()` — аналогично `click`, плюс проверка текущего состояния чекбокса;
- `hover()` — Attached, Visible, Stable, Receives Events;
- `selectOption()` — Attached, Visible, Enabled.

Для действий, не требующих взаимодействия с видимым состоянием элемента "как пользователь" (например, `textContent()`, `getAttribute()`), Playwright обычно ожидает только присоединения элемента к DOM (Attached), не проверяя видимость.

Если по истечении таймаута хотя бы одна проверка не проходит, Playwright выбрасывает исключение (`TimeoutError`) с подробным сообщением, указывающим, какая именно проверка не была пройдена (например, "element is not visible" или "element is outside of the viewport") — это заметно упрощает отладку по сравнению с общими ошибками таймаута в Selenium.

Важно отличать auto-waiting от `waitForSelector`/`waitForURL` и подобных явных методов ожидания: auto-waiting встроен неявно в каждое действие и не требует отдельного вызова, тогда как явные `waitForX` методы используются, когда нужно дождаться состояния, не привязанного напрямую к конкретному действию (например, дождаться навигации, ответа сети или произвольного JS-условия).

## Пример / когда применяется

```javascript
// Playwright сам дождётся, что кнопка видима, не перекрыта и включена
await page.getByRole('button', { name: 'Купить' }).click();

// Явное указание таймаута для конкретного действия
await page.getByText('Загрузка завершена').click({ timeout: 60000 });
```

Пример ситуации, где auto-waiting спасает от гонки: кнопка "Отправить" временно задизейблена, пока не заполнены обязательные поля формы (состояние `disabled` снимается через JS после валидации). Явный клик по кнопке в Playwright автоматически подождёт, пока `disabled` не исчезнет, вместо немедленного падения теста.

## На что смотрит интервьюер

- Способность перечислить конкретные actionability checks (Attached, Visible, Stable, Receives Events, Enabled, Editable), а не общую фразу "он сам ждёт".
- Понимание, что набор проверок зависит от конкретного действия.
- Знание таймаута по умолчанию (30 секунд) и возможности его переопределить.
- Понимание разницы между неявным auto-waiting (встроен в действия) и явными методами ожидания (`waitForSelector` и т.п.).
- Красный флаг — путаница auto-waiting с `implicitlyWait` из Selenium (в Selenium это глобальный таймер поиска элемента без проверки конкретных условий готовности, тогда как в Playwright проверяется целый набор конкретных условий).
