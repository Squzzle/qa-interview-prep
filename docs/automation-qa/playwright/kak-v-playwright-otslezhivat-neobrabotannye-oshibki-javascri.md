# Как в Playwright отслеживать необработанные ошибки JavaScript на странице (`WebError`)?

**Коротко:** Необработанные ошибки JavaScript на странице отслеживаются через подписку на событие `page.on('pageerror', handler)`, которое срабатывает при любом непойманном исключении в контексте браузера; для консольных сообщений (включая `console.error`) используется отдельное событие `page.on('console', ...)`.

## Развёрнутый ответ

По умолчанию Playwright не считает тест упавшим только из-за того, что на странице возникла JS-ошибка — она никак не блокирует выполнение теста автоматически. Чтобы отследить такие ошибки, нужно явно подписаться на события страницы:

- **`page.on('pageerror', error => {...})`** — срабатывает при необработанном исключении в JavaScript-коде страницы (например, `TypeError: cannot read property of undefined`, ошибка в обработчике события, необработанный reject промиса). Обработчик получает объект ошибки с сообщением и стеком.
- **`page.on('console', msg => {...})`** — срабатывает при любом вызове `console.log`, `console.warn`, `console.error` и т.д. на странице; через `msg.type()` можно отфильтровать именно `console.error` сообщения, которые часто указывают на проблему, даже если формально это не выброшенное исключение.

Типичный паттерн — собирать все ошибки в массив на протяжении теста и в конце (или в хуке `afterEach`) проверять, что массив пуст, либо явно фейлить тест при появлении критичной ошибки. Это позволяет ловить "тихие" баги фронтенда, которые не проявляются в UI напрямую (например, ошибка в неиспользуемом сейчас код-пути), но могут сигнализировать о проблемах с качеством кода.

Термин `WebError` в формулировке вопроса, вероятно, отсылает к событию `pageerror` (в некоторых версиях документации и API типизация ошибки называется схожим образом) — суть в том же: это ошибка, возникшая в JS-контексте самой веб-страницы, а не в Node.js-коде теста.

Такой подход особенно полезен в связке с CI: логирование всех перехваченных ошибок в отчёт теста даёт дополнительную диагностическую информацию даже для формально "зелёных" тестов.

## Пример / когда применяется

```javascript
const pageErrors = [];
const consoleErrors = [];

page.on('pageerror', (error) => {
  pageErrors.push(error.message);
});

page.on('console', (msg) => {
  if (msg.type() === 'error') {
    consoleErrors.push(msg.text());
  }
});

await page.goto('https://example.com');
await page.click('#buy-button');

// Явная проверка после выполнения сценария
expect(pageErrors, 'На странице возникли необработанные JS-ошибки').toHaveLength(0);
expect(consoleErrors, 'В консоли есть ошибки').toHaveLength(0);
```

Применяется в регрессионных и smoke-тестах, где помимо проверки видимого поведения UI важно убедиться, что страница не генерирует скрытых ошибок — это особенно ценно при рефакторинге фронтенда или обновлении сторонних библиотек.

## На что смотрит интервьюер

Интервьюер проверяет, знает ли кандидат, что JS-ошибки на странице сами по себе не приводят к падению теста и что для их обнаружения нужна явная подписка на `pageerror`/`console`. Частая ошибка — путать `pageerror` (необработанные исключения) с `console` (все сообщения консоли, включая обычные логи), что приводит либо к пропуску реальных ошибок, либо к ложным срабатываниям на безобидных `console.log`. Также могут спросить про накопление ошибок в массив и финальную проверку в конце теста, а также про интеграцию такой проверки в общий helper/fixture для переиспользования во всех тестах проекта.
