# Как в Playwright замокать ответ API?

**Коротко:** Для мокирования ответа API используется `page.route()` в связке с `route.fulfill()`, который возвращает заранее заданный статус, заголовки и тело ответа без реального обращения к серверу.

## Развёрнутый ответ

Мокирование ответа API в Playwright реализуется через перехват запроса методом `page.route(urlPattern, handler)` и вызов `route.fulfill({...})` внутри обработчика. `fulfill` принимает объект с полями: `status` (HTTP-код), `contentType`, `headers`, `body` (строка или буфер) или `json` (для автоматической сериализации объекта в JSON с нужным `content-type`).

Такой подход позволяет полностью изолировать frontend-тесты от backend: тест не зависит от доступности реального API, от состояния тестовых данных на сервере, от сетевых задержек. Это особенно полезно для проверки edge-кейсов UI — ошибок сервера (500), пустых списков, таймаутов, специфичных структур данных, которые сложно или дорого воспроизвести на реальном бэкенде.

Существует два основных сценария:

1. **Полная подмена ответа** — запрос вообще не уходит на реальный сервер, `fulfill` сразу возвращает сформированные данные.
2. **Модификация реального ответа** — сначала выполняется настоящий запрос через `route.fetch()`, полученный ответ парсится и частично изменяется, затем возвращается модифицированная версия через `route.fulfill({ response, json: modifiedData })`. Это полезно, когда нужно изменить только часть данных, сохранив остальную структуру реального ответа.

Также можно использовать `page.routeFromHAR()` для воспроизведения ответов из ранее записанного HAR-файла — это удобно для стабильных регрессионных тестов, где нужно гарантированно одинаковое поведение сервера при каждом прогоне.

## Пример / когда применяется

```javascript
// Полная подмена ответа API
await page.route('**/api/products', async (route) => {
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify([
      { id: 1, name: 'Test Product', price: 100 },
    ]),
  });
});

await page.goto('https://example.com/shop');

// Модификация реального ответа
await page.route('**/api/user/profile', async (route) => {
  const response = await route.fetch();
  const json = await response.json();
  json.isPremium = true;
  await route.fulfill({ response, json });
});
```

Применяется при тестировании UI-логики, зависящей от разных состояний API: пустая корзина, ошибка 404/500, долгая загрузка, специфичные роли пользователя.

## На что смотрит интервьюер

Проверяется знание метода `route.fulfill()` и его параметров, понимание разницы между полной подменой и частичной модификацией ответа (`route.fetch()` + `fulfill`), а также знание `routeFromHAR` как альтернативы. Частая ошибка — мокировать данные так, что структура ответа не соответствует реальному контракту API, из-за чего тест проходит, а в проде функциональность ломается. Также важно упомянуть, что моки не заменяют полностью интеграционные и e2e-тесты с реальным API.
