# Как включить запись видео или трассировки для отладки упавших тестов в Playwright?

**Коротко:** Видео и трассировка включаются через настройки `use` в конфигурации Playwright Test (`video`, `trace`) либо программно через `context.tracing.start()`/`stop()` и опцию `recordVideo` при создании контекста.

## Развёрнутый ответ

Playwright предоставляет два независимых механизма для последующего анализа падений: запись видео и трассировка (trace).

**Видео** записывается на уровне браузерного контекста. В конфиге `playwright.config.ts` задаётся опция `use.video`, которая может принимать значения: `'off'`, `'on'` (всегда писать), `'retain-on-failure'` (сохранять видео только при падении теста), `'on-first-retry'` (писать видео только при первом повторном прогоне упавшего теста). Видео сохраняется в формате webm в папке с результатами теста. При программном создании контекста видео включается через `browser.newContext({ recordVideo: { dir: 'videos/' } })`.

**Трассировка** — более мощный инструмент, который записывает снимки DOM-дерева на каждом шаге, сетевые запросы, консольные логи, скриншоты до и после каждого действия, а также исходный код теста с привязкой к таймлайну. Включается через опцию `use.trace` со значениями `'off'`, `'on'`, `'retain-on-failure'`, `'on-first-retry'`. Программно управляется методами `context.tracing.start({ screenshots: true, snapshots: true })` перед тестом и `context.tracing.stop({ path: 'trace.zip' })` после.

Обе опции рекомендуется настраивать на значение `retain-on-failure` в CI, чтобы не тратить диск и время на сборку артефактов для успешных тестов, но при этом иметь полную диагностику для упавших. Собранные трассировки анализируются в Trace Viewer — отдельном инструменте с графическим интерфейсом.

## Пример / когда применяется

```javascript
// playwright.config.ts
export default defineConfig({
  use: {
    video: 'retain-on-failure',
    trace: 'retain-on-failure',
    screenshot: 'only-on-failure',
  },
});
```

```javascript
// Программное управление трассировкой внутри теста
const context = await browser.newContext();
await context.tracing.start({ screenshots: true, snapshots: true });

const page = await context.newPage();
await page.goto('https://example.com');
// ... шаги теста ...

await context.tracing.stop({ path: 'trace.zip' });
```

Применяется в CI/CD пайплайнах, где недоступен интерактивный дебаг: артефакты (видео, trace.zip, скриншоты) прикладываются к отчёту о падении и позволяют разработчику или QA восстановить полную картину без повторного запуска теста.

## На что смотрит интервьюер

Ожидается знание конкретных значений опций (`off`, `on`, `retain-on-failure`, `on-first-retry`) и их практического смысла для баланса между объёмом артефактов и полнотой диагностики. Частая ошибка — включать `video: 'on'` и `trace: 'on'` для всех тестов в CI, что резко увеличивает время и объём хранимых данных. Также спрашивают про отличие видео (визуальная запись) от трассировки (интерактивная пошаговая диагностика с DOM, сетью и консолью) и про то, как трассировка анализируется в Trace Viewer.
