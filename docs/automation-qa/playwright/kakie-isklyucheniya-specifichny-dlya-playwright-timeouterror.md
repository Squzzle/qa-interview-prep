# Какие исключения специфичны для Playwright (`TimeoutError`, `PlaywrightException`)?

**Коротко:** Основное специфичное исключение Playwright — `TimeoutError`, возникающее при превышении времени ожидания действия или события; в Python-версии есть базовый класс `Error` (в некоторых версиях `PlaywrightException`), а в JS/TS ошибки представлены обычными объектами `Error` с характерными сообщениями.

## Развёрнутый ответ

Playwright не вводит большую иерархию собственных типов исключений, как, например, Selenium с его `NoSuchElementException`, `StaleElementReferenceException` и т.д. Вместо этого используется небольшой набор ошибок:

- **`TimeoutError`** — самое частое исключение, выбрасывается, когда действие (клик, ожидание видимости элемента, навигация) не укладывается в заданный таймаут (`timeout` в опциях метода или глобальный `actionTimeout`/`navigationTimeout`). Это может происходить из-за того, что элемент не появился в DOM, остался невидимым, задизейблен, перекрыт другим элементом, либо страница не завершила навигацию.
- **`Error` (в Python — базовый класс, в некоторых версиях документации назывался `PlaywrightException`)** — общая ошибка выполнения операций Playwright, например при попытке взаимодействия с закрытой страницей/контекстом/браузером, при ошибке селектора, при падении браузерного процесса.
- Ошибки, связанные с закрытием ресурсов: попытка вызвать метод на уже закрытой странице (`Target page, context or browser has been closed`) или закрытом браузере.

В отличие от Selenium, Playwright встроенно использует авто-ожидание (auto-waiting) перед каждым действием: он ждёт, пока элемент станет actionable (видимым, стабильным, не задизейбленным, не перекрытым) в течение заданного таймаута, и только после исчерпания времени выбрасывает `TimeoutError`. Это снижает количество "случайных" исключений вроде `ElementNotInteractableException` из Selenium, так как большинство подобных состояний Playwright сам дожидается.

При отладке `TimeoutError` важно смотреть на сообщение об ошибке — Playwright обычно подробно указывает причину (например, "element is not visible", "element is outside of the viewport", "waiting for selector ... to be visible"), что упрощает диагностику по сравнению с более общими сообщениями в других фреймворках.

## Пример / когда применяется

```javascript
try {
  await page.click('#submit-button', { timeout: 5000 });
} catch (error) {
  if (error.name === 'TimeoutError') {
    console.log('Элемент не стал кликабельным за отведённое время');
  }
  throw error;
}
```

```python
from playwright.sync_api import TimeoutError as PlaywrightTimeoutError

try:
    page.click("#submit-button", timeout=5000)
except PlaywrightTimeoutError:
    print("Элемент не стал кликабельным за отведённое время")
```

Обработка таких исключений применяется для формирования понятных сообщений об ошибке в отчётах, для реализации retry-логики на уровне отдельных шагов и для различения "настоящих" багов от инфраструктурных сбоев (например, браузер закрылся неожиданно).

## На что смотрит интервьюер

Интервьюер проверяет понимание, что у Playwright небольшой и предсказуемый набор исключений по сравнению с Selenium, и что большая часть проблем с ожиданием элементов покрывается встроенным auto-waiting механизмом, а не try/except вокруг каждого действия. Частая ошибка — оборачивать много кода в блоки try/except "на всякий случай" вместо того, чтобы настроить корректные таймауты и селекторы. Также могут спросить, как отличить `TimeoutError` от других ошибок и как использовать сообщение об ошибке для диагностики причины (элемент невидим, перекрыт, задизейблен).
