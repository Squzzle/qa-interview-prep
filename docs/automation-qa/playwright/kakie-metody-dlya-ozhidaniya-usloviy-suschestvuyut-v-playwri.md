# Какие методы для ожидания условий существуют в Playwright (`waitForSelector`, `waitForFunction`, `waitForURL`)?

**Коротко:** Помимо неявного auto-waiting внутри действий, Playwright предоставляет набор явных методов ожидания для случаев, не привязанных напрямую к конкретному действию над элементом: `waitForSelector` (состояние элемента), `waitForFunction` (произвольное JS-условие), `waitForURL` (навигация), а также `waitForResponse`/`waitForRequest` (сеть) и `waitForLoadState` (состояние загрузки страницы).

## Развёрнутый ответ

**`page.waitForSelector(selector, options)`** — ожидает, пока элемент, соответствующий селектору, достигнет заданного состояния: `attached` (по умолчанию), `detached`, `visible` или `hidden`. Используется, когда нужно явно синхронизироваться с появлением/исчезновением элемента вне контекста конкретного действия — например, дождаться исчезновения спиннера загрузки перед дальнейшими проверками. Стоит отметить, что в современном API Playwright предпочтительнее использовать `locator.waitFor({ state })` — метод самого `Locator`, а не устаревающий глобальный `page.waitForSelector`.

**`page.waitForFunction(pageFunction, arg, options)`** — выполняет произвольную JavaScript-функцию в контексте браузера с поллингом, пока она не вернёт truthy-значение. Применяется для сложных условий, которые нельзя выразить через простой селектор — например, дождаться, пока значение в `window` достигнет нужного состояния, пока количество элементов в списке не превысит порог, или пока не завершится анимация, отслеживаемая через кастомную JS-переменную.

**`page.waitForURL(url, options)`** — ожидает, пока URL страницы не будет соответствовать заданному значению (строке, шаблону с wildcard или регулярному выражению). Используется после действий, инициирующих навигацию (клик по ссылке, отправка формы), чтобы убедиться, что переход на новую страницу/маршрут SPA действительно произошёл, прежде чем продолжать сценарий.

**`page.waitForResponse(urlOrPredicate, options)`** и **`page.waitForRequest(urlOrPredicate, options)`** — ожидают конкретный сетевой ответ/запрос, соответствующий URL или предикату. Крайне полезны для синхронизации теста с асинхронными AJAX/fetch-запросами, когда нужно дождаться именно ответа от конкретного API-эндпоинта, а не полагаться на косвенные признаки (например, появление элемента после ответа).

**`page.waitForLoadState(state)`** — ожидает состояние загрузки страницы: `load` (полная загрузка, включая ресурсы), `domcontentloaded` (DOM построен) или `networkidle` (нет сетевой активности в течение периода — устаревающий, не рекомендуется для SPA с фоновым polling).

**`page.waitForEvent(event, options)`** — ожидает произвольное событие объекта `Page` или `BrowserContext`, например `'popup'` (открытие нового окна), `'download'` (начало скачивания файла), `'dialog'` (появление `alert`/`confirm`/`prompt`).

Общий принцип для всех этих методов — они дополняют, а не заменяют auto-waiting: auto-waiting работает неявно перед каждым действием над элементом, а явные `waitForX` нужны, когда условие не связано напрямую с готовностью конкретного элемента к взаимодействию (навигация, сеть, кастомная логика, события).

## Пример / когда применяется

```javascript
// Дождаться исчезновения спиннера загрузки
await page.locator('.spinner').waitFor({ state: 'hidden' });

// Дождаться перехода на страницу после логина
await page.getByRole('button', { name: 'Войти' }).click();
await page.waitForURL('**/dashboard');

// Дождаться конкретного сетевого ответа перед проверкой данных на странице
const [response] = await Promise.all([
  page.waitForResponse(resp => resp.url().includes('/api/orders') && resp.status() === 200),
  page.getByRole('button', { name: 'Обновить' }).click(),
]);

// Дождаться произвольного JS-условия
await page.waitForFunction(() => document.querySelectorAll('.item').length >= 10);
```

## На что смотрит интервьюер

- Знание конкретного назначения каждого метода, а не общей фразы "они все ждут элемент".
- Понимание, что `waitForSelector` на уровне `page` считается устаревающим паттерном в пользу `locator.waitFor()`.
- Способность привести пример, где `waitForResponse` необходим (синхронизация с конкретным API-запросом), а не просто "подождать элемент".
- Знание, что `networkidle` не рекомендуется как основной способ ожидания в SPA с постоянным фоновым трафиком (веб-сокеты, аналитика, polling).
- Красный флаг — предложение использовать `waitForTimeout` (фиксированную паузу) как основной способ синхронизации вместо перечисленных условных ожиданий.
