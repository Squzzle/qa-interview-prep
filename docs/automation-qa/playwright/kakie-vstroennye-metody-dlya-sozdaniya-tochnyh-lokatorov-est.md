# Какие встроенные методы для создания точных локаторов есть в Playwright (`page.getByRole()`, `page.getByText()`, `page.getByTestId()`)?

**Коротко:** Playwright предоставляет набор семантических локаторов (`getByRole`, `getByText`, `getByLabel`, `getByPlaceholder`, `getByAltText`, `getByTitle`, `getByTestId`), которые ищут элементы так, как их воспринимает пользователь или accessibility-дерево, а не по хрупким CSS/XPath-селекторам, привязанным к разметке.

## Развёрнутый ответ

Playwright продвигает подход "user-facing locators" — поиск элементов по атрибутам, значимым для конечного пользователя или ассистивных технологий, а не по внутренней структуре HTML/CSS, которая может часто меняться при рефакторинге вёрстки.

Основные встроенные методы:

- **`page.getByRole(role, options)`** — находит элемент по его ARIA-роли (`button`, `link`, `heading`, `checkbox`, `textbox` и т.д.) и, опционально, по доступному имени (`name`) — тексту, который видит пользователь или скринридер. Это наиболее рекомендуемый метод, так как он проверяет реальную семантику интерфейса и его доступность.
- **`page.getByText(text, options)`** — находит элемент по видимому текстовому содержимому (можно точное совпадение, подстроку или регулярное выражение).
- **`page.getByLabel(text)`** — находит элемент формы (input, select и т.д.) по тексту связанного с ним `<label>`.
- **`page.getByPlaceholder(text)`** — находит поле ввода по атрибуту `placeholder`.
- **`page.getByAltText(text)`** — находит элемент (обычно `<img>`) по атрибуту `alt`.
- **`page.getByTitle(text)`** — находит элемент по атрибуту `title`.
- **`page.getByTestId(testId)`** — находит элемент по выделенному тестовому атрибуту (по умолчанию `data-testid`, но имя атрибута настраивается через `testIdAttribute` в конфиге). Используется, когда у элемента нет уникальной семантической роли/текста, либо когда команда сознательно хочет отвязать тесты от текста интерфейса (который может переводиться на разные языки) и от структуры DOM.

Каждый из этих методов возвращает `Locator`, который можно дополнительно уточнять цепочкой методов: `.filter({ hasText })`, `.first()`, `.nth()`, `.and()`, вложенным `.locator()` внутри найденного элемента и т.д.

Иерархия предпочтений, рекомендуемая документацией Playwright: сначала `getByRole` (максимально приближен к тому, как пользователь/скринридер воспринимает страницу), затем текстовые/лейбловые локаторы (`getByText`, `getByLabel`, `getByPlaceholder`), и только при отсутствии подходящей семантики — `getByTestId`, а CSS/XPath-селекторы (`page.locator('css=...')`, `page.locator('xpath=...')`) рассматриваются как крайний случай, поскольку они привязаны к деталям реализации разметки.

## Пример / когда применяется

```javascript
// По роли и доступному имени — самый устойчивый вариант
await page.getByRole('button', { name: 'Отправить' }).click();

// По тексту
await page.getByText('Добро пожаловать').isVisible();

// По лейблу формы
await page.getByLabel('Электронная почта').fill('test@example.com');

// По data-testid, когда семантики недостаточно
await page.getByTestId('checkout-total').textContent();
```

Пример комбинации с фильтрацией — найти кнопку "Удалить" внутри конкретной строки таблицы:

```javascript
await page
  .getByRole('row', { name: 'Заказ #123' })
  .getByRole('button', { name: 'Удалить' })
  .click();
```

## На что смотрит интервьюер

- Знание конкретных названий методов и их назначения, а не общей фразы "там есть удобные локаторы".
- Понимание, что `getByRole` опирается на accessibility-дерево (ARIA-роли), а не только на видимый текст.
- Знание, что `data-testid` можно переопределить через конфиг (`testIdAttribute`).
- Follow-up: "Какой локатор выберете, если у элемента нет ни роли, ни текста, ни testid?" (ответ: CSS-селектор как крайняя мера, желательно попросить разработчиков добавить testid).
- Красный флаг — предложение использовать XPath/CSS как основной метод локации без объяснения, почему это менее устойчиво.
