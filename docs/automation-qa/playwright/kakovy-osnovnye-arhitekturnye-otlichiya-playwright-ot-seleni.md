# Каковы основные архитектурные отличия Playwright от Selenium (WebSocket vs HTTP, встроенные драйверы)?

**Коротко:** Selenium управляет браузером через синхронный HTTP-протокол WebDriver и отдельные драйверы (ChromeDriver, GeckoDriver), а Playwright общается с браузером напрямую по постоянному WebSocket-соединению через протоколы вроде CDP, без промежуточных драйверов.

## Развёрнутый ответ

Архитектура Selenium построена вокруг стандарта W3C WebDriver: тестовый код отправляет HTTP-запросы к серверу драйвера (например, ChromeDriver), а тот транслирует команды в браузер через его нативный протокол автоматизации. Каждая команда — это отдельный HTTP-запрос/ответ (request/response), что добавляет накладные расходы на сериализацию, установление соединения и ожидание ответа. Кроме того, для каждого браузера нужен свой отдельный драйвер, версия которого должна соответствовать версии браузера, что порождает проблему совместимости версий.

Playwright устроен принципиально иначе. Он не использует протокол WebDriver и не требует внешних драйверов, устанавливаемых пользователем отдельно. Вместо этого Playwright запускает браузер и устанавливает с ним постоянное соединение по WebSocket, используя низкоуровневые протоколы, специфичные для каждого движка: Chrome DevTools Protocol (CDP) для Chromium, и собственные патчи протоколов для Firefox (через модифицированный Juggler) и WebKit (через модифицированный протокол удалённой отладки WebKit). Все три браузера при установке Playwright скачиваются в виде уже пропатченных сборок, содержащих нужные хуки для автоматизации "из коробки".

Ключевые следствия такой архитектуры:

- **Постоянное соединение vs повторные запросы.** WebSocket-канал остаётся открытым, что позволяет серверу (браузеру) самому присылать события (например, о загрузке сети, консольных сообщениях, диалогах) без поллинга — это и быстрее, и даёт больше возможностей для перехвата событий в реальном времени.
- **Отсутствие отдельных драйверов.** Не нужно вручную скачивать и обновлять ChromeDriver/GeckoDriver и следить за соответствием версий браузеру — Playwright управляет этим сам через `playwright install`.
- **Единый процесс на уровне браузера, а не вкладки.** Playwright оперирует понятиями `Browser` → `BrowserContext` → `Page`, что даёт изолированные "инкогнито"-подобные контексты внутри одного процесса браузера, тогда как в Selenium для изоляции обычно поднимают отдельные экземпляры драйвера/браузера.
- **Более глубокий доступ к внутренним событиям браузера** — благодаря CDP/аналогам Playwright может перехватывать сетевые запросы, эмулировать геолокацию, менять таймзону и локаль, эмулировать устройства без необходимости менять DesiredCapabilities на уровне драйвера.

Из-за прямого протокольного взаимодействия команды в Playwright выполняются быстрее и с меньшей вероятностью гонок (race conditions), так как встроенный auto-waiting опирается на реальные события DOM/сети, получаемые через тот же канал.

## Пример / когда применяется

| Параметр | Selenium | Playwright |
|---|---|---|
| Протокол | HTTP (W3C WebDriver) | WebSocket (CDP и аналоги) |
| Нужен внешний драйвер | Да (ChromeDriver, GeckoDriver и т.д.) | Нет, браузеры пропатчены и встроены |
| Тип соединения | Запрос-ответ на каждую команду | Постоянное соединение с событиями |
| Синхронизация версий браузер/драйвер | Требуется вручную или через Selenium Manager | Не требуется |

На практике архитектурная разница проявляется в скорости и стабильности: Playwright-тест, ожидающий появления элемента после AJAX-запроса, полагается на событие сети/DOM, пришедшее по WebSocket, тогда как аналогичный Selenium-тест чаще требует явных `WebDriverWait` с поллингом через HTTP.

## На что смотрит интервьюер

- Понимание разницы между протоколом WebDriver (HTTP, request/response) и CDP/WebSocket (постоянное соединение, события).
- Знание, что в Selenium нужны отдельные драйверы под каждый браузер, а в Playwright браузеры уже "пропатчены" и скачиваются вместе с библиотекой.
- Умение связать архитектуру с практическими следствиями: скоростью, стабильностью, встроенным auto-waiting.
- Частая ошибка — путать протокол CDP с "просто более новым API Selenium"; нужно чётко разделять уровни абстракции.
- Follow-up: "А как Playwright работает с Firefox/WebKit, если CDP — это протокол Chromium?" (ответ: используются модифицированные аналоги протокола, а не сам CDP).
