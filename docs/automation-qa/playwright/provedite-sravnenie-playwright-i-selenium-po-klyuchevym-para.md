# Проведите сравнение Playwright и Selenium по ключевым параметрам: auto-waiting, изоляция тестов, скорость, работа с сетью.

**Коротко:** Playwright выигрывает по всем перечисленным параметрам за счёт архитектуры: встроенный умный auto-waiting вместо ручных ожиданий, изолированные `BrowserContext` вместо полноценных инстансов браузера, более высокая скорость благодаря WebSocket-протоколу и нативный перехват/подмена сетевых запросов без прокси.

## Развёрнутый ответ

**Auto-waiting.** В Selenium ожидание готовности элемента реализуется вручную через `WebDriverWait` с `ExpectedConditions` (например, `elementToBeClickable`) или через неявные ожидания (`implicitlyWait`), которые считаются антипаттерном из-за непредсказуемого поведения при смешивании с явными ожиданиями. В Playwright auto-waiting встроен в каждое действие (`click`, `fill`, `check` и т.д.): перед выполнением действия фреймворк сам дожидается, что элемент присоединён к DOM, видим, стабилен (не анимируется), доступен для взаимодействия (не перекрыт) и включён. Это устраняет большинство "flaky" тестов без написания дополнительного кода ожидания.

**Изоляция тестов.** В Selenium для полной изоляции между тестами обычно перезапускают весь экземпляр браузера/драйвера (что дорого по ресурсам и времени) либо чистят куки/localStorage вручную. В Playwright для этого существует `BrowserContext` — лёгкий аналог отдельного профиля браузера (аналог incognito-сессии) внутри одного и того же процесса браузера. Каждый контекст имеет собственные куки, localStorage, кэш и сессионное хранилище, при этом создание нового контекста занимает миллисекунды, в отличие от полного перезапуска браузера.

**Скорость.** За счёт WebSocket-соединения и отсутствия HTTP round-trip на каждую команду, а также благодаря дешёвому созданию `BrowserContext` вместо полного релонча браузера, Playwright обычно выполняет тесты быстрее, особенно при параллельном запуске: несколько контекстов могут работать в рамках одного процесса браузера, снижая расход памяти и CPU по сравнению с параллельным запуском нескольких полноценных инстансов Selenium.

**Работа с сетью.** Selenium исторически не имеет встроенного API для перехвата/подмены сетевых запросов — для этого нужны внешние прокси (BrowserMob Proxy) или недавно появившийся Selenium BiDi/CDP-интеграция с ограниченными возможностями. Playwright предоставляет нативный API `page.route()`, позволяющий перехватывать, модифицировать, блокировать запросы и подделывать (mock) ответы напрямую, без сторонних прокси, а также `page.waitForResponse()`/`waitForRequest()` для точной синхронизации тестов с сетевыми событиями.

## Пример / когда применяется

| Параметр | Selenium | Playwright |
|---|---|---|
| Auto-waiting | Нужны явные `WebDriverWait`/`ExpectedConditions` | Встроено в каждое действие (actionability checks) |
| Изоляция | Полный перезапуск браузера или ручная очистка состояния | `BrowserContext` — лёгкая изоляция за миллисекунды |
| Скорость | Ограничена HTTP round-trip и весом полного браузера | Выше за счёт WebSocket и лёгких контекстов |
| Работа с сетью | Требует внешние прокси или ограниченный BiDi/CDP | Встроенный `page.route()` для перехвата и mock |

Пример перехвата сети в Playwright:

```javascript
await page.route('**/api/users', route =>
  route.fulfill({ status: 200, body: JSON.stringify([{ id: 1, name: 'Test' }]) })
);
await page.goto('https://example.com/users');
```

## На что смотрит интервьюер

- Может ли кандидат раскрыть каждый из четырёх параметров конкретно, а не общими фразами вроде "Playwright быстрее и лучше".
- Знание конкретных механизмов Selenium (`WebDriverWait`, `ExpectedConditions`, `implicitlyWait`) для контраста.
- Понимание, что такое `BrowserContext` и почему он дешевле, чем новый инстанс браузера.
- Знание API `page.route()` и `waitForResponse` для сетевого взаимодействия.
- Красный флаг — заявление, что Selenium "вообще не умеет" ждать элементы или работать с сетью (умеет, но менее удобно и не "из коробки").
