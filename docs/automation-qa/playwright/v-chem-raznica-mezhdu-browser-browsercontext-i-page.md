# В чем разница между `Browser`, `BrowserContext` и `Page` в Playwright?

**Коротко:** `Browser` — это запущенный процесс браузерного движка, `BrowserContext` — изолированная сессия (аналог отдельного профиля/incognito-окна) внутри этого процесса, а `Page` — конкретная вкладка внутри контекста, с которой происходит непосредственное взаимодействие.

## Развёрнутый ответ

Эти три сущности образуют иерархию с разным уровнем изоляции и разной "стоимостью" создания/уничтожения.

**`Browser`** — самый "тяжёлый" объект: реальный запущенный процесс браузерного движка (Chromium/Firefox/WebKit). Его создание и закрытие занимает заметное время (десятки-сотни миллисекунд и больше), поэтому в тестовых сьютах его обычно запускают один раз на весь прогон или на файл тестов, а не пересоздают перед каждым тестом.

**`BrowserContext`** — лёгкая изолированная сессия внутри уже запущенного `Browser`. Это концептуальный аналог отдельного окна в режиме инкогнито: свои куки, свой localStorage/sessionStorage, свой кэш, свои разрешения (permissions), своя геолокация и viewport. Создание нового контекста занимает миллисекунды, что делает его идеальным инструментом изоляции между тестами: вместо перезапуска целого браузера для каждого теста создаётся новый `BrowserContext`, а сам `Browser` переиспользуется. По умолчанию контексты не делят состояние друг с другом — данные одного теста не "утекают" в другой.

**`Page`** — представление одной вкладки (или всплывающего окна) внутри конкретного `BrowserContext`. Именно на уровне `Page` выполняется вся содержательная работа теста: навигация по URL, поиск элементов через `Locator`, клики, ввод текста, ожидания, скриншоты, работа с диалогами (`page.on('dialog', ...)`), доступ к консоли браузера и сети. Один `BrowserContext` может содержать несколько `Page` одновременно — например, если сценарий предполагает открытие ссылки в новой вкладке или всплывающего окна (popup).

Таким образом, отношение "один ко многим" работает на каждом уровне: один `Browser` может иметь много `BrowserContext`, а один `BrowserContext` — много `Page`. Данные и состояние (куки, storage) изолированы на уровне `BrowserContext`, а не `Page` — то есть две вкладки в одном контексте видят одни и те же куки и localStorage, что имитирует поведение двух вкладок в одном реальном окне браузера.

## Пример / когда применяется

| Сущность | Что это | Стоимость создания | Изоляция |
|---|---|---|---|
| `Browser` | Запущенный процесс движка | Высокая | Между процессами |
| `BrowserContext` | Сессия/профиль внутри браузера | Низкая (мс) | Куки, storage, permissions |
| `Page` | Вкладка внутри контекста | Очень низкая | Нет — общий storage с контекстом |

Пример, когда несколько `Page` полезны в одном `BrowserContext` — тест сценария "пригласить друга по ссылке", где одна страница открывает ссылку в новой вкладке:

```javascript
const [newPage] = await Promise.all([
  context.waitForEvent('page'),
  page.click('a[target="_blank"]'),
]);
await newPage.waitForLoadState();
```

Типовая практика в тестовом сьюте: один `Browser` запускается один раз на весь прогон (в `globalSetup` или на уровне воркера), а `BrowserContext` создаётся заново перед каждым тестом и закрывается после него — это даёт изоляцию без накладных расходов на перезапуск браузера.

## На что смотрит интервьюер

- Чёткое разделение уровней: процесс (`Browser`) → сессия/профиль (`BrowserContext`) → вкладка (`Page`).
- Понимание, что изоляция куки/storage происходит именно на уровне `BrowserContext`, а не `Page`.
- Знание, что несколько `Page` в одном контексте разделяют одно и то же хранилище (как реальные вкладки одного окна).
- Follow-up: "Почему для изоляции тестов используют новый `BrowserContext`, а не новый `Browser`?" (дешевле по времени и ресурсам).
- Красный флаг — утверждение, что `Page` и `BrowserContext` — взаимозаменяемые понятия.
