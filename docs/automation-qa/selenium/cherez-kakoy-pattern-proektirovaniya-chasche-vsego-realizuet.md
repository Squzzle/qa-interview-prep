# Через какой паттерн проектирования чаще всего реализуется управление экземпляром WebDriver? В чем его суть (Singleton)?

**Коротко:** Чаще всего используется паттерн Singleton (в сочетании с `ThreadLocal` для потокобезопасности при параллельном запуске), который гарантирует, что в рамках одного потока/теста существует единственный, переиспользуемый экземпляр WebDriver, доступный через общую точку входа.

## Развёрнутый ответ

Singleton — это порождающий паттерн проектирования, суть которого в том, чтобы гарантировать существование только одного экземпляра класса в рамках приложения (или в данном случае — в рамках одного потока выполнения тестов) и предоставить глобальную точку доступа к этому экземпляру.

Применительно к WebDriver, Singleton используется, чтобы:
- Не создавать WebDriver заново в каждом классе или тестовом методе вручную, а получать его через единый статический метод, например `DriverManager.getDriver()`.
- Централизовать логику инициализации, конфигурации Options/Capabilities и закрытия драйвера в одном месте.
- Избежать утечек ресурсов и лишних процессов браузера от случайно созданных дублирующих экземпляров.

Важный нюанс: наивная реализация классического Singleton (простое статическое поле) не работает безопасно при **параллельном запуске тестов** (например, TestNG с `parallel="methods"` или JUnit 5 с parallel execution), потому что несколько потоков будут читать/писать одно и то же статическое поле, и один и тот же экземпляр WebDriver окажется общим для нескольких потоков одновременно — это приведёт к состоянию гонки и падениям тестов из-за смешивания команд между разными сессиями браузера.

Поэтому на практике Singleton комбинируют с `ThreadLocal<WebDriver>`: каждый поток получает свой собственный экземпляр драйвера, хранящийся в отдельной ячейке `ThreadLocal`, но обращение к нему по-прежнему происходит через единственный статический класс-держатель (что и сохраняет суть Singleton — единая точка доступа, просто "единственность" теперь скоуплена на поток, а не на весь JVM-процесс).

Реализация обычно включает:
- Приватный конструктор (недоступен снаружи).
- Статический метод `getDriver()`, который лениво инициализирует драйвер при первом обращении в потоке.
- Метод `quitDriver()` для явного закрытия и удаления значения из `ThreadLocal` (важно вызывать `remove()`, чтобы избежать утечек памяти в пуле потоков).

## Пример / когда применяется

```java
public class DriverManager {
    private static final ThreadLocal<WebDriver> driverThread = new ThreadLocal<>();

    private DriverManager() {}

    public static WebDriver getDriver() {
        if (driverThread.get() == null) {
            ChromeOptions options = new ChromeOptions();
            driverThread.set(new ChromeDriver(options));
        }
        return driverThread.get();
    }

    public static void quitDriver() {
        WebDriver driver = driverThread.get();
        if (driver != null) {
            driver.quit();
            driverThread.remove();
        }
    }
}
```

Такой подход применяется в проектах с Page Object Model, где каждая страница получает драйвер через `DriverManager.getDriver()` без явной передачи через конструкторы, и особенно критичен в фреймворках с параллельным запуском тестов в CI.

## На что смотрит интервьюер

- Понимание сути Singleton (единственный экземпляр + глобальная точка доступа), а не просто "это статическая переменная".
- Знание проблемы гонки при параллельном запуске и решения через `ThreadLocal`.
- Понимание необходимости `remove()` в `ThreadLocal` для предотвращения утечек памяти при переиспользовании потоков (thread pool).
- Умение объяснить, зачем вообще нужен этот паттерн, а не просто создавать драйвер в каждом тесте (переиспользование, централизация конфигурации, поддержка параллелизма).
