# Что такое неявные ожидания (Implicit Wait)? Как они задаются?

**Коротко:** Implicit Wait — это глобальная настройка драйвера, задающая максимальное время, в течение которого `findElement`/`findElements` будет периодически опрашивать DOM в поисках элемента, прежде чем выбросить `NoSuchElementException`; задаётся один раз через `driver.manage().timeouts().implicitlyWait(...)` и действует на всю сессию.

## Развёрнутый ответ

По умолчанию Selenium ищет элемент мгновенно: если в момент вызова `findElement()` элемента ещё нет в DOM (например, страница/асинхронный компонент ещё не догрузились), сразу выбрасывается `NoSuchElementException`. Implicit Wait решает эту проблему на уровне всего драйвера: устанавливается таймаут, в течение которого WebDriver будет с определённым интервалом (обычно опрос происходит внутри самой реализации драйвера) повторно пытаться найти элемент, прежде чем окончательно вернуть ошибку.

Задаётся implicit wait один раз, как правило сразу после создания драйвера:

```java
driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
```

(в старых версиях Selenium — `implicitlyWait(10, TimeUnit.SECONDS)`).

После этого настройка действует **глобально для всей сессии драйвера** — то есть на все последующие вызовы `findElement`/`findElements`, а не только на конкретный элемент.

Ключевые нюансы, которые важно знать:

1. **Implicit Wait — это не "ожидание видимости" или "кликабельности".** Он реагирует только на присутствие элемента в DOM (по сути — на успешность поиска), а не на его видимость, включённость (`enabled`) или готовность к взаимодействию. Поэтому элемент может быть найден, но клик по нему всё равно упадёт с `ElementNotInteractableException`, если элемент скрыт или перекрыт.

2. **Не рекомендуется смешивать Implicit и Explicit Wait в одном тесте/проекте** — это приводит к непредсказуемым и труднообъяснимым таймаутам (подробнее — тема отдельного вопроса), поскольку суммарное время ожидания может стать неочевидным и непредсказуемым.

3. **Implicit Wait задаётся один раз на сессию**, но при необходимости его можно переопределить снова тем же вызовом в любой момент теста (например, временно увеличить/уменьшить).

4. **По умолчанию Implicit Wait равен 0** — то есть отключён, если явно не задан.

5. Начиная с некоторых версий Selenium/W3C-протокола, значение implicit wait передаётся в capabilities сессии как часть объекта `timeouts`.

## Пример / когда применяется

```java
WebDriver driver = new ChromeDriver();
driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));

driver.get("https://example.com");
WebElement button = driver.findElement(By.id("submit")); // будет ждать до 10 секунд появления элемента в DOM
```

Применяется в проектах, где ставка делается на простоту (единая глобальная настройка вместо явных ожиданий на каждый элемент), либо как "подстраховка" в сочетании с явными ожиданиями для конкретных сложных сценариев — хотя большинство экспертов рекомендуют использовать преимущественно Explicit Wait для предсказуемости.

## На что смотрит интервьюер

- Чёткое понимание: implicit wait реагирует только на присутствие элемента в DOM, а не на видимость/кликабельность.
- Знание, что настройка глобальная и задаётся один раз на сессию.
- Понимание рисков смешивания implicit и explicit wait (неочевидное суммарное время ожидания, "рваные" таймауты).
- Знание значения по умолчанию (0 — ожидание отключено).
- Красный флаг: если кандидат путает implicit wait с `Thread.sleep()` или считает, что он ждёт полной готовности элемента к взаимодействию.
