# Что такое Selenoid? Какую инфраструктурную проблему он решает?

**Коротко:** Selenoid — легковесная замена Selenium Grid, запускающая браузеры в изолированных Docker-контейнерах по требованию, что решает проблему тяжёлой, ресурсоёмкой и сложно масштабируемой инфраструктуры для параллельного запуска UI-тестов на разных версиях браузеров.

## Развёрнутый ответ

Классический Selenium Grid требует заранее развёрнутых и поддерживаемых узлов (Selenium Nodes) с установленными браузерами и драйверами конкретных версий, зарегистрированных на Hub. Это создаёт ряд инфраструктурных проблем:
- Нужно вручную поддерживать актуальные версии браузеров и WebDriver-бинарников на каждом узле.
- Постоянно запущенные ноды потребляют ресурсы, даже когда тесты не выполняются.
- Сложно быстро масштабировать количество параллельных сессий под нагрузку CI.
- Сложно изолировать сессии друг от друга — "грязное" состояние браузера может влиять на последующие тесты.

Selenoid (разработан компанией Aerokube) решает эти проблемы за счёт контейнеризации:
- Каждая сессия WebDriver запускает отдельный Docker-контейнер с нужным браузером и версией, указанной в `capabilities` теста (`browserName`, `browserVersion`).
- Контейнеры эфемерны — после завершения сессии (`driver.quit()`) контейнер уничтожается, что гарантирует чистое изолированное состояние для каждого теста и отсутствие "утечки" состояния между прогонами.
- Не требует предварительной установки браузеров на хосте — используются готовые Docker-образы браузеров с заранее настроенными версиями.
- Поддерживает горизонтальное масштабирование — количество параллельных сессий ограничено только ресурсами хоста/кластера, а не количеством заранее зарегистрированных нод.
- Предоставляет встроенный веб-интерфейс Selenoid UI для мониторинга активных сессий, а также запись видео сессий и логов браузера "из коробки" — что упрощает отладку падений тестов на CI без ручной настройки.
- Работает по тому же протоколу WebDriver (JSON Wire Protocol / W3C WebDriver Protocol), поэтому со стороны тестового кода никаких изменений не требуется — достаточно указать Selenoid как Remote URL вместо локального драйвера.

Таким образом, Selenoid решает именно инфраструктурную проблему "браузерной фермы": вместо содержания парка виртуальных машин с предустановленными браузерами команда поддерживает набор Docker-образов и получает изолированные, легко масштабируемые и легко воспроизводимые сессии браузеров по требованию.

## Пример / когда применяется

```java
DesiredCapabilities caps = new DesiredCapabilities();
caps.setBrowserName("chrome");
caps.setVersion("120.0");

WebDriver driver = new RemoteWebDriver(
    new URL("http://localhost:4444/wd/hub"), caps
);
```

Конфигурация образов (`browsers.json` для Selenoid):
```json
{
  "chrome": {
    "default": "120.0",
    "versions": {
      "120.0": {
        "image": "selenoid/chrome:120.0",
        "port": "4444"
      }
    }
  }
}
```

Применяется в CI/CD-пайплайнах, где нужно запускать десятки/сотни параллельных UI-тестов на разных версиях браузеров без содержания парка постоянных виртуальных машин.

## На что смотрит интервьюер

Проверяет понимание именно инфраструктурной проблемы (масштабирование, изоляция, версии браузеров), а не поверхностное "это как Selenium Grid, только лучше". Хороший ответ упоминает контейнеризацию по требованию, эфемерность контейнеров и встроенную запись видео/логов. Follow-up: «чем Selenoid отличается от Selenium Grid 4 с встроенной поддержкой Docker» (Grid 4 частично закрыл этот разрыв, добавив Docker-поддержку, но Selenoid остаётся более легковесным и специализированным решением с богатым UI и записью сессий) и «как обеспечивается изоляция между параллельными тестами» (каждая сессия — отдельный контейнер). Красный флаг — путаница между Selenoid и Selenium Grid как синонимами без понимания архитектурных отличий.
