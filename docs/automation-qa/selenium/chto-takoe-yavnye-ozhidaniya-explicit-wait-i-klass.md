# Что такое явные ожидания (Explicit Wait) и класс `WebDriverWait`? В чем их преимущество?

**Коротко:** Explicit Wait — это точечное ожидание конкретного условия (Expected Condition) для конкретного элемента/ситуации, реализуемое через класс `WebDriverWait`, которое опрашивает состояние с заданным интервалом до наступления условия или истечения таймаута, что делает тесты точнее и предсказуемее, чем глобальный Implicit Wait.

## Развёрнутый ответ

`WebDriverWait` — это класс из пакета `org.openqa.selenium.support.ui`, реализующий паттерн "опрос с интервалом" (polling): вместо того чтобы просто подождать фиксированное время (`Thread.sleep`) или полагаться на глобальную настройку (Implicit Wait), явное ожидание позволяет сказать: "жди именно вот этого конкретного условия, максимум N секунд, проверяя каждые X миллисекунд".

Создаётся экземпляр так:

```java
WebDriverWait wait = new WebDriverWait(driver, Duration.of(10, ChronoUnit.SECONDS));
```

И используется вместе с `ExpectedConditions`:

```java
WebElement element = wait.until(ExpectedConditions.elementToBeClickable(By.id("submit")));
element.click();
```

Внутри `until()` происходит цикл: WebDriverWait периодически (по умолчанию каждые 500 мс) вызывает переданное условие (`ExpectedCondition<T>`, по сути функциональный интерфейс `Function<WebDriver, T>`), и если оно возвращает не `null`/не `false`, ожидание завершается успешно и возвращает результат. Если время вышло — выбрасывается `TimeoutException`.

Преимущества Explicit Wait перед Implicit Wait:

1. **Точность условия.** Можно ждать не просто присутствия в DOM, а именно нужного состояния: видимость, кликабельность, исчезновение, изменение текста, наличие alert и т.д.
2. **Локальность.** Ожидание применяется к конкретному месту в коде теста, а не глобально ко всем поискам элементов — это делает поведение теста предсказуемым и читаемым.
3. **Гибкость таймаута.** Для разных операций можно задавать разное время ожидания (быстрая проверка — короткий таймаут, тяжёлая операция вроде загрузки отчёта — длинный).
4. **Управление игнорируемыми исключениями.** Через `wait.ignoring(StaleElementReferenceException.class)` можно указать, какие исключения не должны прерывать цикл опроса, что повышает устойчивость к "мерцающим" (flaky) сценариям, например при перерисовке DOM.
5. **Явность и читаемость кода** — по коду сразу видно, чего именно ожидает тест в данной точке, что упрощает поддержку и отладку.

По умолчанию `WebDriverWait` игнорирует `NotFoundException` (в частности `NoSuchElementException`) в процессе опроса — то есть если элемент временно не найден, ожидание не падает сразу, а продолжает пытаться до истечения таймаута.

## Пример / когда применяется

```java
WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(15));

// ждём кликабельности кнопки
WebElement submitBtn = wait.until(ExpectedConditions.elementToBeClickable(By.id("submit")));
submitBtn.click();

// ждём появления сообщения об успехе
wait.until(ExpectedConditions.textToBePresentInElementLocated(
        By.cssSelector(".notification"), "Успешно"));
```

Применяется практически в каждом реальном автотесте при взаимодействии с динамическим UI: после клика по кнопке, отправляющей AJAX-запрос, после навигации, при ожидании модальных окон, спиннеров загрузки и т.д. — это основной, рекомендуемый инструмент синхронизации в Selenium.

## На что смотрит интервьюер

- Понимание механизма polling (опрос с интервалом), а не просто "ждёт N секунд".
- Знание, что Explicit Wait привязан к конкретному условию и месту в коде, а не действует глобально.
- Умение объяснить `ignoring()` и работу с `StaleElementReferenceException`.
- Практический опыт: какие Expected Conditions использовались, для каких реальных сценариев (спиннеры, AJAX, модалки).
- Красный флаг: путаница между `WebDriverWait` и `Thread.sleep()`, непонимание, что происходит внутри `until()`.
