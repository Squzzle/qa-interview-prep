# Когда возникает `StaleElementReferenceException` и как с ним бороться?

**Коротко:** Исключение возникает, когда ссылка на `WebElement`, полученная ранее, перестаёт соответствовать актуальному DOM-узлу — например, после перезагрузки страницы, обновления через AJAX/JS-фреймворк или пересоздания элемента; бороться — повторным поиском элемента и оборачиванием в retry-логику или `ExpectedConditions.refreshed`.

## Развёрнутый ответ

`WebElement` в Selenium — это по сути ссылка на конкретный узел DOM, идентифицируемый внутренним ID узла в браузере. `StaleElementReferenceException` выбрасывается, когда этот узел был удалён из DOM или заменён новым, а тест пытается обратиться к старой, уже недействительной ссылке.

Типичные причины:
1. Полная перезагрузка страницы (`page reload`, переход по ссылке) — весь DOM пересоздаётся, все старые ссылки становятся невалидными.
2. Частичное обновление DOM через JavaScript/AJAX (SPA-фреймворки типа React/Angular/Vue часто перерисовывают часть дерева при изменении состояния) — даже визуально идентичный элемент на самом деле новый DOM-узел.
3. Действия, которые меняют структуру страницы: сортировка таблицы, фильтрация списка, удаление/добавление элементов.
4. Взаимодействие с элементом, который был найден до срабатывания анимации или отложенной перерисовки.

Способы борьбы:
- Повторный поиск элемента непосредственно перед взаимодействием, а не хранение ссылки в переменной надолго.
- Использование `ExpectedConditions.refreshed(condition)` вместе с `WebDriverWait`, который повторно ищет элемент, пока условие не станет истинным для нового узла.
- Реализация retry-обёртки (например, метод `retryClick`), который перехватывает `StaleElementReferenceException` и повторяет попытку N раз с небольшим интервалом.
- В Page Object Model — избегать сохранения `WebElement` в полях класса; лучше хранить локатор (`By`) и искать элемент каждый раз через метод.
- Использование библиотек с автоматическим retry, например Selenide, где `$` возвращает "ленивый" прокси-объект, который сам переискивает элемент при каждом обращении.

## Пример / когда применяется

```java
public void clickWithRetry(By locator, int attempts) {
    for (int i = 0; i < attempts; i++) {
        try {
            driver.findElement(locator).click();
            return;
        } catch (StaleElementReferenceException e) {
            if (i == attempts - 1) throw e;
        }
    }
}
```

Или через `WebDriverWait`:
```java
WebElement row = wait.until(
    ExpectedConditions.refreshed(
        ExpectedConditions.presenceOfElementLocated(By.xpath("//tr[3]"))
    )
);
```

## На что смотрит интервьюер

Ожидает конкретное объяснение механики (ссылка на DOM-узел, а не на "элемент вообще"), а не общую фразу "элемент устарел". Хороший ответ включает упоминание Page Object без хранения `WebElement` полей и конкретные механизмы борьбы (`refreshed`, retry). Follow-up: «почему в SPA это встречается чаще» и «как Selenide решает эту проблему из коробки» (через ленивую прокси-реализацию `SelenideElement`). Красный флаг — предложение "просто добавить `Thread.sleep()`" как единственное решение.
