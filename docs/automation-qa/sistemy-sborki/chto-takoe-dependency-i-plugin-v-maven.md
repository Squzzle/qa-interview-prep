# Что такое `dependency` и `plugin` в Maven?

**Коротко:** `dependency` — это внешняя библиотека, код которой используется проектом (например, Selenium, JUnit), а `plugin` — это инструмент, расширяющий возможности самого процесса сборки (компиляция, тестирование, упаковка, отчёты).

## Развёрнутый ответ

Оба понятия описываются в `pom.xml` через координаты groupId/artifactId/version, но выполняют принципиально разные роли.

**Dependency (зависимость)** — это библиотека, которую использует код проекта. Она подключается к classpath на этапе компиляции и/или выполнения. У зависимостей есть атрибут `scope`, определяющий, на каких этапах жизненного цикла она доступна:

- `compile` (по умолчанию) — доступна везде: при компиляции, тестировании, выполнении.
- `test` — доступна только при компиляции и запуске тестов (например, JUnit, TestNG, Mockito, RestAssured в тестовых модулях).
- `provided` — нужна для компиляции, но на этапе выполнения предоставляется окружением (например, servlet-api при деплое в контейнер приложений).
- `runtime` — не нужна для компиляции, но нужна во время выполнения (например, JDBC-драйвер).

**Plugin (плагин)** — это исполняемый компонент, который расширяет саму систему сборки, выполняя конкретные задачи (goals) на определённых фазах жизненного цикла. Maven по своей сути — это фреймворк-оркестратор, а вся реальная работа (компиляция, тестирование, упаковка) выполняется плагинами. Даже базовая компиляция кода реализована как `maven-compiler-plugin`.

Примеры распространённых плагинов в проектах автоматизации:

- `maven-compiler-plugin` — настройка версии Java для компиляции.
- `maven-surefire-plugin` — запуск unit-тестов на фазе test.
- `maven-failsafe-plugin` — запуск интеграционных тестов на фазе integration-test/verify.
- `maven-shade-plugin` / `maven-assembly-plugin` — сборка "толстого" (fat) jar со всеми зависимостями внутри.
- `allure-maven` — генерация Allure-отчётов по результатам тестов.

Плагин может выполнять одну или несколько целей (goals), и каждая цель обычно привязана к определённой фазе жизненного цикла (binding).

## Пример / когда применяется

Пример объявления зависимости с scope `test` и плагина для запуска тестов:

```xml
<dependencies>
    <dependency>
        <groupId>org.seleniumhq.selenium</groupId>
        <artifactId>selenium-java</artifactId>
        <version>4.15.0</version>
        <scope>test</scope>
    </dependency>
</dependencies>

<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-surefire-plugin</artifactId>
            <version>3.1.2</version>
            <configuration>
                <suiteXmlFiles>
                    <suiteXmlFile>testng.xml</suiteXmlFile>
                </suiteXmlFiles>
            </configuration>
        </plugin>
    </plugins>
</build>
```

Здесь Selenium — это dependency (используется в коде тестов), а Surefire — plugin (управляет процессом запуска этих тестов).

## На что смотрит интервьюер

Интервьюер хочет услышать чёткое разделение: dependency — "что использует код", plugin — "что делает сборка". Частая ошибка — путать назначение scope или не знать, что даже компиляция кода реализована через плагин. Хороший ответ включает конкретные примеры плагинов, используемых в тестовых проектах (Surefire, Failsafe, Allure), что показывает практический опыт, а не только теорию.
