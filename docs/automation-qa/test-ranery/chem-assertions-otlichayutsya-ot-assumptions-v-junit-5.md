# Чем `Assertions` отличаются от `Assumptions` в JUnit 5?

**Коротко:** `Assertions` проверяют бизнес-логику и, если условие не выполнено, помечают тест как упавший (failed); `Assumptions` проверяют предварительные условия для запуска теста и, если условие не выполнено, помечают тест как пропущенный (skipped/aborted), а не проваленный.

## Развёрнутый ответ

`org.junit.jupiter.api.Assertions` и `org.junit.jupiter.api.Assumptions` — оба класса содержат статические методы для проверки условий, но семантика провала у них принципиально разная:

- **`Assertions`** (`assertEquals`, `assertTrue`, `assertThrows` и т.д.) — используются для проверки самого результата тестируемой логики. Если условие ложно, выбрасывается `AssertionError`, и JUnit фиксирует статус теста как **FAILED**. Это основной механизм валидации ожидаемого поведения приложения.

- **`Assumptions`** (`assumeTrue`, `assumeFalse`, `assumingThat`) — используются для проверки предварительных условий среды выполнения (окружение, ОС, конфигурация, доступность внешнего сервиса и т.п.), которые не относятся напрямую к тестируемой логике. Если предположение не выполняется, тест прерывается (aborted) и помечается как **SKIPPED**, а не как проваленный — потому что причина не в баге приложения, а в неподходящих условиях для запуска.

Ключевые методы `Assumptions`:

- **`assumeTrue(condition)`** / **`assumeFalse(condition)`** — прерывает тест, если условие не выполнено.
- **`assumingThat(condition, executable)`** — выполняет блок кода только если условие истинно; в отличие от `assumeTrue`, не прерывает весь тест, а просто пропускает конкретный блок внутри теста, продолжая выполнение остального кода.

Практическая разница проявляется в отчётах CI: тесты, упавшие через `Assertions`, требуют внимания разработчиков/QA как баг или регрессия; тесты, пропущенные через `Assumptions`, не считаются провалом сборки и обычно не блокируют pipeline — они просто сигнализируют "этот тест неприменим в данном окружении".

## Пример / когда применяется

```java
@Test
void testOnlyOnLinux() {
    Assumptions.assumeTrue(System.getProperty("os.name").toLowerCase().contains("linux"));

    // код теста выполнится, только если ОС — Linux;
    // на других ОС тест будет помечен как SKIPPED
    Assertions.assertTrue(new File("/etc/hosts").exists());
}

@Test
void conditionalAssertionExample() {
    int value = getEnvironmentDependentValue();

    Assumptions.assumingThat(
        value > 100,
        () -> Assertions.assertEquals(150, value)
    );

    // этот код выполнится в любом случае, независимо от assumingThat
    Assertions.assertNotNull(value);
}
```

Типичное применение `Assumptions` — тесты, зависящие от окружения (ОС, локаль, наличие внешнего сервиса, флаг конфигурации), интеграционные тесты, которые должны выполняться только при доступности определённого стенда.

## На что смотрит интервьюер

- Понимает ли кандидат разницу в статусах результата: FAILED vs SKIPPED/ABORTED.
- Может ли привести реальный пример, когда стоит использовать `Assumptions` вместо `Assertions` (условия окружения, а не бизнес-логика).
- Знает ли, что чрезмерное использование `Assumptions` может маскировать реальные проблемы конфигурации CI (тесты тихо пропускаются, и это может остаться незамеченным).
- Частая ошибка — использовать `assumeTrue` для проверки бизнес-логики, из-за чего реальные баги превращаются в "пропущенные" тесты, а не в "провалы", и команда теряет видимость проблемы.
- Follow-up вопрос — как настроить CI, чтобы отличать skipped-тесты от passed в общей статистике и не терять сигнал о деградации покрытия.
