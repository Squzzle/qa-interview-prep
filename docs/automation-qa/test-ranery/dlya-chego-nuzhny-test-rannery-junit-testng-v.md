# Для чего нужны тест-раннеры (JUnit, TestNG) в автоматизации?

**Коротко:** тест-раннеры — это фреймворки, которые обнаруживают, запускают и управляют жизненным циклом автотестов, собирают результаты и предоставляют отчётность, освобождая инженера от написания собственной инфраструктуры запуска.

## Развёрнутый ответ

Тест-раннер (test runner framework) — это библиотека и инструмент, которые решают несколько задач в автоматизации тестирования:

- **Обнаружение тестов** — автоматический поиск методов, помеченных специальными аннотациями (`@Test` и подобные), без необходимости вручную регистрировать каждый тест.
- **Управление жизненным циклом** — выполнение подготовительных и завершающих действий (setup/teardown) на разных уровнях: перед/после каждого теста, перед/после класса, перед/после всего набора (suite).
- **Организация и группировка** — объединение тестов в наборы (suites), группы, категории, возможность выборочного запуска по тегам или именам.
- **Параметризация** — многократный запуск одного и того же теста с разными наборами входных данных.
- **Управление зависимостями и порядком** — определение приоритетов, зависимостей между тестами (актуально для TestNG).
- **Параллельный запуск** — распределение тестов по потокам/процессам для ускорения прогона.
- **Сбор и агрегация результатов** — формирование отчётов (pass/fail/skip), интеграция с CI/CD системами (Jenkins, GitLab CI и т.д.) через стандартизированные форматы отчётов (например, JUnit XML).
- **Обработка исключений и assert'ов** — стандартизированный механизм проверки условий и падения теста с понятным сообщением об ошибке.

JUnit и TestNG — два самых распространённых тест-раннера для JVM-стека (Java/Kotlin). Они не заменяют инструменты для взаимодействия с приложением (Selenium, REST-клиенты и т.д.), а служат "каркасом", в котором эти инструменты используются: раннер отвечает за то, *когда* и *в каком порядке* выполнится код теста, а сам код теста уже вызывает Selenium, RestAssured и другие библиотеки.

Без тест-раннера пришлось бы писать собственный `main`-метод, вручную вызывать методы тестов, самому ловить исключения и вести подсчёт успешных/неуспешных прогонов — то есть заново изобретать то, что уже давно решено индустриальными фреймворками.

## Пример / когда применяется

Типичный пример использования JUnit 5:

```java
class LoginTest {

    @BeforeEach
    void setUp() {
        // открыть браузер, залогиниться в тестовое окружение
    }

    @Test
    void shouldLoginWithValidCredentials() {
        // выполнение теста и проверка через assert
    }

    @AfterEach
    void tearDown() {
        // закрыть браузер
    }
}
```

Раннер сам найдёт метод `shouldLoginWithValidCredentials`, вызовет `setUp()` перед ним и `tearDown()` после, независимо от результата теста, и включит результат в общий отчёт прогона (например, при запуске через Maven Surefire/Failsafe в CI).

## На что смотрит интервьюер

- Понимает ли кандидат, что тест-раннер — это именно инфраструктура запуска, а не инструмент взаимодействия с UI/API.
- Может ли назвать конкретные возможности (аннотации, параметризация, параллельность, отчётность), а не отделаться общей фразой "чтобы тесты запускать".
- Знает ли разницу в подходах JUnit и TestNG (это часто follow-up вопрос).
- Частая ошибка — путать тест-раннер с фреймворком автоматизации (Selenium/Playwright) или с системой сборки (Maven/Gradle), которая его запускает.
- Красный флаг — неспособность объяснить, зачем нужны setup/teardown хуки и что будет, если их не использовать (дублирование кода, утечки ресурсов, нестабильные тесты).
