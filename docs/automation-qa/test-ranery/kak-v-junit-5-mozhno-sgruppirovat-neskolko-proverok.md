# Как в JUnit 5 можно сгруппировать несколько проверок так, чтобы тест упал один раз со всеми ошибками (`assertAll`)?

**Коротко:** для этого используется метод `Assertions.assertAll(...)`, который выполняет все переданные проверки независимо друг от друга и в случае провала нескольких из них выбрасывает единый `MultipleFailuresError`, содержащий сообщения обо всех упавших проверках сразу.

## Развёрнутый ответ

По умолчанию, если в тесте подряд идут несколько обычных `assert*` вызовов, то при провале первого же из них тест немедленно прерывается исключением, а до остальных проверок выполнение не доходит — соответственно, разработчик видит только первую ошибку, исправляет её, перезапускает тест и узнаёт о следующей ошибке только на следующей итерации. Это неэффективно при большом количестве независимых проверок в одном тесте (например, проверка нескольких полей одного объекта).

`assertAll` решает эту проблему: он принимает набор `Executable` (лямбда-выражений с проверками) и выполняет их все, независимо от того, упала предыдущая проверка или нет. Если хотя бы одна проверка провалилась, в конце выбрасывается единое исключение `MultipleFailuresError`, в котором перечислены сообщения обо всех неудачных проверках сразу — это позволяет за один прогон теста увидеть полную картину, а не чинить проблемы по одной.

Сигнатура метода:

```java
static void assertAll(Executable... executables)
static void assertAll(String heading, Executable... executables)
static void assertAll(Collection<Executable> executables)
```

Параметр `heading` — необязательный заголовок группы проверок, который добавляется в сообщение об ошибке для удобства чтения отчёта.

Важный нюанс: `assertAll` группирует именно **независимые** проверки. Если проверки логически зависят друг от друга (вторая проверка не имеет смысла без успеха первой), группировать их через `assertAll` не стоит — лучше оставить последовательные обычные assert'ы.

## Пример / когда применяется

```java
@Test
void userFieldsAreCorrect() {
    User user = userService.createUser("Ivan", "ivan@example.com", 25);

    Assertions.assertAll("Проверка полей пользователя",
        () -> Assertions.assertEquals("Ivan", user.getName()),
        () -> Assertions.assertEquals("ivan@example.com", user.getEmail()),
        () -> Assertions.assertEquals(25, user.getAge()),
        () -> Assertions.assertNotNull(user.getId())
    );
}
```

Если, например, поля `email` и `age` не совпадают с ожидаемыми, тест упадёт один раз, а в отчёте будет две ошибки сразу — по email и по age, — что экономит время на диагностику по сравнению с последовательными обычными `assertEquals`.

Применяется прежде всего при проверке объектов с множеством полей (DTO, response API-запроса), при валидации нескольких аспектов одного бизнес-сценария в одном логическом тесте.

## На что смотрит интервьюер

- Понимает ли кандидат ключевое отличие `assertAll` от последовательных assert'ов — выполнение всех проверок, даже если предыдущие упали.
- Знает ли про `MultipleFailuresError` и что именно в нём агрегируются все ошибки.
- Может ли аргументировать, когда группировка через `assertAll` уместна, а когда нет (независимые vs зависимые проверки).
- Частая ошибка — пытаться использовать `assertAll` для проверок, которые логически зависят одна от другой (например, сначала проверить, что объект не null, а потом обращаться к его полям, — если вынести это в одну группу `assertAll`, при null-объекте упадут все лямбды с NPE, а не с понятным сообщением).
- Follow-up вопрос — как это соотносится с "soft assertions" в AssertJ (`SoftAssertions`) — концептуально похожий механизм из другой библиотеки.
