# Как в TestNG настроить зависимости между тестами (`dependsOnMethods`)?

**Коротко:** зависимость задаётся атрибутом `dependsOnMethods` внутри `@Test`, куда передаётся имя метода (или нескольких методов), от успешного выполнения которых зависит текущий тест; если метод-зависимость упал или был пропущен, зависимый тест автоматически помечается как SKIPPED, а не выполняется и не падает сам по себе.

## Развёрнутый ответ

TestNG позволяет явно выразить порядок и логическую связь между тестами через два родственных атрибута:

- **`dependsOnMethods = {"methodName1", "methodName2"}`** — тест выполнится только после успешного завершения указанных методов. Если метод содержится в другом классе, имя указывается с полным путём к классу (`"com.example.tests.LoginTest.loginTest"`).
- **`dependsOnGroups = {"groupName"}`** — то же самое, но зависимость выражается через принадлежность к группе методов (`groups = {"groupName"}`), а не через конкретные имена методов — удобно, когда зависимость логическая ("после всех smoke-тестов"), а не привязана к конкретному методу.

Ключевое поведение при зависимостях:

- Если тест, от которого зависят другие, **упал**, все зависимые от него тесты автоматически получают статус **SKIPPED** (пропущен), а не запускаются и не проваливаются самостоятельно. Это отражается в отчёте отдельной категорией, отличной от FAILED.
- Есть атрибут `alwaysRun = true`, который заставляет зависимый тест выполниться, даже если тест, от которого он зависит, упал (полезно для тестов очистки/cleanup, которые должны запуститься в любом случае).
- TestNG сам строит граф зависимостей и определяет корректный порядок запуска; при циклической зависимости (A зависит от B, B зависит от A) будет выброшено исключение на этапе анализа.
- Зависимости работают только в пределах одного прогона (suite); нельзя зависеть от метода, который не будет запущен в текущей конфигурации `testng.xml`.

Как и с `priority`, важно понимать: активное использование зависимостей между тестами противоречит принципу независимости автотестов и увеличивает хрупкость набора (падение одного теста лавинообразно "скипает" десятки других). Оправданное применение — это сложные E2E-сценарии, где реально нет смысла продолжать (например, если создание тестового пользователя упало, нет смысла запускать тесты его редактирования).

## Пример / когда применяется

```java
public class UserLifecycleTest {

    @Test
    public void createUser() {
        // создаём пользователя
    }

    @Test(dependsOnMethods = "createUser")
    public void updateUser() {
        // обновляем данные пользователя, созданного в предыдущем тесте
    }

    @Test(dependsOnMethods = "updateUser", alwaysRun = true)
    public void deleteUser() {
        // удаляем пользователя в любом случае, даже если updateUser упал,
        // чтобы не оставлять "мусорные" данные в системе
    }
}
```

Пример зависимости от группы:

```java
@Test(groups = "smoke")
public void applicationIsUp() { /* ... */ }

@Test(dependsOnGroups = "smoke")
public void detailedRegressionTest() {
    // запустится только после успешного прохождения всех smoke-тестов
}
```

## На что смотрит интервьюер

- Знает ли кандидат, что при падении "родительского" теста зависимые тесты помечаются SKIPPED, а не FAILED и не выполняются вовсе.
- Понимает ли назначение `alwaysRun = true` для гарантированного выполнения cleanup-логики.
- Может ли объяснить разницу между `dependsOnMethods` (конкретные методы) и `dependsOnGroups` (логическая группа).
- Частая ошибка — не учитывать, что при переименовании зависимого метода тест сломается на этапе конфигурации (используется строковое имя метода, а не ссылка на него, что не проверяется компилятором).
- Красный флаг — злоупотребление зависимостями для построения длинных цепочек E2E-сценариев вместо изолированных тестов, что делает набор хрупким и сложным в диагностике (одна причина падения "роняет" множество тестов).
- Follow-up вопрос — как это соотносится с параллельным запуском: TestNG корректно учитывает зависимости при параллелизации, не запуская зависимый тест раньше завершения того, от которого он зависит.
