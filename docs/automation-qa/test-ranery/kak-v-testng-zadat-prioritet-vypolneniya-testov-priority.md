# Как в TestNG задать приоритет выполнения тестов (`priority`)?

**Коротко:** приоритет задаётся параметром `priority` внутри аннотации `@Test`, целым числом; тесты выполняются в порядке возрастания приоритета (сначала меньшее значение), а тесты с одинаковым или не заданным приоритетом (по умолчанию 0) выполняются в порядке, не гарантированном спецификацией (обычно — в порядке объявления в скомпилированном классе, но полагаться на это не стоит).

## Развёрнутый ответ

В TestNG аннотация `@Test` поддерживает атрибут `priority` (тип `int`), который задаёт порядок выполнения тестовых методов внутри одного класса (или в рамках suite, если методы разбросаны по нескольким классам, включённым в один `<test>`):

- По умолчанию у всех тестов `priority = 0`.
- Тесты с меньшим значением приоритета выполняются раньше тестов с большим значением.
- Приоритет может быть отрицательным (для запуска теста раньше "нулевых" по умолчанию).
- Если у нескольких методов одинаковый приоритет, порядок между ними TestNG не гарантирует явно (детали зависят от версии/реализации), поэтому опираться на порядок объявления в коде — плохая практика.
- Приоритет — не единственный способ управления порядком: есть также параметр `preserveOrder` в `testng.xml` (для сохранения порядка объявления методов/классов) и явные зависимости через `dependsOnMethods`/`dependsOnGroups`, которые гарантируют порядок сильнее, чем `priority`.

Важный концептуальный момент, который стоит подчеркнуть на собеседовании: полагаться на приоритет для организации тестов — признак антипаттерна в автоматизации. Хорошие автотесты должны быть независимыми друг от друга (принцип изоляции тестов), а порядок выполнения не должен влиять на корректность прогона. Приоритет уместен в узких случаях: например, smoke-тесты должны выполняться первыми, чтобы быстро проверить базовую работоспособность системы перед более длительными и глубокими тестами, либо в специфичных сценариях, где тесты сознательно строят пошаговый сценарий (что тоже лучше явно выражать через `dependsOnMethods`, а не только через приоритет).

## Пример / когда применяется

```java
public class SmokeAndRegressionTest {

    @Test(priority = 0)
    public void applicationIsUp() {
        // smoke-проверка: приложение доступно
    }

    @Test(priority = 1)
    public void loginPageIsAccessible() {
        // проверка доступности страницы логина
    }

    @Test(priority = 2)
    public void loginWithValidCredentials() {
        // более детальный сценарий
    }

    @Test // priority = 0 по умолчанию, порядок относительно других 0-приоритетных не гарантирован
    public void unrelatedTest() {
        // ...
    }
}
```

Применяется, когда важно, чтобы базовые/быстрые проверки (smoke) прошли раньше объёмных регрессионных сценариев — это позволяет быстро "провалить" прогон на раннем этапе (fail fast), не тратя время CI на дальнейшие тесты при недоступности приложения.

## На что смотрит интервьюер

- Знает ли кандидат, что меньшее число приоритета выполняется раньше (частая ошибка — путать местами).
- Понимает ли, что `priority` не гарантирует порядок среди тестов с одинаковым значением.
- Может ли аргументированно объяснить, что жёсткая зависимость от порядка через `priority` — плохая практика, и что для истинных зависимостей между шагами лучше использовать `dependsOnMethods`.
- Красный флаг — если кандидат предлагает выстраивать весь тестовый сценарий (создание пользователя → логин → удаление) только через `priority`, без учёта независимости и идемпотентности тестов.
- Follow-up вопрос — чем `priority` отличается от `dependsOnMethods` и что произойдёт с зависимым тестом, если тест, от которого он зависит, упадёт (он будет пропущен со статусом SKIP, а не FAIL).
