# Назовите основные аннотации JUnit 5 (`@Test`, `@BeforeEach`, `@AfterAll`, `@DisplayName`, `@Disabled`).

**Коротко:** JUnit 5 использует набор аннотаций для пометки тестовых методов, хуков жизненного цикла и метаданных теста; ключевые из них — `@Test`, `@BeforeEach`/`@AfterEach`, `@BeforeAll`/`@AfterAll`, `@DisplayName`, `@Disabled`, `@Tag`, `@Nested`.

## Развёрнутый ответ

Основные аннотации JUnit 5 (модуль `org.junit.jupiter.api`):

- **`@Test`** — помечает метод как тестовый; раннер обнаружит и выполнит его.
- **`@BeforeEach`** — метод выполняется перед каждым тестовым методом в классе (аналог `setUp`). Полезен для подготовки состояния (открытие драйвера, создание объектов).
- **`@AfterEach`** — метод выполняется после каждого тестового метода (аналог `tearDown`), даже если тест упал с исключением. Используется для очистки ресурсов.
- **`@BeforeAll`** — метод выполняется один раз перед всеми тестами класса. Должен быть `static` (если не используется `@TestInstance(Lifecycle.PER_CLASS)`). Подходит для дорогих операций (поднятие контейнера, подключение к БД).
- **`@AfterAll`** — метод выполняется один раз после всех тестов класса, также должен быть `static` по умолчанию. Используется для финальной очистки общих ресурсов.
- **`@DisplayName("...")`** — задаёт человекочитаемое имя теста или класса для отчётов, вместо технического имени метода.
- **`@Disabled("причина")`** — отключает тест или класс от выполнения, с указанием причины (полезно при временно нестабильных или недоработанных тестах).
- **`@Tag("smoke")`** — присваивает тегу/категории для выборочного запуска (например, только smoke-тесты).
- **`@Nested`** — позволяет создавать вложенные тестовые классы для группировки логически связанных тестов с общим контекстом.
- **`@RepeatedTest(n)`** — запускает тест n раз подряд (для проверки стабильности/флакующих тестов).
- **`@ParameterizedTest`** — помечает параметризованный тест (используется вместе с `@ValueSource`, `@CsvSource` и т.д.).
- **`@ExtendWith(...)`** — подключает расширения JUnit 5 (например, Mockito, Spring, кастомные extension-классы).

Важный нюанс по умолчанию жизненного цикла: JUnit 5 создаёт новый экземпляр тестового класса для каждого тестового метода (`Lifecycle.PER_METHOD`), поэтому `@BeforeAll`/`@AfterAll` обязаны быть статическими. Если явно указать `@TestInstance(TestInstance.Lifecycle.PER_CLASS)`, экземпляр класса будет один на весь класс, и тогда `@BeforeAll`/`@AfterAll` можно сделать нестатическими.

## Пример / когда применяется

```java
import org.junit.jupiter.api.*;

@DisplayName("Тесты калькулятора")
class CalculatorTest {

    @BeforeAll
    static void initAll() {
        // разовая инициализация, например запуск тестовой БД
    }

    @BeforeEach
    void init() {
        // подготовка перед каждым тестом
    }

    @Test
    @DisplayName("2 + 2 должно быть равно 4")
    void additionWorks() {
        Assertions.assertEquals(4, 2 + 2);
    }

    @Test
    @Disabled("Функционал ещё не реализован, JIRA-123")
    void divisionNotReadyYet() {
        // временно отключено
    }

    @AfterEach
    void tearDown() {
        // очистка после каждого теста
    }

    @AfterAll
    static void tearDownAll() {
        // финальная очистка, например остановка тестовой БД
    }
}
```

## На что смотрит интервьюер

- Знает ли кандидат разницу между `@BeforeEach`/`@AfterEach` и `@BeforeAll`/`@AfterAll` (частота вызова, требование `static`).
- Понимает ли, зачем нужен `@Disabled` с указанием причины, а не просто закомментированный тест.
- Знает ли про `Lifecycle.PER_CLASS` как способ избежать статических методов.
- Частая ошибка — путать JUnit 4 аннотации (`@Before`, `@After`, `@Ignore`) с JUnit 5 (`@BeforeEach`, `@AfterEach`, `@Disabled`) — это классический follow-up на знание миграции JUnit 4 → 5.
- Красный флаг — незнание, что тест-метод по умолчанию должен быть package-private или public без `static` модификатора.
