# Сравните JUnit 5 и TestNG. В каких случаях какой фреймворк предпочтительнее?

**Коротко:** JUnit 5 — более современный, модульный фреймворк с богатым Extension API, стандарт де-факто для юнит-тестирования и экосистемы Spring; TestNG — более "батарейки включены" решение с встроенными группами, приоритетами, зависимостями, гибким `@DataProvider` и нативным параллелизмом, традиционно популярное в UI/API-автоматизации (Selenium, RestAssured).

## Развёрнутый ответ

**Архитектура и происхождение.** JUnit 5 состоит из трёх модулей: Platform (движок запуска, на котором могут работать разные фреймворки), Jupiter (новый API с аннотациями `@Test` и т.д.) и Vintage (для обратной совместимости с JUnit 3/4). Такая модульность даёт мощный Extension API для написания собственных расширений. TestNG исторически создавался как ответ на ограничения JUnit (в частности, старого JUnit 3/4) и изначально включал функциональность, которую в JUnit пришлось добавлять через сторонние библиотеки или новые версии.

**Параметризация.** JUnit 5 использует `@ParameterizedTest` с разными источниками (`@ValueSource`, `@CsvSource`, `@MethodSource` и т.д.), требуя отдельную зависимость `junit-jupiter-params`. TestNG использует единый механизм `@DataProvider`, изначально встроенный, гибкий для передачи сложных объектов, с поддержкой параллельного выполнения наборов данных.

**Группировка и зависимости.** TestNG имеет встроенную поддержку групп тестов (`groups = {"smoke", "regression"}`), приоритетов (`priority`) и явных зависимостей (`dependsOnMethods`, `dependsOnGroups`) прямо в аннотациях. В JUnit 5 для похожей группировки используется `@Tag` (аналог групп), но встроенных приоритетов и зависимостей между тестами нет — порядок тестов в JUnit по умолчанию не гарантирован и не рекомендуется на него полагаться (при необходимости используется `@TestMethodOrder` с кастомным компаратором, но это считается редким случаем, а не стандартной практикой).

**Иерархия хуков.** У TestNG она шире: method → class → test (из XML) → suite (`@BeforeSuite`/`@AfterSuite`), плюс уровень групп (`@BeforeGroups`/`@AfterGroups`). У JUnit 5: method → class (`@BeforeEach/@AfterEach`, `@BeforeAll/@AfterAll`), без встроенного XML-конфигурируемого уровня suite (хотя есть JUnit Platform Suite API для описания suite'ов в коде через аннотации `@Suite`).

**Конфигурация запуска.** TestNG традиционно конфигурируется через `testng.xml` — декларативное описание suite, групп, параллелизма, параметров окружения. JUnit 5 конфигурация обычно ближе к коду и настройкам системы сборки (Maven/Gradle), без отдельного XML-формата для описания suite (хотя это можно считать как плюсом — меньше "магии" вне кода, — так и минусом — меньше гибкости декларативной конфигурации без перекомпиляции).

**Retry и параллелизм.** У TestNG есть встроенный `IRetryAnalyzer` и родная поддержка параллельного запуска на уровне методов/классов/suite через `testng.xml`. JUnit 5 поддерживает параллельный запуск через конфигурацию (`junit.jupiter.execution.parallel.enabled=true` в `junit-platform.properties`), но retry требует сторонних расширений.

**Экосистема и интеграции.** JUnit 5 — стандарт для Spring Boot, Spring Test, большинства современных Java-библиотек и IDE "из коробки"; лучше интегрируется с mock-фреймворками через Extension API (`@ExtendWith(MockitoExtension.class)`). TestNG исторически прочно закрепился в сфере UI/API автоматизации тестирования (Selenium WebDriver, RestAssured проекты), благодаря встроенным группам, зависимостям и гибкому DataProvider, которые удобны именно для организации больших наборов end-to-end тестов.

**Когда предпочтительнее что:**

- **JUnit 5** — для юнит- и интеграционного тестирования кода приложения, особенно в экосистеме Spring/Spring Boot; когда важна тесная интеграция с современными IDE, инструментами покрытия кода, mock-библиотеками; когда нужен богатый Extension API для кастомизации.
- **TestNG** — для крупных наборов автотестов UI/API, где важны встроенные группы (smoke/regression/e2e), явные зависимости между шагами сценария, гибкая параметризация сложных объектов и декларативная XML-конфигурация параллельного запуска без дополнительных плагинов.

На практике выбор часто определяется не столько техническими ограничениями (оба фреймворка достаточно зрелые и покрывают 90% типовых задач), сколько историческими договорённостями в команде, интеграцией с уже существующим стеком (Spring — почти всегда JUnit 5) и личным опытом лида/команды.

## Пример / когда применяется

| Критерий | JUnit 5 | TestNG |
|---|---|---|
| Группы тестов | `@Tag` | `groups` в `@Test` |
| Приоритет | нет встроенного | `priority` |
| Зависимости между тестами | нет встроенных | `dependsOnMethods`, `dependsOnGroups` |
| Параметризация | `@ParameterizedTest` + источники | `@DataProvider` |
| Retry | сторонние расширения | `IRetryAnalyzer` встроенный |
| Конфигурация suite | код / JUnit Platform Suite API | `testng.xml` |
| Экосистема | Spring, современные Java-проекты | UI/API автоматизация (Selenium, RestAssured) |

## На что смотрит интервьюер

- Может ли кандидат сравнить фреймворки не абстрактно ("оба хорошие"), а по конкретным техническим отличиям (группы, зависимости, приоритеты, retry).
- Понимает ли, что выбор часто продиктован экосистемой проекта (Spring Boot почти всегда тянет за собой JUnit 5), а не только "личными предпочтениями".
- Знает ли сильные стороны TestNG именно в контексте автоматизации UI/API (зависимости, группы, DataProvider), а не только в теории.
- Частая ошибка — утверждение, что один фреймворк однозначно "лучше" другого без контекста задачи.
- Follow-up вопрос — какой фреймворк использовался на предыдущем проекте и почему был выбран именно он; как бы кандидат мигрировал большой набор тестов с TestNG на JUnit 5 (или наоборот) и какие возможности пришлось бы компенсировать сторонними библиотеками.
