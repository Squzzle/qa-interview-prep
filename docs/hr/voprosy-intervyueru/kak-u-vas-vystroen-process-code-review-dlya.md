# Как у вас выстроен процесс code review для автотестов?

**Коротко:** Это вопрос, который стоит задать интервьюеру, чтобы понять зрелость инженерной культуры команды и то, насколько серьёзно относятся к качеству кода автотестов, а не только к качеству продукта.

## Развёрнутый ответ

Этот вопрос кандидат задаёт сам, чтобы выяснить реальные практики команды. Он показывает интервьюеру, что кандидат воспринимает автотесты как полноценный код, который требует таких же практик разработки, как и продуктовый код: ревью, стандарты, рефакторинг.

Хороший ответ интервьюера обычно включает несколько составляющих:

- Кто проводит ревью — только тимлид/QA-лид, или ревью делает вся команда, включая разработчиков.
- Есть ли обязательные чек-листы или гайдлайны (naming conventions, паттерны Page Object / Screenplay, работа с локаторами, обработка ожиданий и флаки-тестов).
- Используется ли pull request flow (GitHub/GitLab/Bitbucket) с обязательным approve перед мержем.
- Есть ли автоматические проверки перед ревью: линтеры, статический анализ, запуск тестов в CI как gate для PR.
- Как быстро в среднем проходит ревью и сколько итераций обычно требуется.

Если интервьюер отвечает расплывчато («ну, кто-то посмотрит иногда») — это красный флаг, говорящий об отсутствии culture качества и возможных проблемах с поддерживаемостью тестового фреймворка в будущем.

## Пример / когда применяется

Вопрос уместно задать на финальном этапе интервью, когда даётся слово кандидату («у вас есть вопросы к нам?»). Особенно полезен, если кандидат претендует на позицию с ростом до QA Lead/SDET, так как процесс код-ревью — прямая зона ответственности такой роли.

Пример уточняющего диалога:

```text
Кандидат: Как у вас устроено код-ревью для автотестов?
Интервьюер: Мы используем GitLab MR, обязательны два approve — от QA и от одного разработчика,
плюс CI прогоняет линтер и smoke-набор перед мержем.
Кандидат: А есть ли гайдлайн по паттернам написания тестов (например, Page Object)?
Интервьюер: Да, есть внутренний wiki с конвенциями и примерами.
```

## На что смотрит интервьюер

Интервьюер (в данном случае сам кандидат, задающий вопрос) должен показать, что понимает: автотесты — это код, требующий тех же практик, что и продакшн-код. Частая ошибка — не задавать подобные вопросы вовсе, из-за чего складывается впечатление пассивности и отсутствия интереса к качеству процессов. Follow-up, который может возникнуть в ответ, — уточнение про метрики качества тестов (flaky rate, время прогона, покрытие). Красный флаг для самого кандидата — если он не может объяснить, зачем вообще важен код-ревью для тестов.
