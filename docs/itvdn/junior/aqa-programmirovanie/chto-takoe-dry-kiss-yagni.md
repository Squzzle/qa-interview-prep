# Что такое DRY, KISS, YAGNI?

**Коротко:** это три взаимодополняющих принципа хорошего дизайна кода: не дублировать логику (DRY), делать решения максимально простыми (KISS) и не реализовывать функциональность, которая пока не нужна (YAGNI).

## Развёрнутый ответ

**DRY — Don't Repeat Yourself.** Каждая часть знания (бизнес-логика, правило, алгоритм) должна иметь единственное, однозначное представление в системе. Если один и тот же кусок логики скопирован в нескольких местах, при изменении требований придётся вносить правки во всех копиях, а если забыть одну — появится баг. Решение — выносить повторяющийся код в отдельные методы, классы, константы или утилитные функции.

**KISS — Keep It Simple, Stupid.** Решение должно быть настолько простым, насколько это возможно для решения задачи, без избыточной сложности, лишней абстракции или «умного» кода ради самого ума. Простой код проще читать, тестировать, поддерживать и передавать другим разработчикам. Излишняя инженерия (over-engineering) — типичное нарушение KISS.

**YAGNI — You Aren't Gonna Need It.** Не стоит реализовывать функциональность «про запас», основываясь на предположении, что она понадобится в будущем. Реализация невостребованных возможностей увеличивает сложность кодовой базы, время разработки и площадь для багов, при этом часто такая функциональность так и остаётся невостребованной или требования к ней меняются к моменту, когда она реально понадобится.

Все три принципа работают вместе: DRY уменьшает дублирование, KISS упрощает конкретные решения, а YAGNI ограничивает объём того, что вообще нужно реализовывать.

## Пример / когда применяется

Нарушение DRY в автотестах и исправление через вынесение общего кода:

```java
// Нарушение DRY: одинаковая логика логина скопирована в каждом тесте
@Test
void testCase1() {
    driver.findElement(By.id("login")).sendKeys("user1");
    driver.findElement(By.id("password")).sendKeys("pass1");
    driver.findElement(By.id("submit")).click();
    // ... шаги теста
}

@Test
void testCase2() {
    driver.findElement(By.id("login")).sendKeys("user2");
    driver.findElement(By.id("password")).sendKeys("pass2");
    driver.findElement(By.id("submit")).click();
    // ... шаги теста
}

// Исправление: вынесли общий шаг в метод (DRY) и использовали простой Page Object (KISS)
void login(String user, String password) {
    driver.findElement(By.id("login")).sendKeys(user);
    driver.findElement(By.id("password")).sendKeys(password);
    driver.findElement(By.id("submit")).click();
}
```

Пример нарушения YAGNI: тестовый фреймворк с самого начала проектируют с поддержкой пяти разных браузерных драйверов, кастомной системой плагинов и конфигурируемым отчётом на трёх языках, хотя команда использует только Chrome и один язык отчётов — это лишняя сложность, которую стоило бы добавить только тогда, когда появится реальная потребность.

## На что смотрит интервьюер

- Понимание, что принципы не абсолютны и могут конфликтовать — например, чрезмерное следование DRY (вынесение всего в общие абстракции) иногда нарушает KISS, создавая излишне сложные и связанные (coupled) абстракции.
- Умение привести пример из собственного опыта написания автотестов, а не только теоретическое определение.
- Follow-up: «Приведите пример, когда DRY может навредить» (преждевременное обобщение двух похожих, но концептуально разных кусков кода в одну абстракцию), «Как YAGNI соотносится с гибкой разработкой (Agile)?» (Agile поощряет итеративную разработку только нужного на текущий момент функционала).
- Красный флаг — кандидат воспринимает эти принципы как жёсткие правила без исключений, не понимая контекст их применения.
