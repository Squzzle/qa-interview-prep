# Что такое интерфейс? Что такое абстрактный класс? Чем они отличаются?

**Коротко:** интерфейс описывает контракт (набор методов, которые обязан реализовать класс), а абстрактный класс — это класс, который может содержать как абстрактные, так и уже реализованные методы, а также поля с состоянием. Класс может реализовать много интерфейсов, но наследовать только один абстрактный класс.

## Развёрнутый ответ

**Интерфейс** (`interface`) — это тип, который определяет набор методов без реализации (до Java 8) — по сути, контракт: «любой класс, реализующий этот интерфейс, обязан предоставить реализацию перечисленных методов». Начиная с Java 8 интерфейсы могут содержать `default` и `static` методы с телом, а с Java 9 — `private` методы для внутреннего переиспользования кода внутри интерфейса. Все поля в интерфейсе неявно `public static final` (константы), у интерфейса нет собственного состояния (полей экземпляра) и нет конструкторов. Класс может реализовывать (`implements`) сразу несколько интерфейсов — это способ сымитировать множественное наследование поведения.

**Абстрактный класс** (`abstract class`) — это класс, который нельзя инстанцировать напрямую (нельзя написать `new AbstractClass()`), но который может содержать:
- абстрактные методы (без реализации, обязательные к переопределению в наследниках);
- обычные методы с полной реализацией;
- поля любого уровня доступа и любое состояние;
- конструкторы (вызываются через `super()` из наследника).

Класс может наследовать только один абстрактный (или любой другой) класс — в Java нет множественного наследования классов, только одиночное.

Ключевые отличия:

| Критерий | Интерфейс | Абстрактный класс |
|---|---|---|
| Наследование | Множественное (`implements` нескольких) | Одиночное (`extends` одного) |
| Состояние (поля) | Только константы (`public static final`) | Любые поля экземпляра |
| Реализация методов | `default`/`static`/`private` методы с телом (Java 8+) | Любые методы с телом |
| Конструктор | Нет | Есть |
| Модификаторы доступа методов | `public` (по умолчанию) | Любые (`private`, `protected`, `public`) |
| Назначение | Описать «что умеет делать» объект (контракт) | Описать «чем является» объект + дать общую реализацию |

## Пример / когда применяется

```java
interface Flyable {
    void fly(); // абстрактный метод контракта

    default void land() { // метод с реализацией по умолчанию (Java 8+)
        System.out.println("Приземление стандартным способом");
    }
}

abstract class Bird {
    protected String name;

    public Bird(String name) {
        this.name = name;
    }

    public abstract String makeSound(); // должно быть реализовано наследником

    public void eat() { // общая реализация для всех птиц
        System.out.println(name + " клюёт зерно");
    }
}

class Eagle extends Bird implements Flyable {
    public Eagle(String name) {
        super(name);
    }

    @Override
    public String makeSound() {
        return "Клёкот";
    }

    @Override
    public void fly() {
        System.out.println(name + " летит высоко");
    }
}
```

Интерфейсы используются, когда нужно описать поведение, общее для несвязанных иерархий классов (например, `Comparable`, `Runnable`, `Serializable`). Абстрактные классы — когда есть общая логика и состояние, которые нужно переиспользовать (например, базовый класс `BaseTest` в тестовом фреймворке с общей настройкой `@BeforeEach`, наследуемый всеми тест-классами).

## На что смотрит интервьюер

- Знание, что с Java 8 интерфейсы получили `default`/`static` методы, а с Java 9 — `private` методы, и понимание, зачем это добавили (эволюция API без поломки обратной совместимости).
- Умение объяснить, почему в Java нет множественного наследования классов (проблема ромбовидного наследования — diamond problem) и как интерфейсы её решают.
- Частая ошибка — утверждение, что в интерфейсах вообще не может быть реализации методов (устаревшее для Java 8+).
- Follow-up: «Может ли абстрактный класс не иметь ни одного абстрактного метода?» (да, может), «Может ли интерфейс наследовать другой интерфейс?» (да, через `extends`, причём может наследовать несколько интерфейсов).
