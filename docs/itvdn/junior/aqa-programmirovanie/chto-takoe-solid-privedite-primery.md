# Что такое SOLID? Приведите примеры.

**Коротко:** SOLID — пять принципов объектно-ориентированного проектирования, которые делают код более гибким, поддерживаемым и устойчивым к изменениям: Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion.

## Развёрнутый ответ

**S — Single Responsibility Principle (принцип единственной ответственности).** Класс должен иметь только одну причину для изменения, то есть решать только одну задачу. Если класс одновременно читает данные, обрабатывает бизнес-логику и формирует отчёт — это нарушение принципа: изменения в одной из областей потребуют трогать один и тот же класс по разным причинам.

**O — Open/Closed Principle (принцип открытости/закрытости).** Классы должны быть открыты для расширения, но закрыты для модификации. Новую функциональность стоит добавлять через наследование или композицию (новые классы, реализующие общий интерфейс), а не через изменение уже работающего и протестированного кода.

**L — Liskov Substitution Principle (принцип подстановки Барбары Лисков).** Объекты дочернего класса должны быть взаимозаменяемы с объектами родительского класса без нарушения корректности программы. Если наследник меняет ожидаемое поведение родителя (например, выбрасывает исключение там, где родитель этого не делал, или сужает контракт метода), это нарушает LSP.

**I — Interface Segregation Principle (принцип разделения интерфейсов).** Клиенты не должны зависеть от методов, которые они не используют. Лучше несколько маленьких специализированных интерфейсов, чем один большой «толстый» интерфейс, вынуждающий классы реализовывать ненужные методы.

**D — Dependency Inversion Principle (принцип инверсии зависимостей).** Модули верхнего уровня не должны зависеть от модулей нижнего уровня — оба должны зависеть от абстракций. Конкретные реализации внедряются извне (например, через конструктор — Dependency Injection), а не создаются жёстко внутри класса через `new`.

## Пример / когда применяется

Нарушение LSP и Interface Segregation, а затем исправление:

```java
// Нарушение LSP: Ostrich не умеет летать, но обязан реализовать fly()
interface Bird {
    void fly();
    void eat();
}

class Ostrich implements Bird {
    @Override
    public void fly() {
        throw new UnsupportedOperationException("Страусы не летают");
    }

    @Override
    public void eat() { /* ... */ }
}

// Исправление через Interface Segregation
interface Eater {
    void eat();
}

interface Flyer {
    void fly();
}

class Sparrow implements Eater, Flyer {
    public void eat() { /* ... */ }
    public void fly() { /* ... */ }
}

class Ostrich2 implements Eater { // не реализует Flyer — контракт честный
    public void eat() { /* ... */ }
}
```

Пример Dependency Inversion в контексте автотестов — внедрение драйвера через конструктор вместо жёсткого создания внутри класса:

```java
interface Browser {
    void open(String url);
}

class ChromeBrowser implements Browser {
    public void open(String url) { /* Selenium-логика */ }
}

class LoginTest {
    private final Browser browser;

    // зависимость внедряется извне, а не создаётся через new ChromeBrowser()
    public LoginTest(Browser browser) {
        this.browser = browser;
    }

    public void run() {
        browser.open("https://example.com/login");
    }
}
```

## На что смотрит интервьюер

- Способность привести собственный пример нарушения принципа и показать, как его исправить, а не просто зачитать определения.
- Частая ошибка — путать Open/Closed с простым «не трогай старый код никогда»; на самом деле принцип про расширение через абстракции.
- Follow-up: «Как SOLID связан с паттернами проектирования?» (многие паттерны — Strategy, Factory, Decorator — прямое следствие применения SOLID), «Приведите пример нарушения LSP из реальной практики автотестов» (например, метод `PageObject.click()` в наследнике выбрасывает исключение вместо клика).
- Красный флаг — кандидат знает только расшифровку аббревиатуры, но не может объяснить практическую пользу хотя бы одного принципа.
