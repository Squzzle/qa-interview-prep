# Какая разница между String, Stringbuffer и Stringbuilder?

**Коротко:** `String` — неизменяемый (immutable) объект, любое «изменение» создаёт новую строку в памяти; `StringBuilder` и `StringBuffer` — изменяемые (mutable) классы для эффективной работы со строками, отличающиеся друг от друга только тем, что `StringBuffer` синхронизирован (потокобезопасен), а `StringBuilder` — нет и потому быстрее.

## Развёрнутый ответ

**`String`** в Java — immutable-класс: после создания объекта его внутреннее содержимое изменить нельзя. Любая операция, которая выглядит как изменение строки (`concat()`, `+`, `replace()`), на самом деле создаёт новый объект `String` в памяти, оставляя исходный без изменений. Это приводит к избыточному созданию объектов при многократной конкатенации в цикле, что негативно сказывается на производительности и нагрузке на сборщик мусора (`GC`). Неизменяемость даёт и плюсы: строки безопасны для использования в многопоточной среде без синхронизации и могут безопасно кэшироваться в **String Pool** — специальной области памяти, где одинаковые строковые литералы переиспользуются.

**`StringBuilder`** — изменяемый (mutable) класс, представляющий собой обёртку над массивом символов, который может динамически расширяться. Методы вроде `append()`, `insert()`, `delete()`, `reverse()` изменяют внутреннее состояние того же объекта, не создавая новые объекты при каждой операции. Не синхронизирован — не потокобезопасен, но за счёт этого быстрее `StringBuffer` в однопоточном сценарии. Появился в Java 5 как замена `StringBuffer` там, где многопоточность не требуется.

**`StringBuffer`** — функционально идентичен `StringBuilder` (тот же набор методов), но все его методы синхронизированы (`synchronized`), что делает его потокобезопасным при работе из нескольких потоков одновременно — ценой некоторого снижения производительности из-за накладных расходов на синхронизацию.

Сравнительная таблица:

| Критерий | `String` | `StringBuilder` | `StringBuffer` |
|---|---|---|---|
| Изменяемость | Immutable | Mutable | Mutable |
| Потокобезопасность | Да (за счёт immutability) | Нет | Да (synchronized) |
| Производительность при частых изменениях | Низкая (создание новых объектов) | Высокая | Средняя (накладные расходы синхронизации) |
| Год появления | С самого начала Java | Java 5 | С самого начала Java |
| Когда использовать | Строка не меняется или меняется редко | Частые изменения в одном потоке | Частые изменения из нескольких потоков |

## Пример / когда применяется

```java
// Неэффективно: на каждой итерации создаётся новый объект String
String result = "";
for (int i = 0; i < 1000; i++) {
    result += i; // создаёт 1000 промежуточных объектов String
}

// Эффективно: один изменяемый буфер
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i);
}
String result2 = sb.toString();
```

```java
// Потокобезопасный вариант при работе из нескольких потоков
StringBuffer sharedBuffer = new StringBuffer();

Runnable appendTask = () -> sharedBuffer.append("data ");
Thread t1 = new Thread(appendTask);
Thread t2 = new Thread(appendTask);
t1.start();
t2.start();
```

В автотестах `StringBuilder` часто применяется при формировании динамических локаторов (XPath/CSS) или тела запросов из нескольких частей внутри одного потока выполнения теста.

## На что смотрит интервьюер

- Понимание, почему `String` immutable и какие плюсы это даёт (безопасность, String Pool, хэш-кэширование для использования в качестве ключей `HashMap`).
- Знание, что разница между `StringBuilder` и `StringBuffer` — исключительно в синхронизации, а не в наборе методов.
- Follow-up: «Что такое String Pool и как он работает?», «Почему `String` — хороший кандидат для ключа `HashMap`?» (иммутабельность гарантирует неизменность хэш-кода), «Когда `+` для строк компилируется в `StringBuilder` автоматически?» (компилятор Java при конкатенации в одном выражении сам использует `StringBuilder`, но не в цикле с переприсваиванием).
- Красный флаг — кандидат утверждает, что `StringBuilder` быстрее `StringBuffer` из-за «другого алгоритма», а не понимает истинную причину (отсутствие синхронизации).
