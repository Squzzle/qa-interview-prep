# Какая разница между Thread class и Runnable interface?

**Коротко:** `Thread` — конкретный класс, представляющий поток выполнения, а `Runnable` — функциональный интерфейс, описывающий задачу для выполнения в потоке; наследование от `Thread` занимает единственный слот наследования и жёстко связывает задачу с потоком, тогда как реализация `Runnable` — более гибкий подход, разделяющий задачу и механизм её выполнения.

## Развёрнутый ответ

Есть два способа создать и запустить поток в Java:

**1. Наследование от класса `Thread`** — создаётся подкласс, переопределяющий метод `run()`, и его экземпляр напрямую является потоком:

```java
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Поток выполняется: " + Thread.currentThread().getName());
    }
}
```

Недостаток: поскольку в Java нет множественного наследования классов, если класс уже наследует что-то другое, он не сможет одновременно наследовать `Thread`.

**2. Реализация интерфейса `Runnable`** — создаётся класс, реализующий `Runnable` и его единственный метод `run()`, а затем объект передаётся в конструктор `Thread`:

```java
class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("Задача выполняется: " + Thread.currentThread().getName());
    }
}

Thread thread = new Thread(new MyTask());
thread.start();
```

Ключевые различия:

| Критерий | `extends Thread` | `implements Runnable` |
|---|---|---|
| Наследование | Занимает единственный слот наследования класса | Не занимает — можно наследовать другой класс |
| Разделение ответственности | Задача и механизм потока смешаны в одном классе | Задача (`Runnable`) отделена от потока (`Thread`) |
| Повторное использование | Каждый новый поток требует нового объекта `Thread`-наследника | Один и тот же `Runnable` можно передать в разные потоки или в `ExecutorService` |
| Совместимость с пулами потоков | Плохо сочетается с `ExecutorService` | Хорошо подходит для `ExecutorService.submit(Runnable)` |
| Рекомендация | Использовать редко, только если нужно переопределить поведение самого `Thread` | Предпочтительный подход в большинстве случаев |

Важный нюанс: запуск потока должен выполняться через метод `start()`, а не через прямой вызов `run()`. Вызов `run()` напрямую выполнит код синхронно в текущем потоке, не создавая новый поток выполнения — это одна из самых частых ошибок джуниоров.

## Пример / когда применяется

Современный подход — использование `Runnable` вместе с `ExecutorService`, что позволяет управлять пулом потоков, не заботясь о ручном создании и жизненном цикле каждого `Thread`:

```java
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class Main {
    public static void main(String[] args) {
        ExecutorService executor = Executors.newFixedThreadPool(2);
        Runnable task = () -> System.out.println("Задача в пуле потоков: " + Thread.currentThread().getName());

        executor.submit(task);
        executor.submit(task);
        executor.shutdown();
    }
}
```

В контексте автотестов `Runnable`/`ExecutorService` применяется для параллельного запуска нескольких сценариев проверки (например, параллельная проверка нескольких API-эндпоинтов) без создания отдельного класса-наследника `Thread` под каждую задачу.

## На что смотрит интервьюер

- Понимание, что `Runnable` — предпочтительный подход в реальных проектах, а `extends Thread` используется редко.
- Знание разницы между вызовом `start()` и `run()` — частый практический вопрос-ловушка.
- Follow-up: «Что вернёт `Callable` в отличие от `Runnable`?» (`Callable` может вернуть результат и выбросить checked-исключение, `run()` в `Runnable` — `void` и не выбрасывает checked-исключений), «Как безопасно остановить поток?» (через флаг-переменную или `interrupt()`, а не через устаревший `Thread.stop()`).
- Красный флаг — кандидат не знает, что вызов `run()` не создаёт новый поток, или не может объяснить, зачем вообще используется `Runnable`, если можно наследовать `Thread`.
