# Какие паттерны GOF вам известны? Приведите примеры их использования.

**Коротко:** паттерны «Банды четырёх» (Gang of Four) — 23 классических решения типовых задач проектирования, разделённые на три группы: порождающие (создание объектов), структурные (композиция классов) и поведенческие (взаимодействие объектов).

## Развёрнутый ответ

Паттерны GOF описаны в книге «Design Patterns: Elements of Reusable Object-Oriented Software» (1994) четырёх авторов и делятся на три категории:

**Порождающие паттерны (creational)** — отвечают за гибкое и контролируемое создание объектов:
- **Singleton** — гарантирует, что у класса есть только один экземпляр, и предоставляет глобальную точку доступа к нему.
- **Factory Method** — определяет интерфейс для создания объекта, но позволяет подклассам решать, объект какого класса создавать.
- **Abstract Factory** — предоставляет интерфейс для создания семейств связанных объектов без указания их конкретных классов.
- **Builder** — отделяет пошаговое конструирование сложного объекта от его представления.

**Структурные паттерны (structural)** — отвечают за композицию классов и объектов в более крупные структуры:
- **Adapter** — преобразует интерфейс одного класса в интерфейс, ожидаемый клиентом.
- **Decorator** — динамически добавляет объекту новую функциональность без изменения его класса.
- **Facade** — предоставляет упрощённый унифицированный интерфейс к сложной подсистеме.
- **Proxy** — предоставляет объект-заместитель, контролирующий доступ к другому объекту.

**Поведенческие паттерны (behavioral)** — отвечают за взаимодействие и распределение ответственности между объектами:
- **Strategy** — определяет семейство взаимозаменяемых алгоритмов и позволяет выбирать нужный в рантайме.
- **Observer** — определяет зависимость «один-ко-многим», при которой изменение состояния одного объекта уведомляет всех зависимых.
- **Template Method** — определяет скелет алгоритма в методе базового класса, позволяя подклассам переопределять отдельные шаги.
- **Command** — инкапсулирует запрос в виде объекта, позволяя параметризовать методы, ставить запросы в очередь и логировать их.

## Пример / когда применяется

**Singleton** — часто используется для конфигурации тестового окружения или единственного экземпляра `WebDriver`:

```java
public class DriverManager {
    private static WebDriver instance;

    private DriverManager() {}

    public static WebDriver getDriver() {
        if (instance == null) {
            instance = new ChromeDriver();
        }
        return instance;
    }
}
```

**Builder** — активно применяется для построения тестовых данных:

```java
public class UserBuilder {
    private String name;
    private String email;

    public UserBuilder withName(String name) {
        this.name = name;
        return this;
    }

    public UserBuilder withEmail(String email) {
        this.email = email;
        return this;
    }

    public User build() {
        return new User(name, email);
    }
}

// использование
User user = new UserBuilder().withName("Иван").withEmail("ivan@test.com").build();
```

**Template Method** — основа для Page Object паттерна в автотестах: базовый класс `BasePage` определяет общий алгоритм ожидания загрузки страницы, а конкретные страницы переопределяют шаг проверки уникального элемента.

**Strategy** — выбор стратегии сравнения ожидаемого и фактического результата в зависимости от типа проверки (точное совпадение, содержит подстроку, регулярное выражение), инкапсулированной в отдельные классы с общим интерфейсом.

## На что смотрит интервьюер

- Умение не просто назвать паттерны по памяти, а объяснить, какую проблему конкретный паттерн решает, и привести пример из практики автоматизации тестирования.
- Частая ошибка — путать Factory Method и Abstract Factory, либо называть Singleton «всегда плохой практикой» без понимания компромиссов (глобальное состояние усложняет unit-тестирование).
- Follow-up: «Какие паттерны вы использовали в своём тестовом фреймворке?», «Чем Adapter отличается от Facade?» (Adapter меняет интерфейс под ожидания клиента, Facade упрощает доступ к сложной подсистеме, не обязательно меняя интерфейсы).
- Красный флаг — кандидат перечисляет паттерны без понимания категорий (порождающие/структурные/поведенческие) и не может связать их с реальными задачами в тестировании.
