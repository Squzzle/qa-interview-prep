# Разница между final, finally и finalize?

**Коротко:** `final` — модификатор, запрещающий изменение (класса, метода или переменной); `finally` — блок в конструкции `try-catch`, который выполняется всегда, независимо от исключения; `finalize()` — устаревший метод объекта, который вызывался сборщиком мусора перед уничтожением объекта.

## Развёрнутый ответ

**`final`** — ключевое слово-модификатор, применяемое к трём разным сущностям с разным эффектом:
- `final` **переменная** — значение можно присвоить только один раз; для примитивов это делает значение константой, для ссылочных типов — запрещает переприсвоение ссылки (но не запрещает изменение состояния самого объекта, на который ссылка указывает).
- `final` **метод** — метод нельзя переопределить (`override`) в подклассах.
- `final` **класс** — класс нельзя унаследовать (`extends`); пример из стандартной библиотеки — класс `String` объявлен `final`.

**`finally`** — блок кода в конструкции `try-catch-finally`, который выполняется всегда после завершения блока `try`, независимо от того, было выброшено исключение или нет, и независимо от того, было ли оно перехвачено. Используется для гарантированного освобождения ресурсов (закрытие файлов, соединений с базой данных, драйвера браузера в автотестах). Единственные случаи, когда `finally` может не выполниться — вызов `System.exit()` внутри `try` или аварийное завершение JVM.

**`finalize()`** — защищённый метод класса `Object`, который сборщик мусора (Garbage Collector) мог вызвать перед физическим удалением объекта из памяти, чтобы дать объекту шанс освободить внешние ресурсы. Метод считается устаревшим (`deprecated` начиная с Java 9, а начиная с Java 18 — помечен для удаления) из-за непредсказуемости момента вызова (GC не гарантирует, когда и вызовет ли вообще `finalize()`), проблем с производительностью и потенциальных ошибок при «воскрешении» объекта внутри `finalize()`. Вместо него рекомендуется использовать интерфейс `AutoCloseable` и конструкцию `try-with-resources`, либо `java.lang.ref.Cleaner`.

## Пример / когда применяется

```java
// final переменная, метод и класс
final class Constants { // класс нельзя унаследовать
    public static final double PI = 3.14159; // константа

    public final int getVersion() { // метод нельзя переопределить
        return 1;
    }
}
```

```java
// finally — гарантированное освобождение ресурса в автотестах
WebDriver driver = new ChromeDriver();
try {
    driver.get("https://example.com");
    // шаги теста, которые могут выбросить исключение
} catch (Exception e) {
    System.out.println("Ошибка теста: " + e.getMessage());
} finally {
    driver.quit(); // выполнится в любом случае — драйвер не «зависнет»
}
```

```java
// Современная замена finalize() — try-with-resources
class FileResource implements AutoCloseable {
    public void use() {
        System.out.println("Используем ресурс");
    }

    @Override
    public void close() {
        System.out.println("Ресурс закрыт");
    }
}

try (FileResource resource = new FileResource()) {
    resource.use();
} // close() вызовется автоматически, без ненадёжного finalize()
```

## На что смотрит интервьюер

- Знание, что все три термина не связаны друг с другом напрямую и относятся к разным механизмам языка — частая ловушка на собеседовании, где кандидаты путают их из-за похожего звучания.
- Понимание, что `finalize()` устарел и не должен использоваться в новом коде.
- Follow-up: «В каких случаях `finally` может не выполниться?» (`System.exit()`, аварийное завершение JVM, бесконечный цикл или зависание в самом `try`/`catch`), «Можно ли изменить поля объекта, на который ссылается `final`-переменная?» (да, `final` фиксирует только саму ссылку, а не состояние объекта).
- Красный флаг — кандидат путает `finally` с `finalize()` по смыслу назначения или не может объяснить, почему `driver.quit()` в автотестах правильно размещать именно в `finally`.
