# Что такое Selenium Grid?

**Коротко:** Selenium Grid — это компонент экосистемы Selenium, позволяющий распределённо и параллельно запускать тесты на разных машинах, браузерах, версиях и операционных системах через единую точку входа.

## Развёрнутый ответ

В основе Selenium Grid лежит архитектура **Hub-Node** (в Selenium 4 переработана в более гибкую модель с ролями Router, Distributor, Session Map, Node), которая решает две задачи: масштабирование (запуск многих тестов параллельно) и кроссплатформенное покрытие (запуск на разных ОС и версиях браузеров без необходимости иметь их локально).

**Архитектура Selenium 3 (классическая):**
- **Hub** — центральный узел, принимающий запросы от тестов и распределяющий их по нодам.
- **Node** — машина (физическая или виртуальная), на которой реально запускается браузер и выполняется тест.

**Архитектура Selenium 4 (актуальная):**
Grid переработан в набор независимых компонентов, которые можно запускать как единый процесс ("Standalone") или как распределённую систему:
- **Router** — принимает входящие запросы и маршрутизирует их к нужному компоненту.
- **Distributor** — отслеживает доступные ноды и их возможности (capabilities), решает, на какой ноде запустить новую сессию.
- **Session Map** — база данных, хранящая соответствие между активной сессией и нодой, на которой она выполняется.
- **Node** — worker-машина, где физически запускаются браузеры.
- **New Session Queue** — очередь запросов на создание новых сессий, если все ноды заняты.

Тест обращается не напрямую к браузеру, а к Grid (через `RemoteWebDriver`), указывая нужные **capabilities** (браузер, версия, ОС), а Grid сам находит подходящую ноду и создаёт там сессию.

**Зачем нужен Grid:**

1. **Параллельный запуск** — сотни тестов можно выполнить одновременно на разных нодах, значительно сокращая общее время прогона (особенно критично для больших regression-наборов в CI/CD).
2. **Кроссбраузерное/кроссплатформенное тестирование** — можно проверить поведение приложения в Chrome на Windows, Safari на macOS и Firefox на Linux, не имея всех этих окружений локально.
3. **Централизация инфраструктуры** — тестовые машины/браузеры настраиваются один раз на нодах, а не на каждой машине разработчика/CI-агента.
4. **Масштабируемость в облаке** — Grid часто разворачивают в Docker-контейнерах или Kubernetes (например, через Zalando/Aerokube Selenoid, или официальный Docker Selenium), что упрощает горизонтальное масштабирование.

Также существуют облачные аналоги/надстройки над Grid — BrowserStack, Sauce Labs, LambdaTest — которые предоставляют управляемую инфраструктуру Grid как сервис, снимая необходимость поддерживать собственные ноды.

## Пример / когда применяется

Подключение теста к удалённому Grid через `RemoteWebDriver`:

```java
DesiredCapabilities capabilities = new DesiredCapabilities();
capabilities.setBrowserName("chrome");
capabilities.setPlatform(Platform.LINUX);

WebDriver driver = new RemoteWebDriver(
        new URL("http://grid-hub-address:4444/wd/hub"),
        capabilities
);

driver.get("https://example.com");
driver.quit();
```

Запуск Standalone Grid локально через Docker (Selenium 4):

```bash
docker run -d -p 4444:4444 --shm-size="2g" selenium/standalone-chrome:latest
```

Grid применяется, когда команда имеет большой набор регрессионных тестов (сотни-тысячи кейсов) и не может позволить прогонять их последовательно из-за времени выполнения, либо когда нужно гарантировать поддержку приложения в разных браузерных окружениях перед релизом.

## На что смотрит интервьюер

- Знает ли кандидат разницу между архитектурой Hub-Node (Selenium 3) и Router-Distributor-Node (Selenium 4).
- Понимает ли роль `RemoteWebDriver` и capabilities при работе с Grid.
- Может ли назвать практические причины использования Grid (не просто "для параллельности", а с пониманием кроссплатформенного покрытия).
- В курсе ли про Docker Selenium/Selenoid или облачные альтернативы (BrowserStack, Sauce Labs) как индустриальный стандарт вместо ручного поднятия нод.
- Красный флаг: путает Selenium Grid с параллельным запуском тестов на уровне test runner'а (TestNG/JUnit параллелизм) — это разные уровни абстракции, которые часто комбинируют вместе.
