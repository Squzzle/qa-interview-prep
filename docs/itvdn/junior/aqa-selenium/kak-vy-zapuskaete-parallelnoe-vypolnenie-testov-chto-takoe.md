# Как вы запускаете параллельное выполнение тестов? Что такое ThreadLocal?

**Коротко:** Параллельный запуск тестов организуется на уровне test runner'а (TestNG, JUnit 5) через конфигурацию параллелизма по потокам, а безопасный доступ к драйверу в многопоточной среде обеспечивается через `ThreadLocal<WebDriver>`, который хранит отдельный экземпляр драйвера для каждого потока.

## Развёрнутый ответ

При запуске тестов последовательно каждый тест может использовать общий (статический) экземпляр `WebDriver` без проблем. Но как только тесты начинают выполняться параллельно (в разных потоках одновременно), общий `static WebDriver` становится источником состояния гонки (race condition): несколько потоков будут обращаться к одному и тому же браузеру одновременно, что приведёт к непредсказуемым падениям и смешению состояний тестов.

**Настройка параллелизма на уровне test runner'а:**

- **TestNG** позволяет настроить параллелизм в `testng.xml` через атрибут `parallel` (`methods`, `classes`, `tests`, `instances`) и `thread-count`, задающий количество потоков.
- **JUnit 5** поддерживает параллельное выполнение через `junit-platform.properties`, включая `junit.jupiter.execution.parallel.enabled=true` и настройку стратегии (`fixed`, `dynamic`).
- Также параллелизм часто настраивают на уровне build-инструмента — Maven Surefire/Failsafe plugin (`parallel`, `threadCount`) или Gradle (`maxParallelForks`).

**Что такое ThreadLocal и зачем он нужен:**

`ThreadLocal<T>` — класс из `java.lang`, который предоставляет каждому потоку собственную, изолированную копию переменной. Когда несколько потоков обращаются к `ThreadLocal<WebDriver>.get()`, каждый получает свой собственный экземпляр драйвера, не видя и не влияя на экземпляры других потоков.

Типичная реализация паттерна для параллельного запуска Selenium-тестов:

```java
public class DriverManager {
    private static ThreadLocal<WebDriver> driverThreadLocal = new ThreadLocal<>();

    public static void setDriver(WebDriver driver) {
        driverThreadLocal.set(driver);
    }

    public static WebDriver getDriver() {
        return driverThreadLocal.get();
    }

    public static void removeDriver() {
        driverThreadLocal.remove();
    }
}
```

Каждый поток теста при инициализации создаёт свой драйвер и сохраняет его в `ThreadLocal`, а по завершении теста обязательно вызывает `remove()` — это важно, чтобы избежать утечек памяти, так как `ThreadLocal` привязан к потоку, а потоки в пуле часто переиспользуются между разными тестами.

**Дополнительные аспекты параллельного запуска:**

- Каждый тест должен быть **изолирован и независим** — недопустимо разделять состояние между тестами (общие данные, общая сессия логина), иначе параллельный запуск приведёт к нестабильным результатам.
- При работе с Selenium Grid параллельный запуск даёт дополнительный выигрыш — тесты распределяются не только по потокам одной машины, но и по разным нодам физически.
- Нужно учитывать ресурсы CI-агента (CPU, память) — слишком большое количество параллельных браузерных сессий может привести к деградации производительности и ложным падениям тестов из-за таймаутов.

## Пример / когда применяется

Настройка параллелизма в `testng.xml`:

```xml
<suite name="RegressionSuite" parallel="methods" thread-count="4">
    <test name="LoginTests">
        <classes>
            <class name="tests.LoginTest"/>
        </classes>
    </test>
</suite>
```

Использование ThreadLocal в базовом классе тестов:

```java
public class BaseTest {
    protected static ThreadLocal<WebDriver> driver = new ThreadLocal<>();

    @BeforeMethod
    public void setUp() {
        driver.set(new ChromeDriver());
    }

    @AfterMethod
    public void tearDown() {
        driver.get().quit();
        driver.remove();
    }
}
```

Такой подход применяется в любом проекте, где regression suite насчитывает десятки-сотни тестов и последовательный прогон занимает неприемлемо много времени (например, больше часа в CI pipeline).

## На что смотрит интервьюер

- Понимает ли кандидат, почему `static WebDriver` без `ThreadLocal` ломается при параллельном запуске (race condition).
- Знает ли конкретные механизмы настройки параллелизма хотя бы в одном test runner'е (TestNG или JUnit 5).
- Помнит ли про обязательный вызов `remove()` у `ThreadLocal` для предотвращения утечек памяти при переиспользовании потоков в пуле.
- Понимает ли требование независимости и изоляции тестов друг от друга как обязательное условие для безопасного параллелизма.
- Красный флаг: предложение просто "запустить несколько JVM процессов" без понимания механизма ThreadLocal и настроек test runner'а, либо непонимание, зачем вообще нужен ThreadLocal, если "можно просто создавать новый драйвер в каждом методе".
