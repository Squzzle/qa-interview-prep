# Какие виды локаторов страницы существуют? Каковы их преимущества и недостатки?

**Коротко:** Selenium поддерживает 8 стратегий поиска элементов: id, name, className, tagName, linkText, partialLinkText, cssSelector и xpath — у каждой свой баланс скорости, читаемости и устойчивости к изменениям вёрстки.

## Развёрнутый ответ

Локатор — это стратегия, по которой Selenium находит элемент в DOM. В классе `By` определены следующие методы:

- **`By.id`** — поиск по атрибуту `id`. Самый быстрый и надёжный способ, так как `id` теоретически должен быть уникальным на странице. Недостаток — не все элементы имеют `id`, и он может генерироваться динамически (например, React/Angular-приложения часто добавляют случайные суффиксы).
- **`By.name`** — поиск по атрибуту `name`. Часто применяется для полей форм. Недостаток — атрибут не уникален, может встречаться у нескольких элементов.
- **`By.className`** — поиск по CSS-классу. Проблема в том, что классы часто используются для стилизации и повторяются у множества элементов, а также могут содержать несколько классов через пробел, что усложняет точный поиск.
- **`By.tagName`** — поиск по HTML-тегу (`div`, `input`, `a`). Практически никогда не уникален, используется в связке с другими локаторами или для получения списка элементов.
- **`By.linkText`** — точный поиск ссылки (`<a>`) по её видимому тексту. Хрупкий локатор — ломается при любом изменении текста, включая локализацию.
- **`By.partialLinkText`** — поиск ссылки по частичному совпадению текста. Более гибкий, но менее предсказуемый — может найти не тот элемент, если текст встречается в нескольких ссылках.
- **`By.cssSelector`** — поиск по CSS-селектору. Быстрый (браузер использует нативный движок CSS), гибкий, поддерживает сложные комбинации (атрибуты, потомки, псевдоклассы). Не умеет искать по тексту элемента и не может двигаться "вверх" по DOM (к родителю).
- **`By.xpath`** — поиск по XPath-выражению. Самый мощный и универсальный: умеет искать по тексту, двигаться к родительским и соседним элементам, использовать сложную логику (`contains`, `starts-with`, оси `following-sibling` и т.д.). Недостатки — медленнее CSS в некоторых браузерах (особенно IE, но для современных браузеров разница минимальна), менее читаем, легче написать хрупкий или неоднозначный локатор.

Общее правило приоритета (best practice): `id` → `cssSelector` → `xpath` (когда нужен поиск по тексту или движение вверх по DOM). `linkText`/`partialLinkText`/`className`/`tagName` используют реже, обычно в комбинации с другими локаторами или для простых кейсов.

Также важно различать **абсолютный** и **относительный** XPath: абсолютный (`/html/body/div[1]/div[2]`) крайне хрупкий и не рекомендуется — любое изменение структуры страницы его ломает. Относительный (`//div[@class='card']//button`) значительно устойчивее.

## Пример / когда применяется

```java
// By.id — самый надёжный, если id статичен
driver.findElement(By.id("submitBtn"));

// By.cssSelector — гибкий и быстрый
driver.findElement(By.cssSelector("input[type='email']"));

// By.xpath — когда нужен поиск по тексту или родителю
driver.findElement(By.xpath("//button[text()='Оформить заказ']"));
driver.findElement(By.xpath("//label[text()='Email']/following-sibling::input"));

// By.linkText — точное совпадение текста ссылки
driver.findElement(By.linkText("Забыли пароль?"));
```

Пример из практики: форма без `id` и `name`, но с уникальным текстом кнопки — здесь оправдан XPath с `text()`. Если же вёрстка стабильна и элементы имеют осмысленные `data-*` атрибуты (например, `data-testid`), лучший локатор — CSS-селектор по этому атрибуту: `[data-testid='submit-button']`.

## На что смотрит интервьюер

- Знает ли кандидат все 8 стратегий или путается, называя только id/class/xpath.
- Понимает ли разницу между абсолютным и относительным XPath и почему абсолютный — антипаттерн.
- Может ли аргументированно сравнить CSS-селектор и XPath (не просто "xpath медленнее", а с пониманием, когда CSS не справится — поиск по тексту, движение к родителю).
- Знает ли о `data-testid`/`data-qa` атрибутах как современном подходе к стабильным локаторам.
- Красный флаг: утверждение, что XPath "всегда медленнее и его нужно избегать" без понимания контекста — это устаревший миф, применимый в основном к старому Internet Explorer.
