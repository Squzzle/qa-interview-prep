# Что такое CI?

**Коротко:** CI (Continuous Integration, непрерывная интеграция) — практика разработки, при которой изменения кода часто (несколько раз в день) вливаются в общую ветку, а каждое такое изменение автоматически собирается и проверяется тестами, чтобы как можно раньше обнаруживать конфликты и ошибки.

## Развёрнутый ответ

Continuous Integration — это инженерная практика и одновременно набор инструментов, реализующих идею: разработчики регулярно (в идеале — несколько раз в день) интегрируют свои изменения в общий репозиторий (обычно через Pull/Merge Request в основную ветку), и при каждой такой интеграции автоматически запускается пайплайн, включающий как минимум сборку проекта и прогон автотестов.

Цель CI — как можно раньше обнаружить проблему интеграции: конфликты кода, ошибки компиляции, падение тестов из-за несовместимых изменений разных разработчиков. Чем дольше ветка с изменениями существует изолированно (без частой интеграции с основной), тем сложнее и дороже потом её слить — эта проблема называется "merge hell" и как раз то, против чего направлен CI.

Типичный CI-пайплайн для проекта с автотестами включает такие шаги:
1. Checkout кода из репозитория по триггеру (push, Pull Request, merge, по расписанию).
2. Установка зависимостей и окружения (JDK, Node.js, браузеры для UI-тестов и т.д.).
3. Сборка проекта (компиляция, статический анализ кода — linters, SonarQube).
4. Запуск unit-тестов, а затем интеграционных/UI-тестов (может быть в параллельных джобах).
5. Формирование отчётов (JUnit XML, Allure) и публикация результатов.
6. Опционально — сборка артефакта (jar, Docker-образ) и его публикация в реестр.
7. Уведомление команды о результате (Slack, email, статус в Pull Request).

Инструменты CI: Jenkins (классический self-hosted сервер автоматизации), GitHub Actions, GitLab CI/CD, CircleCI, TeamCity, Bamboo, Azure Pipelines. Конфигурация пайплайна обычно описывается декларативно как код (YAML-файл в репозитории — принцип "Pipeline as Code"), что позволяет версионировать сам процесс сборки вместе с кодом проекта.

Важно отличать CI от смежных, но более широких практик:
- **Continuous Delivery (CD)** — расширение CI, при котором после успешного прохождения всех проверок сборка автоматически готовится к развёртыванию (например, публикуется артефакт), но финальный деплой в продакшн может требовать ручного подтверждения.
- **Continuous Deployment** — ещё один шаг вперёд: каждое успешное изменение автоматически разворачивается в продакшн без ручного вмешательства.

Для AQA-инженера CI — ключевая точка интеграции автотестов: именно CI-пайплайн решает, когда запускать какие тесты (smoke — на каждый Pull Request, полный regression — по расписанию ночью или перед релизом), как агрегировать и хранить отчёты, и как быстро сообщать команде о найденных регрессиях.

## Пример / когда применяется

Минимальный пример пайплайна на GitHub Actions, который собирает проект и прогоняет автотесты при каждом push и Pull Request:

```yaml
name: CI Pipeline
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up JDK
        uses: actions/setup-java@v4
        with:
          java-version: "17"
          distribution: "temurin"

      - name: Build with Maven
        run: mvn -B clean compile

      - name: Run tests
        run: mvn test

      - name: Publish test report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: target/surefire-reports
```

Аналогичный минимальный `.gitlab-ci.yml` для GitLab CI:

```yaml
stages:
  - build
  - test

build:
  stage: build
  script:
    - mvn -B clean compile

test:
  stage: test
  script:
    - mvn test
  artifacts:
    when: always
    paths:
      - target/surefire-reports
```

## На что смотрит интервьюер

Интервьюер проверяет, отличает ли кандидат CI от CD/Continuous Deployment и понимает ли он практическую цель CI (раннее обнаружение проблем интеграции за счёт частых, автоматически проверяемых слияний), а не просто "автоматический запуск тестов". Хороший ответ включает конкретные инструменты (Jenkins, GitHub Actions, GitLab CI) и пример структуры пайплайна. Частый follow-up — как в CI встроены автотесты и как формируются/публикуются отчёты о прогоне; красный флаг — если кандидат не может связать CI с реальным опытом настройки или хотя бы чтения такого пайплайна.
