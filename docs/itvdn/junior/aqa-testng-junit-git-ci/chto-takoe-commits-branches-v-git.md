# Что такое commits, branches в Git?

**Коротко:** Commit — это зафиксированный снимок состояния файлов проекта с уникальным идентификатором (хешем), автором, сообщением и ссылкой на родительский коммит; branch — именованный подвижный указатель на конкретный коммит, который позволяет вести параллельную независимую линию разработки.

## Развёрнутый ответ

**Commit.** Коммит — это неизменяемая (immutable) запись в истории репозитория, фиксирующая состояние всех отслеживаемых файлов на момент выполнения `git commit`. Каждый коммит содержит:
- ссылку на дерево файлов (`tree`) — снимок структуры каталогов и файлов;
- ссылку на один или несколько родительских коммитов (у первого коммита родителя нет, у коммита слияния — два и более родителя);
- метаданные: автор, комиттер, дата, сообщение коммита;
- собственный уникальный идентификатор — SHA-хеш, вычисленный от содержимого коммита (включая хеш родителя), поэтому изменение любого коммита в истории меняет хеши всех последующих коммитов.

Последовательность коммитов, связанных через ссылки на родителя, образует направленный ациклический граф (DAG) — историю проекта. Благодаря тому, что каждый коммит ссылается на родителя, а не наоборот, история линейно "читается" назад от текущего состояния.

**Branch.** Ветка в Git — это не копия файлов, а просто файл, содержащий хеш коммита, на который она указывает (например, `.git/refs/heads/main`). Когда делается новый коммит, находясь на определённой ветке, Git обновляет указатель этой ветки на новый коммит. Это делает создание, переключение и удаление веток крайне дешёвыми операциями по сравнению с системами, где ветвление подразумевает копирование всего дерева файлов.

Ветки используются для изоляции разработки: например, основная ветка (`main`/`master`) хранит стабильный код, а новая функциональность разрабатывается в отдельной ветке (`feature/...`), не затрагивая основную линию до момента готовности и code review. После завершения работы ветка вливается в основную через `merge` (с созданием merge-коммита с двумя родителями) или через `rebase` (перенос коммитов поверх другой ветки с переписыванием истории, делая её линейной).

Специальный указатель `HEAD` обозначает, на каком коммите/ветке в данный момент находится рабочая директория. Обычно `HEAD` указывает на ветку (например, `feature/x`), а та, в свою очередь — на конкретный коммит; такое состояние называется "attached HEAD". Если явно перейти на конкретный коммит (`git checkout <hash>`), получаем "detached HEAD" — коммиты в этом состоянии не принадлежат ни одной ветке и могут быть потеряны при сборке мусора, если не создать для них ветку.

## Пример / когда применяется

```bash
# создание новой ветки от текущего состояния main
git checkout main
git checkout -b feature/parallel-tests

# внесение изменений и коммит
git add src/test/java/ParallelTest.java
git commit -m "Add parallel execution config for TestNG"

# просмотр графа коммитов и веток
git log --oneline --graph --all

# слияние ветки обратно в main
git checkout main
git merge feature/parallel-tests

# удаление отработавшей ветки
git branch -d feature/parallel-tests
```

Пример графа истории после нескольких коммитов и слияния:

```
* a1b2c3d (main) Merge feature/parallel-tests
|\
| * e4f5g6h (feature/parallel-tests) Add parallel execution config
|/
* 9i8h7g6 Initial commit
```

## На что смотрит интервьюер

Кандидат должен чётко объяснить, что коммит — это снимок с уникальным хешем и ссылкой на родителя, а ветка — это лёгкий, подвижный указатель на коммит, а не отдельная копия репозитория. Часто спрашивают про detached HEAD и риски работы в этом состоянии (коммиты могут "потеряться" без ссылки от ветки), а также про разницу между merge и rebase применительно к веткам. Красный флаг — если кандидат путает ветку с отдельной директорией/копией проекта или не понимает, почему коммиты образуют граф, а не простой линейный список.
