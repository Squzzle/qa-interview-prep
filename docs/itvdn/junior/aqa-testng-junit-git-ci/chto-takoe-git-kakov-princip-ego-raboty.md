# Что такое Git? Каков принцип его работы?

**Коротко:** Git — распределённая система контроля версий, которая хранит историю проекта как последовательность снимков (snapshots) состояния файлов, а не как набор дельт-изменений; каждый коммит ссылается на родительский, образуя граф, и у каждого разработчика есть полная копия репозитория со всей историей.

## Развёрнутый ответ

Git — самая распространённая распределённая система контроля версий (DVCS), созданная Линусом Торвальдсом в 2005 году для разработки ядра Linux. В отличие от централизованных систем (SVN), Git не хранит данные как список изменений файлов по сравнению с предыдущей версией. Вместо этого при каждом коммите Git делает снимок (snapshot) состояния всех отслеживаемых файлов на данный момент. Если файл не менялся, Git не хранит его копию заново, а просто ссылается на уже сохранённую ранее версию — это делает хранение эффективным.

Ключевые концепции внутреннего устройства Git:
- **Объектная модель.** Git хранит четыре типа объектов: `blob` (содержимое файла), `tree` (структура каталога, ссылки на blob и другие tree), `commit` (метаданные коммита: автор, сообщение, ссылка на дерево и на родительский коммит/коммиты) и `tag` (именованная ссылка на конкретный коммит). Все объекты адресуются по SHA-1 (в новых версиях — опционально SHA-256) хешу своего содержимого, что гарантирует целостность истории — изменение любого объекта меняет и весь путь хешей после него.
- **Три области.** Working directory (рабочая директория с реальными файлами) → staging area / index (промежуточная область, куда попадают изменения командой `git add`, формируя будущий коммит) → repository (`.git`, где хранится история коммитов после `git commit`).
- **Ветки (branches)** — по сути просто именованные указатели на конкретный коммит. Создание ветки — дешёвая операция (создаётся файл с 40-символьным хешем), в отличие от некоторых других VCS, где ветка — это копия всего дерева файлов.
- **HEAD** — указатель на текущий коммит/ветку, с которой идёт работа.
- **Распределённость.** Каждый клон репозитория (`git clone`) содержит полную историю всех коммитов, а не только "рабочую копию" последней версии. Это позволяет работать полностью офлайн (коммитить, смотреть историю, создавать ветки) и синхронизироваться с удалёнными репозиториями (`origin`) только при `push`/`pull`/`fetch`.
- **Слияние (merge) и перебазирование (rebase)** — два основных способа интегрировать изменения из одной ветки в другую. Merge создаёт коммит слияния с двумя родителями, сохраняя историю "как было". Rebase переносит коммиты одной ветки поверх другой, переписывая историю и делая её линейной.

## Пример / когда применяется

Базовый рабочий цикл в Git:

```bash
git init                          # инициализация репозитория
git clone <url>                   # клонирование удалённого репозитория
git status                        # какие файлы изменены
git add file.java                 # добавить изменения в staging area
git commit -m "Add login test"    # зафиксировать снимок как коммит
git log --oneline --graph         # посмотреть историю коммитов
git branch feature/new-tests      # создать ветку
git checkout feature/new-tests    # переключиться на ветку
git push origin feature/new-tests # отправить ветку на удалённый сервер
git pull                          # получить и влить изменения с сервера
```

Просмотр объекта коммита изнутри (для понимания внутреннего устройства):

```bash
git cat-file -p HEAD        # содержимое коммита: tree, parent, author, message
git cat-file -p HEAD^{tree}  # содержимое дерева файлов этого коммита
```

## На что смотрит интервьюер

Хороший ответ отличает Git от "просто хранилища файлов с версиями" — кандидат должен объяснить модель снимков (а не дельт), объектную модель (blob/tree/commit) и то, что ветка — это лёгкий указатель, а не копия файлов. Частый уточняющий вопрос — разница между `merge` и `rebase`, а также что такое staging area и зачем он нужен (позволяет собрать коммит не из всех измененных файлов сразу, а выборочно). Красный флаг — если кандидат знает только команды "по памяти" (`add`, `commit`, `push`), но не может объяснить, что происходит внутри при этих операциях.
