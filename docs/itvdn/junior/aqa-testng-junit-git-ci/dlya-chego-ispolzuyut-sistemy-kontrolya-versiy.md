# Для чего используют системы контроля версий?

**Коротко:** Системы контроля версий (VCS) фиксируют историю изменений файлов проекта, позволяют нескольким людям работать над одним кодом параллельно, безопасно объединять изменения, откатываться к предыдущим состояниям и понимать, кто, когда и зачем внёс каждое изменение.

## Развёрнутый ответ

Система контроля версий (Version Control System, VCS) — инструмент, который хранит историю изменений файлов (обычно исходного кода) во времени. Каждое сохранённое состояние (commit/revision) фиксирует, что именно изменилось, кто это сделал, когда и с каким комментарием.

Основные задачи, которые решает VCS:
- **История изменений и трассируемость.** Можно посмотреть, кто и когда внёс конкретное изменение, и почему (commit message). Это важно при разборе багов ("когда это сломалось?") и при code review.
- **Совместная работа.** Несколько разработчиков/тестировщиков могут одновременно работать над одним проектом, не переписывая изменения друг друга. VCS предоставляет механизмы слияния (merge) параллельных изменений.
- **Откат изменений (rollback).** Если новое изменение сломало функциональность, можно быстро вернуться к рабочей версии кода.
- **Ветвление (branching).** Позволяет разрабатывать новую функциональность или чинить баги в изолированной ветке, не затрагивая стабильную версию кода в основной ветке, а затем интегрировать изменения через merge/pull request.
- **Резервное копирование.** Распределённые VCS (Git) хранят полную историю у каждого разработчика локально, что снижает риск потери данных при поломке центрального сервера.
- **Основа для CI/CD.** Практически все современные пайплайны непрерывной интеграции и доставки триггерятся событиями в VCS (push, merge в определённую ветку, создание тега).

Различают два основных типа VCS:
- **Централизованные** (например, SVN, Perforce) — единый центральный репозиторий, к которому клиенты обращаются за историей и для сохранения изменений.
- **Распределённые** (например, Git, Mercurial) — каждый разработчик имеет полную копию репозитория со всей историей локально, что даёт возможность работать офлайн и повышает отказоустойчивость.

Для тестировщика/AQA-инженера VCS важна не только для хранения кода автотестов, но и для организации совместной работы над тестовой базой, тест-планами (если они хранятся как код) и для интеграции автотестов в CI/CD-пайплайн, где сборка и запуск тестов триггерятся изменениями в репозитории.

## Пример / когда применяется

Типичный рабочий процесс с использованием VCS (Git) в команде:

1. Разработчик создаёт новую ветку от `main`: `git checkout -b feature/login-tests`.
2. Пишет автотесты, коммитит изменения с понятными сообщениями: `git commit -m "Add login smoke tests"`.
3. Отправляет ветку на сервер: `git push origin feature/login-tests`.
4. Создаёт Pull Request/Merge Request для code review.
5. После одобрения коллегами ветка сливается (merge) в `main`, что автоматически триггерит CI-пайплайн, который запускает автотесты и деплой.

Если позже выяснится, что какой-то коммит сломал сборку, можно быстро найти проблемный коммит через `git bisect` или откатить его через `git revert`.

## На что смотрит интервьюер

Интервьюер ожидает, что кандидат назовёт не только "хранение кода", но и конкретные сценарии: совместную работу, ветвление, откат, интеграцию с CI/CD. Плюсом будет упоминание разницы между централизованными и распределёнными VCS и понимание, зачем нужна именно распределённая модель (офлайн-работа, надёжность, скорость локальных операций). Красный флаг — если кандидат воспринимает VCS исключительно как "облачное хранилище файлов", не понимая механизмов истории, веток и слияния.
