# Для чего нужны GitHub, GitLab и другие, базирующиеся на Git, вебхостинги проектов?

**Коротко:** GitHub, GitLab, Bitbucket и подобные сервисы — это веб-платформы, надстроенные над Git, которые предоставляют централизованное удалённое хранилище репозиториев плюс инструменты совместной работы: Pull/Merge Request, code review, issue tracking, CI/CD, управление доступом и документацию проекта.

## Развёрнутый ответ

Сам Git — это только инструмент командной строки/протокол для распределённого контроля версий; он не предоставляет веб-интерфейс, систему обсуждения кода или управление задачами. Git-хостинги (GitHub, GitLab, Bitbucket, Gitea, Azure DevOps Repos и т.п.) берут Git как транспортный протокол хранения истории и надстраивают над ним слой сервисов, нужных команде для полноценной совместной разработки:

- **Централизованная точка синхронизации.** Хотя Git распределён и каждый разработчик хранит полную историю локально, команде нужна общая "точка правды" (`origin`), куда все пушат и откуда все тянут изменения — это и есть роль хостинга.
- **Pull Request / Merge Request.** Механизм предложения изменений: разработчик оформляет ветку как запрос на слияние в основную ветку, коллеги оставляют комментарии построчно к коду (code review), обсуждают и одобряют изменения перед мержем.
- **Управление доступом и правами.** Роли (owner, maintainer, developer, reporter), защищённые ветки (branch protection: запрет прямого push в `main`, обязательные проверки перед мержем), права на отдельные репозитории или группы репозиториев.
- **Issue tracking / Project management.** Ведение задач, багов, досок (kanban-board), связывание issue с коммитами и Pull Request.
- **CI/CD "из коробки".** GitHub Actions, GitLab CI/CD и подобные встроенные пайплайны запускают сборку, тесты и деплой автоматически по событиям в репозитории (push, merge, тег).
- **Дополнительные сервисы:** wiki и документация проекта, хранение артефактов и Docker-образов (GitHub Packages, GitLab Container Registry), управление релизами и тегами, security-сканирование зависимостей (Dependabot, GitLab Dependency Scanning), веб-хуки для интеграции с внешними системами (Slack, Jira).

Для AQA/QA-инженера такие платформы важны как минимум по трём причинам: хранение и версионирование кода автотестов совместно с командой разработки; организация code review автотестов через Pull Request (чтобы плохие/нестабильные тесты не попадали в основную ветку без проверки); и запуск автотестов в CI-пайплайне при каждом изменении кода (например, GitHub Actions или GitLab CI триггерит прогон regression-набора при каждом Pull Request).

## Пример / когда применяется

Типичный процесс работы с Pull Request на GitHub:

```bash
git checkout -b feature/add-api-tests
# ... пишем автотесты ...
git commit -m "Add API tests for /users endpoint"
git push origin feature/add-api-tests
```

Далее в веб-интерфейсе GitHub создаётся Pull Request из ветки `feature/add-api-tests` в `main`. GitHub Actions автоматически запускает workflow (например, `mvn test`), проверяя, что новые тесты и вся сборка проходят успешно. Коллеги оставляют комментарии к коду, после одобрения (`Approve`) и прохождения всех проверок ветку можно слить в `main` — часто с автоматическим squash-merge для сохранения линейной истории.

Пример минимального workflow-файла GitHub Actions, запускающего тесты при каждом Pull Request:

```yaml
name: CI
on:
  pull_request:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: "17"
      - run: mvn test
```

## На что смотрит интервьюер

Кандидат должен чётко разделять Git (протокол/инструмент версионирования) и хостинг (веб-сервис с дополнительными возможностями) — это частая путаница у начинающих. Хороший ответ упоминает Pull/Merge Request и code review как ключевую ценность таких платформ для командной разработки, а не только "место, где хранится код". Плюсом будет понимание встроенного CI/CD (GitHub Actions/GitLab CI) и того, как он связан с автотестами — именно события в хостинге чаще всего триггерят запуск тестового набора.
