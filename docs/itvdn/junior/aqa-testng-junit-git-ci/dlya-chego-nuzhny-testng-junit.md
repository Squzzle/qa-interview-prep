# Для чего нужны TestNG/JUnit?

**Коротко:** TestNG и JUnit — это фреймворки для написания и запуска автоматизированных тестов на Java (unit-, интеграционных, end-to-end), которые берут на себя организацию тестов, управление жизненным циклом, проверку результатов и формирование отчётов.

## Развёрнутый ответ

TestNG и JUnit — тестовые фреймворки для платформы Java (JUnit также используется в Kotlin/Scala-проектах). Их основная задача — снять с разработчика/тестировщика рутину по организации тестового кода: обнаружение тестовых методов, их запуск в нужном порядке, подготовку и очистку окружения, проверку утверждений (assertions), группировку и параллельный запуск, а также генерацию отчётов о результатах.

Ключевые возможности, которые предоставляют оба фреймворка:
- Аннотации для описания жизненного цикла теста (`@Test`, методы подготовки/очистки перед и после тестов и наборов тестов).
- Библиотека assertions для проверки ожидаемого и фактического результата.
- Возможность группировать тесты и запускать их выборочно (по тегам/группам, по классам, по пакетам).
- Поддержка параметризованных тестов — запуск одного теста с разным набором входных данных.
- Интеграция со сборщиками (Maven, Gradle) и CI-системами (Jenkins, GitLab CI, GitHub Actions) — единый формат отчётов (например, JUnit XML), который понимают почти все CI-инструменты.
- Возможность запуска тестов параллельно и управления зависимостями между тестами.

Исторически JUnit — более старый и распространённый в мире Java unit-тестирования фреймворк (особенно JUnit 5 / Jupiter с модульной архитектурой: Platform, Jupiter, Vintage). TestNG появился позже и изначально создавался с прицелом на более сложные сценарии — интеграционное и end-to-end тестирование, поддержку явных зависимостей между тестами (`dependsOnMethods`), гибкую конфигурацию через XML-файлы (`testng.xml`), встроенную поддержку параллельного запуска и групп тестов "из коробки".

В контексте автоматизации тестирования (AQA) оба фреймворка чаще всего используются не для unit-тестов кода приложения, а как "движок" для UI/API-автотестов поверх Selenium, REST Assured и т.п.: они управляют запуском тестовых сценариев, проверяют условия (assert), обеспечивают up/down окружения (драйвер браузера, соединение с БД) и интегрируются с CI/CD и системами отчётности (Allure, ExtentReports).

## Пример / когда применяется

Пример структуры теста на TestNG с параметризацией и группами:

```java
import org.testng.annotations.*;
import static org.testng.Assert.assertEquals;

public class LoginTest {

    @BeforeMethod
    public void setUp() {
        // инициализация драйвера браузера
    }

    @Test(groups = {"smoke"}, dataProvider = "credentials")
    public void loginWithValidCredentials(String user, String pass) {
        // выполнение логина
        assertEquals(actualResult, expectedResult);
    }

    @DataProvider(name = "credentials")
    public Object[][] credentials() {
        return new Object[][] {
            {"user1", "pass1"},
            {"user2", "pass2"}
        };
    }

    @AfterMethod
    public void tearDown() {
        // закрытие драйвера
    }
}
```

Аналог на JUnit 5:

```java
import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.assertEquals;

class LoginTest {

    @BeforeEach
    void setUp() {
        // инициализация драйвера браузера
    }

    @Test
    @Tag("smoke")
    void loginWithValidCredentials() {
        assertEquals(expectedResult, actualResult);
    }

    @AfterEach
    void tearDown() {
        // закрытие драйвера
    }
}
```

Такие тесты запускаются локально из IDE, через сборщик (`mvn test`, `gradle test`) или в CI-пайплайне, а результаты агрегируются в отчёт.

## На что смотрит интервьюер

Частые ошибки кандидатов: путают фреймворк для запуска тестов (TestNG/JUnit) с библиотекой для работы с браузером (Selenium) — важно чётко разделять эти уровни. Также часто не могут назвать конкретные отличия TestNG от JUnit (группы и зависимости тестов, параллельный запуск через `testng.xml` против JUnit 5 Platform/Extensions). Красный флаг — если кандидат не понимает, зачем вообще нужен фреймворк, и предлагает писать тесты как обычный `main`-метод с `if`. Хороший ответ также упоминает интеграцию с CI и отчётностью, а не только сам факт запуска тестов.
