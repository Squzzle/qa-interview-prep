# Что нужно проверять при использовании сканера отпечатка/Face ID?

**Коротко:** Нужно проверять корректность распознавания (успех/отказ), обработку ошибок и лимитов попыток, поведение при смене/удалении биометрических данных, fallback на PIN/пароль, безопасность хранения данных и корректную работу на разных устройствах/ОС-версиях.

## Развёрнутый ответ

Тестирование биометрической аутентификации (Touch ID/сканер отпечатка на Android, Face ID на iOS) требует внимания как к функциональным сценариям, так и к безопасности и UX. Основные зоны проверки:

**1. Позитивные сценарии:**
- Успешная авторизация зарегистрированным отпечатком/лицом.
- Корректный переход в приложение/подтверждение действия (например, оплаты) после успешной биометрии.

**2. Негативные сценарии:**
- Неверный отпечаток/незарегистрированное лицо — должен быть отказ с понятным сообщением.
- Превышение лимита неудачных попыток — система должна блокировать биометрию и предлагать fallback (PIN-код, пароль, графический ключ) согласно политике ОС (обычно 5 неудачных попыток на Face ID/Touch ID).
- Отсутствие зарегистрированной биометрии на устройстве — приложение должно корректно обрабатывать этот случай (предложить альтернативный способ входа, а не крашиться).
- Отключение датчика на уровне ОС во время работы приложения (например, пользователь зашёл в Settings и отключил Face ID) — приложение должно адекватно реагировать при следующей попытке.

**3. Fallback-механизмы:**
- Проверка перехода на альтернативный способ аутентификации (PIN, пароль, паттерн) при недоступности биометрии.
- Проверка, что fallback предлагается корректно и не создаёт дыру в безопасности (например, слишком лёгкий обход).

**4. Изменение биометрических данных:**
- Добавление нового отпечатка/лица после того, как оно уже было привязано к аккаунту в приложении — по правилам безопасности приложение должно инвалидировать сохранённый биометрический ключ (Android: `KeyStore` привязывает ключ к текущему набору биометрии, при изменении набора ключ становится недействителен — это нужно протестировать явно).
- Удаление всех отпечатков/лиц с устройства — приложение должно перевести пользователя на альтернативный метод входа.

**5. Безопасность:**
- Данные отпечатка/лица никогда не должны передаваться в приложение или на сервер — только результат аутентификации (успех/отказ) через системный API (BiometricPrompt на Android, LocalAuthentication framework на iOS). Тестировщик должен убедиться, что приложение использует официальный API, а не пытается получить "сырые" биометрические данные.
- Проверка, что при использовании биометрии для критичных операций (оплата, доступ к банковским данным) есть корректная деградация до более сильной аутентификации при подозрительных условиях.

**6. Кроссплатформенные и девайс-специфичные нюансы:**
- Различия в UI системного диалога биометрии на разных версиях Android (BiometricPrompt появился в API 28+, до этого — FingerprintManager, deprecated).
- Проверка на разных устройствах: разные сканеры (в экране, сбоку, на задней панели) — часто отличается стабильность и скорость распознавания.
- На iOS — проверка Face ID при разных условиях освещения, ношении маски (Face ID с маской начиная с iOS 15.4 на iPhone с Face ID может работать в очках), альтернативные внешние виды (макияж, борода, очки).

**7. Accessibility:**
- Проверка, что для пользователей, не способных использовать биометрию (например, ампутация пальцев), всегда доступен альтернативный метод входа.

## Пример / когда применяется

Чек-лист для тест-кейсов биометрии в банковском приложении:

| Сценарий | Ожидаемый результат |
|---|---|
| Успешный отпечаток | Вход выполнен, переход на главный экран |
| Неверный отпечаток (1-4 попытка) | Показано сообщение об ошибке, попытка засчитана |
| 5-я неверная попытка подряд | Биометрия временно заблокирована, предложен PIN |
| Биометрия не настроена на устройстве | Экран биометрии не показывается, сразу предлагается PIN/пароль |
| Пользователь добавил новый отпечаток после привязки в приложении | Сохранённый биометрический ключ инвалидирован, требуется повторный вход через пароль + повторная привязка биометрии |
| Отключение Face ID в настройках ОС во время использования приложения | При следующей попытке входа приложение предлагает fallback, а не крашится |

## На что смотрит интервьюер

- Понимание, что тестируется не сам сканер (это hardware), а интеграция приложения с системным API биометрии.
- Знание про инвалидацию ключей при изменении набора биометрических данных — это частый пропущенный сценарий у новичков.
- Понимание security-аспекта: приложение никогда не должно получать доступ к "сырым" биометрическим данным.
- Знание об ограничении числа попыток и fallback-механизме.
- Red flag — если кандидат говорит только "проверить, что палец распознаётся", не упоминая негативные сценарии, безопасность и fallback.
