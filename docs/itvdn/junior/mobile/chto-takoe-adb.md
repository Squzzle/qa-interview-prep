# Что такое ADB?

**Коротко:** ADB (Android Debug Bridge) — консольная утилита из Android SDK Platform Tools, позволяющая взаимодействовать с Android-устройством или эмулятором через компьютер: устанавливать приложения, снимать логи, выполнять shell-команды, управлять файловой системой устройства.

## Развёрнутый ответ

ADB работает по клиент-серверной архитектуре и состоит из трёх компонентов:

1. **Клиент** — работает на машине разработчика/тестировщика, отправляет команды.
2. **Сервер (adb server)** — фоновый процесс на компьютере, управляющий соединением между клиентом и устройствами.
3. **Демон (adbd)** — фоновый процесс, работающий непосредственно на Android-устройстве или эмуляторе, принимающий команды от сервера.

Для тестировщика ADB — один из ключевых инструментов ежедневной работы, так как позволяет:

- **Устанавливать/удалять приложения**: `adb install`, `adb uninstall`.
- **Снимать логи** приложения и системы: `adb logcat` — критично для отладки крашей и воспроизведения багов.
- **Выполнять shell-команды** на устройстве: `adb shell` (запускает интерактивную shell-сессию, доступ к файловой системе, `pm` — package manager, `am` — activity manager).
- **Управлять файлами**: `adb push` (загрузить файл на устройство), `adb pull` (скачать файл с устройства) — полезно для получения БД приложения, конфигов, скриншотов.
- **Управлять сетевыми настройками**, эмулировать разные условия (в связке с другими инструментами).
- **Снимать скриншоты и запись экрана**: `adb shell screencap`, `adb shell screenrecord`.
- **Управлять разрешениями (permissions)** приложения без взаимодействия с UI: `adb shell pm grant/revoke`.
- **Эмулировать нажатия клавиш и события**: `adb shell input tap/swipe/keyevent` — используется как в ручном, так и в автоматизированном тестировании (в том числе Appium использует ADB "под капотом" для Android).

ADB также широко используется автоматизированными фреймворками — Appium, Espresso и другие инструменты фактически строятся поверх ADB для взаимодействия с устройством на Android.

## Пример / когда применяется

```bash
# Проверить список подключённых устройств/эмуляторов
adb devices

# Установить APK на устройство
adb install -r app-debug.apk

# Снять логи приложения в реальном времени, отфильтровав по тегу
adb logcat | grep "MyAppTag"

# Сохранить логи в файл для приложения к баг-репорту
adb logcat -d > device_log.txt

# Получить список установленных пакетов, содержащих "myapp"
adb shell pm list packages | grep myapp

# Выдать приложению разрешение на камеру без UI-взаимодействия
adb shell pm grant com.example.app android.permission.CAMERA

# Сделать скриншот экрана устройства и сохранить локально
adb shell screencap -p /sdcard/screen.png
adb pull /sdcard/screen.png ./screen.png
```

Типичный кейс использования в работе QA: приложение упало на устройстве, но воспроизвести баг стабильно не получается. Тестировщик подключает устройство по ADB, запускает `adb logcat`, воспроизводит шаги и прикладывает полученный стектрейс к баг-репорту — это резко ускоряет коммуникацию с разработчиками.

## На что смотрит интервьюер

- Знание базовых команд ADB (`install`, `logcat`, `shell`, `push`/`pull`) — не в теории, а по факту использования в работе.
- Понимание клиент-серверной архитектуры ADB (хотя бы на базовом уровне).
- Умение объяснить, зачем ADB нужен именно тестировщику (а не только разработчику) — снятие логов, воспроизведение и локализация багов.
- Red flag — если кандидат знает ADB только "по названию", но не может привести ни одной реальной команды или сценария использования.
