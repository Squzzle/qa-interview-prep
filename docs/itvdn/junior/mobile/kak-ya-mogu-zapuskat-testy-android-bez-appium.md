# Как я могу запускать тесты Android без Appium?

**Коротко:** Есть несколько альтернатив Appium для автоматизации Android: нативные фреймворки Espresso и UI Automator (Google), а также сторонние решения вроде Robotium, Robolectric (для unit-уровня), и облачные/BDD-обёртки поверх этих инструментов; выбор зависит от того, нужен ли доступ к исходному коду приложения и какой уровень тестирования требуется.

## Развёрнутый ответ

Appium — универсальный кроссплатформенный инструмент (через WebDriver протокол), но он не единственный вариант, и в ряде случаев нативные инструменты предпочтительнее:

**1. Espresso** — официальный фреймворк от Google для UI-тестирования Android, часть Android Jetpack. Работает изнутри приложения (требует исходный код и интеграцию тестов в проект), синхронизируется с UI-потоком автоматически, что делает тесты значительно более стабильными (меньше flaky-тестов) по сравнению с Appium. Пишется на Kotlin/Java, тесты запускаются как instrumented-тесты через Gradle.

**2. UI Automator** — тоже от Google, но в отличие от Espresso может тестировать не только своё приложение, а весь экран устройства, включая системные приложения и взаимодействие между несколькими приложениями (например, проверка push-уведомлений, переключение между приложениями). Часто используется в связке с Espresso (`UiAutomator` + `Espresso` через `AndroidX Test`).

**3. Robolectric** — фреймворк для запуска Android unit-тестов на JVM без эмулятора/устройства, значительно ускоряет прогон тестов (не требует загрузки эмулятора), но не даёт полноценного end-to-end покрытия реального UI-рендеринга на устройстве.

**4. Robotium** — более старый opensource-фреймворк для UI-тестирования Android (предшественник современных решений), сейчас используется редко, так как Espresso полностью его вытеснил.

**5. Firebase Test Lab / облачные фермы устройств** — сами по себе не альтернатива инструменту автоматизации, а среда выполнения: тесты, написанные на Espresso/UI Automator, можно прогонять на реальных облачных устройствах через Firebase Test Lab без необходимости содержать собственный парк девайсов.

**Ключевое отличие Espresso/UI Automator от Appium:**
- Appium управляет приложением "снаружи" через WebDriver-протокол, эмулируя пользователя, независимо от языка написания самого приложения — удобно, когда нет доступа к исходному коду или нужна кроссплатформенность (Android + iOS одним фреймворком).
- Espresso работает "изнутри" — тесты компилируются вместе с приложением, что даёт более быстрые и стабильные (менее flaky) тесты за счёт автоматической синхронизации с UI-потоком, но требует доступа к коду и не является кроссплатформенным.

## Пример / когда применяется

Пример простого Espresso-теста на Kotlin:

```kotlin
@RunWith(AndroidJUnit4::class)
class LoginTest {

    @get:Rule
    val activityRule = ActivityScenarioRule(LoginActivity::class.java)

    @Test
    fun successfulLogin_navigatesToHomeScreen() {
        onView(withId(R.id.usernameField))
            .perform(typeText("testuser"), closeSoftKeyboard())
        onView(withId(R.id.passwordField))
            .perform(typeText("password123"), closeSoftKeyboard())
        onView(withId(R.id.loginButton)).perform(click())

        onView(withId(R.id.homeScreenTitle))
            .check(matches(isDisplayed()))
    }
}
```

Пример использования UI Automator для взаимодействия с системным диалогом (например, разрешением на доступ к камере), который недоступен для Espresso, так как выходит за пределы вашего приложения:

```kotlin
val device = UiDevice.getInstance(InstrumentationRegistry.getInstrumentation())
val allowButton = device.wait(
    Until.findObject(By.text("Allow")), 5000
)
allowButton?.click()
```

Когда стоит выбрать Espresso/UI Automator вместо Appium: команда разработки и QA работает в одной кодовой базе, важна максимальная стабильность и скорость прогона тестов в CI, кроссплатформенность (iOS) не требуется или тестируется отдельным набором инструментов (XCUITest).

## На что смотрит интервьюер

- Знание конкретных альтернатив (Espresso, UI Automator), а не общий ответ "можно и без Appium как-нибудь".
- Понимание принципиальной разницы между "изнутри" (Espresso) и "снаружи" (Appium) подходами к автоматизации.
- Понимание компромиссов: стабильность/скорость (Espresso) против кроссплатформенности и независимости от исходного кода (Appium).
- Знание, что UI Automator нужен для сценариев, выходящих за пределы одного приложения (системные диалоги, переключение между приложениями).
- Red flag — если кандидат не может назвать ни одной конкретной альтернативы или путает Espresso с Appium по принципу работы.
