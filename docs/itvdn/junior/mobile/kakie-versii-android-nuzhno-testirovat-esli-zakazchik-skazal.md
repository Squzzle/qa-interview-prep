# Какие версии Android нужно тестировать, если заказчик сказал поддерживать с версии 5.0?

**Коротко:** Нужно тестировать не только минимальную (5.0/Lollipop) и максимальную актуальную версию, а репрезентативную выборку промежуточных версий с учётом их доли на рынке, отличий в API/UI и особенностей вендорских прошивок.

## Развёрнутый ответ

Требование "поддерживать с версии 5.0" (Android Lollipop, API level 21) означает, что приложение должно корректно работать на диапазоне от Android 5.0 до самой актуальной версии на момент тестирования (например, Android 15/16). Это довольно широкий диапазон, охватывающий более 10 лет эволюции платформы, поэтому подход "протестировать всё" физически нереалистичен — нужно строить **репрезентативную матрицу**.

Принципы построения матрицы:

1. **Граничные версии** — обязательно тестируем минимальную (5.0) и максимальную актуальную версию, так как именно на границах диапазона чаще всего проявляются баги (устаревшие API, deprecated-методы, или наоборот новые ограничения/permissions).
2. **Ключевые "переломные" версии** — версии, где Google вносил значимые архитектурные изменения:
   - Android 6.0 (Marshmallow) — runtime permissions.
   - Android 7.0 (Nougat) — multi-window, изменения в работе с сертификатами (Network Security Config).
   - Android 8.0 (Oreo) — background execution limits, notification channels.
   - Android 9.0 (Pie) — ограничения на использование HTTP (cleartext traffic по умолчанию запрещён).
   - Android 10 — scoped storage, тёмная тема, изменения в разрешениях на геолокацию.
   - Android 11-12 — дальнейшее ужесточение scoped storage, package visibility.
   - Android 13-14 — гранулярные разрешения на медиа, notification permission.
3. **Доля рынка** — по данным аналитики выбираются версии, которые реально используют пользователи приложения (не тратим ресурсы на версии с долей &lt;1%, если это не оговорено отдельно).
4. **Вендорские особенности** — учитываем не только "чистый" Android (AOSP), но и кастомизации крупных вендоров (Samsung One UI, Xiaomi MIUI), так как они могут вести себя иначе даже на одной версии Android.
5. **Форм-факторы** — версия ОС тестируется в сочетании с разными размерами экранов/плотностью пикселей, так как баги часто возникают на стыке версии ОС и адаптивности верстки.

На практике итоговая матрица может выглядеть так: 5.0, 6.0, 8.0, 10, 12, 14 (актуальная) — то есть не все версии подряд, а ключевые точки, дающие максимальное покрытие рисков при разумных трудозатратах.

## Пример / когда применяется

Пример матрицы устройств для регрессионного тестирования (заказчик просит поддержку 5.0+):

| Версия Android | API Level | Почему включена |
|---|---|---|
| 5.0 | 21 | Минимальная поддерживаемая (граница) |
| 6.0 | 23 | Runtime permissions |
| 8.1 | 27 | Background limits, notification channels |
| 9.0 | 28 | Запрет cleartext traffic по умолчанию |
| 10 | 29 | Scoped storage |
| 13/14 (актуальная) | 33/34 | Максимальная поддерживаемая, новые permissions |

При наличии автоматизации имеет смысл прогонять smoke-тесты на всех точках матрицы через облачные фермы устройств (BrowserStack, Firebase Test Lab), а полный регресс — на 2-3 ключевых версиях.

## На что смотрит интервьюер

- Понимает ли кандидат, что нельзя тестировать "все версии подряд" — нужен риск-ориентированный подход.
- Знает ли про ключевые архитектурные изменения Android (permissions, scoped storage, cleartext traffic) — это показывает реальный опыт, а не теорию.
- Упоминает ли вендорскую фрагментацию (Samsung, Xiaomi) как отдельный риск.
- Red flag — ответ "тестируем только последнюю версию" или "тестируем все версии подряд без анализа" — оба варианта показывают отсутствие практического понимания.
