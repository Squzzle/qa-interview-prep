# Каков формат файлов сборок приложений для Android и iOS?

**Коротко:** Для Android — это APK и AAB (Android App Bundle), для iOS — IPA; для отладочных/тестовых целей также используются форматы вроде AAB→APKS (bundletool) и IPA для ad-hoc/enterprise распространения.

## Развёрнутый ответ

**Android:**
- **APK (Android Package Kit)** — исполняемый файл приложения, по сути ZIP-архив, содержащий скомпилированный код (`classes.dex`), ресурсы, манифест (`AndroidManifest.xml`), нативные библиотеки (`.so`) и подпись. Это классический формат для установки приложения на устройство напрямую (например, через `adb install`).
- **AAB (Android App Bundle)** — современный формат публикации в Google Play, введён Google как обязательный для новых приложений с августа 2021 года. AAB не устанавливается напрямую на устройство — вместо этого Google Play генерирует из него оптимизированные APK под конкретную конфигурацию устройства (архитектура процессора, язык, плотность экрана) — это называется Dynamic Delivery, что уменьшает размер итогового APK для пользователя.
- Для локального тестирования AAB QA может использовать утилиту **bundletool**, которая генерирует `.apks` файл (набор APK для разных конфигураций) из AAB.

**iOS:**
- **IPA (iOS App Store Package)** — архив (тоже по сути ZIP), содержащий скомпилированное приложение (`.app` bundle), ресурсы, Info.plist, подпись (provisioning profile + code signing). Это единственный формат для распространения iOS-приложений — как через App Store, так и для внутреннего тестирования (TestFlight, ad-hoc, enterprise-дистрибуция).
- В отличие от Android, у iOS нет отдельного "бандл-формата" для магазина — Apple использует собственную систему App Thinning на своей стороне для оптимизации доставки под конкретное устройство, но разработчику для загрузки в App Store Connect всё равно нужен единый IPA.

Для QA важно понимать разницу: если нужно установить сборку вручную на тестовое устройство Android — обычно просят APK (а не AAB, так как AAB напрямую не устанавливается). Для iOS тестовая сборка приходит как IPA через TestFlight или напрямую (для jailbreak/enterprise-девайсов).

## Пример / когда применяется

```bash
# Установка APK на Android-устройство через ADB
adb install -r app-release.apk

# Генерация APKS-набора из AAB через bundletool для локального тестирования
bundletool build-apks --bundle=app-release.aab --output=app-release.apks --local-testing

# Установка сгенерированного APK на подключённое устройство
bundletool install-apks --apks=app-release.apks
```

Для iOS сборка распространяется чаще всего так:
- Через **TestFlight** — Apple сама конвертирует загруженный в App Store Connect IPA/архив в тестовую сборку для внешних/внутренних тестеров.
- Через **ad-hoc IPA** — вручную устанавливается на зарегистрированные UDID устройств (например, через Xcode или сторонние сервисы типа Diawi).

## На что смотрит интервьюер

- Знание, что AAB — это формат публикации, а не установки, и в чём разница с APK.
- Понимание, зачем Google ввёл AAB (уменьшение размера приложения через Dynamic Delivery/App Thinning).
- Знание инструментов для работы со сборками (`adb install`, `bundletool`).
- Для iOS — знание, что IPA единственный формат дистрибуции, и понимание разницы между TestFlight, ad-hoc и enterprise-распространением.
- Red flag — если кандидат не различает APK и AAB или не знает, что AAB нельзя установить напрямую через `adb install`.
