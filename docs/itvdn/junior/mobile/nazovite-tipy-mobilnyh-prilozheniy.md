# Назовите типы мобильных приложений.

**Коротко:** Существует три основных типа мобильных приложений — нативные (Native), веб-приложения (Web/Mobile Web) и гибридные (Hybrid), а также кроссплатформенные приложения на базе Flutter/React Native, которые часто выделяют в отдельную категорию.

## Развёрнутый ответ

**1. Нативные приложения (Native apps)**
Разрабатываются под конкретную платформу с использованием "родных" языков и SDK: Kotlin/Java для Android, Swift/Objective-C для iOS. Устанавливаются через официальные магазины (Google Play, App Store).
- Плюсы: максимальная производительность, полный доступ к возможностям устройства (камера, датчики, Bluetooth, NFC), лучший UX, соответствие нативным гайдлайнам платформы (Material Design / Human Interface Guidelines).
- Минусы: разработка и поддержка отдельно под каждую платформу — дороже и дольше.

**2. Веб-приложения (Web / Mobile Web apps)**
Работают через браузер устройства, не требуют установки, разрабатываются на HTML/CSS/JavaScript. Часто реализуются как Progressive Web App (PWA) — с возможностью добавления на домашний экран, офлайн-режимом через Service Workers, push-уведомлениями.
- Плюсы: одна кодовая база под все платформы, не нужно проходить модерацию магазинов приложений, мгновенное обновление (не нужно ждать одобрения store).
- Минусы: ограниченный доступ к нативным возможностям устройства, зависимость от браузера и сети, обычно менее производительны.

**3. Гибридные приложения (Hybrid apps)**
Пишутся на веб-технологиях (HTML/CSS/JS), но упаковываются в нативную оболочку (обычно через WebView) и устанавливаются как обычное приложение через магазин. Технологии: Apache Cordova/PhoneGap, Ionic.
- Плюсы: одна кодовая база, доступ к части нативных API через плагины, дешевле нативной разработки.
- Минусы: производительность и UX хуже нативных, ограничения WebView, зависимость от плагинов для доступа к железу.

**4. Кроссплатформенные приложения (Cross-platform)**
Отдельная категория, которую иногда объединяют с гибридными, но технически она ближе к нативной по производительности: React Native, Flutter, Kotlin Multiplatform. Код пишется один раз на общем языке/фреймворке (Dart, JS/TS, Kotlin), но компилируется в нативный код или использует нативные компоненты рендеринга, а не WebView.
- Плюсы: близкая к нативной производительность, единая кодовая база, доступ к нативным модулям.
- Минусы: иногда требуется писать platform-specific код для сложных кейсов, зависимость от зрелости фреймворка.

## Пример / когда применяется

| Тип | Пример | Когда выбирают |
|---|---|---|
| Native | Instagram (core), банковские приложения | Высокие требования к производительности, безопасности, глубокая интеграция с ОС |
| Web/PWA | Twitter Lite, Starbucks PWA | Быстрый запуск, минимальный бюджет, нет необходимости в store |
| Hybrid | Ранние версии Uber, некоторые корпоративные приложения | Быстрый MVP, ограниченный бюджет, несложный UI |
| Cross-platform | Facebook Ads Manager (React Native), Google Ads (Flutter) | Нужна нативная производительность + экономия на разработке под 2 платформы |

Для QA-инженера тип приложения напрямую влияет на выбор инструментов автоматизации: Native → Espresso/XCUITest/Appium; Web → Selenium/Playwright (мобильный браузер); Hybrid/Cross-platform → Appium (с учётом WebView-контекстов) или Detox (для React Native).

## На что смотрит интервьюер

- Знание всех основных типов, а не только "нативные и гибридные".
- Понимание плюсов/минусов каждого типа, а не просто заучивание названий.
- Умение привести реальные примеры приложений.
- Понимание, как тип приложения влияет на стратегию и инструменты тестирования — это отличает практика от теоретика.
- Red flag — путаница между "гибридным" и "кроссплатформенным" без понимания технической разницы (WebView vs нативный рендеринг).
