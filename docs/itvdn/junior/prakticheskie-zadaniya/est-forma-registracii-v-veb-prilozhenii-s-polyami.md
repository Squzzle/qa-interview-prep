# Есть форма регистрации в веб-приложении с полями (first name, last name, username, password, repeat password) и кнопкой Register. Какие проверки нужно провести?

**Коротко:** нужно проверить обязательность и валидацию каждого поля по отдельности, совпадение password/repeat password, граничные значения длины, поведение кнопки Register, а также security-аспекты (XSS, SQL-инъекции, надёжность пароля) и UX (сообщения об ошибках, маскирование пароля).

## Развёрнутый ответ

Проверки для формы регистрации логично разбить по полям и по сквозным (кросс-полевым) сценариям.

**По полю First name / Last name:**
- обязательность поля (что произойдёт при пустом значении);
- допустимые символы (буквы, дефис, апостроф для составных имён) и реакция на цифры, спецсимволы, эмодзи;
- граничные значения длины (минимальная — 1-2 символа, максимальная — по спецификации, обычно 50-255);
- обрезание пробелов по краям (trim);
- поддержка Unicode (кириллица, иероглифы, диакритика).

**По полю Username:**
- обязательность;
- уникальность — попытка зарегистрировать уже существующий username;
- допустимые символы (буквы, цифры, подчёркивание, точка) и запрет спецсимволов/пробелов;
- граничные значения длины (мин/макс);
- регистр-независимость или чувствительность к регистру при проверке уникальности;
- реакция на SQL-инъекции и XSS-payload'ы.

**По полю Password и Repeat password:**
- обязательность обоих полей;
- совпадение password и repeat password (позитивный и негативный кейс);
- требования к сложности пароля (минимальная длина, наличие цифр/спецсимволов/заглавных букв — если заявлено в требованиях);
- маскирование ввода (точки/звёздочки) и наличие переключателя "показать пароль";
- реакция на слишком короткий/слишком длинный пароль;
- проверка, что пароль не хранится и не логируется в открытом виде (это скорее нефункциональная/security проверка, но её стоит упомянуть).

**Кнопка Register:**
- состояние кнопки (disabled, пока не заполнены обязательные поля, либо активна всегда с валидацией по клику);
- повторное нажатие (защита от двойной отправки формы, дублирования аккаунта);
- поведение при потере соединения с сервером во время отправки.

**Сквозные сценарии:**
- полностью пустая форма — отправка должна быть заблокирована с показом всех необходимых ошибок;
- частично заполненная форма;
- успешная регистрация с валидными данными — редирект/сообщение об успехе, письмо подтверждения (если есть);
- проверка, что после успешной регистрации нельзя зарегистрироваться повторно с тем же username;
- локализация и корректность текстов ошибок.

## Пример / когда применяется

Мини-таблица тест-кейсов:

| Поле | Негативный сценарий | Ожидаемый результат |
|---|---|---|
| Username | Уже занят | Ошибка "Username уже используется" |
| Password/Repeat | Не совпадают | Ошибка "Пароли не совпадают" |
| First name | Пустое | Ошибка "Обязательное поле" |
| Password | Короче минимума (например, 3 символа) | Ошибка "Минимум 8 символов" |
| Username | Содержит `<script>alert(1)</script>` | Payload экранируется, аккаунт не создаётся с исполняемым скриптом |

Такой подход применяется на этапе тест-дизайна перед регрессионным тестированием форм регистрации в любом веб-приложении.

## На что смотрит интервьюер

Ожидается, что кандидат не ограничится общими словами "проверю обязательные поля", а разложит проверки по каждому полю отдельно и отдельно выделит кросс-полевую валидацию (совпадение паролей, уникальность username). Плюсом будет упоминание security-аспектов (инъекции, хранение пароля) и UX-деталей (disabled-кнопка, показать пароль). Минус — если кандидат забудет про граничные значения длины или про повторную отправку формы.
