# Как бы вы провели smoke-testing для приложения типа Telegram?

**Коротко:** smoke-тест для мессенджера проверяет только критичный "happy path" основных функций — установку/запуск, регистрацию/логин, отправку и получение сообщения, работу медиа и push-уведомлений — без глубокого покрытия граничных случаев, чтобы быстро понять, стабильна ли сборка для дальнейшего тестирования.

## Развёрнутый ответ

Smoke-testing ("дымовое" тестирование) — это поверхностная проверка ключевой функциональности сразу после получения новой сборки, цель которой — убедиться, что приложение в принципе работоспособно и не "рассыпается" на базовых сценариях, прежде чем тратить время на полноценное регрессионное тестирование. Основные принципы: быстрота выполнения (обычно 15-30 минут), покрытие только критичных путей (без edge-кейсов), бинарный результат (сборка принимается или отклоняется — go/no-go).

Для приложения типа Telegram набор smoke-кейсов логично сгруппировать по ключевым модулям:

1. **Установка и запуск** — приложение устанавливается и запускается без крашей на целевых ОС/устройствах.
2. **Аутентификация** — вход по номеру телефона, получение и ввод SMS/кода подтверждения, успешный вход в существующий аккаунт.
3. **Список чатов** — открывается главный экран, отображается список диалогов.
4. **Отправка сообщения** — открыть чат, отправить текстовое сообщение, сообщение доставляется собеседнику.
5. **Получение сообщения** — входящее сообщение отображается в реальном времени, приходит push-уведомление.
6. **Медиаконтент** — отправка/получение фото или файла в чате (базовая проверка, без всех форматов).
7. **Создание чата/группы** — создание нового диалога или группового чата с одним участником.
8. **Голосовой/видеозвонок** — инициация звонка (базовая проверка, что звонок устанавливается).
9. **Настройки профиля** — открытие экрана настроек, отображение имени и аватара пользователя.
10. **Синхронизация между устройствами** (если применимо) — сообщение, отправленное с одного устройства, появляется на другом.

Каждый пункт проверяется только по happy path — без дублирующихся негативных сценариев, без разных типов файлов, без тестирования edge-кейсов (это задача для функционального/регрессионного тестирования).

## Пример / когда применяется

Пример краткого smoke-чек-листа (10 минут на прогон):

| # | Проверка | Ожидаемый результат |
|---|---|---|
| 1 | Установка приложения | Приложение устанавливается без ошибок |
| 2 | Запуск приложения | Открывается без краша |
| 3 | Вход по номеру телефона | Успешная авторизация |
| 4 | Открытие списка чатов | Список отображается |
| 5 | Отправка текстового сообщения | Сообщение доставлено |
| 6 | Получение сообщения от другого пользователя | Push-уведомление получено |
| 7 | Отправка фото | Фото отправлено и отображается |
| 8 | Создание группового чата | Группа создана успешно |
| 9 | Инициация звонка | Звонок устанавливается |
| 10 | Открытие настроек профиля | Экран настроек открывается |

Если хотя бы один критичный пункт (например, отправка сообщения) не работает — сборка отклоняется и возвращается разработчикам без дальнейшего тестирования.

## На что смотрит интервьюер

Интервьюер оценивает, понимает ли кандидат разницу между smoke-, sanity- и полным регрессионным тестированием — частая ошибка новичков в том, что они начинают перечислять глубокие негативные сценарии (граничные значения длины сообщения, все форматы файлов и т.д.), превращая smoke-тест в полноценный regression-набор. Хороший ответ демонстрирует умение выделить именно критичные, "деньгообразующие" сценарии приложения и обосновать, почему выбраны именно они.
