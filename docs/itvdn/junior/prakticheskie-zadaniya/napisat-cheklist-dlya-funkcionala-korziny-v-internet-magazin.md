# Написать чеклист для функционала корзины в интернет-магазине.

**Коротко:** чеклист корзины должен покрывать добавление/удаление товаров, изменение количества, пересчёт цены и скидок, сохранение состояния между сессиями, лимиты и граничные значения, а также поведение при недоступности товара.

## Развёрнутый ответ

Чеклист — это перечень пунктов проверки без детальных шагов и ожидаемых результатов (в отличие от тест-кейса). Для корзины его удобно группировать по функциональным блокам:

1. **Добавление товара в корзину**
   - Добавление одного товара, нескольких разных товаров.
   - Повторное добавление уже добавленного товара (увеличивается количество, а не создаётся дубликат строки).
   - Добавление товара, которого нет в наличии / товара с ограничением "не более N штук".
   - Добавление максимально возможного количества (по остатку на складе) и попытка превысить его.
   - Добавление товара с вариациями (размер, цвет) — должны создаваться отдельные позиции.

2. **Изменение количества товара**
   - Увеличение/уменьшение количества через инкремент/декремент и через ручной ввод в поле.
   - Ввод 0 — должна ли позиция удаляться.
   - Ввод отрицательного числа, дробного числа, нечислового значения.
   - Ввод количества больше, чем есть на складе.
   - Пересчёт итоговой суммы после изменения количества.

3. **Удаление товара**
   - Удаление одной позиции, удаление всех позиций (пустая корзина).
   - Отмена удаления, если есть такая функция ("Undo").
   - Корректное отображение пустой корзины (сообщение, отключённая кнопка "Оформить заказ").

4. **Расчёт стоимости**
   - Корректность суммы по каждой позиции (цена × количество).
   - Корректность общей суммы корзины.
   - Применение скидок, промокодов, акций (в том числе истёкших/невалидных промокодов).
   - Учёт стоимости доставки, налогов.
   - Округление дробных сумм.

5. **Сохранение состояния**
   - Сохранение корзины при обновлении страницы.
   - Сохранение корзины для неавторизованного пользователя (через cookies/localStorage) и её объединение с корзиной после авторизации.
   - Синхронизация корзины между устройствами для авторизованного пользователя.
   - Поведение корзины после истечения сессии.

6. **Изменение состояния товара во время нахождения в корзине**
   - Товар закончился на складе, пока лежал в корзине.
   - Изменилась цена товара, пока он лежал в корзине.
   - Товар был снят с продажи/удалён из каталога.

7. **Кросс-функциональные проверки**
   - Отображение количества товаров в иконке корзины в header.
   - Переход из корзины к оформлению заказа и обратно с сохранением данных.
   - Поведение при работе в нескольких вкладках браузера одновременно.
   - Локализация (валюта, язык) для разных регионов.
   - Доступность (a11y): управление корзиной с клавиатуры, ARIA-атрибуты.
   - Производительность при большом количестве позиций в корзине (50+ товаров).

## Пример / когда применяется

Такой чеклист используется на этапе разработки тест-плана перед началом функционального тестирования e-commerce модуля, а также как основа для написания подробных тест-кейсов. Пример превращения пункта чеклиста в тест-кейс:

| Шаг | Действие | Ожидаемый результат |
|---|---|---|
| 1 | Открыть карточку товара с остатком 3 шт. | Карточка отображается |
| 2 | Установить количество 5 и нажать "В корзину" | Система не позволяет добавить больше остатка, показывает сообщение "Доступно только 3 шт." |
| 3 | Установить количество 3 и добавить в корзину | Товар добавлен в корзину в количестве 3 шт. |

## На что смотрит интервьюер

- Умение структурировать чеклист по логическим блокам, а не хаотичный список.
- Учёт граничных значений (0, максимум, дробные и отрицательные числа).
- Понимание разницы между позитивными и негативными сценариями.
- Учитывает ли кандидат неочевидные кейсы: смена цены/наличия товара "в фоне", гостевая корзина vs авторизованная, многопользовательские сценарии.
- Частая ошибка: кандидат перечисляет только happy path и забывает про edge cases, интеграцию с другими модулями (склад, промокоды, доставка) и кросс-браузерность/адаптивность.
