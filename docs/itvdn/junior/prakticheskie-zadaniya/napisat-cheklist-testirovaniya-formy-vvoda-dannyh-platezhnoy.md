# Написать чеклист тестирования формы ввода данных платежной карты.

**Коротко:** чеклист должен покрывать проверку каждого поля формы (номер карты, срок действия, CVV, имя владельца), их взаимосвязь, безопасность (маскирование, PCI DSS), UX-поведение и обработку ошибок платёжного шлюза.

## Развёрнутый ответ

Форма ввода карты обычно состоит из полей: **номер карты**, **срок действия (MM/YY)**, **CVV/CVC**, **имя держателя карты**. Чеклист логично строить по полям, затем — по кросс-функциональным и security-аспектам.

1. **Поле "Номер карты"**
   - Принимает только цифры, длина 13–19 символов (в зависимости от платёжной системы).
   - Автоматическое определение платёжной системы (Visa, MasterCard, Mir и т. д.) по BIN и отображение соответствующей иконки.
   - Проверка контрольной суммы по алгоритму Луна (Luhn).
   - Автоформатирование по группам цифр (например, 4-4-4-4).
   - Вставка номера через copy-paste, в том числе с пробелами/дефисами.
   - Ввод букв, спецсимволов, слишком короткого/длинного номера.
   - Маскирование номера при отображении (кроме последних 4 цифр) после ввода/на re-render.

2. **Поле "Срок действия"**
   - Формат MM/YY или отдельные поля месяц/год.
   - Отклонение просроченной даты (месяц/год в прошлом).
   - Граничные значения: текущий месяц/год (валидно), месяц 13, месяц 00.
   - Отклонение нечисловых символов.

3. **Поле "CVV/CVC"**
   - Длина 3 цифры (4 для Amex).
   - Поле маскируется при вводе (точки/звёздочки).
   - Не должно сохраняться после отправки формы или при возврате на страницу (требование PCI DSS).
   - Отклонение букв и спецсимволов.

4. **Поле "Имя держателя карты"**
   - Допустимые символы (обычно латиница, пробел, дефис).
   - Ограничение длины.
   - Проверка пустого значения.
   - Транслитерация/локализация (разрешить ли кириллицу).

5. **Взаимодействие полей и отправка формы**
   - Кнопка "Оплатить" неактивна, пока не заполнены все обязательные поля.
   - Клиентская валидация всех полей перед отправкой на сервер.
   - Серверная валидация (нельзя полагаться только на клиентскую).
   - Порядок табуляции (Tab) между полями логичен.
   - Автопереход фокуса в следующее поле при заполнении текущего (если реализовано).

6. **Обработка ответа платёжного шлюза**
   - Успешная оплата — корректное сообщение и переход к следующему шагу.
   - Отклонение банком (недостаточно средств, карта заблокирована) — понятное сообщение об ошибке.
   - Таймаут/недоступность платёжного шлюза — обработка без потери данных заказа и без двойного списания.
   - Повторная отправка формы (двойной клик) не должна вызывать двойное списание.
   - 3-D Secure / SMS-подтверждение — переход и возврат корректны.

7. **Безопасность и соответствие требованиям**
   - Передача данных только по HTTPS.
   - Данные карты не сохраняются в логах, localStorage, истории браузера, autocomplete отключён для CVV.
   - Соответствие требованиям PCI DSS (данные карты не хранятся на стороне клиента дольше необходимого).
   - Защита от автозаполнения чужими сохранёнными картами.

8. **UX / доступность**
   - Понятные сообщения об ошибках рядом с конкретным полем.
   - Работа с клавиатуры, ARIA-атрибуты для скринридеров.
   - Корректное отображение на мобильных устройствах (numeric keypad для номера карты).
   - Локализация валюты и суммы.

## Пример / когда применяется

Такой чеклист применяется при тестировании формы оплаты в интернет-магазине перед релизом функционала checkout, а также как базис для чек-листов при аудите безопасности платёжных форм. Пример перевода пункта в тест-кейс:

| Шаг | Действие | Ожидаемый результат |
|---|---|---|
| 1 | Ввести номер карты, не проходящий проверку Луна (например, `4111 1111 1111 1112`) | Поле подсвечивается ошибкой, кнопка "Оплатить" неактивна |
| 2 | Ввести корректный номер `4111 1111 1111 1111` | Ошибка исчезает, определяется иконка Visa |

## На что смотрит интервьюер

- Знание алгоритма Луна и умение объяснить, зачем он нужен.
- Понимание требований PCI DSS хотя бы на базовом уровне (CVV не хранится, маскирование номера).
- Учёт серверной валидации в дополнение к клиентской — частая ошибка junior-специалистов полагаться только на фронтенд.
- Обработка нестандартных сценариев: таймаут шлюза, повторный клик по "Оплатить", 3-D Secure.
- Понимание, что нельзя тестировать реальными картами в проде — используются тестовые карты песочницы платёжной системы.
