# Определите необходимое количество функциональных тест-кейсов, чтобы проверить Log in форму.

**Коротко:** для типовой формы логина (поля login/username, password, кнопка Login, ссылка "забыли пароль", чекбокс "запомнить меня") достаточным минимумом считается около 12-18 функциональных тест-кейсов, покрывающих позитивные, негативные и граничные сценарии.

## Развёрнутый ответ

Количество тест-кейсов зависит от сложности формы, но при декомпозиции по классам эквивалентности и граничным значениям для стандартной формы логина обычно выделяют следующие группы проверок:

1. **Позитивные сценарии** — вход с валидным логином и паролем; вход с валидными данными и включённым "запомнить меня"; вход по Enter вместо клика по кнопке.
2. **Негативные сценарии по полю логина** — пустое поле логина; несуществующий логин; логин с пробелами в начале/конце; логин с недопустимыми символами (если есть ограничения); превышение максимальной длины.
3. **Негативные сценарии по полю пароля** — пустое поле пароля; неверный пароль; пароль с учётом регистра (case sensitivity); превышение максимальной длины пароля.
4. **Оба поля пустые** — попытка входа без заполнения формы вообще.
5. **UI/UX проверки** — маскирование пароля (точки/звёздочки), наличие иконки "показать пароль", placeholder-тексты, состояние кнопки Login (disabled/enabled в зависимости от заполненности полей).
6. **Безопасность** — блокировка аккаунта после N неудачных попыток (brute-force protection), отсутствие информации о том, что именно неверно — логин или пароль (чтобы не давать подсказку атакующему), проверка SQL-инъекций и XSS в полях ввода.
7. **Функциональные ссылки/элементы** — переход по ссылке "Забыли пароль", переход на страницу регистрации, работа чекбокса "запомнить меня" (проверка, что сессия сохраняется после закрытия браузера).
8. **Кросс-браузерность и адаптивность** — отображение формы в разных браузерах и на разных разрешениях экрана (обычно выносится в отдельный чек-лист, а не считается отдельным функциональным кейсом).

Итоговое число зависит от глубины декомпозиции: если считать "по одному кейсу на комбинацию условий", получится порядка 15-20 кейсов; если группировать похожие негативные проверки в один параметризованный кейс, число может сократиться до 8-10. Важно не столько точное число, сколько демонстрация системного подхода к покрытию.

## Пример / когда применяется

Пример краткого чек-листа (12 кейсов) для формы логина:

| # | Тест-кейс | Ожидаемый результат |
|---|---|---|
| 1 | Валидный логин + валидный пароль | Успешный вход, редирект на главную |
| 2 | Валидный логин + неверный пароль | Сообщение об ошибке, вход запрещён |
| 3 | Несуществующий логин + любой пароль | Сообщение об ошибке |
| 4 | Пустой логин + валидный пароль | Валидационная ошибка "поле обязательно" |
| 5 | Валидный логин + пустой пароль | Валидационная ошибка "поле обязательно" |
| 6 | Оба поля пустые | Кнопка Login неактивна или ошибка при клике |
| 7 | Логин с пробелами по краям | Пробелы обрезаются (trim), вход успешен |
| 8 | Пароль чувствителен к регистру | Вход с неверным регистром пароля отклоняется |
| 9 | Маскирование пароля | Символы отображаются как точки/звёздочки |
| 10 | "Запомнить меня" включён | Сессия сохраняется после перезапуска браузера |
| 11 | Многократный неверный ввод пароля | Аккаунт блокируется / показывается CAPTCHA |
| 12 | Ввод SQL-инъекции в поле логина | Приложение не падает, вход отклонён |

## На что смотрит интервьюер

Интервьюер оценивает не точное число (правильного ответа "15" или "20" не существует), а умение декомпозировать функциональность на классы эквивалентности, граничные значения и негативные сценарии. Частая ошибка кандидата — назвать только 2-3 очевидных позитивных кейса и забыть про security-проверки (brute-force, инъекции), UI-детали (маскирование пароля) и edge-кейсы (пробелы, регистр). Хороший ответ демонстрирует структурированное мышление: сначала группы кейсов, потом конкретные примеры.
