# Поле username должно быть обязательным, но оно не является обязательным. Приведите пример баг-репорта, созданного для этой ошибки.

**Коротко:** баг-репорт должен содержать заголовок, окружение, шаги воспроизведения, фактический и ожидаемый результат, severity/priority и приложения (скриншот, лог), сформулированные так, чтобы разработчик мог однозначно воспроизвести и исправить дефект.

## Развёрнутый ответ

Качественный баг-репорт строится по стандартной структуре, независимо от используемой баг-трекинговой системы (Jira, Azure DevOps, TestRail и т.д.):

- **ID** — присваивается системой автоматически.
- **Title (заголовок)** — короткое, но информативное описание сути проблемы, желательно в формате "что сломано + при каком условии".
- **Environment (окружение)** — версия приложения, ОС, браузер, устройство, тестовый стенд (staging/prod).
- **Preconditions (предусловия)** — состояние системы перед началом теста (например, пользователь на странице регистрации).
- **Steps to reproduce (шаги воспроизведения)** — пронумерованные, минимальные и достаточные шаги.
- **Actual result (фактический результат)** — что происходит на самом деле.
- **Expected result (ожидаемый результат)** — что должно происходить согласно требованиям/спецификации.
- **Severity** — насколько сильно баг влияет на систему технически (Critical/Major/Minor/Trivial).
- **Priority** — насколько срочно нужно исправить (High/Medium/Low).
- **Attachments** — скриншот, видео, лог консоли/сервера.

Для конкретного случая "поле username должно быть обязательным, но не является таковым" severity обычно **Major** (нарушение бизнес-логики валидации данных, может привести к появлению записей с пустым username в базе), а priority зависит от того, насколько это критично для релиза — часто **Medium-High**, так как напрямую влияет на целостность данных.

## Пример / когда применяется

Пример оформленного баг-репорта:

```
Title: Форма регистрации позволяет создать аккаунт с пустым полем Username

Environment: Chrome 126, Windows 11, staging.example.com, build 2.4.1

Preconditions: Пользователь открыл страницу регистрации (/register)

Steps to reproduce:
1. Открыть страницу /register
2. Заполнить поля First name, Last name, Password, Repeat password валидными значениями
3. Оставить поле Username пустым
4. Нажать кнопку Register

Actual result: Форма отправляется успешно, аккаунт создаётся с пустым username,
пользователь перенаправляется на главную страницу

Expected result: Согласно требованиям (см. п. 3.2 функциональной спецификации),
поле Username является обязательным. Форма должна показать валидационную ошибку
"Username является обязательным полем" и не отправлять запрос на сервер

Severity: Major
Priority: High

Attachments: screenshot_empty_username.png, network_log.har
```

## На что смотрит интервьюер

Интервьюер проверяет, знает ли кандидат стандартную структуру баг-репорта и умеет ли чётко разделять "фактический" и "ожидаемый" результат — это самая частая ошибка новичков (смешивание описания шагов с ожидаемым результатом). Также важно, чтобы кандидат корректно расставил severity/priority и обосновал их, а не поставил наугад. Красный флаг — если кандидат не упоминает окружение или ссылку на требования/спецификацию, из-за чего баг-репорт становится субъективным мнением, а не объективным фактом.
