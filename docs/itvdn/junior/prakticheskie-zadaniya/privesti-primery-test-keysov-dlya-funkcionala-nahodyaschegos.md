# Привести примеры тест-кейсов для функционала, находящегося на нескольких страницах проекта (например, поле поиска).

**Коротко:** для сквозного функционала важно проверять не только работу на каждой отдельной странице, а согласованность поведения, состояния и результатов между всеми страницами, где он присутствует.

## Развёрнутый ответ

Функционал вроде поля поиска обычно вынесен в общий header/layout и присутствует на всех страницах сайта. Ключевая идея тестирования такого сквозного элемента — проверить не только его работу локально, но и:

- **Консистентность отображения**: поле поиска выглядит и ведёт себя одинаково на всех страницах (главная, каталог, страница товара, корзина, профиль).
- **Консистентность поведения**: одинаковый алгоритм поиска, одинаковая обработка ошибок независимо от того, с какой страницы инициирован поиск.
- **Сохранение состояния при навigации**: введённый, но не отправленный текст в поле поиска — должен ли он сохраняться при переходе на другую страницу (обычно нет, если не сохранено намеренно) или пропадать.
- **Переход между страницами**: после поиска пользователь попадает на страницу результатов — корректна ли работа кнопки "Назад" браузера (возврат к предыдущей странице с сохранённым состоянием поиска или без).
- **Синхронизация с другими компонентами**: например, поиск по каталогу должен учитывать применённые фильтры/сортировку, если он вызван со страницы каталога.

## Пример / когда применяется

Примеры тест-кейсов для поля поиска, доступного на нескольких страницах:

| № | Тест-кейс | Шаги | Ожидаемый результат |
|---|---|---|---|
| 1 | Поиск с главной страницы | Открыть главную страницу, ввести запрос "телефон", нажать Enter | Открывается страница результатов поиска с релевантными товарами |
| 2 | Поиск со страницы каталога | Открыть каталог "Электроника", ввести запрос "наушники", нажать Enter | Результаты поиска отображаются, при этом категория "Электроника" учитывается как контекст (если предусмотрено) или сбрасывается на общий поиск (в зависимости от требований) |
| 3 | Поиск со страницы товара | Открыть карточку товара, ввести другой запрос в поле поиска, нажать Enter | Переход на страницу результатов без потери данных корзины/сессии |
| 4 | Идентичность результатов | Выполнить одинаковый запрос "ноутбук" с главной и из каталога | Результаты поиска идентичны при одинаковых условиях (без активных фильтров каталога) |
| 5 | Сохранение состояния поля | Ввести текст в поле поиска на главной, не отправляя, перейти на другую страницу | Поле поиска на новой странице пустое (если не предусмотрено сохранение черновика) |
| 6 | Работа с историей браузера | Выполнить поиск, перейти на страницу результата товара, нажать "Назад" | Отображается страница результатов поиска с тем же запросом и позицией скролла |
| 7 | Поиск на разных устройствах/раскладках | Проверить работу поля поиска в мобильной версии на всех тех же страницах | Поведение и результаты идентичны десктопной версии с поправкой на UI |
| 8 | Очистка поля поиска (крестик) | На разных страницах нажать иконку очистки поля | Поле очищается одинаково на всех страницах, фокус остаётся в поле |
| 9 | Автодополнение/подсказки | Ввести частичный запрос на разных страницах | Список подсказок формируется одинаковым алгоритмом независимо от страницы |

## На что смотрит интервьюер

- Понимает ли кандидат, что тестирование сквозного функционала — это не "повторить один и тот же тест-кейс N раз для каждой страницы", а именно проверка консистентности между страницами.
- Учёт сохранения/потери состояния при переходах, работы истории браузера (back/forward).
- Внимание к тому, что результаты и поведение должны быть идентичны независимо от точки входа, если не заявлено обратное (контекстный поиск).
- Частая ошибка: кандидат ограничивается только "поиск работает на странице А" и не поднимает вопрос синхронизации между страницами — это как раз то, что проверяет данный вопрос.
