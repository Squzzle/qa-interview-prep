# Что такое Black/Grey/White Box Testing?

**Коротко:** Это классификация методов тестирования по уровню знания внутреннего устройства системы: Black Box — тестирование без знания кода (только через интерфейс), White Box — с полным знанием кода и структуры (тестируется сама логика реализации), Grey Box — промежуточный вариант с частичным знанием внутреннего устройства.

## Развёрнутый ответ

**Black Box Testing (тестирование чёрного ящика)** — тестировщик не имеет доступа к исходному коду и не знает внутреннюю реализацию системы. Тестирование строится исключительно на основе требований, спецификаций и ожидаемого поведения "снаружи": вход → выход. Основные техники: эквивалентное разбиение (equivalence partitioning), анализ граничных значений (boundary value analysis), таблицы решений, диаграммы причинно-следственных связей. Этим методом чаще всего занимаются QA-инженеры/тестировщики без глубоких знаний программирования.

**White Box Testing (тестирование белого ящика, structural/glass box testing)** — тестировщик имеет полный доступ к исходному коду и строит тесты на основе внутренней структуры программы: путей выполнения, ветвлений, циклов. Техники: покрытие операторов (statement coverage), покрытие ветвей (branch coverage), покрытие условий (condition coverage), покрытие путей (path coverage). Обычно выполняется разработчиками или тестировщиками с навыками программирования (unit-тесты, code review, статический анализ).

**Grey Box Testing (тестирование серого ящика)** — гибридный подход: тестировщик имеет частичное знание внутреннего устройства (например, знает структуру базы данных, архитектуру API, алгоритмы), но тестирует систему в основном через внешний интерфейс, как в black box. Позволяет создавать более осмысленные тест-кейсы, зная, например, какие граничные условия заложены в коде, но не тестируя код напрямую построчно.

Сравнение по критериям:

| Критерий | Black Box | Grey Box | White Box |
|---|---|---|---|
| Знание кода | Нет | Частичное | Полное |
| Кто выполняет | QA | QA / QA-automation | Разработчик / SDET |
| Основа тестов | Требования | Требования + архитектура | Исходный код |
| Примеры техник | Equivalence partitioning, BVA | Тестирование API + знание схемы БД | Unit-тесты, code coverage |

## Пример / когда применяется

**Black Box**: тестировщик проверяет форму регистрации, вводя валидные/невалидные email, не зная, как реализована валидация внутри — только по документации и ожидаемому поведению.

**White Box**: разработчик пишет unit-тест для метода `calculateDiscount()`, специально подбирая входные данные так, чтобы покрыть все ветки `if/else` внутри метода, включая редкие edge-case ветки.

**Grey Box**: тестировщик знает, что скидка на сайте хранится в БД и пересчитывается через триггер, поэтому целенаправленно создаёт тестовые данные в БД для проверки граничных случаев через UI, не читая сам код триггера.

## На что смотрит интервьюер

- Умение чётко разграничить, кто обычно выполняет каждый вид (Black Box — QA, White Box — разработчики, Grey Box — часто QA automation).
- Знание конкретных техник для каждого вида, а не только определений.
- Может спросить про частный случай Grey Box — тестирование API при известной схеме БД.
- Частая ошибка — путают White Box Testing с Unit Testing (White Box — это подход/методология, Unit Testing — уровень тестирования; они пересекаются, но не тождественны).
