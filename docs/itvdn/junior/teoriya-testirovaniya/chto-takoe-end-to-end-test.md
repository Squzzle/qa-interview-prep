# Что такое End-to-End тест?

**Коротко:** End-to-End (E2E) тестирование проверяет работу всей системы целиком от начала до конца, эмулируя реальный сценарий пользователя через все интегрированные компоненты (UI, backend, база данных, внешние сервисы), в условиях, максимально приближенных к production.

## Развёрнутый ответ

E2E-тестирование находится на верхнем уровне пирамиды тестирования (Test Pyramid) — после unit-тестов и интеграционных тестов. Его цель — убедиться, что вся цепочка компонентов системы работает согласованно как единое целое с точки зрения конечного пользователя, включая реальные (или максимально приближенные к реальным) внешние интеграции: базы данных, очереди сообщений, сторонние API, платёжные шлюзы.

Ключевые особенности:

- Тестируется полный бизнес-сценарий "от и до" (например, от регистрации пользователя до получения товара), а не отдельная функция или связка двух модулей.
- Выполняется на стенде, максимально приближенном к production (staging), часто через UI, эмулируя реальные действия пользователя.
- Обычно медленнее и дороже в поддержке, чем unit- и integration-тесты (согласно пирамиде тестирования их должно быть меньше всего).
- Часто автоматизируется с помощью инструментов вроде Selenium, Playwright, Cypress для веб; Appium для мобильных приложений.
- Более хрупкие (flaky) тесты из-за большого числа зависимостей — падение любого звена цепочки (сеть, БД, сторонний сервис) может привести к ложному провалу теста.

Отличие от Integration Testing: интеграционное тестирование проверяет взаимодействие между отдельными модулями/сервисами (например, "Корзина" ↔ "Оплата"), а E2E проверяет весь путь целиком через все слои системы, включая внешние интеграции и реальный UI.

## Пример / когда применяется

Пример сценария E2E-теста для интернет-магазина: пользователь открывает сайт → регистрируется → находит товар через поиск → добавляет в корзину → применяет промокод → оформляет заказ → вводит данные оплаты → получает подтверждение заказа на email → заказ появляется в личном кабинете и в admin-панели.

```javascript
// Пример E2E теста на Playwright
test('пользователь может оформить заказ', async ({ page }) => {
  await page.goto('https://shop.example.com');
  await page.click('text=Войти');
  await page.fill('#email', 'user@test.com');
  await page.fill('#password', 'Password123');
  await page.click('#login-submit');

  await page.fill('#search', 'ноутбук');
  await page.click('.product-card >> nth=0');
  await page.click('text=Добавить в корзину');
  await page.click('text=Оформить заказ');
  await page.fill('#promo-code', 'SUMMER20');
  await page.click('text=Подтвердить оплату');

  await expect(page.locator('.order-confirmation')).toContainText('Заказ оформлен');
});
```

Применяется для критичных бизнес-процессов (checkout, регистрация, оплата), где важно убедиться, что вся цепочка систем работает согласованно перед релизом.

## На что смотрит интервьюер

- Понимание места E2E в пирамиде тестирования — их должно быть немного, они дорогие и медленные, покрывают только критичные сквозные сценарии.
- Умение отличить E2E от Integration testing (весь путь целиком vs. взаимодействие пары модулей).
- Знание проблемы flaky-тестов в E2E и способов борьбы с ней (retry-механизмы, стабильные селекторы, изоляция тестовых данных, ожидания вместо sleep).
- Знание инструментов (Selenium, Playwright, Cypress, Appium).
- Follow-up: "Почему нельзя строить всю стратегию тестирования только на E2E-тестах?" (медленно, дорого, хрупко, сложно локализовать причину падения).
