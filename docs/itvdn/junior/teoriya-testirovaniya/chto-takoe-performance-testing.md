# Что такое Performance Testing?

**Коротко:** Performance Testing — это вид нефункционального тестирования, который проверяет скорость отклика, стабильность, масштабируемость и использование ресурсов системой при определённой нагрузке; включает подвиды Load, Stress, Spike, Endurance (Soak) и Scalability testing.

## Развёрнутый ответ

Performance Testing отвечает не на вопрос "правильно ли работает система", а на вопрос "насколько быстро и стабильно она работает" под разной нагрузкой. Основные метрики: время отклика (response time), пропускная способность (throughput, requests per second), использование CPU/памяти/сети, количество ошибок под нагрузкой (error rate), время до первого байта (TTFB).

Основные подвиды:

- **Load Testing (нагрузочное тестирование)** — проверка поведения системы при ожидаемой (планируемой) нагрузке, например 1000 одновременных пользователей, соответствующей реальным условиям эксплуатации.
- **Stress Testing (стресс-тестирование)** — постепенное увеличение нагрузки выше нормы до тех пор, пока система не начнёт деградировать или не "упадёт", чтобы найти предел прочности и точку отказа.
- **Spike Testing** — резкое кратковременное увеличение нагрузки (скачок) для проверки, как система реагирует на внезапный всплеск трафика (например, во время распродажи).
- **Endurance / Soak Testing** — тестирование при умеренной нагрузке в течение продолжительного времени (часы, дни), чтобы выявить утечки памяти, деградацию производительности со временем.
- **Scalability Testing** — проверка способности системы масштабироваться (горизонтально/вертикально) при росте нагрузки, сохраняя приемлемые показатели производительности.
- **Volume Testing** — проверка поведения системы при работе с большими объёмами данных (например, огромная БД), а не большим числом пользователей.

Performance Testing обычно выполняют с помощью специализированных инструментов: JMeter, Gatling, k6, LoadRunner, Locust. Тестирование строится по фазам: определение целевых показателей (SLA), подготовка тестового окружения близкого к production, написание сценариев нагрузки, выполнение, анализ узких мест (bottlenecks), формирование отчёта с рекомендациями по оптимизации.

## Пример / когда применяется

Пример: интернет-магазин ожидает Black Friday с трёхкратным увеличением трафика. Проводят Load Testing на ожидаемых 5000 пользователей, затем Stress Testing, постепенно увеличивая нагрузку до 15000+, чтобы найти точку, при которой время отклика превышает 3 секунды или начинают появляться ошибки 500. Дополнительно проводят Spike Testing, симулируя резкий скачок при старте акции.

```bash
# Пример простого сценария нагрузки на k6
k6 run --vus 500 --duration 10m load-test.js
```

## На что смотрит интервьюер

- Умение чётко разграничить Load / Stress / Spike / Soak / Scalability — частая ошибка путать Load и Stress testing.
- Знание конкретных метрик (response time, throughput, error rate), а не общих фраз "тестируем скорость".
- Знание инструментов (JMeter, k6, Gatling, LoadRunner).
- Follow-up вопрос: "Как определить целевые показатели производительности перед началом тестирования?" (SLA/NFR, бизнес-требования).
- Могут спросить про разницу между Performance Testing и Performance Engineering (проактивная оптимизация архитектуры vs. постфактум-проверка).
