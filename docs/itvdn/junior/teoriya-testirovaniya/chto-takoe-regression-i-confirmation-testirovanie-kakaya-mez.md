# Что такое Regression и Confirmation тестирование, какая между ними разница?

**Коротко:** Regression testing проверяет, что новые изменения не сломали ранее работавшую функциональность; Confirmation testing (он же retesting) проверяет, что конкретный ранее найденный дефект действительно исправлен. Это разные по цели виды тестирования, которые часто выполняются вместе после исправления бага.

## Развёрнутый ответ

**Regression Testing (регрессионное тестирование)** — это повторное выполнение уже существующих тестов (или их части) после внесения изменений в код (новая функциональность, исправление багов, рефакторинг, обновление зависимостей, изменение конфигурации), чтобы убедиться, что эти изменения не привнесли новых дефектов в ранее работавшую функциональность и не "сломали" существующее поведение системы. Регрессионное тестирование не проверяет само исправление — оно проверяет всё остальное вокруг него.

**Confirmation Testing / Retesting (подтверждающее / повторное тестирование)** — это целенаправленная проверка того, что конкретный дефект, зафиксированный ранее (например, в баг-трекере), действительно был исправлен и больше не воспроизводится по тем же шагам. Retesting выполняется по тем же шагам, что привели к обнаружению исходного дефекта.

**Ключевые различия:**

| Критерий | Confirmation testing (retesting) | Regression testing |
|---|---|---|
| Цель | Убедиться, что конкретный баг исправлен | Убедиться, что остальная функциональность не сломалась |
| Область охвата | Узкая — конкретный сценарий/дефект | Широкая — вся система или связанные модули |
| Основа тест-кейсов | Шаги воспроизведения исходного бага | Существующий набор регрессионных тестов/чек-листов |
| Может ли быть автоматизирован | Да, но часто выполняется вручную для проверки одного конкретного случая | Да, отлично подходит для автоматизации из-за повторяемости |
| Когда выполняется | После получения фикса от разработчика | После любых изменений в кодовой базе (не только фиксов) |

Важный нюанс: эти два вида тестирования обычно выполняются последовательно и дополняют друг друга. Сначала проводится confirmation testing — проверяется, что баг исправлен. Затем проводится regression testing — проверяется, что исправление не сломало ничего рядом (побочные эффекты фикса). Пропуск regression testing после фикса — частая причина появления новых багов, вызванных самим исправлением ("fix broke something else").

## Пример / когда применяется

Пример: в приложении был найден баг — при применении промокода со скидкой 100% итоговая сумма заказа становится отрицательной. Разработчик исправляет логику расчёта скидки.

- **Confirmation testing**: QA повторяет точные шаги из баг-репорта — применяет тот же промокод к той же корзине — и убеждается, что сумма теперь корректно равна нулю, а не отрицательна.
- **Regression testing**: QA дополнительно проверяет смежную функциональность — применение обычных промокодов (10%, 50%), расчёт стоимости доставки, применение нескольких промокодов подряд, оформление заказа без промокода — чтобы убедиться, что исправление логики скидки не затронуло эти сценарии.

## На что смотрит интервьюер

Интервьюер проверяет, понимает ли кандидат, что это не синонимы, а два дополняющих друг друга процесса с разной целью и охватом. Хороший ответ включает таблицу или явное сравнение "confirmation — проверяем конкретный фикс, regression — проверяем всё вокруг".

Частая ошибка — считать confirmation testing частью regression testing или наоборот, либо использовать термины взаимозаменяемо.
