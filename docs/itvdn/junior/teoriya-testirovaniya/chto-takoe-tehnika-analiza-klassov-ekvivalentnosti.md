# Что такое техника анализа классов эквивалентности?

**Коротко:** Equivalence Partitioning — это техника тест-дизайна, при которой множество входных (или выходных) данных разбивается на классы эквивалентности, внутри которых система, предположительно, ведёт себя одинаково, и из каждого класса для тестирования выбирается один репрезентативный элемент.

## Развёрнутый ответ

Идея техники основана на предположении, что если один представитель класса эквивалентности обработан системой корректно (или некорректно), то и все остальные значения этого класса будут обработаны так же — то есть тестировать каждое отдельное значение избыточно. Это позволяет существенно сократить количество тест-кейсов при сохранении разумного уровня покрытия.

Классы эквивалентности бывают:
- **Валидные (positive)** — данные, которые система должна принять и корректно обработать.
- **Невалидные (negative)** — данные, которые система должна отклонить с соответствующей ошибкой/сообщением.

Классы могут строиться на основе:
- диапазонов значений (числовые диапазоны, длина строки);
- множеств допустимых значений (перечисления, списки);
- булевых условий (да/нет, включено/выключено);
- форматов данных (email, номер телефона, дата).

Важные нюансы:
1. Для одного входного параметра может существовать несколько валидных и несколько невалидных классов одновременно.
2. Технику применяют не только к входным данным, но и к выходным результатам, и к внутренним данным (например, содержимому БД).
3. Часто эту технику комбинируют с анализом граничных значений (Boundary Value Analysis) — сначала строятся классы эквивалентности, а затем проверяются именно границы этих классов, так как ошибки статистически чаще возникают именно там.
4. Технику относят к black-box методам — тестировщику не нужен доступ к коду, только к спецификации/требованиям.

Ценность техники — в балансе между полнотой покрытия и трудозатратами: вместо перебора всех возможных значений (что физически невозможно для больших диапазонов) тестировщик получает минимальный, но статистически обоснованный набор тестов.

## Пример / когда применяется

Пример: поле "Возраст пользователя при регистрации" принимает целые числа от 18 до 100.

| Класс | Диапазон | Тип | Пример значения для теста |
|---|---|---|---|
| 1 | &lt; 18 | Невалидный | 10 |
| 2 | 18–100 | Валидный | 45 |
| 3 | > 100 | Невалидный | 150 |
| 4 | нечисловое значение | Невалидный | "abc" |
| 5 | пустое поле | Невалидный | "" |

Вместо проверки всех значений от 1 до 200 (200 тестов) достаточно 5 тестов — по одному представителю каждого класса, что даёт достаточное покрытие с минимальными трудозатратами.

Техника применяется практически везде, где есть входные поля с ограничениями: формы регистрации, фильтры поиска, параметры API-запросов, конфигурационные настройки.

## На что смотрит интервьюер

Интервьюер часто просит прямо на собеседовании построить классы эквивалентности для конкретного поля — важно правильно выделить и валидные, и невалидные классы, не забыть про граничные и особые случаи (пустое значение, null, нечисловой ввод).

Частые ошибки: путать классы эквивалентности с граничными значениями, забывать про невалидные классы, выделять избыточное число классов (например, делить диапазон 18–100 на несколько валидных классов без оснований).
