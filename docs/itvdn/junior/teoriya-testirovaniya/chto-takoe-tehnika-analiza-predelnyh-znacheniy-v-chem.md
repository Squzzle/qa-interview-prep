# Что такое техника анализа предельных значений? В чем ценность этой техники?

**Коротко:** Boundary Value Analysis (BVA) — техника тест-дизайна, при которой тестируются значения на границах классов эквивалентности и рядом с ними, поскольку статистика дефектов показывает, что ошибки чаще всего допускаются именно на границах диапазонов.

## Развёрнутый ответ

Анализ граничных значений — логическое продолжение эквивалентного разбиения (Equivalence Partitioning). Если EP говорит "достаточно проверить один представитель класса", то BVA уточняет: "этим представителем должна быть граница класса, а не случайное значение из середины", потому что именно на границах разработчики чаще всего допускают ошибки off-by-one (использование `<` вместо `<=`, `>` вместо `>=` и т.п.).

Для диапазона значений обычно проверяют:
- **минимальное допустимое значение (min)**;
- **значение чуть меньше минимума (min - 1)** — должно быть отклонено;
- **значение чуть больше минимума (min + 1)** — опционально, для дополнительной уверенности;
- **максимальное допустимое значение (max)**;
- **значение чуть больше максимума (max + 1)** — должно быть отклонено;
- **значение чуть меньше максимума (max - 1)** — опционально.

Существует два подхода:
- **2-точечный BVA (two-point)** — проверяются только сама граница и одно соседнее значение за её пределами (min, min-1, max, max+1);
- **3-точечный BVA (three-point)** — дополнительно проверяются значения непосредственно внутри диапазона у границы (min, min+1, max-1, max).

**Ценность техники:**
1. Она находит наиболее частый класс дефектов реальных программ — ошибки в условиях сравнения (`>` вместо `>=`, неверные константы циклов).
2. Она требует минимального числа тестов по сравнению с полным перебором, обеспечивая при этом высокую вероятность обнаружения дефекта.
3. Она применима не только к числовым диапазонам, но и к длине строк, размеру файлов, количеству элементов в списке, датам, размерам буфера памяти.
4. Она хорошо сочетается с эквивалентным разбиением — вместе эти две техники дают компактный, но качественный набор тестов практически для любого поля ввода.

## Пример / когда применяется

Пример: поле "Пароль" должно содержать от 8 до 20 символов.

| Значение | Длина | Ожидаемый результат |
|---|---|---|
| 7 символов | min − 1 | Ошибка валидации |
| 8 символов | min | Принято |
| 9 символов | min + 1 | Принято |
| 19 символов | max − 1 | Принято |
| 20 символов | max | Принято |
| 21 символ | max + 1 | Ошибка валидации |

Такой набор из 6 тестов с высокой вероятностью находит распространённую ошибку реализации, например, `if (length < 8)` вместо `if (length <= 8)`, приводящую к тому, что пароль из 8 символов ошибочно отклоняется.

Техника особенно активно применяется при тестировании форм с ограничением длины/количества, числовых полей (возраст, сумма платежа, количество товара в корзине), пагинации (первая/последняя страница), лимитов API (rate limits).

## На что смотрит интервьюер

Интервьюер обычно просит применить технику к конкретному примеру прямо на собеседовании и проверяет, помнит ли кандидат все ключевые точки (min-1, min, min+1, max-1, max, max+1), а не только min и max.

Частая ошибка — забывать про значения "чуть за пределами границы" (min-1, max+1), которые как раз и находят off-by-one ошибки, либо путать эту технику с эквивалентным разбиением.
