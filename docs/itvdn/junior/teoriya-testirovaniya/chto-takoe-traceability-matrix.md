# Что такое Traceability Matrix?

**Коротко:** Traceability Matrix (матрица трассируемости, RTM — Requirements Traceability Matrix) — это документ, который связывает требования с тест-кейсами (и часто с дефектами), обеспечивая полное покрытие требований тестами и возможность отследить, какое требование какими тестами проверено.

## Развёрнутый ответ

RTM решает главную задачу: убедиться, что каждое требование к продукту покрыто хотя бы одним тест-кейсом, и что нет "лишних" тест-кейсов, не привязанных ни к одному требованию. Это критично для проектов с формальной документацией (например, в регулируемых отраслях — медицина, финансы, авиация), где нужно доказать полноту тестового покрытия.

Обычно матрица представляет собой таблицу со столбцами:

- ID требования (Requirement ID)
- Описание требования
- ID связанного тест-кейса (или нескольких)
- Статус выполнения теста (Passed/Failed/Blocked/Not Run)
- ID связанного дефекта (если тест провалился)

Типы трассируемости:

- **Forward traceability** — от требования к тест-кейсам: проверяет, что каждое требование протестировано.
- **Backward traceability** — от тест-кейса к требованию: проверяет, что каждый тест-кейс действительно относится к какому-то требованию (нет избыточных/неактуальных тестов).
- **Bidirectional traceability** — сочетание обоих направлений, наиболее полный вариант.

RTM также может связывать требования с бизнес-целями/user story (для Agile-проектов, где вместо формальных SRS используются user story и acceptance criteria) и с дефектами, найденными при выполнении тестов — что позволяет оценить, какие требования наиболее проблемные.

## Пример / когда применяется

Пример таблицы RTM:

| Req ID | Требование | Test Case ID | Статус | Defect ID |
|---|---|---|---|---|
| REQ-001 | Пользователь может войти по email и паролю | TC-101, TC-102 | Passed | — |
| REQ-002 | Система блокирует аккаунт после 5 неверных попыток входа | TC-103 | Failed | BUG-45 |
| REQ-003 | Пароль должен содержать минимум 8 символов | TC-104 | Passed | — |

Применяется в проектах с чёткими требованиями (SRS, ТЗ) для аудита покрытия перед релизом, а также при регуляторных проверках (например, для медицинского ПО, где аудитор требует доказать, что 100% требований покрыты тестами).

## На что смотрит интервьюер

- Понимание разницы между forward и backward traceability.
- Знание, зачем нужна матрица трассируемости — не просто "документ ради документа", а инструмент контроля покрытия и оценки impact-анализа при изменении требований.
- Может спросить, как RTM используется при impact-анализе — если требование меняется, какие тест-кейсы нужно пересмотреть.
- В Agile-контексте — может спросить, как трассируемость реализуется без формального SRS (через связку user story ↔ acceptance criteria ↔ тест-кейсы в таск-трекере, например Jira + Xray/Zephyr).
