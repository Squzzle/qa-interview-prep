# Какая разница между чеклистом и тест-кейсами?

**Коротко:** тест-кейс — это детальный формализованный сценарий с шагами, входными данными и ожидаемым результатом, а чек-лист — укрупнённый список пунктов для проверки без пошаговой детализации; чек-лист быстрее в создании, тест-кейс — более воспроизводим и точен.

## Развёрнутый ответ

**Тест-кейс (Test Case)** — это формализованный документ, описывающий один конкретный сценарий проверки. Как правило, включает:

- уникальный идентификатор;
- название/цель;
- предусловия (preconditions);
- шаги выполнения (steps) — пронумерованный список действий;
- тестовые данные;
- ожидаемый результат для каждого шага или для сценария в целом;
- фактический результат (заполняется при выполнении);
- статус (Passed/Failed/Blocked);
- приоритет и связь с требованием (traceability).

**Чек-лист (Checklist)** — это более лёгкий, укрупнённый список пунктов ("что нужно проверить"), без детального описания шагов и без явного разделения на входные данные и ожидаемый результат. Чек-лист отвечает на вопрос "что проверить", а не "как именно это проверить шаг за шагом".

Ключевые различия:

| Критерий | Тест-кейс | Чек-лист |
|---|---|---|
| Уровень детализации | Высокий: шаги, данные, ожидаемый результат | Низкий: краткий пункт проверки |
| Воспроизводимость | Высокая — любой исполнитель получит одинаковый результат | Ниже — зависит от опыта и трактовки исполнителя |
| Скорость создания и поддержки | Медленнее, дороже поддерживать | Быстрее создавать и обновлять |
| Пригодность для автоматизации | Хорошо подходит как основа для автотестов | Плохо — не хватает конкретики |
| Использование | Регрессионные наборы, критичная функциональность, регулируемые проекты | Smoke-тестирование, exploratory-тестирование, быстрые проверки в agile |
| Гибкость при исследовательском тестировании | Низкая — жёстко фиксирует шаги | Высокая — оставляет пространство для импровизации тестировщика |

Оба артефакта решают одну задачу — систематизировать проверку функциональности, но на разных уровнях формализации. Выбор между ними — это компромисс между скоростью и точностью/воспроизводимостью.

## Пример / когда применяется

Тест-кейс для формы логина:

```
ID: TC-014
Название: Успешная авторизация с валидными данными
Предусловие: пользователь зарегистрирован, аккаунт активен
Шаги:
  1. Открыть страницу логина
  2. Ввести валидный email
  3. Ввести валидный пароль
  4. Нажать "Войти"
Ожидаемый результат: пользователь авторизован, произошёл редирект на dashboard
```

Чек-лист для той же формы:

```
- Логин с валидными данными
- Логин с невалидным паролем
- Логин с пустыми полями
- Восстановление пароля
- Блокировка после N неверных попыток
```

Чек-листы обычно применяются при exploratory-тестировании и smoke-проверках, где важна скорость и гибкость. Тест-кейсы используются для регрессионных наборов и критичной функциональности, где важна точная воспроизводимость и возможность передать выполнение другому человеку без потери контекста.

## На что смотрит интервьюер

- Понимание, что выбор формата — это компромисс, а не универсально "правильный" вариант.
- Знание типовой структуры тест-кейса (шаги, ожидаемый результат, предусловия).
- Способность привести пример, когда чек-лист предпочтительнее тест-кейса (и наоборот).
- Красный флаг: утверждение, что чек-лист и тест-кейс — это одно и то же, или что один из форматов всегда лучше другого.
