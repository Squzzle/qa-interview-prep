# Какие есть атрибуты баг-репорта? Какие основные поля для заполнения?

**Коротко:** качественный баг-репорт должен содержать заголовок, шаги воспроизведения, ожидаемый и фактический результат, приоритет, серьёзность, окружение, вложения и статус — то есть всю информацию, необходимую разработчику для быстрого воспроизведения и исправления дефекта без дополнительных уточняющих вопросов.

## Развёрнутый ответ

Баг-репорт (defect report) — это формализованный документ, фиксирующий обнаруженный дефект. Основные атрибуты/поля:

1. **ID (идентификатор)** — уникальный номер, присваиваемый системой баг-трекинга (Jira, Bugzilla, TestRail и т.д.).
2. **Title / Summary (заголовок)** — краткое, но информативное описание сути проблемы, по которому можно сразу понять контекст (например, «Ошибка 500 при оформлении заказа с промокодом на сумму больше 10000 руб.»).
3. **Description (описание)** — подробное описание проблемы.
4. **Steps to Reproduce (шаги воспроизведения)** — пронумерованная последовательность действий, точно приводящая к дефекту.
5. **Expected Result (ожидаемый результат)** — как система должна была себя вести согласно требованиям.
6. **Actual Result (фактический результат)** — как система повела себя на самом деле.
7. **Severity (серьёзность)** — техническая оценка влияния дефекта на систему (Blocker, Critical, Major, Minor, Trivial).
8. **Priority (приоритет)** — бизнес-оценка срочности исправления (High, Medium, Low), устанавливается обычно менеджером/владельцем продукта.
9. **Environment (окружение)** — версия ОС, браузера, устройства, версия сборки/билда, URL стенда, где воспроизведён баг.
10. **Attachments (вложения)** — скриншоты, видео, логи, дампы БД, HAR-файлы сетевых запросов.
11. **Status (статус)** — текущее состояние дефекта в жизненном цикле (New, Open, In Progress, Reopened, Resolved, Closed, Rejected).
12. **Assignee (исполнитель)** — кто назначен на исправление.
13. **Reporter (автор)** — кто обнаружил и завёл баг.
14. **Reproducibility (воспроизводимость)** — всегда/иногда/один раз (Always/Sometimes/Once).
15. **Component/Module (компонент)** — какой модуль системы затронут.
16. **Labels/Tags** — дополнительная классификация (например, regression, UI, backend).

Хороший баг-репорт должен соответствовать принципу «одна проблема — один репорт» и быть написан так, чтобы разработчик мог воспроизвести дефект без обращения к автору репорта.

## Пример / когда применяется

Пример заполненного баг-репорта:

```text
ID: BUG-4521
Title: 500 Internal Server Error при применении промокода к заказу > 10000 руб.
Environment: Prod-подобный stage, Chrome 126, Windows 11
Priority: High
Severity: Critical
Steps to Reproduce:
  1. Добавить в корзину товар на сумму 12000 руб.
  2. Перейти к оформлению заказа.
  3. Ввести промокод "SUMMER2026".
  4. Нажать "Применить".
Expected Result: Промокод применяется, сумма скидки отображается корректно.
Actual Result: Сервер возвращает 500 Internal Server Error, страница не загружается.
Attachments: screenshot_500.png, network.har
Reproducibility: Always
Status: New
```

Атрибуты используются на всех этапах: от заведения бага QA-инженером до анализа метрик качества релиза (например, количество Critical-багов, среднее время исправления по приоритетам).

## На что смотрит интервьюер

- Полнота списка: упомянуты ли Severity и Priority отдельно (частая путаница на younger уровнях).
- Понимание, что Steps to Reproduce должны быть воспроизводимы «с нуля», без домыслов разработчика.
- Знание жизненного цикла статусов бага (New → Open → Fixed → Retest → Closed/Reopened).
- Красный флаг: перечисление только 2-3 полей (например, «название и описание») без структурированного подхода.
