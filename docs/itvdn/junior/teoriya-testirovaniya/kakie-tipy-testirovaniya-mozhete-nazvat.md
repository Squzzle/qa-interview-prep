# Какие типы тестирования можете назвать?

**Коротко:** Типы тестирования принято делить по объекту проверки (функциональное и нефункциональное), по доступу к коду (белый/чёрный/серый ящик), по степени автоматизации (ручное и автоматизированное) и по отношению к изменениям в коде (регрессионное, дымовое, санитарное).

## Развёрнутый ответ

Классификация типов тестирования многомерна, поэтому на собеседовании стоит структурировать ответ по нескольким осям.

**1. По объекту тестирования:**
- **Функциональное тестирование** — проверка того, что система делает то, что должна, согласно требованиям (бизнес-логика, пользовательские сценарии, API).
- **Нефункциональное тестирование** — проверка характеристик системы, не связанных напрямую с функциями: производительность (performance/load/stress testing), удобство использования (usability), безопасность (security), совместимость (compatibility), надёжность (reliability), локализация (localization).

**2. По доступу к внутренней структуре кода:**
- **Black box (чёрный ящик)** — тестировщик не видит код, работает только с входами/выходами и требованиями.
- **White box (белый ящик)** — тестировщик имеет доступ к коду и проверяет внутреннюю логику (используется, например, при unit-тестировании).
- **Grey box (серый ящик)** — частичное знание внутренней структуры, например знание архитектуры БД при тестировании через UI.

**3. По степени автоматизации:**
- **Ручное тестирование (Manual testing)** — выполняется человеком без скриптов.
- **Автоматизированное тестирование (Automation testing)** — выполняется с помощью скриптов и фреймворков (Selenium, Playwright, Cypress, JUnit и т.д.).

**4. По отношению к изменениям в коде и целям проверки:**
- **Smoke testing (дымовое)** — быстрая проверка базовой работоспособности сборки перед более глубоким тестированием.
- **Sanity testing (санитарное)** — узконаправленная проверка конкретной функциональности после небольшого изменения.
- **Regression testing (регрессионное)** — проверка, что новые изменения не сломали существующую функциональность.
- **Retesting (повторное тестирование)** — проверка, что конкретный дефект исправлен.

**5. Другие важные виды:**
- **Exploratory testing (исследовательское)** — одновременное проектирование и выполнение тестов без заранее написанных сценариев, основанное на опыте тестировщика.
- **Acceptance testing (приёмочное)** — проверка готовности продукта к принятию заказчиком/пользователем (UAT).
- **A/B testing** — сравнение двух вариантов фичи на реальных пользователях.
- **Ad-hoc testing** — неформальное тестирование без подготовки, "на удачу".

## Пример / когда применяется

Пример: перед релизом мобильного приложения банка команда сначала проводит smoke testing (запускается ли приложение, открывается ли главный экран), затем функциональное тестирование новой фичи (перевод по номеру телефона), затем регрессионное тестирование существующих модулей (история операций, вход в приложение), после этого — нагрузочное тестирование (performance testing) на пиковую нагрузку в дни зарплаты, и в конце — security testing перед выходом в прод, так как приложение работает с деньгами.

## На что смотрит интервьюер

Интервьюер оценивает, умеет ли кандидат структурировать классификацию (не просто вывалить список из 15 терминов без системы), а также правильность определений — особенно частая ошибка: путать smoke и sanity testing, или считать regression testing синонимом retesting.
