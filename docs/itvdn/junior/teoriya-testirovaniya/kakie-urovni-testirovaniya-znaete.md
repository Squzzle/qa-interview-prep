# Какие уровни тестирования знаете?

**Коротко:** Классические уровни тестирования — модульное (unit), интеграционное (integration), системное (system) и приёмочное (acceptance) тестирование; они образуют пирамиду от узкой, быстрой проверки отдельных компонентов до широкой проверки всей системы с точки зрения бизнеса.

## Развёрнутый ответ

Уровни тестирования (test levels) отражают, на каком масштабе системы проводится проверка, и обычно соответствуют этапам разработки (V-модель).

1. **Unit testing (модульное тестирование).** Проверка наименьших изолированных частей кода — функций, методов, классов. Обычно выполняется разработчиками с использованием фреймворков (JUnit, NUnit, Jest, pytest). Использует технику белого ящика, зависимости изолируются с помощью моков/стабов. Самый дешёвый и быстрый уровень для нахождения дефектов.

2. **Integration testing (интеграционное тестирование).** Проверка взаимодействия между несколькими модулями/компонентами или между системой и внешними сервисами (БД, API, очереди сообщений). Существуют разные стратегии: big bang (интеграция всех модулей сразу), инкрементальная (top-down, bottom-up), контрактное тестирование (contract testing) для микросервисов.

3. **System testing (системное тестирование).** Проверка полностью собранной системы целиком на соответствие функциональным и нефункциональным требованиям. Проводится в окружении, максимально близком к продакшену. Здесь тестируются end-to-end сценарии, производительность, безопасность, совместимость.

4. **Acceptance testing (приёмочное тестирование).** Финальная проверка того, что система готова к передаче/использованию заказчиком или конечным пользователем. Подвиды:
   - **UAT (User Acceptance Testing)** — проверка бизнес-пользователями, что продукт решает их задачи;
   - **Operational Acceptance Testing (OAT)** — проверка готовности к эксплуатации (бэкапы, восстановление, мониторинг);
   - **Contractual/Regulatory Acceptance Testing** — проверка соответствия договорным или законодательным требованиям;
   - **Alpha/Beta testing** — тестирование ограниченной группой пользователей до публичного релиза.

Эти уровни формируют так называемую "пирамиду тестирования" (Test Pyramid): в основании — много быстрых дешёвых unit-тестов, выше — меньшее число интеграционных тестов, на вершине — небольшое число медленных и дорогих E2E/UI тестов. Такая структура обеспечивает баланс между скоростью обратной связи и полнотой покрытия.

## Пример / когда применяется

Пример: разрабатывается сервис бронирования отелей. Разработчик пишет unit-тесты для функции расчёта итоговой стоимости бронирования с учётом скидок. Затем QA/разработчик проверяет интеграцию сервиса бронирования с сервисом оплаты и с внешним API отеля (integration testing). После сборки всего приложения QA-инженер проверяет полный сценарий "поиск отеля → бронирование → оплата → подтверждение" в тестовой среде (system testing). Перед релизом бизнес-заказчик тестирует, что новый функционал соответствует его ожиданиям (UAT).

## На что смотрит интервьюер

Интервьюер проверяет, понимает ли кандидат иерархию уровней и их связь с пирамидой тестирования, а также кто обычно отвечает за каждый уровень (разработчики — unit, QA — system/acceptance, совместно — integration).

Частые ошибки: путать уровни тестирования с типами тестирования (например, называть "регрессионное тестирование" уровнем), не знать, что такое V-модель.
