# Приведите пример хорошего тест-кейса.

**Коротко:** хороший тест-кейс атомарен, воспроизводим, содержит чёткие предусловия, пронумерованные шаги, конкретные тестовые данные и однозначный ожидаемый результат, а также связан с требованием и приоритетом.

## Развёрнутый ответ

Хороший тест-кейс отвечает следующим критериям:

- **Атомарность** — проверяет один сценарий/условие, а не несколько разных проверок сразу.
- **Однозначность** — любой исполнитель, прочитав кейс, выполнит одни и те же действия и получит один и тот же ожидаемый результат.
- **Воспроизводимость** — чёткие предусловия и тестовые данные, исключающие случайность.
- **Полнота** — присутствуют все обязательные поля: ID, название, приоритет, предусловия, шаги, ожидаемый результат.
- **Трассируемость** — связь с требованием/пользовательской историей, чтобы можно было понять, какое требование покрывает кейс.
- **Актуальность** — кейс соответствует текущей версии функциональности, а не устаревшей.
- **Независимость** — кейс не должен зависеть от порядка выполнения других кейсов (либо зависимость явно указана в предусловиях).

Обязательные поля хорошего тест-кейса:

1. ID (уникальный идентификатор).
2. Название (кратко описывает суть проверки).
3. Приоритет/важность (High/Medium/Low).
4. Предусловия (Preconditions).
5. Шаги выполнения (Steps) — пронумерованный список.
6. Тестовые данные (Test Data).
7. Ожидаемый результат (Expected Result).
8. Фактический результат и статус (заполняются при прогоне).
9. Связанное требование (Requirement/User Story ID).

## Пример / когда применяется

Пример хорошего тест-кейса для проверки формы регистрации на сайте:

```
ID: TC-102
Название: Регистрация нового пользователя с валидными данными
Приоритет: High
Требование: US-021 "Регистрация пользователя"

Предусловия:
  - Пользователь не авторизован
  - Email test.user@example.com отсутствует в базе

Тестовые данные:
  - Email: test.user@example.com
  - Пароль: Qwerty123!
  - Имя: Иван

Шаги:
  1. Открыть страницу /register
  2. Ввести email "test.user@example.com" в поле Email
  3. Ввести пароль "Qwerty123!" в поле Password
  4. Ввести "Иван" в поле Имя
  5. Нажать кнопку "Зарегистрироваться"

Ожидаемый результат:
  - Пользователь создан в системе
  - Отображается сообщение "Регистрация прошла успешно"
  - Произошёл редирект на страницу /dashboard
  - На email отправлено письмо с подтверждением
```

Такой кейс можно передать junior-тестировщику или использовать как основу для написания автотеста — в нём нет двусмысленности ни в шагах, ни в ожидаемом результате.

Для сравнения — пример **плохого** тест-кейса:

```
Название: Проверить регистрацию
Шаги: зарегистрироваться и проверить, что всё работает
Ожидаемый результат: всё ок
```

Такой кейс невоспроизводим: неясно, какие данные вводить, что значит "всё ок", и какой конкретно результат считается успешным.

## На что смотрит интервьюер

- Умение привести структурированный, а не абстрактный пример.
- Наличие конкретных данных (email, пароль) вместо общих фраз "ввести валидные данные".
- Чёткость и проверяемость ожидаемого результата (не "всё работает", а конкретные наблюдаемые эффекты).
- Указание приоритета и связи с требованием — многие кандидаты забывают об этих полях.
- Красный флаг: тест-кейс без ожидаемого результата или без предусловий, либо кейс, проверяющий сразу несколько несвязанных сценариев.
