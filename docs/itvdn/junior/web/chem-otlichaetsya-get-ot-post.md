# Чем отличается GET от POST?

**Коротко:** GET используется для получения данных, передаёт параметры в URL, идемпотентен и кешируется; POST используется для отправки/создания данных, передаёт их в теле запроса, не идемпотентен и обычно не кешируется.

## Развёрнутый ответ

**GET** — метод для получения ресурса с сервера без изменения его состояния.
- Параметры передаются в строке запроса (query string), например `?id=42&sort=asc`.
- Идемпотентен — повторный вызов с теми же параметрами не меняет состояние сервера и возвращает тот же результат.
- Безопасен (safe method) — не должен вызывать побочных эффектов на сервере.
- Кешируется браузерами и прокси по умолчанию.
- Имеет ограничение на длину URL (обычно около 2000 символов в зависимости от браузера/сервера).
- Остаётся в истории браузера, в логах сервера — не подходит для передачи конфиденциальных данных (паролей, токенов).

**POST** — метод для отправки данных на сервер, обычно для создания нового ресурса или выполнения действия с побочными эффектами.
- Данные передаются в теле запроса (`body`), а не в URL.
- Не идемпотентен — повторный вызов может создать несколько ресурсов (например, повторная отправка формы создаст два одинаковых заказа).
- Не кешируется по умолчанию.
- Нет строгого ограничения на размер передаваемых данных (лимит зависит от конфигурации сервера).
- Подходит для передачи конфиденциальных данных, так как они не попадают в URL и историю браузера (хотя это не заменяет шифрование через HTTPS).

Дополнительные различия:

| Критерий | GET | POST |
|---|---|---|
| Назначение | получение данных | создание/отправка данных |
| Место передачи данных | URL (query string) | тело запроса |
| Идемпотентность | да | нет |
| Безопасность (safe) | да | нет |
| Кеширование | да | нет (по умолчанию) |
| Ограничение размера данных | есть (длина URL) | практически нет |
| Видимость в истории/логах | да | нет |

## Пример / когда применяется

GET-запрос:
```
GET /api/products?category=phones&sort=price HTTP/1.1
Host: example.com
```

POST-запрос:
```
POST /api/products HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "name": "iPhone 16",
  "price": 999
}
```

GET используется для поиска товаров, получения списка пользователей, чтения профиля. POST используется для создания заказа, регистрации пользователя, отправки формы обратной связи, загрузки файла.

## На что смотрит интервьюер

- Знание идемпотентности как ключевого различия (частая ошибка — не упомянуть это свойство вообще).
- Понимание, что GET не должен использоваться для операций с побочными эффектами (нарушение принципа "safe method").
- Знание, что данные в GET видны в URL, что критично для безопасности (не передавать пароли через GET).
- Follow-up вопросы: "Что такое идемпотентность?", "Почему нельзя передавать пароль в GET-запросе?", "Чем PUT отличается от POST?".
- Красный флаг: ответ "GET для чтения, POST для записи" без упоминания идемпотентности, кеширования и ограничений по размеру данных.
