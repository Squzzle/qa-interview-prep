# Что такое кэш?

**Коротко:** Кэш — это промежуточное хранилище данных, которое сохраняет копию ранее полученных или вычисленных результатов, чтобы ускорить последующий доступ к ним и снизить нагрузку на сеть/сервер/процессор.

## Развёрнутый ответ

Кэширование — оптимизация, при которой часто используемые или ресурсоёмкие данные сохраняются во временном хранилище (кэше), чтобы избежать повторного обращения к первоисточнику (серверу, базе данных, диску) за теми же данными.

В контексте веба выделяют несколько уровней кэширования:

- **Кэш браузера (HTTP-кэш)** — браузер сохраняет статические ресурсы (HTML, CSS, JS, изображения) на диске пользователя. Управляется HTTP-заголовками ответа сервера: `Cache-Control` (например, `max-age=3600` — кэшировать час), `ETag` и `Last-Modified` (для условной валидации — сервер отвечает `304 Not Modified`, если ресурс не изменился), `Expires` (устаревший, но всё ещё встречающийся заголовок).
- **Прокси-кэш / CDN-кэш** — промежуточные серверы (например, Cloudflare, CDN) кэшируют ответы ближе к пользователю географически, снижая нагрузку на origin-сервер и задержку.
- **Серверный кэш** — сервер приложения кэширует результаты дорогих вычислений или запросов к базе данных (например, в Redis или Memcached), чтобы не выполнять их заново при каждом запросе.
- **Кэш базы данных** — сама СУБД может кэшировать результаты часто выполняемых запросов или страницы данных в памяти.

Ключевая проблема кэширования — **инвалидация (сброс) кэша**: нужно гарантировать, что пользователь не увидит устаревшие данные после их изменения. Частые стратегии: TTL (время жизни записи), явная инвалидация при изменении данных, версионирование URL ресурсов (например, добавление хеша в имя файла `app.abc123.js`).

## Пример / когда применяется

Пример заголовков ответа сервера для статического файла:

```
GET /static/app.js HTTP/1.1

HTTP/1.1 200 OK
Cache-Control: max-age=86400, public
ETag: "33a64df551"
```

При повторном запросе браузер отправит `If-None-Match: "33a64df551"`, и если файл не изменился, сервер ответит `304 Not Modified` без передачи тела — экономия трафика и времени.

Пример тест-кейса для проверки кэширования:
1. Загрузить страницу первый раз, зафиксировать время загрузки и наличие ресурса в кэше (DevTools → Network → колонка Size показывает "from disk cache").
2. Изменить контент на сервере (например, обновить статью), обновить страницу без hard refresh и убедиться, что пользователь видит актуальные данные, а не устаревшую версию из кэша — если версия ресурса не изменилась в URL, это может быть багом инвалидации кэша.
3. Проверить поведение при `Ctrl+F5` (hard refresh, игнорирующий кэш) в сравнении с обычным обновлением.

## На что смотрит интервьюер

- Понимание разницы между уровнями кэширования (браузер, CDN, сервер, БД), а не только "кэш браузера".
- Знание конкретных HTTP-заголовков, отвечающих за кэширование (`Cache-Control`, `ETag`, `Expires`).
- Понимание проблемы инвалидации кэша и её практических последствий для тестирования (устаревшие данные после деплоя, "user видит старую версию сайта").
- Follow-up: "Как проверить, что ресурс закэширован в браузере?", "Что делать, если после релиза пользователи видят старую версию сайта?", "Чем ETag отличается от Last-Modified?".
- Красный флаг — если кандидат путает кэш с cookies или сессией.
