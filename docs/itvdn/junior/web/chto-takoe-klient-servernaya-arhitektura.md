# Что такое клиент-серверная архитектура?

**Коротко:** это модель взаимодействия, где один компонент (клиент) инициирует запросы, а другой (сервер) их обрабатывает и возвращает ответ; клиент и сервер разделены логически и физически и общаются по сети через определённый протокол.

## Развёрнутый ответ

Клиент-серверная архитектура — это способ организации распределённой системы, в котором функциональность разделена между двумя типами участников:

- **Клиент** — инициирует запрос, отвечает за представление данных пользователю (UI/UX), не хранит бизнес-логику или хранит её частично (тонкий/толстый клиент).
- **Сервер** — пассивно ожидает запросы, обрабатывает бизнес-логику, работает с базой данных, возвращает клиенту ответ.

Ключевые принципы такой архитектуры:

- **Разделение ответственности** — клиент отвечает за отображение, сервер за обработку и хранение данных.
- **Асимметричность взаимодействия** — клиент всегда инициатор, сервер всегда отвечающая сторона (в классической модели, без учёта WebSocket/Server-Sent Events, где возможна инициация от сервера).
- **Множественность клиентов** — один сервер может обслуживать множество клиентов одновременно.
- **Использование сетевых протоколов** — взаимодействие идёт по протоколам вроде HTTP/HTTPS, TCP, WebSocket.
- **Масштабируемость** — сервер можно масштабировать независимо от клиентов (горизонтально или вертикально).

Различают несколько видов клиентов:

- **Тонкий клиент** — минимальная логика на стороне клиента, вся обработка происходит на сервере (классический пример — браузер, отображающий серверный HTML).
- **Толстый (богатый) клиент** — значительная часть логики выполняется на клиенте (SPA-приложения на React/Angular/Vue, десктопные приложения).

Также существует многоуровневая (n-tier) архитектура, где помимо клиента и сервера выделяют дополнительные слои: сервер приложений, сервер базы данных, балансировщик нагрузки, кеш-сервер и т.д.

## Пример / когда применяется

Классический пример: браузер (клиент) отправляет HTTP GET-запрос на веб-сервер (Nginx/Apache), который обрабатывает запрос, обращается к базе данных, формирует HTML или JSON и отправляет ответ обратно.

Пример на уровне HTTP-обмена:

```
Клиент -> Сервер:
GET /api/users/42 HTTP/1.1
Host: example.com

Сервер -> Клиент:
HTTP/1.1 200 OK
Content-Type: application/json

{ "id": 42, "name": "Ivan" }
```

Клиент-серверная архитектура применяется практически везде: веб-приложения, мобильные приложения с backend, банковские системы, почтовые клиенты (SMTP/IMAP), DNS-система и т.д.

## На что смотрит интервьюер

- Понимание, что клиент и сервер — это роли, а не конкретные устройства (сервер тоже может быть клиентом для другого сервера, как в микросервисах).
- Знание отличий тонкого и толстого клиента.
- Способность привести реальный пример из практики (например, тестируемое приложение с фронтом и backend API).
- Частая ошибка: путать клиент-серверную архитектуру с peer-to-peer или считать, что сервер обязательно физическая машина, а не программный процесс.
