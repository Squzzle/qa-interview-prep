# Что такое REST API, SOAP? В чем разница?

**Коротко:** REST — это архитектурный стиль для построения API поверх HTTP с использованием стандартных методов и обычно JSON, а SOAP — строгий протокол обмена сообщениями на основе XML со своей спецификацией конвертов, заголовков и стандартов безопасности.

## Развёрнутый ответ

**REST (Representational State Transfer)** — архитектурный стиль, предложенный Роем Филдингом, описывающий набор принципов построения распределённых систем:

- Клиент-серверная модель.
- Отсутствие состояния (statelessness) — каждый запрос содержит всю информацию, необходимую для его обработки, сервер не хранит сессию клиента между запросами.
- Кешируемость ответов.
- Единообразный интерфейс — использование стандартных HTTP-методов (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`) и работа с ресурсами через URI.
- Слоистая система (можно добавлять прокси, балансировщики, шлюзы без изменения контракта клиент-сервер).
- Опционально HATEOAS (Hypermedia as the Engine of Application State).

REST API обычно использует JSON (реже XML) как формат данных, а не требует строгой спецификации сообщений — это скорее набор рекомендаций, чем жёсткий стандарт.

**SOAP (Simple Object Access Protocol)** — протокол обмена структурированными сообщениями, основанный на XML. Каждое SOAP-сообщение оформляется в виде "конверта" (Envelope), содержащего заголовок (Header) и тело (Body). SOAP имеет строгую спецификацию, описываемую через WSDL (Web Services Description Language), что позволяет генерировать клиентский код автоматически.

Ключевые отличия:

| Критерий | REST | SOAP |
|---|---|---|
| Тип | архитектурный стиль | протокол |
| Формат данных | JSON, XML, любой | только XML |
| Транспорт | обычно HTTP/HTTPS | HTTP, SMTP, TCP и др. |
| Состояние | stateless | может быть stateful |
| Описание контракта | OpenAPI/Swagger (опционально) | WSDL (обязательно) |
| Безопасность | HTTPS, OAuth, JWT | WS-Security (встроенная спецификация) |
| Производительность | легче, меньше накладных расходов | тяжелее из-за XML и конвертов |
| Гибкость | высокая | низкая, строгая типизация |
| Обработка ошибок | HTTP-коды состояния | стандартизированный `Fault`-элемент внутри XML |

REST проще в изучении и использовании, лучше подходит для веб и мобильных приложений. SOAP используется в системах с высокими требованиями к безопасности, транзакционности и формальному контракту — банковские системы, платёжные шлюзы, корпоративные интеграции (например, стандарт ebXML).

## Пример / когда применяется

Пример REST-запроса:

```
GET /api/users/42 HTTP/1.1
Host: example.com
Accept: application/json
```

Пример SOAP-запроса:

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
  <soapenv:Header/>
  <soapenv:Body>
    <getUser>
      <id>42</id>
    </getUser>
  </soapenv:Body>
</soapenv:Envelope>
```

REST применяется в большинстве современных веб- и мобильных API (Twitter API, GitHub API). SOAP до сих пор используется в банковском секторе, страховании, государственных системах, где важна строгая типизация и встроенная безопасность (WS-Security, WS-ReliableMessaging).

## На что смотрит интервьюер

- Понимание, что REST — это стиль, а SOAP — протокол (частая путаница у джунов).
- Знание, что REST не обязательно использует JSON, а может использовать и XML.
- Умение назвать конкретные отличия (WSDL vs OpenAPI, stateless vs возможность stateful).
- Follow-up вопросы: "Что такое HATEOAS?", "Как тестировать SOAP-сервисы?", "Чем GraphQL отличается от REST?".
- Красный флаг: если кандидат говорит "REST — это формат передачи данных" или путает REST с JSON.
