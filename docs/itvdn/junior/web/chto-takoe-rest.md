# Что такое REST?

**Коротко:** REST (Representational State Transfer) — это архитектурный стиль построения распределённых систем (чаще всего веб-API), основанный на работе с ресурсами через стандартные HTTP-методы, единообразных URL-адресах ресурсов и отсутствии состояния сессии на сервере между запросами.

## Развёрнутый ответ

REST был описан Роем Филдингом в его диссертации 2000 года как набор архитектурных ограничений (constraints), которым должна соответствовать система, чтобы называться RESTful:

1. **Клиент-серверная архитектура.** Клиент и сервер разделены и могут развиваться независимо, взаимодействуя только через согласованный интерфейс.

2. **Отсутствие состояния (statelessness).** Каждый запрос от клиента к серверу должен содержать всю информацию, необходимую для его обработки (например, токен авторизации в заголовке); сервер не хранит состояние клиентской сессии между запросами.

3. **Кэшируемость (cacheable).** Ответы сервера должны явно указывать, можно ли их кэшировать (через заголовки `Cache-Control`, `ETag`), чтобы повысить производительность.

4. **Единообразный интерфейс (uniform interface).** Ключевое ограничение REST, включающее:
   - идентификацию ресурсов через URI (например, `/users/42`);
   - манипуляцию ресурсами через представления (JSON/XML) и стандартные HTTP-методы (`GET` — получить, `POST` — создать, `PUT`/`PATCH` — обновить, `DELETE` — удалить);
   - самоописываемые сообщения (правильные HTTP-заголовки, статус-коды);
   - HATEOAS (Hypermedia as the Engine of Application State) — ответ может содержать ссылки на связанные действия/ресурсы (на практике многие "RESTful" API этим ограничением пренебрегают).

5. **Многоуровневая система (layered system).** Между клиентом и сервером могут быть промежуточные слои (прокси, балансировщики, шлюзы), прозрачные для клиента.

6. **Код по требованию (code on demand)** — опциональное ограничение, позволяющее серверу передавать клиенту исполняемый код (например, JavaScript).

На практике термином "REST API" или "RESTful API" часто называют любой HTTP-based API, работающий с JSON и использующий стандартные HTTP-методы, даже если он не строго соответствует всем шести ограничениям (особенно HATEOAS почти никогда не реализуется полностью) — это важный нюанс, который стоит упомянуть на собеседовании, показывая, что кандидат понимает разницу между "RESTful в строгом смысле" и "REST-подобным API, который используется в индустрии".

Для тестировщика REST API важен тем, что тестирование API строится вокруг проверки: правильных HTTP-методов и статус-кодов (200, 201, 400, 401, 403, 404, 500), корректности структуры JSON-ответа (схема, обязательные поля), идемпотентности методов (`GET`, `PUT`, `DELETE` должны быть идемпотентны — повторный вызов с теми же параметрами не должен менять результат, в отличие от `POST`).

## Пример / когда применяется

Пример типичного REST API для управления пользователями:

```
GET    /api/users        -> получить список пользователей (200 OK)
GET    /api/users/42     -> получить пользователя с id=42 (200 OK / 404 Not Found)
POST   /api/users        -> создать нового пользователя (201 Created)
PUT    /api/users/42     -> полностью обновить пользователя 42 (200 OK)
PATCH  /api/users/42     -> частично обновить пользователя 42 (200 OK)
DELETE /api/users/42     -> удалить пользователя 42 (204 No Content)
```

Пример проверки ответа в автотесте (Java + RestAssured):

```java
given()
  .header("Authorization", "Bearer " + token)
.when()
  .get("/api/users/42")
.then()
  .statusCode(200)
  .body("id", equalTo(42))
  .body("email", notNullValue());
```

## На что смотрит интервьюер

Интервьюер проверяет знание базовых принципов REST (statelessness, ресурсы через URI, стандартные HTTP-методы), а также умение критично оценить, что реальные API редко полностью соответствуют строгому REST (в частности, HATEOAS). Часто спрашивают про разницу REST vs SOAP, идемпотентность методов, разницу PUT и PATCH. Красный флаг — если кандидат не может назвать статус-коды или путает `PUT` и `POST` по семантике (создание vs полное обновление).
