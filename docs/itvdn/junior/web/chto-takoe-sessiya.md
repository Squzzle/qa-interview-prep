# Что такое сессия?

**Коротко:** Сессия — это период взаимодействия пользователя с веб-приложением, в течение которого сервер сохраняет состояние пользователя (например, факт авторизации) между отдельными HTTP-запросами, поскольку сам протокол HTTP не хранит состояние (stateless).

## Развёрнутый ответ

HTTP — протокол без сохранения состояния (stateless): каждый запрос обрабатывается независимо, и сервер по умолчанию "не помнит" предыдущие запросы того же клиента. Чтобы связать несколько запросов одного пользователя (например, чтобы не заставлять его логиниться на каждой странице), используется механизм сессий.

Классическая реализация:
1. Пользователь логинится, сервер создаёт запись сессии (обычно в памяти, Redis или базе данных) с уникальным идентификатором — **session ID**.
2. Session ID передаётся клиенту, как правило, через cookie (например, `JSESSIONID`, `connect.sid`).
3. При каждом следующем запросе браузер автоматически отправляет эту cookie серверу.
4. Сервер по session ID находит сохранённые данные (например, id пользователя, его права) и понимает, кто сделал запрос, без повторной аутентификации.

Сессия имеет время жизни (**session timeout**) — если пользователь неактивен дольше заданного времени, сессия истекает, и требуется повторный вход.

Альтернативный подход — **stateless-сессии на основе токенов** (например, JWT — JSON Web Token): вся необходимая информация о пользователе закодирована и подписана внутри самого токена, который клиент передаёт в заголовке `Authorization`. Сервер не хранит состояние сессии на своей стороне (или хранит минимально), а лишь проверяет подпись и срок действия токена. Такой подход лучше масштабируется горизонтально (не нужно синхронизировать хранилище сессий между серверами).

Важно не путать сессию и cookie: cookie — это лишь один из механизмов транспорта, с помощью которого чаще всего передаётся идентификатор сессии, но не единственный (session ID теоретически можно передавать и в URL, хотя это небезопасно).

## Пример / когда применяется

Пример работы cookie-based сессии в HTTP-заголовках:

```
# После логина сервер отвечает:
HTTP/1.1 200 OK
Set-Cookie: JSESSIONID=A1B2C3D4E5; HttpOnly; Secure; Path=/

# При следующем запросе браузер отправляет:
GET /profile HTTP/1.1
Cookie: JSESSIONID=A1B2C3D4E5
```

Тест-кейсы для проверки сессии:
- Проверить, что после логина сессия действительно создаётся (появляется cookie/токен) и пользователь получает доступ к защищённым страницам.
- Проверить, что после логаута сессия действительно инвалидируется на сервере (повторное использование старого session ID/токена должно возвращать `401 Unauthorized`), а не только удаляется cookie на клиенте.
- Проверить поведение при истечении сессии по таймауту — пользователя должно перенаправить на страницу логина.
- Проверить, что параллельные сессии одного пользователя в разных браузерах обрабатываются корректно (в зависимости от бизнес-требований — допускаются несколько активных сессий или нет).

## На что смотрит интервьюер

- Понимание, что сессия решает проблему stateless-природы HTTP.
- Умение отличить сессию от cookie и от токена (JWT) — частый вопрос на смешение понятий.
- Понимание жизненного цикла сессии: создание, использование, истечение/логаут.
- Follow-up: "Чем отличается session-based авторизация от JWT?", "Как протестировать логаут?", "Что произойдёт, если удалить cookie сессии вручную во время работы?".
- Красный флаг — если кандидат утверждает, что "сессия — это и есть cookie", не понимая, что cookie лишь транспортный механизм для идентификатора сессии.
