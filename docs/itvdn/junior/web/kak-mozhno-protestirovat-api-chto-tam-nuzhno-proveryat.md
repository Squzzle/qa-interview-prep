# Как можно протестировать API, что там нужно проверять?

**Коротко:** API тестируют вручную (Postman, curl) и автоматизированно (REST Assured, Pytest+requests, Playwright API-тесты), проверяя корректность статус-кодов, структуры и содержимого ответа, обработку ошибок, производительность, безопасность и соответствие контракту.

## Развёрнутый ответ

Тестирование API можно разделить по способу выполнения и по типу проверок.

**Способы тестирования:**

- **Ручное тестирование** — через Postman, Insomnia, curl. Подходит для исследовательского тестирования, дебага, разовых проверок.
- **Автоматизированное тестирование** — с помощью фреймворков: REST Assured (Java), Pytest + requests (Python), Supertest (Node.js), Playwright/Cypress (для API внутри e2e). Автотесты запускаются в CI/CD на каждый коммит/релиз.
- **Контрактное тестирование** — проверка соответствия API заявленной спецификации (OpenAPI/Swagger), инструменты вроде Pact для consumer-driven contract testing.
- **Нагрузочное тестирование** — JMeter, Gatling, k6 — проверка поведения API под нагрузкой (RPS, время отклика, деградация).
- **Тестирование безопасности** — проверка на SQL-инъекции, XSS, некорректную авторизацию (OWASP ZAP, Burp Suite).

**Что нужно проверять в ответе API:**

1. **Статус-код** — соответствует ли ожидаемому (`200 OK`, `201 Created`, `400 Bad Request`, `401 Unauthorized`, `404 Not Found`, `500 Internal Server Error` и т.д.).
2. **Структура ответа (схема)** — соответствие JSON Schema или контракту (наличие обязательных полей, правильные типы данных).
3. **Содержимое (данные)** — корректность значений, соответствие бизнес-логике (например, при создании пользователя возвращается корректный `id`).
4. **Заголовки ответа** — `Content-Type`, `Cache-Control`, CORS-заголовки, кастомные заголовки.
5. **Обработка невалидных входных данных** — что возвращает API при некорректных параметрах, отсутствующих обязательных полях, неверных типах данных.
6. **Граничные значения** — пустые строки, максимальные длины, специальные символы, null-значения.
7. **Авторизация и аутентификация** — доступ с валидным/невалидным/отсутствующим токеном, доступ с недостаточными правами.
8. **Идемпотентность** — повторный вызов `PUT`/`DELETE` должен давать предсказуемый результат.
9. **Производительность** — время ответа в пределах SLA.
10. **Согласованность данных** — например, после `POST` данные действительно появляются при последующем `GET`.
11. **Пагинация, фильтрация, сортировка** — если API их поддерживает.
12. **Версионирование API** — совместимость при смене версий (`/v1/`, `/v2/`).

## Пример / когда применяется

Пример автотеста на Java с REST Assured:

```java
@Test
public void shouldReturnUserById() {
    given()
        .header("Authorization", "Bearer " + token)
    .when()
        .get("/api/users/42")
    .then()
        .statusCode(200)
        .body("id", equalTo(42))
        .body("name", notNullValue())
        .contentType(ContentType.JSON);
}

@Test
public void shouldReturn404ForNonExistentUser() {
    given()
        .header("Authorization", "Bearer " + token)
    .when()
        .get("/api/users/999999")
    .then()
        .statusCode(404)
        .body("error", equalTo("User not found"));
}
```

Пример проверки через curl вручную:

```bash
curl -i -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name": ""}'
# Ожидается 400 Bad Request, так как имя не может быть пустым
```

## На что смотрит интервьюер

- Способность назвать не только "проверяю статус-код", а полный список: схему, данные, заголовки, ошибки, безопасность, производительность.
- Понимание разницы между позитивными и негативными тест-кейсами.
- Знание конкретных инструментов (Postman, REST Assured, JMeter) и умение объяснить их назначение.
- Follow-up вопросы: "Как бы вы тестировали идемпотентность DELETE-запроса?", "Что такое контрактное тестирование?", "Как автоматизировать проверку JSON Schema?".
- Красный флаг: если кандидат называет только "отправляю запрос и смотрю, что вернулось 200".
