# Как происходит шифрование?

**Коротко:** Шифрование — это преобразование данных в нечитаемый вид с помощью алгоритма и ключа так, чтобы восстановить исходные данные мог только тот, у кого есть нужный ключ; различают симметричное (один ключ) и асимметричное (пара ключей: публичный/приватный) шифрование.

## Развёрнутый ответ

Шифрование (encryption) — процесс преобразования открытого текста (plaintext) в шифротекст (ciphertext) с помощью математического алгоритма (шифра) и ключа. Обратный процесс — расшифрование (decryption) — восстанавливает исходные данные, но выполнить его без ключа практически невозможно за разумное время.

Существует два основных типа шифрования:

- **Симметричное шифрование** — для шифрования и расшифрования используется один и тот же секретный ключ. Примеры алгоритмов: AES, DES, 3DES, ChaCha20. Плюс — высокая скорость работы, минус — проблема безопасной передачи ключа между сторонами.
- **Асимметричное шифрование** — используется пара математически связанных ключей: публичный (открытый), которым шифруют данные или проверяют подпись, и приватный (закрытый), которым расшифровывают данные или создают подпись. Примеры: RSA, ECC, DSA. Плюс — не нужно заранее обмениваться секретом, минус — работает значительно медленнее симметричного.

На практике оба подхода комбинируют — это называется гибридным шифрованием: асимметричным алгоритмом шифруют случайно сгенерированный симметричный ключ (сессионный ключ), а сами данные шифруют уже быстрым симметричным алгоритмом. Именно так устроен протокол TLS/SSL, лежащий в основе HTTPS.

Отдельно стоит упомянуть хеширование — это не шифрование, а односторонняя функция: из данных получают хеш фиксированной длины, но обратно данные восстановить нельзя (используется для паролей, контроля целостности, ЭЦП).

## Пример / когда применяется

Классический пример — установка HTTPS-соединения (упрощённо):

1. Клиент обращается к серверу, сервер присылает сертификат с публичным ключом.
2. Клиент проверяет сертификат (через цепочку доверия к CA) и генерирует случайный симметричный ключ сессии.
3. Клиент шифрует этот сессионный ключ публичным ключом сервера (асимметрично) и отправляет серверу.
4. Сервер расшифровывает сессионный ключ своим приватным ключом.
5. Дальнейший обмен данными идёт уже быстрым симметричным шифрованием (например, AES-256) с использованием этого сессионного ключа.

Как тестировщику это важно, например, при проверке, что:
- сайт использует актуальный TLS (1.2/1.3), а не устаревший SSL/TLS 1.0;
- пароли в базе хранятся не в открытом виде, а хешируются (bcrypt/argon2), а не просто "шифруются";
- чувствительные данные (токены, номера карт) не передаются по HTTP.

## На что смотрит интервьюер

- Путает ли кандидат шифрование и хеширование (частая ошибка — "пароли шифруются", хотя правильно — хешируются).
- Знает ли разницу симметричного и асимметричного шифрования и может привести примеры алгоритмов.
- Понимает ли, зачем нужен гибридный подход в TLS, а не только асимметричное шифрование.
- Follow-up вопросы: "Чем отличается HTTP от HTTPS?", "Что такое SSL-сертификат?", "Как проверить, что соединение защищено?".
- Красный флаг — если кандидат не может объяснить простыми словами, зачем вообще нужно шифрование (конфиденциальность, целостность, аутентификация).
