# Как расшифровывается CRUD?

**Коротко:** CRUD — это аббревиатура от Create, Read, Update, Delete — четыре базовые операции над данными, которые лежат в основе большинства API и систем хранения информации.

## Развёрнутый ответ

CRUD расшифровывается как:

- **Create (Создание)** — добавление новой записи/ресурса. В REST API соответствует методу `POST`, в SQL — оператору `INSERT`.
- **Read (Чтение)** — получение существующих данных без их изменения. В REST соответствует методу `GET`, в SQL — `SELECT`.
- **Update (Обновление)** — изменение существующей записи. В REST соответствует методам `PUT` (полная замена ресурса) или `PATCH` (частичное обновление), в SQL — `UPDATE`.
- **Delete (Удаление)** — удаление записи/ресурса. В REST соответствует методу `DELETE`, в SQL — `DELETE`.

CRUD — это базовая модель для описания жизненного цикла данных в любой информационной системе: от простого TODO-приложения до сложной ERP-системы. Практически любое приложение, работающее с базой данных, реализует эти четыре операции в том или ином виде — через UI, API или напрямую через SQL-запросы.

Важный нюанс для тестировщика: CRUD-операции — это минимальный набор функциональности, который обязательно нужно протестировать при работе с любой сущностью в системе (пользователь, заказ, товар и т.д.), включая:
- позитивные сценарии (успешное создание/чтение/обновление/удаление);
- негативные сценарии (создание с невалидными данными, обновление несуществующей записи, удаление уже удалённой записи);
- проверку побочных эффектов (например, при удалении родительской сущности — что происходит с дочерними: каскадное удаление, запрет удаления, обнуление связи).

## Пример / когда применяется

Соответствие CRUD и HTTP-методов на примере API управления пользователями:

```
POST   /api/users          -> Create: создать нового пользователя
GET    /api/users/42       -> Read: получить пользователя с id=42
GET    /api/users          -> Read: получить список пользователей
PUT    /api/users/42       -> Update: полностью обновить пользователя
PATCH  /api/users/42       -> Update: частично обновить пользователя
DELETE /api/users/42       -> Delete: удалить пользователя
```

Соответствие в SQL:

```sql
INSERT INTO users (name, email) VALUES ('Ivan', 'ivan@example.com'); -- Create
SELECT * FROM users WHERE id = 42;                                   -- Read
UPDATE users SET name = 'Ivan Petrov' WHERE id = 42;                 -- Update
DELETE FROM users WHERE id = 42;                                     -- Delete
```

## На что смотрит интервьюер

- Знание точной расшифровки (Create, Read, Update, Delete) без ошибок в порядке или значении букв.
- Умение сопоставить CRUD-операции с HTTP-методами и SQL-операторами.
- Понимание, что тестирование CRUD подразумевает не только happy path, но и негативные сценарии, побочные эффекты (каскадное удаление, конфликты при одновременном обновлении).
- Follow-up вопросы: "В чём разница между PUT и PATCH при Update?", "Как бы вы протестировали Delete с точки зрения целостности данных?".
- Красный флаг: путаница между CRUD и HTTP-методами при описании (например, ответ "CRUD — это GET, POST, PUT, DELETE" без указания смысловых операций).
