# Какие бывают виды баз данных?

**Коротко:** Основные виды баз данных — реляционные (SQL) и нереляционные (NoSQL); NoSQL, в свою очередь, делится на документные, key-value, колоночные и графовые; выбор зависит от структуры данных, требований к консистентности и масштабированию.

## Развёрнутый ответ

**Реляционные базы данных (RDBMS)** хранят данные в виде таблиц со строками и столбцами, между таблицами задаются связи (foreign key). Данные структурированы жёсткой схемой. Для работы используется язык SQL. Такие СУБД, как правило, поддерживают ACID-транзакции (атомарность, согласованность, изолированность, устойчивость), что делает их надёжными для финансовых и других критичных систем. Примеры: PostgreSQL, MySQL, Oracle, MS SQL Server, SQLite.

**Нереляционные базы данных (NoSQL)** не требуют жёсткой схемы и лучше подходят для больших объёмов слабоструктурированных данных, горизонтального масштабирования и высокой скорости записи/чтения. Внутри NoSQL выделяют несколько подвидов:

- **Документные** — данные хранятся в виде документов (обычно JSON/BSON), например `MongoDB`, `CouchDB`. Удобны, когда структура данных гибкая и может меняться от записи к записи.
- **Key-Value** — простейшая модель "ключ-значение", максимально быстрый доступ по ключу. Примеры: `Redis`, `DynamoDB`, `Memcached`. Часто используются как кэш или сессионное хранилище.
- **Колоночные (column-family)** — данные хранятся по столбцам, а не строкам, что ускоряет аналитические запросы по большим объёмам данных. Примеры: `Cassandra`, `HBase`.
- **Графовые** — данные представлены как узлы и связи между ними, удобны для моделирования сложных взаимосвязей (соцсети, рекомендательные системы). Примеры: `Neo4j`, `ArangoDB`.

Отдельно можно выделить **in-memory базы данных** (например, Redis, Memcached), которые хранят данные в оперативной памяти для максимальной скорости доступа, и **NewSQL** — гибридные системы, сочетающие масштабируемость NoSQL с ACID-гарантиями SQL (например, `CockroachDB`).

Ключевое отличие в подходе к согласованности: RDBMS обычно следуют модели **ACID**, тогда как многие NoSQL-системы придерживаются модели **BASE** (Basically Available, Soft state, Eventually consistent) — жертвуют строгой консистентностью ради доступности и масштабируемости (см. теорему CAP).

## Пример / когда применяется

| Тип БД | Пример | Когда использовать |
|---|---|---|
| Реляционная | PostgreSQL, MySQL | Финансы, заказы, где важна целостность связей и транзакции |
| Документная | MongoDB | Каталоги товаров с изменяемой структурой атрибутов |
| Key-Value | Redis | Кэш, сессии, счётчики, rate limiting |
| Колоночная | Cassandra | Логи, телеметрия, аналитика big data |
| Графовая | Neo4j | Социальные связи, рекомендации, обнаружение мошенничества |

Как тестировщику важно понимать тип БД проекта, чтобы правильно готовить тестовые данные, проверять целостность после операций (например, при удалении родительской записи в SQL проверяется каскадное удаление/запрет через FK), а в NoSQL — учитывать возможную eventual consistency при проверке результата сразу после записи.

## На что смотрит интервьюер

- Может ли кандидат назвать конкретные примеры СУБД для каждого типа, а не только теорию.
- Понимает ли разницу ACID vs BASE и знает ли про CAP-теорему (хотя бы на базовом уровне).
- Follow-up: "Что такое нормализация?", "Что такое индекс?", "В чём разница между SQL JOIN и связями в MongoDB?".
- Красный флаг — если кандидат утверждает, что NoSQL "всегда лучше" или "всегда быстрее" без понимания компромиссов.
