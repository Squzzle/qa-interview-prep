# Какие есть HTTP-методы?

**Коротко:** Основные HTTP-методы — GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS, TRACE и CONNECT; они различаются по назначению, а также по свойствам идемпотентности и безопасности (safe).

## Развёрнутый ответ

HTTP-метод (глагол) указывает, какое действие клиент хочет выполнить над ресурсом.

- **GET** — получение ресурса. Не должен изменять состояние сервера (safe), идемпотентен (повторный вызов даёт тот же результат). Параметры передаются в строке запроса, тело запроса обычно не используется.
- **POST** — создание нового ресурса или выполнение действия, которое меняет состояние сервера. Не идемпотентен: повторный POST может создать ещё один ресурс (например, ещё один заказ).
- **PUT** — полная замена ресурса по заданному идентификатору. Идемпотентен: сколько раз ни отправь один и тот же PUT-запрос, результат будет одинаковым.
- **PATCH** — частичное обновление ресурса (только указанные поля). В общем случае не гарантированно идемпотентен, но на практике часто реализуется идемпотентно.
- **DELETE** — удаление ресурса. Идемпотентен: повторное удаление уже удалённого ресурса не должно приводить к новому эффекту (хотя код ответа может отличаться, например `404` вместо `200`).
- **HEAD** — аналог GET, но сервер возвращает только заголовки, без тела ответа. Полезен для проверки существования ресурса или его метаданных (например, размер файла) без загрузки содержимого.
- **OPTIONS** — запрос информации о доступных методах и возможностях сервера/ресурса. Используется, в частности, в CORS как preflight-запрос перед "небезопасными" кросс-доменными запросами.
- **TRACE** — возвращает клиенту то же самое сообщение, что было получено сервером, используется для отладки; часто отключается из соображений безопасности.
- **CONNECT** — устанавливает туннель к серверу, обычно для работы через прокси (например, для HTTPS-соединений).

Важные понятия:
- **Safe (безопасный) метод** — не изменяет состояние сервера (GET, HEAD, OPTIONS).
- **Idempotent (идемпотентный) метод** — при многократном повторении даёт тот же эффект, что и один раз (GET, PUT, DELETE, HEAD; POST — нет).

## Пример / когда применяется

Пример REST API для управления пользователями:

```
GET    /api/users        — получить список пользователей
GET    /api/users/1      — получить пользователя с id=1
POST   /api/users        — создать нового пользователя
PUT    /api/users/1      — полностью заменить данные пользователя 1
PATCH  /api/users/1      — обновить только email пользователя 1
DELETE /api/users/1      — удалить пользователя 1
```

С точки зрения тестирования: если сделать `PUT /api/users/1` дважды подряд с одинаковым телом — итоговое состояние ресурса должно быть одинаковым (идемпотентность). А если два одинаковых `POST /api/users` создают два разных ресурса с разными id — это ожидаемое поведение, а не баг.

## На что смотрит интервьюер

- Знание не только GET/POST, но и PUT/PATCH/DELETE/HEAD/OPTIONS.
- Понимание разницы PUT (полная замена) и PATCH (частичное обновление) — частый вопрос.
- Понимание идемпотентности и её практического значения для тестирования (ретраи, повторные запросы, чувствительность к сетевым сбоям).
- Follow-up: "Что такое CORS preflight и какой метод он использует?", "Почему GET не должен иметь тело запроса на практике?", "Как протестировать идемпотентность DELETE?".
- Красный флаг — если кандидат не может объяснить, зачем нужен именно PATCH, если есть PUT.
