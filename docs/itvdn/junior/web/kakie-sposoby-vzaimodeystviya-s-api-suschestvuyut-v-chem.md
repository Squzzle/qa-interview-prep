# Какие способы взаимодействия с API существуют? В чем разница между ними?

**Коротко:** основные подходы к взаимодействию с API — REST, SOAP, GraphQL, gRPC и WebSocket; они различаются форматом данных, моделью запрос-ответ, гибкостью выборки данных и производительностью.

## Развёрнутый ответ

**REST (Representational State Transfer)** — использует стандартные HTTP-методы (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`) для работы с ресурсами, обычно передаёт данные в формате JSON. Прост в реализации и тестировании, но может приводить к избыточной (over-fetching) или недостаточной (under-fetching) выборке данных.

**SOAP (Simple Object Access Protocol)** — строгий XML-based протокол со своей спецификацией (WSDL), поддерживает встроенные стандарты безопасности и транзакционности (WS-Security, WS-AtomicTransaction). Тяжелее и медленнее REST, но надёжнее в enterprise-средах.

**GraphQL** — язык запросов к API, разработанный Facebook, позволяет клиенту точно указать, какие поля данных ему нужны, что решает проблему over-/under-fetching. Обычно есть единственная конечная точка (`/graphql`), а запрос описывает нужную структуру данных.

**gRPC (Google Remote Procedure Call)** — высокопроизводительный RPC-фреймворк на основе HTTP/2 и бинарного формата Protocol Buffers. Обеспечивает быструю передачу данных, поддерживает потоковую передачу (streaming), строгую типизацию через `.proto`-файлы. Часто используется во внутреннем взаимодействии микросервисов.

**WebSocket** — протокол полнодуплексной связи, позволяющий серверу самостоятельно отправлять данные клиенту без повторных запросов (в отличие от классического HTTP request-response). Используется там, где нужна связь в реальном времени.

Различия между подходами:

| Критерий | REST | SOAP | GraphQL | gRPC |
|---|---|---|---|---|
| Формат данных | JSON/XML | XML | JSON | Protocol Buffers (бинарный) |
| Гибкость выборки | низкая | низкая | высокая (клиент выбирает поля) | средняя |
| Производительность | средняя | низкая | средняя | высокая |
| Обучаемость | простая | сложная | средняя | средняя |
| Потоковая передача | нет (кроме WebSocket) | нет | подписки (subscriptions) | да, встроенная |
| Типизация контракта | опционально (OpenAPI) | обязательно (WSDL) | обязательно (схема GraphQL) | обязательно (`.proto`) |

## Пример / когда применяется

REST-запрос:
```
GET /api/users/42/orders
```

GraphQL-запрос (клиент явно указывает нужные поля):
```graphql
query {
  user(id: 42) {
    name
    orders {
      id
      total
    }
  }
}
```

gRPC-описание метода:
```protobuf
service UserService {
  rpc GetUser (UserRequest) returns (UserResponse);
}
```

REST хорош для публичных API общего назначения; GraphQL — для сложных фронтендов с разнообразными требованиями к данным (мобильное приложение и веб могут запрашивать разные поля через один endpoint); gRPC — для внутреннего взаимодействия микросервисов, где важна производительность; SOAP — для enterprise-интеграций с высокими требованиями к безопасности; WebSocket — для чатов, уведомлений в реальном времени, трейдинговых платформ.

## На что смотрит интервьюер

- Умение сравнить минимум 3 подхода, а не только REST vs SOAP.
- Понимание проблемы over-fetching/under-fetching и того, как её решает GraphQL.
- Знание, что gRPC использует бинарный формат и HTTP/2, что даёт выигрыш в производительности.
- Follow-up вопросы: "Как тестировать GraphQL API?", "В чём минусы GraphQL (сложность кеширования, N+1 проблема)?", "Как тестировать gRPC без специальных инструментов (grpcurl, BloomRPC)?".
- Красный флаг: если кандидат знает только REST и не может назвать альтернативы.
