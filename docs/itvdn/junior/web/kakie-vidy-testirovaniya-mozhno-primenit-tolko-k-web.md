# Какие виды тестирования можно применить только к Web?

**Коротко:** к Web-приложениям специфичны тестирование кроссбраузерности, адаптивной вёрстки (responsive/cross-resolution), SEO-тестирование, тестирование через прямые манипуляции с URL, cookies/localStorage/sessionStorage и HTTP-заголовками, а также проверка поведения при разных настройках браузера (JS выключен, разные разрешения экрана, зум).

## Развёрнутый ответ

В отличие от десктопных и мобильных нативных приложений, веб-приложение выполняется внутри браузера как посредника между пользователем и сервером, и именно из этой архитектуры вытекают специфичные для Web виды и аспекты тестирования:

1. **Кроссбраузерное тестирование (cross-browser testing).** Проверка того, что сайт корректно отображается и работает в разных браузерах (Chrome, Firefox, Safari, Edge) и их версиях, так как разные браузерные движки (Blink, Gecko, WebKit) могут по-разному интерпретировать CSS и JavaScript.

2. **Кроссплатформенное и адаптивное (responsive) тестирование.** Проверка корректности отображения при разных разрешениях экрана, ориентациях и masштабировании — актуально из-за верстки на CSS Grid/Flexbox и media queries, которых нет в нативных приложениях в таком виде.

3. **Тестирование навигации через URL и историю браузера.** Проверка работы прямых ссылок (deep links), корректности поведения кнопок "Назад"/"Вперёд", работы с параметрами запроса (query string), якорями (hash), обработки 404-страниц при невалидных URL.

4. **Тестирование кук, localStorage, sessionStorage и кэша.** Проверка, что сессии, авторизация и пользовательские настройки корректно сохраняются/удаляются, а также поведение приложения при отключённых cookies.

5. **SEO-тестирование.** Проверка мета-тегов, `sitemap.xml`, `robots.txt`, семантической разметки, микроразметки (schema.org) — специфично именно для публичных веб-сайтов, а не для приложений за авторизацией или мобильных апп.

6. **Тестирование доступности (accessibility/a11y) для веб-стандартов** — соответствие WCAG, работа с screen reader через ARIA-атрибуты в контексте DOM.

7. **Тестирование при отключённом JavaScript или блокировщиках рекламы/скриптов** — проверка graceful degradation.

8. **Тестирование HTTP-специфичных аспектов** — корректность статус-кодов, редиректов (301/302), работы HTTPS/SSL-сертификатов, CORS-политик, работы с HTTP-заголовками кэширования.

Стоит отличать эти виды от общих видов тестирования (функциональное, регрессионное, нагрузочное, юзабилити), которые применимы к любому типу ПО (десктоп, мобилка, API) — вопрос именно про то, что уникально для Web из-за природы браузера, HTTP и DOM.

## Пример / когда применяется

Пример чек-листа для кроссбраузерного и адаптивного тестирования интернет-магазина:

| Проверка | Chrome | Firefox | Safari | Мобильный (375px) |
|---|---|---|---|---|
| Отображение каталога | ОК | ОК | Смещена сетка | ОК |
| Кнопка "В корзину" кликабельна | ОК | ОК | ОК | Перекрыта хедером |
| Работа кнопки "Назад" после оформления заказа | ОК | Дублирует заказ | ОК | ОК |

Такой баг ("Назад" дублирует заказ) — классический пример проблемы, специфичной именно для Web, связанной с кэшированием форм браузером при навигации назад.

## На что смотрит интервьюер

Интервьюер хочет услышать конкретные Web-специфичные виды (кроссбраузерность, адаптивность, работа с URL/cookies/кэшем, SEO), а не общий список видов тестирования вообще (функциональное, нагрузочное и т.п., которые применимы к любому ПО). Плюсом будет упоминание инструментов (BrowserStack, Lighthouse, axe для accessibility). Красный флаг — если кандидат перечисляет только общие виды тестирования без связи со спецификой браузера/HTTP/DOM.
