# Какие знаете форматы передачи данных?

**Коротко:** наиболее распространённые форматы передачи данных — JSON, XML, YAML, CSV, Protocol Buffers, а также бинарные форматы вроде MessagePack и Avro; выбор формата зависит от требований к читаемости, компактности и производительности.

## Развёрнутый ответ

Форматы передачи данных можно разделить на текстовые (человекочитаемые) и бинарные (компактные, но нечитаемые напрямую).

**Текстовые форматы:**

- **JSON (JavaScript Object Notation)** — самый распространённый формат для REST API, компактный, легко парсится, поддерживает вложенные структуры, массивы, базовые типы данных.
- **XML (eXtensible Markup Language)** — язык разметки с тегами, атрибутами и поддержкой схем валидации (XSD), используется в SOAP и enterprise-системах.
- **YAML (YAML Ain't Markup Language)** — человекочитаемый формат, часто используется для конфигурационных файлов (Docker Compose, Kubernetes-манифесты, CI/CD пайплайны), поддерживает комментарии, ссылки на другие узлы, менее многословен чем XML.
- **CSV (Comma-Separated Values)** — простой табличный формат, где значения разделены запятыми, используется для экспорта/импорта табличных данных (Excel, отчёты, выгрузки из БД).
- **HTML (HyperText Markup Language)** — формат разметки для отображения содержимого в браузере, иногда передаётся как часть ответа сервера (например, серверный рендеринг).

**Бинарные форматы:**

- **Protocol Buffers (protobuf)** — бинарный формат сериализации от Google, компактнее и быстрее JSON, требует заранее описанной схемы (`.proto`-файл), используется в gRPC.
- **MessagePack** — бинарный аналог JSON, компактнее текстового JSON при той же структуре данных.
- **Avro** — бинарный формат сериализации от Apache, часто используется в системах обработки больших данных (Kafka, Hadoop), поддерживает эволюцию схемы.
- **BSON (Binary JSON)** — бинарное расширение JSON, используется внутри MongoDB для хранения документов.

Выбор формата зависит от контекста:
- если важна читаемость человеком и простота отладки — JSON, YAML, XML;
- если важна максимальная производительность и компактность — Protocol Buffers, Avro, MessagePack;
- если данные табличные и нужен простой экспорт/импорт — CSV;
- если нужна валидация через строгую схему — XML+XSD или Protobuf.

## Пример / когда применяется

JSON (REST API):
```json
{ "id": 42, "name": "Ivan" }
```

YAML (конфигурация Kubernetes):
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
```

CSV (экспорт данных):
```csv
id,name,email
1,Ivan,ivan@example.com
2,Petr,petr@example.com
```

Protocol Buffers (схема):
```protobuf
message User {
  int32 id = 1;
  string name = 2;
}
```

JSON применяется в REST API и мобильных приложениях; YAML — в DevOps-конфигурациях (Docker, Kubernetes, GitHub Actions); CSV — при выгрузке отчётов из тестируемой системы для последующей проверки; Protobuf/gRPC — в высоконагруженных внутренних сервисах, где важна скорость сериализации.

## На что смотрит интервьюер

- Умение назвать не только JSON и XML, но и хотя бы YAML, CSV, и один бинарный формат (Protobuf/MessagePack).
- Понимание компромисса между читаемостью и производительностью при выборе формата.
- Знание практического применения каждого формата (например, YAML для конфигов CI/CD, CSV для отчётов).
- Follow-up вопросы: "Почему Protobuf быстрее JSON?", "Когда бы вы выбрали YAML вместо JSON для конфигурации?", "Как тестировщик может проверить корректность CSV-выгрузки?".
- Красный флаг: ограничение ответа только JSON и XML без упоминания других форматов, используемых в современной разработке.
