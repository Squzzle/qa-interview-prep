# Охарактеризуйте каждый класс status code (1хх; 2xx; 3xx; 4xx; 5xx).

**Коротко:** HTTP-коды состояния делятся на пять классов по первой цифре: 1xx — информационные, 2xx — успех, 3xx — редирект, 4xx — ошибка клиента, 5xx — ошибка сервера.

## Развёрнутый ответ

HTTP status code — трёхзначное число в ответе сервера, сообщающее клиенту результат обработки запроса. Первая цифра определяет класс ответа:

- **1xx (Informational / Информационные)** — запрос принят, обработка продолжается, финальный ответ ещё не готов. Используются редко. Пример: `100 Continue` (сервер готов принять тело запроса), `101 Switching Protocols` (переход, например, на WebSocket).
- **2xx (Success / Успех)** — запрос успешно получен, понят и обработан. Пример: `200 OK` (стандартный успешный ответ), `201 Created` (ресурс создан, обычно ответ на POST), `204 No Content` (успех, но тела ответа нет, например при DELETE).
- **3xx (Redirection / Перенаправление)** — для завершения запроса клиенту нужно выполнить дополнительное действие, обычно перейти по другому адресу. Пример: `301 Moved Permanently` (ресурс перемещён навсегда, обновить закладки/ссылки), `302 Found` (временное перенаправление), `304 Not Modified` (ресурс не изменился, можно использовать кэш).
- **4xx (Client Error / Ошибка клиента)** — запрос содержит ошибку или не может быть обработан по вине клиента. Пример: `400 Bad Request` (некорректный синтаксис запроса), `401 Unauthorized` (нужна аутентификация), `403 Forbidden` (доступ запрещён, даже если аутентификация пройдена), `404 Not Found` (ресурс не найден), `405 Method Not Allowed`, `409 Conflict`, `422 Unprocessable Entity` (валидные синтаксически, но семантически некорректные данные).
- **5xx (Server Error / Ошибка сервера)** — сервер не смог обработать корректный запрос из-за собственной ошибки. Пример: `500 Internal Server Error` (общая ошибка сервера), `502 Bad Gateway` (проблема на промежуточном прокси/шлюзе), `503 Service Unavailable` (сервер временно недоступен, например перегружен или на обслуживании), `504 Gateway Timeout`.

Для тестировщика важно не просто помнить коды, а уметь по коду понять, "на чьей стороне" проблема — это критично при локализации бага (баг на фронте/бэке, проблема сети или инфраструктуры).

## Пример / когда применяется

Практический пример из API-тестирования:

```
POST /api/users
{
  "email": "invalid-email"
}
```

Ожидаемый ответ:

```
HTTP/1.1 400 Bad Request
{
  "error": "email должен быть в корректном формате"
}
```

Если вместо этого сервер вернёт `500 Internal Server Error` — это баг: сервер должен был провалидировать входные данные и вернуть `4xx`, а не падать с необработанным исключением.

Другой пример — при удалении несуществующего ресурса `DELETE /api/users/999` ожидается `404 Not Found`, а не `200 OK`.

## На что смотрит интервьюер

- Знание конкретных распространённых кодов внутри каждого класса (200, 201, 204, 301, 302, 304, 400, 401, 403, 404, 500, 502, 503), а не только общее описание классов.
- Понимание разницы `401` (не авторизован) и `403` (авторизован, но нет прав) — частая путаница.
- Понимание разницы `301` и `302` (постоянный/временный редирект) и их влияния на SEO/кэш.
- Follow-up: "Чем отличается 502 от 504?", "Какой код вернётся при неверном пароле?", "Как бы вы протестировали обработку 500-х ошибок?".
- Красный флаг — если кандидат считает, что любая ошибка — это `500`, независимо от природы проблемы.
