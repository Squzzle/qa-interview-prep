# Как взаимодействуют клиентская библиотека Selenium, драйвер браузера и сам браузер?

**Коротко:** клиентская библиотека формирует HTTP-запросы по протоколу WebDriver (W3C), отправляет их на локальный сервер драйвера браузера, драйвер транслирует команды в нативные вызовы браузера через его встроенный debugging/автоматизационный протокол, а результат возвращается обратно по цепочке.

## Развёрнутый ответ

Архитектура Selenium WebDriver построена на клиент-серверной модели:

1. **Клиентская библиотека (Selenium bindings)** — это код на языке программирования (Java, Python, C#, JS), который использует API `WebDriver` (например, `driver.get(url)`, `driver.findElement(...)`). Каждый такой вызов преобразуется в HTTP-запрос согласно спецификации протокола WebDriver W3C — JSON-объект с описанием команды (endpoint, метод, тело запроса).

2. **Драйвер браузера (browser driver)** — отдельный исполняемый файл (`chromedriver`, `geckodriver`, `msedgedriver`, `safaridriver`), который запускается как локальный HTTP-сервер (обычно на `localhost` со случайным портом). Он принимает HTTP-запросы от клиентской библиотеки, разбирает их и транслирует в команды, понятные конкретному браузеру. Драйвер выступает посредником (proxy) между стандартизированным протоколом и внутренним API браузера.

3. **Браузер** — получает команды от драйвера через собственный протокол автоматизации: у Chrome/Edge это DevTools Protocol (CDP), у Firefox — Marionette. Браузер выполняет реальные действия: открывает страницу, кликает, вводит текст, — и возвращает результат (например, DOM-состояние, скриншот, статус выполнения) обратно драйверу.

4. **Обратный путь.** Драйвер формирует HTTP-ответ (JSON) и отправляет его клиентской библиотеке, которая десериализует его в объект языка программирования (например, `WebElement`).

Ключевой момент архитектуры W3C WebDriver (в отличие от устаревшего JSON Wire Protocol): единый стандартизированный формат команд и ответов, что обеспечивает совместимость между разными языковыми биндингами и разными браузерами без необходимости писать код под конкретный браузер.

## Пример / когда применяется

Пример цепочки для команды клика по кнопке:

```java
WebElement button = driver.findElement(By.id("submit"));
button.click();
```

Что происходит под капотом:

1. Java-клиент сериализует команду `click` с `elementId` в JSON и отправляет `POST http://localhost:9515/session/{sessionId}/element/{elementId}/click` на chromedriver.
2. chromedriver транслирует это в команду CDP `Input.dispatchMouseEvent`.
3. Chrome выполняет клик и возвращает статус.
4. chromedriver отправляет HTTP-ответ `{"value": null}` обратно клиенту, означающий успешное выполнение.

## На что смотрит интервьюер

- Понимание разницы между клиентской библиотекой, драйвером и браузером — многие путают драйвер с самим браузером.
- Знание, что связь идёт по HTTP и протоколу WebDriver (W3C), а не напрямую.
- Упоминание, что у каждого браузера свой драйвер и свой внутренний протокол (CDP, Marionette).
- Follow-up: чем отличается W3C WebDriver от JSON Wire Protocol, что такое Selenium Grid и как он вписывается в эту схему (добавляет ещё один сетевой хоп — Hub/Node).
- Красный флаг: утверждение, что Selenium «сам» управляет браузером напрямую без драйвера.
