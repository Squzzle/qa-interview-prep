# Что такое bash и batch скрипты? Зачем их используют?

**Коротко:** bash-скрипты — это исполняемые текстовые файлы с последовательностью команд Linux/Unix shell (`.sh`), а batch-скрипты — аналог для Windows CMD (`.bat`/`.cmd`); оба служат для автоматизации повторяющихся задач без ручного ввода команд.

## Развёрнутый ответ

Bash (Bourne Again SHell) — самый распространённый интерпретатор командной строки в Linux/macOS/Unix-системах. Bash-скрипт — это текстовый файл, начинающийся со строки `#!/bin/bash` (`shebang`, указывающей, каким интерпретатором выполнять файл), содержащий последовательность shell-команд, условия, циклы, функции и переменные. Bash-скрипты поддерживают полноценные конструкции программирования: условные операторы (`if`/`else`), циклы (`for`, `while`), функции, обработку аргументов командной строки, работу с кодами возврата (`exit code`) для определения успеха/неудачи выполнения.

Batch-скрипты — аналогичный механизм автоматизации для Windows, файлы с расширением `.bat` или `.cmd`, выполняемые интерпретатором `cmd.exe`. Синтаксис принципиально отличается от bash (переменные через `%VAR%`, циклы через `FOR /L` или `FOR /F`, условия через `IF`), а возможности несколько ограниченнее, чем у современных альтернатив вроде PowerShell, но batch остаётся востребованным для простых задач автоматизации на легаси-системах Windows.

Зачем используют оба вида скриптов:
- **Автоматизация рутинных задач.** Запуск последовательности команд без ручного ввода — сборка проекта, запуск тестов, деплой, очистка временных файлов.
- **Воспроизводимость.** Один и тот же скрипт даёт одинаковый результат при каждом запуске, устраняя человеческий фактор и ошибки при ручном вводе команд.
- **Интеграция в CI/CD.** Пайплайны CI (Jenkins, GitLab CI, GitHub Actions) по сути выполняют shell/batch-скрипты на каждом этапе (сборка, тесты, деплой).
- **Обвязка (glue code) для инструментов.** Скрипты часто используются как связующий слой между разными утилитами — например, скрипт, который поднимает Docker-контейнеры, ждёт готовности сервиса, запускает автотесты и собирает отчёт, а затем гасит контейнеры.
- **Условная логика и обработка ошибок.** В отличие от простого списка команд, скрипты позволяют реагировать на код возврата предыдущей команды и принимать решения (например, повторить попытку при таймауте).

## Пример / когда применяется

Bash-скрипт для запуска автотестов с проверкой готовности окружения — типичная задача QA-автоматизатора в CI:

```bash
#!/bin/bash
set -e  # прервать выполнение при первой же ошибке

echo "Поднимаем тестовое окружение..."
docker-compose up -d

echo "Ждём готовности сервиса..."
for i in {1..30}; do
  if curl -s http://localhost:8080/health | grep -q "UP"; then
    echo "Сервис готов"
    break
  fi
  sleep 2
done

echo "Запускаем автотесты..."
mvn test -Dsuite=regression
TEST_EXIT_CODE=$?

echo "Останавливаем окружение..."
docker-compose down

exit $TEST_EXIT_CODE
```

Аналогичный batch-скрипт для Windows, запускающий автотесты и проверяющий результат:

```bat
@echo off
echo Запуск автотестов...
call mvn test -Dsuite=smoke

if %ERRORLEVEL% NEQ 0 (
    echo Тесты упали, код ошибки: %ERRORLEVEL%
    exit /b 1
) else (
    echo Все тесты прошли успешно
)
```

## На что смотрит интервьюер

Интервьюер оценивает, понимает ли кандидат разницу платформ (bash — Linux/Unix/macOS, batch — Windows) и умеет ли объяснить практическую пользу скриптов в контексте CI/CD и автоматизации тестирования, а не только в общих чертах. Хороший сигнал — упоминание кодов возврата (`exit code`, `$?`, `%ERRORLEVEL%`) для интеграции скрипта в пайплайн и принятия решений об успехе/провале этапа. Частая ошибка — не знать, как передать параметры в скрипт (`$1`, `$2` в bash; `%1`, `%2` в batch) или не понимать, зачем нужен `set -e` для прерывания скрипта при ошибке. Может прозвучать вопрос о том, чем PowerShell превосходит batch на Windows.
