# Что такое Configuration Management?

**Коротко:** Configuration Management (управление конфигурациями) — это практика систематического отслеживания, контроля и поддержания согласованного состояния конфигурации систем, окружений и кода на протяжении всего их жизненного цикла, чаще всего реализуемая через декларативные инструменты и подход `Infrastructure as Code`.

## Развёрнутый ответ

Configuration Management решает проблему дрейфа конфигурации (`configuration drift`) — ситуации, когда с течением времени серверы, окружения или сборки начинают отличаться друг от друга из-за ручных изменений, разных версий пакетов, забытых патчей. Без управления конфигурациями окружения (dev, test, staging, prod) постепенно расходятся, что приводит к классической проблеме «работает у меня, но не работает на проде/у тестировщика».

Ключевые принципы Configuration Management:
- **Версионирование конфигурации.** Файлы конфигурации, скрипты установки, манифесты хранятся в системе контроля версий (Git), как и обычный код — это даёт историю изменений, возможность отката и code review.
- **Декларативность.** Инструменты описывают желаемое конечное состояние системы («на сервере должен быть установлен nginx версии 1.25 и открыт порт 443»), а не последовательность императивных команд — инструмент сам определяет, что нужно изменить, чтобы достичь этого состояния.
- **Идемпотентность.** Повторное применение одной и той же конфигурации к системе не должно менять результат — можно безопасно запускать конфигурацию многократно.
- **Автоматизация и повторяемость.** Одна и та же конфигурация применяется одинаково к десяткам или сотням серверов без ручного вмешательства, что устраняет человеческий фактор.

Основные инструменты Configuration Management: `Ansible` (agentless, работает по SSH, конфигурация в YAML), `Puppet` и `Chef` (требуют агента на управляемых машинах, конфигурация в собственном DSL/Ruby), `SaltStack`. Отдельно стоит `Terraform` — он относится скорее к `Provisioning`/`Infrastructure as Code` для создания самих ресурсов (VM, сети, диски), но на практике эти инструменты часто используют вместе: Terraform создаёт инфраструктуру, а Ansible донастраивает её конфигурацию.

## Пример / когда применяется

В контексте QA-автоматизации Configuration Management используют, чтобы гарантированно одинаково настраивать тестовые окружения и CI-агенты — одинаковые версии браузеров, драйверов, переменных окружения на всех раннерах.

```yaml
# Пример Ansible playbook: настройка тестового агента с Chrome и ChromeDriver
- hosts: test_agents
  become: true
  tasks:
    - name: Установить Google Chrome
      apt:
        name: google-chrome-stable
        state: present
        update_cache: true

    - name: Установить нужную версию ChromeDriver
      get_url:
        url: "https://storage.googleapis.com/chrome-for-testing-public/126.0.6478.126/linux64/chromedriver-linux64.zip"
        dest: /tmp/chromedriver.zip

    - name: Распаковать ChromeDriver в /usr/local/bin
      unarchive:
        src: /tmp/chromedriver.zip
        dest: /usr/local/bin
        remote_src: true
```

Такой playbook можно применить к любому количеству CI-агентов, гарантируя, что все они настроены абсолютно одинаково, что убирает флаки тестов из-за расхождения версий браузера/драйвера.

## На что смотрит интервьюер

Интервьюер проверяет, понимает ли кандидат разницу между Configuration Management и Provisioning (управление уже созданными ресурсами vs создание самих ресурсов), а также знает ли конкретные инструменты (Ansible, Puppet, Chef) и умеет объяснить принципы декларативности и идемпотентности своими словами. Частая ошибка — путать Configuration Management с системой контроля версий кода приложения или называть только один инструмент без понимания концепции в целом. Могут спросить про конкретный опыт настройки тестовых окружений через такие инструменты в предыдущих проектах.
