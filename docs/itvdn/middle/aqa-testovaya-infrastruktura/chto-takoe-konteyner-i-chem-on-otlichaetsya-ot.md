# Что такое контейнер и чем он отличается от виртуальной машины?

**Коротко:** контейнер — это изолированный процесс, использующий ядро хостовой ОС и механизмы `namespaces`/`cgroups`, тогда как виртуальная машина эмулирует полноценное железо и запускает собственное независимое ядро ОС поверх гипервизора.

## Развёрнутый ответ

Контейнер — это способ изоляции процессов на уровне операционной системы. Он не содержит своей ОС и своего ядра — все контейнеры на хосте используют одно и то же ядро хостовой операционной системы (обычно Linux). Изоляция достигается механизмами ядра: `namespaces` (изолируют видимость процессов, сети, файловой системы, пользователей, hostname) и `cgroups` (ограничивают и учитывают потребление CPU, памяти, I/O). Файловая система контейнера строится из слоёв образа (image layers) по принципу `copy-on-write`, что делает образы компактными и быстро разворачиваемыми.

Виртуальная машина, в отличие от контейнера, эмулирует полный набор виртуального железа (виртуальный CPU, память, диск, сетевую карту) через гипервизор (`Type 1` — bare-metal, например ESXi, KVM; `Type 2` — работает поверх хостовой ОС, например VirtualBox). На этом виртуальном железе устанавливается полноценная гостевая ОС со своим ядром, драйверами и системными службами, полностью независимая от хостовой ОС.

Ключевые различия:
- **Уровень изоляции.** VM изолирована на уровне гипервизора и железа — самая сильная изоляция, вплоть до разных ОС (Windows-гость на Linux-хосте). Контейнер изолирован на уровне ядра ОС — все контейнеры хоста делят одно ядро, поэтому нельзя запустить Windows-контейнер на Linux-хосте (и наоборот) без дополнительной VM-прослойки.
- **Вес и скорость.** Контейнер запускается за секунды, весит десятки-сотни мегабайт, накладные расходы на ресурсы минимальны. VM запускается за десятки секунд-минуты, весит гигабайты (полная копия ОС), потребляет фиксированный объём выделенных ресурсов.
- **Плотность размещения.** На одном хосте можно запустить в разы больше контейнеров, чем VM, при том же объёме ресурсов.
- **Переносимость.** Контейнерный образ (`Docker image`) переносим между средами с совместимым ядром «как есть» (`build once, run anywhere`), VM переносится как целый образ диска, что тяжелее.
- **Безопасность.** VM даёт более строгую изоляцию (полноценная граница на уровне гипервизора), контейнер теоретически более уязвим к атакам через общее ядро (`container escape`), поэтому в продакшене их часто комбинируют — контейнеры внутри VM.

## Пример / когда применяется

Типичный сценарий в автоматизации тестирования — поднять окружение с приложением, базой данных и очередью сообщений через `docker-compose`, чтобы каждый тест-ран стартовал с чистого изолированного окружения:

```bash
docker run -d --name selenium-hub -p 4444:4444 selenium/hub:latest
docker run -d --name chrome-node --link selenium-hub:hub selenium/node-chrome:latest
docker run -d --name test-db -e POSTGRES_PASSWORD=test -p 5432:5432 postgres:15
```

Виртуальные машины в тестировании чаще используют, когда нужно проверить поведение на разных ОС целиком (Windows Server, разные дистрибутивы Linux) или изолировать окружение на уровне гипервизора для строгих требований безопасности — например, тестовые стенды в закрытом контуре банка.

## На что смотрит интервьюер

Ожидается чёткое понимание, что контейнеры не «лёгкие виртуальные машины» — это разные механизмы изоляции. Частая ошибка кандидатов — сказать, что контейнер «содержит свою мини-ОС». Интервьюер может спросить про `namespaces` и `cgroups` конкретно, про то, почему на Linux-хосте нельзя нативно запустить Windows-контейнер, а также про сценарии использования Docker в CI/CD для автоматизации тестирования (параллельные изолированные запуски тестов, Selenium Grid в контейнерах, `Testcontainers`).
