# Что такое Provisioning?

**Коротко:** Provisioning — это процесс создания, выделения и подготовки к работе вычислительных ресурсов (серверов, виртуальных машин, сетей, хранилищ) с нуля, до того как на них начинается настройка конфигурации приложения.

## Развёрнутый ответ

Provisioning отвечает на вопрос «откуда вообще берётся сервер/ресурс», в отличие от Configuration Management, который отвечает на вопрос «как этот уже существующий сервер настроен». Это первый этап жизненного цикла инфраструктуры: заказать/создать вычислительный ресурс, выделить ему сеть, диск, базовый образ ОС, и только после этого передать управление инструментам конфигурации.

Различают несколько видов provisioning:
- **Server/Resource Provisioning** — создание виртуальной машины, контейнера или физического сервера с нуля: выбор образа ОС, размера (CPU/RAM), диска.
- **Network Provisioning** — настройка сетевой инфраструктуры для новых ресурсов: VPC, подсети, security groups, DNS-записи.
- **User Provisioning** — создание учётных записей пользователей и назначение прав доступа в системе.
- **Cloud Provisioning** — автоматизированное создание облачных ресурсов через API провайдера (AWS, GCP, Azure) вместо ручного создания через консоль.

Основной современный подход к provisioning — `Infrastructure as Code` (IaC), когда инфраструктура описывается декларативно в виде кода и создаётся/изменяется через этот код, а не вручную через UI облачного провайдера. Ключевой инструмент здесь — `Terraform` (мультиоблачный, декларативный, хранит состояние инфраструктуры в state-файле), а также облачно-специфичные инструменты вроде `AWS CloudFormation` или `Pulumi` (provisioning через обычный код на TypeScript/Python/Go вместо DSL).

Важно чётко разграничивать на собеседовании: Provisioning создаёт ресурс (сервер существует), Configuration Management настраивает уже существующий ресурс (на сервере установлен нужный софт и открыты нужные порты). На практике эти этапы часто идут последовательно в одном пайплайне: Terraform создаёт VM → Ansible донастраивает её ПО.

## Пример / когда применяется

В автоматизации тестирования provisioning используют для динамического создания тестовых окружений под конкретный прогон и их последующего удаления, чтобы не платить за простаивающие ресурсы и не накапливать дрейф состояния:

```hcl
# Пример Terraform: создание EC2-инстанса для тестового CI-раннера
resource "aws_instance" "test_runner" {
  ami           = "ami-0c94855ba95c71c99" # Ubuntu 22.04
  instance_type = "t3.medium"

  tags = {
    Name = "qa-selenium-runner"
  }

  vpc_security_group_ids = [aws_security_group.test_sg.id]
}

resource "aws_security_group" "test_sg" {
  name = "qa-test-sg"

  ingress {
    from_port   = 4444
    to_port     = 4444
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

После выполнения `terraform apply` создаётся готовая VM, на которую затем Ansible накатит конфигурацию (браузеры, драйверы), а после завершения тестового рана `terraform destroy` уничтожит ресурс, экономя расходы на инфраструктуру.

## На что смотрит интервьюер

Ключевая проверка — умеет ли кандидат чётко разграничить Provisioning и Configuration Management, а не смешивать их в одно понятие. Хорошим ответом считается упоминание Infrastructure as Code и конкретных инструментов (Terraform, CloudFormation), а также понимание, что provisioning часто является динамическим и временным процессом в CI/CD (создать окружение под тест-ран → выполнить тесты → уничтожить окружение). Частый follow-up — «как это применимо к вашим тестовым окружениям на предыдущем проекте», где ожидается конкретный практический пример, а не только теория.
