# Что такое SSH и как им пользоваться?

**Коротко:** SSH (Secure Shell) — это криптографический сетевой протокол для безопасного удалённого управления машиной через незащищённую сеть, обеспечивающий шифрование трафика, аутентификацию и целостность данных.

## Развёрнутый ответ

SSH заменил устаревшие небезопасные протоколы удалённого доступа (`Telnet`, `rlogin`), которые передавали данные, включая пароли, в открытом виде. SSH работает по модели клиент-сервер: на удалённой машине запущен SSH-демон (`sshd`), слушающий по умолчанию TCP-порт 22, а клиент подключается к нему командой `ssh`.

Ключевые возможности и механизмы SSH:
- **Шифрование трафика.** Весь обмен данными между клиентом и сервером шифруется, что защищает от перехвата (man-in-the-middle) в незащищённых сетях.
- **Аутентификация по паролю.** Простейший способ входа — ввод пароля пользователя удалённой машины, но менее безопасный и неудобный для автоматизации.
- **Аутентификация по ключевой паре (public/private key).** Более безопасный и распространённый в CI/CD способ: генерируется пара ключей (`ssh-keygen`), публичный ключ добавляется на сервер в файл `~/.ssh/authorized_keys`, приватный ключ остаётся у клиента и никогда не передаётся по сети. Аутентификация происходит через криптографическое доказательство владения приватным ключом, без передачи пароля.
- **Проброс портов (port forwarding/tunneling).** SSH позволяет создавать защищённые туннели, перенаправляя трафик с локального порта на удалённый и наоборот — полезно, когда сервис на удалённой машине доступен только локально (`localhost`), но нужно достучаться до него со своей машины.
- **SCP/SFTP.** Поверх SSH-протокола работают утилиты безопасного копирования файлов (`scp`) и защищённой передачи файлов (`sftp`).

Типовые команды использования SSH:
- `ssh user@host` — подключиться к удалённой машине под указанным пользователем.
- `ssh -p 2222 user@host` — подключиться на нестандартный порт.
- `ssh -i ~/.ssh/id_rsa user@host` — подключиться, явно указав приватный ключ.
- `scp file.txt user@host:/remote/path/` — скопировать файл на удалённую машину.
- `ssh -L 8080:localhost:80 user@host` — проброс локального порта 8080 на порт 80 удалённой машины.

## Пример / когда применяется

В автоматизации тестирования SSH используют для удалённого выполнения тестов на CI-агентах, доступа к тестовым серверам для сбора логов, деплоя тестового окружения и настройки CI-раннеров без пароля (по ключу), что критично для неинтерактивных скриптов:

```bash
# Сгенерировать пару SSH-ключей для CI-агента (без пароля, для автоматизации)
ssh-keygen -t ed25519 -f ~/.ssh/ci_deploy_key -N ""

# Скопировать публичный ключ на удалённый тестовый сервер
ssh-copy-id -i ~/.ssh/ci_deploy_key.pub qa-user@test-server.internal

# Подключиться к тестовому серверу по ключу и удалённо запустить прогон тестов
ssh -i ~/.ssh/ci_deploy_key qa-user@test-server.internal \
  "cd /opt/qa/project && mvn test -Dsuite=smoke"

# Скачать лог-файл прогона с удалённого сервера себе на машину для анализа падения
scp -i ~/.ssh/ci_deploy_key qa-user@test-server.internal:/opt/qa/project/target/surefire-reports/*.txt ./logs/
```

## На что смотрит интервьюер

Интервьюер проверяет понимание разницы между аутентификацией по паролю и по ключевой паре, а также умение объяснить, зачем в CI/CD-пайплайнах используют именно ключи, а не пароли (неинтерактивность, безопасность, отсутствие секрета в открытом виде в скриптах). Частая ошибка — путать публичный и приватный ключ местами (публичный кладётся на сервер, приватный остаётся у клиента и никогда не передаётся). Может прозвучать follow-up про проброс портов или про то, как безопасно хранить SSH-ключи в CI-системе (через секреты/vault, а не в коде репозитория).
