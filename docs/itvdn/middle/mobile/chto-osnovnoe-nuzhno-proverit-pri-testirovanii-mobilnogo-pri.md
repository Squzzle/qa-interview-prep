# Что основное нужно проверить при тестировании мобильного приложения?

**Коротко:** проверяют функциональность, UI/UX на разных экранах и ОС, производительность, работу с сетью (в т.ч. её отсутствие), потребление ресурсов (батарея, память, трафик), прерывания (звонки, уведомления), установку/обновление/удаление и специфичные для платформы права доступа.

## Развёрнутый ответ

Тестирование мобильного приложения шире, чем тестирование обычного веб- или десктоп-приложения, потому что добавляется огромное количество факторов окружения. Основные направления проверки:

- **Функциональное тестирование** — соответствие бизнес-логики требованиям: регистрация, авторизация, основные пользовательские сценарии, обработка ошибок.
- **UI/UX и совместимость** — корректное отображение на разных размерах экрана, разрешениях, плотностях пикселей (`dpi`), ориентациях (portrait/landscape), версиях ОС (Android/iOS) и разных производителях устройств (кастомные прошивки, например MIUI, EMUI).
- **Установка, обновление, удаление** (Install/Update/Uninstall тестирование) — чистая установка, обновление с сохранением данных, повторная установка, откат к предыдущей версии, поведение при удалении (очистка данных, кэша).
- **Работа с сетью** — поведение при отсутствии интернета, при слабом/нестабильном соединении (2G/3G/Wi-Fi), при переключении сетей (Wi-Fi ↔ мобильная сеть), таймауты, повторные запросы, офлайн-режим.
- **Прерывания (Interruption testing)** — входящий звонок, SMS, push-уведомление, будильник, разрядка батареи, подключение зарядки, блокировка экрана, сворачивание приложения — приложение должно корректно восстанавливать состояние.
- **Производительность и ресурсы** — потребление CPU, памяти, батареи, трафика; скорость запуска (cold/warm/hot start); отсутствие лагов при скролле/анимациях; отсутствие утечек памяти.
- **Локализация** — корректность переводов, форматов дат/чисел/валют, поддержка RTL-языков, обрезание текста в UI-элементах.
- **Безопасность** — хранение чувствительных данных (токены, пароли), передача данных по HTTPS, права доступа (permissions), защита от реверс-инжиниринга.
- **Совместимость с аппаратными датчиками** — GPS, камера, акселерометр, отпечаток пальца/Face ID, Bluetooth, NFC — в зависимости от функциональности приложения.
- **Push-уведомления** — доставка, отображение, обработка тапа по уведомлению в разных состояниях приложения (foreground/background/killed).
- **Тестирование в разных состояниях устройства** — низкий заряд батареи, мало свободной памяти на устройстве, включённый режим энергосбережения, разные системные настройки (шрифт, тёмная тема, языки).

## Пример / когда применяется

Типичный чек-лист перед релизом мобильного приложения:

| Категория | Примеры проверок |
|---|---|
| Установка | Чистая установка, обновление со старой версии, downgrade, удаление |
| Сеть | Работа без интернета, обрыв связи посреди запроса, медленная сеть |
| Прерывания | Входящий звонок во время видео, сворачивание при загрузке файла |
| UI | Разные размеры экрана, повороты, тёмная/светлая тема |
| Ресурсы | Расход батареи в фоне, память при длительной работе |
| Безопасность | Хранение токена авторизации, HTTPS-only трафик |

## На что смотрит интервьюер

Интервьюер ожидает, что кандидат не ограничится «проверить кнопки и формы», а вспомнит про специфику мобильной платформы: прерывания, сеть, ресурсы, установку/обновление и права доступа. Красный флаг — если кандидат не может назвать ни одного мобильно-специфичного вида тестирования (interruption testing, install/uninstall testing). Частый follow-up: «А как бы вы тестировали именно потребление батареи?» или «Как проверить, что приложение не течёт по памяти?».
