# Что такое утечки памяти? Как найти?

**Коротко:** утечка памяти — это ситуация, когда объект, который больше не нужен приложению, продолжает удерживаться в памяти живой ссылкой и не может быть собран сборщиком мусора, что со временем приводит к росту потребления памяти, замедлению работы и краху приложения с `OutOfMemoryError`.

## Развёрнутый ответ

В управляемых средах (Java/Kotlin на Android, Objective-C/Swift с ARC на iOS) память освобождается автоматически, когда на объект больше не остаётся ссылок. Утечка происходит, когда «мёртвый» с точки зрения бизнес-логики объект (например, уничтоженная активити) на самом деле продолжает быть достижим через какую-то долгоживущую ссылку, и поэтому сборщик мусора не может его удалить.

**Типичные причины утечек на Android:**
- Статические ссылки на `Context`/`Activity`/`View` — например, `static Activity instance` или синглтон, хранящий ссылку на активити.
- Незарегистрированные (не отписанные) листенеры и колбэки — подписка на события (`BroadcastReceiver`, `LocationListener`, EventBus) без соответствующей отписки в `onDestroy`/`onStop`.
- Анонимные внутренние классы и non-static inner классы (в Java), неявно хранящие ссылку на внешний класс (активити), например анонимный `Handler` или `AsyncTask`, переживающий уничтожение активити.
- Незавершённые фоновые потоки/корутины, держащие ссылку на UI-компоненты после того, как экран уже закрыт.
- Утечки через `Bitmap` — не освобождённые вовремя большие изображения.
- Context-зависимые синглтоны, инициализированные Activity Context вместо Application Context.

**Как искать утечки памяти (инструменты и подходы):**

1. **Android Studio Profiler (Memory Profiler)** — визуализация использования кучи в реальном времени, снятие heap dump, поиск объектов, которые не должны быть живы после выполнения сценария (например, N экземпляров уничтоженной активити).
2. **LeakCanary** — библиотека от Square, автоматически детектирующая утечки активити/фрагментов в debug-сборках и показывающая цепочку ссылок (reference chain), которая удерживает объект в памяти.
3. **`adb shell dumpsys meminfo <package>`** — снимок использования памяти процессом приложения (Java heap, Native heap, Graphics и т.д.) в динамике.
4. **Ручное тестирование через повторяющиеся сценарии** — многократное открытие/закрытие одного и того же экрана (например, 30-50 раз) с последующим анализом графика памяти: если память после каждого цикла возвращается примерно к одному уровню — утечки нет, если стабильно растёт — есть утечка.
5. **MAT (Eclipse Memory Analyzer Tool)** — глубокий анализ heap dump файла для нахождения дерева ссылок, которые удерживают объект (Dominator Tree, Path to GC Roots).
6. **Проверка на `OutOfMemoryError` и ANR в crash-репортах** (Firebase Crashlytics, Google Play Console Android Vitals) — по продакшн-статистике можно выявить экраны/сценарии с наибольшим числом OOM-крашей.

Для тестировщика важно уметь не только обнаружить сам факт утечки (рост памяти), но и грамотно составить баг-репорт с шагами воспроизведения, приложенным heap dump или логом LeakCanary, чтобы разработчик мог быстро найти причину.

## Пример / когда применяется

Типичный тест-кейс на обнаружение утечки:

```text
1. Установить debug-сборку приложения с интегрированным LeakCanary.
2. Открыть экран "Профиль" → нажать "Назад" 30 раз подряд (открыть-закрыть).
3. LeakCanary автоматически покажет уведомление с найденной утечкой
   и деревом ссылок вида:
   ProfileActivity ← EventBus$stickyEvents ← static field leaks GC root
4. Приложить снимок уведомления и дерево ссылок в баг-репорт.
```

Пример типичной причины утечки в коде (для контекста тестировщику при общении с разработчиком):

```java
public class MyLocationManager {
    private static MyLocationManager instance;
    private Context context; // утечка, если сюда передана Activity, а не Application context

    public static MyLocationManager getInstance(Context context) {
        if (instance == null) {
            instance = new MyLocationManager(context); // Activity "застревает" в синглтоне навсегда
        }
        return instance;
    }
}
```

## На что смотрит интервьюер

Интервьюер хочет услышать не только определение утечки памяти, но и конкретные инструменты (LeakCanary, Android Profiler, `dumpsys meminfo`) и практический метод её выявления (повторяющиеся сценарии с наблюдением за графиком памяти). Плюсом будет знание типичных паттернов-причин (статические ссылки на Activity, неотписанные листенеры). Красный флаг — общие фразы без конкретных инструментов или непонимание разницы между обычным ростом памяти при использовании и утечкой.
