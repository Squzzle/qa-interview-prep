# Как происходит перехват трафика http/https для мобильных устройств?

**Коротко:** мобильное устройство настраивается на использование прокси-сервера (например, Charles, Fiddler, mitmproxy, Proxyman), который перехватывает HTTP-трафик напрямую, а для HTTPS выполняет MITM-расшифровку через собственный корневой сертификат, установленный в доверенное хранилище устройства.

## Развёрнутый ответ

Перехват трафика нужен, чтобы видеть запросы и ответы приложения к серверу: заголовки, тело запроса/ответа, коды статусов, тайминги — это критично для тестирования API-взаимодействия, отладки багов и security-тестирования.

**Схема для HTTP:**
1. На компьютере запускается прокси-инструмент (Charles Proxy, Fiddler, mitmproxy, HTTP Toolkit, Proxyman и т.п.), который слушает определённый порт.
2. Мобильное устройство (и компьютер) подключаются к одной Wi-Fi сети.
3. В настройках Wi-Fi на устройстве указывается ручной прокси: IP-адрес компьютера и порт прокси-инструмента.
4. Весь HTTP-трафик с устройства идёт через прокси, и он отображается в инструменте в открытом виде.

**Схема для HTTPS (MITM — man-in-the-middle):**
Так как HTTPS шифрует трафик end-to-end, обычный прокси не может прочитать содержимое. Чтобы это обойти:
1. Прокси-инструмент генерирует собственный корневой сертификат (root CA), которым он подписывает «на лету» сертификаты для каждого домена, к которому обращается приложение.
2. Этот корневой сертификат нужно установить и явно доверять ему на мобильном устройстве (Settings → Security → Install certificate, либо через профиль на iOS).
3. После установки сертификата приложение устанавливает HTTPS-соединение не напрямую с сервером, а с прокси, который расшифровывает трафик, показывает его, и заново зашифровывает при пересылке настоящему серверу — устройство «думает», что общается напрямую с сервером, так как ему доверяет сертификат прокси.
4. Начиная с Android 7+ приложения по умолчанию доверяют только системным (предустановленным) CA, а не пользовательским — поэтому многие современные приложения не перехватываются без дополнительных шагов (см. ниже).

**Ограничения и обход SSL Pinning:**
Некоторые приложения используют **certificate/SSL pinning** — жёстко «прибивают» ожидаемый сертификат или публичный ключ сервера внутри кода, и не доверяют даже валидному, но подменённому сертификату прокси. Для тестирования таких приложений обычно требуется:
- собранная debug/staging-версия приложения без pinning;
- патчинг APK (например, через `objection`, Frida) для отключения pinning на rooted-устройстве или эмуляторе;
- либо явная договорённость с разработчиками об отключении pinning в тестовой сборке.

## Пример / когда применяется

Типичная настройка перехвата на Android-эмуляторе через Charles Proxy:

```text
1. Запустить Charles на компьютере, включить Proxy → Proxy Settings (порт 8888).
2. В настройках Wi-Fi эмулятора/устройства указать:
   Proxy hostname: <IP компьютера>
   Proxy port: 8888
3. Открыть в браузере устройства chls.pro/ssl — скачать и установить сертификат Charles.
4. Settings → Security → Trusted credentials → убедиться, что сертификат Charles в списке доверенных.
5. Открыть тестируемое приложение — трафик появляется в Charles в расшифрованном виде.
```

Используется для: отладки API-запросов, проверки корректности передаваемых параметров, тестирования обработки ошибок сервера (эмуляция 4xx/5xx через Charles Rewrite/Map Local), security-аудита (проверка отсутствия чувствительных данных в открытом виде).

## На что смотрит интервьюер

Интервьюер проверяет понимание того, ПОЧЕМУ обычный HTTP прокси не работает «из коробки» для HTTPS и как решается доверие к сертификату. Хороший ответ обязательно упоминает MITM-механизм и необходимость установки корневого сертификата на устройство. Опытный кандидат также знает про SSL pinning как препятствие и способы его обхода (Frida, объектион, патч APK). Красный флаг — если кандидат говорит, что HTTPS "просто перехватывается так же, как HTTP", не упоминая сертификаты.
