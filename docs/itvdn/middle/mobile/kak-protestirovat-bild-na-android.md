# Как протестировать билд на Android?

**Коротко:** тестирование билда включает проверку корректности установки/обновления/удаления на разных версиях Android и устройствах, smoke-тестирование ключевых сценариев, проверку конфигурации сборки (версия, подпись, ProGuard/R8), совместимости и производительности перед передачей билда в дальнейшее тестирование или релиз.

## Развёрнутый ответ

Тестирование билда (build verification / smoke testing) — это первичная проверка новой сборки приложения перед тем, как передавать её на полное регрессионное тестирование. Цель — быстро отсеять «сломанные» сборки, чтобы не тратить время команды на тестирование заведомо нерабочего билда.

**1. Проверка метаданных билда**
- Корректный `versionCode`/`versionName` — соответствуют ожидаемому релизу, инкрементированы относительно предыдущей сборки.
- Правильная подпись (signing) — debug vs release keystore, особенно важно для сборок, которые пойдут в Google Play.
- Корректный флейвор/окружение (dev/staging/prod) — билд смотрит на нужный API-эндпоинт, а не, например, staging-сборка случайно указывает на прод-базу.

**2. Установка / обновление / удаление**
- Чистая установка APK/AAB на «чистое» устройство (без предыдущих версий).
- Обновление с предыдущей продакшн-версии (upgrade path) — данные и настройки пользователя должны сохраниться (см. вопрос про миграцию данных).
- Удаление приложения — проверка, что не остаётся «мусора», и что при последующей установке всё корректно.
- Если приложение использует ProGuard/R8 (обфускация и минификация кода в release-сборке) — обязательно тестировать именно release-билд, а не только debug, так как обфускация может сломать рефлексию, сериализацию (Gson/Moshi) и т.п.

**3. Smoke-тестирование ключевых сценариев**
Прогон краткого набора критичных пользовательских путей (Critical Path): запуск приложения, авторизация, основная бизнес-функция (например, оформление заказа), выход. Если smoke-тест не проходит — билд немедленно возвращается разработчикам, дальнейшее тестирование не начинается.

**4. Совместимость (Compatibility testing)**
- Разные версии Android (минимум поддерживаемая `minSdkVersion` и последняя актуальная).
- Разные производители/прошивки устройств (Samsung, Xiaomi/MIUI, Huawei — известны своими нестандартными ограничениями фоновых процессов).
- Разные размеры экранов и плотности.

**5. Проверка размера и производительности сборки**
- Размер APK/AAB не должен неожиданно вырасти (например, из-за случайно добавленных ресурсов/библиотек).
- Время холодного старта приложения (`cold start time`) не должно деградировать относительно предыдущей сборки.
- Отсутствие явных ANR (Application Not Responding) и крашей при базовой навигации.

**6. Проверка разрешений (permissions)**
Новые/изменённые разрешения в `AndroidManifest.xml` соответствуют заявленной функциональности, запрос разрешений (runtime permissions) корректно обрабатывается на разных версиях Android (модель permissions менялась начиная с Android 6, 10, 11, 13).

**7. Логи и диагностика**
Просмотр `logcat` на предмет неожиданных `Exception`/`Error`/`ANR` даже в тех местах, где приложение внешне «работает нормально» — иногда ошибки перехватываются и скрываются от пользователя, но указывают на проблему.

## Пример / когда применяется

Чек-лист Build Verification Test (BVT) перед началом полного регресса:

```text
[ ] Билд устанавливается на чистое устройство (Android min и max поддерживаемых версий)
[ ] Билд обновляется с предыдущей продакшн-версии без потери данных
[ ] Приложение запускается без краша (cold start)
[ ] Авторизация проходит успешно
[ ] Основной бизнес-сценарий выполняется без ошибок
[ ] logcat не содержит необработанных Exception/ANR при базовом сценарии
[ ] Размер APK/AAB в пределах ожидаемого диапазона
[ ] versionCode/versionName корректны и соответствуют релизу
```

Пример команды для установки и мониторинга логов при smoke-тестировании:

```bash
adb install -r app-release.apk
adb logcat *:E | grep com.example.app
```

## На что смотрит интервьюер

Интервьюер оценивает, понимает ли кандидат разницу между полным регрессом и быстрым build verification (смоук-тестом), и знает ли конкретные аспекты, специфичные для Android-сборок: signing, ProGuard/R8, upgrade path, permissions модель. Плюсом будет упоминание практических команд (`adb install`, `logcat`) и метрик (cold start time, размер APK). Красный флаг — если кандидат описывает тестирование билда так же, как полное регрессионное тестирование, не выделяя приоритет быстрой проверки критичных путей.
