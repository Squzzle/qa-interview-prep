# Как работает Android? Какая у него архитектура?

**Коротко:** Android — это Linux-based операционная система со слоистой архитектурой: ядро Linux, аппаратный уровень абстракции (HAL), нативные библиотеки и Android Runtime (ART), Java/Kotlin API Framework и слой приложений, где каждое приложение работает в изолированной песочнице со своим процессом.

## Развёрнутый ответ

Архитектуру Android принято представлять в виде пяти слоёв, от низкого уровня к высокому:

1. **Linux Kernel** — модифицированное ядро Linux, отвечает за низкоуровневое управление: драйверы устройств (камера, дисплей, Bluetooth, Wi-Fi), управление памятью, процессами и потоками, энергопотреблением, безопасностью (SELinux). Также содержит специфичные для Android подсистемы: Binder (IPC-механизм), low memory killer, wakelocks.

2. **Hardware Abstraction Layer (HAL)** — набор стандартизированных интерфейсов, которые позволяют Android Framework обращаться к аппаратным возможностям устройства (камера, сенсоры, аудио) без знания деталей реализации конкретного производителя. Это даёт вендорам гибкость в реализации драйверов, не затрагивая верхние уровни системы.

3. **Native Libraries и Android Runtime (ART)**:
   - Нативные библиотеки на C/C++: libc, WebKit, SQLite, OpenGL ES, Media Framework, SSL и др.
   - **ART (Android Runtime)** — заменил Dalvik начиная с Android 5.0. Выполняет байт-код приложений (формат `.dex`), использует **AOT (Ahead-Of-Time)** компиляцию совместно с JIT (Just-In-Time) для оптимального баланса времени установки/запуска и производительности. Каждое приложение запускается в собственном экземпляре виртуальной машины и в отдельном Linux-процессе с уникальным UID — это база песочницы безопасности Android.

4. **Java API Framework (Application Framework)** — набор Java/Kotlin API и системных сервисов, через которые приложения взаимодействуют с системой:
   - **Activity Manager** — управление жизненным циклом Activity и стеком задач.
   - **Window Manager** — управление окнами и отображением.
   - **Content Providers** — обмен данными между приложениями.
   - **View System** — построение UI (View, ViewGroup).
   - **Package Manager** — установка, обновление, информация о приложениях.
   - **Telephony, Resource, Location, Notification Manager** и другие.

5. **System Applications / Applications Layer** — предустановленные и пользовательские приложения (Контакты, SMS, браузер, сторонние приложения), написанные на Kotlin/Java и использующие Framework API.

Ключевые архитектурные принципы:

- **Изоляция процессов (sandboxing)**: каждое приложение получает уникальный Linux UID, работает в своём процессе и своей ART-инстанции; доступ к чужим данным ограничен правами файловой системы и permissions.
- **Binder IPC** — основной механизм межпроцессного взаимодействия, на котором строятся Intent, ContentProvider, AIDL-сервисы.
- **Компонентная модель приложения**: Activity, Service, BroadcastReceiver, ContentProvider — каждое приложение состоит из этих компонентов, управляемых системой, а не имеет единой точки входа `main()`.
- **Управление жизненным циклом процессов системой**: Android может убить фоновый процесс приложения при нехватке памяти (Low Memory Killer), приоритет процессов зависит от видимости и важности (foreground, visible, service, background, empty).

## Пример / когда применяется

Понимание архитектуры важно при отладке проблем производительности и памяти: например, если приложение часто убивается системой в фоне, тестировщик должен понимать концепцию приоритетов процессов и OOM killer, чтобы правильно классифицировать баг (это ожидаемое поведение ОС или утечка памяти в приложении).

```text
Applications Layer          -> System Apps, User Apps
Java API Framework          -> Activity/Window/Package Manager, Content Providers
Native Libraries + ART      -> libc, SQLite, OpenGL ES, ART (AOT+JIT), Zygote
Hardware Abstraction Layer  -> Camera HAL, Audio HAL, Sensors HAL
Linux Kernel                -> Drivers, Binder, Power Management, SELinux
```

При тестировании производительности можно использовать `adb shell dumpsys meminfo <package>` для анализа потребления памяти конкретным процессом приложения, что напрямую опирается на понимание процессной модели Android.

## На что смотрит интервьюер

- Умение назвать слои архитектуры не просто перечислением, а с пониманием, зачем нужен каждый (например, зачем HAL, а не прямой доступ к драйверам).
- Знание разницы между Dalvik и ART, понимание AOT/JIT компиляции.
- Понимание модели безопасности: sandbox, permissions, UID на приложение — частый вопрос для QA, тестирующих security-аспекты.
- Знание жизненного цикла процесса и приоритетов (почему приложение может быть убито в фоне) — важно для баг-репортов о "пропавших" данных или неожиданных рестартах.
- Красный флаг: путаница между Activity lifecycle (уровень приложения) и process lifecycle (уровень системы) — это разные вещи, и кандидаты часто их смешивают.
- Follow-up вопросы: что такое Zygote и зачем он нужен, как работает Binder, чем ART отличается от JVM.
