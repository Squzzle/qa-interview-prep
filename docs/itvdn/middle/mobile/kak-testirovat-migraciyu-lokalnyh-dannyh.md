# Как тестировать миграцию локальных данных?

**Коротко:** тестирование миграции проверяет, что при обновлении приложения (и, соответственно, схемы локальной БД или формата хранения) существующие данные пользователя корректно преобразуются в новую структуру без потери и повреждения, а также что приложение корректно обрабатывает откат и промежуточные версии.

## Развёрнутый ответ

Миграция локальных данных происходит, когда новая версия приложения меняет структуру хранения данных: добавляет/удаляет колонки в SQLite-таблице, меняет схему Room/Core Data, переименовывает поля, меняет формат сериализации. Если миграция реализована некорректно, пользователь после обновления может потерять данные, получить краш или увидеть повреждённые данные.

Основные направления тестирования:

**1. Проверка миграции «через одну версию» (N → N+1)**
Установить предыдущую версию приложения, создать репрезентативный набор данных (в идеале — покрывающий все edge-case: пустые поля, максимальные значения, спецсимволы, большие объёмы), обновить приложение до новой версии, проверить, что данные сохранились и корректно отображаются.

**2. Проверка миграции «через несколько версий» (N → N+k, skip-версии)**
Пользователи часто не обновляются вовремя и переходят сразу через несколько релизов. Нужно убедиться, что цепочка миграций (миграция v1→v2→v3→v4) отрабатывает последовательно и корректно, а не только прямой путь v1→v4.

**3. Тестирование отсутствия миграции / чистой установки**
Убедиться, что на чистой установке (без старых данных) новая схема создаётся корректно без ошибок, ссылающихся на несуществующие миграции.

**4. Тестирование граничных и проблемных данных**
- Данные с `null`/пустыми значениями в полях, которые в новой схеме стали обязательными (`NOT NULL`) — нужна стратегия дефолтных значений.
- Большие объёмы данных — проверка производительности миграции (не должна блокировать UI надолго, не должна приводить к ANR).
- Повреждённые/некорректные данные из старой версии — приложение не должно падать, а должна быть стратегия graceful degradation.

**5. Проверка отказоустойчивости миграции**
Что произойдёт, если миграция прервётся (приложение убито системой, разрядилась батарея, кончилась память посреди процесса)? Хорошая реализация использует транзакции, чтобы либо миграция полностью применилась, либо полностью откатилась, не оставляя БД в промежуточном состоянии.

**6. Регрессионное тестирование после миграции**
После миграции — полный набор функциональных проверок, что все фичи, зависящие от мигрированных данных, работают как прежде.

**7. Проверка даунгрейда (downgrade)**
Что происходит, если пользователь откатывается на старую версию приложения (актуально для внутреннего/бета-тестирования) — как правило, старая версия не понимает новую схему, и нужно понимать ожидаемое поведение (краш с понятным сообщением, либо блокировка отката).

## Пример / когда применяется

Пример тест-кейса для Room-миграции на Android:

```kotlin
// Пример проверки миграции с помощью MigrationTestHelper (androidx.room:room-testing)
@Test
fun migrate1To2_preservesExistingData() {
    var db = helper.createDatabase(TEST_DB, 1).apply {
        execSQL("INSERT INTO users (id, name) VALUES (1, 'Alice')")
        close()
    }

    db = helper.runMigrationsAndValidate(TEST_DB, 2, true, MIGRATION_1_2)

    val cursor = db.query("SELECT * FROM users WHERE id = 1")
    assertTrue(cursor.moveToFirst())
    assertEquals("Alice", cursor.getString(cursor.getColumnIndex("name")))
}
```

Ручной сценарий: установить версию 1.5 из архива APK, заполнить приложение данными (заметки, избранное, история), обновить до текущей версии через `adb install -r`, проверить, что все данные на месте и приложение не крашится при первом запуске после обновления.

## На что смотрит интервьюер

Интервьюер ждёт от кандидата понимания, что миграция — это не только «данные не потерялись», но целый набор сценариев: skip-миграции, откат, повреждённые данные, отказоустойчивость и производительность на больших объёмах. Плюсом будет упоминание конкретных инструментов (Room `MigrationTestHelper`, тестирование через архивные APK разных версий). Красный флаг — если кандидат ограничивается фразой «просто обновить приложение и посмотреть, что данные есть».
