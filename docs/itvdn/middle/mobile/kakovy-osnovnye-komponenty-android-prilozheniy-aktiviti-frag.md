# Каковы основные компоненты Android-приложений (активити / фрагмент / сервис / интент-фильтр)?

**Коротко:** ключевые компоненты Android-приложения — Activity (экран UI), Fragment (переиспользуемая часть UI внутри активити), Service (фоновая работа без UI), Broadcast Receiver (реакция на системные/приложенческие события), Content Provider (обмен данными между приложениями) и Intent/Intent Filter (механизм взаимодействия между компонентами).

## Развёрнутый ответ

Android-приложение строится из декларативно зарегистрированных в манифесте компонентов, каждый из которых имеет свой жизненный цикл, управляемый системой:

**Activity (активити)** — представляет один экран с пользовательским интерфейсом. У активити есть жизненный цикл (`onCreate → onStart → onResume → onPause → onStop → onDestroy`), который управляется системой в зависимости от видимости и фокуса экрана. Каждый экран приложения обычно соответствует одной активити (или одному «хосту» для набора фрагментов в современном подходе single-activity).

**Fragment (фрагмент)** — модульная, переиспользуемая часть пользовательского интерфейса, которая живёт внутри активити (или другого фрагмента). Имеет собственный жизненный цикл, привязанный к жизненному циклу хостящей активити, но с дополнительными состояниями (`onAttach`, `onCreateView`, `onViewCreated`, `onDestroyView`, `onDetach`). Используется для построения адаптивных UI (например, list+detail на планшетах) и для разделения сложного экрана на переиспользуемые блоки.

**Service (сервис)** — компонент, выполняющий длительные операции в фоне без пользовательского интерфейса, даже если пользователь перешёл в другое приложение. Бывает:
- обычный (started) сервис — запускается и работает независимо от компонента, который его запустил;
- bound-сервис — предоставляет клиент-серверный интерфейс другим компонентам;
- начиная с Android 8+ фоновые сервисы сильно ограничены — предпочтение отдаётся `WorkManager` или foreground-сервисам с уведомлением.

**Broadcast Receiver** — компонент, который подписывается и реагирует на широковещательные события — системные (низкий заряд батареи, изменение сети, завершение загрузки устройства) или кастомные, отправленные другими компонентами/приложениями через `Intent`.

**Content Provider** — управляет доступом к структурированному набору данных приложения и предоставляет стандартизированный интерфейс для чтения/записи этих данных другими приложениями (например, доступ к контактам, медиатеке через `ContentResolver`).

**Intent** — асинхронный «мессендж»-объект, который используется для запроса действия у другого компонента: запуск активити, запуск сервиса, доставка broadcast. Бывает явный (explicit — с указанием конкретного класса-получателя) и неявный (implicit — с указанием действия/данных, без конкретного получателя).

**Intent Filter** — декларация в манифесте (или динамически в коде), которая описывает, какие неявные intent'ы способен обработать компонент (по `action`, `category`, `data`). Именно intent-фильтры позволяют системе находить подходящий компонент (в том числе из другого приложения) для обработки неявного intent'а — например, открытие диалога «Поделиться».

Все эти компоненты декларируются в `AndroidManifest.xml` (кроме динамически регистрируемых broadcast receiver'ов) и управляются системным процессом Zygote/ActivityManager, а не самим приложением напрямую.

## Пример / когда применяется

```xml
<activity android:name=".ShareActivity">
    <intent-filter>
        <action android:name="android.intent.action.SEND" />
        <category android:name="android.intent.category.DEFAULT" />
        <data android:mimeType="text/plain" />
    </intent-filter>
</activity>
```

Такой intent-filter означает: «эта активити умеет принимать текст, отправленный из другого приложения через системное меню Share». Тестировщику важно проверять подобные сценарии: расшаривание из соцсетей в приложение, обработку deep link'ов (`<data android:scheme="myapp" android:host="product" />`), и поведение при получении broadcast (например, реакцию на смену сети или получение push-уведомления).

## На что смотрит интервьюер

Интервьюер оценивает, различает ли кандидат назначение каждого компонента и понимает ли их взаимодействие через `Intent`/`Intent Filter`, а не просто перечисляет названия. Полезно упомянуть современные ограничения (background execution limits, foreground service с Android 8+, необходимость `WorkManager`). Частый follow-up: «В чём разница между явным и неявным Intent?» или «Как работает deep linking через intent-filter?»
