# Опишите жизненный цикл активити.

**Коротко:** активити проходит через набор коллбэков — `onCreate`, `onStart`, `onResume`, `onPause`, `onStop`, `onDestroy` (и опционально `onRestart`), которые вызываются системой в зависимости от того, создаётся, становится видимой, получает фокус, теряет фокус, скрывается или уничтожается активити.

## Развёрнутый ответ

Жизненный цикл активити управляется системой Android, а не самим приложением — система вызывает соответствующие методы в ответ на действия пользователя, системные события (звонок, поворот экрана) или нехватку ресурсов.

Основные состояния и коллбэки:

- **`onCreate()`** — вызывается один раз при создании активити. Здесь инициализируется UI (`setContentView`), восстанавливается сохранённое состояние (`savedInstanceState`), настраиваются вью-модели, адаптеры и т.д.
- **`onStart()`** — активити становится видимой пользователю, но ещё не обязательно в фокусе (может быть частично перекрыта).
- **`onResume()`** — активити находится «на переднем плане» и в фокусе, пользователь может с ней взаимодействовать. Именно здесь принято запускать анимации, камеру, сенсоры.
- **`onPause()`** — активити частично теряет фокус (например, появился диалог поверх неё, или начался переход на другую активити). Это последняя гарантированно вызываемая точка перед возможным уничтожением процесса — здесь сохраняют лёгкие, некритичные для UX данные и останавливают ресурсоёмкие операции (камера, анимации).
- **`onStop()`** — активити полностью не видна пользователю (свернули приложение, открыли другую активити поверх). Здесь освобождают более тяжёлые ресурсы.
- **`onRestart()`** — вызывается перед повторным `onStart()`, если активити возвращается из состояния `Stopped` (пользователь вернулся назад к этой активити).
- **`onDestroy()`** — финальный коллбэк перед уничтожением активити — либо потому что она сама себя завершает (`finish()`), либо система уничтожает её (поворот экрана, нехватка памяти, завершение работы системой).

Дополнительные важные методы:
- **`onSaveInstanceState(Bundle)`** — вызывается перед возможным уничтожением активити системой (не при явном `finish()`), чтобы сохранить UI-состояние (например, введённый текст, позицию скролла) для последующего восстановления.
- **`onRestoreInstanceState(Bundle)`** — восстанавливает сохранённое состояние после пересоздания активити.

Важный нюанс жизненного цикла: при повороте экрана (или другом изменении конфигурации) по умолчанию активити полностью уничтожается и создаётся заново (`onDestroy` → `onCreate`), если явно не указано иное (`android:configChanges`) — поэтому корректная работа с `onSaveInstanceState`/ViewModel критична для отсутствия потери данных пользователя.

Современный подход (Jetpack) рекомендует хранить UI-состояние в `ViewModel`, которая переживает пересоздание активити при повороте экрана (но не при уничтожении процесса системой), а долгоживущие данные — в `SavedStateHandle` или персистентном хранилище.

## Пример / когда применяется

Последовательность вызовов при типичных сценариях:

| Сценарий | Последовательность |
|---|---|
| Первый запуск активити | `onCreate → onStart → onResume` |
| Нажатие Home (сворачивание) | `onPause → onStop` |
| Возврат в приложение | `onRestart → onStart → onResume` |
| Открытие диалога поверх активити | `onPause` (только) |
| Поворот экрана | `onPause → onStop → onDestroy → onCreate → onStart → onResume` |
| Нажатие «Назад» (завершение) | `onPause → onStop → onDestroy` |

Тестировщик должен уметь воспроизводить каждый из этих сценариев (в том числе с включённой опцией «Не сохранять активности», см. соответствующий вопрос) и проверять, что состояние UI и данные корректно сохраняются/восстанавливаются.

## На что смотрит интервьюер

Ожидается точное знание порядка методов и разницы между `onPause`/`onStop`, а также понимание, зачем нужен `onSaveInstanceState` и как поворот экрана влияет на жизненный цикл. Плюсом будет упоминание `ViewModel` как современного решения проблемы сохранения состояния. Красный флаг — путаница в порядке коллбэков или незнание, что при повороте экрана активити по умолчанию пересоздаётся полностью. Частый follow-up: «Что будет с данными в обычной переменной класса активити при повороте экрана без `onSaveInstanceState`?»
