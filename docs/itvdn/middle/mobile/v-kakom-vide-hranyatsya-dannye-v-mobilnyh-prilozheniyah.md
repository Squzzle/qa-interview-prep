# В каком виде хранятся данные в мобильных приложениях локально?

**Коротко:** локально данные хранятся в виде пар «ключ-значение» (`SharedPreferences`/`UserDefaults`), реляционных или объектных баз данных (`SQLite`, `Room`, `Core Data`, `Realm`), файлов во внутреннем/внешнем хранилище и в защищённых хранилищах для чувствительных данных (`Keystore`/`Keychain`).

## Развёрнутый ответ

Мобильные ОС предоставляют несколько уровней и механизмов локального хранения, каждый со своим назначением и уровнем защиты:

**1. Key-Value хранилища:**
- Android: `SharedPreferences` — XML-файлы с примитивными парами ключ-значение (строки, числа, булевы значения); хранятся в `/data/data/<package>/shared_prefs/`.
- iOS: `UserDefaults` (`.plist` файлы) — аналогичное назначение.
- Используются для настроек приложения, флагов первого запуска, простых пользовательских предпочтений.

**2. Базы данных:**
- Android: `SQLite` напрямую или через ORM `Room`; также `Realm` (кроссплатформенная NoSQL БД).
- iOS: `Core Data` (обёртка над SQLite/иными сторами), `Realm`.
- Используются для структурированных данных: кэш списков, офлайн-режим, история, локальный кэш API-ответов.

**3. Файловая система:**
- Внутреннее хранилище приложения (`/data/data/<package>/files/` на Android, `Documents`/`Library` на iOS) — доступно только самому приложению.
- Внешнее хранилище (SD-карта, публичная память на Android) — может быть доступно другим приложениям и пользователю через файловый менеджер; требует разрешений.
- Кэш-директория (`cache/`) — временные файлы, которые система может очищать при нехватке места.
- Используется для медиафайлов, скачанных документов, временных файлов.

**4. Защищённое хранилище для чувствительных данных:**
- Android: `Keystore` (аппаратно поддерживаемое хранилище криптографических ключей) в сочетании с `EncryptedSharedPreferences`/`EncryptedFile` из Jetpack Security.
- iOS: `Keychain` — зашифрованное системное хранилище для паролей, токенов, сертификатов.
- Данные здесь хранятся в зашифрованном виде и защищены от чтения другими приложениями и (частично) от root/jailbreak-доступа.

**5. Куки и веб-хранилища** — если приложение использует `WebView`, данные могут храниться также в `WebView storage` (cookies, `localStorage`), что важно учитывать при тестировании гибридных приложений.

Важный аспект для QA: не все данные должны храниться одинаково — пароли, токены авторизации, платёжные данные **не должны** попадать в обычные `SharedPreferences`/`UserDefaults` в открытом виде — это уязвимость. Такие данные должны идти через `Keystore`/`Keychain` или как минимум через шифрование.

## Пример / когда применяется

Что стоит проверить на rooted-устройстве/эмуляторе при security-тестировании:

```bash
# Проверка содержимого SharedPreferences приложения (rooted device / emulator)
adb shell run-as com.example.app cat shared_prefs/app_prefs.xml

# Просмотр локальной SQLite-базы
adb shell run-as com.example.app cat databases/app.db
```

Если в выводе видно `password=12345` или `auth_token=eyJhbGci...` в открытом виде — это баг безопасности, который должен быть заведён с высоким приоритетом.

## На что смотрит интервьюер

Интервьюер проверяет, знает ли кандидат конкретные механизмы (`SharedPreferences`, `SQLite`/`Room`, `Keystore`/`Keychain`), а не отвечает общими словами «в базе данных». Ценится понимание, что для чувствительных данных нужны отдельные защищённые механизмы, и что тестировщик обязан проверять, не «утекают» ли пароли/токены в открытые хранилища. Частый follow-up: «Как бы вы проверили, что приложение не хранит пароль в открытом виде?»
