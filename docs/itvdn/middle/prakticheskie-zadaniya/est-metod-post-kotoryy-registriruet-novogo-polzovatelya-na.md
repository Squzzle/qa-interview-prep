# Есть метод POST, который регистрирует нового пользователя на сайте, есть тело запроса, содержащее данные о почте, телефоне, имени пользователя и адресе проживания. Какие кейсы для проверки можете привести?

**Коротко:** нужно проверить позитивные сценарии (валидная регистрация, корректный код ответа и данные в БД), негативные сценарии по каждому полю (граничные значения, невалидный формат, пустые/null значения), а также кросс-полевые, security и нефункциональные проверки (дубликаты, идемпотентность, производительность).

## Развёрнутый ответ

Тестирование POST-метода регистрации разбивается на несколько уровней:

**1. Позитивные кейсы (Happy path):**
- Регистрация со всеми валидными полями — ожидаем `200/201 Created`, тело ответа содержит `id` пользователя, данные в БД совпадают с отправленными.
- Регистрация с минимально обязательными полями (если часть полей опциональна).
- Проверка, что пароль (если есть) хранится в захэшированном виде, а не в открытом тексте.
- Проверка отправки письма/SMS подтверждения (если предусмотрено бизнес-логикой).

**2. Валидация полей (негативные кейсы):**
- Email: без `@`, без домена, двойной `@@`, кириллица в адресе, пустая строка, `null`, превышение длины (RFC — 254 символа), email с уже существующим в системе адресом (дубликат), email с пробелами по краям (trim behavior), SQL/HTML-инъекции в поле.
- Телефон: неверный формат (буквы вместо цифр), отсутствие кода страны, слишком короткий/длинный номер, спецсимволы, локальный/международный формат (+380... vs 0...).
- Имя пользователя: пустое значение, спецсимволы, эмодзи, превышение допустимой длины, уже занятое имя, только пробелы, XSS-payload (`<script>`), Unicode-нормализация.
- Адрес: отсутствие обязательных подполей (город, индекс), некорректный формат индекса, слишком длинная строка, отсутствие адреса вовсе (если поле опционально — проверить, что регистрация проходит).

**3. Граничные значения (Boundary Value Analysis):** для каждого поля — минимальная/максимальная допустимая длина минус 1, ровно на границе, плюс 1 символ.

**4. Кросс-полевые проверки:** одновременная невалидность нескольких полей — приложение должно вернуть агрегированный список ошибок, а не только первую найденную.

**5. Технические/HTTP-аспекты:**
- Отсутствие обязательного поля в JSON вообще (не пустая строка, а отсутствие ключа).
- Неверный `Content-Type` (не `application/json`).
- Некорректный JSON (синтаксическая ошибка) — ожидаем `400 Bad Request`.
- Повторный (дублирующий) запрос с тем же телом — идемпотентность, отсутствие дублей в БД.
- Проверка кодов ответа: `201` при успехе, `400` при валидационных ошибках, `409 Conflict` при дубликате email, `500` не должен возникать на пользовательских ошибках.

**6. Security:**
- SQL-инъекции, XSS в текстовых полях.
- Rate limiting / защита от брутфорса при массовой регистрации.
- Проверка, что в ответе API не утекают чувствительные данные (пароль, внутренние ID).

**7. Нефункциональные:**
- Нагрузочное тестирование одновременной регистрации нескольких пользователей с одинаковым email (race condition).
- Логирование и мониторинг ошибок регистрации.

## Пример / когда применяется

Пример структурированного набора тест-кейсов в табличном виде:

| № | Поле | Входные данные | Ожидаемый результат |
|---|------|-----------------|----------------------|
| 1 | email | `user@test.com` (валидный) | `201 Created` |
| 2 | email | `user@@test.com` | `400`, сообщение об ошибке формата |
| 3 | email | уже существует в системе | `409 Conflict` |
| 4 | phone | `+380501234567` | `201 Created` |
| 5 | phone | `abc12345` | `400`, ошибка валидации |
| 6 | username | пустая строка | `400` |
| 7 | address | отсутствует (опционально) | `201 Created`, поле `null` в БД |
| 8 | body | некорректный JSON | `400 Bad Request` |

Такой подход применяется при автоматизации API-тестов (например, в Postman/RestAssured), где каждая строка таблицы превращается в отдельный параметризованный тест.

## На что смотрит интервьюер

Интервьюер ожидает, что кандидат не ограничится 2-3 очевидными кейсами, а системно пройдётся по каждому полю с применением техник тест-дизайна (эквивалентные классы, граничные значения). Красный флаг — если кандидат забывает про security-аспекты (инъекции, XSS) и про проверку кодов ответа HTTP. Частый follow-up: "А как бы вы это автоматизировали?", "Что делать с дублирующимися email при параллельных запросах?", "Как проверить, что пароль не хранится в открытом виде?".
