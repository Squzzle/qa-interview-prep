# Есть веб-приложение интернет-магазина (регистрация, логин, поиск товаров, корзина и покупки). Программу поддерживают следующие браузеры: Chrome, Safari, Edge. У нас есть ограниченное время на тестирование. Расскажите, как вы будете проверять приложение?

**Коротко:** нужно применить риск-ориентированный подход — приоритизировать критичные бизнес-flows (регистрация → логин → поиск → корзина → покупка) как end-to-end сценарии на основном браузере (Chrome), затем провести кросс-браузерную проверку ключевых сценариев на Safari и Edge, оставив второстепенную функциональность и edge-кейсы на потом при наличии времени.

## Развёрнутый ответ

При ограниченном времени тестирование строится по принципу "от общего к частному" с фокусом на бизнес-критичность:

**1. Анализ рисков и приоритизация.** Определяем core user journey: регистрация → логин → поиск товара → добавление в корзину → оформление покупки (оплата). Это денежный путь пользователя (revenue path), баги в нём наиболее критичны для бизнеса.

**2. Smoke-тестирование на основном браузере (Chrome).** Сначала быстро проверяем, что вся цепочка работает end-to-end хотя бы на одном браузере — это даёт уверенность, что билд в принципе рабочий, прежде чем тратить время на кросс-браузерность.

**3. Детальное тестирование core-функциональности на Chrome:**
- Регистрация: валидация полей, успешная регистрация, дублирующийся email.
- Логин: успешный вход, неверный пароль, блокировка после N попыток.
- Поиск: релевантность результатов, фильтры, сортировка, пустой результат поиска.
- Корзина: добавление/удаление товара, изменение количества, пересчёт суммы, применение промокода.
- Покупка: оформление заказа, выбор способа оплаты/доставки, подтверждение заказа.

**4. Кросс-браузерная проверка (Safari, Edge) — только критичных сценариев.** Не дублируем весь объём тестов на каждом браузере — фокусируемся на:
- Визуальном рендеринге (вёрстка, шрифты, отступы — известные проблемные места для Safari, например flexbox/grid-баги).
- Функциональности критичного пути (логин, корзина, оплата) — именно там баги наиболее дорогие.
- Специфичных для браузера особенностях (Safari — cookies/localStorage политика, autofill форм; Edge — устаревшие рендер-движки в старых версиях).

**5. Приоритизация по времени (если времени совсем мало):**
- День/час 1: Smoke test на Chrome (core flow работает в принципе).
- Далее: Детальное тестирование core flow на Chrome.
- Далее: Core flow на Safari и Edge (только критичные шаги, не весь регресс).
- Если время останется: второстепенная функциональность (профиль пользователя, история заказов, edge-кейсы валидации) и полный регресс на всех браузерах.

**6. Использование инструментов для ускорения.** Если есть автотесты (Selenium/Playwright с кросс-браузерным запуском) — прогоняем их параллельно на всех трёх браузерах, пока ведём ручное exploratory-тестирование.

**7. Коммуникация рисков.** Явно фиксируем и сообщаем команде, что именно было протестировано полностью, что частично, а что не проверялось вовсе, чтобы решение о релизе принималось осознанно.

## Пример / когда применяется

Пример тайм-плана при доступных 4 часах на тестирование:

| Время | Активность | Браузер |
|---|---|---|
| 0:00–0:20 | Smoke test core flow | Chrome |
| 0:20–1:30 | Детальное тестирование регистрации, логина, корзины, покупки | Chrome |
| 1:30–2:15 | Core flow (регистрация → покупка) | Safari |
| 2:15–3:00 | Core flow (регистрация → покупка) | Edge |
| 3:00–3:30 | Визуальная проверка вёрстки на всех трёх браузерах | Chrome/Safari/Edge |
| 3:30–4:00 | Exploratory testing второстепенных функций, если время позволяет | Chrome |

Такой подход применяется в реальных условиях перед срочным релизом, хотфиксом или ограниченным окном для регресса перед крупной распродажей (Black Friday и т.п.), где core purchase flow важнее второстепенных функций.

## На что смотрит интервьюер

Интервьюер оценивает умение приоритизировать в условиях дефицита ресурсов и понимание бизнес-ценности разных частей приложения (покупка важнее, чем, например, редактирование профиля). Важно показать, что кандидат не будет пытаться равномерно "размазать" тестирование по всем браузерам и функциям, а сфокусируется на критичном пути. Красный флаг — попытка тестировать всё подряд без приоритизации или игнорирование кросс-браузерности вовсе. Частые follow-up: "А если баг найден только в Safari и времени на фикс нет — что делаете?", "Как бы вы использовали автоматизацию, если бы она была?", "Как расставить приоритеты, если у вас 3 разных стейкхолдера с разными ожиданиями?".
