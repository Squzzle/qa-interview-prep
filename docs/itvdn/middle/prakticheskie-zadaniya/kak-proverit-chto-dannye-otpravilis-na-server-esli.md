# Как проверить, что данные отправились на сервер, если у нас нет доступа к бэкенду?

**Коротко:** используем инструменты перехвата и анализа сетевого трафика на стороне клиента (DevTools Network, proxy-инструменты типа Charles/Fiddler/Proxyman), а также наблюдаем за косвенными признаками успешной отправки — статус-коды ответа, изменение UI, side-эффекты (email, уведомления), доступный публичный API.

## Развёрнутый ответ

Отсутствие доступа к бэкенду (серверным логам, БД, коду) не означает отсутствие возможности верифицировать факт отправки данных — вся необходимая информация проходит через клиент, и её можно перехватить и проанализировать на границе "клиент — сеть".

Основные инструменты и подходы:

1. **Браузерные DevTools (вкладка Network).** Открыть Network до совершения действия, отправить форму, найти соответствующий запрос (по методу POST/PUT и по URL). Проверить: что именно ушло в теле запроса (Payload/Request body), какие заголовки (Content-Type, Authorization), какой пришёл статус-код (200/201 — успех, 4xx — ошибка клиента, 5xx — ошибка сервера) и что содержится в теле ответа (Response). Это самый доступный и часто используемый способ.

2. **Proxy-инструменты (Charles Proxy, Fiddler, Proxyman, mitmproxy).** Позволяют перехватывать трафик не только браузера, но и мобильных приложений, десктопных клиентов. Полезны, когда нужно перехватить HTTPS-трафик (с установкой корневого сертификата), посмотреть весь обмен целиком, включая заголовки, куки, тайминги, а также модифицировать запросы/ответы для тестирования (man-in-the-middle тестирование).

3. **Постман/Insomnia для повторной отправки и верификации.** Скопировать запрос как cURL из DevTools и выполнить его вручную в Postman, изучая ответ подробно, либо составить последующий GET-запрос для проверки, что данные сохранились (если есть публичный read-эндпоинт).

4. **Наблюдение за косвенными признаками (side-effects) в самом приложении:**
   - Появление созданной записи в другом разделе интерфейса (список, история заказов, профиль).
   - Изменение счётчиков/статусов на странице (например, "Заказ №123 создан").
   - Получение email/SMS/push-уведомления, которое должно генерироваться сервером в ответ на сохранение данных.
   - Изменение состояния кнопки/формы (disabled после успешной отправки, редирект на страницу успеха).

5. **HAR-файлы (HTTP Archive).** Экспорт полного лога сетевых запросов из DevTools в HAR-файл — удобно для приложения к баг-репорту и для передачи разработчикам как доказательство.

6. **Логи на стороне мобильного приложения / логи трекеров (Charles/Proxyman для мобильных, adb logcat для Android)** — если тестируется мобильное приложение, а не веб.

7. **Публичные/партнёрские API того же продукта**, если они существуют (например, мобильное приложение может иметь свой API для чтения тех же данных, что были отправлены с веба).

Важно подчеркнуть ограничение этого подхода: без доступа к бэкенду мы можем убедиться, что "клиент отправил запрос и получил ответ, похожий на успех", но не можем на 100% гарантировать, что данные корректно и окончательно сохранились в БД (могла произойти асинхронная ошибка после ответа, откат транзакции и т.п.) — для полной уверенности нужно либо получить доступ хотя бы для чтения (read-only доступ к БД или staging-окружению), либо полагаться на согласованные с бэкенд-командой контракты (API-документация, договорённость о кодах ответов).

## Пример / когда применяется

Пример: тестируется форма подписки на рассылку на публичном сайте, доступа к серверу и БД нет.
1. Открываем DevTools → Network, фильтруем по XHR/Fetch.
2. Заполняем форму, нажимаем Submit.
3. Находим запрос `POST /api/subscribe`, смотрим Payload: `{ "email": "test@example.com" }`.
4. Смотрим ответ: статус `200 OK`, тело `{ "success": true, "subscriberId": "abc123" }`.
5. Убеждаемся, что UI показал сообщение "Вы успешно подписаны" (side-effect).
6. Дополнительно, если есть публичный эндпоинт проверки статуса подписки, выполняем `GET /api/subscribe/status?email=test@example.com` через Postman и убеждаемся в статусе "subscribed".

```bash
curl -X GET "https://example.com/api/subscribe/status?email=test@example.com"
```

## На что смотрит интервьюер

Интервьюер оценивает практическое владение инструментами тестирования "чёрного ящика" (black-box testing) на сетевом уровне — знание DevTools Network, понимание HTTP-статусов, умение работать с Postman и proxy-инструментами. Хороший ответ обязательно упомянет анализ сетевого трафика как основной инструмент и признает ограничение метода (нельзя гарантировать 100% сохранение без доступа хотя бы для чтения). Красный флаг — ответ "никак не проверить" или полагание исключительно на визуальные подсказки UI без анализа реального запроса/ответа. Частый follow-up: "А если API зашифрован/обфусцирован?" (ответ: MITM-proxy с сертификатом, анализ по факту side-эффектов) и "Чем HAR-файл полезен разработчику?" (полное воспроизводимое доказательство запроса для дебага).
