# На что следует акцентировать внимание при автоматизации методов API? Что следует проверять?

**Коротко:** при автоматизации API нужно проверять код ответа, схему и содержимое тела ответа, заголовки, время отклика, поведение при некорректных данных, а также независимость и воспроизводимость тестов.

## Развёрнутый ответ

Автоматизация API-тестов должна покрывать несколько уровней проверки:

**1. HTTP-статус код.** Каждый тест обязан явно проверять код ответа (`200`, `201`, `400`, `401`, `403`, `404`, `409`, `500`) — соответствие ожидаемому поведению для конкретного сценария.

**2. Тело ответа (Response Body):**
- Соответствие JSON-схеме (schema validation) — типы полей, обязательность, отсутствие лишних/недостающих полей.
- Корректность значений — сравнение с ожидаемыми данными, а не только "что-то вернулось".
- Проверка бизнес-логики: например, что рассчитанная цена, статус заказа, количество элементов в коллекции соответствуют ожиданиям.

**3. Заголовки (Headers):** `Content-Type`, `Cache-Control`, кастомные заголовки авторизации/трассировки (`X-Request-Id`), CORS-заголовки при необходимости.

**4. Время ответа (Response Time):** базовые SLA-проверки — метод не должен отвечать дольше N миллисекунд, особенно критично для highload API.

**5. Негативные сценарии:**
- Невалидные/отсутствующие обязательные параметры.
- Некорректные типы данных (строка вместо числа).
- Проверка авторизации/аутентификации: запрос без токена, с истёкшим токеном, с токеном другого пользователя (проверка доступа — IDOR).
- Проверка лимитов (rate limiting, пагинация, максимальный размер payload).

**6. Идемпотентность и side-эффекты:** для `POST`/`PUT`/`DELETE` — что происходит при повторном вызове; для `GET` — что метод не изменяет состояние сервера.

**7. Независимость тестов:** каждый тест должен создавать свои тестовые данные (или использовать моки/тестовую БД) и не зависеть от порядка выполнения или результатов других тестов — иначе тесты становятся хрупкими (flaky).

**8. Интеграция в CI/CD:** тесты должны быть быстрыми, детерминированными, с чёткими логами при падении, запускаться на каждый пуш/PR.

**9. Тестовые данные и окружение:** использование тестового окружения/staging, очистка данных после тестов (teardown), генерация уникальных данных (email с timestamp) во избежание конфликтов.

**10. Контрактное тестирование (Contract Testing):** при микросервисной архитектуре — проверка, что API не нарушает контракт с потребителями (например, с помощью Pact).

## Пример / когда применяется

Пример на RestAssured (Java) для проверки эндпоинта получения пользователя:

```java
@Test
void shouldReturnUserById() {
    given()
        .header("Authorization", "Bearer " + token)
    .when()
        .get("/api/users/{id}", 123)
    .then()
        .statusCode(200)
        .contentType(ContentType.JSON)
        .body("id", equalTo(123))
        .body("email", matchesPattern(".+@.+\\..+"))
        .time(lessThan(500L));
}

@Test
void shouldReturn401WithoutToken() {
    given()
    .when()
        .get("/api/users/{id}", 123)
    .then()
        .statusCode(401);
}
```

Такой подход применяется в любом API-фреймворке (Postman/Newman, RestAssured, Karate, supertest для Node.js) и является базой для регрессионного набора тестов API в CI.

## На что смотрит интервьюер

Интервьюер оценивает, понимает ли кандидат, что автоматизация API — это не только проверка кода `200`. Важно упомянуть schema validation, security-аспекты (авторизация, IDOR), независимость тестов и интеграцию в CI. Красный флаг — если кандидат говорит только "проверяю, что ответ пришёл" без деталей. Частые follow-up вопросы: "Как вы проверяете JSON-схему?", "Как избежать flaky-тестов при работе с реальной БД?", "Как тестировать асинхронные API (например, с вебхуками)?".
