# Написать предельные значения для ввода в форму оплаты товара на сайте.

**Коротко:** для формы оплаты граничные значения нужно определить для каждого поля отдельно (сумма, номер карты, срок действия, CVV, имя держателя) — минимум, минимум−1, максимум, максимум+1, а также специфические для домена платежей значения (0, отрицательные числа, дробные копейки, истёкший срок действия карты).

## Развёрнутый ответ

Boundary Value Analysis (анализ граничных значений) — техника тест-дизайна, основанная на наблюдении, что ошибки чаще всего возникают именно на границах допустимых диапазонов, а не в их середине. Для формы оплаты товара нужно применить эту технику к каждому полю ввода отдельно, а также учесть предметную специфику платёжных систем.

**Поле "Сумма к оплате" (если сумма редактируется, например, донат или пополнение баланса):**
- Минимально допустимая сумма (например, если минимум 1 грн/руб): значения `0.99`, `1.00`, `1.01`.
- Максимально допустимая сумма (например, лимит платежа 150 000): значения `149999.99`, `150000.00`, `150000.01`.
- `0` — должно быть отклонено.
- Отрицательное число (`-100`) — должно быть отклонено.
- Дробные значения с избыточной точностью (`10.999` при допустимых двух знаках после запятой).
- Очень большое число (переполнение типа, например `9999999999999`).
- Нечисловой ввод в числовое поле (если поле текстовое, а не строго numeric input).

**Поле "Номер карты" (обычно 16 цифр для большинства платёжных систем, 15 для Amex):**
- 15 цифр (для карт Amex — валидное граничное значение).
- 16 цифр (стандарт Visa/Mastercard) — валидное значение.
- 14 цифр — должно быть отклонено (меньше минимума).
- 17 цифр — должно быть отклонено (больше максимума).
- Номер карты, не проходящий проверку по алгоритму Луна (Luhn algorithm) — должен быть отклонён как невалидный, даже если длина корректна.
- Номер карты с буквами/спецсимволами вместо цифр.

**Поле "Срок действия карты" (MM/YY):**
- Месяц `01` и `12` — валидные граничные значения.
- Месяц `00` и `13` — невалидные значения (граница снизу и сверху).
- Текущий месяц/год (карта истекает "прямо сейчас") — граничный случай, зависит от бизнес-правила (обычно карта с текущим месяцем ещё действительна до конца месяца).
- Месяц/год в прошлом (`01/20` при текущей дате 2026) — должно быть отклонено как просроченная карта.
- Месяц/год слишком далеко в будущем (например, `01/99` — обычно карты не выпускаются на 70+ лет вперёд) — граничная проверка на разумный диапазон.

**Поле "CVV/CVC" (обычно 3 цифры, 4 для Amex):**
- 3 цифры — валидное значение для большинства карт.
- 4 цифры — валидное значение для Amex.
- 2 цифры — должно быть отклонено (меньше минимума).
- 5 цифр — должно быть отклонено (больше максимума).
- Ведущий ноль (`007`) — должен приниматься как валидное значение (частая ошибка в валидации, которая теряет ведущие нули при приведении к числу).

**Поле "Имя держателя карты":**
- Минимальная длина (1 символ) и пустое поле (0 символов).
- Максимальная длина, разрешённая платёжной системой (обычно 26 символов) — значения на границе (25, 26, 27 символов).
- Только пробелы.
- Цифры и спецсимволы вместо букв (обычно должны быть отклонены).

## Пример / когда применяется

Таблица примера граничных значений для поля "Сумма к оплате" при допустимом диапазоне от 1.00 до 150000.00:

| № | Входное значение | Тип границы | Ожидаемый результат |
|---|---|---|---|
| 1 | 0.99 | min − 0.01 | Отклонено: "Сумма меньше минимальной" |
| 2 | 1.00 | min | Принято |
| 3 | 1.01 | min + 0.01 | Принято |
| 4 | 149999.99 | max − 0.01 | Принято |
| 5 | 150000.00 | max | Принято |
| 6 | 150000.01 | max + 0.01 | Отклонено: "Сумма превышает лимит" |
| 7 | 0 | ниже допустимого | Отклонено |
| 8 | -50 | отрицательное значение | Отклонено |

```ts
function validatePaymentAmount(amount: number): boolean {
  const MIN = 1.00;
  const MAX = 150000.00;
  if (Number.isNaN(amount)) return false;
  return amount >= MIN && amount <= MAX;
}
```

## На что смотрит интервьюер

Интервьюер проверяет владение техникой Boundary Value Analysis на практике: min, min−1, min+1, max, max−1, max+1 — а не просто интуитивный набор "больших и маленьких чисел". Хороший ответ учитывает специфику домена (алгоритм Луна для номера карты, разная длина CVV для разных платёжных систем, бизнес-правило про "текущий месяц срока действия ещё валиден"). Красный флаг — ограничиться только суммой платежа и не вспомнить про номер карты/CVV/срок действия, либо перепутать граничный анализ с эквивалентным разбиением классов. Частый follow-up: "В чём разница между Boundary Value Analysis и Equivalence Partitioning?" (граничный анализ фокусируется на краях диапазона, разбиение на классы эквивалентности — на представителях каждого класса значений, включая граничные как отдельный случай).
