# Предположим, что после нажатия кнопки submit страница перезагружается и ранее введенные данные исчезают. Как проверить, что информация отправлена ​​в базу данных?

**Коротко:** нужно проверять не UI, а фактическое состояние системы — через прямой доступ к БД (SQL-запрос), через API/бэкенд-логи, через сетевой трафик (DevTools/прокси) и через косвенные признаки в другом UI, где эти данные должны отобразиться.

## Развёрнутый ответ

Раз UI не показывает результат после перезагрузки (данные формы очищаются), нужно верифицировать результат на уровне, отличном от исходной формы. Возможные подходы, от наиболее точного к наименее:

1. **Прямой доступ к базе данных.** Если есть доступ (dev/staging окружение, права QA), выполнить SQL-запрос напрямую к таблице, куда должны были попасть данные (например, `SELECT * FROM users WHERE email = '...'`). Это самый надёжный способ — проверяется именно то, что записано в БД: значения полей, кодировка, отсутствие дублей, корректность связанных записей (внешние ключи, timestamps).

2. **Проверка через сетевой трафик (Network tab в DevTools, Charles Proxy, Fiddler, Postman/Insomnia для повторной отправки).** Даже без доступа к БД можно посмотреть, какой запрос ушёл на сервер (метод, endpoint, тело запроса) и какой пришёл ответ (код статуса, тело ответа). Код 200/201 с телом, содержащим сохранённые данные (например, `id` новой записи), — косвенное подтверждение, что сервер принял и обработал данные.

3. **Проверка через другой UI/интерфейс той же системы.** Если данные из формы должны где-то отображаться (профиль пользователя, список заказов, админ-панель), нужно зайти туда и убедиться, что новая запись появилась.

4. **Проверка через API напрямую (GET-запрос к соответствующему endpoint).** Например, если форма отправляла `POST /api/users`, можно вызвать `GET /api/users/{id}` или `GET /api/users?email=...`, используя id/email из ответа на POST-запрос, чтобы убедиться, что данные действительно сохранились, а не просто был возвращён "успешный" ответ без реального сохранения.

5. **Проверка логов бэкенда** (application logs, access logs), если есть доступ — там может быть подтверждение факта записи в БД или ошибка, которая не была показана в UI.

6. **Проверка через side-эффекты**: если после сохранения должно прийти письмо (email-подтверждение), push-уведомление или сработать интеграция с другой системой — их появление тоже косвенно подтверждает успешную обработку данных.

Важный нюанс: код ответа 200 от сервера не гарантирует, что данные реально попали в БД (может быть ошибка на уровне записи, транзакция могла откатиться, могла сохраниться "не та" версия данных). Поэтому наиболее надёжная проверка — либо прямой доступ к БД, либо GET-запрос сразу после POST, подтверждающий персистентность данных.

## Пример / когда применяется

Пример: форма "Добавить отзыв" отправляет `POST /api/reviews`, страница перезагружается, поле "текст отзыва" пустое. Проверка:
1. Открыть DevTools → Network до отправки, зафиксировать запрос `POST /api/reviews` с телом `{ "productId": 42, "text": "Отличный товар" }`, посмотреть ответ — например, `{ "id": 501, "status": "created" }`.
2. Выполнить `GET /api/reviews/501` и убедиться, что вернулся тот же текст.
3. Если есть доступ к БД: `SELECT * FROM reviews WHERE id = 501;` — проверить, что `text = 'Отличный товар'`, `product_id = 42`, `created_at` заполнен.
4. Открыть страницу товара в UI и убедиться, что отзыв отображается в списке отзывов.

```sql
SELECT id, product_id, text, created_at
FROM reviews
WHERE id = 501;
```

## На что смотрит интервьюер

Интервьюер проверяет, понимает ли кандидат разницу между "UI ничего не показал" и "данные не сохранились" — и что тестировать нужно систему целиком, а не полагаться только на визуальную обратную связь. Хороший ответ обязательно упомянет либо прямой доступ к БД, либо проверку сетевого трафика/API — это база для QA даже без специализации в автоматизации. Красный флаг — ответ "не могу проверить, если нет доступа к бэкенду" без попытки предложить альтернативы (сетевой трафик, повторный GET, другой UI). Частый follow-up: именно этот вопрос — "А если у нас нет доступа к бэкенду?" (см. отдельный вопрос про это).
