# Протестовать функционал банкомата с помощью техники State Transition Diagram.

**Коротко:** банкомат моделируется как конечный автомат с состояниями (ожидание карты, ввод PIN, главное меню, выбор операции, выдача денег, ошибка, изъятие карты) и переходами между ними по событиям (вставка карты, ввод PIN, выбор суммы, таймаут, ошибка); тест-кейсы строятся как на валидные переходы, так и на недопустимые (0-switch и N-switch coverage).

## Развёрнутый ответ

State Transition Testing (тестирование переходов состояний) — техника тест-дизайна из "чёрного ящика", применяемая для систем, поведение которых зависит от текущего состояния и истории предыдущих действий, а не только от текущего ввода. Банкомат — классический пример такой системы.

**Шаг 1: определить состояния (states) банкомата:**
1. `Idle` — ожидание вставки карты.
2. `CardInserted` — карта вставлена, ожидание ввода PIN.
3. `PinValidation` — идёт проверка PIN.
4. `MainMenu` — PIN верный, показано главное меню (баланс, снятие, перевод и т.д.).
5. `AmountSelection` — выбор суммы для снятия.
6. `Processing` — обработка операции сервером/банком.
7. `CashDispensing` — выдача наличных.
8. `PrintingReceipt` — печать чека (опционально).
9. `CardEjection` — возврат карты.
10. `ErrorState` — ошибка (неверный PIN, недостаточно средств, нет связи с банком и т.п.).
11. `CardRetained` — карта изъята (после N неверных попыток PIN).

**Шаг 2: определить события/действия (triggers), вызывающие переходы:**
- Вставка карты.
- Ввод PIN (корректный / некорректный).
- Выбор операции в меню.
- Ввод суммы (валидной / превышающей баланс / превышающей лимит выдачи / некратной номиналу купюр).
- Подтверждение операции.
- Отмена операции (нажатие Cancel).
- Таймаут бездействия.
- Извлечение карты пользователем.
- Сбой связи с банковским сервером.

**Шаг 3: построить таблицу переходов состояний (State Transition Table)** — для каждой пары "текущее состояние + событие" указать целевое состояние. Это и есть основа диаграммы.

**Шаг 4: спроектировать тест-кейсы на основе диаграммы:**
- **Валидные переходы (0-switch coverage)** — проверка каждого отдельного перехода, указанного в диаграмме (например, `Idle → CardInserted` при вставке карты).
- **Последовательности переходов (1-switch, N-switch coverage)** — проверка цепочек из двух и более последовательных переходов (например, `Idle → CardInserted → PinValidation → MainMenu`).
- **Недопустимые/непредусмотренные переходы** — попытка вызвать событие, недопустимое в текущем состоянии (например, попытка выбрать сумму снятия, находясь в состоянии `Idle` без вставленной карты, — система не должна допускать такой переход).
- **Граничные и защитные сценарии**: 3 неверных ввода PIN подряд → переход в `CardRetained`; таймаут в состоянии `MainMenu` без действий → возврат в `Idle` с выдачей карты; извлечение карты вручную на любом этапе → корректный возврат в `Idle`.
- **Сценарии сбоя**: потеря связи с банком в состоянии `Processing` → переход в `ErrorState` с последующим возвратом карты, а не зависание или выдача денег без списания.

## Пример / когда применяется

Упрощённая таблица переходов состояний:

| Текущее состояние | Событие | Следующее состояние |
|---|---|---|
| Idle | Вставлена карта | CardInserted |
| CardInserted | Введён верный PIN | MainMenu |
| CardInserted | Введён неверный PIN (попытка 1-2) | CardInserted (запрос повторного ввода) |
| CardInserted | Введён неверный PIN (3-я попытка) | CardRetained |
| MainMenu | Выбрано "Снять наличные" | AmountSelection |
| AmountSelection | Введена сумма ≤ баланса и ≤ дневного лимита | Processing |
| AmountSelection | Введена сумма > баланса | ErrorState → MainMenu |
| Processing | Успешная обработка банком | CashDispensing |
| Processing | Сбой связи с банком | ErrorState → CardEjection |
| CashDispensing | Деньги выданы | CardEjection |
| CardEjection | Карта извлечена пользователем | Idle |
| MainMenu | Таймаут (нет действий N секунд) | CardEjection → Idle |

Тест-кейсы, построенные по этой таблице:
1. Вставить карту → ввести верный PIN → выбрать "Снять наличные" → ввести сумму 500 грн (кратную номиналу, ≤ баланса) → подтвердить → проверить выдачу денег и возврат карты.
2. Вставить карту → ввести неверный PIN трижды подряд → проверить, что карта изъята (`CardRetained`), а не возвращена.
3. Вставить карту → ввести верный PIN → выбрать снятие → указать сумму больше баланса → проверить сообщение об ошибке и возврат в `MainMenu` без списания средств.
4. Вставить карту → ввести верный PIN → выбрать снятие → указать сумму → в момент `Processing` симулировать обрыв связи с банком → проверить, что деньги не выданы, карта возвращена, средства не списаны (нет "зависших" транзакций).
5. Вставить карту → находиться в `MainMenu` дольше таймаута без действий → проверить автоматический возврат карты и переход в `Idle`.

## На что смотрит интервьюер

Интервьюер оценивает знание техники тест-дизайна State Transition Testing именно как формального метода: умение построить состояния, события, таблицу/диаграмму переходов, а не просто описать банкомат "своими словами". Ключевой момент, который выделяет сильного кандидата — упоминание проверки недопустимых переходов и защитных сценариев (изъятие карты, сбой связи, таймаут), а не только happy path снятия денег. Красный флаг — путаница между State Transition Testing и обычным сценарным (use case) тестированием, отсутствие понятия "недопустимый переход". Частый follow-up: "Что такое 0-switch и 1-switch coverage?" (проверка отдельных переходов против проверки цепочек из двух переходов) и "Как бы вы автоматизировали такой набор тестов?" (например, через модель конечного автомата в тестовом фреймворке).
