# Составить Smoke Test Suite для DOU.ua.

**Коротко:** smoke-набор для DOU.ua (портал вакансий/новостей для IT-сообщества) должен проверять доступность сайта, ключевые пользовательские сценарии (поиск вакансий, чтение статей, авторизация, отклик на вакансию) на верхнем уровне — "система работает в целом", без глубокого регресса.

## Развёрнутый ответ

Smoke Testing (дымовое тестирование) — это поверхностная, быстрая проверка критически важных функций системы сразу после сборки/деплоя, чтобы убедиться, что билд достаточно стабилен для дальнейшего полноценного тестирования. Цель smoke-набора — не найти все баги, а быстро отсечь "сломанные" сборки: если базовая функциональность не работает, дальнейшее тестирование бессмысленно.

Для составления smoke suite нужно сначала определить ключевые модули/пользовательские сценарии сайта DOU.ua (портал для IT-специалистов Украины: новости, статьи, вакансии, форум, компании, зарплатный опрос):

1. **Доступность сайта и базовая загрузка:**
   - Главная страница dou.ua открывается и возвращает 200 OK.
   - Основные статические ресурсы (CSS, JS, изображения) загружаются без ошибок в консоли.
   - Основное меню навигации (Новости, Вакансии, Форум, Компании и т.д.) отображается и кликабельно.

2. **Раздел вакансий (ключевая бизнес-функция портала):**
   - Страница списка вакансий открывается.
   - Поиск вакансий по ключевому слову (например, "QA") возвращает релевантные результаты.
   - Фильтрация вакансий (по городу, специализации, опыту) работает.
   - Открытие карточки конкретной вакансии показывает описание, требования, кнопку "Откликнуться".
   - Отклик на вакансию (базовый happy path, если доступен без полной верификации) не выдаёт ошибку 500.

3. **Раздел новостей/статей:**
   - Список последних новостей отображается на главной.
   - Открытие отдельной статьи показывает текст, дату, автора.
   - Комментарии под статьёй загружаются (если применимо).

4. **Регистрация/авторизация пользователя:**
   - Форма входа открывается.
   - Вход с валидными тестовыми учётными данными проходит успешно.
   - Выход из аккаунта работает.

5. **Форум:**
   - Список тем форума открывается.
   - Открытие конкретной темы показывает сообщения.

6. **Раздел компаний:**
   - Список компаний открывается.
   - Профиль конкретной компании отображается с описанием и списком вакансий.

7. **Поиск по сайту (глобальный):**
   - Поисковая строка возвращает результаты по общему запросу.

8. **Мобильная адаптивность (базовая проверка):**
   - Главная страница корректно отображается на мобильном разрешении экрана.

9. **Кросс-браузерная проверка (минимальный набор):**
   - Основные сценарии (открытие главной, поиск вакансии, открытие статьи) проверяются хотя бы в Chrome и Safari/Firefox.

Важно: smoke-набор должен быть быстрым (обычно 15-30 минут ручного прогона или несколько минут для автоматизированного) и покрывать только "happy path" критичных модулей, без глубокого перебора граничных значений и негативных сценариев — это задача полноценного regression/functional testing.

## Пример / когда применяется

Таблица примера smoke-набора для чек-листа:

| № | Модуль | Проверка | Ожидаемый результат |
|---|---|---|---|
| 1 | Главная | Открыть dou.ua | Страница загружается, статус 200, нет ошибок в консоли |
| 2 | Вакансии | Найти вакансию по запросу "QA Engineer" | Отображается список релевантных вакансий |
| 3 | Вакансии | Открыть карточку вакансии | Отображаются описание, требования, кнопка "Откликнуться" |
| 4 | Новости | Открыть последнюю статью с главной | Статья открывается, текст отображается корректно |
| 5 | Авторизация | Войти под тестовым аккаунтом | Успешный вход, отображается имя пользователя в шапке |
| 6 | Форум | Открыть раздел форума | Список тем отображается |
| 7 | Компании | Открыть профиль компании | Отображается карточка компании с вакансиями |
| 8 | Поиск | Ввести запрос в глобальный поиск | Возвращаются релевантные результаты |

Применяется smoke suite сразу после каждого деплоя на staging/prod и как первый шаг перед полноценным regression-циклом — если smoke падает, билд возвращается разработчикам без дальнейшего тестирования ("build acceptance test").

## На что смотрит интервьюер

Интервьюер проверяет, понимает ли кандидат разницу между smoke-тестированием и полноценным regression/functional тестированием (smoke — быстро, поверхностно, только критичные сценарии; regression — глубоко и полно). Хороший ответ структурирован по бизнес-модулям конкретного сайта (а не абстрактный "проверить, что сайт открывается"), показывает понимание специфики продукта (для DOU.ua это вакансии и статьи как основная ценность). Красный флаг — smoke suite из одного пункта ("сайт открывается") или, наоборот, слишком глубокий набор с граничными значениями (что уже не smoke, а functional testing). Частый follow-up: "Как бы вы автоматизировали этот smoke suite?" (например, Playwright/Selenium с CI-интеграцией, запуск после каждого деплоя).
