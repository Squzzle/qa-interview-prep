# Веб-страница с полями e-mail, password и кнопкой submit. Назовите отрицательные тест-кейсы, по которым можно проверить эту страницу.

**Коротко:** негативные тест-кейсы должны покрывать некорректный формат данных, граничные значения, пустые поля, спецсимволы/инъекции, поведение при сетевых и серверных ошибках, а также UX/security-аспекты (блокировка после N попыток, отображение пароля, XSS/SQLi).

## Развёрнутый ответ

Негативное тестирование формы с полями "e-mail", "password" и кнопкой "submit" направлено на проверку того, что система корректно обрабатывает некорректный, неполный или вредоносный ввод, не даёт сбоев и не создаёт уязвимостей. Такие кейсы делятся на несколько категорий:

**Валидация поля e-mail:**
- Пустое поле email.
- Email без символа `@` (например, `useremail.com`).
- Email без домена (`user@`).
- Email без имени пользователя (`@domain.com`).
- Email с двумя `@` (`user@@domain.com`).
- Email с пробелами внутри или по краям.
- Email со спецсимволами, не допустимыми стандартом (например, `user!#$%@domain.com` в незаэкранированном виде).
- Email с недопустимо длинной строкой (превышение максимальной длины поля/стандарта, например 320+ символов).
- Email с кириллицей/юникодом, если система не поддерживает IDN.
- Email, состоящий только из пробелов.
- Email с SQL-инъекцией (`' OR '1'='1`) или XSS-пейлоадом (`<script>alert(1)</script>`) — вместо выполнения скрипта или ошибки 500 должна быть корректная валидация/экранирование.

**Валидация поля password:**
- Пустое поле password.
- Пароль короче минимальной длины (например, 1-2 символа при минимуме 8).
- Пароль длиннее максимально допустимой длины.
- Пароль, состоящий только из пробелов.
- Пароль без учёта требований к сложности (если есть политика — только цифры, без спецсимволов и заглавных букв).
- Пароль с недопустимыми символами (если есть ограничения) или, наоборот, проверка что все "легальные" спецсимволы принимаются.
- Пароль с SQL-инъекцией/XSS-пейлоадом.

**Комбинированные и общие кейсы:**
- Оба поля пустые, нажатие submit.
- Заполнено только одно поле.
- Верный email, но неверный пароль (и наоборот) — проверка сообщения об ошибке (не должно раскрывать, какое именно поле неверно, из соображений безопасности — "неверный логин или пароль", а не "пароль неверный").
- Множественные быстрые нажатия submit (double-click) — не должно быть дублирующих запросов/дублирующих сессий.
- Ввод данных существующего, но заблокированного/неактивного пользователя.
- Превышение количества попыток входа — должна сработать блокировка/капча (rate limiting, protection от brute-force).
- Потеря соединения с сервером во время отправки — должна быть понятная ошибка, а не зависание UI.
- Ответ сервера с ошибкой 500/timeout — форма должна показать сообщение об ошибке, не терять введённые данные там, где это уместно (кроме пароля — его из соображений безопасности принято очищать).
- Проверка на XSS через оба поля при выводе значения обратно в UI (например, в сообщении об ошибке).
- Проверка, что пароль не отображается в открытом виде (нет утечки в консоль браузера, network tab не должен передавать пароль в открытом виде без HTTPS).
- Copy-paste длинной строки с невидимыми символами (zero-width characters, emoji) в оба поля.
- Автозаполнение браузером некорректных/старых данных.
- Отключённый JavaScript — серверная валидация должна работать независимо от клиентской.

## Пример / когда применяется

Пример тест-кейса в табличной форме:

| № | Входные данные (email / password) | Ожидаемый результат |
|---|---|---|
| 1 | `""` / `""` | Ошибка "Заполните обязательные поля", submit заблокирован |
| 2 | `test@test` / `Passw0rd!` | Ошибка валидации формата email |
| 3 | `test@test.com` / `""` | Ошибка "Введите пароль" |
| 4 | `test@test.com` / `123` (короче минимума) | Ошибка "Пароль должен содержать минимум N символов" |
| 5 | `' OR 1=1 --@test.com` / `anypass` | Запрос отклонён валидацией, нет SQL-ошибки, нет доступа |
| 6 | Верный email / неверный пароль (5-й раз подряд) | Аккаунт временно заблокирован / показана капча |

## На что смотрит интервьюер

Интервьюер оценивает системность мышления: разбивает ли кандидат кейсы по категориям (валидация формата, граничные значения, security, UX, сетевые ошибки), а не выдаёт хаотичный список из 3-4 очевидных пунктов. Хороший кандидат обязательно вспомнит про SQL-инъекции/XSS, brute-force защиту и поведение при сетевых сбоях — это то, что часто упускают junior-специалисты. Красный флаг — ограничиться только "email без @" и "пустой пароль", не подумав про безопасность и граничные значения. Частые follow-up вопросы: "А как бы вы автоматизировали эти кейсы?" и "Какие из этих кейсов high priority, а какие low?".
