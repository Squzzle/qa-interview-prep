# Что такое A/B тестирование?

**Коротко:** A/B тестирование — это метод сравнения двух (или более) вариантов продукта на реальных пользователях, разделённых на случайные группы, для определения, какой вариант лучше по заданной метрике, на основе статистически значимых данных.

## Развёрнутый ответ

A/B тестирование (split testing) — это экспериментальный метод, при котором пользователи случайным образом делятся на две (или более) группы: контрольную (A, текущая версия) и тестовую (B, новый вариант). Обе группы одновременно взаимодействуют со своей версией продукта, после чего сравниваются ключевые метрики для определения, какой вариант эффективнее.

Ключевые компоненты A/B теста:

**1. Гипотеза.** Формулируется чёткая гипотеза с ожидаемым эффектом: "Если изменить цвет кнопки CTA с серого на зелёный, то конверсия в покупку увеличится, потому что зелёный цвет более заметен и ассоциируется с действием."

**2. Метрика (success metric).** Определяется измеримый показатель успеха — конверсия, CTR, среднее время на странице, retention, revenue per user. Важно заранее определить и метрики "здоровья" (guardrail metrics), чтобы улучшение основной метрики не ухудшило другие важные показатели.

**3. Случайное разделение выборки (randomization).** Пользователи распределяются случайно между группами (обычно по user ID/session ID через хэш-функцию), чтобы группы были статистически сопоставимы и результат не был искажён внешними факторами.

**4. Размер выборки и длительность.** Заранее рассчитывается необходимый размер выборки (sample size) с учётом ожидаемого эффекта и требуемой статистической мощности (power), чтобы результат был статистически значим, а не случайным шумом.

**5. Статистическая значимость (statistical significance).** По завершении теста результаты проверяются на статистическую значимость (обычно p-value &lt; 0.05), чтобы убедиться, что наблюдаемая разница не случайна.

Роль QA в A/B тестировании:
- проверка корректности механизма разделения трафика (пользователь стабильно попадает в одну и ту же группу при повторных визитах);
- проверка, что обе версии (A и B) функционально корректны и не содержат багов;
- проверка корректности сбора и логирования аналитических событий (event tracking) для обеих групп — без этого метрики нельзя будет посчитать;
- проверка edge cases: что происходит, если пользователь одновременно попадает в несколько экспериментов (feature flag conflicts).

## Пример / когда применяется

Пример: интернет-магазин тестирует новый дизайн страницы товара.
- Группа A (50% трафика) видит текущую страницу с кнопкой "Купить" серого цвета.
- Группа B (50% трафика) видит новую версию с зелёной кнопкой и укороченным описанием.
- Метрика успеха: конверсия в добавление в корзину.
- Guardrail-метрика: средний чек (чтобы убедиться, что рост конверсии не происходит за счёт снижения среднего чека).
- Тест длится 2 недели, набирается статистически значимая выборка (например, по 50 000 пользователей в каждой группе).
- По результатам: конверсия в группе B выросла на 8% при p-value 0.02 — эффект статистически значим, средний чек не изменился → раскатывают вариант B на 100% пользователей.

```text
Группа A (control): 50% трафика → серая кнопка, конверсия 3.2%
Группа B (variant): 50% трафика → зелёная кнопка, конверсия 3.5%
p-value = 0.02 → результат статистически значим
```

## На что смотрит интервьюер

Ожидается упоминание не только механики "показали разным пользователям разное", но и понятий статистической значимости, guardrail-метрик и роли QA в проверке корректности разделения трафика и трекинга событий. Слабый ответ ограничивается фразой "это когда сравнивают две версии сайта". Возможные follow-up: "как убедиться, что результат A/B теста статистически значим, а не случаен?" или "что вы будете тестировать в системе feature flags/экспериментов?".
