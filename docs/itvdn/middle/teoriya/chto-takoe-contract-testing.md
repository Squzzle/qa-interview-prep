# Что такое contract testing?

**Коротко:** Contract testing — это проверка того, что взаимодействие между двумя сервисами (consumer и provider) соответствует согласованному "контракту" (формату запросов/ответов), позволяющая тестировать интеграции независимо, без необходимости деплоить оба сервиса вместе в одном окружении.

## Развёрнутый ответ

Contract testing (контрактное тестирование) решает проблему, характерную для микросервисной архитектуры: как убедиться, что сервис A (consumer, потребитель) и сервис B (provider, поставщик API) продолжают корректно взаимодействовать, если изменения в них происходят независимо, разными командами, с разной скоростью релизов, без полноценного end-to-end окружения со всеми сервисами одновременно.

Традиционный подход — E2E-интеграционное тестирование всей системы целиком — становится дорогим, медленным и хрупким при росте числа микросервисов (десятки сервисов, комбинаторный взрыв зависимостей, нестабильные окружения). Contract testing предлагает альтернативу: тестировать пару consumer-provider изолированно, проверяя не всю бизнес-логику, а соответствие согласованному контракту — структуре запроса и ответа (поля, типы, обязательность, статус-коды).

Основная модель — **Consumer-Driven Contract testing (CDC)**:

1. **Consumer** (потребитель API, например, frontend-сервис) пишет тест, который описывает, каких запросов и ответов он ожидает от provider'а. Этот тест генерирует **контракт** (файл-артефакт, обычно в формате JSON) — описание ожидаемого взаимодействия.
2. Контракт публикуется в **Broker** (централизованное хранилище контрактов, например Pact Broker).
3. **Provider** (например, backend-сервис) при своей сборке в CI забирает контракт из Broker и проверяет, что его реальный API действительно соответствует ожиданиям всех consumer'ов (verification stage) — т.е. что изменения в provider не сломают ни одного из потребителей.
4. Если provider меняет свой API так, что нарушает контракт какого-либо consumer'а, verification-тест падает в CI ещё до релиза provider'а — команда узнаёт об этом рано (shift-left), не дожидаясь E2E-тестов или продакшн-инцидента.

Преимущества contract testing:
- Не нужно поднимать все зависимые сервисы одновременно для тестирования интеграции — каждый сервис тестируется изолированно.
- Быстрая обратная связь: тесты выполняются в CI каждого сервиса независимо, без общего staging-окружения.
- Раннее обнаружение breaking changes в API до релиза, а не после интеграции в проде.
- Меньше "flaky" тестов по сравнению с полноценным E2E.

Ограничения: contract testing НЕ проверяет полную бизнес-логику и производительность — только структурное и поведенческое соответствие контракту. E2E-тесты критичных сквозных сценариев всё равно нужны, но в существенно меньшем объёме (согласно принципу тестовой пирамиды, где contract tests занимают промежуточный слой между unit и E2E).

Основные инструменты: **Pact** (самый популярный, поддерживает множество языков — Java, JS, Python, .NET, Pact Broker для хранения контрактов), **Spring Cloud Contract** (популярен в Java/Spring-экосистеме, альтернативный подход provider-driven), **Postman/Newman** с схемами для более простых случаев.

## Пример / когда применяется

Пример: сервис "Frontend" (consumer) ожидает от сервиса "Order Service" (provider) следующий ответ на `GET /orders/{id}`:

```json
{
  "id": 123,
  "status": "SHIPPED",
  "totalAmount": 49.99
}
```

Consumer-тест на Pact (псевдокод на JS):

```javascript
describe('Order Service contract', () => {
  it('returns order details', async () => {
    await provider.addInteraction({
      state: 'order 123 exists',
      uponReceiving: 'a request for order 123',
      withRequest: { method: 'GET', path: '/orders/123' },
      willRespondWith: {
        status: 200,
        body: { id: 123, status: 'SHIPPED', totalAmount: 49.99 },
      },
    });
  });
});
```

Этот тест генерирует контракт, который публикуется в Pact Broker. Когда команда Order Service меняет структуру ответа (например, переименовывает `totalAmount` в `total`), verification-тест на стороне provider падает в CI, и команда узнаёт о breaking change до деплоя.

Contract testing особенно применим в микросервисной архитектуре с множеством независимых команд, при частых релизах, и когда полноценное E2E-окружение дорого поддерживать.

## На что смотрит интервьюер

Интервьюер проверяет понимание проблемы, которую решает contract testing (хрупкость и дороговизна полного E2E в микросервисах), а не просто определение термина. Ожидается упоминание consumer-driven подхода, роли Broker'а и хотя бы одного инструмента (Pact). Частая ошибка — путать contract testing с обычным API-тестированием схемы ответа (JSON Schema validation) одного сервиса — contract testing принципиально включает две стороны (consumer и provider) и механизм verification на стороне provider. Плюс в ответе — понимание места contract-тестов в тестовой пирамиде.
