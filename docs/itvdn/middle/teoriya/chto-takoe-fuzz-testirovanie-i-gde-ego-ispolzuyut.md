# Что такое fuzz-тестирование и где его используют?

**Коротко:** Fuzz-тестирование (fuzzing) — это техника автоматизированного тестирования, при которой в систему подаются большие объёмы случайных, некорректных, граничных или специально сгенерированных данных с целью найти сбои, падения, утечки памяти и уязвимости, не выявляемые обычными позитивными тестами.

## Развёрнутый ответ

Fuzzing реализуется с помощью специальных инструментов (фаззеров), которые генерируют входные данные по одному из подходов:

- **Mutation-based (мутационный)** — берётся валидный образец входных данных (файл, HTTP-запрос, строка) и случайным образом изменяется: инвертируются биты, вставляются спецсимволы, обрезается длина, меняются значения полей.
- **Generation-based (генеративный)** — данные генерируются с нуля на основе формальной модели формата (грамматики, схемы протокола), что позволяет глубже проверять сложные форматы (JSON, XML, бинарные протоколы).
- **Coverage-guided (управляемый покрытием кода)** — наиболее современный подход (AFL, libFuzzer, honggfuzz): фаззер инструментирует бинарник, отслеживает, какие ветки кода были выполнены, и мутирует входные данные так, чтобы максимизировать покрытие кода и находить новые пути выполнения.

Цель fuzzing — не проверка бизнес-логики "работает ли фича правильно", а поиск того, что происходит при выходе за границы ожидаемого поведения: падения приложения (crash), необработанные исключения, зависания, переполнения буфера, утечки памяти, некорректная обработка кодировок, уязвимости типа SQL-инъекций, XSS, десериализации.

Fuzzing особенно эффективен там, где приложение принимает сложные, слабоструктурированные или бинарные входные данные: парсеры файлов, сетевые протоколы, компиляторы, API, форматы изображений/видео.

## Пример / когда применяется

Типичные области применения:

- Тестирование безопасности API и веб-приложений (передача некорректных JSON/XML, SQL-инъекции, переполнение полей).
- Тестирование парсеров файлов (PDF, изображения, аудио/видео кодеки) — именно так находят большинство CVE в библиотеках вроде libpng, FFmpeg.
- Тестирование сетевых протоколов и драйверов.
- Аудит безопасности библиотек с открытым исходным кодом (Google OSS-Fuzz непрерывно фаззит сотни open-source проектов).

Пример простого fuzz-теста на Python с использованием библиотеки `atheris` (аналог libFuzzer для Python):

```python
import atheris
import sys

def parse_input(data: bytes):
    # тестируемая функция парсинга
    if len(data) > 4 and data[:4] == b"\x00\x01\x02\x03":
        if data[4:5] == b"\xff":
            raise ValueError("Found the bug!")

def TestOneInput(data):
    parse_input(data)

atheris.Setup(sys.argv, TestOneInput)
atheris.Fuzz()
```

## На что смотрит интервьюер

Проверяется, не путает ли кандидат fuzzing с обычным негативным тестированием "вручную подобранных" некорректных значений — ключевое отличие fuzzing в масштабе, автоматизации и случайности/мутациях входных данных. Хороший ответ упоминает конкретные инструменты (AFL, libFuzzer, OSS-Fuzz, Peach, Burp Intruder для веб) и связь fuzzing с security-тестированием и DevSecOps. Красный флаг — если кандидат не может объяснить разницу между mutation-based и coverage-guided подходами или не знает, что fuzzing обычно ищет crash/hang, а не бизнес-баги.
