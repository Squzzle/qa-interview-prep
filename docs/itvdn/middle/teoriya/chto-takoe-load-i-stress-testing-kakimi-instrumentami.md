# Что такое load и stress testing? Какими инструментами пользуются для их выполнения?

**Коротко:** Load testing проверяет поведение системы под ожидаемой/плановой нагрузкой (укладывается ли она в SLA), а stress testing намеренно превышает эти пределы, чтобы найти точку отказа и оценить деградацию и восстановление системы; инструменты для обоих видов — JMeter, Gatling, k6, LoadRunner.

## Развёрнутый ответ

Load testing и stress testing — оба являются подвидами performance-тестирования, но решают разные задачи и часто путаются между собой на собеседованиях, поэтому важно чётко разграничивать их цель.

**Load testing (нагрузочное тестирование)** — проверка поведения системы при ожидаемой, реалистичной нагрузке, соответствующей нормальным или пиковым, но предсказуемым условиям эксплуатации (например, среднее и максимальное число одновременных пользователей по бизнес-прогнозу). Цель — убедиться, что система соответствует SLA/NFR по времени отклика, пропускной способности и стабильности при плановой нагрузке, без деградации.

Пример вопроса, на который отвечает load testing: "Выдержит ли система 5000 одновременных пользователей с response time p95 &lt; 500ms, как это заявлено в требованиях?"

**Stress testing (стресс-тестирование)** — целенаправленное превышение нормальной нагрузки, зачастую до предела и за него, чтобы найти:
- **breaking point** — точку, при которой система перестаёт корректно работать (ошибки, критическое замедление, отказ);
- как система деградирует под чрезмерной нагрузкой (graceful degradation vs полный крах);
- восстанавливается ли система после снятия нагрузки (recovery testing) — не остаются ли "зависшие" соединения, утечки ресурсов, неконсистентные данные.

Пример вопроса для stress testing: "При какой нагрузке система перестанет отвечать, и как она себя поведёт — упадёт полностью, начнёт отклонять новые запросы (circuit breaker) или медленно деградирует?"

Ключевые различия:

| Критерий | Load testing | Stress testing |
|---|---|---|
| Цель | Подтвердить соответствие SLA при ожидаемой нагрузке | Найти предел и точку отказа |
| Уровень нагрузки | В пределах нормы/плана | Превышающий нормальный, до отказа и выше |
| Что ищем | Соответствие NFR (response time, throughput) | Breaking point, деградацию, recovery |
| Типичный результат | "Проходит/не проходит SLA" | "Система падает при X RPS, требуется Y секунд на восстановление" |

Часто эти виды тестирования проводятся последовательно: сначала load testing для подтверждения базовых требований, затем stress testing для понимания запаса прочности и планирования capacity (сколько ресурсов нужно на рост бизнеса, автоскейлинг).

Инструменты для выполнения обоих видов идентичны, разница только в конфигурации сценария нагрузки (профиле нагрузки — количество виртуальных пользователей, ramp-up/ramp-down):

- **Apache JMeter** — гибкая настройка Thread Group для постепенного увеличения нагрузки (ramp-up period), плагины для мониторинга.
- **Gatling** — DSL позволяет легко описать сценарий постепенного наращивания (`rampUsers`, `constantUsersPerSec`, `incrementUsersPerSec`).
- **k6** — поддержка stages в конфигурации для описания профиля нагрузки (постепенный рост до пика и стресс-уровня).
- **LoadRunner** — enterprise-решение с готовыми сценариями load/stress.
- **Locust** — гибкая настройка через код количества пользователей и hatch rate.
- APM-инструменты (Grafana/Prometheus, New Relic) — обязательны при stress testing, чтобы видеть, что именно отказывает первым (CPU, память, connection pool, диск).

## Пример / когда применяется

Пример конфигурации stress-теста в k6 с постепенным наращиванием нагрузки выше нормы:

```javascript
export const options = {
  stages: [
    { duration: '2m', target: 100 },   // load testing: нормальная нагрузка
    { duration: '2m', target: 500 },   // приближение к пределу
    { duration: '2m', target: 1500 },  // stress testing: превышение нормы
    { duration: '2m', target: 0 },     // recovery — снятие нагрузки
  ],
};
```

Load testing применяется регулярно перед каждым крупным релизом для подтверждения, что производительность не деградировала. Stress testing применяется реже — перед крупными событиями (распродажи, маркетинговые кампании) или при капасити-планировании инфраструктуры, чтобы понять запас прочности системы.

## На что смотрит интервьюер

Главная проверка — не путает ли кандидат load и stress testing (частейшая ошибка на собеседованиях). Хороший ответ чётко разграничивает цель ("подтвердить SLA" vs "найти предел"), упоминает recovery testing как часть стресс-тестирования и знает, что оба вида используют одни и те же инструменты, различаясь лишь профилем нагрузки. Плюс балл — если кандидат упомянет мониторинг ресурсов (APM) как необходимую часть анализа, а не просто фиксацию response time.
