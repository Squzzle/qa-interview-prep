# Что такое mock/stub? Какие знаете инструменты для работы с ними?

**Коротко:** Mock и stub — это тестовые двойники (test doubles), заменяющие реальные зависимости в тестах: stub возвращает заранее заданные данные, а mock дополнительно проверяет, как именно тестируемый код с ним взаимодействовал.

## Развёрнутый ответ

Test double — общий термин (по классификации Джерарда Мезароша) для объектов, заменяющих реальные зависимости в тестах. Основные виды:

- **Dummy** — объект-заглушка, передаётся, потому что нужен по сигнатуре, но никогда реально не используется в тесте.
- **Stub** — возвращает заранее заданные (hardcoded) ответы на вызовы, не имеет "интеллекта", просто предоставляет данные, необходимые для теста, не проверяя взаимодействие.
- **Mock** — объект, который не только возвращает данные, но и в конце теста проверяется на то, был ли вызван нужный метод, с какими аргументами и сколько раз (verify). Mock используется, когда важно проверить *поведение* (behavior verification), а не только результат.
- **Fake** — рабочая, но упрощённая реализация зависимости (например, in-memory база данных вместо реальной СУБД) — работает "по-настоящему", но не подходит для production.
- **Spy** — обёртка вокруг реального объекта, которая записывает, как он был вызван, при этом делегируя реальный вызов исходному объекту (частичный mock).

Ключевое отличие stub от mock: stub используется для *state verification* ("что вернул объект и что получилось в результате"), а mock — для *behavior verification* ("был ли вызван метод X с параметром Y ровно 1 раз"). Злоупотребление mock-верификацией без нужды делает тесты хрупкими (over-specification) — тест начинает ломаться при рефакторинге, даже если поведение системы не изменилось.

Зачем нужны mock/stub:
- изоляция тестируемого юнита от внешних зависимостей (БД, сеть, файловая система, сторонние API) — тест становится быстрым и детерминированным;
- возможность смоделировать редкие/сложные условия (сетевая ошибка, таймаут, конкретный edge case ответа API), которые сложно воспроизвести с реальной зависимостью;
- ускорение тестов (не нужно ждать реальные I/O операции);
- тестирование в изоляции, когда реальная зависимость ещё не готова (параллельная разработка).

Инструменты:

**Для unit-тестирования (Java):** Mockito, EasyMock, JMock, PowerMock (для legacy кода со статическими методами/final классами).

**Для unit-тестирования (JS/TS):** Jest (встроенные `jest.mock`, `jest.fn`), Sinon.js, ts-mockito.

**Для unit-тестирования (Python):** `unittest.mock`, pytest-mock.

**Для мокирования HTTP/API на уровне интеграционных тестов:** WireMock, MockServer, Nock (Node.js), Postman Mock Server, Hoverfly.

**Для контрактного тестирования (contract testing) микросервисов:** Pact — позволяет и потребителю, и поставщику API верифицировать соответствие контракту без реального взаимодействия.

## Пример / когда применяется

Пример на Java с Mockito — тестирование сервиса, зависящего от `PaymentGateway`:

```java
@Test
void shouldReturnErrorWhenPaymentGatewayFails() {
    PaymentGateway gatewayMock = mock(PaymentGateway.class);
    when(gatewayMock.charge(any(Order.class))).thenThrow(new PaymentException("Timeout"));

    OrderService service = new OrderService(gatewayMock);
    OrderResult result = service.checkout(new Order(100));

    assertEquals(OrderStatus.FAILED, result.getStatus());
    verify(gatewayMock, times(1)).charge(any(Order.class));
}
```

Здесь `gatewayMock` выступает в роли stub (задаёт поведение `when...thenThrow`) и одновременно проверяется как mock (`verify(...times(1))`) — на практике границы между stub и mock часто размыты в одном объекте, что и создаёт неоднозначность терминологии на собеседованиях.

Пример WireMock для интеграционного теста, мокирующего внешний REST API:

```json
{
  "request": { "method": "GET", "url": "/api/currency/USD" },
  "response": { "status": 200, "jsonBody": { "rate": 90.5 } }
}
```

## На что смотрит интервьюер

Ожидается чёткое разграничение stub (state verification) и mock (behavior verification), а не использование терминов как синонимов. Хорошим сигналом является упоминание проблемы over-mocking и хрупкости тестов при чрезмерной behavior-верификации. Кандидат должен назвать конкретные инструменты для своего стека, а не только "Mockito" в общем.
