# Что такое performance testing? Какими инструментами пользуются для его выполнения?

**Коротко:** Performance testing — это нефункциональное тестирование, оценивающее скорость отклика, пропускную способность и стабильность системы под различными видами нагрузки; основные инструменты — JMeter, Gatling, k6, LoadRunner, Locust, а также APM-системы (Grafana, Prometheus, New Relic) для мониторинга во время теста.

## Развёрнутый ответ

Performance testing (тестирование производительности) — это зонтичный термин, объединяющий несколько видов нефункционального тестирования, цель которых — оценить, как система ведёт себя с точки зрения скорости, отзывчивости, стабильности и масштабируемости под разными условиями нагрузки.

Ключевые метрики, которые измеряются:

- **Response time (время отклика)** — время от отправки запроса до получения ответа, часто измеряется как среднее и в перцентилях (p50, p90, p95, p99 — важно смотреть именно перцентили, а не только среднее, так как среднее скрывает выбросы).
- **Throughput (пропускная способность)** — количество запросов/транзакций в единицу времени (RPS, TPS).
- **Latency** — задержка сети/обработки до начала ответа.
- **Error rate** — процент неуспешных запросов под нагрузкой.
- **Утилизация ресурсов** — CPU, память, диск, сеть на сервере во время теста.
- **Concurrency** — количество одновременных пользователей/соединений, которое система способна обслуживать.

Разновидности performance-тестирования (важно отличать на собеседовании):

- **Load testing** — проверка поведения системы при ожидаемой/плановой нагрузке (например, 500 одновременных пользователей).
- **Stress testing** — постепенное увеличение нагрузки до отказа системы, чтобы найти точку разрушения (breaking point) и понять, как система деградирует и восстанавливается.
- **Spike testing** — резкий скачок нагрузки (например, в 10 раз выше нормы за секунды) для проверки устойчивости к внезапным всплескам.
- **Soak/Endurance testing** — длительное тестирование под средней нагрузкой (часы, дни) для выявления утечек памяти, деградации со временем.
- **Scalability testing** — проверка, насколько линейно система масштабируется при добавлении ресурсов (горизонтально/вертикально).
- **Volume testing** — проверка поведения при больших объёмах данных в базе/хранилище.

Процесс: определение NFR (non-functional requirements) и SLA до старта, подготовка тестового окружения (по возможности приближенного к production), написание сценариев нагрузки, постепенное увеличение нагрузки с мониторингом метрик, анализ узких мест (bottlenecks), формирование отчёта с рекомендациями.

Основные инструменты:

- **Apache JMeter** — самый распространённый open-source инструмент, поддерживает HTTP, JDBC, JMS, FTP и другие протоколы, богатая экосистема плагинов, GUI и CLI режимы.
- **Gatling** — Scala-based инструмент с DSL для написания сценариев как код, хорошая визуализация отчётов, эффективен по ресурсам (асинхронная модель на Akka/Netty).
- **k6** — современный инструмент от Grafana Labs, сценарии на JavaScript, отлично интегрируется в CI/CD, лёгкий и удобный для cloud-native нагрузочного тестирования.
- **LoadRunner (Micro Focus)** — коммерческий enterprise-инструмент, поддерживает широкий спектр протоколов, используется в крупных организациях.
- **Locust** — Python-based инструмент, сценарии описываются кодом, поддерживает распределённое нагрузочное тестирование.
- **Apache Bench (ab), wrk** — простые CLI-утилиты для быстрых базовых замеров.
- **Grafana + Prometheus, New Relic, Datadog, AppDynamics (APM)** — используются не для генерации нагрузки, а для мониторинга состояния системы (CPU, память, latency по сервисам) во время нагрузочного теста, что критично для выявления bottleneck.

## Пример / когда применяется

Пример базового сценария на k6:

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  vus: 100,           // 100 виртуальных пользователей
  duration: '5m',     // длительность теста
};

export default function () {
  const res = http.get('https://api.example.com/products');
  check(res, {
    'status is 200': (r) => r.status === 200,
    'response time < 300ms': (r) => r.timings.duration < 300,
  });
  sleep(1);
}
```

Performance testing применяется перед крупными релизами, распродажами (Black Friday), запуском маркетинговых кампаний, а также регулярно как часть CI/CD (performance regression testing), чтобы вовремя заметить деградацию с новым релизом.

## На что смотрит интервьюер

Интервьюер проверяет умение различать виды нагрузочного тестирования (load/stress/spike/soak) — частая ошибка кандидатов путать их между собой или считать синонимами. Также важно понимание, что средний response time — плохая метрика, и нужно смотреть перцентили. Плюс за упоминание, что performance testing требует не только генератора нагрузки, но и мониторинга ресурсов сервера (APM) для поиска bottleneck, а не только измерения времени ответа клиента.
