# Что такое таблица решений/decision table и как её можно использовать?

**Коротко:** Таблица решений — техника тест-дизайна, которая систематизирует все возможные комбинации входных условий и соответствующих им действий/результатов системы, помогая не пропустить комбинации при тестировании сложной бизнес-логики.

## Развёрнутый ответ

Decision table (таблица решений) — это способ представить бизнес-правила в виде таблицы, где строки — это условия (conditions) и действия (actions), а столбцы — правила (rules), то есть уникальные комбинации значений условий и соответствующий им результат.

Структура таблицы:
- **Conditions (условия)** — входные параметры, влияющие на поведение системы (например, "возраст >= 18", "есть купон", "товар в наличии").
- **Condition values** — возможные значения условий (обычно true/false, но могут быть и множественные значения).
- **Actions (действия/результаты)** — что должна сделать система при данной комбинации условий.
- **Rules (правила)** — столбцы, каждый из которых соответствует одной уникальной комбинации условий и её результату.

Технику применяют, когда:
- логика системы описывается несколькими независимыми (или частично зависимыми) булевыми/дискретными условиями;
- нужно гарантированно покрыть все комбинации условий, включая неочевидные;
- бизнес-правила сложны для восприятия в виде обычного текстового описания (легко упустить edge case).

Количество правил в полной таблице для n бинарных условий равно 2^n. При большом числе условий таблицу оптимизируют, объединяя правила с одинаковым результатом и незначащими условиями (используя "-"/"any" для условий, которые не влияют на конкретное правило) — это называется collapsed/consolidated decision table.

Таблица решений тесно связана с методом причинно-следственных связей (cause-effect graphing) — по сути, decision table является табличным представлением такого графа.

## Пример / когда применяется

Пример: логика скидки в интернет-магазине зависит от трёх условий: (1) сумма заказа >= 3000 руб., (2) есть промокод, (3) пользователь — постоянный клиент.

| Условие | R1 | R2 | R3 | R4 | R5 | R6 | R7 | R8 |
|---|---|---|---|---|---|---|---|---|
| Сумма >= 3000 | Да | Да | Да | Да | Нет | Нет | Нет | Нет |
| Есть промокод | Да | Да | Нет | Нет | Да | Да | Нет | Нет |
| Постоянный клиент | Да | Нет | Да | Нет | Да | Нет | Да | Нет |
| **Результат: скидка** | 15% | 10% | 10% | 5% | 10% | 5% | 5% | 0% |

Каждый столбец (R1-R8) превращается в отдельный тест-кейс, что гарантирует покрытие всех комбинаций бизнес-правила без пропуска граничных случаев (например, "постоянный клиент без промокода и с маленькой суммой").

## На что смотрит интервьюер

Ожидается, что кандидат объяснит структуру (условия/действия/правила), приведёт формулу 2^n и покажет практический пример, а не только определение "это таблица с условиями". Частая ошибка — путать decision table с state transition table (таблицей переходов состояний), которая описывает не комбинации условий, а последовательные состояния системы. Могут спросить про оптимизацию большой таблицы (collapsing) и про то, чем decision table отличается от pairwise-тестирования.
