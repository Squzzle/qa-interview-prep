# Что такое тестирование API? Какими инструментами пользуются для его выполнения?

**Коротко:** API-тестирование — это проверка бизнес-логики, контракта, безопасности и производительности программного интерфейса (REST, SOAP, GraphQL, gRPC) на уровне запросов/ответов, без обращения к UI; основные инструменты — Postman, REST Assured, SoapUI, curl, Swagger/OpenAPI, а для автоматизации — фреймворки на Java/Python/JS.

## Развёрнутый ответ

API-тестирование — это уровень тестирования, направленный на проверку программных интерфейсов (Application Programming Interface), через которые взаимодействуют разные компоненты системы (клиент-сервер, микросервисы между собой). В отличие от UI-тестирования, здесь проверяется бизнес-логика, обработка данных, статусы ответов и контракты напрямую, минуя графический интерфейс — это делает тесты быстрее, стабильнее (меньше "flaky" из-за верстки/анимаций) и дешевле в поддержке.

Что проверяется при API-тестировании:

- **Функциональность** — соответствие ответа ожидаемому бизнес-поведению (корректные данные, правильные вычисления).
- **HTTP-статус коды** — 200, 201, 400, 401, 403, 404, 409, 500 и соответствие их семантике (не возвращает ли API 200 при ошибке).
- **Структура и схема ответа (contract)** — соответствие JSON/XML схеме (JSON Schema, OpenAPI/Swagger спецификации), типы полей, обязательность полей.
- **Валидация входных данных** — граничные значения, некорректные типы, отсутствующие обязательные поля, SQL-инъекции и XSS в параметрах.
- **Аутентификация и авторизация** — корректная работа токенов (JWT, OAuth2), проверка доступа по ролям, истечение токена.
- **Производительность на уровне отдельного эндпоинта** — время ответа, поведение при большом объёме данных в ответе.
- **Идемпотентность и side-эффекты** — например, повторный `POST` не должен создавать дубликат, если это не предусмотрено; `PUT`/`DELETE` должны быть идемпотентны.
- **Обработка ошибок** — корректность и информативность error-response при некорректных запросах.
- **Совместимость версий API** — обратная совместимость при версионировании (`/v1/`, `/v2/`).

Основные инструменты:

- **Postman** — самый популярный инструмент для ручного и полуавтоматизированного тестирования REST API: коллекции запросов, переменные окружения, встроенные тесты на JavaScript (`pm.test`), Newman для запуска коллекций в CI/CD.
- **REST Assured** — Java-библиотека для автоматизации тестирования REST API, интегрируется с JUnit/TestNG, поддерживает fluent-синтаксис (given/when/then) и JSON Path для валидации ответов.
- **SoapUI / ReadyAPI** — инструмент для тестирования SOAP и REST сервисов, поддержка сложных сценариев с data-driven тестированием.
- **curl / HTTPie** — консольные утилиты для быстрой ручной проверки запросов.
- **Swagger UI / OpenAPI** — используется не только для документации, но и для валидации, что реализация соответствует контракту (schema validation), а также для генерации тестов.
- **Insomnia** — альтернатива Postman, легче и проще в некоторых сценариях.
- **Karate DSL** — фреймворк на базе Gherkin-подобного синтаксиса специально для API-тестирования, совмещает читаемость BDD и мощь программного тестирования.
- **k6, JMeter, Gatling** — инструменты для нагрузочного тестирования API (переход от функционального к performance-тестированию).
- **Python: requests + pytest**, **JS: axios/supertest + Jest/Mocha** — популярные стеки для автоматизации API-тестов в соответствующих экосистемах.
- **Pact** — для contract-тестирования между микросервисами (consumer-driven contracts).

## Пример / когда применяется

Пример теста на REST Assured для проверки эндпоинта получения пользователя:

```java
given()
    .header("Authorization", "Bearer " + token)
.when()
    .get("/api/users/{id}", 42)
.then()
    .statusCode(200)
    .body("id", equalTo(42))
    .body("email", matchesPattern(".+@.+\\..+"))
    .body("$", matchesJsonSchemaInClasspath("user-schema.json"));
```

API-тестирование применяется на всех этапах: разработчики покрывают эндпоинты unit/интеграционными тестами ещё до готовности UI (что позволяет тестировать бизнес-логику параллельно с фронтендом), QA-инженеры пишут E2E API-тесты для критичных сценариев (регистрация → логин → создание заказа), а перед релизом микросервисной архитектуры применяется contract testing, чтобы убедиться, что изменение одного сервиса не сломает потребителей его API.

## На что смотрит интервьюер

Интервьюер ожидает не просто перечисление инструментов, а понимание, ЧТО именно проверяется на уровне API (статусы, схема, авторизация, идемпотентность), и почему API-тестирование выгоднее UI-тестирования по скорости и стабильности (пирамида тестирования). Частая ошибка — назвать только Postman и не знать программных фреймворков для автоматизации (REST Assured, Karate). Плюс в ответе — упоминание contract testing и различий REST/SOAP/GraphQL подходов к тестированию.
