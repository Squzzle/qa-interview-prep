# Что знаете о тестировании нагрузки? В каком случае следует проводить такое тестирование? На каком этапе готовности продукта?

**Коротко:** Нагрузочное тестирование (load testing) проверяет поведение системы под ожидаемой и пиковой нагрузкой, а проводится оно тогда, когда система (или её ключевые компоненты) уже стабильна функционально и доступна в среде, близкой к production.

## Развёрнутый ответ

Нагрузочное тестирование — вид performance-тестирования, цель которого определить, как система ведёт себя при заданном количестве одновременных пользователей/запросов, и выявить узкие места (bottlenecks) до того, как их обнаружат реальные пользователи.

Разновидности нагрузочного/производительного тестирования, которые часто путают:
- **Load testing** — проверка поведения системы при ожидаемой нагрузке (типичный и пиковый трафик).
- **Stress testing** — намеренное превышение нагрузки сверх допустимой, чтобы найти точку отказа и проверить graceful degradation.
- **Spike testing** — резкий скачок нагрузки (например, при флеш-распродаже) и оценка реакции системы.
- **Endurance/Soak testing** — длительная работа под средней нагрузкой для выявления утечек памяти, деградации производительности со временем.
- **Volume testing** — проверка поведения при больших объёмах данных (не пользователей).
- **Scalability testing** — проверка, насколько линейно система масштабируется при добавлении ресурсов.

Ключевые метрики: время отклика (response time), пропускная способность (throughput, RPS/TPS), использование ресурсов (CPU, RAM, I/O), количество ошибок под нагрузкой, время восстановления после пиковой нагрузки.

Когда проводить: нагрузочное тестирование целесообразно, когда:
- ожидается значительный рост числа пользователей (маркетинговая кампания, сезонные распродажи, релиз новой платформы);
- приложение критично к времени отклика (финтех, e-commerce, стриминг);
- меняется архитектура (переход на микросервисы, смена БД, добавление кэширования) — нужно подтвердить, что изменения не деградировали производительность;
- есть SLA по производительности, которые нужно формально подтвердить.

На каком этапе готовности продукта: нагрузочное тестирование обычно проводится не раньше, чем система функционально стабильна — то есть после завершения основного цикла функционального тестирования, когда ключевые сценарии (critical path) работают корректно. Проводить его на самых ранних стадиях (при нестабильном функционале) бессмысленно — результаты будут нерепрезентативны, а баги будут "маскировать" реальные проблемы производительности. Идеально — на staging/pre-production окружении, максимально близком по конфигурации к production, незадолго до релиза или перед ожидаемым пиком нагрузки. При этом отдельные компоненты (например, конкретный API endpoint или запрос к БД) можно нагрузочно тестировать и раньше, в рамках модульного performance-тестирования (shift-left подход, continuous performance testing в CI/CD).

## Пример / когда применяется

Пример сценария: интернет-магазин готовится к "чёрной пятнице". За 2-3 недели до события команда:
1. Готовит staging-окружение, максимально идентичное production по инфраструктуре.
2. С помощью JMeter/Gatling/k6 моделирует ожидаемый пиковый трафик (например, x5 от обычного).
3. Постепенно увеличивает нагрузку (ramp-up), фиксируя метрики response time и error rate.
4. Проводит stress test — превышает ожидаемый пик, чтобы найти точку отказа (например, при 1000 RPS начинает расти латентность БД).
5. По результатам оптимизирует индексы БД, добавляет кэширование, настраивает автоскейлинг.

```bash
# Пример простого нагрузочного теста через k6
k6 run --vus 500 --duration 5m load-test.js
```

## На что смотрит интервьюер

Ошибка кандидатов — путать load, stress и volume тестирование или говорить, что нагрузочное тестирование "делается в самом конце после всего". Интервьюер ждёт понимания, что: (1) есть разные виды performance-тестирования с разными целями, (2) время проведения зависит от готовности функционала и цели теста, (3) кандидат может назвать конкретные метрики и инструменты (JMeter, Gatling, k6, LoadRunner). Частый follow-up: "чем отличается throughput от latency?" или "что делать, если под нагрузкой начали падать 500-е ошибки?".
