# Для какого функционала труднее всего написать тест-кейсы?

**Коротко:** Сложнее всего писать тест-кейсы для функционала с недетерминированным поведением (асинхронность, конкурентность, ML/AI), для интеграций с внешними системами, а также для UX/визуальных и security-аспектов, где сложно формализовать "ожидаемый результат".

## Развёрнутый ответ

Тест-кейс подразумевает предсказуемую пару "шаги → ожидаемый результат". Сложность возникает там, где ожидаемый результат трудно однозначно зафиксировать или где поведение системы нестабильно между прогонами.

Основные категории такого функционала:

- **Асинхронный и многопоточный код** — race condition, deadlock, порядок обработки событий в очередях. Результат зависит от таймингов, которые сложно контролировать и воспроизводить детерминированно в тест-кейсе.
- **Интеграции с внешними системами / третьими сторонами** — платёжные шлюзы, банковские API, сторонние сервисы доставки. Поведение зависит от внешней стороны, тестовые окружения нестабильны, ожидаемый результат может меняться без предупреждения.
- **Функционал на основе Machine Learning / AI** — рекомендательные системы, чат-боты, поиск с ранжированием. Нет единственного "правильного" ответа — оценка идёт через метрики (precision/recall, relevance score), а не через точное совпадение.
- **Производительность и нагрузка** — результат зависит от окружения, инфраструктуры, конкурентной нагрузки; тест-кейс с фиксированными шагами плохо описывает "система должна выдержать 1000 RPS с latency &lt; 200ms" — тут нужны отдельные NFR-критерии, а не классический шаг-результат.
- **UX/визуальный/удобство использования** — "удобно ли пользоваться" — субъективный критерий, который плохо формализуется в бинарный pass/fail.
- **Security-тестирование** — многие уязвимости не имеют предсказуемого шаблона атаки — нужен творческий подход (аналог exploratory), а не фиксированный сценарий.
- **Данные с большим количеством комбинаций (комбинаторный взрыв)** — например, форма с 10 независимыми полями валидации — полное покрытие комбинаций нереалистично, приходится применять pairwise/orthogonal array вместо явного перечисления в кейсах.
- **Legacy-системы без документации** — неизвестно, какое поведение является "ожидаемым", а какое — багом, ставшим частью бизнес-логики.

## Пример / когда применяется

Пример: тестирование системы уведомлений, где push-уведомление должно прийти "в течение нескольких секунд" после события в очереди Kafka с несколькими consumer-группами. Написать классический тест-кейс "Шаг 1: отправить событие. Шаг 2: проверить, что уведомление пришло" недостаточно — нужно учитывать retry-логику, идемпотентность, возможную задержку и порядок обработки. Здесь вместо жёсткого кейса используют property-based тестирование или тестирование с ретраями и допуском по времени (eventual consistency check).

Другой пример — тестирование рекомендательного блока "вам может понравиться": вместо точного тест-кейса используют критерии оценки релевантности (top-N precision) на выборке пользователей.

## На что смотрит интервьюер

Интервьюер проверяет, умеет ли кандидат выйти за рамки шаблонного "шаг-результат" мышления и распознать классы функционала, требующие иных подходов к верификации (метрики, exploratory, property-based тестирование, NFR). Слабый ответ — просто "сложный функционал сложно тестировать" без конкретики. Хороший ответ приводит 2-3 разных класса причин сложности (недетерминированность, внешние зависимости, субъективность критериев) с примерами и называет альтернативные техники верификации.
