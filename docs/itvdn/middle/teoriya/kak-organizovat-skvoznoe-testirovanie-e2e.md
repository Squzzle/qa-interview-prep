# Как организовать сквозное тестирование (e2e)?

**Коротко:** E2E-тестирование проверяет полный бизнес-сценарий от начала до конца через все взаимодействующие компоненты системы, и его организация требует стабильного окружения, управляемых тестовых данных и продуманной стратегии стабильности (anti-flakiness).

## Развёрнутый ответ

End-to-end (e2e) тестирование — это проверка полного пользовательского сценария (workflow), проходящего через все слои и компоненты системы (UI, backend, БД, внешние интеграции, очереди сообщений), имитируя реальное поведение пользователя от начала до конца.

Этапы организации e2e-тестирования:

**1. Определение критичных сквозных сценариев.** Не нужно покрывать e2e-тестами всё — это дорого и медленно. Выбираются критичные для бизнеса пути (critical user journeys): регистрация → заказ → оплата → доставка, например.

**2. Подготовка тестового окружения.** Окружение должно максимально повторять production по конфигурации, но быть изолированным (staging/UAT). Внешние зависимости, которые нельзя тестировать напрямую (платёжные системы, сторонние API), заменяются на sandbox-версии или моки/стабы.

**3. Управление тестовыми данными.** Данные должны быть предсказуемыми и изолированными между прогонами — либо через сиды (seed data), либо через генерацию уникальных данных на каждый запуск (test data builders), либо через очистку состояния (teardown) после теста.

**4. Выбор инструментов автоматизации.** Для UI e2e: Selenium WebDriver, Cypress, Playwright. Для API-уровня e2e (проверка цепочки сервисов без UI): REST-assured, Postman/Newman, готовые HTTP-клиенты в связке с оркестратором (JUnit/TestNG/pytest).

**5. Обеспечение стабильности (anti-flakiness).** E2E-тесты подвержены "флакам" из-за асинхронности, сетевых задержек, состояния окружения. Меры: явные ожидания (explicit waits) вместо sleep, retry-механизмы, изоляция тестов друг от друга, параллельный запуск с изолированными данными.

**6. Интеграция в CI/CD.** E2E-тесты обычно запускаются реже unit/integration-тестов (не на каждый коммит, а на стадии перед деплоем или ночным прогоном), так как они медленнее и дороже по ресурсам. Часто используется пирамида тестирования (test pyramid), где e2e — верхний, самый малочисленный слой.

**7. Мониторинг и отчётность.** Результаты публикуются в dashboard (Allure, ReportPortal), с логами, скриншотами и видео при падении теста для быстрой диагностики.

## Пример / когда применяется

Пример e2e-сценария для интернет-магазина: пользователь регистрируется → добавляет товар в корзину → применяет промокод → оформляет заказ → оплачивает картой (через sandbox платёжного провайдера) → получает email-подтверждение → заказ появляется в админ-панели со статусом "Оплачен".

```ts
// Пример e2e-теста на Playwright
test("успешное оформление заказа с промокодом", async ({ page }) => {
  await page.goto("/login");
  await page.fill("#email", "user@test.com");
  await page.fill("#password", "Password123");
  await page.click("#login-btn");

  await page.goto("/catalog/item-42");
  await page.click("#add-to-cart");
  await page.goto("/cart");
  await page.fill("#promo-code", "SALE10");
  await page.click("#apply-promo");
  await page.click("#checkout");

  await page.fill("#card-number", "4242424242424242");
  await page.click("#pay-btn");

  await expect(page.locator("#order-status")).toHaveText("Оплачен");
});
```

## На что смотрит интервьюер

Ожидается понимание, что e2e-тесты — дорогой и хрупкий, но необходимый верхний уровень пирамиды тестирования, и что их нужно ограничивать только критичными сценариями. Красный флаг — предложение покрыть e2e-тестами всё приложение. Интервьюер также проверяет знание проблемы флакующих тестов и способов борьбы с ней. Возможные follow-up: "как бы вы боролись с нестабильностью e2e-тестов в CI?" или "как e2e соотносится с integration и unit тестами (test pyramid)?".
