# Как посчитать Cyclomatic complexity?

**Коротко:** Цикломатическая сложность вычисляется по формуле `M = E - N + 2P` (рёбра минус узлы плюс удвоенное число компонент связности графа потока управления), либо проще — `M = количество точек принятия решений + 1`.

## Развёрнутый ответ

Cyclomatic Complexity (цикломатическая сложность, метрика Томаса Маккейба, 1976 год) — это количественная метрика структурной сложности программы, отражающая число независимых линейных путей выполнения через код. Она строится на графе потока управления (control flow graph, CFG): узлы графа — блоки кода, рёбра — переходы между ними.

Формальная формула:

`M = E - N + 2P`

где:
- `E` — количество рёбер (edges) в графе потока управления;
- `N` — количество узлов (nodes);
- `P` — количество компонент связности (обычно `P = 1` для одной функции/метода без недостижимого кода).

Практическая (эквивалентная) формула, которой пользуются на практике без построения графа:

`M = D + 1`

где `D` — количество точек принятия решений (decision points): `if`, `else if`, `case` в `switch`, `for`, `while`, `do-while`, тернарный оператор `?:`, логические операторы `&&` и `||` в условиях (каждый добавляет отдельный путь), `catch` в обработке исключений.

Интерпретация значений (по Маккейбу):
- 1–10 — простой код, низкий риск, легко тестируется;
- 11–20 — умеренно сложный, требует внимания;
- 21–50 — сложный, высокий риск дефектов, рекомендуется рефакторинг;
- более 50 — крайне рискованный, практически непроверяемый и неподдерживаемый код.

Важный практический смысл метрики для QA: цикломатическая сложность указывает **минимальное количество тест-кейсов**, необходимых для покрытия всех независимых путей выполнения (path coverage / basis path testing). Если `M = 5`, значит, нужно минимум 5 тестов, чтобы пройти по каждому независимому пути хотя бы один раз.

## Пример / когда применяется

Пример метода на псевдокоде:

```java
int classify(int age, boolean hasDiscount) {
    if (age < 18) {                 // decision point 1
        return 0;
    } else if (age >= 18 && age < 65) { // decision point 2 (if) + decision point 3 (&&)
        if (hasDiscount) {           // decision point 4
            return 1;
        }
        return 2;
    } else {
        return 3;
    }
}
```

Считаем точки принятия решений: `if (age < 18)`, `else if (...)`, `&&`, `if (hasDiscount)` — итого 4 точки.

`M = D + 1 = 4 + 1 = 5`

Значит, для полного покрытия независимых путей потребуется минимум 5 тест-кейсов (например: возраст 10; возраст 30 без скидки; возраст 30 со скидкой; возраст 70; и граничное значение 18 или 65 отдельно, в зависимости от разбиения путей).

На практике эту метрику считают статические анализаторы кода (SonarQube, PMD, ESLint-плагины типа `complexity`, CodeClimate) — вручную считать нужно уметь только для базового понимания на собеседовании.

## На что смотрит интервьюер

Интервьюер проверяет, знает ли кандидат хотя бы одну из формул наизусть и понимает ли связь метрики с тестовым покрытием (basis path testing) — это часто упускают, называя цикломатическую сложность просто "метрикой сложности кода без практического применения для тестировщика". Частая ошибка — забыть добавить `+1` в формуле или не посчитать логические операторы `&&`/`||` как отдельные точки решения. Хороший ответ также упоминает инструменты статического анализа и то, что высокая сложность коррелирует с большей вероятностью дефектов.
