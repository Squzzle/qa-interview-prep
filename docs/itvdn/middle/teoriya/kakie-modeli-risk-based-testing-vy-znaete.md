# Какие модели risk-based testing вы знаете?

**Коротко:** Основные подходы risk-based testing — оценка риска через матрицу "вероятность × влияние" (Probability × Impact), модель PRAM (Product Risk Analysis Model), FMEA (Failure Mode and Effects Analysis), а также общий фреймворк ISTQB, разделяющий риски на риски продукта и риски проекта.

## Развёрнутый ответ

Risk-based testing (RBT) — подход к тестированию, при котором приоритезация тестовых усилий (что тестировать в первую очередь, насколько глубоко, с какой периодичностью) строится на основе анализа рисков, а не равномерного покрытия всего функционала.

Базовая формула оценки риска, общая для большинства моделей:

`Уровень риска = Вероятность возникновения дефекта × Влияние (Impact/Severity) дефекта`

Основные модели и подходы:

1. **Матрица "Вероятность × Влияние" (Probability/Likelihood × Impact Matrix)** — самый распространённый и простой подход. Каждому функциональному модулю или требованию присваивается оценка вероятности отказа (насколько вероятно, что там есть баг — по сложности, новизне кода, истории дефектов) и оценка влияния (насколько критичны последствия для бизнеса/пользователя). Комбинация даёт уровень приоритета тестирования: High/High — тестируем в первую очередь и глубоко, Low/Low — можно тестировать поверхностно или пропустить.

2. **PRAM (Product Risk Analysis Model)** — формализованный ISTQB-подход, при котором риски продукта классифицируются по категориям (функциональные, производительность, безопасность, usability и т.д.), каждой присваивается вес, и на основе весов строится тестовое покрытие и распределение ресурсов.

3. **FMEA (Failure Mode and Effects Analysis)** — заимствован из инженерии/производства. Для каждого потенциального режима отказа (failure mode) оценивается три параметра: Severity (тяжесть последствий), Occurrence (вероятность возникновения), Detection (вероятность обнаружения дефекта до релиза). Перемножение даёт RPN (Risk Priority Number) `= Severity × Occurrence × Detection`. Чем выше RPN, тем выше приоритет тестирования этого сценария.

4. **ISTQB-классификация рисков на два уровня**:
   - **Product risks (риски продукта)** — риски, что продукт не будет соответствовать ожиданиям или содержит дефекты (сложность кода, новизна технологии, история дефектов в модуле, количество изменений).
   - **Project risks (риски проекта)** — организационные риски, влияющие на процесс тестирования (нехватка времени, ресурсов, квалификации команды, нестабильные требования, проблемы с окружением).

5. **Метод MoSCoW / приоритизация по бизнес-критичности** — часто применяется совместно с RBT для определения, какие требования "Must have" требуют максимального покрытия тестами, а "Could have" — минимального.

Практический процесс risk-based testing обычно включает: идентификацию рисков (мозговой штурм с командой, PM, аналитиками), оценку (probability × impact), приоритизацию, распределение тестовых усилий пропорционально риску, и периодическую переоценку рисков по ходу проекта.

## Пример / когда применяется

Пример применения матрицы Probability × Impact для e-commerce приложения:

| Модуль | Вероятность дефекта | Влияние | Приоритет тестирования |
|---|---|---|---|
| Оплата (checkout) | Средняя | Критическое | Высокий — глубокое тестирование, автоматизация регрессии |
| Поиск товаров | Высокая (частые изменения) | Среднее | Средний |
| Страница "О компании" | Низкая | Низкое | Низкий — беглая проверка |

Пример FMEA для той же оплаты: Severity=9 (потеря денег клиента), Occurrence=4 (редко, но случается при интеграции с новым платёжным провайдером), Detection=3 (трудно обнаружить без специального теста) → `RPN = 9×4×3 = 108` — высокий приоритет, требует дополнительных тестов граничных случаев и мониторинга.

Risk-based testing особенно применим при ограниченных сроках/ресурсах (когда невозможно покрыть 100% функционала) и в проектах с высокой ценой дефекта (финтех, медицина, авиация).

## На что смотрит интервьюер

Интервьюер хочет услышать конкретные названия моделей (а не общие слова "мы оцениваем риски"), понимание формулы вероятность × влияние, и в идеале — знание FMEA с RPN как более формализованного количественного подхода. Плюс — умение привести живой пример приоритизации. Частая ошибка кандидатов — путать risk-based testing с приоритизацией багов (severity/priority у дефектов) вместо приоритизации самого тестового покрытия. Также ожидается знание, что риски бывают product risks и project risks — многие забывают про вторую категорию.
