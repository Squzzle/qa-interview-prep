# Какие тест-кейсы можно сдать для тестирования баз данных?

**Коротко:** Тестирование БД охватывает проверку структуры схемы, целостности данных, корректности CRUD-операций, транзакционности, производительности запросов и безопасности хранимых данных.

## Развёрнутый ответ

Тест-кейсы для тестирования базы данных группируются по нескольким направлениям.

**1. Проверка схемы (schema/structural testing):**
- соответствие структуры таблиц, типов данных, длин полей проектной документации;
- наличие и корректность первичных/внешних ключей (PK/FK), уникальных ограничений (UNIQUE), NOT NULL;
- корректность индексов (наличие на часто запрашиваемых полях, отсутствие избыточных);
- корректность связей между таблицами (правильная кардинальность: 1:1, 1:N, N:M).

**2. Проверка целостности данных (data integrity):**
- ссылочная целостность — нельзя создать запись со ссылкой на несуществующий внешний ключ;
- каскадные операции (ON DELETE CASCADE / SET NULL / RESTRICT) работают согласно бизнес-логике;
- ограничения (constraints), например CHECK-ограничения на диапазон значений, соблюдаются;
- отсутствие дублирующихся записей там, где должна быть уникальность.

**3. Проверка CRUD-операций:**
- Create — вставка валидных и невалидных данных (граничные значения, спецсимволы, NULL там, где не должно быть);
- Read — корректность выборок, JOIN-ов, фильтров, сортировки, пагинации;
- Update — корректное обновление записи и связанных данных, отсутствие побочных эффектов на другие записи;
- Delete — мягкое/жёсткое удаление, корректная обработка связанных записей.

**4. Проверка транзакционности (ACID):**
- атомарность — при ошибке в середине транзакции все изменения откатываются (rollback);
- согласованность — данные остаются в валидном состоянии после транзакции;
- изолированность — параллельные транзакции не мешают друг другу (проверка уровня изоляции, dirty read/phantom read);
- устойчивость — данные сохраняются после commit даже при сбое системы сразу после него.

**5. Проверка производительности (performance testing БД):**
- время выполнения тяжёлых/часто используемых запросов (через EXPLAIN/EXPLAIN ANALYZE);
- поведение при больших объёмах данных (volume testing);
- эффективность использования индексов, отсутствие full table scan там, где не нужно.

**6. Проверка процедур миграции и хранимых процедур:**
- корректность миграционных скриптов (up/down миграции), отсутствие потери данных;
- корректность работы триггеров, хранимых процедур, представлений (views).

**7. Проверка безопасности:**
- защита от SQL-инъекций;
- корректность прав доступа (роли, права на чтение/запись для разных пользователей БД);
- шифрование чувствительных данных (пароли, персональные данные).

## Пример / когда применяется

Пример тест-кейсов для таблицы `orders` со связью к `users` (FK `user_id`):

| № | Тест-кейс | Ожидаемый результат |
|---|---|---|
| 1 | Вставить заказ с несуществующим `user_id` | Ошибка нарушения FK-ограничения |
| 2 | Удалить пользователя, у которого есть заказы (без ON DELETE CASCADE) | Ошибка, удаление запрещено |
| 3 | Вставить заказ с отрицательной суммой при CHECK `amount >= 0` | Ошибка нарушения CHECK constraint |
| 4 | Выполнить перевод денег между счетами в рамках одной транзакции при сбое на втором шаге | Все изменения откатываются (rollback) |
| 5 | Выполнить SELECT с JOIN `orders` и `users` по 1 млн записей | Запрос выполняется в пределах SLA (например, &lt; 200 мс) с использованием индекса |

```sql
-- Пример проверки, что запрос использует индекс, а не full scan
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42;
```

## На что смотрит интервьюер

Ожидается структурированный ответ, охватывающий не только "проверить, что данные сохранились", но и целостность, транзакционность, производительность и безопасность. Слабый ответ — только примеры на уровне "добавили запись, проверили что она есть". Хорошим сигналом является упоминание ACID и конкретных SQL-конструкций (EXPLAIN, CHECK, FK). Возможные follow-up вопросы: "как бы вы протестировали хранимую процедуру?" или "как проверить, что миграция БД не привела к потере данных?".
