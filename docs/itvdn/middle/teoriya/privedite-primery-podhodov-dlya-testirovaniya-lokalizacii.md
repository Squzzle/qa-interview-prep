# Приведите примеры подходов для тестирования локализации.

**Коротко:** Тестирование локализации проверяет корректность адаптации продукта под конкретный язык и регион (переводы, форматы, культурные особенности), а тестирование интернационализации (i18n) проверяет, что архитектура продукта в принципе способна поддерживать разные локали.

## Развёрнутый ответ

Важно различать два связанных, но разных вида тестирования:
- **Internationalization testing (i18n)** — проверка, что приложение архитектурно готово к работе с разными языками/регионами без изменения кода (используется вынесение строк в ресурсы, поддержка Unicode, гибкие layout).
- **Localization testing (l10n)** — проверка корректности конкретной локализации (перевода и адаптации) под конкретный язык/регион.

Основные подходы и области проверки локализации:

**1. Лингвистическая проверка (linguistic testing):**
- корректность и точность перевода (отсутствие машинного перевода "в лоб", сохранение смысла и тона);
- отсутствие непереведённых строк (hardcoded strings) и "ключей" вместо текста (например, `HOME_PAGE_TITLE` вместо реального текста);
- корректность склонений, множественного числа (pluralization rules отличаются между языками — в русском 3 формы, в английском 2, в некоторых языках больше);
- отсутствие обрезанного текста из-за разной длины строк на разных языках (например, немецкий текст обычно длиннее английского на 30%).

**2. Проверка форматов (locale-specific formatting):**
- форматы даты и времени (DD.MM.YYYY vs MM/DD/YYYY);
- форматы чисел и валют (разделители разрядов: 1,000.00 vs 1.000,00; символ и позиция валюты);
- форматы адресов, телефонных номеров, почтовых индексов;
- единицы измерения (метрическая/имперская система).

**3. Проверка UI/визуального отображения:**
- корректность отображения layout при разной длине текста (переполнение, обрезание, наложение элементов);
- поддержка RTL (right-to-left) языков — арабский, иврит: зеркальное отображение интерфейса, выравнивание, порядок элементов;
- корректность отображения шрифтов и спецсимволов (кириллица, иероглифы, диакритические знаки);
- корректность кодировки (UTF-8) — отсутствие "кракозябр" (mojibake).

**4. Функциональная проверка с учётом локали:**
- сортировка данных с учётом алфавита конкретного языка (например, сортировка с учётом ё, умляутов);
- поиск и фильтрация с учётом локали (регистронезависимость, диакритика);
- корректная работа с часовыми поясами при выборе региона.

**5. Культурная адаптация (cultural appropriateness):**
- корректность используемых иконок, цветов, изображений с учётом культурных особенностей региона;
- соответствие локальному законодательству (например, обязательные согласия/дисклеймеры для GDPR в ЕС).

**6. Псевдолокализация (pseudo-localization):** техника, при которой строки автоматически заменяются на псевдо-переведённые версии (с добавлением спецсимволов, увеличением длины) ещё до реального перевода — позволяет на раннем этапе найти hardcoded-строки и проблемы с layout без ожидания готового перевода.

## Пример / когда применяется

Пример тест-кейсов для локализации приложения на немецкий и арабский:

| № | Проверка | Ожидаемый результат |
|---|---|---|
| 1 | Открыть страницу оформления заказа на немецком | Все строки переведены, кнопка "Bestellen" не обрезана |
| 2 | Ввести дату рождения в форме на локали en-US | Формат MM/DD/YYYY принимается корректно |
| 3 | Переключить интерфейс на арабский | Layout зеркалируется (RTL), меню и кнопки справа налево |
| 4 | Отобразить цену на локали de-DE | Формат "1.234,56 €" |
| 5 | Запустить псевдолокализацию | Обнаружены 3 hardcoded-строки, не попавшие в файл переводов |

## На что смотрит интервьюер

Хороший ответ различает i18n и l10n, приводит конкретные примеры форматов (даты, числа, валюты) и упоминает RTL-языки и pluralization rules — это выдаёт реальный опыт, а не поверхностное знание. Слабый ответ ограничивается фразой "проверить, что текст переведён". Возможные follow-up: "как бы вы тестировали приложение с RTL-языком, если сами его не знаете?" или "что такое псевдолокализация и зачем она нужна?".
