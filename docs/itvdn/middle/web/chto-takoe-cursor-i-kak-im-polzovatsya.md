# Что такое cursor и как им пользоваться?

**Коротко:** Курсор — это объект базы данных, позволяющий построчно обходить результат запроса и обрабатывать каждую строку по отдельности, в отличие от обычной множественной (set-based) обработки в SQL.

## Развёрнутый ответ

По умолчанию SQL — язык множественных операций (set-based): запросы `SELECT`, `UPDATE`, `DELETE` обрабатывают сразу весь набор строк. Курсор — это механизм, позволяющий "отойти" от такого подхода и обрабатывать результат запроса построчно, императивно, как в обычном языке программирования (цикл с текущей позицией).

Жизненный цикл курсора состоит из нескольких обязательных шагов:

1. **DECLARE** — объявление курсора и связанного с ним запроса.
2. **OPEN** — выполнение запроса и формирование результирующего набора, на который курсор будет указывать.
3. **FETCH** — построчное извлечение данных из курсора в переменные, обычно в цикле, пока не будет достигнут конец набора (`@@FETCH_STATUS <> 0` в T-SQL).
4. **CLOSE** — освобождение результирующего набора (но объявление курсора ещё существует).
5. **DEALLOCATE** — полное удаление курсора и освобождение ресурсов.

Основной недостаток курсоров — низкая производительность на больших объёмах данных: построчная обработка на порядки медленнее set-based операций, курсор удерживает блокировки и ресурсы дольше, увеличивает нагрузку на журнал транзакций и хуже масштабируется. Поэтому в большинстве случаев курсоры считаются антипаттерном там, где задачу можно решить обычным `UPDATE`/`SELECT` с `JOIN`, оконными функциями или CTE.

Курсоры оправданы в узком классе задач:

- построчный вызов хранимой процедуры/внешней системы для каждой строки, где нельзя обойтись множественной операцией;
- сложная процедурная логика, которую невозможно выразить декларативным SQL (например, последовательные вычисления с зависимостью от предыдущей обработанной строки, не сводимые к оконным функциям);
- административные скрипты обслуживания (например, обход всех баз данных на сервере для выполнения одной и той же операции над каждой).

## Пример / когда применяется

Пример курсора в T-SQL, который обходит таблицу и выводит имена клиентов:

```sql
DECLARE @CustomerName NVARCHAR(100);

DECLARE customer_cursor CURSOR FOR
    SELECT name FROM customers WHERE is_active = 1;

OPEN customer_cursor;
FETCH NEXT FROM customer_cursor INTO @CustomerName;

WHILE @@FETCH_STATUS = 0
BEGIN
    PRINT 'Обработка клиента: ' + @CustomerName;
    -- построчная логика, например вызов хранимой процедуры
    FETCH NEXT FROM customer_cursor INTO @CustomerName;
END

CLOSE customer_cursor;
DEALLOCATE customer_cursor;
```

Эквивалентную задачу массового обновления той же таблицы обычно решают без курсора, множественной операцией:

```sql
UPDATE customers
SET status = 'reviewed'
WHERE is_active = 1;
```

## На что смотрит интервьюер

Интервьюер проверяет понимание того, что курсоры — это "процедурный" подход внутри декларативного SQL, и умение объяснить, почему их стоит избегать при возможности заменить set-based операцией (JOIN, оконные функции, CTE). Частая ошибка — забыть `DEALLOCATE`/`CLOSE`, что приводит к утечке ресурсов. Красный флаг — использование курсора там, где задача тривиально решается обычным `UPDATE`/`SELECT`, без понимания последствий для производительности.
