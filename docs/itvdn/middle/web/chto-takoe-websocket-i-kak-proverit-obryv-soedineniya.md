# Что такое WebSocket и как проверить обрыв соединения?

**Коротко:** WebSocket — протокол полнодуплексной связи поверх TCP, устанавливающий постоянное соединение между клиентом и сервером для обмена сообщениями в реальном времени; обрыв соединения проверяют через симуляцию разрыва сети, закрытие сокета вручную и отслеживание событий `close`/`error` и механизма переподключения (reconnect) с проверкой heartbeat/ping-pong.

## Развёрнутый ответ

WebSocket (RFC 6455) — сетевой протокол, который начинается с HTTP-запроса с заголовком `Upgrade: websocket`, после чего соединение "апгрейдится" с HTTP на постоянный TCP-канал, по которому обе стороны (клиент и сервер) могут в любой момент отправлять сообщения друг другу без необходимости повторного запроса (в отличие от классического request-response HTTP). Это делает WebSocket подходящим для чатов, live-уведомлений, онлайн-игр, торговых терминалов с котировками в реальном времени.

Жизненный цикл WebSocket-соединения на клиенте описывается событиями:
- `open` — соединение установлено.
- `message` — получено сообщение от сервера.
- `error` — произошла ошибка соединения.
- `close` — соединение закрыто (с кодом и причиной, например `1000` — нормальное закрытие, `1006` — аномальное закрытие без явного close-фрейма, часто означающее обрыв сети).

Обрыв соединения может произойти по разным причинам: сеть недоступна, сервер упал, таймаут неактивности, явное закрытие одной из сторон. Поскольку TCP не всегда сразу сигнализирует о разрыве (например, при внезапном отключении сети без корректного FIN-пакета), для надёжного обнаружения обрыва используется механизм **heartbeat** (ping/pong): сервер и клиент периодически обмениваются служебными ping/pong фреймами, и если pong не получен за отведённое время — соединение считается разорванным и закрывается принудительно.

Способы проверки обрыва соединения при тестировании:
1. **Отключение сети физически или через DevTools** (Network → Offline) — проверить, что клиент фиксирует `close`/`error` событие и пытается переподключиться.
2. **Симуляция обрыва на уровне прокси** — с помощью инструментов вроде Charles Proxy, Toxiproxy или `tc`/`iptables` можно эмулировать потерю пакетов, задержку, полный разрыв TCP-соединения.
3. **Принудительное закрытие сокета на сервере** — проверить, как клиент реагирует на серверный `close` с разными кодами закрытия.
4. **Проверка heartbeat-механизма** — убедиться, что при отсутствии ping/pong в течение таймаута соединение действительно закрывается, а не "зависает" в неопределённом состоянии.
5. **Проверка reconnect-логики** — после разрыва клиент должен попытаться переподключиться (обычно с экспоненциальной задержкой) и восстановить состояние (например, дозапросить пропущенные сообщения).

## Пример / когда применяется

Пример клиентского кода с обработкой обрыва и переподключением:

```js
function connect() {
  const ws = new WebSocket('wss://example.com/socket');

  ws.onopen = () => console.log('Соединение установлено');
  ws.onmessage = (event) => console.log('Получено:', event.data);

  ws.onclose = (event) => {
    console.warn(`Соединение закрыто, код: ${event.code}`);
    setTimeout(connect, 3000); // реконнект через 3 секунды
  };

  ws.onerror = (err) => console.error('Ошибка соединения', err);
}
connect();
```

Тест-кейс для ручной проверки: открыть DevTools → Network → включить "Offline" во время активной WebSocket-сессии → убедиться, что срабатывает `onclose` с кодом `1006`, а клиент инициирует переподключение после восстановления сети.

## На что смотрит интервьюер

Ожидается понимание, что WebSocket — не просто "быстрый HTTP", а отдельный протокол с апгрейдом соединения и полным дуплексом. Частая ошибка — не знать про heartbeat/ping-pong механизм и не уметь объяснить, почему TCP-соединение не всегда мгновенно сигнализирует об обрыве. Хороший ответ включает конкретные инструменты симуляции обрыва сети (DevTools Offline, Toxiproxy, отключение Wi-Fi) и разбор кодов закрытия (`1000`, `1006`, `1001`).
