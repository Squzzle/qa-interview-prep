# Как происходит авторизация на сервере?

**Коротко:** Сервер после аутентификации пользователя проверяет по его ролям/правам (хранящимся в сессии, токене или БД), разрешён ли доступ к запрошенному ресурсу или действию, и возвращает `200` при успехе либо `401`/`403` при отказе.

## Развёрнутый ответ

Процесс авторизации на сервере обычно состоит из следующих шагов:

1. Клиент отправляет запрос с идентификатором сессии (cookie) или токеном (например, `Authorization: Bearer <JWT>`).
2. Сервер (middleware/фильтр) извлекает из токена или из хранилища сессий информацию о пользователе: его ID, роли, права, скоупы.
3. Сервер сверяет эти данные с правилами доступа для запрошенного ресурса/эндпоинта. Правила могут быть заданы:
   - **RBAC** — доступ определяется ролью (`admin`, `manager`, `user`);
   - **ABAC** — доступ определяется набором атрибутов (время суток, IP, отдел, статус подписки);
   - **ACL** — явный список разрешений на конкретный объект (например, «пользователь X может редактировать документ Y»).
4. Если проверка пройдена — сервер выполняет запрошенное действие и возвращает `200 OK` (или соответствующий успешный код). Если проверка не пройдена — возвращает `403 Forbidden`. Если токен/сессия отсутствует или невалидна — `401 Unauthorized`.

Технически это часто реализовано через middleware (Express, ASP.NET Core), интерцепторы (Spring Security), guards (NestJS) или декораторы/аннотации (`@PreAuthorize`, `@RolesAllowed`). Проверка выполняется до того, как запрос попадёт в бизнес-логику контроллера, чтобы не тратить ресурсы на обработку заведомо запрещённого запроса.

Важный нюанс: авторизация должна проверяться не только на уровне маршрута («может ли пользователь вызывать этот эндпоинт»), но и на уровне конкретных данных («может ли пользователь работать именно с этим объектом») — иначе возникает уязвимость IDOR (Insecure Direct Object Reference), когда пользователь меняет ID в URL и получает доступ к чужим данным.

## Пример / когда применяется

Пример потока авторизации в REST API с JWT:

```bash
curl -X GET https://api.example.com/orders/42 \
  -H "Authorization: Bearer eyJhbGciOi..."
```

Псевдокод серверного middleware:

```ts
function authorize(requiredRole: string) {
  return (req, res, next) => {
    const token = req.headers.authorization?.split(' ')[1];
    const payload = verifyJwt(token); // выбросит ошибку -> 401
    if (!payload) return res.status(401).json({ error: 'Unauthorized' });

    const order = getOrderById(req.params.id);
    if (payload.role !== requiredRole && payload.userId !== order.ownerId) {
      return res.status(403).json({ error: 'Forbidden' });
    }
    next();
  };
}
```

## На что смотрит интервьюер

Ожидается понимание разницы между проверкой роли на уровне эндпоинта и проверкой владения конкретным ресурсом (IDOR). Частая ошибка — говорить только про роли и забывать про атрибутную/ресурсную авторизацию. Хороший ответ также упоминает, где хранится информация о правах (JWT claims, сессия на сервере, отдельный сервис авторизации типа OPA/Keycloak) и как тестировщик может проверить корректность авторизации (тесты с подменой ID, тесты с понижением роли, тесты с истёкшим/подделанным токеном).
