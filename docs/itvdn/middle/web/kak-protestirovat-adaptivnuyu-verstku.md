# Как протестировать адаптивную верстку?

**Коротко:** Адаптивную вёрстку тестируют на разных размерах экрана и разрешениях (desktop, tablet, mobile), используя эмуляцию устройств в DevTools, реальные устройства и проверку контрольных точек (breakpoints), ориентации, масштабирования и корректности отображения контента без потери функциональности.

## Развёрнутый ответ

Тестирование адаптивной (responsive) вёрстки включает несколько уровней проверки:

1. **Проверка breakpoints** — контрольных точек, на которых меняется layout (обычно это границы вроде 320px, 768px, 1024px, 1440px). На каждой контрольной точке нужно убедиться, что элементы не наезжают друг на друга, текст не обрезается, изображения масштабируются корректно.
2. **Эмуляция устройств** — использование Device Toolbar в Chrome DevTools или аналогичных инструментов для быстрой проверки на пресетах (iPhone, iPad, Android-устройства разных размеров) и в режиме произвольного разрешения (Responsive mode).
3. **Тестирование на реальных устройствах** — эмуляция не всегда точно повторяет поведение реального рендеринга (особенно для touch-событий, зума, viewport на iOS Safari), поэтому критичные сценарии стоит проверять на реальных телефонах/планшетах.
4. **Проверка ориентации** — портретная и альбомная ориентация должны одинаково корректно отображать контент.
5. **Проверка меню и навигации** — на мобильных версиях часто применяется другой паттерн навигации (гамбургер-меню), нужно убедиться, что он открывается/закрывается, не блокирует контент.
6. **Проверка изображений и медиа** — использование `srcset`/`picture` для отдачи разных размеров изображений, отсутствие горизонтального скролла из-за непропорциональных картинок.
7. **Проверка типографики** — читаемость текста на маленьких экранах (размер шрифта, межстрочный интервал), отсутствие обрезки длинных слов/URL.
8. **Кроссбраузерное тестирование** — разные браузеры (особенно мобильный Safari) по-разному интерпретируют `viewport meta tag`, флексы, грид, что может давать расхождения.
9. **Проверка производительности** — на мобильных сетях (3G/4G эмуляция в Network throttling) страница должна оставаться пригодной для использования.
10. **Автоматизация** — визуальное регрессионное тестирование (Percy, BackstopJS, Chromatic) для скриншот-сравнения вёрстки на разных viewport между релизами.

## Пример / когда применяется

Чек-лист ручного тестирования конкретной страницы:

| Разрешение | Устройство (пример) | Что проверять |
|---|---|---|
| 320–480px | iPhone SE | Нет горизонтального скролла, меню превращается в гамбургер |
| 768px | iPad (portrait) | Grid перестраивается в 2 колонки, кнопки кликабельны (44x44px) |
| 1024px | iPad (landscape) | Layout близок к desktop, но с адаптацией отступов |
| 1440px+ | Desktop | Максимальная ширина контента (`max-width`), контент не растягивается на весь экран |

Пример автоматизированной проверки через Playwright с разными viewport:

```ts
import { test, expect, devices } from '@playwright/test';

test('мобильное меню открывается на маленьком экране', async ({ browser }) => {
  const context = await browser.newContext({ ...devices['iPhone 12'] });
  const page = await context.newPage();
  await page.goto('https://example.com');
  await page.click('[data-testid="burger-menu"]');
  await expect(page.locator('[data-testid="mobile-nav"]')).toBeVisible();
});
```

## На что смотрит интервьюер

Интервьюер ожидает не только «открыть DevTools и посмотреть», но и знание конкретных контрольных точек, паттернов адаптивности (mobile-first, `srcset`, флекс/грид), проверку реальных устройств вместо только эмуляции, а также знание инструментов визуальной регрессии. Частая ошибка кандидатов — ограничиваться фразой «проверю на разных разрешениях», не называя конкретных инструментов и критериев (нет горизонтального скролла, читаемость, кликабельность элементов, touch-таргеты).
