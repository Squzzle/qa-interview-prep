# Как выполнить Debug страницы в браузере?

**Коротко:** Отладка страницы выполняется через встроенные DevTools браузера (Console, Elements, Network, Sources) — просмотр DOM, стилей, сетевых запросов, логов и пошаговое выполнение JS-кода с точками останова.

## Развёрнутый ответ

DevTools (Chrome DevTools, Firefox Developer Tools) — основной инструмент отладки веб-страниц. Ключевые вкладки и их назначение:

- **Elements/Inspector** — просмотр и live-редактирование DOM-дерева и CSS-стилей. Позволяет проверить, применились ли нужные стили, найти неверную вёрстку, посмотреть computed-стили и box model элемента.
- **Console** — вывод логов (`console.log`, `console.error`, `console.warn`), выполнение произвольного JS-кода в контексте страницы, просмотр непойманных исключений и стектрейсов.
- **Network** — просмотр всех сетевых запросов: статус-коды, тайминги, заголовки, тело запроса/ответа, размер payload. Позволяет проверить, действительно ли отправляется ожидаемый запрос и что приходит в ответе, включая WebSocket-фреймы.
- **Sources** — просмотр исходного JS/CSS кода, установка breakpoints (точек останова), пошаговое выполнение (`step over`, `step into`, `step out`), просмотр значений переменных в момент останова, conditional breakpoints (срабатывающие только при определённом условии).
- **Application/Storage** — просмотр и редактирование `localStorage`, `sessionStorage`, cookies, IndexedDB, Service Workers, кэша.
- **Performance** — профилирование производительности: запись таймлайна выполнения скриптов, рендеринга, layout/reflow, поиск узких мест.
- **Lighthouse/Audits** — автоматизированный аудит производительности, доступности, SEO.

Дополнительные техники отладки: точки останова прямо в коде (`debugger;`), условные breakpoints по клику в Sources, breakpoint на изменение DOM-узла (DOM breakpoints), breakpoint на XHR/fetch запрос, отслеживание событий (Event Listener Breakpoints).

Для мобильной отладки используется режим Device Toolbar (эмуляция устройств) или удалённая отладка (`chrome://inspect` для реального Android-устройства, Safari Web Inspector для iOS).

## Пример / когда применяется

Пример: страница не отправляет ожидаемый запрос при клике на кнопку «Купить».

1. Открыть DevTools (`F12` или `Cmd+Option+I`).
2. Перейти во вкладку Sources, найти обработчик клика, поставить breakpoint внутри функции.
3. Кликнуть на кнопку — выполнение остановится на breakpoint.
4. Пошагово (`F10`/`F11`) пройти код, проверяя значения переменных в панели Scope.
5. Параллельно проверить вкладку Network — отправился ли запрос, какой у него статус и тело ответа.
6. Если запрос не отправился — проверить Console на наличие исключений, которые прервали выполнение раньше.

```js
function onBuyClick() {
  debugger; // выполнение остановится здесь при открытых DevTools
  const payload = buildOrderPayload();
  fetch('/api/orders', { method: 'POST', body: JSON.stringify(payload) });
}
```

## На что смотрит интервьюер

Ожидается практическое знание конкретных вкладок DevTools, а не общая фраза «открыть консоль». Хороший кандидат упоминает breakpoints, Network-вкладку для анализа запросов/ответов и умеет объяснить разницу между `step over` и `step into`. Также интервьюер может спросить про отладку производительности (Performance tab) или про отладку мобильной вёрстки — здесь важно упомянуть эмуляцию устройств и удалённую отладку.
