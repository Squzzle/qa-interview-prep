# Какая разница между методами PUT и PATCH?

**Коротко:** PUT полностью заменяет ресурс данными из тела запроса и идемпотентен; PATCH применяет частичное изменение к существующему ресурсу и по умолчанию не гарантированно идемпотентен.

## Развёрнутый ответ

**PUT**:
- Семантика — "заменить ресурс целиком по указанному URI данными из тела запроса". Если ресурс не существует по данному URI, некоторые реализации API создают его (upsert-поведение), но строго по спецификации PUT просто заменяет представление ресурса.
- Идемпотентен: повторная отправка одного и того же PUT-запроса несколько раз приводит к тому же конечному состоянию ресурса, что и один раз.
- Тело запроса должно содержать полное представление ресурса — если клиент не передаст какое-то поле, оно, как правило, будет обнулено/удалено (в зависимости от реализации), потому что сервер интерпретирует тело как новое полное состояние объекта.

**PATCH**:
- Семантика — "применить частичное изменение к существующему ресурсу". Клиент передаёт только те поля, которые нужно изменить, остальные остаются нетронутыми.
- Не гарантированно идемпотентен по спецификации (RFC 5789): например, PATCH-операция вида "увеличить счётчик на 1" при повторном применении даст разный результат. Однако на практике многие PATCH-реализации (передача набора полей для замены) фактически идемпотентны.
- Формат тела запроса может быть либо просто набором полей для обновления (`{ "email": "new@example.com" }`), либо использовать стандартизированные форматы патчей, такие как JSON Patch (RFC 6902, набор операций `add`/`remove`/`replace`) или JSON Merge Patch (RFC 7396).

Ключевое практическое отличие: с PUT клиент обязан прислать весь объект целиком (даже неизменившиеся поля), тогда как с PATCH клиент присылает только дельту изменений — это экономит трафик и снижает риск случайно затереть поля, о которых клиент не знал.

## Пример / когда применяется

Дано: ресурс пользователя `{ "id": 1, "name": "Иван", "email": "ivan@example.com", "age": 30 }`.

PUT-запрос — обязан содержать все поля, иначе часть данных потеряется:

```
PUT /api/users/1 HTTP/1.1
Content-Type: application/json

{
  "name": "Иван",
  "email": "new-ivan@example.com",
  "age": 30
}
```

PATCH-запрос — передаём только изменяемое поле:

```
PATCH /api/users/1 HTTP/1.1
Content-Type: application/json

{
  "email": "new-ivan@example.com"
}
```

Если клиент по ошибке отправит PUT без поля `age`, результат может стать `{ "id": 1, "name": "Иван", "email": "new-ivan@example.com", "age": null }` — это частая причина багов, когда фронтенд использует PUT вместо PATCH и не досылает все поля формы.

Таблица сравнения:

| Критерий | PUT | PATCH |
|---|---|---|
| Действие | Полная замена ресурса | Частичное обновление |
| Идемпотентность | Да | Не гарантирована |
| Тело запроса | Полный объект | Только изменяемые поля |
| Риск потери данных | Высокий при неполном теле | Низкий |

## На что смотрит интервьюер

Ожидается точное понимание идемпотентности PUT против PATCH и практических последствий (потеря данных при неполном PUT-запросе). Хороший ответ упоминает стандарты JSON Patch/JSON Merge Patch для PATCH. Частая ошибка кандидатов — говорить, что "PUT для создания, PATCH для обновления", что неточно: оба метода применяются к обновлению существующего ресурса, разница именно в полноте замены и идемпотентности, а создание ресурса через PUT — лишь частный случай upsert-поведения в некоторых API.
