# Какие статус-коды ошибок бывают? Может ли сервер отправить код 400, если проблема на его стороне?

**Коротко:** HTTP-коды ошибок делятся на клиентские (`4xx`) и серверные (`5xx`); формально сервер не должен отправлять `400` при собственной проблеме — это семантически неверно, хотя на практике такое встречается из-за небрежной реализации.

## Развёрнутый ответ

Коды ошибок HTTP делятся на два класса:

**4xx — ошибки клиента** (проблема в запросе, который отправил клиент):
- `400 Bad Request` — некорректный синтаксис запроса или невалидные данные.
- `401 Unauthorized` — отсутствует или невалидна аутентификация.
- `403 Forbidden` — аутентификация есть, но прав недостаточно.
- `404 Not Found` — ресурс не найден.
- `405 Method Not Allowed` — метод не поддерживается для данного ресурса.
- `409 Conflict` — конфликт состояния ресурса (например, дублирование).
- `422 Unprocessable Entity` — синтаксически верный запрос, но семантически некорректные данные (ошибка валидации).
- `429 Too Many Requests` — превышен лимит запросов (rate limiting).

**5xx — ошибки сервера** (проблема на стороне сервера при обработке корректного запроса):
- `500 Internal Server Error` — общая необработанная ошибка на сервере.
- `502 Bad Gateway` — сервер, выступающий как прокси/шлюз, получил невалидный ответ от вышестоящего сервера.
- `503 Service Unavailable` — сервер временно не может обработать запрос (перегрузка, обслуживание).
- `504 Gateway Timeout` — вышестоящий сервер не ответил вовремя.

По спецификации (RFC 9110) семантика строгая: `4xx` означает, что запрос содержит ошибку и клиент должен исправить его перед повторной отправкой; `5xx` означает, что запрос был валиден, но сервер не смог его обработать по собственной вине. Поэтому если проблема действительно на стороне сервера (например, не удалось подключиться к БД, упал внутренний сервис, ошибка в коде обработчика), корректный ответ — один из `5xx`, а не `400`.

Однако на практике сервер может отправить `400`, даже если корневая причина — на его стороне: например, если сервис-бэкенд вернул некорректно сформированные данные, и API-шлюз, не разобравшись, интерпретирует их как «плохой запрос» и транслирует `400` клиенту. Это ошибка проектирования API, а не корректное поведение — грамотный сервис должен возвращать `5xx` в такой ситуации, а `400` использовать только когда виноват именно клиент (невалидный JSON, отсутствующее обязательное поле, неверный формат данных).

## Пример / когда применяется

Пример некорректного поведения API: клиент отправляет полностью валидный запрос на создание заказа, но внутри сервиса падает соединение с базой данных. Разработчик по ошибке оборачивает исключение в общий catch-блок и всегда возвращает `400`:

```ts
app.post('/orders', async (req, res) => {
  try {
    const order = await db.createOrder(req.body);
    res.status(201).json(order);
  } catch (e) {
    // Ошибка: всегда 400, даже если проблема в БД
    res.status(400).json({ error: e.message });
  }
});
```

Правильная реализация должна различать типы ошибок:

```ts
app.post('/orders', async (req, res) => {
  const validation = validateOrderSchema(req.body);
  if (!validation.ok) {
    return res.status(400).json({ error: validation.errors }); // ошибка клиента
  }
  try {
    const order = await db.createOrder(req.body);
    res.status(201).json(order);
  } catch (e) {
    return res.status(500).json({ error: 'Internal error' }); // ошибка сервера
  }
});
```

## На что смотрит интервьюер

Интервьюер проверяет знание конкретных кодов и их семантики, а не только деления на «4xx/5xx». Хороший ответ — умение объяснить, что `400` формально предназначен именно для клиентских ошибок, а его использование при внутренних сбоях сервера — антипаттерн, встречающийся из-за плохой обработки исключений. Как тестировщик, важно уметь провоцировать и проверять такие ситуации (падение зависимого сервиса, некорректная обработка исключений) и указывать разработчикам на несоответствие кода ответа реальной причине ошибки.
