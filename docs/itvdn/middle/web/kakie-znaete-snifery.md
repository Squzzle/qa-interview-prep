# Какие знаете сниферы?

**Коротко:** Сниффер — это инструмент перехвата и анализа сетевого трафика; наиболее известные — Wireshark, tcpdump, Fiddler, Charles Proxy и встроенные DevTools браузера.

## Развёрнутый ответ

Сниффер (packet sniffer, анализатор трафика) — программа или устройство, которое перехватывает пакеты данных, проходящие через сетевой интерфейс, и позволяет их разобрать по протоколам (Ethernet, IP, TCP/UDP, HTTP, TLS и т.д.). Работа сниффера основана на переводе сетевого адаптера в "неразборчивый" режим (promiscuous mode), в котором интерфейс принимает все пакеты в сегменте сети, а не только адресованные конкретному устройству. В коммутируемых сетях для перехвата чужого трафика дополнительно применяют ARP-spoofing, зеркалирование портов (port mirroring / SPAN) или man-in-the-middle прокси.

Основные инструменты, которые стоит знать QA-инженеру:

- **Wireshark** — самый популярный десктопный анализатор с графическим интерфейсом, поддерживает сотни протоколов, мощные фильтры отображения (`http.request.method == "POST"`), возможность собрать TCP-стрим целиком, экспорт объектов (файлов, изображений) из трафика.
- **tcpdump** — консольный сниффер для Unix-систем, часто используется на серверах без GUI; результат можно сохранить в `.pcap` и открыть в Wireshark для детального анализа.
- **Fiddler** — HTTP(S)-прокси-сниффер для Windows, ориентирован на веб-трафик, удобен для отладки REST API, подмены запросов/ответов (composer), троттлинга скорости соединения.
- **Charles Proxy** — кроссплатформенный аналог Fiddler, широко используется на macOS, поддерживает SSL-проксирование, breakpoints, throttling, запись мобильного трафика.
- **mitmproxy** — консольный/скриптуемый HTTPS-прокси на Python, удобен для автоматизации перехвата и модификации запросов.
- **Browser DevTools (вкладка Network)** — встроенный минимальный сниффер для анализа HTTP-запросов, тайминга загрузки, WebSocket-сообщений прямо в браузере.

Для перехвата HTTPS-трафика большинству инструментов требуется установка собственного корневого сертификата в систему/браузер, чтобы выполнять "легальный" MITM и расшифровывать TLS.

## Пример / когда применяется

Типичный кейс для QA: воспроизвести баг, при котором мобильное приложение отправляет некорректный запрос. Тестировщик поднимает Charles Proxy на ноутбуке, настраивает прокси на телефоне, устанавливает SSL-сертификат Charles на устройство и записывает сессию — в логе видно точный JSON тела запроса, заголовки авторизации и код ответа сервера.

Другой пример — использование `tcpdump` на сервере для диагностики сетевой проблемы:

```bash
tcpdump -i eth0 -w capture.pcap port 443
```

Файл `capture.pcap` затем открывается в Wireshark, где применяется фильтр `tls.handshake.type == 1` для анализа ClientHello.

## На что смотрит интервьюер

Интервьюер проверяет, понимает ли кандидат разницу между сниффером трафика (Wireshark/tcpdump на уровне пакетов) и HTTP-прокси-инструментами (Fiddler/Charles/mitmproxy), а также знает ли принцип работы с HTTPS (необходимость сертификата для расшифровки). Частая ошибка — путать сниффер с обычным логированием запросов внутри приложения. Красный флаг — незнание, зачем нужен promiscuous mode и почему в коммутируемой сети сниффер по умолчанию видит только свой трафик. Возможные follow-up: "как перехватить HTTPS-трафик?", "чем отличается SPAN-порт от ARP-spoofing?", "как найти утечку персональных данных в трафике?".
