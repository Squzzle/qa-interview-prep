# Какова разница между DROP и TRUNCATE?

**Коротко:** `DROP` удаляет объект (таблицу) целиком вместе со структурой, индексами, правами и метаданными; `TRUNCATE` только быстро очищает данные, сохраняя структуру таблицы.

## Развёрнутый ответ

`DROP TABLE` — это команда языка определения данных (DDL), которая полностью удаляет таблицу из базы данных: её структуру (столбцы, типы), данные, индексы, ограничения (constraints), триггеры и связанные с ней права доступа. После выполнения `DROP` таблица перестаёт существовать, и для повторной работы с данными потребуется заново создать таблицу командой `CREATE TABLE`.

`TRUNCATE TABLE` также относится к DDL, но удаляет только строки данных, оставляя структуру таблицы, индексы, ограничения и права доступа нетронутыми. `TRUNCATE` работает значительно быстрее, чем `DELETE FROM table` без условия, потому что:

- не генерирует построчные записи в журнале транзакций (в большинстве СУБД логируется только освобождение страниц/экстентов, а не удаление каждой строки);
- не активирует построчные (`FOR EACH ROW`) триггеры `DELETE`;
- физически освобождает выделенные таблице страницы данных, а не помечает строки как удалённые по одной.

Ключевые отличия:

| Критерий | DROP | TRUNCATE |
|---|---|---|
| Что удаляется | Таблица целиком (структура + данные) | Только данные |
| Откат (ROLLBACK) | В PostgreSQL/SQL Server — да, в транзакции; в MySQL с InnoDB — DDL авто-коммитится | Аналогично DROP по возможности отката |
| Триггеры DELETE | Не применимо | Не срабатывают (построчные) |
| Автоинкремент (identity) | Удаляется вместе с таблицей | Обычно сбрасывается в 0/1 |
| Скорость | Быстро (метаданные) | Быстро (без построчного логирования) |
| Нужны права | DROP на объект | DELETE/ALTER на таблицу (зависит от СУБД) |
| Внешние ключи | Нельзя дропнуть, если есть зависимые FK (без CASCADE) | Часто нельзя truncate таблицу, на которую ссылаются FK, без специальных опций |

Важно отличать оба варианта от `DELETE FROM table` — это команда DML, которая удаляет строки построчно, может использовать `WHERE`, поддерживает триггеры и полноценно логируется, но работает медленнее на больших объёмах.

## Пример / когда применяется

`DROP` применяют, когда таблица больше не нужна вообще — например, при удалении устаревшей временной таблицы после миграции:

```sql
DROP TABLE staging_import_2025;
```

`TRUNCATE` применяют, когда нужно быстро очистить таблицу перед повторной загрузкой данных (например, в ETL-процессе), сохранив её схему и права:

```sql
TRUNCATE TABLE staging_import_2025;
```

Если нужно удалить только часть строк по условию — используют `DELETE`:

```sql
DELETE FROM orders WHERE status = 'cancelled';
```

## На что смотрит интервьюер

Интервьюер ждёт чёткого разграничения трёх команд: `DROP` (структура+данные), `TRUNCATE` (только данные, быстро) и `DELETE` (построчно, с условием, медленнее, но откатываемо и с триггерами). Частая ошибка — утверждение, что `TRUNCATE` нельзя откатить ни в одной СУБД (в PostgreSQL и SQL Server это DDL внутри транзакции откатывается). Красный флаг — неспособность объяснить, почему `TRUNCATE` быстрее `DELETE`. Возможные follow-up: "что будет с autoincrement после TRUNCATE?", "можно ли truncate таблицу с внешними ключами?", "что произойдёт, если сделать DROP таблицы, на которую есть FK?".
