# Какова разница между методами GET и POST?

**Коротко:** GET запрашивает данные и передаёт параметры в URL, является идемпотентным и кешируемым; POST отправляет данные в теле запроса для создания/изменения ресурса, не идемпотентен по умолчанию и обычно не кешируется.

## Развёрнутый ответ

**GET**:
- Предназначен для получения данных без изменения состояния сервера.
- Параметры передаются в строке запроса (query string), например `?id=42&sort=asc`.
- Идемпотентен — повторный одинаковый запрос не меняет состояние сервера и возвращает тот же результат (если данные на сервере не изменились между запросами).
- Безопасен (safe) в терминах HTTP-семантики — не должен вызывать побочных эффектов.
- Кешируется браузером и промежуточными прокси по умолчанию.
- Ограничение на длину URL (обычно 2000-8000 символов в зависимости от браузера/сервера), что ограничивает объём передаваемых данных.
- Параметры видны в URL, истории браузера, логах сервера — не подходит для передачи чувствительных данных (паролей, токенов) без дополнительного шифрования уровня транспорта.
- Можно добавить в закладки, передать ссылкой.

**POST**:
- Предназначен для отправки данных на сервер, обычно с целью создания нового ресурса или выполнения действия с побочным эффектом.
- Данные передаются в теле запроса (body), не ограничены длиной URL и не видны в адресной строке.
- Не идемпотентен по умолчанию — повторная отправка одного и того же POST-запроса может создать несколько записей (например, повторный клик по кнопке "Оформить заказ" без защиты от даблклика создаст два заказа).
- Не кешируется браузером по умолчанию.
- Поддерживает разные типы содержимого тела: `application/json`, `application/x-www-form-urlencoded`, `multipart/form-data` (для загрузки файлов).
- Нельзя добавить в закладки как самостоятельное действие, нельзя повторить простым обновлением страницы без предупреждения браузера о повторной отправке данных.

## Пример / когда применяется

Пример GET-запроса для получения списка товаров:

```
GET /api/products?category=phones&sort=price_asc HTTP/1.1
Host: example.com
```

Пример POST-запроса для создания заказа:

```
POST /api/orders HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "productId": 42,
  "quantity": 2
}
```

Таблица сравнения:

| Критерий | GET | POST |
|---|---|---|
| Назначение | Получение данных | Создание/изменение данных |
| Расположение параметров | URL (query string) | Тело запроса |
| Идемпотентность | Да | Нет (по умолчанию) |
| Кешируется | Да | Нет |
| Ограничение длины | Есть (URL) | Практически нет |
| Видимость данных | В URL, истории, логах | Не в URL |
| Пригоден для чувствительных данных | Нет | Да (совместно с HTTPS) |

## На что смотрит интервьюер

Ожидается понимание не только "GET для чтения, POST для записи", но и терминов идемпотентности и безопасности (safe methods) в терминах HTTP-семантики, а также практических следствий (кеширование, ограничение длины URL, риск дублирования при повторной отправке POST). Частая ошибка — утверждать, что GET небезопасен для передачи паролей "потому что нешифрованный", путая шифрование транспорта (HTTPS шифрует и GET, и POST одинаково) с видимостью параметров в URL/логах/истории браузера — именно в этом реальная причина не передавать чувствительные данные через GET.
