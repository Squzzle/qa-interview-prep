# Каковы есть основные виды уязвимости веб-приложений?

**Коротко:** Основные уязвимости веб-приложений описаны в OWASP Top 10 и включают инъекции (SQL/NoSQL/OS), нарушение аутентификации/авторизации, XSS, CSRF, небезопасную десериализацию, использование компонентов с известными уязвимостями и неправильную конфигурацию безопасности.

## Развёрнутый ответ

Наиболее авторитетный источник классификации уязвимостей веб-приложений — OWASP Top 10. Основные категории:

- **Injection (инъекции)** — SQL Injection, NoSQL Injection, Command Injection: злоумышленник передаёт вредоносный код через входные данные, которые интерпретируются как часть запроса/команды из-за отсутствия параметризации и экранирования.
- **Broken Authentication** — слабые пароли, отсутствие блокировки после множества неудачных попыток входа, небезопасное хранение сессионных токенов, отсутствие MFA.
- **Broken Access Control (нарушение авторизации)** — пользователь может получить доступ к чужим данным или функциям, к которым не должен иметь доступ (например, IDOR — Insecure Direct Object Reference, когда меняя ID в URL можно получить доступ к чужому ресурсу).
- **Cross-Site Scripting (XSS)** — внедрение вредоносного JavaScript-кода в страницу, который выполняется в браузере жертвы. Виды: Stored XSS (сохранённый на сервере), Reflected XSS (отражённый через параметр запроса), DOM-based XSS (эксплуатация через клиентский JS без обращения к серверу).
- **Cross-Site Request Forgery (CSRF)** — заставляет аутентифицированного пользователя невольно выполнить нежелательное действие на другом сайте (например, перевод денег) через поддельный запрос, использующий его активную сессию.
- **Security Misconfiguration** — небезопасные настройки по умолчанию, открытые административные панели, избыточные права, подробные сообщения об ошибках, раскрывающие внутреннюю структуру системы.
- **Sensitive Data Exposure / Cryptographic Failures** — передача и хранение конфиденциальных данных (пароли, номера карт) без должного шифрования (HTTPS, хеширование с солью).
- **XML External Entities (XXE)** — эксплуатация парсеров XML, позволяющая читать произвольные файлы на сервере или выполнять SSRF через внешние сущности.
- **Insecure Deserialization** — десериализация непроверенных данных, позволяющая выполнить произвольный код или подменить объект.
- **Using Components with Known Vulnerabilities** — использование устаревших библиотек/фреймворков с публично известными CVE.
- **Insufficient Logging & Monitoring** — отсутствие логирования подозрительной активности не позволяет вовремя обнаружить и отреагировать на атаку.
- **Server-Side Request Forgery (SSRF)** — сервер по указанию злоумышленника выполняет запрос к внутреннему ресурсу, недоступному напрямую извне.

## Пример / когда применяется

Пример SQL Injection в уязвимом коде:

```sql
-- Уязвимый запрос, построенный конкатенацией строк
SELECT * FROM users WHERE username = 'admin' AND password = '' OR '1'='1';
```

Если приложение подставляет пользовательский ввод напрямую в SQL-строку, ввод `' OR '1'='1` в поле пароля обходит проверку аутентификации.

Пример Reflected XSS: URL вида `https://example.com/search?q=<script>alert(document.cookie)</script>` — если сервер возвращает значение `q` без экранирования в HTML-ответе, скрипт выполнится в браузере жертвы.

Тестировщик проверяет такие уязвимости через инструменты вроде OWASP ZAP, Burp Suite, а также вручную составленными негативными тест-кейсами (специальные символы, `' OR 1=1--`, `<script>` теги, длинные payload для проверки переполнения буфера).

## На что смотрит интервьюер

Ожидается знание конкретных категорий OWASP Top 10, а не общая фраза "уязвимости бывают разные". Хороший кандидат приводит конкретные примеры (SQLi, XSS, CSRF, IDOR) и объясняет механизм эксплуатации хотя бы одной-двух из них, а также способы защиты (параметризованные запросы, экранирование вывода, CSRF-токены, Content Security Policy). Частая ошибка — путать XSS и CSRF или не знать разницы между Stored и Reflected XSS.
