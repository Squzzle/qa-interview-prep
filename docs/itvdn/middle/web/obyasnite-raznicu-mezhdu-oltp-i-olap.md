# Объясните разницу между OLTP и OLAP.

**Коротко:** OLTP (Online Transaction Processing) — системы для быстрой обработки множества коротких транзакций (текущая операционная деятельность), а OLAP (Online Analytical Processing) — системы для сложного многомерного анализа больших объёмов исторических данных.

## Развёрнутый ответ

**OLTP** ориентирован на обработку транзакций реального времени: создание заказа, оплата, регистрация пользователя, обновление остатков на складе. Характеристики OLTP-систем:

- Множество коротких, простых операций (`INSERT`/`UPDATE`/`DELETE`/точечный `SELECT`), выполняемых конкурентно многими пользователями одновременно.
- Схема данных обычно нормализована (3NF), чтобы избежать избыточности и обеспечить целостность данных при записи.
- Требования: низкая задержка отклика (миллисекунды), высокая пропускная способность транзакций, строгие свойства ACID.
- Примеры: банковские системы, системы бронирования, интернет-магазины, CRM.

**OLAP** ориентирован на анализ и построение отчётности на основе больших объёмов агрегированных данных, часто накопленных за длительный период:

- Небольшое число сложных, "тяжёлых" запросов на чтение (сложные `JOIN`, `GROUP BY`, оконные функции, агрегации) с малым количеством одновременных пользователей (аналитики, BI-инструменты).
- Схема данных денормализована и оптимизирована под чтение — типичные схемы "звезда" (star schema) или "снежинка" (snowflake schema) с таблицами фактов (fact tables) и измерений (dimension tables).
- Данные обычно загружаются пакетно (batch ETL) из OLTP-систем в хранилище данных (data warehouse), а не изменяются напрямую пользователями.
- Примеры: Google BigQuery, ClickHouse, Snowflake, Amazon Redshift, аналитические кубы SSAS.

Сравнительная таблица:

| Критерий | OLTP | OLAP |
|---|---|---|
| Назначение | Операционные транзакции | Аналитика, отчётность |
| Тип запросов | Простые, короткие | Сложные, агрегирующие |
| Объём данных за запрос | Единицы строк | Миллионы строк |
| Нормализация схемы | Высокая (3NF) | Низкая (star/snowflake) |
| Частота изменений | Постоянно (реальное время) | Пакетно (ETL, по расписанию) |
| Пользователи | Множество, конкурентно | Немного аналитиков/BI-систем |
| Приоритет | Целостность и скорость записи | Скорость чтения и агрегации |

## Пример / когда применяется

OLTP-пример: при оформлении заказа в интернет-магазине выполняется транзакция, вставляющая строку в `orders`, уменьшающая остаток в `inventory` и создающая запись в `payments` — всё в рамках одной ACID-транзакции с минимальной задержкой.

OLAP-пример: в конце дня ETL-процесс выгружает данные из OLTP-базы заказов в хранилище данных, где аналитик выполняет запрос вида:

```sql
SELECT region, DATE_TRUNC('month', order_date) AS month, SUM(amount) AS total_revenue
FROM fact_orders fo
JOIN dim_region dr ON fo.region_id = dr.id
GROUP BY region, month
ORDER BY month;
```

Такой запрос затрагивает миллионы строк и был бы слишком тяжёлым для боевой OLTP-базы, поэтому его выполняют в отдельном OLAP-хранилище.

## На что смотрит интервьюер

Интервьюер проверяет понимание архитектурного разделения: зачем аналитические нагрузки выносят из операционной базы в отдельное хранилище (снижение нагрузки на продакшн, разная оптимизация схемы под чтение/запись). Частая ошибка — путать OLAP-кубы (многомерный анализ) с обычным аналитическим SQL-запросом к той же OLTP-базе. Красный флаг — незнание схем "звезда"/"снежинка" и понятий fact/dimension таблиц. Follow-up: "почему нельзя просто гонять аналитические запросы на продакшн OLTP-базе?", "что такое ETL и зачем он нужен?", "приведите пример схемы 'звезда'".
