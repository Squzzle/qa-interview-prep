# Вспомните разные типы репликации в SQL Server?

**Коротко:** SQL Server поддерживает четыре основных типа репликации: снапшот (Snapshot), транзакционную (Transactional), слиянием (Merge) и одноранговую (Peer-to-Peer), каждая — под свой сценарий консистентности и направления синхронизации.

## Развёрнутый ответ

Репликация в SQL Server — механизм копирования и синхронизации данных между базами данных (издатель — Publisher, дистрибьютор — Distributor, подписчик — Subscriber), обычно для распределения нагрузки на чтение, обеспечения отказоустойчивости или интеграции географически распределённых систем.

- **Снапшот-репликация (Snapshot Replication)** — самая простая: периодически делается полный снимок (снапшот) данных публикации и целиком копируется подписчикам. Не отслеживает промежуточные изменения, подходит для редко изменяемых данных или небольших объёмов, где приемлема задержка между обновлениями. Наименее ресурсоёмкая в плане постоянного отслеживания изменений, но при больших объёмах данных дорогая по трафику при каждой синхронизации.

- **Транзакционная репликация (Transactional Replication)** — отслеживает изменения в журнале транзакций издателя и передаёт их подписчикам почти в реальном времени (обычно с задержкой в секунды). Изначальная синхронизация выполняется снапшотом, а далее применяются только инкрементальные изменения (INSERT/UPDATE/DELETE) в том порядке, в котором они произошли. Подписчики обычно доступны только на чтение. Наиболее распространённый тип для сценариев "один издатель — много читающих подписчиков" (например, репортинг-серверы).

- **Репликация слиянием (Merge Replication)** — позволяет и издателю, и подписчикам вносить изменения независимо (в том числе в офлайне), а затем синхронизировать и объединять их, разрешая конфликты по заданным правилам (например, "приоритет издателя" или кастомная логика разрешения конфликтов). Используется в сценариях с мобильными/удалёнными клиентами, которые периодически подключаются к сети (например, торговые представители с ноутбуками).

- **Одноранговая репликация (Peer-to-Peer Replication)** — разновидность транзакционной репликации, где все узлы равноправны (peers) и могут одновременно принимать чтение и запись, а изменения распространяются между всеми узлами. Применяется для горизонтального масштабирования записи и повышения доступности между несколькими дата-центрами, но требует аккуратного проектирования во избежание конфликтов (обычно избегают одновременной записи одних и тех же строк на разных узлах).

Дополнительно стоит упомянуть **Bidirectional Replication** (двунаправленную транзакционную репликацию между двумя серверами) как отдельно настраиваемый вариант, хотя формально она строится поверх транзакционной модели.

## Пример / когда применяется

Транзакционная репликация типично используется, когда основная OLTP-база (издатель) обслуживает запись, а несколько read-only реплик используются для построения отчётов, чтобы не нагружать боевую базу тяжёлыми SELECT-запросами:

```sql
-- Упрощённо: настройка публикации через T-SQL
EXEC sp_replicationdboption @dbname = 'SalesDB', @optname = 'publish', @value = 'true';
EXEC sp_addpublication @publication = 'SalesPublication', @status = 'active';
EXEC sp_addarticle @publication = 'SalesPublication', @article = 'Orders', @source_object = 'Orders';
```

Merge-репликация уместна, когда полевые агенты вносят заказы в локальную копию базы на ноутбуке без связи с сервером, а вечером синхронизируются с центральной базой, и конфликты (например, изменение одного и того же заказа с двух сторон) разрешаются автоматически по правилам приоритета.

## На что смотрит интервьюер

Интервьюер проверяет умение подобрать правильный тип репликации под сценарий: read-only отчётность → транзакционная; офлайн-клиенты с двусторонними изменениями → merge; масштабирование записи между ЦОД → peer-to-peer; редко меняющиеся справочники → snapshot. Частая ошибка — путать репликацию с резервным копированием (backup) или с Always On Availability Groups, которые решают задачу отказоустойчивости иначе (на уровне всей базы, а не отдельных таблиц/публикаций). Красный флаг — незнание, что merge-репликация — единственный тип, разрешающий изменения на подписчике с последующим разрешением конфликтов.
