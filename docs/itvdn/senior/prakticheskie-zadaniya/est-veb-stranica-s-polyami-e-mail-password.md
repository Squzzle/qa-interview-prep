# Есть веб-страница с полями e-mail, password и кнопкой submit... Как проверить, что данные отправлены в базу данных?

**Коротко:** проверка идёт на трёх уровнях — через UI/API (косвенно, по видимому результату), через прямой доступ к базе данных (напрямую, SQL-запросом), и через логи/мониторинг сетевого трафика, при этом обязательно проверяется корректность и безопасность хранения данных (хеширование пароля).

## Развёрнутый ответ

Самый прямой и надёжный способ — **прямая проверка базы данных**. После заполнения формы и нажатия submit нужно подключиться к БД (через клиента типа DBeaver, psql, mysql CLI или административную панель) и выполнить SQL-запрос к соответствующей таблице, например `SELECT * FROM users WHERE email = 'test@example.com';`, и убедиться, что запись появилась, поля заполнены корректно, а пароль хранится в захешированном виде (bcrypt/argon2/scrypt), а не в открытом тексте — это одновременно и функциональная, и security-проверка.

**Проверка через API/сетевой трафик.** Через DevTools (вкладка Network) или прокси (Postman, Fiddler, Charles) посмотреть, какой именно запрос отправляется на backend при сабмите формы, какой код ответа возвращается (`200`/`201` при успехе), и что содержится в теле ответа (например, ID созданного пользователя). Если API возвращает ID, можно затем обратиться к другому эндпоинту (`GET /users/{id}`) и убедиться, что данные сохранились и доступны для чтения — это косвенное, но валидное подтверждение записи в БД без прямого доступа к ней.

**Проверка через UI (косвенная).** Если после регистрации приложение делает логин или показывает профиль пользователя — можно проверить, что данные "прижились", попытавшись авторизоваться с этим email/паролем или увидев данные в личном кабинете. Это самый слабый по надёжности способ (может маскировать баги кэширования), но иногда единственный доступный, если нет доступа к БД/API у тестировщика.

**Проверка через логи приложения.** Логи backend-сервиса могут содержать записи о выполненном SQL INSERT/UPDATE, что дополнительно подтверждает факт записи и помогает диагностировать, если запись не удалась (например, ошибка constraint violation, timeout соединения с БД).

**Дополнительные аспекты, которые стоит проверить:**
- Транзакционность: если регистрация состоит из нескольких операций (создание пользователя + создание профиля + отправка email), важно убедиться, что либо всё выполнилось, либо ничего (атомарность), а не осталась "полусозданная" запись.
- Уникальность: повторная попытка регистрации с тем же email не создаёт вторую запись.
- Корректность типов и форматов данных в БД (например, email хранится в нижнем регистре, если так задумано бизнес-логикой).
- Отсутствие чувствительных данных в открытом виде — пароль обязательно должен быть хеширован, а не храниться plain text.

## Пример / когда применяется

Пример последовательности действий тестировщика:

```sql
-- После заполнения формы регистрации с email test@example.com
SELECT id, email, password_hash, created_at
FROM users
WHERE email = 'test@example.com';
```

Ожидаемый результат: одна строка, `password_hash` — это хеш (например, начинается с `$2b$` для bcrypt), а не исходный пароль в открытом виде; `created_at` соответствует времени регистрации.

Альтернатива без прямого доступа к БД — через API:

```bash
# 1. Отправка формы регистрации (эмулируем через API)
curl -X POST https://app.example.com/api/register \
  -H "Content-Type: application/json" \
  -d '{"email": "test@example.com", "password": "StrongPass123"}'
# Ответ: {"id": 42, "email": "test@example.com"}

# 2. Проверка, что запись реально существует и доступна
curl https://app.example.com/api/users/42
# Ответ должен содержать email test@example.com
```

## На что смотрит интервьюер

Интервьюер ждёт, что кандидат не ограничится ответом "зайду в базу и посмотрю", а расскажет про несколько уровней проверки (БД, API, UI, логи) и, что важно, поднимет тему безопасности хранения пароля (хеширование). Красный флаг — кандидат не упоминает, что пароль в БД не должен храниться в открытом виде, или не знает, как обратиться к базе данных напрямую. Follow-up вопросы: "а что, если у вас нет доступа к базе данных?", "как проверить, что пароль захеширован правильным алгоритмом?", "что делать, если запись в БД появилась, но email не пришёл пользователю?"
