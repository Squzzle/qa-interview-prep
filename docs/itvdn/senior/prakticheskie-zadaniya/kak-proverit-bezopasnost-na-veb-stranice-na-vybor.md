# Как проверить безопасность на веб-странице (на выбор)?

**Коротко:** базовая проверка безопасности веб-страницы включает тестирование на инъекции (SQL/NoSQL), XSS, некорректную аутентификацию/авторизацию, утечку чувствительных данных, отсутствие HTTPS/security-заголовков и CSRF — с использованием как ручных проверок, так и специализированных инструментов (Burp Suite, OWASP ZAP).

## Развёрнутый ответ

Security-тестирование веб-страницы стоит структурировать вокруг известной методологии, например OWASP Top 10 — списка самых распространённых веб-уязвимостей. QA-инженер (не обязательно security-специалист) должен уметь проверить хотя бы базовый набор рисков.

**Инъекции (Injection).** Проверка полей ввода на SQL-инъекции (`' OR '1'='1' --`), NoSQL-инъекции, command injection. Цель — убедиться, что ввод экранируется/параметризуется на бэкенде и не приводит к выполнению произвольного кода или обходу аутентификации.

**Межсайтовый скриптинг (XSS).** Ввод скриптовых пейлоадов в текстовые поля (`<script>alert(1)</script>`, `<img src=x onerror=alert(1)>`) и проверка, что они не выполняются при отображении на странице (экранирование HTML на выходе). Проверяется как отражённый (reflected), так и хранимый (stored) XSS — например, если комментарий с XSS-пейлоадом сохраняется и отображается другим пользователям.

**Аутентификация и управление сессией.** Проверка, что пароли хранятся захешированными, что сессионные токены/cookie имеют флаги `HttpOnly`, `Secure`, `SameSite`, что после логаута сессия реально инвалидируется на сервере (а не только удаляется токен на клиенте), что нет возможности перебора пароля без ограничения количества попыток (brute-force protection, rate limiting), что нет утечки информации через сообщения об ошибках ("пользователь не найден" vs "неверный пароль" — оба должны быть одинаковыми, чтобы не раскрывать существование аккаунта).

**Авторизация / контроль доступа (Broken Access Control).** Проверка IDOR (Insecure Direct Object Reference) — попытка получить доступ к чужим данным, изменив ID в URL/запросе (`/api/orders/123` → `/api/orders/124`), проверка, что обычный пользователь не может выполнить действия администратора через прямой вызов API в обход UI.

**CSRF (Cross-Site Request Forgery).** Проверка наличия CSRF-токенов на формах, изменяющих состояние (POST/PUT/DELETE), проверка, что запрос без корректного токена отклоняется.

**Передача и хранение чувствительных данных.** Проверка использования HTTPS (а не HTTP), отсутствие чувствительных данных в URL (query params с паролями/токенами логируются серверами и прокси), проверка security-заголовков ответа сервера: `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Strict-Transport-Security`.

**Конфигурация и раскрытие информации.** Проверка отсутствия отладочной информации в продакшене (stack trace ошибок, версии фреймворков в заголовках ответа), проверка, что директории/файлы конфигурации недоступны напрямую (`/.env`, `/.git/`, `/admin` без авторизации).

Инструменты для практической проверки: браузерные DevTools для базовых проверок, Burp Suite или OWASP ZAP для перехвата и модификации трафика, автоматического сканирования на известные уязвимости, `sqlmap` для проверки SQL-инъекций, ручное тестирование security-заголовков через `curl -I`.

## Пример / когда применяется

Пример практической проверки на XSS в поле ввода:

```text
Ввести в текстовое поле (например, "имя пользователя"):
<script>alert(document.cookie)</script>

Ожидаемый результат: скрипт НЕ выполняется, значение отображается как текст
(экранированное <script> в виде &lt;script&gt;)
```

Пример проверки security-заголовков через `curl`:

```bash
curl -I https://example.com

# Ожидаем увидеть в ответе:
# Strict-Transport-Security: max-age=31536000; includeSubDomains
# X-Content-Type-Options: nosniff
# X-Frame-Options: DENY
# Content-Security-Policy: default-src 'self'
```

Пример проверки IDOR:

```bash
# Авторизован как пользователь с id=123, пробуем получить чужой заказ
curl -H "Authorization: Bearer <token_user_123>" \
  https://api.example.com/orders/124
# Ожидается 403 Forbidden, а не данные чужого заказа
```

## На что смотрит интервьюер

Интервьюер ждёт хотя бы 2-3 конкретные категории уязвимостей с примерами проверки (не абстрактное "проверю на уязвимости"), знание OWASP Top 10 как ориентира и понимание, что security-тестирование — это не только про хакерские инъекции, но и про правильную конфигурацию (заголовки, HTTPS, флаги cookie). Красный флаг — кандидат называет только XSS/SQL-инъекции и не упоминает авторизацию/IDOR/security-заголовки. Плюс к ответу — упоминание инструментов (Burp Suite, OWASP ZAP, sqlmap).
