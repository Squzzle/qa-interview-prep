# Какое минимальное количество тест-кейсов необходимо, чтобы убедиться в корректной работе этой веб-страницы?

**Коротко:** "минимальное" не значит "одно число" — правильный ответ определяется через категории проверок (позитив, граничные значения, негатив, безопасность, UX/доступность), и для формы из 2-3 полей разумный минимум составляет порядка 10-15 тест-кейсов, покрывающих ключевые классы эквивалентности и граничные условия.

## Развёрнутый ответ

Отвечать конкретным числом без контекста ("нужно 5 тест-кейсов") — плохая практика: правильный подход — показать методологию оценки достаточности покрытия через техники тест-дизайна: классы эквивалентности (equivalence partitioning), анализ граничных значений (boundary value analysis), таблицы принятия решений (decision table) для комбинаций полей.

Для веб-страницы с полями email, password и кнопкой submit минимальный, но осмысленный набор тест-кейсов должен покрывать:

**Позитивные сценарии (2-3 кейса).** Успешная регистрация с валидными email и паролем, успешная регистрация с email на границе допустимой длины, успешная регистрация с паролем на границе минимальной/максимальной длины.

**Граничные значения email (2-3 кейса).** Минимально короткий валидный email (`a@b.co`), email на границе максимальной длины поля, email с максимально длинным доменом/локальной частью в пределах спецификации.

**Граничные значения password (2-3 кейса).** Пароль ровно на минимальной длине (проходит), пароль на 1 символ короче минимума (не проходит), пароль на максимальной допустимой длине.

**Негативные сценарии по email (2-3 кейса).** Пустое поле email, некорректный формат (`test@`, `test.com`), email уже зарегистрирован в системе.

**Негативные сценарии по password (1-2 кейса).** Пустое поле, пароль без учёта требований политики безопасности (например, без цифр, если это требуется).

**Комбинаторные/UX сценарии (1-2 кейса).** Оба поля пустые (submit неактивен или показывает обе ошибки сразу), фокус/табуляция между полями работает корректно, кнопка submit неактивна до заполнения обязательных полей (если так по дизайну).

**Технические/безопасность (1-2 кейса).** Проверка, что пароль маскируется на экране (type="password"), проверка защиты от SQL-инъекции/XSS в полях.

Итого разумный минимум — около 12-15 тест-кейсов для покрытия основных рисков без избыточности. Важно подчеркнуть: "минимальное количество" зависит от критичности функционала (для формы логина/регистрации, через которую проходят все пользователи, минимум должен быть выше, чем для второстепенной формы), а также от того, есть ли автоматизация (при наличии автотестов "минимум для ручной проверки" может быть меньше — только smoke-набор).

## Пример / когда применяется

Пример компактной таблицы тест-кейсов (сокращённый вариант для собеседования):

| № | Кейс | Ожидаемый результат |
|---|---|---|
| 1 | Валидные email и пароль | Успешная регистрация |
| 2 | Пустой email | Ошибка валидации, submit заблокирован |
| 3 | Пустой password | Ошибка валидации |
| 4 | Некорректный формат email | Ошибка "неверный формат email" |
| 5 | Пароль короче минимума на 1 символ | Ошибка валидации |
| 6 | Пароль на границе минимума | Успех |
| 7 | Email уже зарегистрирован | Ошибка "email занят" |
| 8 | Email на границе макс. длины | Успех |
| 9 | Пароль маскируется на экране | `type="password"` подтверждён |
| 10 | XSS-пейлоад в email | Экранирование/отклонение, скрипт не выполняется |
| 11 | Двойной клик по submit | Одна запись, не дублируется |
| 12 | Табуляция между полями | Корректный порядок фокуса |

## На что смотрит интервьюер

Интервьюер проверяет, использует ли кандидат техники тест-дизайна (классы эквивалентности, граничные значения) вместо интуитивного перечисления кейсов "из головы", и понимает ли разницу между "минимально достаточным" и "исчерпывающим" покрытием. Красный флаг — конкретное число без обоснования ("нужно 3 теста") или, наоборот, попытка перечислить бесконечное количество кейсов без приоритизации. Хороший ответ структурирует кейсы по категориям и явно связывает "минимальность" с риск-ориентированным подходом. Follow-up: "а как изменится минимум, если это форма для медицинского приложения с высокими требованиями к безопасности?"
