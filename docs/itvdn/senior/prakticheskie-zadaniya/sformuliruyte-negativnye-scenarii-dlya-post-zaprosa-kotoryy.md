# Сформулируйте негативные сценарии для POST-запроса, который создаёт нового пользователя.

**Коротко:** негативные сценарии проверяют, что API корректно отклоняет некорректные, неполные, дублирующиеся или потенциально опасные данные, возвращая правильный код ошибки и понятное сообщение, а не создаёт "битую" запись и не падает с 500-й ошибкой.

## Развёрнутый ответ

Негативное тестирование POST-запроса на создание пользователя строится вокруг нескольких групп проверок.

**Валидация обязательных полей.** Отправка запроса без обязательного поля (например, без `email` или `password`), с пустой строкой вместо значения, с `null` вместо значения, с полем неверного типа данных (число вместо строки, массив вместо строки). Ожидаемый результат — код `400 Bad Request` с телом ответа, описывающим, какое поле некорректно.

**Валидация формата и границ значений.** Некорректный формат email (`test@`, `test.com`, `test@@mail.com`), слишком короткий или слишком длинный пароль (короче минимальной длины политики безопасности, длиннее максимальной допустимой), превышение максимальной длины строковых полей (имя длиной 10000 символов), отрицательные числа там, где ожидается положительное значение (например, возраст = -5), недопустимые символы в полях, которые должны содержать только буквы/цифры.

**Дубликаты и конфликты данных.** Попытка создать пользователя с email, который уже зарегистрирован — ожидается `409 Conflict` (а не `500` и не молчаливое создание дубликата). Регистрация с одинаковым email в разном регистре (`Test@mail.com` vs `test@mail.com`), если по бизнес-логике email должен быть уникален без учёта регистра.

**Безопасность и инъекции.** SQL-инъекции и NoSQL-инъекции в текстовых полях (`' OR '1'='1`), XSS-пейлоады в имени пользователя (`<script>alert(1)</script>`), передача лишних полей, которых нет в схеме (например, `"role": "admin"` — попытка mass assignment, чтобы получить повышенные права), передача JSON с некорректным синтаксисом (битый JSON) — ожидается `400`, а не `500`.

**Технические и протокольные аномалии.** Отсутствие заголовка `Content-Type: application/json`, отправка данных в неверном формате (form-data вместо JSON), пустое тело запроса, превышение допустимого размера тела запроса, повторная отправка одного и того же запроса (проверка идемпотентности/защиты от двойного клика — не должно создаваться два пользователя).

**Авторизация и доступ.** Если эндпоинт защищён — запрос без токена авторизации, с истёкшим токеном, с токеном недостаточных прав. Ожидается `401`/`403`, а не создание пользователя.

Для каждого сценария важно проверять не только код ответа, но и то, что состояние системы не изменилось: пользователь не появился в базе данных, не отправилось письмо подтверждения, не создался связанный объект (профиль, корзина и т.д.).

## Пример / когда применяется

Пример набора негативных тест-кейсов в табличном виде:

| № | Входные данные | Ожидаемый результат |
|---|---|---|
| 1 | `email` отсутствует | 400, сообщение "email is required" |
| 2 | `email = "not-an-email"` | 400, сообщение о неверном формате |
| 3 | `password = "123"` (короче минимума) | 400, сообщение о требованиях к паролю |
| 4 | `email` уже существует в БД | 409 Conflict |
| 5 | Лишнее поле `"role": "admin"` | 400 либо поле игнорируется без повышения прав |
| 6 | Тело запроса — невалидный JSON | 400, а не 500 |
| 7 | `name = "<script>alert(1)</script>"` | 400 или экранирование, без выполнения скрипта |
| 8 | Повторная отправка идентичного запроса подряд | второй пользователь не создаётся (или явная идемпотентность) |

Пример на `curl` для проверки отсутствия обязательного поля:

```bash
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"password": "StrongPass123"}'
# Ожидается: 400 Bad Request, тело содержит указание на отсутствие email
```

## На что смотрит интервьюер

Интервьюер ждёт, что кандидат не ограничится одним-двумя очевидными кейсами ("пустое поле"), а покажет системное мышление: валидация, безопасность, идемпотентность, коды ответов, состояние БД после запроса. Красный флаг — если кандидат проверяет только код ответа и не говорит о содержимом сообщения об ошибке и о том, что данные реально не попали в систему. Частый follow-up: "а как проверить, что после невалидного запроса ничего не изменилось в базе?" и "чем отличается 400 от 422 от 409?".
