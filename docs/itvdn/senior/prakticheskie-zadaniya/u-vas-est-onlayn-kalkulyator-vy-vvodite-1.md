# У вас есть онлайн-калькулятор. Вы вводите 1+1 и получаете 3. Расскажите, как вы будете искать причину проблемы.

**Коротко:** нужно системно локализовать проблему по слоям приложения — от воспроизведения и изоляции окружения (клиент/сеть/сервер) до анализа логов, сетевых запросов и кода, двигаясь от "черного ящика" к "белому ящику".

## Развёрнутый ответ

Первый шаг — **воспроизведение и изоляция**. Нужно убедиться, что баг стабильно воспроизводится: повторить действие несколько раз, проверить в разных браузерах, в режиме инкогнито (чтобы исключить кэш и расширения браузера), на разных устройствах и с разным вводом (`2+2`, `0+1`, отрицательные числа, дробные числа) — это поможет понять, специфична ли проблема именно для `1+1` или это системная ошибка вычислений.

Второй шаг — **анализ на стороне клиента (frontend)**. Открыть DevTools браузера: вкладку Console — нет ли JS-ошибок; вкладку Network — что именно отправляется на сервер (если вычисление идёт через API) и что приходит в ответе; проверить, не изменяются ли значения на пути от инпута к вычислению из-за багов в JS (например, конкатенация строк вместо сложения чисел: `"1" + "1"` даёт `"11"`, а не `2` — классическая ошибка приведения типов, хотя в вопросе результат "3", что скорее указывает на логическую, а не строковую ошибку). Проверить, не влияет ли verstka/дебаунс/повторный вызов обработчика события (например, обработчик клика сработал дважды и прибавил лишнюю единицу).

Третий шаг — **если вычисление идёт на backend** — проверить сетевой запрос напрямую через Postman/curl с теми же параметрами, чтобы исключить фронтенд из уравнения. Если API возвращает "3" — проблема на сервере; если API возвращает "2", а фронт показывает "3" — проблема в клиентском коде (например, к результату с сервера прибавляется какое-то смещение или счётчик).

Четвёртый шаг — **анализ логики вычислений**. Если проблема на сервере — смотреть исходный код функции сложения, юнит-тесты (если есть, почему они не поймали баг), логи приложения на предмет промежуточных значений. Частые причины подобных багов: округление с ошибкой, добавление какой-то константы "по ошибке" (например, дебаг-код `result + 1` забыли убрать), баг в парсинге ввода (если инпут "1" интерпретируется как "1" + скрытый пробел/символ, который парсится как ещё одна единица), проблема кэширования (старый закэшированный результат объединяется с новым).

Пятый шаг — **проверка окружения и данных**. Не идёт ли двойная отправка формы (submit сработал дважды), нет ли проблемы с состоянием (state) в SPA-приложении, где предыдущий результат не сбрасывается и складывается с новым.

Весь процесс — это классический "бинарный поиск" причины: последовательно исключать компоненты системы (клиент → сеть → сервер → бизнес-логика → данные), пока не будет локализован конкретный участок кода, где происходит ошибка.

## Пример / когда применяется

Практический чек-лист действий по порядку:

1. Повторить `1+1` несколько раз, попробовать другие числа.
2. Открыть DevTools → Console — проверить JS-ошибки.
3. Открыть DevTools → Network — посмотреть, что отправляется/приходит.
4. Если API — вызвать эндпоинт напрямую через `curl`/Postman в обход UI.
5. Сравнить результат UI и результат "чистого" API-вызова.
6. Если баг на сервере — смотреть код, логи, юнит-тесты функции сложения.
7. Проверить, не срабатывает ли обработчик клика/сабмита дважды.
8. Зафиксировать шаги воспроизведения и создать баг-репорт с точным окружением.

```bash
curl -X POST https://calculator.example.com/api/sum \
  -H "Content-Type: application/json" \
  -d '{"a": 1, "b": 1}'
# Если ответ {"result": 3} — баг на сервере
# Если ответ {"result": 2} — баг на клиенте (в UI отображается неверно)
```

## На что смотрит интервьюер

Интервьюер проверяет умение мыслить системно и декомпозировать проблему, а не сразу лезть в код "наугад". Красный флаг — ответ "открою код и посмотрю", без предварительной изоляции проблемы через воспроизведение и разделение фронт/бэк. Хороший ответ показывает знание инструментов (DevTools, Postman/curl), понимание типовых причин (приведение типов, двойной обработчик, кэш, забытый дебаг-код) и системный подход "от простого к сложному". Частый follow-up: "а если баг воспроизводится только у одного пользователя?" — ожидается ответ про локаль, кэш браузера, cookies, версию приложения на клиенте.
