# Что такое Definition of Done?

**Коротко:** Definition of Done (DoD) — это согласованный командой набор критериев, которым должна соответствовать любая задача/user story, чтобы считаться полностью завершённой, включая требования к тестированию, документации и качеству кода.

## Развёрнутый ответ

Definition of Done — это чек-лист, применяемый ко всем элементам бэклога (user story, задачам), который определяет общее понимание "готовности" в команде. DoD снижает субъективность оценки завершённости работы и предотвращает накопление технического долга и скрытого недоделанного функционала.

Типичные пункты DoD:

- Код написан и соответствует стандартам кодирования (code style, code review пройден).
- Unit-тесты написаны и проходят (с заданным порогом покрытия).
- Функциональность протестирована QA (функциональные, регрессионные тесты).
- Автотесты для новой функциональности написаны и добавлены в CI-пайплайн.
- Все критичные и блокирующие баги исправлены.
- Документация обновлена (API docs, release notes, пользовательская документация при необходимости).
- Требования по производительности и безопасности проверены (если применимо).
- Функциональность продемонстрирована и принята product owner'ом.
- Изменения смёржены в основную ветку и задеплоены в тестовое/staging окружение.

Важно отличать DoD от **Acceptance Criteria (AC)**: AC специфичны для конкретной user story и описывают, что именно должно работать; DoD — это общие, универсальные критерии качества, применимые ко всем историям в команде/проекте.

DoD может быть многоуровневым: DoD для user story, DoD для спринта, DoD для релиза — каждый уровень добавляет свои критерии (например, для релиза добавляются регрессионное тестирование всего продукта, нагрузочное тестирование, security review).

QA играет ключевую роль в формировании и защите DoD: именно тестировщик чаще всего настаивает на включении пунктов про тестовое покрытие и качество, и именно QA проверяет соответствие DoD перед закрытием задачи.

## Пример / когда применяется

Пример DoD для user story в Scrum-команде:

```
- [ ] Код написан и прошёл code review (минимум 1 approve)
- [ ] Unit-тесты покрывают новую логику (coverage >= 80%)
- [ ] Функциональные тест-кейсы выполнены, критичных багов нет
- [ ] Автотесты на happy path добавлены в regression suite
- [ ] Story продемонстрирована на demo и принята PO
- [ ] Задеплоено на staging без ошибок
```

Если хотя бы один пункт не выполнен, задача не может быть перенесена в статус "Done" — она остаётся в работе или возвращается в бэклог.

## На что смотрит интервьюер

- Кандидат должен чётко отличать DoD от Acceptance Criteria — частая путаница на собеседованиях.
- Красный флаг: ответ вида "DoD — это когда баги закрыты" — слишком узко, не отражает многоаспектность.
- Интервьюер оценивает понимание роли QA в формировании DoD, а не только исполнении.
- Follow-up вопросы: "Кто утверждает DoD в команде?", "Что делать, если разработчик считает задачу done, а DoD не выполнен полностью?", "Как DoD меняется от спринта к релизу?".
