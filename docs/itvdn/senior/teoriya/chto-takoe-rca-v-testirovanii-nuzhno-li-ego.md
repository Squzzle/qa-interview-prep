# Что такое RCA в тестировании? Нужно ли его проводить?

**Коротко:** RCA (Root Cause Analysis, анализ первопричин) — это процесс выявления фундаментальной причины возникновения дефекта или инцидента, а не только его симптомов; проводить его нужно обязательно, особенно для критичных и повторяющихся проблем, поскольку без RCA команда лечит следствия, а не причины.

## Развёрнутый ответ

RCA — это систематический подход к поиску первопричины проблемы с целью её устранения и предотвращения повторения в будущем, а не просто исправления конкретного бага. В контексте QA/тестирования RCA применяется как к найденным дефектам, так и к процессным сбоям (например, "почему баг дошёл до продакшена, если у нас есть тестирование").

Основные методики RCA:

1. **"5 почему" (5 Whys).** Последовательное задавание вопроса "почему это произошло?" пять раз (или до тех пор, пока не найдена корневая причина), углубляясь от симптома к первопричине.

2. **Диаграмма Исикавы (Fishbone/Cause-and-Effect Diagram).** Визуализация возможных причин проблемы по категориям: люди, процесс, технологии, окружение и т.д.

3. **Fault Tree Analysis.** Логическое дерево событий, приводящих к отказу системы, полезно для сложных технических инцидентов.

4. **Pareto-анализ.** Определение, какие 20% причин вызывают 80% проблем — помогает приоритизировать усилия.

Зачем RCA нужен в тестировании:

- **Предотвращение повторения дефектов.** Если баг вызван непониманием требований, а не просто ошибкой в коде, исправление одного бага не решит проблему — нужно менять процесс уточнения требований.
- **Улучшение процесса тестирования.** Если критичный баг "проскочил" в продакшен, RCA покажет пробел в тест-покрытии, окружении, или в самом процессе релиза.
- **Классификация дефектов по причинам** (defect causal analysis) — например, ошибки требований, ошибки дизайна, ошибки кодирования, ошибки конфигурации — помогает видеть системные проблемы.
- **Основа для метрик качества процесса**, таких как Defect Removal Efficiency, и для continuous improvement в Agile-ретроспективах.

RCA не всегда нужен для каждого мелкого бага — это было бы неэффективно с точки зрения затрат времени. RCA целесообразен для:

- критичных production-инцидентов;
- повторяющихся (регрессирующих) багов;
- багов, вызвавших значительные финансовые/репутационные потери;
- систематических проблем процесса (например, регулярные пропуски багов одного типа).

## Пример / когда применяется

Пример применения "5 почему" после production-инцидента (баг с двойным списанием денег):

```
Проблема: У пользователей произошло двойное списание при оплате.

1. Почему? — Запрос на оплату отправился дважды.
2. Почему? — Кнопка "Оплатить" не была заблокирована после первого клика.
3. Почему? — Frontend не реализовал debounce/disable на кнопке.
4. Почему? — Это не было указано в Acceptance Criteria user story.
5. Почему? — В команде нет стандартного чек-листа UI-паттернов
   для критичных операций (double-submit protection).

Корневая причина: отсутствие стандартизированного
чек-листа/гайдлайна для критичных UI-действий.

Корректирующее действие: добавить пункт про
защиту от double-submit в общий Definition of Done
и UI-гайдлайны команды.
```

## На что смотрит интервьюер

- Кандидат должен показать разницу между "исправить симптом" и "устранить причину".
- Ожидается знание конкретных техник (5 Whys, Fishbone), а не только термина.
- Важно ответить на вторую часть вопроса — "нужно ли" — с аргументацией, когда RCA оправдан, а когда избыточен.
- Красный флаг: "RCA нужен для каждого бага" — показывает непонимание баланса затрат и пользы.
- Follow-up: "Приведите пример RCA из вашего опыта", "Как RCA связан с ретроспективой спринта?".
