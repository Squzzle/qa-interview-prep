# Что такое сокеты и как их тестировать, вручную и автоматизировано? Зачем их используют?

**Коротко:** Сокет (socket) — программный интерфейс для сетевого обмена данными между процессами по протоколам `TCP`/`UDP`; используются там, где нужна постоянная двусторонняя связь в реальном времени (чаты, торговые платформы, игры, стриминг), а тестируются как вручную (через специализированные клиенты типа `wscat`/`Postman`), так и автоматизированно (через клиентские библиотеки на языке тестов, отправляющие сообщения и проверяющие ответы).

## Развёрнутый ответ

**Сокет** — это конечная точка сетевого соединения, программная абстракция, которая позволяет двум процессам (на одной машине или на разных) обмениваться данными по сети через определённый протокол транспортного уровня. Разновидности:

- **TCP-сокеты** — надёжное, гарантированное доставкой, упорядоченное соединение (используется, например, в HTTP, WebSocket).
- **UDP-сокеты** — быстрые, но без гарантии доставки и порядка (используются в стриминге видео, VoIP, некоторых играх).
- **WebSocket** — протокол поверх TCP, обеспечивающий полнодуплексную (двустороннюю) связь между клиентом и сервером через одно долгоживущее соединение — в отличие от классического HTTP (запрос-ответ), сервер может сам инициативно отправлять данные клиенту.

**Зачем используются:** там, где нужен обмен данными в реальном времени с низкой задержкой и без накладных расходов на повторное установление соединения для каждого сообщения: чаты и мессенджеры, торговые/биржевые платформы (обновление котировок), онлайн-игры, live-трансляции, системы мониторинга и телеметрии, push-уведомления.

**Как тестировать вручную:**
1. Использовать специализированные клиенты: `wscat` (командная строка для WebSocket), Postman (поддерживает WebSocket-запросы), браузерные расширения (Simple WebSocket Client), либо вкладку Network → WS в DevTools браузера.
2. Проверить установление соединения (handshake): корректные ли заголовки `Upgrade: websocket`, `Connection: Upgrade`, статус ответа `101 Switching Protocols`.
3. Отправить тестовые сообщения и проверить корректность ответа/broadcast другим подключённым клиентам.
4. Проверить обработку разрыва соединения (сервер должен корректно освобождать ресурсы, уведомлять других участников в чате об отключении пользователя).
5. Проверить поведение при некорректных данных (невалидный JSON, слишком большое сообщение, неожиданный тип фрейма).
6. Проверить безопасность: работает ли аутентификация при установлении соединения, есть ли ограничение по частоте сообщений (rate limiting), защищено ли соединение (`wss://` вместо `ws://`).
7. Проверить поведение при множественных одновременных соединениях (несколько вкладок/клиентов).

**Как тестировать автоматизированно:**
- Использовать клиентские библиотеки на языке тестового фреймворка, например Java (`Java-WebSocket`, `Tyrus`), Python (`websockets`, `websocket-client`), JS (`ws`, встроенный `WebSocket` API).
- Написать тест, который: открывает соединение → отправляет сообщение → ассертит полученный ответ (по содержимому, по времени, по количеству событий) → закрывает соединение.
- Для нагрузочного тестирования сокетов — специализированные инструменты: Gatling (поддерживает WebSocket-сценарии), Artillery, k6 (модуль `ws`).
- Для интеграционных тестов с брокерами сообщений в реальном времени также применяют тестирование на уровне протокола (например, для соединений через message queue).

## Пример / когда применяется

Пример автоматизированного теста WebSocket на Java с библиотекой `Java-WebSocket`:

```java
class ChatWebSocketTest {

    @Test
    void shouldReceiveBroadcastMessage() throws Exception {
        CountDownLatch latch = new CountDownLatch(1);
        AtomicReference<String> received = new AtomicReference<>();

        WebSocketClient client = new WebSocketClient(new URI("ws://localhost:8080/chat")) {
            @Override
            public void onMessage(String message) {
                received.set(message);
                latch.countDown();
            }
            @Override public void onOpen(ServerHandshake handshake) {}
            @Override public void onClose(int code, String reason, boolean remote) {}
            @Override public void onError(Exception ex) {}
        };

        client.connectBlocking();
        client.send("{\"type\":\"join\",\"user\":\"tester\"}");

        boolean gotResponse = latch.await(5, TimeUnit.SECONDS);
        assertTrue(gotResponse, "Сообщение не было получено за 5 секунд");
        assertTrue(received.get().contains("tester присоединился к чату"));

        client.closeBlocking();
    }
}
```

Пример ручной проверки через `wscat`:

```bash
wscat -c ws://localhost:8080/chat
# После подключения:
> {"type":"message","text":"Привет"}
< {"type":"message","from":"user1","text":"Привет"}
```

## На что смотрит интервьюер

- Понимает ли кандидат разницу между HTTP (запрос-ответ) и WebSocket (постоянное двустороннее соединение), а также разницу TCP/UDP.
- Знает ли конкретные инструменты для ручного тестирования сокетов (не только "открою DevTools").
- Понимает ли, что для автоматизации нужен клиент, умеющий работать асинхронно (ожидание сообщений с таймаутом, а не мгновенный ассерт).
- Упоминает ли проверки не только happy path, но и обработку разрывов соединения, некорректных данных, безопасности (`wss://`, аутентификация).
- Красный флаг — путаница между сокетами и обычными REST API-запросами, непонимание, зачем вообще нужно постоянное соединение вместо повторных HTTP-запросов (polling).
