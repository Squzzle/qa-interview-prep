# Для чего нужен протокол RTP?

**Коротко:** RTP (Real-time Transport Protocol) используется для доставки аудио- и видеоданных в реальном времени по IP-сетям (VoIP-звонки, видеоконференции, стриминг), обеспечивая нумерацию пакетов, временные метки и информацию для восстановления порядка и синхронизации, но не гарантируя доставку.

## Развёрнутый ответ

RTP описан в RFC 3550 и работает поверх транспортного протокола, обычно UDP (реже TCP), поскольку для медиатрафика в реальном времени критична низкая задержка, а не 100%-я надёжность доставки — потерянный кадр видео проще пропустить, чем ждать его повторной передачи, которая приведёт к задержке и "заиканию".

RTP решает несколько важных задач, которых не хватает "голому" UDP:

- **Порядковые номера (sequence number)** — позволяют получателю обнаружить потерю пакетов и восстановить правильный порядок, если пакеты пришли не по порядку (что типично для IP-сетей).
- **Временные метки (timestamp)** — используются для синхронизации воспроизведения (джиттер-буфер) и для синхронизации нескольких потоков (например, аудио и видео одного звонка).
- **Идентификатор источника (SSRC)** — позволяет различать несколько потоков в рамках одной сессии (например, несколько участников конференции).
- **Поле типа полезной нагрузки (payload type)** — указывает, какой кодек используется (G.711, G.729, H.264, VP8 и т.д.), что позволяет декодеру правильно интерпретировать данные.

RTP всегда работает в связке с RTCP (RTP Control Protocol) — управляющим протоколом, который передаёт статистику качества передачи (потери пакетов, джиттер, round-trip time) и используется для мониторинга и адаптации качества связи.

Сам RTP не устанавливает и не управляет сессией — за это отвечают протоколы сигнализации, такие как SIP или H.323, которые договариваются о параметрах медиапотока (кодеки, порты) через SDP, после чего непосредственно медиаданные передаются по RTP.

Для тестировщика знание RTP важно при тестировании VoIP-систем, колл-центров, видеоконференций: нужно уметь проверять качество передачи голоса (MOS-оценка), потерю пакетов, джиттер, задержку, а также анализировать RTP-трафик через Wireshark для диагностики проблем со звуком.

## Пример / когда применяется

Типичный сценарий: тестирование IP-телефонии. SIP устанавливает соединение между двумя абонентами и согласовывает через SDP параметры (например, кодек G.711, UDP-порт 16000-16010 для медиапотока). После установления сессии голос передаётся пакетами RTP по согласованным портам.

Пример заголовка RTP-пакета (упрощённо, поля из RFC 3550):

```
Version: 2
Padding: 0
Extension: 0
CSRC count: 0
Marker: 0
Payload Type: 0 (PCMU / G.711 μ-law)
Sequence Number: 34521
Timestamp: 160000
SSRC: 0x1a2b3c4d
```

Анализ RTP-трафика в Wireshark:

```bash
# фильтр в Wireshark
rtp

# анализ качества звонка
Telephony -> RTP -> Stream Analysis
```

## На что смотрит интервьюер

- Понимает ли кандидат, почему RTP использует UDP, а не TCP (компромисс между задержкой и надёжностью).
- Знает ли разницу между RTP (передача данных) и RTCP (контроль качества) — частый вопрос на уточнение.
- Может ли объяснить связь RTP с сигнальными протоколами (SIP/SDP) — RTP сам по себе не устанавливает сессию.
- Красный флаг: если кандидат считает, что RTP гарантирует доставку пакетов, как TCP.
