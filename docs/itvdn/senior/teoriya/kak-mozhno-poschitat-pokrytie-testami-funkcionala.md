# Как можно посчитать покрытие тестами функционала?

**Коротко:** покрытие функционала тестами считается как отношение количества требований/сценариев, покрытых хотя бы одним тест-кейсом, к общему числу требований/сценариев, а покрытие кода — как отношение выполненных строк/веток/условий кода к их общему числу при прогоне тестов; оба вида метрик дополняют друг друга.

## Развёрнутый ответ

Существует два принципиально разных уровня покрытия, которые часто путают: покрытие требований (functional/requirements coverage) и покрытие кода (code coverage). Оба важны, но отвечают на разные вопросы.

Покрытие требований/функционала:
- Формула: `Requirements Coverage = (Число требований, покрытых тест-кейсами / Общее число требований) * 100%`.
- Строится через матрицу трассируемости (Requirements Traceability Matrix, RTM), где каждому требованию или пользовательскому сценарию сопоставляются один или несколько тест-кейсов.
- Показывает, насколько полно тестовые сценарии закрывают функциональные и нефункциональные требования (включая edge-кейсы, негативные сценарии, граничные значения).
- Считается вручную или через инструменты управления тестами (TestRail, Xray, Zephyr, qTest), которые автоматически строят отчёт трассируемости.

Покрытие кода (code coverage):
- Line coverage — доля выполненных строк кода при прогоне тестов.
- Branch coverage — доля выполненных веток условий (if/else, switch).
- Statement/Condition coverage — более детальные метрики, учитывающие все комбинации логических условий.
- Считается автоматически инструментами (JaCoCo для Java, Istanbul/nyc для JS, Coverage.py для Python) в связке с unit- и интеграционными тестами.
- Важный нюанс: 100% code coverage не означает отсутствие багов — покрытие показывает, что код выполнился, а не что результат проверен корректно (можно выполнить строку кода без единой осмысленной проверки/ассерта).

Дополнительные виды покрытия:
- Risk coverage — покрытие тестами зон с наивысшим риском (даже если формально не 100% требований покрыто, критичные риски должны быть закрыты).
- Coverage по браузерам/устройствам/окружениям — актуально для кросс-браузерного/кросс-платформенного тестирования.

Практический подход к расчёту покрытия функционала без специализированных инструментов:
1. Составить список всех функциональных требований/пользовательских историй.
2. Для каждого требования указать, есть ли тест-кейс (позитивный, негативный, граничный).
3. Посчитать процент требований, у которых есть хотя бы один тест-кейс, и отдельно — процент требований с полным покрытием (позитив + негатив + граничные значения).

## Пример / когда применяется

Пример матрицы трассируемости:

| Требование | Тест-кейсы (позитив) | Тест-кейсы (негатив) | Покрыто |
|---|---|---|---|
| Авторизация по email/паролю | TC-001 | TC-002, TC-003 | Да |
| Восстановление пароля | TC-004 | — | Частично |
| Двухфакторная аутентификация | — | — | Нет |

Покрытие функционала = 2 из 3 (частично или полностью) = 67%. Метрика применяется при подготовке test summary report, при аудите качества тестового покрытия перед крупным релизом и при обосновании необходимости написания дополнительных тест-кейсов.

## На что смотрит интервьюер

Частая ошибка кандидатов — путать code coverage и requirements coverage, либо утверждать, что высокий процент покрытия кода гарантирует отсутствие багов. Хороший ответ чётко разделяет эти понятия и объясняет ограничения каждого. Красный флаг — незнание, как считается покрытие практически (через RTM или инструменты). Follow-up вопросы: почему 100% code coverage не гарантирует качество, как вы определяете, что покрытие функционала достаточно для релиза.
