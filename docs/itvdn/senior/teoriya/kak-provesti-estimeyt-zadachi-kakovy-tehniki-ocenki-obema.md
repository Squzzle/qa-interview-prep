# Как провести эстимейт задачи? Каковы техники оценки объема тестирования существуют?

**Коротко:** эстимейт проводится через декомпозицию задачи на подзадачи с последующей оценкой каждой из них (экспертным методом, по аналогии или через групповые техники вроде Planning Poker), с обязательным учётом рисков, неопределённости и буфера на непредвиденные ситуации.

## Развёрнутый ответ

Эстимейт тестирования — это прогноз трудозатрат и времени, необходимых для проверки функционала. Точная оценка невозможна, поэтому цель — дать достаточно надёжный диапазон, на который можно опираться при планировании.

Общий алгоритм проведения эстимейта:

1. Декомпозиция задачи. Крупная фича разбивается на более мелкие тестируемые единицы: подготовка тестовых данных, написание тест-кейсов, ручное тестирование, автоматизация, регресс, оформление отчёта.
2. Оценка каждой подзадачи по отдельности — оценивать мелкие части точнее, чем задачу целиком.
3. Учёт факторов неопределённости: новизна функционала, сложность интеграций, качество требований, доступность тестового окружения и тестовых данных.
4. Добавление буфера (обычно 15-30%) на риски и непредвиденные задачи.
5. Валидация оценки — сверка с похожими задачами в прошлом (историческими данными) и, при возможности, обсуждение с командой.

Основные техники оценки объёма тестирования:

- Экспертная оценка (Expert judgement) — оценка на основе опыта эксперта в аналогичных задачах; быстро, но субъективно.
- Оценка по аналогии (Analogy-based estimation) — сравнение с похожей задачей, выполненной ранее, с корректировкой на отличия.
- Декомпозиция (Work Breakdown Structure, WBS) — разбиение на мелкие задачи с последующим суммированием оценок; снижает погрешность.
- Planning Poker — групповая техника оценки в scrum-командах, где каждый участник независимо даёт оценку (в story points или часах), после чего расхождения обсуждаются до консенсуса; снижает эффект якорения и группового давления.
- Три точки (Three-point estimation / PERT) — рассчитывается оптимистичная (O), пессимистичная (P) и наиболее вероятная (M) оценки, итоговая формула: `E = (O + 4M + P) / 6`. Позволяет учесть неопределённость и получить более взвешенный результат.
- Test Point Analysis (TPA) — метод, специфичный для тестирования: оценка строится на основе сложности функциональных точек продукта и характеристик качества (аналог Function Point Analysis, адаптированный под тестирование).
- Процент от времени разработки — эмпирическое правило (например, тестирование = 30-50% от времени разработки фичи), используется как грубая быстрая оценка на ранних стадиях, но не заменяет детальный эстимейт.

Важный нюанс: любая оценка должна пересматриваться по ходу работы (re-estimation) — как только становится доступна новая информация (например, требования изменились или обнаружена скрытая сложность интеграции).

## Пример / когда применяется

Пример применения PERT для оценки тестирования новой фичи:
- Оптимистичная оценка: 2 дня
- Наиболее вероятная: 4 дня
- Пессимистичная: 8 дней
- Итог: `E = (2 + 4*4 + 8) / 6 = 26 / 6 ≈ 4.3 дня`

Planning Poker применяется на sprint planning в Scrum-командах для оценки в story points. WBS и экспертная оценка чаще применяются при оценке крупных релизов или проектов на этапе предпродажи (pre-sale estimation).

## На что смотрит интервьюер

Слабый ответ — называется только "экспертная оценка" без структурированного подхода к декомпозиции. Сильный ответ демонстрирует знание нескольких техник и понимание, когда какую применять (Planning Poker для agile-команды, WBS для крупного фиксированного проекта). Красный флаг — эстимейт без учёта рисков и буфера, что на практике почти всегда приводит к превышению сроков.
