# Как вы преодолеете трудности из-за отсутствия надлежащей документации для тестирования?

**Коротко:** отсутствие документации компенсируется активным сбором информации из доступных источников — кода, UI, коммуникации со стейкхолдерами и exploratory testing, а также параллельным восстановлением минимально необходимой документации.

## Развёрнутый ответ

Отсутствие требований — типичная ситуация в реальных проектах, особенно в стартапах и на legacy-проектах. QA-инженер не должен ждать идеальной документации, а должен уметь работать в условиях неопределённости, используя альтернативные источники знаний о продукте.

Основные шаги:

1. **Сбор информации из всех доступных источников.** Это могут быть: исходный код и коммиты (git log, pull request'ы), тикеты в баг-трекере, переписка в Slack/почте, старые версии документации, макеты в Figma, API-спецификации (Swagger/OpenAPI), базы знаний (Confluence), а также сам работающий продукт.

2. **Коммуникация со стейкхолдерами.** Прямые интервью с продакт-менеджером, бизнес-аналитиком, разработчиками и, если возможно, с конечными пользователями. Важно задавать уточняющие вопросы и фиксировать ответы письменно — это становится де-факто документацией.

3. **Exploratory testing.** Свободное исследование приложения помогает построить ментальную модель системы: какие есть модули, сценарии, ограничения. Результаты фиксируются в виде заметок, mind-map или чек-листов.

4. **Reverse engineering требований.** По UI и поведению системы восстанавливаются бизнес-правила: что происходит при валидных и невалидных данных, какие есть роли и права доступа.

5. **Составление собственной минимальной документации.** По итогам исследования QA сам создаёт test charter, чек-листы, карту рисков и, по возможности, согласовывает их с командой — это закрывает пробел для будущих итераций и для новых членов команды.

6. **Риск-ориентированный подход.** В условиях недостатка документации приоритет отдаётся тестированию критичных бизнес-сценариев (happy path, платежи, безопасность), а не попытке покрыть всё исчерпывающе.

## Пример / когда применяется

Типичный кейс: senior QA приходит на legacy-проект, где актуальной документации нет уже два года. План действий:

| Шаг | Действие | Результат |
|---|---|---|
| 1 | Exploratory-сессия 2-3 дня | Mind-map модулей приложения |
| 2 | Интервью с тимлидом и support-командой | Список известных проблемных зон |
| 3 | Анализ тикетов за последние 6 месяцев | Список регрессионных багов и хрупких мест |
| 4 | Составление risk-based test plan | Приоритизированный список сценариев для регресса |
| 5 | Фиксация найденного в Confluence | Базовая документация для команды |

## На что смотрит интервьюер

- Пассивный ответ вида "буду ждать, пока напишут документацию" — красный флаг, показывает неготовность работать в реальных условиях.
- Интервьюер ожидает конкретный план действий, а не общие фразы.
- Важно упомянуть коммуникацию с людьми, а не только технические артефакты — многие кандидаты забывают про "человеческий" источник информации.
- Follow-up вопросы: "А если стейкхолдеры недоступны?", "Как вы приоритизируете, что тестировать в первую очередь без требований?", "Как убедить менеджмент выделить время на восстановление документации?".
