# Какой подход вы используете для Test Cases Review?

**Коротко:** Test Cases Review — это процесс проверки тест-кейсов на полноту, корректность и соответствие требованиям силами коллег (peer review) или тимлида, обычно проводится по чек-листу и включает проверку покрытия, ясности шагов и соответствия критериям приёмки.

## Развёрнутый ответ

Ревью тест-кейсов — важная практика обеспечения качества самого тестирования (тестирование тестов). Она снижает риск пропуска сценариев, дублирования усилий и низкого качества тест-кейсов ещё до начала их выполнения.

Подход к ревью обычно включает следующие этапы:

1. **Определение критериев ревью (чек-лист).** Например:
   - Соответствует ли тест-кейс требованию/acceptance criteria?
   - Есть ли чёткие предусловия (preconditions)?
   - Шаги воспроизведения однозначны и атомарны?
   - Указан ожидаемый результат (expected result)?
   - Покрыты ли позитивные, негативные и граничные сценарии (boundary/edge cases)?
   - Нет ли дублирования с другими тест-кейсами?
   - Присвоены ли приоритет и категория (smoke/regression/functional)?
   - Есть ли привязка к требованию/user story (traceability)?

2. **Выбор формата ревью.** Может быть: peer review (кросс-проверка между QA-инженерами), review тимлидом/senior QA, или совместное ревью с бизнес-аналитиком/разработчиком для проверки понимания требований.

3. **Инструменты.** TestRail, Zephyr, Xray, Qase — большинство TMS поддерживают статусы ревью (Draft/In Review/Approved) и комментарии inline.

4. **Обратная связь и итерации.** Автор тест-кейса дорабатывает по комментариям, ревьюер повторно проверяет — процесс похож на code review в разработке.

5. **Приоритизация ревью по риску.** Не все тест-кейсы требуют одинаково глубокого ревью — критичные (payment flow, security) проверяются тщательнее, чем второстепенные UI-детали.

6. **Ревью на разных этапах.** Ревью может проходить как до выполнения (перед стартом тестирования, чтобы убедиться в покрытии), так и периодически (чтобы актуализировать устаревшие кейсы при изменении требований — см. связанный вопрос про частоту ревью документации).

Также важен подход "review by design" — тест-кейсы пишутся сразу с расчётом на понятность для другого читателя: чёткие заголовки, структура Given-When-Then или Steps/Expected Result, минимум неоднозначных формулировок.

## Пример / когда применяется

Чек-лист для ревью тест-кейса (используется как шаблон в TMS):

```
[ ] Название описывает суть проверки (не "Test 1", а конкретика)
[ ] Есть ссылка на требование/user story (traceability ID)
[ ] Preconditions указаны и достаточны
[ ] Шаги атомарны, воспроизводимы без вопросов
[ ] Expected result однозначен и проверяем
[ ] Присутствуют негативные и граничные сценарии
[ ] Приоритет и тип теста указаны (smoke/regression/etc.)
[ ] Нет дублирования с существующими тест-кейсами
[ ] Тест-кейс не зависит от состояния, оставленного другим тестом
```

Пример workflow в Jira/TestRail: автор переводит тест-кейс в статус "Ready for Review" → назначенный ревьюер (senior QA) проверяет по чек-листу → оставляет комментарии или апрувит → после исправлений кейс переходит в "Approved" и включается в test suite.

## На что смотрит интервьюер

- Ожидается структурированный подход с конкретным чек-листом, а не общее "смотрю, чтобы всё было нормально".
- Важно упомянуть traceability к требованиям — частый пропуск у junior/middle кандидатов.
- Красный флаг: отсутствие упоминания негативных/граничных сценариев в критериях ревью.
- Follow-up: "Как вы поступите, если автор тест-кейса не согласен с вашими замечаниями?", "Как часто нужно проводить ревью существующих тест-кейсов?".
