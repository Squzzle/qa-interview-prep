# Когда следует делать стресс-тестирование на проектах? От чего отталкиваться, когда строите сценарий для такого тестирования? Что учесть при выборе инструмента?

**Коротко:** Стресс-тестирование нужно проводить перед крупными нагрузочными событиями (маркетинговые акции, релизы с ожидаемым ростом трафика), при изменении архитектуры, а также периодически для критичных высоконагруженных систем; сценарий строится на основе реальных бизнес-профилей нагрузки (production-логи, аналитика), а инструмент выбирается исходя из протокола системы, масштаба нагрузки, стоимости и интеграции с CI/CD.

## Развёрнутый ответ

**Стресс-тестирование (stress testing)** — вид нагрузочного тестирования, цель которого — определить предел работоспособности системы, найти точку отказа (breaking point) и проверить, как система деградирует и восстанавливается после перегрузки (в отличие от load-тестирования, которое проверяет поведение под ожидаемой нагрузкой, и от soak/endurance-тестирования, проверяющего стабильность при длительной работе).

**Когда следует проводить:**
1. Перед событиями с ожидаемым скачком трафика: распродажи (Black Friday), маркетинговые кампании, релизы новых популярных функций, публичные анонсы.
2. После существенных изменений архитектуры (переход на микросервисы, смена БД, добавление кэширующего слоя) — нужно валидировать, что система не деградировала по производительности.
3. Периодически для критичных систем (финтех, e-commerce, healthcare) как часть регулярного цикла обеспечения качества — не только реактивно, но и проактивно.
4. Перед миграцией в новую инфраструктуру (переезд в облако, смена дата-центра, апгрейд версии СУБД).
5. При планировании capacity planning — чтобы понять, сколько ресурсов (серверов, реплик БД) потребуется для будущего роста аудитории.

**От чего отталкиваться при построении сценария:**
- **Реальные production-данные:** анализ логов сервера, аналитики (Google Analytics, APM-системы типа New Relic/Datadog) — какие эндпоинты наиболее нагружены, какое соотношение типов запросов (чтение/запись), какой пиковый RPS (requests per second) наблюдался исторически.
- **Бизнес-профиль пользователя:** типичный путь пользователя (user journey) — например, для e-commerce: 70% — просмотр каталога, 20% — добавление в корзину, 10% — оформление заказа. Сценарий стресс-теста должен отражать эту пропорцию, а не бить в один эндпоинт.
- **Пиковые сезонные факторы:** ожидаемый рост трафика относительно baseline (например, "в 5 раз больше обычного RPS во время акции").
- **Критичные бизнес-транзакции:** money path (оплата, оформление заказа) должен быть протестирован в первую очередь, так как его отказ наиболее дорог для бизнеса.
- **Постепенное нарастание нагрузки (ramp-up):** сценарий должен наращивать нагрузку не мгновенно, а ступенчато, чтобы зафиксировать именно точку деградации/отказа, а не просто "уронить" систему сразу.
- **Метрики для отслеживания:** время ответа (latency, percentile — p95/p99), throughput, error rate, использование CPU/memory/DB connections, поведение очередей.

**Что учесть при выборе инструмента:**
1. **Протокол системы** — HTTP/REST, WebSocket, gRPC, очереди сообщений (Kafka, RabbitMQ) — не все инструменты одинаково хорошо поддерживают все протоколы.
2. **Масштаб нагрузки** — нужно ли генерировать нагрузку с одной машины или требуется распределённая генерация (кластер нагрузочных генераторов) для достижения нужного RPS.
3. **Скриптование и гибкость** — насколько удобно описывать сложные сценарии (условная логика, корреляция данных между запросами, параметризация).
4. **Интеграция с CI/CD** — можно ли встроить нагрузочные тесты в пайплайн для регулярного прогона (например, JMeter/Gatling/k6 как шаг в CI).
5. **Стоимость** — open-source (JMeter, k6, Gatling Community) vs enterprise-решения (LoadRunner, BlazeMeter, Gatling Enterprise) с их поддержкой и облачной генерацией нагрузки.
6. **Отчётность и мониторинг** — насколько удобны встроенные отчёты, интеграция с Grafana/APM-системами для сопоставления клиентской нагрузки с серверными метриками.
7. **Порог входа команды** — JMeter (GUI + XML) проще для новичков, Gatling/k6 (код на Scala/JS) требуют программистских навыков, но дают лучшую поддерживаемость и версионирование сценариев в Git.

## Пример / когда применяется

Пример сценария стресс-теста на k6 для API оформления заказа с постепенным нарастанием нагрузки:

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '2m', target: 100 },   // разгон до 100 VU
    { duration: '5m', target: 500 },   // разгон до 500 VU
    { duration: '5m', target: 1500 },  // ищем точку отказа
    { duration: '3m', target: 0 },     // спад нагрузки
  ],
  thresholds: {
    http_req_duration: ['p(95)<800'], // 95% запросов быстрее 800мс
    http_req_failed: ['rate<0.05'],   // ошибок меньше 5%
  },
};

export default function () {
  const res = http.post('https://api.example.com/orders', JSON.stringify({
    userId: 42,
    items: [{ sku: 'ABC123', qty: 1 }],
  }), { headers: { 'Content-Type': 'application/json' } });

  check(res, {
    'status is 201': (r) => r.status === 201,
    'response time < 1s': (r) => r.timings.duration < 1000,
  });

  sleep(1);
}
```

Такой сценарий позволяет найти, при каком количестве виртуальных пользователей начинает расти error rate или latency выходит за приемлемый порог (`p95 < 800ms`), и как система себя ведёт после спада нагрузки (восстанавливается ли до нормальных показателей).

## На что смотрит интервьюер

- Различает ли кандидат стресс-тестирование, load-тестирование, spike-тестирование и soak/endurance-тестирование — частая путаница терминов на собеседованиях.
- Понимает ли важность построения сценария на основе реальных production-данных, а не произвольных цифр "из головы".
- Упоминает ли метрики (percentile latency, error rate, throughput), а не только "сервер упал/не упал".
- Учитывает ли при выборе инструмента не только сам факт генерации нагрузки, но и протокол, масштабируемость, интеграцию с CI/CD и стоимость.
- Красный флаг — если кандидат предлагает "просто максимально быстро долбить запросами" без плана ramp-up и без определения конкретных метрик успеха/провала.
