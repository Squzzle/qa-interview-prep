# На основе чего нужно составлять стратегию для проведения тестирования нагрузки?

**Коротко:** стратегия нагрузочного тестирования строится на основе бизнес-требований к производительности (SLA/NFR), профиля реальной нагрузки пользователей, критичных сценариев использования и характеристик production-окружения.

## Развёрнутый ответ

Нагрузочное тестирование (Performance/Load Testing) без чёткой стратегии превращается в бессмысленное "погонять систему нагрузкой и посмотреть, что будет". Правильная стратегия должна опираться на следующие источники и факторы:

1. **Нефункциональные требования (NFR) и SLA.** Заранее определённые бизнес-требования: например, "система должна выдерживать 1000 одновременных пользователей с временем отклика &lt; 2 секунд на 95-м перцентиле". Без этих цифр невозможно определить критерии успеха/провала теста.

2. **Профиль реальной нагрузки (Production traffic analysis).** Анализ реальных логов/аналитики: сколько пользователей в пиковые часы, какие операции выполняются чаще всего, каково соотношение чтения/записи, сезонность (например, распродажи, начало месяца для биллинга).

3. **Критичные бизнес-сценарии.** Не вся система тестируется с одинаковой интенсивностью — приоритет отдаётся сценариям с максимальным бизнес-влиянием: checkout, авторизация, поиск, генерация отчётов.

4. **Архитектура системы.** Понимание узких мест: база данных, внешние интеграции, очереди сообщений, кэширование — стратегия должна включать точки, где вероятны боттлнеки.

5. **Типы нагрузочных тестов, которые нужно провести:**
   - **Load Testing** — поведение системы при ожидаемой нагрузке.
   - **Stress Testing** — поведение системы за пределами нормальной нагрузки, поиск точки отказа.
   - **Spike Testing** — резкие всплески нагрузки (например, старт распродажи).
   - **Soak/Endurance Testing** — длительная нагрузка для выявления утечек памяти и деградации со временем.
   - **Scalability Testing** — проверка, насколько система масштабируется при увеличении ресурсов.

6. **Тестовое окружение.** Окружение должно максимально соответствовать production по конфигурации (количество инстансов, характеристики БД), иначе результаты будут нерепрезентативны.

7. **Тестовые данные.** Объём и характер данных должны отражать реальный production (например, объём таблиц в БД, размер файлов).

8. **Инструменты.** Выбор инструмента (JMeter, Gatling, k6, Locust) зависит от протокола (HTTP, WebSocket, gRPC), необходимости распределённой нагрузки и интеграции с CI/CD.

9. **Мониторинг и метрики.** Стратегия должна включать, какие метрики собираются во время теста: response time, throughput, error rate, CPU/memory/DB connections — это позволяет находить не только факт деградации, но и её причину.

## Пример / когда применяется

Пример структуры стратегии нагрузочного тестирования для интернет-магазина перед "Чёрной пятницей":

```
1. Цель: подтвердить, что система выдержит 5x
   обычной пиковой нагрузки (на основе аналитики
   прошлого года).

2. Целевые метрики (из NFR):
   - Response time p95 < 1.5s на checkout
   - Error rate < 0.1%
   - Throughput: 2000 RPS на каталог, 300 RPS на checkout

3. Сценарии:
   - Просмотр каталога (60% трафика)
   - Добавление в корзину (20%)
   - Checkout и оплата (15%)
   - Поиск (5%)

4. Типы тестов:
   - Load test на 5x baseline
   - Spike test — резкий рост за 1 минуту
   - Soak test — 4 часа на 3x baseline

5. Окружение: staging, идентичный prod
   по инфраструктуре (autoscaling отключен
   для контроля переменных)

6. Инструмент: k6 + Grafana/Prometheus для мониторинга
```

## На что смотрит интервьюер

- Ожидается упоминание NFR/SLA как отправной точки — без этого "стратегия" превращается в угадывание.
- Важно показать знание разных типов нагрузочных тестов (не только Load, но Stress/Spike/Soak).
- Красный флаг: ответ ограничивается упоминанием только инструмента (JMeter) без методологии.
- Follow-up: "Как вы определите реалистичный профиль нагрузки, если у продукта ещё нет реального трафика?", "Что делать, если тестовое окружение слабее production?".
