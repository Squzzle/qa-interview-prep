# Расскажите об алгоритмах шифрования трафика.

**Коротко:** Шифрование трафика делится на симметричное (один ключ для шифрования и расшифровки, быстрое — например `AES`) и асимметричное (пара публичный/приватный ключ, медленнее, но решает проблему безопасного обмена ключами — например `RSA`, `ECC`); на практике в протоколе `TLS`, который защищает большинство сетевого трафика, эти два подхода комбинируются, плюс используется хеширование (`SHA-256`) для проверки целостности.

## Развёрнутый ответ

**Симметричное шифрование.** Один и тот же ключ используется и для шифрования, и для расшифровки данных. Плюсы: высокая скорость, подходит для шифрования больших объёмов данных. Минус: сторонам нужно каким-то образом безопасно обменяться общим ключом заранее, что само по себе является проблемой в открытой сети.
- **AES (Advanced Encryption Standard)** — современный стандарт, блочный шифр с размером ключа 128/192/256 бит. Используется повсеместно (TLS, VPN, шифрование дисков).
- **ChaCha20** — потоковый шифр, альтернатива AES, особенно эффективен на устройствах без аппаратного ускорения AES (мобильные процессоры), используется, например, в TLS 1.3 и WireGuard.
- Устаревшие/небезопасные: `DES`, `3DES`, `RC4` — не должны использоваться в новых системах.

**Асимметричное шифрование (шифрование с открытым ключом).** Используется пара ключей: публичный (можно передавать открыто) и приватный (хранится в секрете). Данные, зашифрованные публичным ключом, может расшифровать только владелец приватного ключа (и наоборот — для цифровых подписей). Минус: вычислительно намного медленнее симметричного, поэтому не используется для шифрования больших потоков данных напрямую.
- **RSA** — классический алгоритм на основе сложности факторизации больших чисел, размер ключа обычно 2048-4096 бит.
- **ECC (Elliptic Curve Cryptography)** — на основе эллиптических кривых, даёт эквивалентную RSA стойкость при значительно меньшем размере ключа (например, ECC 256 бит ~ RSA 3072 бит), быстрее и экономичнее — используется всё активнее (ECDSA для подписей, ECDH для обмена ключами).
- **Diffie-Hellman (DH) / ECDHE** — протокол согласования общего секретного ключа по открытому каналу без его прямой передачи; ECDHE (Elliptic Curve Diffie-Hellman Ephemeral) даёт forward secrecy — компрометация долгосрочного ключа не позволяет расшифровать перехваченный ранее трафик.

**Хеширование** — не шифрование в чистом виде (нет обратной операции), но критичный элемент защиты трафика: обеспечивает проверку целостности данных (что данные не были изменены при передаче). Современный стандарт — `SHA-256`/`SHA-3`. Устаревшие/небезопасные: `MD5`, `SHA-1` — не должны применяться для криптографических целей из-за найденных коллизий.

**Как это комбинируется в TLS (протокол защиты трафика в HTTPS):**
1. **TLS handshake:** клиент и сервер согласуют версию протокола и набор алгоритмов (cipher suite).
2. **Асимметричная криптография** используется на этапе установления соединения: сервер предоставляет сертификат с публичным ключом (подписанный удостоверяющим центром — Certificate Authority), стороны через алгоритм типа ECDHE согласуют общий секретный ключ, не передавая его в открытом виде.
3. **Симметричное шифрование** (обычно AES-256-GCM или ChaCha20-Poly1305) используется дальше для шифрования всего фактического трафика — это быстро и эффективно для больших объёмов данных.
4. **HMAC/AEAD** (например, режим `GCM` объединяет шифрование и проверку целостности) обеспечивает, что данные не были подменены по пути (защита от man-in-the-middle).

TLS 1.3 (актуальная версия) упростил handshake (меньше round-trip'ов), убрал поддержку устаревших небезопасных алгоритмов (RC4, MD5, статический RSA-обмен ключами без forward secrecy) и сделал forward secrecy обязательным.

## Пример / когда применяется

С точки зрения QA важно уметь проверить, что трафик действительно защищён:

```bash
# Проверка версии TLS и cipher suite сервера
openssl s_client -connect example.com:443 -tls1_2

# Проверка сертификата
curl -vI https://example.com 2>&1 | grep -E "SSL|TLS|subject|issuer"
```

Типичные проверки в тест-плане безопасности:
- Приложение не позволяет установить соединение по устаревшим протоколам (`SSLv3`, `TLS 1.0`, `TLS 1.1`) — должен приниматься только `TLS 1.2+`.
- Сертификат действителен, не истёк, выдан доверенным CA, соответствует домену.
- Чувствительные данные (пароли, токены, номера карт) никогда не передаются по `http://`, только по `https://`.
- Заголовок `Strict-Transport-Security (HSTS)` присутствует, чтобы браузер не допускал downgrade до HTTP.
- Cookies с чувствительными данными помечены флагами `Secure` и `HttpOnly`.

## На что смотрит интервьюер

- Понимает ли кандидат разницу между симметричным и асимметричным шифрованием и почему они комбинируются в TLS, а не используется что-то одно.
- Знает ли конкретные современные алгоритмы (AES, ECC, ECDHE) и отличает их от устаревших/небезопасных (DES, RC4, MD5, SHA-1).
- Понимает ли роль хеширования как проверки целостности, а не шифрования данных.
- Может ли объяснить (хотя бы верхнеуровнево) механизм TLS handshake и понятие forward secrecy.
- Умеет ли перевести теорию в практические проверки безопасности трафика (HSTS, cookie flags, версия TLS, валидность сертификата) — это особенно ценится на Senior QA позиции, где теория должна конвертироваться в тестируемые критерии.
- Красный флаг — путаница терминов "шифрование" и "хеширование", либо ответ, ограничивающийся одним словом "HTTPS" без понимания механизма.
