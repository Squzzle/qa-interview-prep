# TOP OWASP: какие знаете уязвимости и методы защиты?

**Коротко:** OWASP Top 10 — авторитетный список самых критичных категорий уязвимостей веб-приложений (актуальная версия 2021 года), включающий нарушения контроля доступа, криптографические сбои, инъекции, небезопасный дизайн, неверную конфигурацию безопасности и другие; для каждой категории существуют конкретные практики защиты (валидация ввода, параметризованные запросы, принцип наименьших привилегий, шифрование, security-заголовки).

## Развёрнутый ответ

OWASP (Open Worldwide Application Security Project) Top 10 — регулярно обновляемый рейтинг наиболее критичных рисков безопасности веб-приложений. Актуальная версия — OWASP Top 10:2021:

1. **A01: Broken Access Control (нарушение контроля доступа).** Пользователь получает доступ к данным или функциям, на которые у него нет прав (например, изменение `id` в URL для доступа к чужому аккаунту — IDOR). Защита: проверка прав на каждом запросе на стороне сервера, принцип наименьших привилегий, deny-by-default политика.

2. **A02: Cryptographic Failures (криптографические сбои).** Передача или хранение чувствительных данных без шифрования либо со слабыми алгоритмами (например, MD5 для паролей). Защита: TLS для передачи данных, современные алгоритмы хеширования паролей (bcrypt, argon2), шифрование чувствительных данных в покое (at rest).

3. **A03: Injection (инъекции).** SQL-инъекции, NoSQL-инъекции, command injection, LDAP-инъекции — когда непроверенный пользовательский ввод исполняется как код/запрос. Защита: параметризованные запросы (prepared statements), ORM с безопасным экранированием, валидация и санитизация ввода.

4. **A04: Insecure Design (небезопасный дизайн).** Архитектурные просчёты, заложенные ещё на этапе проектирования (отсутствие threat modeling, отсутствие rate limiting для восстановления пароля). Защита: threat modeling на этапе дизайна, secure design patterns.

5. **A05: Security Misconfiguration (неверная конфигурация безопасности).** Дефолтные пароли, включённые debug-режимы в проде, лишние открытые порты, подробные сообщения об ошибках, раскрывающие внутреннее устройство системы. Защита: hardening чек-листы, автоматизированные проверки конфигурации, отключение ненужных фич.

6. **A06: Vulnerable and Outdated Components (уязвимые и устаревшие компоненты).** Использование библиотек с известными CVE. Защита: регулярный аудит зависимостей (`npm audit`, `OWASP Dependency-Check`, Snyk), своевременные обновления.

7. **A07: Identification and Authentication Failures (сбои идентификации и аутентификации).** Слабые пароли, отсутствие блокировки после N неудачных попыток входа, небезопасное хранение сессий. Защита: MFA, политика надёжных паролей, безопасное управление сессиями (httpOnly/secure cookie), rate limiting на login.

8. **A08: Software and Data Integrity Failures (нарушение целостности ПО и данных).** Например, небезопасная десериализация, отсутствие проверки подписи для обновлений/CI-пайплайнов (атака на supply chain). Защита: проверка цифровых подписей, безопасная десериализация, контроль зависимостей в CI/CD.

9. **A09: Security Logging and Monitoring Failures (недостаточное логирование и мониторинг).** Отсутствие логов о попытках атак не позволяет вовремя обнаружить инцидент. Защита: централизованное логирование, алертинг на подозрительную активность, аудит логов.

10. **A10: Server-Side Request Forgery (SSRF).** Сервер по запросу атакующего делает HTTP-запрос к произвольному (в том числе внутреннему) ресурсу. Защита: whitelisting разрешённых адресов, отключение доступа к метаданным облака, сегментация сети.

С точки зрения QA важно не просто знать список, а уметь тестировать эти уязвимости на базовом уровне (проверка на SQL-инъекции через `' OR '1'='1`, проверка на XSS через `<script>alert(1)</script>`, проверка прав доступа через подмену `id` в запросе — IDOR).

## Пример / когда применяется

Пример базового ручного теста на SQL-инъекцию в форме логина:

```
Username: admin' --
Password: любой
```

Если приложение уязвимо (строит запрос конкатенацией строк без параметризации), такой ввод может обойти проверку пароля.

Правильная защита на стороне разработки (Java, `PreparedStatement`):

```java
String sql = "SELECT * FROM users WHERE username = ? AND password_hash = ?";
PreparedStatement stmt = connection.prepareStatement(sql);
stmt.setString(1, username);
stmt.setString(2, hashPassword(password));
```

Пример проверки на IDOR: авторизоваться под пользователем A, скопировать URL просмотра заказа `/orders/1001`, изменить ID на `1002` (принадлежащий пользователю B) и убедиться, что сервер возвращает `403 Forbidden`, а не данные чужого заказа.

## На что смотрит интервьюер

- Может ли кандидат назвать хотя бы 4-5 категорий из актуального OWASP Top 10 2021, а не устаревшего списка 2013/2017 года.
- Понимает ли разницу между уязвимостью и методом эксплуатации (например, Broken Access Control как категория и IDOR как конкретный тип атаки внутри неё).
- Умеет ли предложить конкретный метод защиты, а не общие слова "нужно всё проверять".
- Может ли привести пример практического теста (payload для SQLi/XSS, сценарий проверки IDOR).
- Красный флаг — путаница между OWASP Top 10 для веб-приложений и OWASP Mobile Top 10 / API Security Top 10 без понимания, что это разные списки.
