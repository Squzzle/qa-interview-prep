# В чем смысл юнит-тестов?

**Коротко:** Юнит-тесты проверяют корректность отдельного изолированного модуля кода (функции, метода, класса) без внешних зависимостей, дают быструю обратную связь разработчику, фиксируют ожидаемое поведение и служат защитой от регрессий при рефакторинге.

## Развёрнутый ответ

Юнит-тест (unit test) — это автоматизированный тест, проверяющий наименьшую тестируемую единицу кода в изоляции от остальной системы: одну функцию, метод или класс. Ключевая идея — изоляция: все внешние зависимости (база данных, сеть, файловая система, другие сервисы) заменяются на моки, стабы или фейки (`Mock`, `Stub`, `Fake` объекты), чтобы тест проверял именно логику модуля, а не поведение его окружения.

Основной смысл юнит-тестов:

1. **Быстрая обратная связь.** Юнит-тесты выполняются за миллисекунды, их можно запускать сотнями и тысячами при каждом коммите или даже при сохранении файла. Это резко отличает их от E2E-тестов, которые могут выполняться минутами.

2. **Локализация дефектов.** Если юнит-тест падает, разработчик точно знает, в каком именно модуле проблема — не нужно искать по всей системе.

3. **Документирование поведения.** Хорошо написанные тесты читаются как спецификация: "если на вход подать X, метод должен вернуть Y" или "при таком состоянии должно быть выброшено исключение Z".

4. **Безопасный рефакторинг.** Наличие юнит-тестов позволяет менять внутреннюю реализацию модуля, не боясь сломать поведение, — тесты моментально сигнализируют о регрессии.

5. **Дизайн кода через тестируемость.** Написание юнит-тестов (особенно в подходе TDD — Test-Driven Development) подталкивает к слабому связыванию (low coupling) и внедрению зависимостей (Dependency Injection), потому что немодульный, монолитный код сложно тестировать в изоляции.

6. **Экономический эффект.** Согласно классической кривой стоимости дефектов, чем раньше найдена ошибка, тем дешевле её исправление. Юнит-тесты ловят баги на самом раннем этапе — при написании кода.

Важно понимать пирамиду тестирования (test pyramid): юнит-тестов должно быть больше всего (широкое основание), затем интеграционные тесты, и на вершине — немногочисленные E2E/UI-тесты. Это связано с балансом скорости, стоимости поддержки и стабильности.

## Пример / когда применяется

Пример юнит-теста на Java с JUnit 5 и Mockito для сервиса, который зависит от репозитория:

```java
class DiscountServiceTest {

    @Test
    void shouldApply10PercentDiscountForVipUser() {
        // Arrange
        UserRepository repoMock = mock(UserRepository.class);
        when(repoMock.findById(1L)).thenReturn(new User(1L, true)); // isVip = true
        DiscountService service = new DiscountService(repoMock);

        // Act
        BigDecimal result = service.calculateDiscount(1L, new BigDecimal("100"));

        // Assert
        assertEquals(new BigDecimal("90.00"), result);
    }

    @Test
    void shouldThrowExceptionWhenUserNotFound() {
        UserRepository repoMock = mock(UserRepository.class);
        when(repoMock.findById(99L)).thenReturn(null);
        DiscountService service = new DiscountService(repoMock);

        assertThrows(UserNotFoundException.class,
                () -> service.calculateDiscount(99L, new BigDecimal("100")));
    }
}
```

Здесь `UserRepository` замокан — тест не ходит в реальную БД, проверяется только логика расчёта скидки.

Юнит-тесты применяются повсеместно: для бизнес-логики, валидаторов, утилитных функций, парсеров, алгоритмов — везде, где есть чистая (или почти чистая) логика, которую можно изолировать.

## На что смотрит интервьюер

- Понимает ли кандидат разницу между юнит-, интеграционным и E2E-тестом (границы изоляции).
- Знает ли принципы `F.I.R.S.T.` (Fast, Independent, Repeatable, Self-validating, Timely) — качества хорошего юнит-теста.
- Умеет ли объяснить разницу между Mock, Stub и Fake.
- Не путает ли юнит-тестирование с модульным тестированием "через UI".
- Понимает ли связь юнит-тестов с тестируемым дизайном кода (SOLID, Dependency Injection).
- Красный флаг — если кандидат считает, что юнит-тесты пишет только разработчик и QA к ним не имеет отношения: на Senior-позиции ожидается понимание, что QA должен уметь читать, ревьюить и иногда писать юнит-тесты, а также оценивать их покрытие и качество.
