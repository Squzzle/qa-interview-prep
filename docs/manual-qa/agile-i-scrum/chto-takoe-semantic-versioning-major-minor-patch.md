# Что такое Semantic Versioning (MAJOR.MINOR.PATCH)?

**Коротко:** Semantic Versioning (SemVer) — это стандарт нумерации версий вида MAJOR.MINOR.PATCH, где каждая часть номера версии однозначно сигнализирует характер изменений и их влияние на обратную совместимость.

## Развёрнутый ответ

Формат версии: ```MAJOR.MINOR.PATCH``` (например, ```2.5.1```). Правила изменения каждой части строго определены спецификацией semver.org:

- **MAJOR** увеличивается при внесении несовместимых изменений API (breaking changes) — то, что работало раньше, может перестать работать или изменить поведение;
- **MINOR** увеличивается при добавлении новой функциональности с сохранением обратной совместимости — старый функционал продолжает работать так же;
- **PATCH** увеличивается при исправлении багов, не затрагивающем публичный интерфейс/поведение и без добавления новой функциональности;

При увеличении MAJOR — MINOR и PATCH сбрасываются в 0. При увеличении MINOR — PATCH сбрасывается в 0.

Дополнительно спецификация допускает:
- пре-релизные версии через дефис: ```1.0.0-alpha```, ```1.0.0-beta.2``` — обозначают нестабильные версии, которые имеют более низкий приоритет, чем соответствующая релизная версия;
- метаданные сборки через плюс: ```1.0.0+20250109``` — не влияют на приоритет версии при сравнении;
- версия ```0.y.z``` считается начальной разработкой — публичный API ещё нестабилен и может меняться в любой момент.

Значение для тестирования: по номеру версии тестировщик и релиз-менеджер сразу понимают объём и риск изменений. Major-релиз требует полного регрессионного прогона и внимания к обратной совместимости (в том числе к внешним интеграциям и API-контрактам); minor — тестирования новой функциональности плюс smoke по существующей; patch — точечной регрессии вокруг исправленного дефекта.

## Пример / когда применяется

Пример: версия API была ```2.3.4```.
- Исправили баг парсинга даты, не меняя поведение API → ```2.3.5``` (PATCH).
- Добавили новый необязательный параметр в эндпоинт, не ломая старых клиентов → ```2.4.0``` (MINOR).
- Изменили формат ответа эндпоинта, удалив старое поле → ```3.0.0``` (MAJOR), так как это ломает существующих потребителей API.

| Изменение | Что растёт | Пример |
|---|---|---|
| Исправление бага, поведение не меняется | PATCH | 2.3.4 → 2.3.5 |
| Новая фича, совместимость сохранена | MINOR | 2.3.5 → 2.4.0 |
| Breaking change | MAJOR | 2.4.0 → 3.0.0 |

## На что смотрит интервьюер

Частые ошибки: кандидат путает MINOR и PATCH или не может привести пример breaking change; не знает, что PATCH и MINOR обнуляются при повышении старшего разряда; не связывает версионирование с объёмом регрессионного тестирования. Возможные follow-up вопросы: как SemVer помогает планировать тестирование релиза, что такое breaking change применительно к REST API, как вы протестировали бы обратную совместимость перед major-релизом.
