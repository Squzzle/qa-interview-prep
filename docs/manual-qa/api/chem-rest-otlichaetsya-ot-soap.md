# Чем REST отличается от SOAP?

**Коротко:** REST — это гибкий архитектурный стиль поверх HTTP с форматом данных JSON/XML, а SOAP — строгий протокол обмена XML-сообщениями со встроенными стандартами безопасности и транзакций; REST проще и легче, SOAP надёжнее и формальнее.

## Развёрнутый ответ

Основные отличия можно сгруппировать по нескольким аспектам:

- **Тип сущности.** REST — архитектурный стиль (набор принципов), SOAP — протокол с формальной спецификацией.
- **Формат данных.** REST обычно использует JSON (реже XML, YAML), SOAP — только XML.
- **Транспорт.** REST жёстко привязан к HTTP/HTTPS, SOAP может работать поверх разных транспортов (HTTP, SMTP, JMS).
- **Контракт/документация.** REST описывается через OpenAPI/Swagger (не обязателен формально), SOAP всегда сопровождается WSDL — строгим машиночитаемым контрактом.
- **Гибкость vs строгость.** REST гибче, легче реализовать и расширить; SOAP строже типизирован через XSD, что снижает риск несовместимости, но усложняет разработку.
- **Вес сообщений.** SOAP-сообщения избыточны (много XML-обвязки), REST/JSON компактнее — меньше трафика, выше скорость.
- **Безопасность.** SOAP имеет встроенный стандарт WS-Security на уровне сообщения; REST полагается на транспортный уровень (HTTPS/TLS) и внешние механизмы (OAuth, JWT).
- **Транзакции.** SOAP поддерживает распределённые транзакции (WS-AtomicTransaction) "из коробки"; в REST это обычно реализуется на уровне бизнес-логики приложения.
- **Кэширование.** REST-запросы (особенно GET) могут кэшироваться стандартными средствами HTTP; SOAP-запросы обычно идут через POST и не кэшируются автоматически.
- **Идемпотентность и работа с ресурсами.** REST явно оперирует ресурсами через URI и HTTP-методы; SOAP оперирует вызовом операций/методов (похоже на RPC), а не ресурсами.

## Пример / когда применяется

| Критерий | REST | SOAP |
|---|---|---|
| Формат данных | JSON, XML | Только XML |
| Транспорт | HTTP/HTTPS | HTTP, SMTP, JMS и др. |
| Контракт | OpenAPI (не обязателен) | WSDL (обязателен) |
| Вес сообщений | Компактные | Избыточные |
| Безопасность | HTTPS, OAuth, JWT | WS-Security (в сообщении) |
| Типичное применение | Публичные API, мобильные бэкенды | Банки, госсектор, enterprise-интеграции |

Пример: сервис онлайн-магазина для мобильного приложения выберет REST — важна скорость разработки и лёгкость. Банковская система межбанковских переводов выберет SOAP — важны строгая типизация, встроенная безопасность и гарантии транзакций.

## На что смотрит интервьюер

- Может ли кандидат назвать не менее 4-5 конкретных отличий, а не только "SOAP это XML, а REST это JSON".
- Понимает ли, что выбор REST vs SOAP — это компромисс между простотой и строгостью/надёжностью.
- Follow-up: "какой протокол вы бы выбрали для платёжной системы и почему".
- Красный флаг: категоричное утверждение "REST всегда лучше SOAP" без понимания контекста применения.
