# Чем синхронное взаимодействие с API отличается от асинхронного?

**Коротко:** При синхронном взаимодействии клиент отправляет запрос и блокируется в ожидании ответа с готовым результатом в рамках одного HTTP-соединения; при асинхронном — сервер сразу подтверждает приём запроса, а результат обработки приходит позже другим способом (опрос, коллбэк, событие).

## Развёрнутый ответ

**Синхронное взаимодействие** — классическая модель "запрос-ответ" (request-response): клиент отправляет HTTP-запрос и ждёт, пока сервер полностью обработает его и вернёт финальный результат в том же ответе. Это подходит для быстрых операций (чтение данных, простая валидация, короткие вычисления), где задержка не критична. Клиенту не нужно ничего дополнительно реализовывать — он получает результат сразу.

**Асинхронное взаимодействие** применяется, когда обработка запроса занимает много времени (генерация отчёта, обработка видео, сложные вычисления, интеграция с внешними системами) и держать HTTP-соединение открытым нецелесообразно или невозможно (таймауты прокси, балансировщиков). В этом случае сервер:

1. Принимает запрос и сразу отвечает статусом вроде `202 Accepted`, возвращая идентификатор задачи (`task_id`, `job_id`).
2. Обрабатывает задачу в фоне (в очереди, воркерах).
3. Результат клиент получает одним из способов: периодически опрашивая статус задачи (polling), получая уведомление через WebSocket, или через webhook — callback-запрос от сервера на заранее заданный URL клиента.

Ключевые отличия:

| Критерий | Синхронное | Асинхронное |
|---|---|---|
| Ожидание ответа | Клиент блокируется до получения результата | Клиент получает подтверждение сразу, результат — позже |
| Длительность операции | Короткие операции | Длительные/тяжёлые операции |
| Сложность реализации | Проще | Сложнее (нужен механизм доставки результата) |
| Устойчивость к таймаутам | Ограничена таймаутом соединения | Не зависит от длительности обработки |
| Типичные коды ответа | 200 OK | 202 Accepted + отдельный статус-эндпоинт |

С точки зрения тестирования, асинхронные сценарии требуют отдельного внимания: нужно проверять корректность статуса задачи на разных этапах (pending, in_progress, completed, failed), идемпотентность повторных запросов, обработку таймаутов и retries, а также содержимое итогового результата после завершения обработки.

## Пример / когда применяется

Синхронный запрос — получение профиля пользователя:

```
GET /api/users/42
200 OK
{ "id": 42, "name": "Иван" }
```

Асинхронный запрос — запуск генерации отчёта:

```
POST /api/reports
202 Accepted
{ "task_id": "abc123", "status": "pending" }
```

Далее клиент опрашивает статус:

```
GET /api/reports/abc123
200 OK
{ "task_id": "abc123", "status": "completed", "result_url": "https://.../report.pdf" }
```

## На что смотрит интервьюер

- Понимание, что асинхронность нужна не "для красоты", а из-за долгих операций и ограничений по таймаутам.
- Знание типичных кодов ответа (`202 Accepted`) и паттернов (task/job с отдельным статус-эндпоинтом).
- Умение назвать способы получения результата асинхронной операции: polling, webhook, WebSocket.
- Понимание, как тестировать асинхронные процессы: проверка промежуточных статусов, таймаутов, повторных запросов, обработку ошибок в фоновой задаче.
- Красный флаг — путаница между "асинхронным API" и просто "медленным API".
