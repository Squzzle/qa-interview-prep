# Что такое API (Application Programming Interface)?

**Коротко:** API (Application Programming Interface, программный интерфейс приложения) — это набор правил и контрактов, по которым одна программа может обращаться к функциям или данным другой программы, не зная деталей её внутренней реализации.

## Развёрнутый ответ

API определяет, какие запросы можно отправить системе, в каком формате, какие параметры обязательны, а какие данные придут в ответ. Это своего рода "договор" между поставщиком функциональности (сервером, библиотекой, операционной системой) и потребителем (клиентом, другим приложением, разработчиком).

Ключевая идея API — инкапсуляция: внутренняя логика системы скрыта, наружу выставлены только точки взаимодействия (эндпоинты, методы, функции). Это позволяет менять реализацию, не ломая клиентов, если контракт (интерфейс) остаётся прежним.

API бывают разных уровней:
- Библиотечные (SDK) API — функции и классы внутри одного языка программирования.
- Веб-API (Web API) — взаимодействие через HTTP/HTTPS между разными системами, чаще всего именно про них говорят на собеседованиях по QA.
- Операционные API — интерфейсы ОС для работы с файлами, памятью, устройствами.

Для тестировщика важно понимать API прежде всего как HTTP/сетевой интерфейс: клиент отправляет запрос (метод, URL, заголовки, тело), сервер возвращает ответ (код статуса, заголовки, тело), и задача тестировщика — проверить, что поведение соответствует спецификации (документации API) во всех сценариях, включая граничные и негативные.

## Пример / когда применяется

Пример: мобильное приложение погоды не хранит данные о погоде само — оно обращается к API стороннего сервиса, отправляя запрос вида:

```
GET https://api.weather.com/v3/wx/forecast?geocode=55.75,37.62&apiKey=XXXX
```

и получает JSON-ответ с прогнозом. Приложению не важно, как устроена внутренняя база данных сервиса погоды — важен только контракт запрос/ответ.

Тестирование API применяется, когда нужно проверить бизнес-логику до появления UI, покрыть интеграции между микросервисами, ускорить регрессионное тестирование (API-тесты быстрее UI-тестов).

## На что смотрит интервьюер

- Не путает ли кандидат API с конкретной реализацией (REST) — API это более широкое понятие.
- Понимает ли кандидат, что API — это контракт, а не просто "способ обмена данными".
- Может ли привести пример из практики (тестирование конкретного эндпоинта).
- Красный флаг: ответ "API — это когда фронтенд стучится на бэкенд" без понимания сути интерфейса и инкапсуляции.
