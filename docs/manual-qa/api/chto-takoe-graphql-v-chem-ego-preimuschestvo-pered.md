# Что такое GraphQL? В чем его преимущество перед REST?

**Коротко:** GraphQL — язык запросов и рантайм для API, который позволяет клиенту точно указывать, какие данные и в каком объёме ему нужны, получая всё за один запрос вместо нескольких запросов к разным REST-эндпоинтам.

## Развёрнутый ответ

GraphQL был разработан Facebook и открыт как спецификация в 2015 году. В отличие от REST, где сервер жёстко определяет структуру каждого эндпоинта и набор возвращаемых полей, в GraphQL есть единая точка входа (обычно один URL, например `/graphql`), а клиент сам формирует запрос, описывая нужные поля и вложенные связи.

Основные компоненты GraphQL:
- **Schema (схема)** — описание типов данных, полей и связей между ними на сервере.
- **Query** — операция чтения данных.
- **Mutation** — операция изменения данных (создание, обновление, удаление).
- **Subscription** — операция подписки на события в реальном времени.
- **Resolver** — функция на сервере, которая знает, как получить значение конкретного поля.

Преимущества GraphQL перед REST:
1. **Отсутствие over-fetching и under-fetching.** В REST клиент часто получает больше полей, чем нужно (over-fetching), либо вынужден делать несколько запросов, чтобы собрать все нужные данные (under-fetching). В GraphQL клиент запрашивает ровно то, что нужно, за один запрос.
2. **Один эндпоинт вместо множества.** Не нужно поддерживать десятки REST-маршрутов под разные комбинации данных — есть единая схема.
3. **Строгая типизация схемы.** Схема описывает все возможные типы, поля и их типы данных, что упрощает валидацию и автогенерацию документации/клиентского кода.
4. **Гибкая агрегация данных.** Удобно объединять данные из нескольких источников/микросервисов в одном ответе.
5. **Самодокументируемость.** Схему можно интроспектировать (introspection query) — GraphQL Playground/GraphiQL позволяют исследовать API прямо "изнутри".

Недостатки/нюансы, которые стоит знать: сложнее кэшировать ответы стандартными HTTP-механизмами (обычно все запросы идут через POST на один URL), сложнее реализовать rate limiting и мониторинг по "эндпоинтам" (все запросы идут в одну точку), возможны проблемы производительности при глубоко вложенных запросах (N+1 проблема), требуется больше усилий на проектирование схемы и защиту от чрезмерно "дорогих" запросов.

## Пример / когда применяется

Пример GraphQL-запроса, который получает имя пользователя и заголовки только трёх последних постов за один запрос:

```graphql
query {
  user(id: "123") {
    name
    posts(limit: 3) {
      title
    }
  }
}
```

Эквивалент в REST потребовал бы минимум двух запросов: `GET /users/123` и `GET /users/123/posts?limit=3`, причём ответ на первый запрос мог бы содержать лишние поля (email, адрес и т.д.), которые клиенту не нужны.

GraphQL особенно уместен, когда одни и те же данные потребляются разными клиентами (веб, мобильное приложение, смарт-часы) с разными требованиями к набору полей, либо когда нужно агрегировать данные из множества микросервисов в единый ответ (GraphQL-шлюз/BFF).

## На что смотрит интервьюер

- Понимает ли кандидат конкретные термины over-fetching/under-fetching и может привести пример.
- Знает ли о недостатках GraphQL (кэширование, N+1, сложность безопасности), а не только о преимуществах.
- Может ли сравнить подходы к тестированию: в REST тестируется много эндпоинтов, в GraphQL — комбинации запросов к единой схеме.
- Красный флаг: утверждение, что GraphQL всегда лучше REST и должен полностью его заменить — это показывает недостаток практического опыта.
