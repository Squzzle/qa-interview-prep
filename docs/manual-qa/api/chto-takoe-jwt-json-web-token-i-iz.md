# Что такое JWT (JSON Web Token) и из каких частей он состоит?

**Коротко:** JWT — это компактный, самодостаточный токен для передачи данных между сторонами в виде JSON, подписанный (и опционально зашифрованный) для проверки подлинности; состоит из трёх частей: заголовка (header), полезной нагрузки (payload) и подписи (signature), разделённых точками.

## Развёрнутый ответ

JWT (JSON Web Token) — открытый стандарт (RFC 7519) для представления утверждений (claims) в виде JSON-объекта, который передаётся как строка в кодировке Base64URL. Токен состоит из трёх частей, разделённых точкой (`.`): `header.payload.signature`.

1. **Header (заголовок)** — содержит метаданные: тип токена (`typ: JWT`) и алгоритм подписи (`alg`, например `HS256` или `RS256`). Кодируется в Base64URL.
2. **Payload (полезная нагрузка)** — содержит claims (утверждения) — данные о пользователе и метаданные токена. Различают:
   - зарегистрированные claims (registered claims): `iss` (issuer, кто выдал токен), `sub` (subject, идентификатор пользователя), `exp` (expiration time, время истечения), `iat` (issued at, время выдачи), `aud` (audience, для кого предназначен);
   - публичные claims — общепринятые, но не зарезервированные стандартом;
   - приватные (кастомные) claims — специфичные для конкретного приложения (например, `role`, `email`).
   Payload также кодируется в Base64URL, но НЕ шифруется — это важно: любой может декодировать payload и прочитать его содержимое, поэтому в него нельзя класть секретные данные (пароли и т.п.).
3. **Signature (подпись)** — вычисляется как результат применения алгоритма подписи (например, HMAC-SHA256 или RSA) к строке `base64(header) + "." + base64(payload)` с использованием секретного ключа (для симметричных алгоритмов, HS256) или приватного ключа (для асимметричных, RS256). Подпись позволяет серверу проверить, что токен не был изменён после выдачи, и что он выдан доверенным источником.

Ключевое преимущество JWT — самодостаточность: сервер может проверить подлинность токена, просто пересчитав подпись, без обращения к базе данных или хранилищу сессий. Это упрощает горизонтальное масштабирование (stateless-авторизация), но создаёт нюанс: пока не истёк срок действия (`exp`), отозвать JWT "досрочно" сложно (нужны дополнительные механизмы — blacklist токенов, короткое время жизни + refresh token).

## Пример / когда применяется

Пример JWT-токена (условный, сокращённый):

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0IiwibmFtZSI6Ikl2YW4iLCJleHAiOjE3MjAwMDAwMDB9.4f8h2s9d...
```

Декодированный header:

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

Декодированный payload:

```json
{
  "sub": "1234",
  "name": "Ivan",
  "exp": 1720000000
}
```

При каждом запросе клиент передаёт токен в заголовке `Authorization: Bearer <jwt>`, сервер проверяет подпись и срок действия (`exp`), после чего доверяет данным из payload (например, `sub` — id пользователя) без дополнительного похода в базу.

## На что смотрит интервьюер

- Знает ли кандидат все три части токена и их назначение (особенно то, что payload не шифруется, а лишь кодируется).
- Понимает ли разницу между кодированием (Base64URL) и шифрованием — частая путаница.
- Знает ли про claim `exp` и проблему невозможности мгновенного отзыва JWT.
- Follow-up: "как протестировать поведение при истёкшем токене", "что будет, если изменить payload вручную — сработает ли токен" (ответ: нет, подпись перестанет совпадать).
- Красный флаг: утверждение, что данные в JWT "зашифрованы и недоступны для чтения" — на самом деле payload легко декодируется, это критичный момент для тестирования безопасности.
