# Что такое коллекция (Collection) в Postman и зачем она нужна?

**Коротко:** Коллекция в Postman — это упорядоченный набор связанных HTTP-запросов (и вложенных папок с запросами), объединённых логикой конкретного API или сценария, который можно хранить, версионировать, переиспользовать и запускать целиком.

## Развёрнутый ответ

Коллекция — базовая единица организации работы в Postman. Вместо хранения десятков разрозненных запросов кандидат группирует их в одну структуру, что даёт следующие преимущества:

- **Организация и структурирование** — запросы можно объединять в папки по модулям/фичам API (например, "Users", "Orders", "Auth"), что упрощает навигацию в больших проектах.
- **Переиспользование** — коллекцию можно экспортировать в JSON-файл и импортировать в другой проект, поделиться с командой, положить в систему контроля версий (Git), что обеспечивает воспроизводимость тестового набора.
- **Коллекционные переменные** — значения, общие для всех запросов внутри коллекции (например, `base_url`), не завязанные на конкретное окружение.
- **Скрипты на уровне коллекции** — pre-request и test-скрипты можно задать не только для отдельного запроса, но и на уровне всей коллекции или папки, и они будут выполняться для каждого вложенного запроса (например, общая проверка, что статус ответа не 5xx, или общая авторизация).
- **Массовый запуск (Collection Runner)** — всю коллекцию (или отдельную папку) можно запустить последовательно как единый тестовый прогон, с указанием количества итераций и внешних данных (CSV/JSON) для параметризации.
- **Автоматизация через Newman** — коллекция экспортируется и запускается из командной строки/CI-пайплайна без графического интерфейса, что превращает набор ручных проверок в автоматизированный регресс.
- **Документирование API** — на основе коллекции Postman может сгенерировать читаемую документацию с примерами запросов и ответов, которой могут пользоваться другие команды-потребители API.
- **Общий доступ и совместная работа** — коллекции можно шарить внутри рабочего пространства (Workspace) команды, что упрощает синхронизацию тестовых сценариев между разработчиками и QA.

Таким образом коллекция решает задачу превращения набора разрозненных ручных запросов в управляемый, воспроизводимый и автоматизируемый тестовый актив.

## Пример / когда применяется

Структура типичной коллекции для тестирования API интернет-магазина:

```
Orders API (Collection)
├── Auth
│   └── POST /login
├── Users
│   ├── GET /users/{id}
│   └── POST /users
└── Orders
    ├── POST /orders
    ├── GET /orders/{id}
    └── DELETE /orders/{id}
```

Запуск коллекции целиком через Newman в CI:

```bash
newman run "Orders API.postman_collection.json" \
  -e staging.postman_environment.json \
  --reporters cli,junit
```

Коллекция применяется, когда нужно системно протестировать целый API (регрессионный набор), поделиться готовым набором запросов с командой или включить проверки API в автоматический pipeline.

## На что смотрит интервьюер

- Понимание коллекции как основной единицы организации и переиспользования тестовых запросов, а не просто "папки с запросами".
- Знание, что скрипты и переменные можно задавать на уровне коллекции, а не только внутри каждого отдельного запроса.
- Умение объяснить связь коллекции с Collection Runner и Newman для автоматизации.
- Понимание практики хранения коллекций в системе контроля версий для командной работы.
- Красный флаг — если кандидат не может объяснить, зачем группировать запросы, кроме "для порядка", и не знает про Collection Runner/Newman.
