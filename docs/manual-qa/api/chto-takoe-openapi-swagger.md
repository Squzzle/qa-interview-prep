# Что такое OpenAPI (Swagger)?

**Коротко:** OpenAPI Specification — это стандарт машиночитаемого описания REST API в формате YAML/JSON; Swagger — исторически первое и наиболее известное семейство инструментов (Swagger UI, Swagger Editor, Swagger Codegen) для работы с этой спецификацией.

## Развёрнутый ответ

Изначально спецификация называлась Swagger Specification, была создана компанией SmartBear. В 2015 году она была передана в открытое сообщество (OpenAPI Initiative под Linux Foundation) и переименована в OpenAPI Specification (OAS). Сейчас термины часто используют как синонимы, но формально: OpenAPI — сама спецификация/стандарт, Swagger — набор инструментов вокруг него.

OpenAPI-документ (обычно файл `openapi.yaml` или `swagger.json`) описывает:
- **info** — метаданные API: название, версия, описание.
- **servers** — базовые URL, по которым доступен API (например, dev/staging/production окружения).
- **paths** — список эндпоинтов, HTTP-методов, параметров запроса, тел запроса/ответа.
- **components/schemas** — переиспользуемые модели данных (структуры объектов).
- **security** — схемы авторизации (API-key, Bearer-токен, OAuth2 и т.д.).

Ключевые инструменты экосистемы Swagger/OpenAPI:
- **Swagger UI** — генерирует интерактивную веб-страницу документации, где можно прямо из браузера отправлять тестовые запросы к API.
- **Swagger Editor** — редактор для написания и валидации OpenAPI-спецификации.
- **Swagger Codegen / OpenAPI Generator** — генерация клиентского кода (SDK) и серверных заглушек на основе спецификации.

Для тестировщика OpenAPI-документ — это основной источник для проектирования тест-кейсов: он показывает обязательные и опциональные поля, допустимые значения (enum), форматы (email, date-time), а также используется для контрактного тестирования (проверка, что реальный ответ сервера соответствует заявленной схеме) и генерации моков/автотестов инструментами вроде Postman, Dredd, Schemathesis.

## Пример / когда применяется

Фрагмент OpenAPI-спецификации:

```yaml
openapi: 3.0.0
info:
  title: User API
  version: 1.0.0
servers:
  - url: https://api.example.com/v1
paths:
  /users:
    post:
      summary: Создать пользователя
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [name, email]
              properties:
                name:
                  type: string
                email:
                  type: string
                  format: email
      responses:
        '201':
          description: Пользователь создан
        '400':
          description: Некорректные данные
```

По этому фрагменту тестировщик сразу видит, что поля `name` и `email` обязательны, `email` должен соответствовать формату email, и может составить тест-кейсы на отсутствие обязательных полей, некорректный формат email, а также проверить ожидаемые коды 201 и 400.

## На что смотрит интервьюер

- Знает ли кандидат разницу между OpenAPI (спецификация) и Swagger (инструменты) хотя бы на уровне истории/происхождения.
- Умеет ли кандидат читать OpenAPI-документ и извлекать из него тест-кейсы.
- Знает ли о контрактном тестировании на основе OpenAPI.
- Follow-up: "как вы будете тестировать API, если реализация расходится с OpenAPI-документацией".
- Красный флаг: считает OpenAPI и Swagger двумя независимыми, никак не связанными вещами.
