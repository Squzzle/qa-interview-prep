# Что такое пагинация в API и зачем она нужна?

**Коротко:** Пагинация — это разбиение большого набора данных в ответе API на части (страницы), чтобы не отдавать клиенту весь массив данных за один запрос.

## Развёрнутый ответ

Когда API возвращает коллекцию ресурсов (например, список заказов, пользователей, товаров), объём данных может быть очень большим — тысячи и миллионы записей. Если отдавать всё сразу, это приводит к нескольким проблемам:

- Долгий ответ сервера и высокая нагрузка на БД (большие выборки, сортировки, джойны).
- Большой объём трафика и медленная передача данных по сети.
- Высокое потребление памяти на клиенте и на сервере при формировании ответа.
- Неудобство для конечного пользователя интерфейса — невозможно физически отобразить миллион строк на экране.

Пагинация решает эти проблемы, позволяя клиенту запрашивать данные порциями (страницами), явно указывая, какую часть данных он хочет получить. Сервер обрабатывает только запрошенный фрагмент, что снижает нагрузку и ускоряет ответ.

Существует несколько подходов к реализации пагинации:

- **Offset-based (постраничная через смещение)** — клиент указывает номер страницы или смещение и размер страницы. Проста в реализации, но на больших объёмах данных может деградировать по производительности и давать нестабильные результаты при одновременных вставках/удалениях записей.
- **Cursor-based (курсорная)** — вместо номера страницы используется указатель (курсор) на последний полученный элемент, обычно это значение уникального поля (например, id или timestamp). Работает стабильнее при изменяющихся данных и эффективнее на больших таблицах.
- **Keyset pagination** — разновидность курсорной пагинации, где фильтрация идёт по значению ключа (`WHERE id > last_id LIMIT N`), что позволяет избежать дорогого `OFFSET` в БД.

Помимо самих данных, ответ обычно содержит метаинформацию: общее количество записей, номер текущей страницы, ссылки на следующую/предыдущую страницу (в духе HATEOAS), либо курсор для следующего запроса.

## Пример / когда применяется

Типичный ответ API со списком с пагинацией:

```json
{
  "data": [
    { "id": 101, "name": "Товар A" },
    { "id": 102, "name": "Товар B" }
  ],
  "meta": {
    "page": 2,
    "per_page": 20,
    "total": 143,
    "total_pages": 8
  }
}
```

Пример запроса с cursor-based пагинацией:

```
GET /api/orders?limit=20&cursor=eyJpZCI6MTAyfQ==
```

Пагинация применяется практически везде, где отдаётся список: списки заказов, комментариев, логов, пользователей, результатов поиска.

## На что смотрит интервьюер

- Понимает ли кандидат разницу между offset-based и cursor-based пагинацией, знает ли их плюсы и минусы.
- Может ли объяснить проблему производительности `OFFSET` на больших таблицах.
- Знает ли, как тестировать пагинацию: граничные случаи (первая/последняя страница, пустой результат, дублирование записей при вставке новых данных между запросами страниц, некорректные значения `limit`/`page`, отрицательные числа, превышение максимального лимита).
- Красный флаг — если кандидат не может объяснить, зачем вообще нужна пагинация, помимо "чтобы было удобно".
