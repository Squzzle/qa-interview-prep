# Что такое REST API? Какие принципы лежат в его основе (RESTful)?

**Коротко:** REST (Representational State Transfer) — архитектурный стиль построения API поверх HTTP, где ресурсы идентифицируются через URL и обрабатываются стандартными HTTP-методами; RESTful-сервис следует набору принципов (клиент-сервер, отсутствие состояния, единообразный интерфейс и др.).

## Развёрнутый ответ

REST был предложен Роем Филдингом в 2000 году как набор ограничений (constraints) для построения распределённых систем. Сервис называется RESTful, если он соблюдает следующие принципы:

1. **Клиент-сервер** — разделение ответственности: клиент занимается UI и пользовательским опытом, сервер — хранением данных и бизнес-логикой; они независимо развиваются.
2. **Отсутствие состояния (Stateless)** — каждый запрос от клиента к серверу должен содержать всю информацию, необходимую для его обработки; сервер не хранит состояние сессии клиента между запросами.
3. **Кэшируемость (Cacheable)** — ответы должны явно помечаться как кэшируемые или нет, чтобы клиент мог повторно использовать данные и снижать нагрузку.
4. **Единообразный интерфейс (Uniform Interface)** — ключевой принцип REST, включает:
   - идентификацию ресурсов через URI (например, `/users/123`);
   - манипуляцию ресурсами через представления (JSON/XML), а не напрямую;
   - самоописываемые сообщения (заголовки, коды статуса, Content-Type);
   - HATEOAS (Hypermedia As The Engine Of Application State) — ответ может содержать ссылки на связанные действия/ресурсы.
5. **Слоистая система (Layered System)** — клиент не обязан знать, взаимодействует ли он напрямую с сервером или через промежуточные слои (прокси, балансировщики, шлюзы).
6. **Код по требованию (Code on Demand, опционально)** — сервер может передавать клиенту исполняемый код (например, JavaScript).

На практике REST API использует стандартные HTTP-методы применительно к ресурсам: GET (получить), POST (создать), PUT/PATCH (обновить), DELETE (удалить), а также HTTP-коды статуса для передачи результата операции (200, 201, 400, 404, 500 и т.д.).

Важно: многие сервисы называют себя "REST API", но фактически не соблюдают все принципы (например, хранят состояние сессии или не используют HATEOAS) — на практике под REST часто понимают просто "HTTP API с JSON", это стоит уточнять на собеседовании.

## Пример / когда применяется

Пример RESTful-запросов для ресурса "пользователь":

```
GET    /users        — получить список пользователей
GET    /users/123    — получить пользователя с id=123
POST   /users        — создать нового пользователя
PUT    /users/123    — полностью обновить пользователя 123
PATCH  /users/123    — частично обновить пользователя 123
DELETE /users/123    — удалить пользователя 123
```

Пример ответа сервера:

```json
{
  "id": 123,
  "name": "Ivan Petrov",
  "email": "ivan@example.com"
}
```

REST применяется практически везде, где нужен простой, понятный, легко тестируемый API для веб- и мобильных клиентов.

## На что смотрит интервьюер

- Знает ли кандидат конкретные принципы REST, а не только "GET/POST/PUT/DELETE".
- Понимает ли разницу между "RESTful API" и "просто HTTP API с JSON".
- Может ли объяснить stateless и зачем он нужен.
- Follow-up: "чем PUT отличается от PATCH", "что такое идемпотентность метода".
- Красный флаг: путает REST с форматом JSON или считает, что REST обязательно означает JSON (JSON — это только распространённый формат представления, не требование REST).
