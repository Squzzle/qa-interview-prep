# Что такое RESTful-системы?

**Коротко:** RESTful-системы — это API, построенные по архитектурному стилю REST (Representational State Transfer), где взаимодействие с ресурсами происходит через стандартные HTTP-методы, а сами ресурсы идентифицируются URI.

## Развёрнутый ответ

REST (Representational State Transfer) — архитектурный стиль построения распределённых систем, предложенный Роем Филдингом в диссертации 2000 года. RESTful-системой называют API или веб-сервис, который следует принципам REST. Ключевые принципы:

- **Клиент-серверная архитектура** — разделение ответственности между клиентом (UI/логика представления) и сервером (хранение данных, бизнес-логика).
- **Отсутствие состояния (Stateless)** — каждый запрос от клиента содержит всю информацию, необходимую для его обработки; сервер не хранит контекст сессии между запросами (аутентификация передаётся, например, токеном в каждом запросе).
- **Кэшируемость (Cacheable)** — ответы должны явно помечаться как кэшируемые или нет, что повышает производительность.
- **Единообразный интерфейс (Uniform Interface)** — включает:
  - идентификацию ресурсов через URI (например, `/users/123`);
  - манипуляцию ресурсами через представления (JSON/XML);
  - самоописываемые сообщения (заголовки, типы содержимого);
  - HATEOAS (Hypermedia as the Engine of Application State) — ответы могут содержать ссылки на связанные действия/ресурсы.
- **Слоистая система (Layered System)** — клиент не обязан знать, взаимодействует ли он напрямую с сервером или через промежуточные слои (балансировщики, прокси, шлюзы).
- **Code on Demand (опционально)** — сервер может передавать клиенту исполняемый код (например, JavaScript).

На практике RESTful API использует стандартные HTTP-методы для операций над ресурсами:

- GET — получение ресурса;
- POST — создание нового ресурса;
- PUT — полное обновление ресурса;
- PATCH — частичное обновление;
- DELETE — удаление ресурса.

Ресурсы представляются в виде существительных во множественном числе (`/orders`, `/orders/{id}`), а не глаголов, статус выполнения операции возвращается через HTTP-коды ответа (200, 201, 204, 400, 404, 500 и т.д.).

Важно отличать "RESTful" от "REST-like" или "просто HTTP API" — многие API называют себя REST, но не соблюдают все принципы (например, хранят состояние сессии на сервере или не поддерживают HATEOAS) — таких часто называют "RESTish" или "прагматичный REST".

## Пример / когда применяется

Пример RESTful-запросов для управления ресурсом "заказы":

```text
GET    /orders          -> получить список заказов
GET    /orders/42       -> получить заказ с id=42
POST   /orders          -> создать новый заказ (тело запроса содержит данные)
PUT    /orders/42       -> полностью обновить заказ 42
PATCH  /orders/42       -> частично обновить заказ 42 (например, только статус)
DELETE /orders/42       -> удалить заказ 42
```

Пример ответа с HATEOAS-ссылками:

```json
{
  "id": 42,
  "status": "processing",
  "links": [
    { "rel": "cancel", "href": "/orders/42/cancel" },
    { "rel": "customer", "href": "/customers/7" }
  ]
}
```

RESTful-подход применяется практически везде, где нужен простой, кэшируемый и масштабируемый API поверх HTTP: мобильные бэкенды, микросервисы, публичные API (GitHub API, Stripe API и т.д.).

## На что смотрит интервьюер

- Знает ли кандидат все 6 принципов REST, а не только "используются HTTP-методы".
- Понимает ли разницу между PUT и PATCH (полное vs частичное обновление) — частая путаница.
- Может ли объяснить принцип Stateless и почему он важен для масштабирования.
- Знает ли про HATEOAS — многие кандидаты забывают об этом принципе, хотя он часть "полноценного" REST (уровень 3 модели зрелости Ричардсона — Richardson Maturity Model).
- Красный флаг: кандидат говорит "REST — это просто JSON по HTTP" без понимания архитектурных ограничений.
