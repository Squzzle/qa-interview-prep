# Что такое SoapUI?

**Коротко:** SoapUI — инструмент для функционального, нагрузочного и security-тестирования веб-сервисов, изначально ориентированный на SOAP/WSDL, а позже расширенный поддержкой REST API.

## Развёрнутый ответ

SoapUI — это специализированный инструмент тестирования API, который позволяет создавать, отправлять и проверять запросы к веб-сервисам без написания кода на клиентской стороне (хотя поддерживается и скриптинг на Groovy). Изначально инструмент создавался для тестирования SOAP-сервисов: он умеет импортировать WSDL-описание сервиса и автоматически генерировать на его основе шаблоны запросов для всех доступных операций.

Основные возможности SoapUI:

- Импорт WSDL и автогенерация тестовых запросов (Mock-сервисы тоже можно создавать на основе WSDL).
- Поддержка REST API (импорт по URL, Swagger/OpenAPI-спецификации).
- Составление тест-кейсов (TestSuite → TestCase → TestStep) с проверками (Assertions): по содержимому ответа, по XPath/JSONPath, по времени отклика, по схеме (Schema Compliance), по кодам ответа.
- Возможность параметризации запросов через свойства (Properties) и передачи данных из внешних источников (Data Source Test Step, Excel/CSV/БД).
- Поддержка сценарного тестирования: цепочки запросов с передачей значений между шагами.
- Groovy Script Step — для написания произвольной логики, кастомных проверок, работы с переменными окружения.
- Нагрузочное тестирование (Load Tests) — запуск тест-кейсов в несколько потоков с заданной стратегией нагрузки.
- Security-тестирование — встроенные сканы на SQL-инъекции, XML-бомбы, некорректную обработку данных.
- Mock-сервисы — позволяют "заглушить" реальный сервис и эмулировать его ответы, что удобно для параллельной разработки и тестирования интеграций.

Существует бесплатная версия SoapUI Open Source и коммерческая ReadyAPI (с расширенными возможностями: интеграция с CI/CD, расширенное репортинг, поддержка команд).

## Пример / когда применяется

Типичный сценарий: есть SOAP-сервис получения данных о клиенте по ID. Тестировщик:

1. Импортирует WSDL сервиса в SoapUI — инструмент автоматически строит запросы для всех операций.
2. Заполняет тело запроса нужными параметрами (например, ID клиента).
3. Добавляет Assertion на соответствие XSD-схеме ответа и на конкретные значения полей.
4. Оборачивает запрос в TestCase и параметризует ID клиента через Data Source, чтобы прогнать несколько наборов данных.

Пример тела SOAP-запроса, который создаётся в SoapUI:

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
  <soapenv:Header/>
  <soapenv:Body>
    <cli:GetCustomer xmlns:cli="http://example.com/customer">
      <cli:CustomerId>12345</cli:CustomerId>
    </cli:GetCustomer>
  </soapenv:Body>
</soapenv:Envelope>
```

SoapUI применяется, когда нужно быстро протестировать SOAP/REST API без разработки автоматизации на языке программирования, а также как инструмент для ручной проверки бэкенда, mock-сервисов и нагрузочных тестов на ранних этапах.

## На что смотрит интервьюер

- Понимает ли кандидат разницу между SoapUI (в первую очередь для SOAP) и Postman (изначально для REST), и может ли объяснить, когда какой инструмент уместнее.
- Знает ли про Assertions, TestSuite/TestCase структуру, Mock-сервисы.
- Путают SoapUI с самим протоколом SOAP — важно чётко разграничивать инструмент и протокол.
- Может ли объяснить работу с WSDL и генерацию запросов на его основе.
- Красный флаг: кандидат утверждает, что SoapUI работает только с SOAP и не поддерживает REST — это устаревшее и неточное утверждение.
