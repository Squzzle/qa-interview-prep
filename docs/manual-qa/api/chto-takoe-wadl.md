# Что такое WADL?

**Коротко:** WADL (Web Application Description Language) — XML-формат для машиночитаемого описания REST API, аналог WSDL для SOAP, но менее популярный и сейчас практически вытесненный OpenAPI/Swagger.

## Развёрнутый ответ

WADL (Web Application Description Language) — это язык описания на основе XML, предложенный Sun Microsystems в 2009 году для формального описания HTTP-based (в первую очередь REST) веб-приложений и API. По аналогии с тем, как WSDL описывает SOAP-сервисы (операции, типы сообщений, привязки), WADL должен был стать стандартом для описания REST-ресурсов: их URI, поддерживаемых HTTP-методов, параметров запроса, форматов тела запроса/ответа.

Структура WADL-документа включает элементы:

- `<application>` — корневой элемент;
- `<resources>` — базовый URL и вложенные ресурсы;
- `<resource>` — конкретный ресурс с указанием path;
- `<method>` — HTTP-метод (GET, POST и т.д.), доступный для ресурса;
- `<request>`/`<response>` — описание параметров запроса и возможных ответов;
- `<param>` — параметры (query, path, header) с указанием типа и обязательности.

Несмотря на то, что WADL был стандартизирован и поддерживался некоторыми фреймворками (например, JAX-RS/Jersey могли автогенерировать WADL для своих REST-эндпоинтов), он так и не стал по-настоящему широко распространённым стандартом. Основные причины:

- REST по своей природе более гибок и менее формализован, чем SOAP, поэтому строгое XML-описание часто воспринималось избыточным;
- отсутствовала мощная экосистема инструментов вокруг WADL по сравнению с тем, что впоследствии появилось у OpenAPI/Swagger;
- сообщество разработчиков предпочло более лёгкие и удобочитаемые форматы — сначала Swagger (позже переименован в OpenAPI Specification), который использует JSON/YAML вместо XML и де-факто стал отраслевым стандартом описания REST API.

Сегодня WADL используется редко и в основном встречается в legacy-системах или в некоторых корпоративных Java-стеках (JAX-RS). Для новых проектов практически повсеместно применяется OpenAPI Specification (Swagger) или RAML.

## Пример / когда применяется

Пример фрагмента WADL, описывающего ресурс для получения информации о клиенте:

```xml
<application xmlns="http://wadl.dev.java.net/2009/02">
  <resources base="http://example.com/api">
    <resource path="/customers/{id}">
      <param name="id" style="template" type="xsd:int" required="true"/>
      <method name="GET">
        <response>
          <representation mediaType="application/json"/>
        </response>
      </method>
    </resource>
  </resources>
</application>
```

WADL применялся, когда нужно было автоматически сгенерировать клиентский код или документацию для REST API на Java-стеке (JAX-RS автоматически публикует WADL по адресу вида `/application.wadl`). На собеседовании эта тема чаще всего всплывает как сравнение с WSDL/OpenAPI, а не как активно используемая технология.

## На что смотрит интервьюер

- Понимает ли кандидат, что WADL — это исторический аналог WSDL для REST, и может провести аналогию SOAP+WSDL vs REST+WADL/OpenAPI.
- Знает ли, что WADL практически вытеснен OpenAPI/Swagger, и может объяснить почему (удобство, JSON/YAML, экосистема инструментов).
- Не путает ли WADL с WSDL по названию (частая ошибка из-за похожих аббревиатур).
- Красный флаг: кандидат утверждает, что WADL активно используется в современных проектах, не зная о доминировании OpenAPI.
