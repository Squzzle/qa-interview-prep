# Что такое WebHook?

**Коротко:** WebHook — это механизм обратного вызова, при котором сервер сам отправляет HTTP-запрос на заранее указанный клиентом URL при наступлении определённого события, вместо того чтобы клиент постоянно опрашивал сервер.

## Развёрнутый ответ

Обычно взаимодействие с API строится по модели "клиент инициирует запрос". WebHook переворачивает эту модель: клиент заранее регистрирует у сервера-поставщика URL своего собственного эндпоинта (часто называемого callback URL) и список событий, на которые он хочет подписаться. Когда на стороне сервера происходит нужное событие (например, оплата прошла успешно, статус доставки изменился, завершилась сборка в CI), сервер сам формирует HTTP-запрос (как правило, POST с телом в формате JSON) и отправляет его на зарегистрированный URL клиента.

Основные особенности и требования к реализации:

- **Клиент должен иметь публично доступный HTTP-эндпоинт**, который может принять входящий запрос от внешней системы (в отличие от polling/WebSocket, где клиент сам выступает инициатором).
- **Идемпотентность.** Поставщик webhook может доставить одно и то же событие несколько раз (например, из-за сетевых сбоев или ретраев), поэтому обработчик на стороне клиента должен уметь безопасно обрабатывать дубликаты — обычно по уникальному идентификатору события.
- **Подтверждение получения.** Обработчик должен быстро вернуть успешный HTTP-статус (обычно `200 OK`), иначе отправитель посчитает доставку неудачной и повторит попытку согласно своей retry-политике.
- **Безопасность.** Так как эндпоинт публичный, нужно проверять подлинность запроса — обычно через подпись (HMAC-подпись в заголовке, которую клиент проверяет с использованием общего секрета), а также использовать HTTPS.
- **Порядок доставки не гарантирован.** События могут прийти не в том порядке, в котором произошли, поэтому в теле события часто передают временную метку или порядковый номер.

WebHook широко применяется в интеграциях между сервисами: платёжные системы (Stripe, PayPal) уведомляют о статусе транзакции, CI/CD системы (GitHub, GitLab) уведомляют о событиях репозитория, мессенджеры и CRM уведомляют о новых сообщениях.

## Пример / когда применяется

Регистрация webhook в стороннем сервисе:

```json
POST /api/webhooks/subscribe
{
  "url": "https://myapp.example.com/webhooks/payments",
  "events": ["payment.succeeded", "payment.refunded"],
  "secret": "whsec_xxx"
}
```

Пример входящего webhook-запроса от платёжного сервиса:

```json
POST /webhooks/payments HTTP/1.1
Host: myapp.example.com
X-Signature: sha256=8f434346...

{
  "event": "payment.succeeded",
  "event_id": "evt_9f3a",
  "amount": 1500,
  "currency": "RUB",
  "order_id": "order_42"
}
```

Обработчик должен: проверить подпись `X-Signature`, проверить `event_id` на дубликат, обработать событие и вернуть `200 OK`.

## На что смотрит интервьюер

- Понимание, что инициатор запроса при webhook — сервер-поставщик, а не клиент (это главное отличие от polling).
- Знание практических рисков: необходимость идемпотентности, проверки подписи, обработки ретраев и таймаутов.
- Понимание, как тестировать webhook: эмуляция входящих событий, проверка обработки дубликатов, проверка сценария недоступности эндпоинта клиента и последующих ретраев.
- Красный флаг — если кандидат путает webhook с обычным REST-запросом от клиента или не понимает необходимости валидации подлинности входящих запросов.
