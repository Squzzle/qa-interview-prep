# Что такое XSD (XML Schema Definition) в контексте SOAP?

**Коротко:** XSD — язык описания структуры и типов данных XML-документа; в SOAP-сервисах XSD задаёт формат и типы элементов запросов/ответов, а WSDL ссылается на XSD для описания структуры сообщений.

## Развёрнутый ответ

XSD (XML Schema Definition) — это стандарт W3C для описания структуры, типов данных и ограничений XML-документов. XSD-документ сам является XML-файлом и определяет:

- какие элементы и атрибуты допустимы в документе;
- их порядок и вложенность;
- типы данных (строка, число, дата, перечисление и т.д.);
- ограничения (обязательность полей, минимальную/максимальную длину, регулярные выражения, диапазоны значений);
- пространства имён (namespaces), используемые в документе.

В контексте SOAP-веб-сервисов XSD играет ключевую роль: WSDL-описание сервиса (Web Services Description Language) содержит секцию `types`, где либо напрямую задаётся, либо импортируется XSD-схема, описывающая структуру всех сообщений (запросов и ответов), которыми обмениваются клиент и сервер. Таким образом, XSD выступает "контрактом данных" — он фиксирует, какие поля обязательны, какого они типа и в каком порядке должны идти.

Практическое значение XSD для тестирования:

- **Валидация ответа/запроса по схеме** — можно автоматически проверить, что XML-сообщение соответствует контракту (Schema Compliance), это стандартный вид Assertion в SoapUI/ReadyAPI.
- **Генерация тестовых данных** — зная типы и ограничения полей из XSD, можно генерировать корректные или намеренно некорректные (для негативных тестов) данные.
- **Обнаружение breaking changes** — при изменении контракта сервиса можно сравнивать версии XSD, чтобы понять, какие изменения обратно несовместимы (например, добавление обязательного поля).

Структура XSD включает такие элементы, как `xs:element`, `xs:complexType`, `xs:simpleType`, `xs:sequence`, `xs:attribute`, `xs:restriction`.

## Пример / когда применяется

Пример простого XSD, описывающего элемент запроса на получение клиента:

```xml
<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema"
           targetNamespace="http://example.com/customer"
           xmlns:tns="http://example.com/customer"
           elementFormDefault="qualified">

  <xs:element name="GetCustomerRequest">
    <xs:complexType>
      <xs:sequence>
        <xs:element name="CustomerId" type="xs:int" minOccurs="1"/>
        <xs:element name="IncludeOrders" type="xs:boolean" minOccurs="0"/>
      </xs:sequence>
    </xs:complexType>
  </xs:element>

</xs:schema>
```

Такая схема гарантирует, что `CustomerId` обязателен и должен быть целым числом, а `IncludeOrders` — необязательное булево поле. В SoapUI на основе этой схемы можно добавить Assertion "Schema Compliance", который автоматически провалит тест, если реальный ответ сервиса не соответствует описанной структуре (например, отсутствует обязательное поле или неверный тип данных).

## На что смотрит интервьюер

- Понимает ли кандидат, что XSD описывает структуру данных, а WSDL — описывает сам сервис (операции, порты, привязки) и ссылается на XSD для типов сообщений — частая путаница между WSDL и XSD.
- Знает ли, зачем нужна валидация по схеме в автотестах (Schema Compliance) и как это ускоряет обнаружение контрактных багов.
- Может ли привести примеры типов ограничений в XSD (minOccurs/maxOccurs, restriction, pattern).
- Красный флаг: кандидат путает XSD с XML или не понимает разницы между схемой и самим документом.
