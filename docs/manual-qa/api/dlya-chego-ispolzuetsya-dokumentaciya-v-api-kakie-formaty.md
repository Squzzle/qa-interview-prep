# Для чего используется документация в API? Какие форматы документации знаете (OpenAPI, WSDL)?

**Коротко:** Документация API описывает контракт взаимодействия — доступные операции, форматы запросов и ответов, коды ошибок — и нужна разработчикам, тестировщикам и интеграторам, чтобы работать с API без доступа к его исходному коду; основные форматы — OpenAPI (Swagger) для REST и WSDL для SOAP.

## Развёрнутый ответ

Документация API решает несколько задач:
- Даёт разработчикам клиентской стороны понимание, как формировать запросы и обрабатывать ответы, без изучения серверного кода.
- Служит источником правды (single source of truth) для тестировщиков при составлении тест-кейсов: какие поля обязательны, какие типы данных ожидаются, какие коды ошибок возможны.
- Позволяет автоматизировать генерацию клиентского кода, mock-серверов, тестов на основе спецификации.
- Ускоряет онбординг новых членов команды и интеграцию с внешними партнёрами.

Основные форматы документации:

- **OpenAPI Specification (ранее Swagger)** — машиночитаемое описание REST API в формате YAML или JSON: эндпоинты, методы, параметры, схемы тел запросов/ответов, коды статуса, схемы авторизации. На основе OpenAPI можно автоматически сгенерировать интерактивную документацию (Swagger UI), клиентские SDK, мок-серверы, и даже автотесты (контрактное тестирование).
- **WSDL (Web Services Description Language)** — XML-формат описания SOAP-сервисов: какие операции доступны, какие параметры и типы данных они принимают и возвращают, адрес сервиса (endpoint).
- **RAML (RESTful API Modeling Language)** — альтернативный формат описания REST API, менее распространён сегодня, чем OpenAPI.
- **API Blueprint** — ещё один формат описания REST API на основе Markdown.
- **Postman Collections** — не формальная спецификация, но часто используется как практическая "живая документация" с примерами реальных запросов.

Хорошая документация должна включать: описание аутентификации/авторизации, версионирование API, примеры запросов и ответов, описание ошибок и кодов статуса, лимиты (rate limiting), changelog изменений.

## Пример / когда применяется

Фрагмент простого OpenAPI-описания эндпоинта:

```yaml
paths:
  /users/{id}:
    get:
      summary: Получить пользователя по id
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: integer
      responses:
        '200':
          description: Успешный ответ
          content:
            application/json:
              schema:
                type: object
                properties:
                  id:
                    type: integer
                  name:
                    type: string
        '404':
          description: Пользователь не найден
```

На основе такого описания тестировщик сразу видит: путь `/users/{id}`, тип параметра `id`, возможные ответы 200 и 404, структуру тела ответа — и может составить тест-кейсы на позитивные и негативные сценарии (несуществующий id, нечисловой id и т.д.).

## На что смотрит интервьюер

- Знает ли кандидат разницу между OpenAPI (для REST) и WSDL (для SOAP).
- Понимает ли, зачем документация нужна именно тестировщику (не только разработчику).
- Может ли объяснить, как документация используется для контрактного тестирования и генерации моков.
- Follow-up: "что делать, если документация расходится с реальным поведением API" (эталонный ответ — заводить баг на расхождение, документация тоже часть продукта).
- Красный флаг: восприятие документации как формальности, а не рабочего инструмента тестирования.
