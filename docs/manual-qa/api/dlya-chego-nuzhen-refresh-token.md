# Для чего нужен Refresh Token?

**Коротко:** Refresh Token — это долгоживущий токен, который используется для получения нового Access Token без повторной аутентификации пользователя, когда срок действия текущего Access Token истёк.

## Развёрнутый ответ

В большинстве современных схем авторизации (особенно в OAuth 2.0) используется пара токенов:

- **Access Token** — короткоживущий токен (обычно от нескольких минут до пары часов), который передаётся с каждым запросом к защищённому API для подтверждения прав доступа.
- **Refresh Token** — долгоживущий токен (дни, недели или месяцы), который сам по себе НЕ используется для доступа к ресурсам API, а только для получения нового Access Token, когда предыдущий истёк.

Зачем нужно такое разделение:
1. **Безопасность.** Access Token живёт недолго, поэтому если он будет перехвачен злоумышленником, окно для его использования ограничено. Refresh Token хранится более защищённо (например, в httpOnly cookie или secure storage) и передаётся значительно реже — только на эндпоинт обновления токена, а не на каждый обычный запрос к API.
2. **Удобство пользователя (UX).** Без refresh-механизма пользователю пришлось бы заново вводить логин и пароль каждый раз, когда истекает Access Token. Refresh Token позволяет продлевать сессию "невидимо" для пользователя, пока сам Refresh Token действителен.
3. **Возможность отзыва доступа.** Так как для получения нового Access Token клиент обязан обратиться к серверу с Refresh Token, сервер может проверить в этот момент, не был ли Refresh Token отозван (например, пользователь вышел из аккаунта на всех устройствах, сменил пароль, администратор заблокировал сессию), и отказать в выдаче нового Access Token.
4. **Ротация токенов (Refresh Token Rotation).** Продвинутая практика безопасности: при каждом обновлении Access Token сервер выдаёт также новый Refresh Token и инвалидирует старый. Если старый Refresh Token будет использован повторно (признак кражи), сервер может заподозрить компрометацию и отозвать всю цепочку токенов.

Типичный флоу: клиент делает запрос к API с Access Token → получает 401 Unauthorized (токен истёк) → клиент отправляет Refresh Token на специальный эндпоинт (например, `/oauth/token` с `grant_type=refresh_token`) → получает новый Access Token (и, возможно, новый Refresh Token) → повторяет исходный запрос уже с новым Access Token.

## Пример / когда применяется

Пример запроса обновления токена:

```
POST /oauth/token HTTP/1.1
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&refresh_token=def502008a3f...&client_id=my-app
```

Пример ответа:

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "def502009b4e...",
  "expires_in": 3600,
  "token_type": "Bearer"
}
```

Такой механизм применяется практически во всех продуктах с длительными пользовательскими сессиями: мобильные приложения (пользователь не хочет логиниться каждый час), SPA-приложения, интеграции через OAuth 2.0 с внешними провайдерами (вход через Google/GitHub).

## На что смотрит интервьюер

- Понимает ли кандидат разницу в назначении и времени жизни Access и Refresh токенов.
- Знает ли, зачем вообще нужно разделение (а не выдавать сразу один долгоживущий токен) — ключевой ответ про безопасность и возможность отзыва.
- Знает ли о ротации Refresh Token как практике повышения безопасности.
- Follow-up: "что должен вернуть сервер, если Refresh Token истёк или отозван" (ответ: 401, пользователь должен пройти повторный логин), "как бы вы протестировали сценарий одновременного использования одного и того же Refresh Token дважды".
- Красный флаг: путает Refresh Token с Access Token или считает, что Refresh Token тоже используется напрямую для доступа к данным API.
