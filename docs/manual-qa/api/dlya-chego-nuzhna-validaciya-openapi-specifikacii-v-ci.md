# Для чего нужна валидация OpenAPI спецификации в CI/CD?

**Коротко:** Валидация OpenAPI-спецификации в CI/CD автоматически проверяет, что документация API синтаксически корректна, соответствует стандартам и не разошлась с реальным поведением сервиса, предотвращая поставку сломанной или устаревшей документации и breaking changes для потребителей API.

## Развёрнутый ответ

OpenAPI-спецификация (ранее Swagger) — это машиночитаемое описание контракта API: эндпоинты, методы, параметры, форматы запросов и ответов, коды ошибок. Она используется не только как документация для людей, но и как источник для генерации клиентских SDK, серверных заглушек (mock-серверов), автотестов контракта и интерактивной документации (Swagger UI, Redoc).

Если спецификация содержит ошибки или расходится с реальным поведением API, это приводит к серьёзным проблемам:

- Разработчики, интегрирующиеся с API по документации, получают некорректные ожидания и пишут неработающий код.
- Автоматически сгенерированные клиенты/тесты на основе спецификации ломаются или дают ложные результаты.
- Breaking changes (несовместимые изменения контракта) могут попасть в продакшен незамеченными и сломать интеграции у клиентов API.

Поэтому валидацию спецификации встраивают в CI/CD pipeline как обязательный шаг, который выполняется автоматически при каждом изменении кода или самой спецификации. Валидация обычно включает несколько уровней:

1. **Синтаксическая валидация** — проверка, что файл соответствует формату OpenAPI/YAML/JSON schema (нет опечаток, соблюдена структура).
2. **Проверка стиля и best practices (linting)** — например, что все эндпоинты имеют описания, коды ошибок задокументированы, используются согласованные соглашения об именовании (это делают такие инструменты, как Spectral).
3. **Contract testing** — проверка, что реальные ответы API соответствуют описанному контракту (это делают инструменты вроде Dredd, Schemathesis, Pact).
4. **Diff/breaking change detection** — сравнение текущей версии спецификации с предыдущей, чтобы обнаружить несовместимые изменения (удалённые поля, изменённые типы данных, обязательные новые параметры) до релиза.

Встраивание этих проверок в CI позволяет "падать" сборке при обнаружении проблем ещё до мержа в основную ветку, а не узнавать о рассинхронизации документации и кода постфактум от разгневанных потребителей API.

## Пример / когда применяется

Пример шага в CI-пайплайне (псевдо-YAML):

```yaml
jobs:
  validate-api-spec:
    steps:
      - name: Lint OpenAPI spec
        run: npx @stoplight/spectral-cli lint openapi.yaml
      - name: Contract test against running service
        run: npx dredd openapi.yaml http://localhost:3000
```

Такая проверка применяется при каждом Pull Request, затрагивающем API-контракт или сам файл спецификации, чтобы не допустить слияние несовместимых изменений.

## На что смотрит интервьюер

- Понимание разницы между "спецификация просто существует" и "спецификация реально соответствует поведению сервиса".
- Знание, что валидация в CI ловит проблемы на раннем этапе, а не после релиза.
- Способность назвать конкретные категории проверок: синтаксис, стиль (linting), контрактные тесты, breaking changes.
- Понимание последствий для потребителей API при рассинхронизации документации и реализации.
- Красный флаг — если кандидат считает, что OpenAPI-спецификация нужна "только для документации" и не связывает её с автотестами и генерацией кода.
