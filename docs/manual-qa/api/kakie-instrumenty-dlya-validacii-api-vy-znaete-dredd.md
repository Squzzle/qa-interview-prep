# Какие инструменты для валидации API вы знаете (Dredd, Spectral)?

**Коротко:** Наиболее известные инструменты — Dredd и Schemathesis (контрактное тестирование API по спецификации), Spectral (линтер OpenAPI-спецификаций), Pact (contract testing между потребителем и поставщиком), а также встроенные валидаторы Swagger/OpenAPI (Swagger Validator, Redocly CLI).

## Развёрнутый ответ

Инструменты для валидации API можно условно разделить на несколько категорий по тому, что именно они проверяют.

**Линтеры спецификации (проверка самой OpenAPI-документации на корректность и стиль):**
- **Spectral** — самый популярный линтер для OpenAPI/AsyncAPI. Проверяет структуру спецификации на соответствие правилам: обязательные описания, согласованность именования, наличие кодов ошибок и т.д. Правила настраиваются (ruleset), можно писать свои.
- **Redocly CLI** — похожий инструмент от команды Redoc, тоже умеет линтить и подсвечивать проблемы в спецификации, а также строить документацию.

**Инструменты контрактного тестирования (проверка, что реальный API соответствует описанной спецификации):**
- **Dredd** — запускает реальные HTTP-запросы к работающему сервису на основе примеров из OpenAPI/API Blueprint спецификации и сверяет фактические ответы с ожидаемыми по контракту.
- **Schemathesis** — инструмент на базе property-based тестирования: генерирует множество тестовых запросов на основе схемы (в том числе граничные и невалидные случаи) и проверяет, что ответы сервера соответствуют объявленным схемам и статусам.
- **Pact** — реализует подход "consumer-driven contract testing": потребитель API описывает ожидания от поставщика в виде контракта, а затем этот контракт проверяется на стороне поставщика в CI, гарантируя совместимость без необходимости поднимать интеграционное окружение целиком.

**Инструменты для валидации отдельных запросов/ответов во время ручного или автотестирования:**
- Валидация JSON Schema в тестах (библиотеки вроде `ajv` для JS, `jsonschema` для Python) — проверка, что фактический ответ соответствует ожидаемой схеме данных.
- Postman/Newman со встроенными скриптами тестов (`pm.test`), которые могут проверять и структуру, и статус-коды ответа.

Использование этих инструментов в связке даёт полное покрытие: Spectral гарантирует, что сама спецификация написана правильно и по стандартам, а Dredd/Schemathesis/Pact гарантируют, что реализация API реально соответствует этой спецификации.

## Пример / когда применяется

Пример запуска Spectral для линтинга спецификации:

```bash
npx @stoplight/spectral-cli lint openapi.yaml
```

Пример запуска Dredd для контрактного тестирования запущенного сервиса:

```bash
dredd openapi.yaml http://localhost:3000 --hookfiles=./hooks.js
```

Пример property-based тестирования со Schemathesis:

```bash
schemathesis run --url http://localhost:3000 openapi.yaml
```

Такие проверки обычно включают в pipeline CI/CD на этапе Pull Request или перед деплоем, чтобы не допустить рассинхронизации контракта и реализации.

## На что смотрит интервьюер

- Умение назвать хотя бы 2-3 инструмента и объяснить, чем они отличаются друг от друга (линтинг vs контрактное тестирование).
- Понимание разницы между статической проверкой самой спецификации (Spectral) и динамической проверкой соответствия реального API этой спецификации (Dredd, Schemathesis).
- Знание концепции consumer-driven contract testing (Pact) как более продвинутого подхода для микросервисных архитектур.
- Красный флаг — если кандидат называет инструменты, но не может объяснить, что конкретно каждый из них делает.
