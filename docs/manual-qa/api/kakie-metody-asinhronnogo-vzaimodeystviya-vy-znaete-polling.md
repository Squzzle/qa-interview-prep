# Какие методы асинхронного взаимодействия вы знаете (Polling, WebSocket, WebHooks)?

**Коротко:** Основные методы — Polling (периодический опрос сервера), Long Polling, WebSocket (постоянное двустороннее соединение) и WebHooks (сервер сам уведомляет клиента через HTTP-callback).

## Развёрнутый ответ

**Polling (короткий опрос).** Клиент с определённой периодичностью (например, раз в 2-5 секунд) отправляет запрос на сервер, чтобы проверить, не изменилось ли состояние или не готов ли результат. Просто в реализации, но неэффективно: создаёт лишнюю нагрузку на сеть и сервер, а также вносит задержку между реальным появлением результата и его получением клиентом (задержка равна интервалу опроса).

**Long Polling (длинный опрос).** Модификация обычного polling: клиент отправляет запрос, а сервер не отвечает сразу, а держит соединение открытым до тех пор, пока не появится новое событие/данные (или не истечёт таймаут), и только тогда отвечает. После получения ответа клиент сразу отправляет новый запрос. Это снижает количество "пустых" ответов по сравнению с обычным polling, но всё ещё требует постоянного пересоздания соединений.

**WebSocket.** Протокол, устанавливающий постоянное двустороннее (full-duplex) соединение поверх TCP между клиентом и сервером после начального HTTP-хендшейка (`Upgrade: websocket`). После установления соединения обе стороны могут отправлять сообщения друг другу в любой момент без необходимости повторных запросов. Подходит для реального времени: чаты, биржевые котировки, онлайн-игры, live-уведомления. Требует поддержки на обеих сторонах и более сложной инфраструктуры (балансировка stateful-соединений).

**WebHooks.** Механизм, при котором сервер сам инициирует HTTP-запрос (обычно POST) на заранее зарегистрированный клиентом URL, когда происходит определённое событие. Клиенту не нужно ничего опрашивать — он просто поднимает эндпоинт-приёмник и ждёт входящих вызовов. Часто используется для интеграций между системами (например, уведомление от платёжного шлюза об изменении статуса платежа). Требует, чтобы клиент имел публично доступный URL и корректно обрабатывал возможные повторные доставки (идемпотентность), а также проверял подлинность запроса (например, через подпись в заголовке).

Также существуют смежные технологии: **Server-Sent Events (SSE)** — однонаправленный канал от сервера к клиенту поверх HTTP, проще WebSocket, но без обратного канала от клиента; **очереди сообщений** (Kafka, RabbitMQ) — используются внутри распределённых систем для асинхронного обмена между сервисами.

## Пример / когда применяется

Сравнение подходов:

| Метод | Кто инициирует | Задержка | Нагрузка на сеть | Типичный кейс |
|---|---|---|---|---|
| Polling | Клиент, регулярно | Средняя/высокая | Высокая | Простая проверка статуса задачи |
| Long Polling | Клиент, но сервер держит соединение | Низкая | Средняя | Чат без WebSocket |
| WebSocket | Обе стороны | Минимальная | Низкая (одно соединение) | Реальное время, чаты, торговые терминалы |
| WebHook | Сервер | Минимальная (по факту события) | Низкая | Уведомления от внешних сервисов (платежи, CI/CD) |

Пример регистрации webhook при создании подписки:

```json
POST /api/webhooks
{
  "url": "https://client.example.com/webhook-handler",
  "events": ["payment.succeeded", "payment.failed"]
}
```

## На что смотрит интервьюер

- Умение чётко разграничить polling, long polling, WebSocket и webhook — а не просто перечислить слова.
- Понимание, кто инициирует соединение/запрос в каждом случае (это ключевое отличие webhook от остальных).
- Знание практических нюансов webhook: необходимость идемпотентной обработки повторных доставок, проверки подписи запроса, обработки недоступности приёмника (retry-политика на стороне отправителя).
- Способность привести реальный пример использования каждого метода.
- Красный флаг — если кандидат считает webhook и polling одним и тем же, либо не может объяснить, почему WebSocket эффективнее обычного polling.
