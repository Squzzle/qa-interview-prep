# Какие параметры обычно используются для пагинации (limit, offset, page)?

**Коротко:** Чаще всего встречаются пары `limit`/`offset`, `page`/`per_page` (или `page_size`) и параметры курсорной пагинации `cursor`/`next_cursor`, `after`/`before`.

## Развёрнутый ответ

В зависимости от подхода к пагинации используются разные наборы query-параметров:

- **`limit` и `offset`** — классический подход, пришедший из SQL. `limit` задаёт максимальное количество записей в ответе, `offset` — сколько записей пропустить от начала выборки. Например, `limit=20&offset=40` вернёт записи с 41 по 60.
- **`page` и `per_page` (или `page_size`, `size`)** — более человекопонятный вариант: `page` — номер страницы (обычно нумерация с 1), `per_page` — количество элементов на странице. Внутри сервер обычно транслирует это в `offset = (page - 1) * per_page`.
- **Курсорная пагинация: `cursor`, `after`, `before`, `next`/`prev`** — вместо числового смещения передаётся непрозрачный токен (курсор), который указывает на позицию в наборе данных относительно последнего полученного элемента. Сервер в ответе возвращает `next_cursor` для получения следующей порции.
- **Дополнительные параметры**, которые часто идут вместе с пагинацией: `sort`/`order_by` (сортировка, важно для консистентности пагинации), `total_count` в ответе (общее количество записей), `has_more`/`has_next` (флаг наличия следующей страницы).

Важный нюанс с точки зрения тестирования и проектирования API: без стабильной сортировки (`ORDER BY` с уникальным полем) offset-пагинация может возвращать дубли или пропускать записи при параллельных изменениях данных. Поэтому на бэкенде обычно добавляют сортировку по уникальному полю (id, created_at + id) как tie-breaker.

Также стоит учитывать ограничения: сервер обычно задаёт максимальное значение `limit` (например, не больше 100), чтобы клиент не мог одним запросом выгрузить всю базу данных.

## Пример / когда применяется

Offset-based запрос:

```
GET /api/users?limit=20&offset=60
```

Page-based запрос:

```
GET /api/users?page=4&per_page=20
```

Курсорный запрос:

```
GET /api/users?limit=20&after=eyJpZCI6MjAwfQ==
```

Пример ответа с курсором:

```json
{
  "data": [ { "id": 221 }, { "id": 222 } ],
  "next_cursor": "eyJpZCI6MjIyfQ==",
  "has_more": true
}
```

## На что смотрит интервьюер

- Знает ли кандидат разницу между `page`/`per_page` и `limit`/`offset` (по сути одно и то же, но разный уровень абстракции).
- Понимает ли проблему нестабильной сортировки и дублирования/пропуска записей при пагинации на "живых" данных.
- Проверяет ли кандидат в тестах граничные значения: `limit=0`, отрицательные значения, `offset` больше общего количества записей, отсутствие параметров (значения по умолчанию), превышение максимального лимита.
- Красный флаг — если кандидат путает `offset` и `page`, не понимая, что это разные единицы измерения (записи vs страницы).
