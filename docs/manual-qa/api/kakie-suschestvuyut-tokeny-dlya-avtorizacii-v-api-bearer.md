# Какие существуют токены для авторизации в API (Bearer, JWT, OAuth)?

**Коротко:** Для авторизации в API чаще всего используются Bearer-токены (простой токен доступа в заголовке Authorization), JWT (самодостаточный подписанный токен) и токены, выдаваемые по протоколу OAuth 2.0 (Access Token / Refresh Token), а также API-ключи и Basic Auth для более простых случаев.

## Развёрнутый ответ

Стоит различать понятия "тип авторизации/протокол" и "формат самого токена":

- **Basic Auth** — простейший способ, логин и пароль передаются в заголовке `Authorization: Basic base64(login:password)` при каждом запросе. Небезопасно без HTTPS, редко используется в продакшене.
- **API Key** — статичный ключ, выдаваемый клиенту заранее, передаётся в заголовке (например, `X-API-Key`) или query-параметре. Простой, но плохо подходит для персональной авторизации пользователей — обычно используется для идентификации приложения/партнёра.
- **Bearer Token** — общий термин: токен передаётся в заголовке `Authorization: Bearer <token>`. Сам токен может быть произвольной строкой (opaque token, значение которой ничего не говорит клиенту, проверяется только на сервере) либо структурированным, например JWT.
- **JWT (JSON Web Token)** — самодостаточный (self-contained) токен, содержащий закодированные данные о пользователе и подпись, позволяющую серверу проверить его подлинность без обращения к базе данных на каждый запрос.
- **OAuth 2.0** — это протокол авторизации (framework), а не тип токена как таковой. Он описывает, как клиент получает Access Token (и опционально Refresh Token) от сервера авторизации, используя разные "grant types" (Authorization Code, Client Credentials, Password Grant — устаревший, Implicit — устаревший). Полученный Access Token затем передаётся как Bearer-токен в запросах к ресурсному серверу.
- **OpenID Connect (OIDC)** — надстройка над OAuth 2.0, добавляющая ID Token (тоже JWT) специально для аутентификации (подтверждения личности пользователя), в отличие от OAuth, который решает задачу авторизации (выдачи прав доступа).

Важно понимать разницу между аутентификацией (кто ты) и авторизацией (что тебе разрешено) — OAuth 2.0 изначально протокол именно авторизации, а не аутентификации, хотя на практике его часто используют и для того, и для другого через OIDC.

## Пример / когда применяется

Пример передачи Bearer-токена в заголовке HTTP-запроса:

```
GET /api/orders HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0In0.abc123signature
```

Типичный сценарий с OAuth 2.0 (Authorization Code Flow):
1. Пользователь логинится через сервер авторизации (например, вход через Google).
2. Сервер авторизации возвращает клиенту authorization code.
3. Клиент обменивает code на Access Token и Refresh Token через сервер авторизации.
4. Клиент использует Access Token как Bearer-токен для запросов к API.
5. Когда Access Token истекает, клиент использует Refresh Token, чтобы получить новый Access Token без повторного логина пользователя.

## На что смотрит интервьюер

- Разделяет ли кандидат понятия "протокол авторизации" (OAuth) и "формат токена" (JWT) — частая ошибка ставить их в один ряд как равнозначные вещи.
- Знает ли, что Bearer — это способ передачи токена, а не конкретный формат.
- Может ли объяснить разницу между аутентификацией и авторизацией.
- Follow-up: "как бы вы протестировали API с истёкшим/невалидным токеном", "что вернёт сервер при отсутствии заголовка Authorization".
- Красный флаг: путает Bearer/JWT/OAuth как три взаимоисключающих альтернативы, не понимая, что они работают на разных уровнях и часто используются вместе.
