# Какие виды API вы знаете (по архитектуре/протоколу)?

**Коротко:** Основные виды веб-API — REST, SOAP, GraphQL, gRPC, а также WebSocket API для потоковой/событийной передачи данных; они отличаются протоколом, форматом данных и стилем взаимодействия.

## Развёрнутый ответ

По архитектурному стилю и протоколу API можно разделить на несколько групп:

- **REST (Representational State Transfer)** — архитектурный стиль поверх HTTP, использует стандартные методы (GET, POST, PUT, PATCH, DELETE), данные обычно передаются в формате JSON, реже XML. Самый распространённый вид API в вебе на сегодня.
- **SOAP (Simple Object Access Protocol)** — протокол обмена структурированными сообщениями в формате XML, работает поверх HTTP, SMTP и других транспортов, имеет строгую спецификацию (WSDL) и встроенную поддержку транзакций, безопасности (WS-Security).
- **GraphQL** — язык запросов и рантайм для API, позволяет клиенту самому описывать, какие поля данных ему нужны, в отличие от REST, где набор полей фиксирован сервером.
- **gRPC** — высокопроизводительный RPC-фреймворк от Google, использует HTTP/2 и бинарный формат Protocol Buffers (protobuf), часто применяется во внутренних микросервисных коммуникациях.
- **WebSocket API** — обеспечивает постоянное двунаправленное соединение между клиентом и сервером для передачи данных в реальном времени (чаты, биржевые котировки, онлайн-игры).
- **RPC (Remote Procedure Call)** в широком смысле, включая XML-RPC, JSON-RPC — вызов удалённых процедур как будто локальных функций.

По степени открытости API также делят на: публичные (открытые всем), партнёрские (доступны определённым партнёрам по соглашению) и внутренние/приватные (используются только внутри компании между собственными сервисами).

## Пример / когда применяется

| Вид API | Формат данных | Транспорт | Типичное применение |
|---|---|---|---|
| REST | JSON/XML | HTTP | Публичные веб-сервисы, мобильные бэкенды |
| SOAP | XML | HTTP, SMTP | Банковские, корпоративные, госсистемы |
| GraphQL | JSON | HTTP | Гибкая выборка данных, агрегация из нескольких источников |
| gRPC | Protobuf (бинарный) | HTTP/2 | Микросервисы внутри компании |
| WebSocket | JSON/бинарный | TCP (постоянное соединение) | Чаты, real-time уведомления |

Например, интернет-магазин может использовать REST API для взаимодействия мобильного приложения с сервером, GraphQL — для агрегации данных на странице товара из нескольких сервисов (цена, отзывы, наличие), а WebSocket — для отслеживания статуса заказа в реальном времени.

## На что смотрит интервьюер

- Знает ли кандидат больше, чем "REST и SOAP", то есть ориентируется ли в современном стеке (GraphQL, gRPC).
- Может ли объяснить принципиальную разницу, а не просто перечислить названия.
- Понимает ли, когда какой вид API уместнее применять.
- Красный флаг: путает протокол (HTTP) с архитектурным стилем (REST) или считает SOAP устаревшим и "неправильным" без понимания его сфер применения (финтех, госсектор).
