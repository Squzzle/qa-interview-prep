# Какова иерархия (приоритет) переменных в Postman?

**Коротко:** От наивысшего к низшему приоритету: локальные переменные (Local) → переменные окружения (Environment) → коллекционные переменные (Collection) → глобальные переменные (Global); при совпадении имён используется значение переменной с более высоким приоритетом.

## Развёрнутый ответ

Postman позволяет задавать переменные на нескольких уровнях, чтобы гибко управлять конфигурацией запросов (URL, токены, тестовые данные) без дублирования значений в каждом запросе. Уровни, от самого узкого (и приоритетного) к самому широкому:

1. **Local variables (локальные переменные)** — существуют только в рамках выполнения одного запроса или скрипта (задаются через `pm.variables.set()` в pre-request/test-скрипте) и не сохраняются после завершения запроса. Имеют наивысший приоритет.
2. **Environment variables (переменные окружения)** — привязаны к конкретному выбранному окружению (например, "Dev", "Staging", "Production"). Позволяют переключаться между разными наборами конфигурации одной и той же коллекции.
3. **Collection variables (переменные коллекции)** — доступны для всех запросов внутри конкретной коллекции, независимо от выбранного окружения. Полезны для значений, общих для коллекции, но не зависящих от окружения.
4. **Global variables (глобальные переменные)** — доступны во всех коллекциях и окружениях в рамках рабочего пространства Postman. Имеют наименьший приоритет.

Если одна и та же переменная (по имени) определена сразу на нескольких уровнях, Postman подставит значение с наивысшим приоритетом — то есть значение из более "узкой" области видимости перекрывает значение из более "широкой". Это позволяет, например, временно переопределить значение переменной окружения локальной переменной прямо в pre-request скрипте для конкретного теста, не трогая общую конфигурацию.

Понимание этой иерархии критично при отладке: если запрос использует "не то" значение переменной, нужно проверять её значение на каждом уровне сверху вниз (local → environment → collection → global), чтобы найти, где именно задано перекрывающее значение.

## Пример / когда применяется

Пример проверки итогового значения переменной в скрипте:

```javascript
// Получить значение с учётом всей иерархии (то, что реально будет подставлено)
const baseUrl = pm.variables.get("base_url");
console.log(baseUrl);
```

Пример намеренного переопределения в pre-request скрипте:

```javascript
// Локально переопределяем токен только для этого запроса,
// не изменяя переменную окружения
pm.variables.set("auth_token", "temporary-debug-token");
```

Иерархия применяется, когда нужно, например, использовать общий `base_url` для всей коллекции (Collection variable), но для конкретного окружения (Dev/Staging) переопределить его через Environment variable, а для одного отладочного запроса — временно подменить локальной переменной.

## На что смотрит интервьюер

- Точное знание порядка приоритета: Local > Environment > Collection > Global.
- Понимание, что переменные с одинаковым именем на разных уровнях не конфликтуют аварийно, а просто перекрываются по приоритету.
- Умение объяснить практический сценарий использования каждого уровня.
- Красный флаг — если кандидат путает порядок приоритета или не знает о существовании коллекционных переменных как отдельного уровня.
