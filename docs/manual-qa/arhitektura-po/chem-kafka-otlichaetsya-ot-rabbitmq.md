# Чем Kafka отличается от RabbitMQ?

**Коротко:** Kafka — это распределённый упорядоченный лог событий с высокой пропускной способностью и возможностью повторного чтения истории, а RabbitMQ — классический брокер очередей с гибкой маршрутизацией и удалением сообщения после подтверждённой обработки.

## Развёрнутый ответ

Kafka и RabbitMQ решают схожую задачу — асинхронный обмен сообщениями между сервисами, но построены на принципиально разных архитектурных моделях.

**Модель хранения данных.** Kafka хранит сообщения как упорядоченный, неизменяемый (append-only) лог на диске в течение настраиваемого времени (retention), независимо от того, прочитаны они или нет. Консьюмер сам управляет своей позицией (offset) и может перечитать историю заново. RabbitMQ, напротив, удаляет сообщение из очереди сразу после того, как получатель подтвердил (acknowledged) его обработку — модель классической очереди "получил и удалил".

**Маршрутизация.** RabbitMQ реализует богатую модель маршрутизации через exchange (direct, topic, fanout, headers), что позволяет гибко направлять сообщения в разные очереди по сложным правилам. В Kafka маршрутизация проще: сообщение публикуется в топик, а распределение по партициям происходит по ключу сообщения; сложная маршрутизация на уровне брокера отсутствует.

**Пропускная способность и масштабируемость.** Kafka спроектирована для очень высокой пропускной способности (миллионы сообщений в секунду) за счёт последовательной записи на диск и партиционирования. RabbitMQ обычно показывает меньшую пропускную способность, но обеспечивает более тонкий контроль доставки (приоритеты сообщений, TTL, отложенная доставка).

**Модель потребления.** В Kafka несколько независимых consumer group могут читать один и тот же топик параллельно и независимо, каждая со своим offset. В RabbitMQ сообщение из очереди обычно получает только один потребитель (если не настроен fanout-exchange с несколькими очередями), после чего сообщение удаляется.

**Порядок сообщений.** Kafka гарантирует порядок в пределах одной партиции. RabbitMQ гарантирует порядок в пределах одной очереди при одном потребителе.

**Гарантии доставки и повторная обработка.** Kafka по умолчанию поддерживает at-least-once с возможностью настройки exactly-once semantics (через транзакции продюсера). RabbitMQ также поддерживает at-least-once через ack/nack и dead-letter очереди.

**Типичные сценарии использования.** Kafka чаще выбирают для потоковой аналитики, event sourcing, журналирования событий, интеграции больших объёмов данных между множеством систем. RabbitMQ чаще выбирают для классических задач очередей задач (task queue), RPC-подобных сценариев и когда важна сложная маршрутизация сообщений.

## Пример / когда применяется

Kafka — платформа сбора логов и метрик со всех микросервисов компании, где нужно, чтобы несколько разных систем (мониторинг, аналитика, архивирование) независимо читали один и тот же поток событий, и была возможность переиграть историю при добавлении нового потребителя.

RabbitMQ — очередь задач для генерации PDF-документов: клиент кладёт задачу в очередь, один из свободных воркеров забирает задачу, обрабатывает и подтверждает (ack); если воркер упал — задача возвращается в очередь для другого воркера.

| Критерий | Kafka | RabbitMQ |
|---|---|---|
| Модель | Распределённый лог событий | Очередь сообщений |
| Хранение после чтения | Сохраняется (retention) | Удаляется после ack |
| Маршрутизация | Простая (топик + партиция по ключу) | Гибкая (exchange: direct/topic/fanout) |
| Пропускная способность | Очень высокая | Средняя/высокая |
| Множественные независимые читатели | Да (consumer groups) | Нужны отдельные очереди на fanout |

## На что смотрит интервьюер

Интервьюер проверяет, может ли кандидат аргументированно выбрать инструмент под конкретную задачу, а не просто перечислить факты. Частая ошибка — утверждать, что "Kafka всегда лучше" или "RabbitMQ устарел" без понимания компромиссов. Follow-up вопросы: "когда вы бы выбрали RabbitMQ вместо Kafka?", "как в Kafka реализовать приоритет сообщений, которого нет из коробки?", "что такое exactly-once semantics и как она достигается в Kafka?". Красный флаг — незнание, что в Kafka сообщение не удаляется после чтения, а в RabbitMQ обычно удаляется.
