# Чем отличается L4 балансировка от L7?

**Коротко:** L4-балансировка работает на транспортном уровне (TCP/UDP), принимая решения по IP-адресу и порту, а L7-балансировка работает на прикладном уровне (HTTP/HTTPS), анализируя содержимое запроса — URL, заголовки, cookies.

## Развёрнутый ответ

L4 (Layer 4, транспортный уровень модели OSI) — балансировщик работает с TCP/UDP соединениями. Он видит только IP-адреса источника/назначения и порты, не заглядывая внутрь содержимого пакета (payload). Решение о том, на какой backend направить соединение, принимается один раз при установлении TCP-соединения, дальше все пакеты этого соединения идут на один и тот же сервер. L4-балансировка быстрее и менее ресурсоёмка, так как не требует разбора прикладного протокола, но она не может принимать решения на основе содержимого HTTP-запроса (URL, заголовков, cookies).

L7 (Layer 7, прикладной уровень) — балансировщик работает на уровне приложения и понимает конкретный протокол (чаще всего HTTP/HTTPS, а также gRPC, WebSocket). Он способен анализировать содержимое запроса: URL-путь, HTTP-заголовки, cookies, метод запроса, — и на основе этого принимать более гибкие решения о маршрутизации. Например, направлять запросы `/api/*` на один пул серверов, а `/static/*` — на другой; выполнять SSL-терминацию, сжатие, кэширование, добавление/изменение заголовков, A/B-тестирование по cookie. L7 требует больше вычислительных ресурсов (разбор протокола, возможно расшифровка TLS), но даёт значительно больше гибкости.

Ключевые отличия:

- Уровень OSI: L4 — транспортный (TCP/UDP), L7 — прикладной (HTTP и др.).
- Видимость данных: L4 не видит содержимое запроса, L7 видит полностью (URL, заголовки, тело при необходимости).
- Гибкость маршрутизации: L4 — только по IP/порту, L7 — по контенту запроса (path-based, host-based routing).
- Производительность: L4 быстрее и легче по ресурсам, L7 медленнее из-за разбора протокола, но современные реализации (Envoy, Nginx) достаточно производительны.
- Возможности: L7 умеет SSL-терминацию, сжатие контента, кэширование ответов, канареечные (canary) релизы по заголовкам, L4 этого не умеет.

## Пример / когда применяется

Пример L4: балансировка TCP-соединений к базе данных или к произвольному TCP-сервису, где нет понятия HTTP-запроса (например, AWS Network Load Balancer).

Пример L7: балансировка HTTP-трафика веб-приложения, где нужно направить запросы `/api/v1/orders` на сервис заказов, а `/api/v1/users` — на сервис пользователей, используя один и тот же домен и балансировщик (например, Nginx, HAProxy в режиме L7, AWS Application Load Balancer, Envoy).

| Критерий | L4 | L7 |
|---|---|---|
| Уровень OSI | Транспортный | Прикладной |
| Видит содержимое запроса | Нет | Да |
| Маршрутизация по URL/заголовкам | Невозможна | Возможна |
| Производительность | Выше | Ниже (но приемлема) |
| SSL-терминация | Обычно нет | Да |

## На что смотрит интервьюер

Интервьюер проверяет понимание модели OSI и практическое понимание, зачем нужен каждый тип балансировки. Частая ошибка — путать L7 с "более продвинутым L4" без понимания, что L7 буквально работает с содержимым HTTP-запроса. Follow-up: "Какой тип балансировщика вы бы выбрали для микросервисов с path-based routing?", "Может ли L7-балансировщик выполнять SSL-терминацию, а L4 — нет? Почему?".
