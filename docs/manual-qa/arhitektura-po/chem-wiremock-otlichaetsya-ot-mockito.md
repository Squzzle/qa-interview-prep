# Чем WireMock отличается от Mockito?

**Коротко:** Mockito подменяет объекты и методы внутри JVM-процесса на уровне кода (юнит-тестирование), а WireMock поднимает настоящий HTTP-сервер и имитирует внешний API по сети (интеграционное/контрактное тестирование).

## Развёрнутый ответ

Mockito — это библиотека для создания тестовых дублёров (моков и стабов) на уровне Java-объектов. Она работает внутри процесса тестируемого приложения: с помощью байткод-манипуляций или прокси-объектов подменяет поведение конкретных классов и интерфейсов. Mockito используется в модульных (unit) тестах, где нужно изолировать тестируемый класс от его прямых зависимостей — репозиториев, сервисов, других компонентов — не выходя за пределы JVM и не выполняя реальных сетевых вызовов.

WireMock — это инструмент для эмуляции HTTP-сервисов на уровне сети. Он поднимает реальный (или встроенный) HTTP-сервер, который принимает настоящие HTTP-запросы и возвращает заранее настроенные ответы (стабы) по правилам сопоставления (URL, заголовки, тело запроса). WireMock применяется в интеграционных, компонентных и контрактных тестах, когда приложение реально делает HTTP-вызов (через HttpClient, RestTemplate, Feign, WebClient и т.п.) к внешнему сервису, который недоступен, нестабилен или платный в тестовом окружении.

Ключевые различия:
- **Уровень изоляции**: Mockito подменяет Java-объект в памяти (без сети), WireMock — эмулирует конечную точку по протоколу HTTP/HTTPS (с реальным сетевым стеком, включая сериализацию, заголовки, задержки, коды состояния).
- **Область применения**: Mockito — юнит-тесты отдельного класса; WireMock — интеграционные тесты клиентского кода, который делает реальные сетевые вызовы (проверяется весь стек: сериализация JSON, HTTP-клиент, таймауты, ретраи).
- **Что можно проверить**: WireMock позволяет протестировать реальную сериализацию/десериализацию запроса и ответа, поведение при разных HTTP-кодах, задержках, обрывах соединения — то, что Mockito не видит, так как не выходит за пределы Java-объектов.
- **Скорость и сложность**: тесты на Mockito быстрее и проще (не нужен порт, сервер), тесты с WireMock немного тяжелее, но дают больше уверенности в реальной интеграции.

Часто оба инструмента используются вместе на разных уровнях тестовой пирамиды: Mockito — в модульных тестах бизнес-логики, WireMock — в тестах адаптеров/клиентов внешних API.

## Пример / когда применяется

Mockito — тест сервиса `OrderService`, который зависит от интерфейса `PaymentClient`; PaymentClient мокается, чтобы не делать реальный HTTP-вызов и проверить только логику `OrderService`.

WireMock — тест самого класса `PaymentClient` (или интеграционный тест приложения целиком), где нужно убедиться, что клиент правильно формирует HTTP-запрос к платёжному шлюзу и корректно обрабатывает ответ 200/500/таймаут.

```java
// Настройка WireMock-стаба для внешнего сервиса
stubFor(post(urlEqualTo("/payments"))
    .withRequestBody(containing("orderId"))
    .willReturn(aResponse()
        .withStatus(200)
        .withHeader("Content-Type", "application/json")
        .withBody("{\"status\":\"SUCCESS\"}")));
```

## На что смотрит интервьюер

Интервьюер проверяет понимание уровней тестирования: юнит- против интеграционного тестирования, и умение выбрать правильный инструмент под задачу. Частая ошибка — считать WireMock "просто ещё одной моковой библиотекой" без понимания, что он работает по сети. Follow-up вопросы: "как протестировать таймауты и retry-логику клиента с помощью WireMock?", "можно ли использовать WireMock для контрактного тестирования (consumer-driven contracts)?". Красный флаг — если кандидат не может объяснить, почему нельзя просто замокать HTTP-клиент через Mockito вместо использования WireMock в интеграционном тесте (теряется проверка реальной сериализации и сетевого слоя).
