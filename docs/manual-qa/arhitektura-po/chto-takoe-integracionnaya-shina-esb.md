# Что такое **интеграционная шина (ESB)**?

**Коротко:** ESB (Enterprise Service Bus) — это центральный программный компонент интеграционной архитектуры, который связывает разнородные системы, беря на себя маршрутизацию, преобразование форматов данных, оркестрацию и посредничество между сервисами через единую точку интеграции.

## Развёрнутый ответ

Интеграционная шина предприятия (Enterprise Service Bus, ESB) — это архитектурный паттерн и класс middleware-решений, предназначенный для интеграции множества разнородных информационных систем (legacy-приложений, ERP, CRM, микросервисов, внешних партнёров) через единый центральный узел коммуникации, вместо создания прямых связей "точка-точка" между каждой парой систем.

Основные функции ESB:

- **Маршрутизация сообщений (routing)** — определение, куда должно быть направлено сообщение, на основе содержимого, заголовков или бизнес-правил.
- **Трансформация данных (transformation)** — преобразование формата сообщения между системами (например, из XML в JSON, из одной схемы данных в другую), чтобы разнородные системы могли обмениваться данными без прямой доработки друг под друга.
- **Протокольный медиатор (protocol mediation)** — преобразование между разными транспортными протоколами (SOAP, HTTP, JMS, FTP и т.п.).
- **Оркестрация бизнес-процессов** — координация последовательности вызовов нескольких сервисов для реализации сложного бизнес-сценария.
- **Обеспечение надёжности** — гарантированная доставка сообщений, повторные попытки, обработка ошибок, журналирование.
- **Безопасность и мониторинг** — централизованная точка для аутентификации, авторизации, аудита и наблюдения за трафиком между системами.

ESB исторически возник как решение проблемы "интеграционного спагетти" (spaghetti integration), когда при большом количестве систем количество прямых связей между ними растёт квадратично и становится неуправляемым. Вместо этого все системы подключаются к единой шине, и число интеграций становится линейным относительно числа систем.

Важный архитектурный нюанс: ESB — это централизованный, часто "толстый" (smart) компонент, в который выносится значительная часть бизнес-логики интеграции (маршрутизация, трансформация, оркестрация). Это отличает его от современного подхода микросервисной архитектуры, который придерживается принципа "smart endpoints, dumb pipes" (Мартин Фаулер) — умная логика остаётся в самих сервисах, а транспортный слой (например, простой брокер сообщений) остаётся максимально простым. Чрезмерная централизация логики в ESB со временем часто превращает шину в узкое место (single point of failure), сложный в сопровождении монолит интеграции и источник высокой связанности между командами, что стало одной из причин перехода индустрии к более лёгким подходам (брокеры сообщений, API Gateway, событийная архитектура).

## Пример / когда применяется

Крупная организация с десятками legacy-систем (SAP, устаревшая CRM, внешние банковские API, собственные ERP-модули), которым нужно обмениваться данными о клиентах и заказах. Вместо того чтобы писать N×(N-1) прямых интеграций между системами, все они подключаются к единой ESB (например, на базе Mule ESB, IBM Integration Bus, Apache ServiceMix), которая занимается маршрутизацией, трансформацией форматов и оркестрацией бизнес-процессов между ними.

| Подход | Число связей при N системах | Централизация логики |
|---|---|---|
| Точка-точка | Порядка N² | Логика размазана по каждой паре |
| ESB | Порядка N | Логика централизована в шине |

## На что смотрит интервьюер

Интервьюер проверяет понимание исторического контекста ESB и того, почему индустрия частично отошла от этого паттерна в пользу микросервисов и лёгких брокеров сообщений. Частая ошибка — путать ESB с обычным брокером сообщений (Kafka/RabbitMQ), не понимая, что ESB — гораздо более "тяжёлый" и умный компонент с трансформацией и оркестрацией. Follow-up вопросы: "чем ESB отличается от API Gateway?", "какие проблемы возникают при чрезмерной централизации логики в ESB?", "как современная микросервисная архитектура решает задачи, которые раньше решал ESB?". Красный флаг — незнание проблемы single point of failure и антипаттерна "умная шина, глупые сервисы" в противовес принципу "smart endpoints, dumb pipes".
