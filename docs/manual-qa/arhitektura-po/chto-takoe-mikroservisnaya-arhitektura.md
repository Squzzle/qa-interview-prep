# Что такое **микросервисная архитектура**?

**Коротко:** микросервисная архитектура — это стиль построения приложения как набора небольших, независимо развёртываемых сервисов, каждый из которых отвечает за свою бизнес-функцию и взаимодействует с другими по сети через чётко определённые API.

## Развёрнутый ответ

Микросервисная архитектура (microservices architecture) предполагает декомпозицию системы на набор автономных сервисов, каждый из которых:

- реализует ограниченную область ответственности (bounded context), обычно вокруг конкретной бизнес-возможности (заказы, платежи, пользователи и т.д.);
- имеет собственную кодовую базу, собственный процесс сборки и деплоя;
- может иметь собственную базу данных (принцип "database per service"), что обеспечивает слабую связанность на уровне данных;
- взаимодействует с другими сервисами через сеть — синхронно (REST, gRPC) или асинхронно (очереди сообщений, события — Kafka, RabbitMQ);
- может быть написан на своём стеке технологий, независимо от остальных сервисов (polyglot persistence, polyglot programming).

Ключевые принципы: единственная ответственность (single responsibility) на уровне сервиса, независимое развёртывание, децентрализованное управление данными, устойчивость к отказам отдельных компонентов (fault isolation), автоматизация инфраструктуры (CI/CD, контейнеризация, оркестрация через Kubernetes).

Для поддержки микросервисной архитектуры обычно требуется дополнительная инфраструктура: service discovery (Eureka, Consul), API Gateway, распределённый трейсинг (Jaeger, Zipkin), централизованное логирование (ELK/EFK), circuit breaker (Hystrix, Resilience4j), контейнеризация (Docker) и оркестрация (Kubernetes).

## Пример / когда применяется

Пример: интернет-магазин разбит на сервисы — «Каталог», «Корзина», «Заказы», «Платежи», «Уведомления», «Пользователи» — каждый со своей БД, своим API и независимым релизным циклом. При этом фронтенд обращается к ним через единый API Gateway.

| Аспект | Микросервисы |
|---|---|
| Масштабируемость | Каждый сервис масштабируется независимо |
| Технологическая свобода | Да, разные стеки для разных сервисов |
| Сложность эксплуатации | Высокая: сеть, мониторинг, трейсинг |
| Команда | Подходит для больших команд с автономными подкомандами |

Микросервисная архитектура применяется, когда система большая, требования к масштабируемости различаются по компонентам, команда достаточно большая, чтобы разделиться на автономные подкоманды по сервисам, и есть зрелая DevOps-культура.

## На что смотрит интервьюер

Интервьюер смотрит, понимает ли кандидат, что микросервисы — это не только "разбить код на файлы поменьше", а полноценное распределённое приложение со всеми вытекающими сложностями: сетевыми задержками, отказоустойчивостью, eventual consistency. Частая ошибка — назвать признаком микросервисов только "маленький размер сервиса", не упомянув независимый деплой и автономию данных.
