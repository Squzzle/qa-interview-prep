# Что такое **мок (Mock)** и **стаб (Stub)** в тестировании? Для чего они нужны?

**Коротко:** и мок, и стаб — это тестовые двойники (test doubles), заменяющие реальную зависимость; стаб просто возвращает заранее заданные ответы, а мок дополнительно проверяет, что к нему обратились ожидаемым образом (проверка поведения/вызовов).

## Развёрнутый ответ

В автоматизации и модульном тестировании реальные зависимости (внешние сервисы, базы данных, сторонние API) часто заменяют упрощёнными объектами-заглушками — тестовыми дублёрами (test doubles). Это делает тесты быстрыми, детерминированными и независимыми от внешней инфраструктуры.

**Стаб (Stub)** — это объект, который возвращает заранее подготовленные (захардкоженные) данные на определённые вызовы, но не проверяет, как именно и сколько раз его вызвали. Стаб используется для того, чтобы система под тестом получила нужные входные данные для проверки собственной логики. Пример: стаб репозитория, который всегда возвращает список из трёх пользователей, независимо от параметров запроса.

**Мок (Mock)** — это объект, который не только возвращает данные, но и позволяет верифицировать поведение: был ли вызван метод, с какими аргументами, сколько раз, в каком порядке. После выполнения теста явно проверяется (verify), что ожидаемые взаимодействия произошли. Моки применяются, когда важен не результат, а сам факт взаимодействия — например, что при оформлении заказа действительно был вызван метод отправки email-уведомления.

Общая классификация тестовых дублёров (по Мартину Фаулеру):
- **Dummy** — объект-заглушка, который передаётся, но никогда реально не используется (нужен только чтобы удовлетворить сигнатуру метода).
- **Fake** — рабочая, но упрощённая реализация (например, база данных в памяти вместо реальной СУБД).
- **Stub** — возвращает заданные ответы, состояние не проверяется.
- **Spy** — похож на стаб, но дополнительно записывает информацию о вызовах, которую потом можно проверить.
- **Mock** — заранее настраивается с ожиданиями поведения и сам проверяет их выполнение (fail fast при несоответствии).

Ключевое отличие в философии тестирования: стабы используются в тестировании состояния (state verification) — "что вернул объект", моки — в тестировании поведения (behavior verification) — "как вызывающий код взаимодействовал с зависимостью". Злоупотребление моками делает тесты хрупкими: если изменить внутреннюю реализацию (не меняя контракт), тест на моках может сломаться, хотя поведение системы для пользователя не изменилось.

## Пример / когда применяется

Стаб: тестируем метод `calculateDiscount(user)`, который вызывает `userRepository.getUser(id)`. Стаб `userRepository` настроен возвращать пользователя с определённым статусом лояльности, чтобы проверить корректность расчёта скидки.

Мок: тестируем метод `placeOrder(order)`, который должен вызвать `notificationService.sendEmail(...)`. Мок `notificationService` настраивается с ожиданием, что метод `sendEmail` будет вызван ровно один раз с определёнными параметрами, и тест падает, если этого не произошло.

Пример на Java с Mockito:

```java
// Стаб — просто задаём поведение
UserRepository userRepoStub = mock(UserRepository.class);
when(userRepoStub.getUser(1)).thenReturn(new User(1, "Gold"));

// Мок — дополнительно проверяем вызов
NotificationService notificationMock = mock(NotificationService.class);
orderService.placeOrder(order);
verify(notificationMock, times(1)).sendEmail(order.getUserEmail());
```

## На что смотрит интервьюер

Интервьюер оценивает, различает ли кандидат тестирование состояния и тестирование поведения, а не просто заучил термины. Частая ошибка — называть моками любые тестовые дублёры без разбора. Хороший ответ включает упоминание Fake, Spy, Dummy и объяснение, когда моки вредят (излишняя связанность теста с реализацией). Follow-up вопросы: "чем стаб отличается от Fake?", "что такое хрупкий тест из-за overmocking?", "как мокать статические методы или финальные классы?". Красный флаг — если кандидат не понимает, зачем вообще нужны тестовые дублёры (изоляция теста, скорость, детерминированность).
