# Что такое **оффсет (offset)** в Kafka?

**Коротко:** оффсет — это уникальный последовательный номер сообщения внутри конкретной партиции топика Kafka, по которому консьюмер отслеживает, какие сообщения он уже прочитал и с какого места продолжать чтение.

## Развёрнутый ответ

В Kafka каждая партиция топика представляет собой упорядоченную, неизменяемую последовательность сообщений. Каждому сообщению внутри партиции присваивается уникальный, монотонно возрастающий числовой идентификатор — оффсет (offset), начиная с 0. Оффсет уникален только в пределах одной партиции: одинаковые значения оффсета в разных партициях относятся к разным сообщениям.

Оффсет выполняет роль "закладки" в потоке сообщений: консьюмер запоминает, до какого оффсета он уже обработал данные, и при следующем чтении продолжает именно с этой позиции. Это принципиально отличает Kafka от классических очередей: сообщение не удаляется после прочтения, а хранится в логе согласно политике retention, и разные консьюмеры (или consumer group) могут независимо находиться на разных оффсетах одного и того же топика.

Ключевые аспекты работы с оффсетами:

- **Хранение оффсета консьюмера**. Kafka хранит текущий закоммиченный (committed) оффсет каждой consumer group в специальном внутреннем топике `__consumer_offsets`. Это позволяет консьюмеру после перезапуска продолжить чтение с места, где он остановился, а не читать всё заново.

- **Коммит оффсета (commit)**. Может быть автоматическим (`enable.auto.commit=true`, оффсет коммитится периодически по таймеру) или ручным (`commitSync`/`commitAsync` в клиенте), что даёт более точный контроль над тем, когда считать сообщение "обработанным" — важно для гарантий at-least-once/at-most-once.

- **Позиционирование**. Консьюмер может явно перемотать чтение на конкретный оффсет (`seek`), на начало партиции (`earliest`) или на конец (`latest`) — это полезно для перечитывания истории событий или пропуска накопленных сообщений.

- **Влияние на гарантии доставки**. Если оффсет закоммичен до фактической обработки сообщения, а обработчик упал — сообщение будет потеряно (at-most-once). Если оффсет коммитится после успешной обработки — при сбое сообщение будет обработано повторно (at-least-once), что требует идемпотентности обработчика.

## Пример / когда применяется

Партиция топика `orders` содержит сообщения с оффсетами 0, 1, 2, 3... Консьюмер прочитал и обработал сообщения до оффсета 5 включительно и закоммитил оффсет 6 (следующий для чтения). Если консьюмер перезапустится, он продолжит чтение именно с оффсета 6, не обрабатывая заново предыдущие сообщения.

```text
Партиция 0: [0][1][2][3][4][5][6][7]...
                              ^
                     committed offset = 6
                     (следующее чтение начнётся отсюда)
```

Пример ручного коммита в консьюмере (псевдокод на основе Kafka Java Client):

```java
ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
for (ConsumerRecord<String, String> record : records) {
    process(record);
}
consumer.commitSync(); // явный коммит после обработки батча
```

## На что смотрит интервьюер

Интервьюер проверяет понимание того, как Kafka реализует отслеживание прогресса чтения и как это влияет на гарантии доставки. Частая ошибка — путать оффсет сообщения (позиция в логе) с идентификатором сообщения по бизнес-смыслу (например, ID заказа). Follow-up вопросы: "что произойдёт, если закоммитить оффсет до обработки сообщения и сервис упадёт?", "как реализовать exactly-once с помощью оффсетов и внешнего хранилища?", "чем автоматический коммит отличается от ручного с точки зрения надёжности?". Красный флаг — незнание, что оффсет уникален только в пределах партиции, а не топика целиком.
