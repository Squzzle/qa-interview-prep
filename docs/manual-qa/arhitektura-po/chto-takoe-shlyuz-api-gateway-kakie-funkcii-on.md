# Что такое **шлюз (API Gateway)**? Какие функции он выполняет?

**Коротко:** API Gateway — это единая входная точка для внешних клиентов в микросервисную систему, которая принимает все запросы, маршрутизирует их к нужным сервисам и берёт на себя сквозные (cross-cutting) задачи: аутентификацию, ограничение частоты запросов, агрегацию ответов, логирование и балансировку нагрузки.

## Развёрнутый ответ

API Gateway — это компонент архитектуры, расположенный на границе системы между внешними клиентами (веб- и мобильными приложениями, партнёрскими системами) и внутренними микросервисами. Вместо того чтобы клиент напрямую обращался к десяткам отдельных сервисов, зная их адреса и протоколы, он взаимодействует с единой точкой входа — шлюзом, который скрывает внутреннюю структуру системы (реализует паттерн Backend for Frontend / Facade на уровне архитектуры).

Основные функции API Gateway:

- **Маршрутизация запросов (routing)** — направление входящего запроса к соответствующему внутреннему микросервису на основе URL, заголовков или других правил.
- **Агрегация ответов (composition)** — объединение данных из нескольких микросервисов в один ответ клиенту, чтобы клиенту не требовалось делать несколько отдельных запросов.
- **Аутентификация и авторизация** — централизованная проверка токенов (JWT, OAuth2), чтобы каждый отдельный сервис не дублировал эту логику.
- **Ограничение частоты запросов (rate limiting) и защита от перегрузки** — throttling, circuit breaker, защита от DDoS.
- **Балансировка нагрузки** — распределение запросов между несколькими экземплярами сервиса.
- **Трансформация протоколов и форматов** — например, преобразование внешнего REST-запроса во внутренний gRPC-вызов.
- **Кэширование ответов** — снижение нагрузки на внутренние сервисы за счёт кэширования часто запрашиваемых данных.
- **Логирование, мониторинг и трассировка** — централизованный сбор метрик и логов по всем входящим запросам.
- **Версионирование API** — поддержка нескольких версий публичного API одновременно.

API Gateway существенно упрощает жизнь клиентов, снимая с них необходимость знать внутреннюю структуру микросервисной системы, и централизует сквозную функциональность, которую иначе пришлось бы дублировать в каждом сервисе. При этом важно понимать риски: сам шлюз может стать единой точкой отказа (single point of failure) и узким местом производительности, поэтому его обычно проектируют отказоустойчивым (несколько реплик за балансировщиком) и стараются не перегружать сложной бизнес-логикой (в отличие от классического ESB).

Популярные реализации: Kong, NGINX, Amazon API Gateway, Spring Cloud Gateway, Zuul, Apigee, Kubernetes Ingress Controller с соответствующими плагинами.

## Пример / когда применяется

Мобильное приложение интернет-магазина отправляет один запрос `GET /api/product-page/{id}` на API Gateway. Шлюз проверяет JWT-токен пользователя, параллельно вызывает сервис каталога товаров, сервис отзывов и сервис рекомендаций, агрегирует три ответа в один JSON и возвращает клиенту — клиенту не нужно знать про существование трёх отдельных микросервисов.

```text
Клиент -> API Gateway
              ├── проверка JWT
              ├── вызов Product Service
              ├── вызов Reviews Service
              ├── вызов Recommendation Service
              └── агрегированный ответ клиенту
```

## На что смотрит интервьюер

Интервьюер оценивает понимание того, зачем нужен единый вход в систему и какие сквозные задачи он централизует. Частая ошибка — сводить API Gateway только к "прокси, который перенаправляет запросы", не упоминая аутентификацию, rate limiting, агрегацию. Follow-up вопросы: "как избежать того, чтобы Gateway стал единой точкой отказа?", "чем API Gateway отличается от Service Mesh?", "как реализовать паттерн Backend for Frontend поверх API Gateway?". Красный флаг — незнание риска превращения шлюза в "мини-монолит" при накоплении в нём бизнес-логики, что нарушает независимость микросервисов.
