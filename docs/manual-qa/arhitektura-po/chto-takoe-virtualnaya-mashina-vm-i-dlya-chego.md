# Что такое **виртуальная машина (VM)** и для чего она используется в тестировании?

**Коротко:** виртуальная машина — это программная эмуляция полноценного компьютера (с отдельной операционной системой, виртуальным диском, сетью и ресурсами), работающая поверх гипервизора на физическом хосте; в тестировании VM используют для изоляции окружений, воспроизведения разных ОС/конфигураций и быстрого отката к чистому состоянию.

## Развёрнутый ответ

Виртуальная машина (VM) — это программно реализованный аналог физического компьютера. Она имеет собственную гостевую операционную систему, виртуальные аппаратные ресурсы (CPU, RAM, диск, сетевой адаптер) и работает изолированно от хост-системы и других VM на этом же оборудовании. Управляет виртуальными машинами гипервизор — специальный слой ПО, который распределяет ресурсы физического хоста между гостевыми ОС. Различают два типа гипервизоров:

- **Тип 1 (bare-metal)** — работает напрямую на "железе" без базовой ОС (например, VMware ESXi, Microsoft Hyper-V, KVM). Используется в серверных и корпоративных инфраструктурах.
- **Тип 2 (hosted)** — запускается как приложение поверх уже установленной ОС (например, VMware Workstation, VirtualBox, Parallels Desktop). Чаще применяется на рабочих станциях тестировщиков и разработчиков.

Ключевые свойства VM, важные для тестирования:

- **Изоляция.** Каждая VM работает в собственном "песочнице" — сбой или заражение одной машины не влияет на хост и другие VM.
- **Полная эмуляция ОС.** Внутри VM можно установить любую операционную систему (Windows разных версий, различные дистрибутивы Linux, macOS через специальные решения), независимо от ОС хоста.
- **Снапшоты (снимки состояния).** Можно сохранить состояние системы в конкретный момент и в любой момент откатиться к нему — это критично для повторяемости тестов.
- **Клонирование.** Легко создать несколько идентичных копий окружения для параллельного тестирования.
- **Более высокое потребление ресурсов** по сравнению с контейнерами, так как каждая VM тащит за собой полную ОС.

Зачем VM нужны именно в тестировании:

1. **Кросс-платформенное и кросс-браузерное тестирование** — проверка работы приложения на разных ОС (Windows 10/11, разные дистрибутивы Linux, старые версии ОС) без покупки отдельного физического оборудования под каждую конфигурацию.
2. **Воспроизведение окружения пользователя** — тестирование на конфигурациях, максимально близких к продакшену или к окружению конкретного клиента.
3. **Изоляция тестового окружения** — чтобы тесты не влияли на рабочую машину тестировщика и не зависели от её состояния.
4. **Быстрый откат к чистому состоянию** через снапшоты — удобно для тестирования установки/удаления ПО, для повторных прогонов с "нуля".
5. **Тестирование деструктивных сценариев** — проверка поведения при сбоях ОС, вирусной активности, повреждении системы — без риска для реального оборудования.
6. **Параллельный запуск тестов** на нескольких VM одновременно для ускорения регрессионного тестирования.

## Пример / когда применяется

Типичный кейс: нужно проверить, что desktop-приложение корректно устанавливается и работает на Windows 10, Windows 11 и Windows Server 2019. Вместо трёх физических машин тестировщик поднимает три VM в VirtualBox или Hyper-V, для каждой делает "чистый" снапшот сразу после установки ОС. Перед каждым циклом тестирования выполняется откат к снапшоту, устанавливается новая сборка приложения, прогоняются тест-кейсы, а затем состояние снова откатывается — так гарантируется, что каждый прогон стартует с идентичного чистого окружения.

Другой пример — тестирование инсталлятора, который должен корректно откатывать изменения при неудачной установке: на VM можно безопасно проверить сценарий "прервать установку на середине" и убедиться, что система осталась в рабочем состоянии, а затем откатиться к снапшоту для повторного теста.

## На что смотрит интервьюер

- Понимает ли кандидат разницу между VM и контейнером (полная ОС и гипервизор vs. общее ядро ОС и изоляция процессов) — это стандартный follow-up вопрос.
- Знает ли кандидат, что такое гипервизор и может ли назвать конкретные инструменты (VirtualBox, VMware, Hyper-V, KVM).
- Понимает ли ценность снапшотов для повторяемости и чистоты тестового окружения — частая ошибка кандидатов не упоминать эту функцию вовсе.
- Может ли связать теорию с практикой: привести реальный пример из тестирования (кросс-платформенность, тестирование инсталляторов, изоляция окружений).
- Красный флаг: если кандидат путает VM с эмулятором мобильного устройства или не может объяснить, зачем вообще изолировать тестовое окружение.
