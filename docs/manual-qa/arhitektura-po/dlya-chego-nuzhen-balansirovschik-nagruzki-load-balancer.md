# Для чего нужен **балансировщик нагрузки (Load Balancer)**?

**Коротко:** балансировщик нагрузки распределяет входящие запросы между несколькими серверами, повышая производительность, отказоустойчивость и масштабируемость системы.

## Развёрнутый ответ

Балансировщик нагрузки (Load Balancer) — это компонент (аппаратный или программный), который стоит перед группой серверов (backend pool) и распределяет входящий трафик между ними по заданному алгоритму. Основные задачи балансировщика:

- Распределение нагрузки. Равномерно (или по определённой стратегии) направляет запросы на разные экземпляры сервера, чтобы ни один сервер не был перегружен, пока другие простаивают.
- Повышение отказоустойчивости (High Availability). Балансировщик регулярно проверяет состояние серверов (health check) и исключает из ротации те, что не отвечают или отвечают с ошибкой, направляя трафик только на живые узлы.
- Горизонтальное масштабирование. Позволяет добавлять новые серверы в пул без изменения клиентской части — приложение масштабируется добавлением новых инстансов "вширь", а не увеличением мощности одного сервера ("вверх").
- Устранение единой точки отказа на уровне серверов приложения (при этом сам балансировщик, если он не резервирован, может стать такой точкой отказа, поэтому балансировщики также резервируют).
- SSL-терминация. Может брать на себя расшифровку HTTPS-трафика, снижая нагрузку на backend-серверы.
- Sticky sessions (session affinity). Может направлять запросы одного и того же пользователя на один и тот же сервер, если это требуется для сохранения состояния сессии.

Балансировщики бывают программные (Nginx, HAProxy, Envoy), аппаратные (F5) и облачные управляемые сервисы (AWS ELB/ALB/NLB, GCP Load Balancer, Azure Load Balancer).

## Пример / когда применяется

Пример: высоконагруженный сайт разворачивает 5 идентичных экземпляров backend-сервера за Nginx-балансировщиком. При росте трафика в "Чёрную пятницу" добавляются ещё 5 инстансов без изменения кода приложения и без простоя. Если один из серверов падает, health check обнаруживает это за секунды и балансировщик перестаёт направлять на него трафик.

```text
Клиент → Load Balancer → [Server 1, Server 2, Server 3] → БД
```

Балансировщик применяется практически в любой продакшн-системе с более чем одним инстансом приложения, а также на уровне API Gateway в микросервисной архитектуре.

## На что смотрит интервьюер

Интервьюер проверяет, понимает ли кандидат, что балансировщик решает не только "распределение трафика", но и задачи отказоустойчивости через health checks. Частая ошибка — не упомянуть health check и не заметить, что сам балансировщик нуждается в резервировании (иначе становится единой точкой отказа).
