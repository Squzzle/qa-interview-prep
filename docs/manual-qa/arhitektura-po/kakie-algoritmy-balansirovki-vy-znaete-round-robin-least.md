# Какие алгоритмы балансировки вы знаете (Round Robin, Least Connections)?

**Коротко:** основные алгоритмы балансировки — Round Robin, Weighted Round Robin, Least Connections, Least Response Time, IP Hash и Random; выбор зависит от однородности серверов и требований к устойчивости сессий.

## Развёрнутый ответ

Основные алгоритмы балансировки нагрузки:

- Round Robin. Запросы распределяются по кругу — по очереди на каждый сервер из пула. Простой алгоритм, хорошо работает, если серверы одинаковы по мощности и запросы примерно равны по стоимости обработки.
- Weighted Round Robin. Аналогично Round Robin, но каждому серверу присваивается вес в зависимости от его производительности — более мощные серверы получают больше запросов.
- Least Connections. Запрос направляется на сервер с наименьшим числом активных соединений в данный момент. Хорошо подходит, когда время обработки запросов сильно различается.
- Weighted Least Connections. Комбинация Least Connections и весов — учитывает и число соединений, и производительность сервера.
- Least Response Time. Запрос направляется на сервер с наименьшим временем отклика (и/или наименьшим числом активных соединений), что учитывает реальную текущую нагрузку сервера.
- IP Hash. Балансировщик вычисляет хэш от IP-адреса клиента и на основе него определяет, на какой сервер направить запрос — обеспечивает "прилипание" клиента к одному серверу (session affinity/sticky sessions) без хранения состояния на балансировщике.
- Random (случайный выбор). Запрос направляется на случайный сервер из пула, иногда с учётом весов (Weighted Random).
- Consistent Hashing. Используется, когда важно минимизировать перераспределение запросов/данных при добавлении или удалении серверов (часто применяется в распределённых кэшах и системах с шардированием).

## Пример / когда применяется

| Алгоритм | Когда применять |
|---|---|
| Round Robin | Однородные серверы, схожая стоимость запросов |
| Weighted Round Robin | Серверы разной мощности |
| Least Connections | Запросы с сильно различающимся временем обработки (например, долгие отчёты и быстрые API-вызовы) |
| IP Hash | Нужна привязка клиента к серверу без хранения сессии на балансировщике |
| Least Response Time | Важна минимальная задержка ответа |

Пример: в кластере из 3 серверов, где один в 2 раза мощнее остальных, используется Weighted Round Robin с весами 2:1:1 — мощный сервер получает вдвое больше запросов. Если приложение хранит сессию пользователя в памяти конкретного сервера (без внешнего session store), используют IP Hash или sticky sessions, чтобы запросы одного пользователя всегда попадали на один и тот же сервер.

## На что смотрит интервьюер

Интервьюер оценивает, понимает ли кандидат не только названия алгоритмов, но и критерии выбора конкретного алгоритма под конкретную задачу. Частая ошибка — сказать, что Round Robin всегда достаточно, не учитывая неоднородность серверов или разную длительность запросов. Follow-up: "Почему IP Hash может привести к неравномерной нагрузке?", "Как решить проблему хранения сессий без sticky sessions?" (ответ: внешнее хранилище сессий, например Redis).
