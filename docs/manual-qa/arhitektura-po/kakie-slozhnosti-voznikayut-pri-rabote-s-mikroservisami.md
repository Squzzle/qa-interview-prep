# Какие сложности возникают при работе с микросервисами?

**Коротко:** микросервисы усложняют систему как распределённую: появляются проблемы согласованности данных, сетевой надёжности, тестирования, мониторинга и организационной координации.

## Развёрнутый ответ

Ключевые сложности микросервисной архитектуры:

- Согласованность данных (data consistency). Так как у каждого сервиса своя база, распределённые транзакции через несколько сервисов невозможны в классическом ACID-смысле, приходится использовать eventual consistency и паттерн Saga с компенсирующими транзакциями.
- Сетевые сбои и задержки. Вызовы между сервисами идут по сети, а не в рамках процесса, значит нужно закладывать таймауты, повторные попытки (retry), circuit breaker, чтобы не получить каскадный отказ.
- Сложность тестирования. Сквозные (end-to-end / интеграционные) тесты требуют поднятия множества сервисов одновременно; растёт роль contract testing (например, Pact) и тестирования в изоляции с моками зависимостей.
- Наблюдаемость (observability). Нужен распределённый трейсинг (Jaeger, Zipkin), централизованное логирование (ELK/EFK) и метрики (Prometheus/Grafana), иначе диагностировать проблему, распределённую по нескольким сервисам, крайне сложно.
- Управление версиями API. Изменение контракта одного сервиса может сломать другие сервисы-потребители, требуется версионирование API и обратная совместимость.
- Инфраструктурные накладные расходы. Нужны service discovery, API Gateway, оркестрация контейнеров (Kubernetes), CI/CD пайплайны на каждый сервис.
- Организационная сложность. Требуется чёткое разграничение зон ответственности между командами, иначе появляются "распределённые монолиты" — сервисы, которые технически разделены, но жёстко связаны и должны деплоиться вместе.
- Безопасность. Увеличивается площадь атаки: каждый сервис — отдельная точка входа, требуется аутентификация и авторизация между сервисами (mTLS, service mesh, OAuth2/JWT).
- Дублирование данных и логики. Для автономности сервисы иногда дублируют часть данных (денормализация), что требует продуманных механизмов синхронизации через события.

## Пример / когда применяется

Пример: при оформлении заказа нужно списать деньги со счёта («Платежи»), уменьшить остаток товара («Склад») и создать запись заказа («Заказы»). В монолите это была бы одна транзакция. В микросервисах используют Saga: если списание денег прошло, а резервирование товара не удалось — выполняется компенсирующая операция (возврат денег) через события в очереди сообщений.

| Проблема | Решение |
|---|---|
| Распределённые транзакции | Saga, event sourcing |
| Каскадные отказы | Circuit breaker, timeout, retry с backoff |
| Диагностика проблем | Distributed tracing, централизованные логи |
| Несовместимость API | Версионирование, contract testing |

## На что смотрит интервьюер

Интервьюер хочет услышать конкретные проблемы и конкретные паттерны их решения, а не общие слова "это сложно". Красный флаг — кандидат не может назвать ни одного паттерна (Saga, circuit breaker, API Gateway). Follow-up: "Как бы вы реализовали транзакцию через несколько сервисов?", "Что такое распределённый монолит и как его избежать?", "Какие инструменты используются для трейсинга?".
