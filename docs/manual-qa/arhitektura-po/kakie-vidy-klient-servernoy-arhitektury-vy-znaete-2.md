# Какие виды клиент-серверной архитектуры вы знаете (2-уровневая, 3-уровневая)?

**Коротко:** различают 1-уровневую (all-in-one), 2-уровневую (клиент напрямую взаимодействует с сервером БД) и 3-уровневую/N-уровневую архитектуру (клиент, сервер приложений, сервер БД разнесены на отдельные уровни).

## Развёрнутый ответ

Основные виды по числу уровней (tier):

- 1-уровневая архитектура (single-tier). Всё приложение — представление, логика и данные — работает на одной машине, без сети. Пример: настольное приложение с локальной базой данных (SQLite-калькулятор бюджета).
- 2-уровневая архитектура (two-tier, классическая клиент-сервер). Клиент содержит UI и, как правило, часть бизнес-логики, и напрямую обращается к серверу баз данных по сети. Такой клиент часто называют "толстым". Пример: desktop-приложение, которое напрямую подключается к серверу Oracle/MS SQL по драйверу ODBC/JDBC.
- 3-уровневая архитектура (three-tier). Добавляется промежуточный уровень — сервер приложений (application server), который содержит бизнес-логику. Клиент (тонкий) обращается к серверу приложений, а тот, в свою очередь, обращается к серверу базы данных. Классический пример — веб-приложение: браузер (уровень представления) → веб/app-сервер (уровень логики) → СУБД (уровень данных).
- N-уровневая архитектура (multi-tier / n-tier). Обобщение трёхуровневой модели: количество уровней увеличивается — добавляются, например, отдельный уровень кэширования, уровень API Gateway, уровень интеграции с внешними системами. Микросервисная архитектура — частный случай n-tier с множеством самостоятельных сервисов на уровне логики.

Преимущество увеличения числа уровней — более чёткое разделение ответственности, возможность масштабировать и обновлять каждый уровень независимо, повышение безопасности (клиент не имеет прямого доступа к базе данных). Недостаток — рост сложности инфраструктуры и увеличение задержки за счёт дополнительных сетевых переходов.

## Пример / когда применяется

| Уровень | Пример | Где логика |
|---|---|---|
| 1-tier | Desktop-калькулятор с локальной БД | На клиенте |
| 2-tier | Толстый клиент + прямой доступ к серверу БД | На клиенте (частично) |
| 3-tier | Браузер + веб-сервер (backend) + СУБД | На сервере приложений |
| N-tier | Веб-клиент + API Gateway + микросервисы + БД + кэш | Распределена по уровням |

2-уровневая архитектура применяется в небольших внутрикорпоративных системах с ограниченным числом пользователей. 3-уровневая и N-уровневая архитектуры используются в большинстве современных веб- и корпоративных приложений, где важны масштабируемость, безопасность и разделение ответственности.

## На что смотрит интервьюер

Интервьюер оценивает понимание того, зачем нужен дополнительный уровень (сервер приложений) между клиентом и БД — обычно это безопасность (клиент не видит БД напрямую) и возможность независимо масштабировать бизнес-логику. Красный флаг — путаница между "тонким/толстым клиентом" и количеством уровней архитектуры.
