# В чем разница между **шлюзом** и **шиной**?

**Коротко:** API Gateway — это единая точка входа для внешних клиентов в систему, ориентированная на маршрутизацию запросов "снаружи внутрь" и сквозные задачи (аутентификация, rate limiting), а интеграционная шина (ESB) — это центральный узел для интеграции внутренних систем друг с другом, ориентированный на трансформацию данных и оркестрацию бизнес-процессов "между системами".

## Развёрнутый ответ

Хотя оба компонента являются "центральными узлами" в архитектуре и внешне могут показаться похожими (единая точка, через которую проходит трафик), они решают разные задачи и применяются в разных сценариях.

**API Gateway** ориентирован на взаимодействие "снаружи внутрь" (north-south traffic): внешний клиент (браузер, мобильное приложение, партнёр) обращается к системе через шлюз, который маршрутизирует запрос к нужному внутреннему микросервису. Основной фокус — на клиентском опыте и сквозной функциональности на границе системы: аутентификация, авторизация, rate limiting, агрегация ответов нескольких сервисов, версионирование публичного API. Как правило, Gateway остаётся достаточно "тонким" и не содержит сложной бизнес-логики трансформации данных между внутренними системами.

**ESB (интеграционная шина)** ориентирован на взаимодействие "система-система" (внутри организации или между внутренними и внешними корпоративными системами), часто между разнородными и не связанными напрямую бизнес-приложениями (legacy ERP, CRM, внешние банковские системы). Основной фокус — на трансформации форматов данных, протокольном посредничестве (SOAP, JMS, FTP, HTTP), маршрутизации на основе содержимого и оркестрации сложных бизнес-процессов между системами. ESB, как правило, "толще" и умнее — в него часто выносится существенная часть интеграционной бизнес-логики.

Ключевые различия:

| Критерий | API Gateway | ESB (шина) |
|---|---|---|
| Направление трафика | Снаружи внутрь (клиент → сервисы) | Между внутренними/корпоративными системами |
| Основная задача | Единая точка входа, сквозные задачи | Интеграция и трансформация данных между системами |
| Логика | Обычно тонкая (routing, auth, rate limit) | Часто "толстая" (transformation, orchestration) |
| Типичный контекст | Микросервисная архитектура | Крупные предприятия с разнородными legacy-системами |
| Риск | Может стать узким местом на входе | Может стать "интеграционным монолитом" |

На практике границы иногда размываются: некоторые ESB-решения умеют выполнять роль API Gateway, а некоторые Gateway-решения обрастают функциями трансформации данных. Тем не менее, по историческому происхождению и типичному месту в архитектуре это разные паттерны: Gateway — про фасад для клиентов микросервисной системы, ESB — про интеграцию разнородных корпоративных систем друг с другом.

## Пример / когда применяется

API Gateway: мобильное приложение банка отправляет запрос через единый шлюз, который проверяет токен клиента, вызывает несколько внутренних микросервисов (счета, транзакции, курсы валют) и агрегирует ответ.

ESB: внутри того же банка ESB связывает старую мейнфреймовую систему учёта счетов, современную CRM и внешнюю систему межбанковских переводов (SWIFT), преобразуя форматы сообщений (например, из фиксированной длины записи мейнфрейма в XML для CRM) и оркеструя последовательность шагов при обработке платежа.

## На что смотрит интервьюер

Интервьюер проверяет, различает ли кандидат назначение и контекст применения двух паттернов, а не считает их синонимами. Частая ошибка — сказать, что "это одно и то же, просто разные названия". Follow-up вопросы: "может ли один компонент выполнять обе роли одновременно и к чему это может привести?", "почему в микросервисной архитектуре предпочитают тонкий Gateway, а не полноценный ESB?", "какие проблемы возникают, если в API Gateway начинают добавлять бизнес-логику трансформации данных, характерную для ESB?". Красный флаг — неспособность объяснить направление трафика (north-south у Gateway против интеграции систем друг с другом у ESB) как ключевой различающий признак.
