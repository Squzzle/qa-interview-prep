# Что такое **классификация дефектов**? По каким параметрам классифицируют?

**Коротко:** Классификация дефектов — это распределение найденных багов по группам согласно определённым критериям (серьёзность, приоритет, причина, область возникновения и т.д.), чтобы упростить их учёт, приоритизацию и анализ.

## Развёрнутый ответ

Классификация дефектов нужна, чтобы команда могла системно управлять большим количеством багов: понимать, что чинить в первую очередь, где сосредоточены основные проблемы, и как улучшить процесс разработки/тестирования на основе накопленной статистики. Основные параметры классификации:

1. **Severity (серьёзность)** — насколько сильно дефект влияет на работоспособность системы с технической точки зрения (Blocker, Critical, Major, Minor, Trivial).
2. **Priority (приоритет)** — насколько срочно нужно исправить дефект с точки зрения бизнеса (High, Medium, Low).
3. **Тип дефекта (Defect Type)** — функциональный, UI/визуальный, производительность, безопасность, совместимость, локализация, данные (data defect), логический и т.д.
4. **Этап обнаружения (Phase Detected)** — на каком этапе жизненного цикла ПО найден дефект: требования, дизайн, кодирование, тестирование, продакшн.
5. **Этап внесения (Phase Injected / Root Cause)** — на каком этапе была допущена первопричина: ошибка требований, ошибка дизайна, ошибка кодирования, ошибка конфигурации окружения.
6. **Область/модуль (Module/Component)** — к какой части системы относится дефект (авторизация, оплата, каталог и т.п.) — используется для анализа "горячих точек" кода.
7. **Природа дефекта** — функциональный vs нефункциональный (производительность, юзабилити, безопасность, надёжность).
8. **Воспроизводимость** — постоянно воспроизводимый (reproducible) vs периодически возникающий (intermittent, "плавающий" баг).
9. **Тип по стандарту ISO/IEC 25010 или ODC (Orthogonal Defect Classification)** — более формализованные схемы, используемые в зрелых процессах (IBM ODC классифицирует по типу дефекта: Assignment, Checking, Interface, Timing, Algorithm и др.).

Классификация используется не только для организации работы, но и для метрик качества: например, соотношение дефектов по типам помогает понять, стоит ли усиливать тестирование производительности или больше внимания уделять граничным значениям.

## Пример / когда применяется

Пример классификации одного дефекта: "Страница профиля пользователя падает с ошибкой 500 при загрузке аватара размером более 10 МБ" — Severity: Critical, Priority: High, Тип: функциональный/обработка данных, Этап обнаружения: системное тестирование, Этап внесения: кодирование (отсутствие проверки размера файла), Модуль: "Профиль пользователя".

Таблица типичных типов дефектов:

| Тип | Пример |
|---|---|
| Функциональный | Кнопка не выполняет действие |
| UI/визуальный | Смещённая вёрстка, неверный цвет |
| Производительность | Долгая загрузка страницы |
| Безопасность | SQL-инъекция, XSS |
| Совместимость | Не работает в Safari |
| Локализация | Неверный перевод строки |
| Данные | Некорректная валидация или сохранение данных |

## На что смотрит интервьюер

Интервьюер проверяет широту кругозора кандидата — способность назвать несколько независимых осей классификации (не только Severity/Priority, но и тип, этап обнаружения, модуль). Красный флаг — если кандидат сводит классификацию только к Severity и Priority, не упоминая типы дефектов. Плюсом будет упоминание ODC или метрик качества, построенных на основе классификации.
