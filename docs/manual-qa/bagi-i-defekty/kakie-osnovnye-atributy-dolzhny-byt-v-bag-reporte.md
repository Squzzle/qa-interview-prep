# Какие основные **атрибуты** должны быть в баг-репорте?

**Коротко:** Качественный баг-репорт должен содержать заголовок, шаги воспроизведения, фактический и ожидаемый результат, окружение, severity/priority, вложения (скриншоты, логи) и метаданные (автор, дата, статус, версия сборки).

## Развёрнутый ответ

Основные атрибуты баг-репорта:

- ID — уникальный идентификатор дефекта в баг-трекере (например, JIRA-1234).
- Title/Summary — короткий, информативный заголовок, отражающий суть проблемы (не "не работает", а конкретика, например "Кнопка Оплатить не активна при пустой корзине").
- Description — подробное описание проблемы.
- Steps to Reproduce (шаги воспроизведения) — пронумерованная последовательность действий для повторения бага.
- Actual Result — что происходит фактически.
- Expected Result — что должно происходить согласно требованиям.
- Environment (окружение) — версия ОС, браузера, устройства, версия сборки/билда, окружение (dev/stage/prod).
- Severity (серьёзность) — насколько дефект критичен технически (Blocker, Critical, Major, Minor, Trivial).
- Priority (приоритет) — насколько срочно нужно исправить с точки зрения бизнеса (High, Medium, Low).
- Attachments — скриншоты, видео, логи, дампы, HAR-файлы, ответы API.
- Reporter/Assignee — кто нашёл баг, кто отвечает за исправление.
- Status — текущий статус (New, Open, In Progress, Resolved, Reopened, Closed, Rejected).
- Labels/Components — модуль или компонент, к которому относится баг, для группировки и статистики.
- Related links — связь с тест-кейсом, требованием, другими дефектами (дубликаты, блокирующие связи).

Дополнительно хорошей практикой считается указывать: частоту воспроизведения (всегда/иногда/редко), версию, в которой баг впервые появился (если известно), и предположительную причину, если она очевидна тестировщику.

## Пример / когда применяется

Пример хорошо оформленного баг-репорта:

| Атрибут | Значение |
|---|---|
| ID | BUG-482 |
| Title | Ошибка 500 при оформлении заказа с промокодом на сумму более 10000 руб |
| Severity | Critical |
| Priority | High |
| Environment | Chrome 126, Windows 11, stage, build 2.14.3 |
| Steps to Reproduce | 1. Добавить товар на сумму более 10000 руб. 2. Применить промокод SALE20. 3. Нажать "Оформить заказ" |
| Actual Result | Сервер возвращает ошибку 500, заказ не создаётся |
| Expected Result | Заказ создаётся с применённой скидкой |
| Attachments | Скриншот ошибки, HAR-файл запроса |

Такой репорт понятен разработчику без дополнительных уточняющих вопросов и позволяет сразу приступить к воспроизведению и отладке.

## На что смотрит интервьюер

- Перечисляет ли кандидат все ключевые атрибуты, а не только "шаги" и "ожидаемый/фактический результат".
- Понимает ли разницу между Severity и Priority и может привести пример, когда они не совпадают (например, орфографическая ошибка на главной странице — низкий Severity, но High Priority из-за репутационных рисков).
- Упоминает ли важность окружения и вложений для быстрой диагностики.
- Красный флаг: не различает Severity и Priority, забывает про окружение или считает, что достаточно просто описать проблему словами без структуры.
