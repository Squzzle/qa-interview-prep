# Опишите **жизненный цикл бага** (основные статусы).

**Коротко:** Жизненный цикл бага (Bug Life Cycle, Defect Life Cycle) — это последовательность статусов, через которые проходит дефект от момента обнаружения до закрытия: New → Assigned → Open/In Progress → Fixed → Retest → Verified → Closed, с возможными ветвлениями в Reopened, Duplicate, Rejected, Deferred.

## Развёрнутый ответ

Жизненный цикл бага описывает движение дефекта по статусам в системе учёта (Jira, Bugzilla, Azure DevOps, TestRail) от момента его обнаружения тестировщиком до финального закрытия. Типичная последовательность:

1. **New (Новый)** — тестировщик обнаружил проблему и завёл баг-репорт.
2. **Assigned (Назначен)** — тимлид или менеджер назначил баг конкретному разработчику.
3. **Open / In Progress (В работе)** — разработчик начал анализ и исправление.
4. **Fixed (Исправлен)** — разработчик внёс исправление и передал баг обратно на проверку.
5. **Retest (На перепроверке)** — тестировщик повторно тестирует функциональность на предмет исправления.
6. **Verified (Подтверждён)** — тестировщик подтвердил, что дефект действительно устранён.
7. **Closed (Закрыт)** — баг окончательно закрыт, изменения зафиксированы (часто с привязкой к релизу/сборке).

Альтернативные и промежуточные ветви:

- **Reopened (Переоткрыт)** — если при ретесте баг воспроизводится снова, статус возвращается на доработку разработчику.
- **Duplicate (Дубликат)** — если баг уже был заведён ранее под другим номером.
- **Rejected / Not a Bug (Отклонён)** — если поведение признано корректным согласно требованиям.
- **Deferred / Postponed (Отложен)** — если исправление откладывается на будущий релиз по решению менеджмента.
- **Cannot Reproduce (Не воспроизводится)** — разработчик не смог повторить дефект по описанным шагам.

Жизненный цикл не всегда линеен: баг может несколько раз переходить между Reopened и Fixed, прежде чем будет окончательно закрыт.

## Пример / когда применяется

Пример движения бага в Jira: тестировщик создаёт баг со статусом New → тимлид ставит Assigned на разработчика → разработчик переводит в In Progress → после фикса в Resolved/Fixed → тестировщик берёт сборку с фиксом и переводит в Retest → если баг не воспроизводится, ставит Verified/Closed; если воспроизводится — Reopened, и цикл повторяется.

Диаграмма (текстовое представление):

```text
New -> Assigned -> In Progress -> Fixed -> Retest -> Verified -> Closed
                                     ^                    |
                                     +------ Reopened <---+
```

## На что смотрит интервьюер

Интервьюер проверяет знание стандартного набора статусов и логики переходов между ними, а также понимание, что цикл не заканчивается на Fixed — обязательна перепроверка тестировщиком. Частая ошибка кандидатов — закрывать баг сразу после Fixed без Retest/Verified, или путать статусы разных инструментов (Jira, Bugzilla). Может последовать уточняющий вопрос: "кто может переоткрыть баг?", "чем Verified отличается от Closed?".
