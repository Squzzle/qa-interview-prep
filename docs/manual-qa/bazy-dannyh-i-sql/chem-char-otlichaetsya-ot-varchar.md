# Чем CHAR отличается от VARCHAR?

**Коротко:** `CHAR(n)` — тип фиксированной длины, всегда занимает n символов, дополняя строку пробелами при необходимости. `VARCHAR(n)` — тип переменной длины, хранит только фактические символы (плюс небольшой служебный заголовок с длиной), занимая место по факту, но не более n символов.

## Развёрнутый ответ

При объявлении столбца `CHAR(n)` СУБД всегда резервирует место под n символов, независимо от фактической длины сохранённой строки. Если строка короче n, она дополняется пробелами справа до нужной длины (при чтении многие СУБД по умолчанию обрезают эти пробелы, но поведение зависит от конкретной системы и настроек).

`VARCHAR(n)` хранит строку переменной длины: в базе фактически занимается место под саму строку плюс небольшой заголовок (обычно 1-2 байта), где хранится её длина. Значение n задаёт лишь максимально допустимую длину, но не резервирует место заранее.

Практические следствия:

- **Использование дискового пространства.** `CHAR` эффективен, когда все значения имеют примерно одинаковую длину (например, код страны из 2 символов, хеш фиксированной длины). `VARCHAR` эффективен для строк переменной длины (имена, описания, email).
- **Производительность.** `CHAR` теоретически может давать небольшой выигрыш в скорости при сравнении и индексации за счёт фиксированного размера записи (упрощается доступ по смещению), но на практике разница в современных СУБД чаще всего незначительна и зависит от конкретной реализации.
- **Хранение пробелов.** При использовании `CHAR` нужно быть внимательным при сравнении строк — конечные пробелы могут игнорироваться или, наоборот, влиять на результат сравнения в зависимости от СУБД. `VARCHAR` хранит строку "как есть", без добавления пробелов.
- **Изменение записи.** Обновление `VARCHAR`-поля на более длинное значение может потребовать перемещения записи на странице данных (out-of-page storage), в то время как `CHAR` уже занимает фиксированный слот.

## Пример / когда применяется

```sql
CREATE TABLE countries (
    code CHAR(2) NOT NULL,       -- всегда ровно 2 символа: 'US', 'RU'
    name VARCHAR(100) NOT NULL   -- длина названия варьируется: 'USA', 'Russian Federation'
);
```

Если сохранить значение `'RU'` в `CHAR(2)`, оно займёт ровно 2 байта. Если сохранить `'US'` в `VARCHAR(100)`, будет физически занято только 2 символа плюс небольшой заголовок, а не 100 символов.

Пример с потенциальной "ловушкой" сравнения:

```sql
-- code CHAR(5) со значением 'US   ' (дополнено пробелами)
SELECT * FROM countries WHERE code = 'US'; -- может сработать из-за автоматического сравнения без учёта хвостовых пробелов
```

## На что смотрит интервьюер

- Понимание ключевого различия: фиксированная длина против переменной, а не просто "оба хранят строки".
- Знание, когда стоит выбрать `CHAR` (короткие значения фиксированной длины: коды, флаги, хеши) вместо `VARCHAR`.
- Понимание, что `n` в `VARCHAR(n)` — это ограничение максимальной длины, а не резервируемое место.
- Частая ошибка — утверждать, что `CHAR` всегда быстрее `VARCHAR` без оговорок; в реальности выигрыш зависит от СУБД и сценария и часто пренебрежимо мал.
- Хороший ответ упоминает нюанс с хвостовыми пробелами при сравнении строк типа `CHAR`.
