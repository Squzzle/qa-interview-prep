# Чем master-slave репликация отличается от master-master?

**Коротко:** В master-slave репликации запись данных возможна только на одном узле (master), а остальные (slave) доступны только для чтения. В master-master (multi-master) записывать данные можно на нескольких узлах одновременно, и изменения синхронизируются между ними в обе стороны.

## Развёрнутый ответ

**Master-slave (primary-replica):**
- Один узел (master) принимает все операции записи (`INSERT`, `UPDATE`, `DELETE`).
- Один или несколько узлов (slave/replica) получают копию изменений от master и используются, как правило, только для чтения.
- Простая модель конфликтов — конфликтов записи не возникает, так как запись идёт только в одну точку.
- Единая точка отказа для записи: если master падает, запись данных невозможна до тех пор, пока одна из реплик не будет вручную или автоматически повышена (promote) до нового master.
- Хорошо подходит для нагрузок с преобладанием чтения над записью (read-heavy).

**Master-master (multi-master):**
- Несколько узлов могут одновременно принимать операции записи.
- Изменения реплицируются между всеми узлами в обе стороны.
- Повышенная доступность на запись — при отказе одного master остальные продолжают принимать запись без ручного переключения.
- Главная сложность — разрешение конфликтов записи (conflict resolution), когда один и тот же объект данных изменяется одновременно на разных узлах. Решается через стратегии типа "last write wins", векторные часы, ручное слияние или бизнес-логику приложения.
- Более сложна в настройке и поддержке, есть риск рассинхронизации данных при сетевых разделениях (network partition), что напрямую связано с теоремой CAP.

Таким образом, ключевое отличие — количество узлов, допускающих запись, и, как следствие, наличие или отсутствие проблемы конфликтов записи.

## Пример / когда применяется

Master-slave типичен для веб-приложений, где нагрузка на чтение значительно превышает нагрузку на запись — например, интернет-магазин с каталогом товаров: одна база принимает заказы (запись), несколько реплик обслуживают показ каталога (чтение).

Master-master применяется, когда нужна запись из разных географических регионов с низкой задержкой (например, глобальный сервис с дата-центрами в Европе и Азии, где в каждом регионе есть свой master, принимающий локальную запись) или когда требуется отсутствие простоя записи при отказе одного узла.

```text
Master-slave:
  [Master] --записи--> (пишет клиент)
     |  реплицирует
     v
  [Slave 1] [Slave 2]  --чтения--> (читает клиент)

Master-master:
  [Master A] <--двунаправленная репликация--> [Master B]
     ^ пишет и читает                  пишет и читает ^
```

## На что смотрит интервьюер

- Понимание, что ключевое отличие — не "количество серверов", а то, сколько узлов принимают запись.
- Знание проблемы конфликтов записи в master-master и хотя бы одного способа их разрешения.
- Понимание компромиссов: master-slave проще, но имеет единую точку отказа для записи; master-master сложнее, но устойчивее.
- Частая ошибка — путать master-master с шардированием: в master-master все узлы хранят одинаковые полные данные, а в шардировании данные разделены на части.
- Хороший ответ приводит примеры инструментов: MySQL Group Replication, Galera Cluster для master-master; стандартная потоковая репликация PostgreSQL для master-slave.
