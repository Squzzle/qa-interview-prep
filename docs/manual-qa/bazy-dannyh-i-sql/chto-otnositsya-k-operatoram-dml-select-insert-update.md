# Что относится к операторам DML (SELECT, INSERT, UPDATE, DELETE)?

**Коротко:** DML (Data Manipulation Language) — группа операторов SQL, отвечающая за работу с содержимым данных внутри уже существующих таблиц: выборку, добавление, изменение и удаление строк. Основные операторы: `SELECT`, `INSERT`, `UPDATE`, `DELETE`.

## Развёрнутый ответ

В отличие от DDL, который меняет структуру базы данных, DML работает с данными внутри таблиц, не затрагивая их схему.

- **`SELECT`** — выбирает (читает) данные из одной или нескольких таблиц. Поддерживает фильтрацию (`WHERE`), сортировку (`ORDER BY`), группировку (`GROUP BY`, `HAVING`), объединение таблиц (`JOIN`), подзапросы и агрегатные функции. Строго говоря, `SELECT` только читает данные и не изменяет их, поэтому в некоторых классификациях его выделяют в отдельную группу DQL (Data Query Language).
- **`INSERT`** — добавляет новые строки в таблицу. Может добавлять одну строку, несколько строк за один запрос, либо результат другого `SELECT`-запроса (`INSERT INTO ... SELECT`).
- **`UPDATE`** — изменяет значения существующих строк по заданному условию (`WHERE`). Без условия `WHERE` изменит все строки таблицы — частая причина ошибок в продакшене.
- **`DELETE`** — удаляет строки из таблицы по условию (`WHERE`). Без `WHERE` удалит все строки, но, в отличие от `TRUNCATE`, делает это построчно, с полным логированием и возможностью отката внутри транзакции.

Все операторы DML (в отличие от большинства DDL) выполняются в рамках транзакций: их можно откатить командой `ROLLBACK`, если они не были зафиксированы `COMMIT`. Это ключевое практическое отличие DML от DDL.

Важные нюансы:
- `UPDATE` и `DELETE` без `WHERE` — один из самых частых источников инцидентов ("уронил прод"), поэтому в реальной практике перед выполнением таких операций часто сначала пишут аналогичный `SELECT` с тем же условием, чтобы проверить, какие строки будут затронуты.
- Производительность `DELETE` на больших таблицах может быть значительно ниже, чем `TRUNCATE`, из-за построчного логирования и работы триггеров.
- `INSERT` может конфликтовать с ограничениями уникальности (`UNIQUE`, `PRIMARY KEY`), для обработки таких случаев есть конструкции вроде `ON CONFLICT` (PostgreSQL) или `ON DUPLICATE KEY UPDATE` (MySQL).

## Пример / когда применяется

```sql
-- SELECT: выборка с фильтрацией и сортировкой
SELECT id, name, salary
FROM employees
WHERE salary > 50000
ORDER BY salary DESC;

-- INSERT: добавление строки
INSERT INTO employees (id, name, salary)
VALUES (101, 'Иван Петров', 60000);

-- UPDATE: изменение существующих строк
UPDATE employees
SET salary = salary * 1.1
WHERE department = 'Engineering';

-- DELETE: удаление строк по условию
DELETE FROM employees
WHERE id = 101;
```

Пример безопасной практики перед массовым изменением:

```sql
-- Сначала проверить, что будет затронуто
SELECT * FROM employees WHERE department = 'Sales';

-- Затем выполнить изменение внутри транзакции
BEGIN;
UPDATE employees SET salary = salary * 1.05 WHERE department = 'Sales';
-- проверить результат, затем
COMMIT;
```

## На что смотрит интервьюер

- Знание всех четырёх основных операторов DML и их назначения.
- Понимание, что DML-операции откатываются в транзакциях, в отличие от большинства DDL-операций.
- Понимание риска `UPDATE`/`DELETE` без `WHERE` и знание практики предварительной проверки через `SELECT`.
- Знание разницы между `DELETE` и `TRUNCATE` по производительности и логированию (частый follow-up).
- Частая ошибка — не упомянуть, что `SELECT` формально тоже относится к DML (или путать, к какой группе он относится), а также забыть про важность транзакций при массовых изменениях.
