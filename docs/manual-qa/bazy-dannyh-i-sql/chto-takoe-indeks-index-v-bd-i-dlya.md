# Что такое индекс (Index) в БД и для чего он нужен?

**Коротко:** индекс — это дополнительная структура данных (обычно B-tree или хэш-таблица), которая ускоряет поиск и выборку строк по значению одного или нескольких столбцов, ценой дополнительного места на диске и замедления операций вставки/обновления/удаления.

## Развёрнутый ответ

Без индекса СУБД для выполнения запроса с условием (`WHERE`) вынуждена выполнять **полное сканирование таблицы (full table scan / sequential scan)** — последовательно просматривать все строки, проверяя условие. При больших объёмах данных это медленно.

Индекс — это отдельная структура данных, которая хранит отсортированные (или иначе организованные) значения индексируемого столбца вместе со ссылками на физическое расположение соответствующих строк в таблице. Это позволяет находить нужные строки за логарифмическое время (O(log n)) вместо линейного (O(n)).

Основные типы индексов:

- **B-tree (сбалансированное дерево)** — самый распространённый тип, эффективен для операций сравнения (`=`, `<`, `>`, `BETWEEN`), сортировки и диапазонных запросов. Используется по умолчанию в большинстве СУБД.
- **Hash-индекс** — эффективен только для точного сравнения на равенство (`=`), не поддерживает диапазонные запросы и сортировку.
- **Составной (многоколоночный) индекс** — строится по нескольким столбцам сразу; порядок столбцов важен, так как индекс эффективен, если запрос использует столбцы слева направо (leftmost prefix).
- **Уникальный индекс (unique index)** — гарантирует отсутствие дубликатов; автоматически создаётся для PRIMARY KEY и UNIQUE-ограничений.
- **Полнотекстовый индекс (full-text index)** — оптимизирован для поиска по тексту (словам, фразам).
- **Покрывающий индекс (covering index)** — включает все столбцы, нужные для запроса, что позволяет СУБД не обращаться к самой таблице.

Компромиссы использования индексов:

- Ускоряют операции `SELECT` с фильтрацией, сортировкой (`ORDER BY`) и соединением (`JOIN`) по индексируемым столбцам.
- Замедляют операции `INSERT`, `UPDATE`, `DELETE`, так как при каждом изменении данных СУБД должна также обновить все связанные индексы.
- Занимают дополнительное место на диске.
- Избыточное количество индексов на таблице может ухудшить общую производительность записи без существенной пользы для чтения.

Поэтому индексы создаются осознанно — обычно на столбцах, часто используемых в `WHERE`, `JOIN`, `ORDER BY`, и особенно на внешних и первичных ключах.

## Пример / когда применяется

```sql
-- Создание индекса на столбце email для ускорения поиска
CREATE INDEX idx_users_email ON users(email);

-- Составной индекс для запросов, фильтрующих по обоим полям
CREATE INDEX idx_orders_customer_status ON orders(customer_id, status);

-- Проверка использования индекса в плане выполнения
EXPLAIN ANALYZE
SELECT * FROM users WHERE email = 'test@example.com';
```

Тестировщик может использовать `EXPLAIN` / `EXPLAIN ANALYZE`, чтобы убедиться, что медленный запрос действительно использует индекс (`Index Scan`), а не выполняет полное сканирование таблицы (`Seq Scan`), что особенно важно при тестировании производительности.

## На что смотрит интервьюер

Интервьюер проверяет понимание того, что индекс — это компромисс между скоростью чтения и скоростью записи, а не "бесплатное" ускорение. Хороший кандидат объясняет принцип работы (B-tree, ускорение поиска) и упоминает негативный эффект на INSERT/UPDATE/DELETE. Красный флаг — совет "добавить индексы на все столбцы", не понимая последствий. Follow-up: "почему не стоит индексировать каждый столбец?", "как проверить, используется ли индекс в запросе?", "что такое составной индекс и как порядок столбцов влияет на его эффективность?".
