# Что такое представление (View)?

**Коротко:** Представление (View) — это виртуальная таблица, основанная на результате заранее сохранённого SQL-запроса. Оно не хранит данные физически, а формирует их "на лету" при обращении.

## Развёрнутый ответ

View создаётся командой `CREATE VIEW` и хранит в себе не сами данные, а определение запроса (`SELECT`). При каждом обращении к представлению СУБД заново выполняет исходный запрос к базовым таблицам и возвращает актуальный результат. Благодаря этому данные в представлении всегда синхронизированы с данными в исходных таблицах.

Основные цели использования View:

- **Упрощение сложных запросов.** Если часто используется один и тот же сложный `JOIN` с несколькими условиями, его можно один раз оформить как View и затем обращаться к нему как к обычной таблице.
- **Ограничение доступа к данным.** Можно создать View, показывающее только часть столбцов или строк исходной таблицы (например, скрыть колонку с зарплатой или паролями), и выдать пользователю права только на это представление, а не на исходную таблицу.
- **Абстракция от структуры данных.** Если структура таблиц меняется, можно адаптировать View так, чтобы внешний интерфейс (набор доступных полей) остался прежним для приложений и пользователей.
- **Повторное использование логики.** Общая бизнес-логика агрегации или фильтрации выносится в один объект.

Есть также **материализованные представления (Materialized View)** — в отличие от обычного View, они физически хранят результат запроса на диске и обновляются по расписанию или вручную (`REFRESH MATERIALIZED VIEW`). Это ускоряет чтение ценой актуальности данных.

Ограничения обычного View: не все представления допускают модификацию данных через них (`INSERT`/`UPDATE`/`DELETE`) — это зависит от сложности запроса (наличие `JOIN`, `GROUP BY`, агрегатных функций обычно делает View доступным только для чтения).

## Пример / когда применяется

Пример создания представления, скрывающего чувствительные данные:

```sql
CREATE VIEW employee_public AS
SELECT id, first_name, last_name, department
FROM employees;
```

Пример представления с объединением таблиц для упрощения повторяющегося запроса:

```sql
CREATE VIEW order_details AS
SELECT o.id AS order_id, c.name AS customer_name, o.total, o.created_at
FROM orders o
JOIN customers c ON c.id = o.customer_id;

SELECT * FROM order_details WHERE total > 1000;
```

## На что смотрит интервьюер

- Понимает ли кандидат, что View — это не физическое хранилище данных, а сохранённый запрос (в отличие от материализованного представления).
- Знает ли разницу между обычным и материализованным View, а также когда стоит выбрать каждый из вариантов.
- Может ли объяснить практическую пользу View: безопасность, упрощение, инкапсуляция логики.
- Частая ошибка — путать View с таблицей и утверждать, что данные "хранятся" в представлении.
- Хороший кандидат упомянет, что производительность View зависит от исходного запроса — сложные View с большим количеством join могут работать медленно, так как выполняются заново при каждом обращении.
