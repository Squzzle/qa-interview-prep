# Что такое реляционная база данных?

**Коротко:** реляционная база данных — это база данных, в которой данные организованы в виде таблиц (отношений), связанных между собой через общие поля (ключи), и обрабатываются с помощью реляционной алгебры и языка SQL.

## Развёрнутый ответ

Реляционная модель была предложена Эдгаром Коддом в 1970 году. В ней данные представляются в виде набора таблиц (relations), где:

- Каждая **таблица** состоит из строк (кортежей, records) и столбцов (атрибутов, fields).
- Каждая строка уникально идентифицируется **первичным ключом (Primary Key)**.
- Связи между таблицами устанавливаются через **внешние ключи (Foreign Key)**, которые ссылаются на первичные ключи других таблиц.
- Данные хранятся без избыточности благодаря процессу **нормализации** (устранение дублирования и аномалий обновления/вставки/удаления).
- Для доступа к данным используется декларативный язык запросов **SQL** (Structured Query Language).

Реляционные СУБД (RDBMS) обычно обеспечивают свойства **ACID**:

- **Atomicity (атомарность)** — транзакция выполняется целиком или не выполняется вовсе.
- **Consistency (согласованность)** — данные всегда переходят из одного корректного состояния в другое.
- **Isolation (изолированность)** — параллельные транзакции не влияют друг на друга некорректным образом.
- **Durability (устойчивость)** — результаты подтверждённой транзакции сохраняются даже при сбое системы.

Преимущества реляционной модели: строгая структура, целостность данных, мощный язык запросов, зрелая экосистема инструментов. Недостатки: сложнее горизонтально масштабировать, менее гибкая схема по сравнению с NoSQL.

## Пример / когда применяется

Пример реляционной структуры интернет-магазина:

```sql
CREATE TABLE customers (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    customer_id INT REFERENCES customers(id),
    total NUMERIC(10,2)
);

SELECT c.name, o.total
FROM customers c
JOIN orders o ON o.customer_id = c.id
WHERE o.total > 1000;
```

Реляционные БД используются там, где критична целостность данных: банковские системы, системы бронирования, ERP, бухгалтерский учёт.

## На что смотрит интервьюер

Проверяется понимание понятий "таблица", "ключ", "связь", "нормализация" и "ACID". Часто интервьюер просит привести пример SQL-запроса с JOIN, чтобы убедиться, что кандидат не просто знает теорию, но и умеет работать с реляционными данными. Красный флаг — путаница между реляционной и нереляционной моделями или неспособность объяснить, зачем нужны внешние ключи. Follow-up: "что такое нормализация?", "какие аномалии решает нормализация?", "что такое ACID?".
