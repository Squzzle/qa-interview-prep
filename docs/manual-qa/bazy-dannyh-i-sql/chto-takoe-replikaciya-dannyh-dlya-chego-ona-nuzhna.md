# Что такое репликация данных? Для чего она нужна?

**Коротко:** Репликация — это процесс копирования и синхронизации данных между несколькими экземплярами базы данных (серверами), чтобы обеспечить отказоустойчивость, доступность и распределение нагрузки.

## Развёрнутый ответ

При репликации изменения, произведённые на одном сервере базы данных (обычно называемом primary/master), автоматически передаются на один или несколько других серверов (replica/slave). В результате несколько копий базы данных содержат одинаковые (или почти одинаковые, с небольшой задержкой) данные.

Основные цели репликации:

- **Отказоустойчивость (High Availability).** Если master-сервер выходит из строя, одна из реплик может быть повышена до нового master, и система продолжит работать без длительного простоя.
- **Масштабирование чтения.** Read-запросы можно распределять между несколькими репликами, снижая нагрузку на основной сервер, который в этом случае обрабатывает в основном запросы записи.
- **Географическое распределение.** Реплики можно размещать ближе к пользователям в разных регионах, снижая задержки (latency).
- **Резервное копирование и аналитика.** На реплике можно выполнять тяжёлые аналитические запросы или снимать бэкапы, не нагружая продуктивный master.

По способу синхронизации репликация бывает:

- **Синхронная** — master ждёт подтверждения от реплики(к) перед фиксацией транзакции. Гарантирует отсутствие потери данных, но увеличивает задержку записи.
- **Асинхронная** — master фиксирует транзакцию не дожидаясь реплики. Быстрее, но есть риск потери последних изменений при отказе master (репликационный лаг).

По архитектуре репликация делится на master-slave (один источник записи) и master-master (несколько узлов, принимающих запись) — это отдельный часто задаваемый вопрос на собеседовании.

## Пример / когда применяется

Типичный сценарий: у высоконагруженного веб-сервиса один master принимает все `INSERT`/`UPDATE`/`DELETE`, а несколько read-реплик обслуживают `SELECT`-запросы приложения (например, отображение каталога товаров).

```sql
-- На master: запись изменения
UPDATE products SET stock = stock - 1 WHERE id = 42;

-- На read-реплике: приложение читает актуальные (с небольшой задержкой) данные
SELECT * FROM products WHERE category = 'electronics';
```

В PostgreSQL для настройки потоковой репликации используется механизм WAL (Write-Ahead Log): реплика подключается к master и получает поток WAL-записей, применяя их к своей копии данных.

## На что смотрит интервьюер

- Понимание разницы между целями репликации (отказоустойчивость, масштабирование чтения) и целями шардирования (масштабирование записи/объёма).
- Знание компромисса между синхронной и асинхронной репликацией: согласованность данных против производительности.
- Понимание понятия "репликационный лаг" (replication lag) и его последствий — например, читающий с реплики клиент может не увидеть только что записанные на master данные.
- Частая ошибка — путать репликацию с резервным копированием (backup): репликация не защищает от логических ошибок (например, случайного `DROP TABLE`), которые мгновенно реплицируются на все узлы.
- Хороший ответ упоминает конкретные механизмы (WAL в PostgreSQL, binlog в MySQL) и роли master/replica.
