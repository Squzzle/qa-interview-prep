# Что такое шардирование (Sharding)? Какую проблему оно решает?

**Коротко:** Шардирование — это горизонтальное разбиение данных на независимые части (шарды), которые хранятся на разных серверах. Оно решает проблему масштабирования, когда объём данных или нагрузка на запись превышают возможности одного сервера.

## Развёрнутый ответ

В отличие от репликации, где каждый узел хранит полную копию всех данных, при шардировании данные делятся на непересекающиеся части (шарды) по определённому ключу (shard key), и каждый шард хранится на отдельном сервере. Например, таблица пользователей может быть разбита так, что пользователи с id от 1 до 1 000 000 находятся на одном сервере, а с id от 1 000 001 до 2 000 000 — на другом.

Шардирование решает проблемы, которые репликация решить не может:

- **Ограничение объёма данных на одном сервере.** Если таблица настолько велика, что не помещается на диск одного сервера (или работает крайне медленно из-за объёма), её нужно физически разделить.
- **Ограничение пропускной способности записи.** Репликация масштабирует чтение, но все записи всё равно идут через один master. Шардирование распределяет и записи между разными серверами, так как каждый шард обрабатывает запись только для своей части данных.
- **Ограничение по производительности индексов и памяти.** Меньший объём данных на шарде — быстрее работают индексы, больше данных помещается в кэш/память.

Стратегии выбора shard key (ключа шардирования):

- **Диапазонное (range-based)** — данные делятся по диапазону значений ключа (например, по дате или по id). Просто, но может привести к неравномерной нагрузке ("горячие" шарды).
- **Хеш-шардирование (hash-based)** — вычисляется хеш от ключа, и по нему определяется шард. Обеспечивает более равномерное распределение, но усложняет диапазонные запросы.
- **По справочнику/директории (directory-based)** — отдельный сервис-каталог хранит соответствие "ключ -> шард".

Сложности шардирования: невозможность (или сложность) выполнения `JOIN` между данными разных шардов, сложность обеспечения транзакций между шардами (распределённые транзакции), необходимость ре-шардирования при росте данных, усложнение агрегатных запросов (`COUNT`, `SUM` по всем шардам требует объединения результатов на уровне приложения).

## Пример / когда применяется

Пример: крупная социальная сеть с миллиардами пользователей шардирует таблицу сообщений по `user_id`, чтобы каждый шард хранил сообщения только подмножества пользователей:

```text
Shard 1: user_id % 4 == 0  -> Server A
Shard 2: user_id % 4 == 1  -> Server B
Shard 3: user_id % 4 == 2  -> Server C
Shard 4: user_id % 4 == 3  -> Server D
```

Запрос при этом маршрутизируется на нужный шард на основе значения `user_id`, вычисленного в приложении или прокси-слое (например, Vitess для MySQL, Citus для PostgreSQL):

```sql
-- Логика маршрутизации на стороне приложения:
-- shard = user_id % 4
SELECT * FROM messages WHERE user_id = 12345;
```

## На что смотрит интервьюер

- Понимание принципиальной разницы между шардированием (разные данные на разных серверах) и репликацией (одинаковые данные на разных серверах) — эти понятия часто путают.
- Умение объяснить, какую именно проблему решает шардирование: масштабирование записи и объёма хранения, а не отказоустойчивость.
- Знание сложностей: межшардовые join, распределённые транзакции, необходимость выбора хорошего shard key.
- Частая ошибка — считать шардирование заменой репликации; на практике их часто комбинируют (каждый шард дополнительно реплицируется для отказоустойчивости).
- Хороший ответ приводит примеры инструментов (Vitess, Citus, MongoDB sharding) и упоминает проблему "горячих" шардов при неудачном выборе ключа.
