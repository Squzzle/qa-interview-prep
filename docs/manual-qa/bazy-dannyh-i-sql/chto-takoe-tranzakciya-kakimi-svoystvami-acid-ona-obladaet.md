# Что такое транзакция? Какими свойствами (ACID) она обладает?

**Коротко:** Транзакция — это последовательность операций с базой данных, которая выполняется как единое неделимое целое: либо все операции применяются успешно, либо ни одна. Свойства ACID (Atomicity, Consistency, Isolation, Durability) гарантируют корректность и надёжность выполнения транзакций.

## Развёрнутый ответ

Транзакция объединяет несколько операций (например, несколько запросов `INSERT`, `UPDATE`, `DELETE`) в один логический блок работы. Она начинается командой `BEGIN` (или `START TRANSACTION`) и завершается либо `COMMIT` (фиксация изменений), либо `ROLLBACK` (отмена всех изменений).

Свойства ACID расшифровываются так:

- **Atomicity (атомарность)** — транзакция выполняется полностью или не выполняется вовсе. Если в середине транзакции произошёл сбой, все уже сделанные изменения откатываются, база возвращается в исходное состояние.
- **Consistency (согласованность)** — транзакция переводит базу данных из одного согласованного (валидного) состояния в другое. Все ограничения целостности (constraints, foreign key, unique, триггеры) должны соблюдаться до и после транзакции.
- **Isolation (изолированность)** — параллельно выполняющиеся транзакции не должны видеть промежуточные, незафиксированные изменения друг друга. Степень изоляции регулируется уровнями изоляции: `READ UNCOMMITTED`, `READ COMMITTED`, `REPEATABLE READ`, `SERIALIZABLE`. Более строгие уровни защищают от аномалий (dirty read, non-repeatable read, phantom read), но снижают параллелизм.
- **Durability (долговечность)** — после фиксации (`COMMIT`) изменения сохраняются даже в случае сбоя системы (отключения питания, падения сервера). Это обеспечивается за счёт журналирования (write-ahead log, WAL) и записи на диск.

Транзакции критичны там, где нужно обеспечить целостность данных при множественных взаимосвязанных изменениях — например, при переводе денег между счетами: списание с одного счёта и зачисление на другой должны произойти вместе или не произойти вовсе.

## Пример / когда применяется

Классический пример — банковский перевод:

```sql
BEGIN;

UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
UPDATE accounts SET balance = balance + 1000 WHERE id = 2;

COMMIT;
```

Если после первого `UPDATE` произойдёт сбой (например, обрыв соединения), благодаря атомарности транзакция откатится, и деньги не "исчезнут" — баланс первого счёта не изменится.

Если нужно отменить операцию вручную (например, обнаружена ошибка в бизнес-логике):

```sql
BEGIN;

UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
-- обнаружена ошибка
ROLLBACK;
```

## На что смотрит интервьюер

- Знает ли кандидат полную расшифровку ACID и может объяснить каждое свойство своими словами, а не просто назвать буквы.
- Понимает ли разницу между уровнями изоляции транзакций и связанными с ними аномалиями (dirty read, non-repeatable read, phantom read, lost update).
- Может ли привести практический пример, где отсутствие транзакции приведёт к проблеме (например, к рассинхронизации данных).
- Частая ошибка — путать Consistency в ACID с Consistency из теоремы CAP, это разные понятия.
- Хороший ответ включает упоминание `COMMIT`/`ROLLBACK` и того, что не все СУБД/движки таблиц поддерживают транзакции (например, MyISAM в MySQL не поддерживает, в отличие от InnoDB).
