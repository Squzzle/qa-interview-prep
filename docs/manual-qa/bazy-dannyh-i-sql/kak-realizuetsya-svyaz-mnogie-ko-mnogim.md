# Как реализуется связь "многие ко многим"?

**Коротко:** связь "многие ко многим" (N:M) в реляционных БД напрямую не поддерживается и реализуется через промежуточную (связующую) таблицу, содержащую внешние ключи на обе связанные таблицы.

## Развёрнутый ответ

Реляционная модель не позволяет одной строке одной таблицы напрямую ссылаться на несколько строк другой таблицы и наоборот с помощью одного внешнего ключа — внешний ключ всегда указывает на одну конкретную строку. Поэтому для реализации связи N:M вводится дополнительная сущность — **связующая (junction / associative / pivot) таблица**.

Такая таблица обычно содержит:

- Внешний ключ на первую таблицу.
- Внешний ключ на вторую таблицу.
- Составной первичный ключ из этих двух внешних ключей (чтобы исключить дублирующиеся связи), либо отдельный суррогатный ключ, если нужно хранить дополнительные атрибуты связи (например, дату создания связи, роль, количество).
- Дополнительные атрибуты, специфичные именно для самой связи, а не для одной из основных сущностей (например, "дата записи на курс", "оценка", "количество товара в заказе").

Связующая таблица фактически превращает связь N:M в две связи 1:N: "таблица A" 1:N "связующая таблица" N:1 "таблица B".

## Пример / когда применяется

Пример: студенты и курсы — студент может посещать несколько курсов, курс может посещать несколько студентов.

```sql
CREATE TABLE students (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE courses (
    id SERIAL PRIMARY KEY,
    title VARCHAR(100)
);

CREATE TABLE enrollments (
    student_id INT REFERENCES students(id),
    course_id INT REFERENCES courses(id),
    enrolled_at TIMESTAMP DEFAULT now(),
    grade INT,
    PRIMARY KEY (student_id, course_id)
);

-- Все курсы конкретного студента
SELECT c.title
FROM courses c
JOIN enrollments e ON e.course_id = c.id
WHERE e.student_id = 1;
```

Другие примеры N:M: "товары - заказы" (через таблицу `order_items` с атрибутами количество и цена), "теги - статьи", "пользователи - роли".

## На что смотрит интервьюер

Интервьюер оценивает понимание того, что N:M требует именно отдельной таблицы, а не просто двух внешних ключей в одной из существующих таблиц. Важно, чтобы кандидат упомянул составной ключ (или уникальный индекс) для предотвращения дубликатов и мог объяснить, зачем в связующей таблице хранить дополнительные атрибуты. Красный флаг — попытка реализовать N:M без промежуточной таблицы (например, через массив ID в одном из полей, что нарушает нормализацию). Follow-up: "как бы вы хранили количество товара в заказе?", "что если нужно, чтобы студент не мог дважды записаться на один курс?".
