# Какие модели баз данных вы знаете?

**Коротко:** основные модели данных — иерархическая, сетевая, реляционная, объектно-ориентированная и документо-ориентированная (NoSQL); сегодня доминирует реляционная модель, дополняемая различными видами NoSQL.

## Развёрнутый ответ

Модель данных определяет, как данные организуются, связываются и обрабатываются в базе данных. Исторически и логически выделяют несколько основных моделей:

- **Иерархическая модель** — данные организованы в виде дерева, где каждый дочерний элемент имеет только одного родителя (пример: IBM IMS). Проста, но плохо описывает связи "многие ко многим".
- **Сетевая модель** — обобщение иерархической, где запись может иметь несколько родителей (пример: CODASYL). Более гибкая, но сложнее в реализации и сопровождении.
- **Реляционная модель** — данные представлены в виде таблиц (отношений), связанных по ключам; основана на реляционной алгебре Эдгара Кодда. Наиболее распространена (MySQL, PostgreSQL, Oracle).
- **Объектно-ориентированная модель** — данные хранятся как объекты, аналогично ООП-языкам, с поддержкой наследования и инкапсуляции (пример: db4o).
- **Документо-ориентированная (NoSQL)** — данные хранятся как документы (JSON/BSON), не требующие жёсткой схемы (MongoDB, CouchDB).
- **Ключ-значение** — простейшая модель, где данные хранятся как пары ключ-значение (Redis, DynamoDB).
- **Колоночная (columnar)** — данные хранятся по столбцам, а не строкам, что ускоряет аналитические запросы (Cassandra, ClickHouse).
- **Графовая модель** — данные представлены как узлы и рёбра, удобна для описания сложных взаимосвязей (Neo4j).

Выбор модели зависит от характера данных, требований к согласованности (consistency), масштабируемости и типа выполняемых запросов.

## Пример / когда применяется

Реляционная модель подходит для банковских систем, где важна строгая целостность данных и связи между таблицами (клиенты, счета, транзакции). Документная модель (MongoDB) удобна для хранения профилей пользователей с изменяемой структурой полей. Графовая модель применяется в социальных сетях для описания связей "друзья друзей" или в системах рекомендаций.

```sql
-- Пример реляционной структуры
CREATE TABLE accounts (
    id SERIAL PRIMARY KEY,
    owner_id INT NOT NULL,
    balance NUMERIC(12,2) NOT NULL DEFAULT 0
);
```

## На что смотрит интервьюер

Интервьюер проверяет широту кругозора кандидата: способность назвать не только реляционную модель, но и альтернативы, а также объяснить, чем они отличаются друг от друга и когда какую стоит использовать. Красный флаг — если кандидат знает только "таблицы" и не может назвать ни одной NoSQL-модели. Follow-up вопросы: "почему в вашем проекте выбрана именно реляционная БД?", "чем документная модель отличается от реляционной?".
