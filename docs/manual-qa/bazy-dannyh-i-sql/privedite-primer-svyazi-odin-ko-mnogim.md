# Приведите пример связи "один ко многим".

**Коротко:** связь "один ко многим" (1:N) — это когда одна запись родительской таблицы может соответствовать нескольким записям дочерней таблицы, но каждая запись дочерней таблицы связана только с одной записью родительской; классический пример — "клиент - заказы".

## Развёрнутый ответ

Связь "один ко многим" реализуется через внешний ключ (Foreign Key) в дочерней таблице, который ссылается на первичный ключ (Primary Key) родительской таблицы. Родительская таблица не хранит никакой информации о количестве связанных записей — вся логика связи выражена через значение внешнего ключа в дочерней таблице.

Признаки связи 1:N:

- В родительской таблице каждая строка уникальна и не повторяется как "источник" для нескольких дочерних строк напрямую в этой же таблице.
- В дочерней таблице значение внешнего ключа может повторяться (несколько заказов у одного клиента), но обычно каждая строка ссылается ровно на одного родителя.
- Такая связь — самая распространённая в реляционных БД, потому что напрямую отражает естественные бизнес-отношения ("один автор — много книг", "одна категория — много товаров", "один отдел — много сотрудников").

Важно: удаление родительской записи при наличии связанных дочерних требует продуманной стратегии — `ON DELETE CASCADE` (удалить все дочерние), `ON DELETE RESTRICT` (запретить удаление, если есть дочерние), `ON DELETE SET NULL` (обнулить внешний ключ).

## Пример / когда применяется

Классический пример: один клиент (customer) может оформить несколько заказов (orders), но каждый заказ принадлежит только одному клиенту.

```sql
CREATE TABLE customers (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL
);

CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    customer_id INT NOT NULL REFERENCES customers(id) ON DELETE CASCADE,
    total NUMERIC(10,2),
    created_at TIMESTAMP DEFAULT now()
);

-- Все заказы конкретного клиента
SELECT o.id, o.total
FROM orders o
WHERE o.customer_id = 1;
```

Другие примеры 1:N: "автор - книги", "категория - товары", "отдел - сотрудники", "пост в блоге - комментарии".

## На что смотрит интервьюер

Интервьюер проверяет, может ли кандидат не просто назвать пример, а объяснить механизм реализации (внешний ключ в дочерней таблице) и последствия для операций удаления/обновления. Красный флаг — путаница направления связи (например, добавление внешнего ключа в родительскую таблицу вместо дочерней). Follow-up: "что произойдёт при удалении клиента, если есть связанные заказы?", "как ограничить одновременное существование дублирующихся связей?".
