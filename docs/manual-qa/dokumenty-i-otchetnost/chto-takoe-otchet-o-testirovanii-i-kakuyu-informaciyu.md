# Что такое **отчет о тестировании** и какую информацию он должен содержать?

**Коротко:** отчёт о тестировании (Test Summary Report / Test Report) — итоговый документ, который фиксирует результаты проведённого тестирования, статус качества продукта и позволяет заинтересованным лицам принять решение о релизе.

## Развёрнутый ответ

Отчёт о тестировании составляется по итогам цикла тестирования (спринта, релиза, тестовой фазы) и является ключевым артефактом для принятия решения "go / no-go". Его основная цель — дать объективную и измеримую картину того, насколько продукт готов к выпуску, какие риски остаются и какие решения нужно принять.

Стандартный отчёт содержит:

- **Общую информацию**: название проекта, версия/билд, период тестирования, авторы отчёта, ссылки на тест-план.
- **Объём тестирования (scope)**: что тестировалось и что было сознательно исключено из тестирования (out of scope) и почему.
- **Статистику по тест-кейсам**: сколько кейсов запланировано, выполнено, пройдено, провалено, заблокировано, пропущено; процент покрытия требований.
- **Статистику по дефектам**: количество найденных багов по severity/priority, сколько исправлено, сколько осталось открытыми, сколько отклонено, распределение по модулям.
- **Критерии входа/выхода**: были ли выполнены критерии выхода (exit criteria) из тест-плана, и если нет — какие отклонения и почему они допустимы.
- **Риски и известные проблемы**: список открытых дефектов, которые пойдут в релиз, с оценкой их влияния на пользователя и бизнес.
- **Выводы и рекомендации**: рекомендация команды — готов ли продукт к релизу, какие условия должны быть соблюдены (например, "релиз возможен при условии фикса критичного бага X").

Отчёт должен быть написан языком, понятным не только тестировщикам, но и менеджерам продукта, стейкхолдерам, поэтому важна визуализация (графики, таблицы) и чёткие формулировки без лишнего технического жаргона.

## Пример / когда применяется

Пример краткой сводки в отчёте:

| Метрика | Значение |
|---|---|
| Всего тест-кейсов | 320 |
| Выполнено | 320 (100%) |
| Пройдено | 298 (93%) |
| Провалено | 15 |
| Заблокировано | 7 |
| Открытых Critical/High багов | 2 |
| Рекомендация | Условный релиз при фиксе 2 критичных дефектов |

Такой отчёт готовится в конце каждого спринта или перед релизом и прикладывается к решению о выпуске версии в продакшн.

## На что смотрит интервьюер

Интервьюер оценивает, понимает ли кандидат разницу между "отчётом о баге" и "отчётом о тестировании" в целом, и умеет ли выделить главное — не просто перечислить пункты, а объяснить, зачем нужен каждый раздел для принятия решения. Плохой ответ — простое перечисление без понимания цели документа. Частые follow-up вопросы: "Кто читает этот отчёт?", "Что делать, если exit criteria не выполнены, но бизнес требует релиз в срок?", "Чем отличается Test Summary Report от Test Status Report (промежуточного статуса)?"
