# Что такое **тест-кейс**? Из каких основных атрибутов он состоит?

**Коротко:** тест-кейс — это документированный набор шагов, предусловий, входных данных и ожидаемого результата для проверки конкретного аспекта функциональности системы; основные атрибуты — ID, название, предусловия, шаги, тестовые данные, ожидаемый результат, приоритет и статус.

## Развёрнутый ответ

Тест-кейс (test case) — это формализованный сценарий проверки, который позволяет однозначно и повторяемо определить, соответствует ли поведение системы ожидаемому результату. Тест-кейсы пишутся на основе требований, пользовательских историй или анализа рисков и служат основным артефактом для ручного и автоматизированного тестирования.

Стандартные атрибуты тест-кейса:

- **ID (идентификатор)** — уникальный номер для трассируемости и ссылок в баг-репортах.
- **Название (Title/Summary)** — краткое описание того, что проверяется.
- **Предусловия (Preconditions)** — состояние системы/данных перед выполнением шагов (например, "пользователь авторизован", "в корзине есть товар").
- **Шаги выполнения (Steps)** — последовательность действий тестировщика.
- **Тестовые данные (Test Data)** — конкретные входные значения, используемые в шагах.
- **Ожидаемый результат (Expected Result)** — что должно произойти после каждого шага или после всего сценария.
- **Фактический результат (Actual Result)** — заполняется при выполнении, для сравнения с ожидаемым.
- **Приоритет (Priority)** — важность теста (например, high/medium/low), влияет на порядок выполнения при ограниченном времени.
- **Статус выполнения (Status)** — pass/fail/blocked/skipped.
- **Связь с требованием (Requirement ID / Traceability)** — ссылка на требование или user story, которое покрывает тест-кейс.
- **Постусловия (Postconditions)** — необязательный атрибут: состояние системы после выполнения теста (например, для очистки данных).
- **Тип теста / категория** — позитивный, негативный, граничный и т.п.

Хороший тест-кейс атомарен (проверяет один аспект), понятен для выполнения другим человеком без пояснений автора и содержит достаточно деталей для воспроизводимости результата.

## Пример / когда применяется

Пример тест-кейса для функции входа в систему:

| Атрибут | Значение |
|---|---|
| ID | TC-LOGIN-014 |
| Название | Успешный вход с корректными учётными данными |
| Предусловие | Пользователь зарегистрирован в системе, аккаунт активен |
| Шаги | 1. Открыть страницу входа. 2. Ввести логин test@mail.com. 3. Ввести пароль Qwerty123!. 4. Нажать "Войти" |
| Ожидаемый результат | Пользователь перенаправлен на главную страницу, отображается приветствие с именем пользователя |
| Приоритет | High |
| Связь с требованием | REQ-AUTH-001 |

## На что смотрит интервьюер

Интервьюер проверяет, знает ли кандидат полный набор атрибутов, а не только "шаги и ожидаемый результат", и понимает ли важность атомарности и трассируемости тест-кейса. Часто просят написать тест-кейс на месте по простому требованию — оценивается структура, полнота предусловий, конкретность ожидаемого результата (не "система работает корректно", а точное описание видимого поведения).
