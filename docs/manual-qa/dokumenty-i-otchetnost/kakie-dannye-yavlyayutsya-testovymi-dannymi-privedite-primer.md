# Какие данные являются **тестовыми данными**? Приведите примеры.

**Коротко:** тестовые данные — это любые входные значения, состояния системы или конфигурации, которые специально подготовлены или сгенерированы для выполнения тест-кейсов и проверки поведения системы в конкретных условиях.

## Развёрнутый ответ

Тестовые данные — это всё, что "скармливается" системе во время теста: значения полей форм, содержимое базы данных, файлы для загрузки, параметры API-запросов, состояния учётных записей и т.д. Качество тестовых данных напрямую влияет на качество тестирования: недостаточно разнообразные данные пропускают дефекты, избыточно сложные — замедляют тестирование.

Тестовые данные принято классифицировать по нескольким признакам:

- **По валидности**: валидные данные (соответствуют бизнес-правилам, ожидаемо обрабатываются успешно) и невалидные (нарушают правила — используются для негативного тестирования).
- **По происхождению**: реальные данные (обезличенная копия продакшн-данных), синтетические/сгенерированные данные (созданные вручную или инструментами типа Faker), граничные значения (edge cases — минимум, максимум, пустое значение, ноль).
- **По назначению**: данные для позитивных сценариев, данные для негативных сценариев, данные для нагрузочного тестирования (большие объёмы), данные для проверки безопасности (SQL-инъекции, XSS-пейлоады).

Примеры конкретных тестовых данных:

- Логин/пароль тестового пользователя (например, valid_user / Qwerty123!).
- Email в разных форматах: корректный (user@example.com), некорректный (user@, @example.com), с недопустимыми символами.
- Числовые значения на границах диапазона: 0, -1, максимально допустимое число, число за пределами диапазона.
- Файлы для загрузки: корректного формата и размера, повреждённый файл, файл превышающий лимит, файл недопустимого расширения.
- Даты: сегодняшняя, прошедшая, будущая, некорректная (31 февраля), граничные даты (например, 29 февраля в невисокосный год).
- Строки: пустая строка, строка с пробелами, очень длинная строка, строка с эмодзи/юникодом, строка со спецсимволами (для проверки инъекций).

Важный нюанс: тестовые данные должны быть управляемыми и воспроизводимыми — то есть перед каждым прогоном тестов данные либо очищаются/пересоздаются (seed-данные), либо тест изолирован от состояния, оставленного предыдущими прогонами. Также при использовании реальных данных обязательно соблюдение требований по защите персональных данных (маскирование, анонимизация).

## Пример / когда применяется

Пример набора тестовых данных для поля "Возраст" (диапазон 18–65):

| Тип данных | Значение | Ожидаемый результат |
|---|---|---|
| Валидное | 30 | Принято |
| Граничное нижнее | 18 | Принято |
| Граничное нижнее невалидное | 17 | Отклонено |
| Граничное верхнее | 65 | Принято |
| Граничное верхнее невалидное | 66 | Отклонено |
| Невалидное (тип) | "abc" | Отклонено с ошибкой валидации |
| Пустое | (пусто) | Отклонено, если поле обязательно |

Такие данные готовятся при тестировании форм регистрации, анкет, любых полей ввода с ограничениями.

## На что смотрит интервьюер

Интервьюер оценивает, умеет ли кандидат мыслить категориями граничных значений и классов эквивалентности при подготовке тестовых данных, а не приводит только "очевидные" валидные примеры. Красный флаг — если кандидат называет только позитивные данные и не упоминает негативные/граничные случаи.
