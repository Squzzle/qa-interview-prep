# Какие методы анализа требований вы знаете (ревью, прототипирование)?

**Коротко:** основные методы анализа требований — рецензирование (review), прототипирование, моделирование (диаграммы, use case), составление таблиц решений и чек-листов, а также интервью и мозговой штурм со стейкхолдерами для выявления и уточнения требований.

## Развёрнутый ответ

Анализ требований — это процесс изучения, уточнения и структурирования требований для выявления неполноты, противоречий и рисков до начала разработки. Основные методы:

- **Рецензирование (review / walkthrough / inspection)** — формальное или неформальное чтение документа группой (аналитик, разработчик, тестировщик, заказчик) с фиксацией замечаний. Может быть формальным (Fagan inspection с ролями и метриками) или неформальным (peer review).

- **Прототипирование (prototyping)** — создание макета (wireframe, кликабельный прототип в Figma) для визуализации требования и получения обратной связи от заказчика до написания кода. Особенно полезно для UI/UX требований, которые сложно описать текстом.

- **Моделирование (modeling)** — построение диаграмм: use case диаграммы, диаграммы состояний (state machine), диаграммы последовательности (sequence diagram), BPMN-схемы бизнес-процессов. Помогает визуализировать сложные сценарии и найти нестыковки в потоках.

- **Таблицы решений (decision tables)** — структурирование правил бизнес-логики через комбинации условий и действий; хорошо выявляет неучтённые комбинации.

- **Мозговой штурм и интервью со стейкхолдерами** — прямое выяснение потребностей, часто в формате воркшопов (например, Event Storming, User Story Mapping).

- **Составление сценариев использования (use cases) и пользовательских историй (user stories)** с критериями приёмки (acceptance criteria) — структурируют требования вокруг задач пользователя.

- **Чек-листы и матрицы трассируемости** — для проверки полноты покрытия функциональности требованиями.

- **Анализ по методу "5 почему" / причинно-следственный анализ** — для уточнения истинной бизнес-потребности за формальной формулировкой требования.

Тестировщик обычно активно участвует в review, построении таблиц решений и анализе прототипов, поскольку эти методы напрямую пересекаются с тест-дизайном.

## Пример / когда применяется

Пример применения таблицы решений при анализе требования "скидка для клиента": анализ комбинаций условий "тип клиента" (новый/постоянный) и "сумма заказа" (меньше/больше порога) сразу выявляет, что аналитик не указал, какая скидка положена новому клиенту с заказом выше порога — пробел, найденный до кодирования.

| Тип клиента | Сумма заказа | Скидка |
|---|---|---|
| Новый | &lt; 5000 | 0% |
| Новый | ≥ 5000 | ? (не указано в требовании) |
| Постоянный | &lt; 5000 | 5% |
| Постоянный | ≥ 5000 | 10% |

## На что смотрит интервьюер

Интервьюер проверяет, знает ли кандидат конкретные методы (а не только слово "ревью"), и может ли объяснить, для какой задачи какой метод подходит (например, прототипирование — для UX, таблицы решений — для сложной бизнес-логики). Часто просят привести пример из личного опыта. Красный флаг — если кандидат называет анализом требований только чтение документа без активного поиска пробелов и противоречий.
