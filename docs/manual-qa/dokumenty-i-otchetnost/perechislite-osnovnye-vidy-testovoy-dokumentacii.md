# Перечислите основные **виды тестовой документации**.

**Коротко:** основные виды тестовой документации — тест-план (test plan), тест-стратегия (test strategy), тест-кейсы, чек-листы, тестовые сценарии (test scenario), отчёт о тестировании (test report/summary), баг-репорты, матрица трассируемости и тестовые данные.

## Развёрнутый ответ

Тестовая документация сопровождает весь процесс тестирования — от планирования до подведения итогов. Основные виды по стандарту IEEE 829 / ISTQB и общепринятой практике:

- **Тест-стратегия (Test Strategy)** — документ верхнего уровня, описывающий общий подход к тестированию для организации или программы проектов: какие виды тестирования применяются, инструменты, роли, метрики качества.

- **Тест-план (Test Plan)** — документ для конкретного проекта/релиза: цели, объём (scope), стратегия, ресурсы, сроки, критерии входа/выхода (entry/exit criteria), риски, окружение.

- **Тестовые сценарии (Test Scenarios)** — высокоуровневое описание того, что нужно протестировать (например, "проверить процесс оформления заказа"), без детального пошагового описания.

- **Тест-кейсы (Test Cases)** — детальные пошаговые инструкции проверки конкретной функциональности с ожидаемым результатом (ID, предусловия, шаги, входные данные, ожидаемый результат, приоритет).

- **Чек-листы (Checklists)** — сжатый список пунктов для проверки без детальных шагов, применяются при регрессии, smoke- и exploratory-тестировании.

- **Матрица трассируемости требований (Requirements Traceability Matrix, RTM)** — таблица связи требований с тест-кейсами, показывающая покрытие и облегчающая анализ влияния изменений.

- **Баг-репорт (Defect/Bug Report)** — документ по каждому найденному дефекту: шаги воспроизведения, ожидаемый/фактический результат, окружение, severity/priority, вложения.

- **Отчёт о тестировании (Test Summary Report / Test Report)** — итоговый документ по циклу тестирования: сколько тест-кейсов выполнено, сколько багов найдено, статус по критериям выхода, риски, рекомендация по релизу.

- **Тестовые данные (Test Data)** — набор данных, необходимых для выполнения тестов (валидные, невалидные, граничные значения).

- **План/отчёт о тестовом окружении (Test Environment specification)** — описание конфигурации стендов для тестирования.

## Пример / когда применяется

Пример иерархии документов в реальном проекте: тест-стратегия определяет, что регрессия автоматизирована, а новые фичи тестируются вручную по тест-кейсам → тест-план фиксирует конкретный релиз 2.5, сроки и критерии выхода → тест-кейсы пишутся по требованиям к релизу 2.5 → по итогам прогона формируется test summary report с рекомендацией "готово к релизу" или "требуется доработка".

## На что смотрит интервьюер

Интервьюер проверяет, различает ли кандидат тест-план и тест-стратегию (частая путаница), а также умеет ли объяснить назначение каждого документа, а не просто перечислить названия. Могут попросить описать содержание конкретного документа (например, "что входит в тест-план?") или спросить, какие документы кандидат реально вёл на предыдущем проекте.
