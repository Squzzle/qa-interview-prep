# В каком документе описываются Use Cases?

**Коротко:** сценарии использования (Use Cases) обычно описываются в специальном документе Use Case Specification (или Use Case Document), который может быть отдельным артефактом либо частью более общего документа функциональных требований (Functional Requirements Specification, FRS/SRS).

## Развёрнутый ответ

Use Case — это описание взаимодействия актора (пользователя или другой системы) с системой для достижения конкретной цели, включающее основной поток (happy path) и альтернативные/исключительные потоки. В зависимости от методологии разработки эти сценарии могут фиксироваться в разных документах:

- **Use Case Specification / Use Case Document** — отдельный документ, целиком посвящённый детальному описанию сценариев использования. Содержит для каждого use case: название, актора(ов), предусловия, основной поток шагов, альтернативные потоки, постусловия, бизнес-правила, связанные требования.
- **Software Requirements Specification (SRS) / Functional Requirements Specification (FRS)** — более общий документ функциональных требований, где Use Cases могут быть включены как раздел наряду с функциональными и нефункциональными требованиями.
- **User Story и Acceptance Criteria** — в Agile-командах формальные Use Case часто заменяются user story ("Как [роль], я хочу [действие], чтобы [цель]") с критериями приёмки, которые фактически выполняют ту же роль, но в более лёгком формате.
- **Диаграммы Use Case (UML Use Case Diagram)** — визуальное представление акторов и сценариев, обычно сопровождает текстовое описание и показывает связи между use cases (include, extend).

Для тестировщика Use Case Specification — один из ключевых источников для написания тест-кейсов, особенно приёмочных и сценарных (end-to-end), поскольку он явно описывает бизнес-логику и альтернативные ветки поведения системы, которые нужно покрыть тестами.

## Пример / когда применяется

Пример структуры Use Case "Оформление заказа":

| Раздел | Содержание |
|---|---|
| Актор | Зарегистрированный пользователь |
| Предусловие | В корзине есть хотя бы один товар |
| Основной поток | 1. Перейти к оформлению 2. Указать адрес доставки 3. Выбрать способ оплаты 4. Подтвердить заказ |
| Альтернативный поток | Оплата картой отклонена банком → показать ошибку, вернуть на шаг выбора оплаты |
| Постусловие | Заказ создан, статус «Ожидает оплаты» |

Такой документ используется на этапе анализа требований и служит основой для дизайна тест-кейсов приёмочного и системного тестирования.

## На что смотрит интервьюер

Интервьюер проверяет, знает ли кандидат, где на практике искать бизнес-сценарии и как соотносятся между собой SRS, Use Case Specification и User Story в разных методологиях. Красный флаг — если кандидат не может назвать ни одного конкретного документа или путает Use Case с тест-кейсом. Возможные follow-up вопросы: "Чем Use Case отличается от User Story?", "Как вы поступите, если Use Case Specification отсутствует в проекте?", "Что такое include/extend отношения между use cases?"
