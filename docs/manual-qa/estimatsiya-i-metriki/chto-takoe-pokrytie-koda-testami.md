# Что такое покрытие кода тестами?

**Коротко:** покрытие кода тестами (Code Coverage) — это метрика, показывающая, какая доля исходного кода приложения была выполнена (затронута) при прогоне автоматизированных тестов, обычно выражается в процентах.

## Развёрнутый ответ

Code Coverage измеряет, насколько тесты "касаются" реального кода программы во время выполнения. Метрика собирается специальными инструментами (например, JaCoCo для Java, Istanbul/nyc для JavaScript, coverage.py для Python), которые инструментируют код и фиксируют, какие его части были выполнены при прогоне тестового набора.

Существует несколько видов покрытия, различающихся по степени детализации:
- **Покрытие строк (Line Coverage)** — процент строк кода, выполненных хотя бы один раз;
- **Покрытие операторов/инструкций (Statement Coverage)** — похоже на line coverage, но учитывает отдельные инструкции, а не физические строки;
- **Покрытие ветвей (Branch Coverage)** — процент выполненных ветвей условных конструкций (if/else, switch); строже, чем line coverage, так как строка может быть выполнена, но не все её ветви проверены;
- **Покрытие условий (Condition Coverage)** — проверяет, что каждое булево под-условие в сложном выражении принимало оба значения (true/false);
- **Покрытие путей (Path Coverage)** — самый строгий вид, проверяет все возможные комбинации путей выполнения через код; на практике почти недостижим для сложного кода из-за комбинаторного взрыва.

Важно понимать ограничения метрики: высокий процент покрытия НЕ гарантирует отсутствие багов и не означает, что тесты содержательные. Тест может "выполнить" строку кода, но не проверить корректность результата (отсутствие assert'ов) или не покрыть все значимые edge cases. Поэтому Code Coverage — это метрика полноты выполнения кода, а не метрика качества тестирования.

Покрытие кода — это метрика, которая обычно измеряется в рамках модульного (unit) и интеграционного тестирования и относится к так называемому "белому ящику" (white-box testing), так как требует доступа к исходному коду. QA-инженеры (особенно ручные тестировщики без доступа к коду) чаще работают с покрытием требований (Requirements Coverage) — метрикой, показывающей, какая доля функциональных требований покрыта тест-кейсами, что концептуально похоже, но измеряется по-другому.

## Пример / когда применяется

Пример: у модуля обработки платежей 500 строк кода. После прогона unit-тестов инструмент показал, что выполнено 425 строк — Line Coverage = 425/500 = 85%. При этом Branch Coverage составил всего 60%, потому что многие ветви обработки ошибок (например, недостаточно средств, недоступность платёжного шлюза) не были протестированы.

Code Coverage применяется как метрика качества автоматизированных unit/integration тестов в CI/CD-пайплайне (часто задаётся минимальный порог, например 80%, ниже которого сборка блокируется), а также при код-ревью и аудите тестового покрытия перед релизом критичного функционала.

## На что смотрит интервьюер

Интервьюер проверяет понимание того, что 100% покрытия не значит "нет багов", и что кандидат различает виды покрытия (line, branch, path). Хороший ответ включает предупреждение о "ложном чувстве безопасности" от высокого процента покрытия без содержательных проверок. Красный флаг — если кандидат считает, что покрытие кода — это то же самое, что покрытие требований, или что 100% покрытие гарантирует отсутствие дефектов. Частые follow-up: "Чем branch coverage отличается от line coverage?", "Может ли покрытие 100% сосуществовать с багами в проде?".
