# Чем куки отличаются от **кэша** браузера?

**Коротко:** Куки — это небольшие данные ключ-значение, которые сервер использует для идентификации пользователя и состояния сессии, а кэш браузера — это сохранённые копии ресурсов (HTML, CSS, JS, изображения), нужные для ускорения загрузки страниц.

## Развёрнутый ответ

Куки и кэш решают принципиально разные задачи и управляются разными механизмами протокола HTTP.

**Куки:**
- Назначение — хранение состояния между запросами (аутентификация, сессии, персонализация, трекинг).
- Управляются заголовками `Set-Cookie` (сервер → браузер) и `Cookie` (браузер → сервер).
- Содержат небольшой объём данных (обычно до 4 КБ на куку).
- Автоматически отправляются с каждым запросом на соответствующий домен.
- Имеют явный срок жизни, атрибуты безопасности (`HttpOnly`, `Secure`, `SameSite`).

**Кэш браузера:**
- Назначение — сохранение копий статических (и иногда динамических) ресурсов, чтобы не загружать их заново по сети, ускоряя работу сайта и снижая нагрузку на сервер.
- Управляется заголовками `Cache-Control`, `Expires`, `ETag`, `Last-Modified`.
- Может хранить большие объёмы данных (файлы, картинки, скрипты, целые страницы).
- Не отправляется на сервер как есть; вместо этого браузер либо использует закэшированную копию без запроса вовсе, либо делает условный запрос (`If-None-Match`/`If-Modified-Since`) и сервер отвечает `304 Not Modified`, если ресурс не изменился.
- Не связан с идентификацией пользователя как таковой.

Ключевое отличие с точки зрения тестирования: очистка куки разлогинит пользователя и сбросит настройки сессии, а очистка кэша просто заставит браузер заново скачать ресурсы с сервера — на аутентификацию это обычно не влияет (хотя может повлиять на скорость и на то, увидит ли пользователь устаревшую версию статики).

## Пример / когда применяется

Частый баг-репорт QA: "пользователь видит старую версию сайта после деплоя" — это обычно проблема кэша (нужно проверить заголовки `Cache-Control`, версионирование файлов через хэш в имени), а не куки. И наоборот: "пользователь не может войти повторно после логаута" — это, скорее всего, проблема с куки (например, сессионная кука не была удалена или инвалидирована на сервере).

```text
Cache-Control: max-age=31536000, immutable   ; кэш статики
Set-Cookie: session_id=abc123; HttpOnly       ; кука сессии
```

## На что смотрит интервьюер

- Понимание, что это независимые механизмы, управляемые разными наборами заголовков.
- Умение привести практический пример различия для дебага бага (устаревшая статика vs проблема авторизации).
- Follow-up: "Что произойдёт, если очистить только куки, но не кэш, и наоборот?", "Как в DevTools отдельно очистить куки и кэш?" (Application tab и Network → Disable cache).
- Красный флаг — если кандидат считает куки частью кэша браузера или наоборот.
