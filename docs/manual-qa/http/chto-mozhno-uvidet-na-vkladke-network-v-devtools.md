# Что можно увидеть на вкладке **Network** в DevTools?

**Коротко:** Вкладка Network показывает все сетевые запросы, которые страница отправляет и получает: URL, метод, статус-код, заголовки, тело запроса/ответа, тайминги и объём переданных данных.

## Развёрнутый ответ

Вкладка Network — основной инструмент для анализа сетевого взаимодействия страницы с сервером. По каждому запросу можно посмотреть:

- **Name / Path** — имя ресурса и путь запроса.
- **Status** — HTTP-статус-код ответа (200, 301, 404, 500 и т.д.) — быстро видно все "красные" (ошибочные) запросы.
- **Type** — тип ресурса (`xhr`, `fetch`, `document`, `script`, `stylesheet`, `img`, `font`, `websocket` и т.д.), можно фильтровать по типу.
- **Initiator** — что инициировало запрос (какой скрипт, какая строка кода или пользовательское действие).
- **Size** — объём переданных данных (отдельно передаваемый по сети и распакованный).
- **Time** — время выполнения запроса (полезно для поиска "медленных" запросов).
- **Waterfall** — визуальная диаграмма таймингов запроса: DNS lookup, установка соединения, TTFB (Time To First Byte), скачивание содержимого.

При клике на конкретный запрос открываются под-вкладки:
- **Headers** — заголовки запроса и ответа, статус-строка, general-информация (URL, метод, remote address).
- **Payload/Request** — тело отправленного запроса (например, JSON тела POST-запроса) и query-параметры.
- **Response** — тело ответа сервера.
- **Preview** — форматированное (например, для JSON — древовидное) представление ответа.
- **Timing** — детальная разбивка по времени (Queueing, DNS, Connecting, TTFB, Content Download).
- **Cookies** — куки, отправленные с запросом и полученные в ответе.

Полезные функции: фильтрация запросов (по типу, по домену, по статусу через поиск), throttling (эмуляция медленной сети), "Preserve log" (сохранение лога при переходах/редиректах), "Disable cache" (принудительный обход кэша при тестировании), экспорт в HAR-файл для передачи разработчикам.

## Пример / когда применяется

Пример практического использования: тестировщик проверяет форму регистрации. После отправки формы он открывает вкладку Network, находит POST-запрос к `/api/register`, проверяет:
- Статус-код ответа (должен быть 200/201, а не 500 или 400).
- Тело запроса (Payload) — правильно ли сформированы данные (email, пароль в правильном формате, нет ли утечки пароля в открытом виде в логах).
- Тело ответа (Response) — что вернул сервер (токен, сообщение об ошибке).
- Заголовки — есть ли корректные CORS-заголовки, куки сессии.

```text
POST /api/register HTTP/1.1
Content-Type: application/json

{"email": "test@example.com", "password": "***"}

Response: 201 Created
{"userId": 42, "token": "eyJhbGciOi..."}
```

## На что смотрит интервьюер

- Умение перечислить конкретные под-вкладки (Headers, Payload, Response, Timing) и объяснить, зачем нужна каждая.
- Понимание разницы между статус-кодами и типами ресурсов (например, что XHR/fetch отличаются от загрузки document/img).
- Follow-up: "Как найти самый медленный запрос на странице?", "Как экспортировать сетевой лог для разработчика?" (Save as HAR), "Как проверить, что запрос ушёл с правильными куки/заголовками авторизации?".
- Красный флаг — если кандидат не может объяснить, где посмотреть тело запроса/ответа или статус-код ошибки.
