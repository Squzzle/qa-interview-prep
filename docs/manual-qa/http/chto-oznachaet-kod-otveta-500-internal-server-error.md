# Что означает код ответа **500 Internal Server Error**?

**Коротко:** `500 Internal Server Error` означает, что на стороне сервера произошла непредвиденная ошибка, из-за которой он не смог обработать в целом корректный запрос.

## Развёрнутый ответ

`500` — общий код группы серверных ошибок (5xx). Он сигнализирует, что проблема не в запросе клиента (в отличие от кодов 4xx), а в самом сервере: необработанное исключение, ошибка в коде бэкенда, недоступность базы данных, деление на ноль, переполнение памяти и так далее. Это своего рода "универсальная" ошибка сервера, когда более конкретный код 5xx неприменим или не был реализован.

Важные отличия от других серверных кодов:

- `502 Bad Gateway` — сервер, выступающий как прокси или шлюз, получил некорректный ответ от вышестоящего сервера.
- `503 Service Unavailable` — сервер временно не может обработать запрос (перегрузка, обслуживание), обычно с указанием `Retry-After`.
- `504 Gateway Timeout` — прокси/шлюз не дождался ответа от вышестоящего сервера за разумное время.

`500` для тестировщика — это всегда сигнал бага, который нужно детально задокументировать: обязательно приложить сам запрос (метод, URL, заголовки, тело), время возникновения и, если возможно, стек-трейс или логи с бэкенда — это критично для разработчиков при отладке.

## Пример / когда применяется

Пример: сервер не смог подключиться к базе данных при обработке запроса.

```text
GET /users/123 HTTP/1.1
Host: api.example.com

HTTP/1.1 500 Internal Server Error
Content-Type: application/json

{
  "error": "Internal server error, please try again later"
}
```

Такой ответ, полученный тестировщиком на любом эндпоинте — всегда основание для бага, независимо от корректности исходного запроса, так как клиент никогда не должен вызывать необработанное падение сервера.

## На что смотрит интервьюер

Смотрят, понимает ли кандидат, что `500` — это всегда баг на стороне сервера, требующий репорта с логами, и умеет ли отличать `500` от `502`/`503`/`504`. Красный флаг — попытка "починить" `500` изменением запроса на стороне клиента вместо эскалации разработчикам; это может временно "спрятать" проблему, но не решает первопричину.
