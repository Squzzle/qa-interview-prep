# Что такое **CORS (Cross-Origin Resource Sharing)**? Какие проблемы он решает?

**Коротко:** CORS — это механизм браузерной безопасности, который позволяет серверу явно разрешить запросы с других доменов (origin), обходя ограничение Same-Origin Policy, при этом не отключая защиту полностью.

## Развёрнутый ответ

По умолчанию браузеры применяют политику Same-Origin Policy (SOP): скрипт, загруженный с одного origin (комбинация протокол + домен + порт), не может свободно читать ответы от запросов к другому origin. Это защищает пользователя от того, чтобы вредоносный сайт из вкладки А не смог тайком обратиться к API банка (сайт Б), используя куки пользователя, и прочитать конфиденциальные данные.

Проблема в том, что легитимных случаев межсайтовых запросов очень много: фронтенд на `app.example.com` должен обращаться к `api.example.com`, CDN, сторонним виджетам и т.д. CORS решает эту проблему — это набор HTTP-заголовков, с помощью которых сервер сообщает браузеру, каким origin'ам, с какими методами и заголовками разрешено обращаться к его ресурсам.

Важно понимать: CORS — это не защита сервера, а инструмент, который сервер использует, чтобы разрешить браузеру показать ответ странице. Сам запрос на сервер в большинстве случаев всё равно уходит (кроме preflight-блокировки) — CORS ограничивает именно то, может ли JS-код на странице прочитать результат. Проверка происходит на стороне браузера, а не сервера — curl или Postman не подчиняются CORS вообще.

Есть два типа запросов:
- **Simple request** — GET/POST/HEAD с ограниченным набором заголовков и типов контента (`text/plain`, `application/x-www-form-urlencoded`, `multipart/form-data`). Такой запрос отправляется сразу, а браузер проверяет ответные заголовки уже после получения ответа.
- **Preflight request** — если запрос "непростой" (например, метод PUT/DELETE, кастомные заголовки типа `Authorization`, тип `application/json`), браузер сначала отправляет служебный запрос методом `OPTIONS`, чтобы узнать, разрешён ли основной запрос, и только потом отправляет реальный запрос.

## Пример / когда применяется

Типичный сценарий: SPA на `https://frontend.com` делает fetch к `https://api.backend.com/users`.

```text
fetch('https://api.backend.com/users', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: 'Ivan' })
})
```

Так как это не simple request (Content-Type application/json), браузер сначала отправит:

```text
OPTIONS /users HTTP/1.1
Origin: https://frontend.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type
```

Сервер должен ответить, например:

```text
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://frontend.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type
```

Если заголовков нет или origin не совпадает — браузер заблокирует доступ к ответу и в консоли появится ошибка вида "has been blocked by CORS policy".

## На что смотрит интервьюер

- Понимание, что CORS реализуется и проверяется браузером, а не сервером, и что это не аналог файрвола.
- Знание разницы между simple и preflight-запросами.
- Частая ошибка кандидатов: путают CORS с CSRF или считают, что `Access-Control-Allow-Origin: *` решает все проблемы (на самом деле с `*` нельзя передавать credentials/куки).
- Хороший ответ включает упоминание, что сам запрос на сервер может выполниться (например, POST с side-effect), даже если браузер потом заблокирует чтение ответа — поэтому CORS не защищает от CSRF-атак сам по себе.
- Возможные follow-up: "Как обойти CORS при разработке?" (прокси, `Access-Control-Allow-Origin` на бэкенде), "Что такое preflight caching через `Access-Control-Max-Age`?".
