# Что такое **CRUD** и как он соотносится с HTTP-методами?

**Коротко:** CRUD (Create, Read, Update, Delete) — базовые операции над данными, которые в REST API стандартно соотносятся с методами POST, GET, PUT/PATCH и DELETE соответственно.

## Развёрнутый ответ

CRUD — это акроним, описывающий четыре базовые операции работы с данными в любой системе хранения (база данных, файловая система, API):

- **Create** — создание новой записи/ресурса;
- **Read** — чтение/получение записи или списка записей;
- **Update** — обновление существующей записи;
- **Delete** — удаление записи.

В контексте REST API эти операции традиционно отображаются на HTTP-методы:

| CRUD-операция | HTTP-метод | Пример |
|---|---|---|
| Create | POST | `POST /api/users` — создать пользователя |
| Read | GET | `GET /api/users/1` — получить пользователя |
| Update | PUT / PATCH | `PUT /api/users/1` — заменить полностью; `PATCH /api/users/1` — обновить частично |
| Delete | DELETE | `DELETE /api/users/1` — удалить пользователя |

Важные нюансы:

- Соответствие CRUD и HTTP-методов — это соглашение (best practice REST), а не жёсткое правило протокола. Ничто технически не запрещает реализовать создание ресурса через GET, но это будет нарушением семантики HTTP и REST-принципов (GET должен быть безопасным и не менять состояние).
- Update может быть выражен как PUT (полная замена) или PATCH (частичное обновление) — важно понимать разницу и выбирать корректный метод в зависимости от того, передаются ли все поля ресурса или только изменяемые.
- Не любой API является CRUD-ориентированным: часто требуются дополнительные операции (например, «отправить письмо», «активировать аккаунт»), которые не укладываются напрямую в CRUD и реализуются как отдельные POST-эндпоинты с действием в пути (`POST /api/users/1/activate`).

## Пример / когда применяется

Тестирование CRUD-функциональности API интернет-магазина для сущности «товар»:

1. `POST /api/products` — создать товар, проверить код 201 и наличие созданного объекта в теле ответа.
2. `GET /api/products/{id}` — получить товар по id, проверить код 200 и соответствие данных.
3. `PUT /api/products/{id}` — обновить товар полностью, проверить, что все поля заменены.
4. `DELETE /api/products/{id}` — удалить товар, затем повторным GET убедиться, что возвращается 404.

## На что смотрит интервьюер

- Понимание, что CRUD — концепция уровня данных, а HTTP-методы — уровня протокола, и их связь — это соглашение, а не строгая спецификация.
- Умение чётко сопоставить каждую CRUD-операцию с методом.
- Различение PUT и PATCH при операции Update.
- Follow-up: «Как бы вы протестировали полный CRUD-цикл для ресурса?», «Что делать, если API не следует REST-соглашениям (например, использует POST для всего)?».
