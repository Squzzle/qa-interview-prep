# Что такое **куки (Cookies)** и для чего они используются?

**Коротко:** Куки — это небольшие фрагменты данных в формате ключ-значение, которые сервер сохраняет в браузере пользователя и которые автоматически отправляются обратно серверу с каждым HTTP-запросом к тому же домену.

## Развёрнутый ответ

HTTP — протокол без сохранения состояния (stateless): каждый запрос независим, сервер не помнит предыдущие запросы того же клиента. Куки были придуманы как способ решить эту проблему — добавить "память" поверх stateless-протокола.

Сервер устанавливает куки через заголовок ответа `Set-Cookie`, а браузер автоматически прикладывает их к каждому последующему запросу на этот домен через заголовок `Cookie`. Куки могут иметь атрибуты, управляющие их поведением и безопасностью:

- `Expires` / `Max-Age` — срок жизни куки.
- `Domain` и `Path` — область действия (для каких доменов/путей куки будут отправляться).
- `Secure` — куки отправляются только по HTTPS.
- `HttpOnly` — куки недоступны из JavaScript (`document.cookie`), что защищает от кражи через XSS.
- `SameSite` (`Strict`, `Lax`, `None`) — ограничивает отправку куки в кросс-сайтовых запросах, защищая от CSRF.

Основные сферы применения куки:
- **Аутентификация и сессии** — хранение session ID или токена, чтобы сервер узнавал пользователя между запросами (например, после логина).
- **Персонализация** — сохранение настроек пользователя (язык, тема).
- **Трекинг и аналитика** — отслеживание поведения пользователя между визитами (в том числе сторонними сервисами — third-party cookies).
- **A/B тестирование** — сохранение того, в какую тестовую группу попал пользователь.

Важное отличие от Web Storage: куки автоматически передаются на сервер с каждым запросом (что увеличивает объём трафика при больших куки), тогда как LocalStorage/SessionStorage доступны только через JS и никогда не уходят на сервер сами по себе.

## Пример / когда применяется

Пример установки куки сервером:

```text
Set-Cookie: session_id=abc123; Max-Age=3600; Path=/; HttpOnly; Secure; SameSite=Lax
```

Далее браузер будет автоматически отправлять эту куку с каждым запросом на тот же домен:

```text
Cookie: session_id=abc123
```

Практический QA-кейс: при тестировании авторизации важно проверить, что сессионная кука имеет флаги `HttpOnly` и `Secure`, а также что после логаута сервер реально инвалидирует сессию (а не просто удаляет куку на клиенте).

## На что смотрит интервьюер

- Понимание разницы между session cookie (без `Expires`, живёт до закрытия браузера) и persistent cookie (с явным сроком жизни).
- Знание атрибутов безопасности `HttpOnly`, `Secure`, `SameSite` и того, от каких атак они защищают (XSS, CSRF, MITM).
- Follow-up: "Что такое third-party cookies и почему браузеры их блокируют?", "Как проверить куки в DevTools?" (Application → Cookies).
- Красный флаг — если кандидат путает куки с кэшем браузера или не может объяснить механизм `Set-Cookie`/`Cookie`.
