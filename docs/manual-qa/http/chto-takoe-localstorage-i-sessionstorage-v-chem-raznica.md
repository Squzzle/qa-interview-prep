# Что такое **LocalStorage** и **SessionStorage**? В чем разница?

**Коротко:** Оба API относятся к Web Storage и позволяют хранить пары ключ-значение в браузере, но LocalStorage хранит данные без срока действия, а SessionStorage — только на время жизни вкладки.

## Развёрнутый ответ

`LocalStorage` и `SessionStorage` — это части спецификации Web Storage API, доступные через `window.localStorage` и `window.sessionStorage`. Оба хранят данные только в виде строк (ключ-значение), объём хранения ограничен (обычно около 5-10 МБ на origin, зависит от браузера), и данные никогда автоматически не отправляются на сервер (в отличие от куки).

Основные различия:

| Характеристика | LocalStorage | SessionStorage |
|---|---|---|
| Время жизни | Бессрочно, пока не удалят вручную или через очистку данных браузера | До закрытия вкладки/окна |
| Область видимости | Общий для всех вкладок и окон одного origin | Отдельный для каждой вкладки, даже если открыт тот же URL |
| Копирование при дублировании вкладки | Не относится | При дублировании вкладки (Ctrl+клик по ссылке, duplicate tab) данные копируются в новую вкладку |
| Отправка на сервер | Нет | Нет |
| Доступ | Синхронный, через JS | Синхронный, через JS |

Оба хранилища доступны только на клиенте, работают в рамках Same-Origin Policy (данные изолированы по origin), и не подходят для хранения чувствительных данных (токенов, паролей), потому что уязвимы для XSS: любой JS-код, выполненный на странице (в том числе внедрённый через XSS), имеет полный доступ к обоим хранилищам.

Также важно понимать разницу с куки: Storage API не имеет атрибутов `Expires`, `HttpOnly`, `Secure`, `SameSite` — то есть его нельзя защитить от чтения через JavaScript, и он не участвует в HTTP-заголовках запроса автоматически.

## Пример / когда применяется

```text
// LocalStorage — сохраняем настройки темы, они должны сохраняться между сессиями
localStorage.setItem('theme', 'dark');
const theme = localStorage.getItem('theme');

// SessionStorage — временные данные многошаговой формы в рамках одной вкладки
sessionStorage.setItem('formStep', '2');
sessionStorage.removeItem('formStep');
```

Типичный кейс использования LocalStorage — кэширование пользовательских настроек или черновиков; SessionStorage — хранение состояния мастера (wizard) или данных, которые не должны "утекать" между вкладками (например, состояние одной конкретной сессии оформления заказа).

## На что смотрит интервьюер

- Знание, что оба хранилища уязвимы для XSS и не должны использоваться для JWT/токенов без дополнительных мер (или что для токенов предпочтительнее HttpOnly cookies).
- Понимание области видимости SessionStorage по вкладкам, а не по origin в целом — частая ошибка кандидатов путать его с LocalStorage в этом плане.
- Follow-up вопросы: "Как это тестируется QA-инженером?" (проверка через DevTools → Application → Storage), "Что произойдёт с данными при приватном/инкогнито режиме?".
- Красный флаг — если кандидат утверждает, что данные Storage API отправляются на сервер автоматически (это не так, в отличие от куки).
