# Что такое **методы HTTP**? Перечислите основные.

**Коротко:** методы HTTP (HTTP verbs) — это глаголы, определяющие действие, которое клиент хочет выполнить над ресурсом; основные — GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS.

## Развёрнутый ответ

Метод HTTP указывается в стартовой строке запроса и сообщает серверу, какое действие требуется выполнить над ресурсом, определённым в URI. Основные методы:

- **GET** — получить представление ресурса. Не должен изменять состояние сервера (безопасный, идемпотентный метод). Тело запроса, как правило, отсутствует.
- **POST** — создать новый ресурс или инициировать обработку данных (например, отправка формы, выполнение операции). Не идемпотентен: повторный вызов может создать несколько ресурсов.
- **PUT** — полностью заменить существующий ресурс данными из тела запроса (или создать его по указанному URI, если не существует). Идемпотентен: повторный вызов с теми же данными даёт тот же результат.
- **PATCH** — частично обновить ресурс, изменив только указанные поля. В общем случае не гарантированно идемпотентен (зависит от реализации).
- **DELETE** — удалить указанный ресурс. Считается идемпотентным (повторное удаление уже удалённого ресурса не меняет состояния, хотя может вернуть другой код статуса).
- **HEAD** — аналог GET, но сервер возвращает только заголовки без тела ответа. Используется для проверки существования ресурса, получения метаданных (например, `Content-Length`) без загрузки самого содержимого.
- **OPTIONS** — запрашивает у сервера список поддерживаемых методов и заголовков для ресурса; широко используется в CORS как preflight-запрос перед «нестандартными» кросс-доменными запросами.
- **TRACE** и **CONNECT** — используются реже: TRACE возвращает эхо полученного запроса для диагностики, CONNECT устанавливает туннель (например, для HTTPS через прокси).

## Пример / когда применяется

Пример REST API для управления пользователями:

| Метод | URI | Действие |
|---|---|---|
| GET | `/api/users` | Получить список пользователей |
| GET | `/api/users/1` | Получить пользователя с id=1 |
| POST | `/api/users` | Создать нового пользователя |
| PUT | `/api/users/1` | Полностью обновить пользователя с id=1 |
| PATCH | `/api/users/1` | Частично обновить (например, только email) |
| DELETE | `/api/users/1` | Удалить пользователя с id=1 |

При тестировании QA-инженер должен проверять не только «счастливый путь», но и корректность обработки методов, не поддерживаемых для конкретного ресурса (сервер должен вернуть 405 Method Not Allowed).

## На что смотрит интервьюер

- Умение перечислить методы и корректно описать их назначение, а не просто зазубренный список.
- Понимание разницы между PUT и PATCH (полная замена vs частичное обновление) — частый вопрос-ловушка.
- Знание роли OPTIONS в CORS preflight-запросах.
- Follow-up: «Что вернёт сервер, если вызвать неподдерживаемый метод?», «Чем отличается PUT от POST при создании ресурса по конкретному URI?».
