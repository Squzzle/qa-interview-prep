# Что такое **публичный**, **приватный** и **сессионный** ключи?

**Коротко:** публичный и приватный ключи — пара для асимметричного шифрования (публичный шифрует/проверяет, приватный расшифровывает/подписывает), а сессионный ключ — общий симметричный ключ, который стороны получают после handshake и используют для шифрования данных в рамках одного соединения.

## Развёрнутый ответ

**Публичный ключ (public key)** — часть асимметричной криптографической пары, которая распространяется открыто, например встроена в SSL/TLS-сертификат сервера. Им можно зашифровать данные так, что расшифровать их сможет только владелец соответствующего приватного ключа, либо проверить цифровую подпись, созданную приватным ключом.

**Приватный ключ (private key)** — вторая часть пары, которая хранится в секрете владельцем (сервером). С его помощью расшифровываются данные, зашифрованные публичным ключом, либо создаётся цифровая подпись, которую любой может проверить публичным ключом. Компрометация приватного ключа означает полную компрометацию защиты — атакующий сможет расшифровывать трафик или выдавать себя за сервер.

Асимметричное шифрование (RSA, ECC) криптографически надёжно, но вычислительно затратно и медленно для передачи больших объёмов данных, поэтому оно используется только на этапе TLS handshake — для аутентификации сервера и согласования общего секрета.

**Сессионный ключ (session key)** — симметричный ключ, который обе стороны получают в результате handshake (через Diffie-Hellman/ECDHE или RSA-обмен). Он используется для быстрого симметричного шифрования (AES, ChaCha20) всего последующего трафика в рамках текущего TLS-соединения/сессии. Сессионный ключ временный: генерируется заново для каждой новой сессии (а в схемах с Perfect Forward Secrecy — уникален и не восстанавливаем даже при компрометации приватного ключа сервера в будущем).

Кратко разница по ролям:

- Публичный/приватный ключ — постоянная пара, привязанная к сертификату/владельцу, используется для аутентификации и первоначального обмена ключами.
- Сессионный ключ — временный, генерируется на каждое соединение, используется для быстрого симметричного шифрования основного трафика.

## Пример / когда применяется

Пример из жизни: сервер интернет-магазина имеет сертификат с публичным ключом, который браузер использует, чтобы зашифровать pre-master secret на этапе handshake — расшифровать его сможет только сервер своим приватным ключом. Из этого секрета обе стороны независимо вычисляют один и тот же сессионный ключ, которым дальше шифруются HTTP-запросы (логин, номер карты и т.д.) — уже быстрым симметричным алгоритмом AES.

## На что смотрит интервьюер

- Понимание, что асимметричное шифрование используется для обмена ключами/аутентификации, а не для шифрования всего трафика.
- Чёткое различие «постоянный ключ, привязанный к сертификату» vs «временный ключ на сессию».
- Знание термина Perfect Forward Secrecy и зачем он нужен.
- Красный флаг — если кандидат путает публичный ключ с сертификатом или не понимает, зачем вообще нужен сессионный ключ, если уже есть асимметричная пара.
