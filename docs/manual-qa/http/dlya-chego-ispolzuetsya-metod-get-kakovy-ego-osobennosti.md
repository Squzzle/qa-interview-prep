# Для чего используется метод **GET**? Каковы его особенности?

**Коротко:** GET используется для получения (чтения) ресурса без изменения состояния сервера; он безопасен, идемпотентен, может кэшироваться и передаёт параметры в URL, а не в теле.

## Развёрнутый ответ

Метод GET предназначен исключительно для получения данных от сервера — например, HTML-страницы, JSON-объекта, файла. Основные особенности:

- **Безопасность (safe)** — GET не должен изменять состояние сервера. Это семантическое соглашение спецификации HTTP: сервер не обязан технически это гарантировать, но нарушение этого правила считается плохой практикой (например, удаление ресурса через GET-запрос — грубая ошибка проектирования API).
- **Идемпотентность** — повторное выполнение одного и того же GET-запроса не меняет состояние сервера и возвращает тот же результат (при условии, что данные на сервере не изменились между запросами по другой причине).
- **Кэшируемость** — GET-ответы могут кэшироваться браузером, прокси, CDN с помощью заголовков `Cache-Control`, `ETag`, `Last-Modified`, что существенно снижает нагрузку на сервер и ускоряет отдачу повторяющихся запросов.
- **Параметры в URL** — данные передаются через query-параметры (`?key=value&key2=value2`), а не в теле запроса. Это делает GET-запросы видимыми в истории браузера, логах сервера, закладках — поэтому GET нельзя использовать для передачи чувствительных данных (паролей, токенов).
- **Ограничение длины** — большинство браузеров и серверов ограничивают длину URL (обычно 2000-8000 символов в зависимости от реализации), поэтому GET не подходит для передачи больших объёмов данных.
- **Отсутствие тела** — спецификация формально не запрещает тело в GET-запросе, но многие серверы, прокси и библиотеки игнорируют или отбрасывают его — на практике тело в GET не используется.

## Пример / когда применяется

Типичный пример — получение списка товаров с фильтрацией и пагинацией:

```text
GET /api/products?category=electronics&page=2&limit=20 HTTP/1.1
Host: shop.example.com
```

При тестировании QA-инженер проверяет: корректную обработку query-параметров (включая граничные и некорректные значения), соблюдение кэширования (заголовки `ETag`/`If-None-Match`, ожидание 304 Not Modified при повторном запросе с тем же ETag), а также отсутствие изменений на сервере при повторных GET-запросах.

## На что смотрит интервьюер

- Понимание, что GET не должен модифицировать данные — и умение объяснить, почему нарушение этого правила является антипаттерном.
- Знание, что параметры GET видны в URL и потому не подходят для передачи секретов.
- Осведомлённость о кэшировании GET-запросов.
