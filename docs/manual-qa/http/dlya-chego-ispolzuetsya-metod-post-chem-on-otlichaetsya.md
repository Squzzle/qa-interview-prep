# Для чего используется метод **POST**? Чем он отличается от GET?

**Коротко:** POST используется для создания ресурса или отправки данных, изменяющих состояние сервера; данные передаются в теле запроса, метод не идемпотентен и не кэшируется по умолчанию, в отличие от GET.

## Развёрнутый ответ

Метод POST предназначен для отправки данных на сервер с целью создания нового ресурса, выполнения действия или обработки данных (например, отправка формы, загрузка файла, инициирование платежа). Ключевые отличия POST от GET:

| Критерий | GET | POST |
|---|---|---|
| Назначение | Получение данных | Создание ресурса / отправка данных |
| Расположение данных | Query-параметры в URL | Тело запроса (body) |
| Идемпотентность | Идемпотентен | Не идемпотентен |
| Безопасность (safe) | Безопасен | Не безопасен (изменяет состояние) |
| Кэширование | Кэшируется по умолчанию | Не кэшируется по умолчанию |
| Ограничение объёма данных | Ограничен длиной URL | Практически не ограничен (зависит от настроек сервера) |
| Видимость данных | В URL, истории браузера, логах | Скрыты в теле (но не зашифрованы без HTTPS) |

Важные нюансы:

- **Не идемпотентность POST** означает, что повторная отправка одного и того же POST-запроса, как правило, создаёт новый ресурс/эффект (например, повторная отправка формы оплаты может создать два платежа). Это критично тестировать — особенно защиту от двойного клика или повторной отправки формы.
- POST может передавать данные в разных форматах: `application/json`, `application/x-www-form-urlencoded`, `multipart/form-data` (для загрузки файлов).
- POST-запросы не сохраняются в истории браузера и не кэшируются стандартно, что делает их подходящими для передачи чувствительных данных (в сочетании с HTTPS).

## Пример / когда применяется

Форма регистрации пользователя реализуется через POST, так как создаёт новую запись и передаёт чувствительные данные (email, пароль):

```text
POST /api/register HTTP/1.1
Host: example.com
Content-Type: application/json
Content-Length: 58

{"email": "user@example.com", "password": "SecurePass1!"}
```

При тестировании QA-инженер проверяет: обработку повторной отправки (не создаётся ли дубликат при двойном клике), корректную валидацию данных в теле, правильные коды ответа (201 при успешном создании, 400 при некорректных данных, 409 при конфликте — например, email уже занят).

## На что смотрит интервьюер

- Чёткое понимание, что POST не идемпотентен, а GET идемпотентен — базовое, но часто путаемое различие.
- Знание, что POST передаёт данные в теле, а не в URL.
- Понимание, что POST-запросы по умолчанию не кэшируются.
