# Как с помощью DevTools эмулировать медленное соединение?

**Коротко:** Во вкладке Network в DevTools есть выпадающий список Throttling (по умолчанию "No throttling"), позволяющий выбрать пресет скорости сети (например, "Slow 3G", "Fast 3G") или создать собственный кастомный профиль с заданной скоростью и задержкой.

## Развёрнутый ответ

DevTools предоставляет встроенный механизм network throttling — искусственное ограничение скорости сети и добавление задержки (latency) для симуляции условий, в которых работают реальные пользователи с плохим интернетом (мобильная сеть, слабый Wi-Fi).

Шаги для эмуляции:
1. Открыть DevTools (F12 или Cmd+Option+I на Mac).
2. Перейти на вкладку **Network**.
3. Найти выпадающий список throttling (обычно рядом с чекбоксом "Disable cache", подписан "No throttling" по умолчанию).
4. Выбрать один из готовых пресетов: **Slow 3G**, **Fast 3G**, **Offline** — либо открыть **Add** / "Custom" и задать собственные параметры: download/upload скорость (в Kbps/Mbps) и latency (задержка в мс).
5. Обновить страницу или выполнить действие — теперь все сетевые запросы будут искусственно замедлены согласно выбранному профилю.

Также throttling можно применить к CPU (отдельная настройка "CPU throttling" на вкладке Performance) — это симулирует не только медленную сеть, но и слабое устройство, что важно для комплексной проверки производительности на низкопроизводительных телефонах.

Важный нюанс: throttling в DevTools эмулирует сеть только для запросов, инициированных именно этой вкладкой браузера (в контексте открытых DevTools), и не идеально точен (это программная эмуляция, а не реальное ограничение канала), но для функционального и UX-тестирования этого обычно достаточно.

## Пример / когда применяется

Практические случаи применения:
- Проверка, показывает ли приложение индикатор загрузки (loader/spinner) при медленном ответе сервера, а не просто "зависает" без обратной связи.
- Проверка обработки таймаутов — что происходит, если запрос выполняется дольше ожидаемого времени (показывается ли сообщение об ошибке).
- Тестирование прогрессивной загрузки изображений (lazy loading, placeholder) на медленном соединении.
- Проверка сценария "Offline" — корректно ли приложение сообщает пользователю об отсутствии сети (особенно важно для PWA с поддержкой офлайн-режима).

```text
DevTools → Network → Throttling: "Slow 3G"
  Download: ~400 Kbps
  Upload: ~400 Kbps
  Latency: ~2000 ms
```

После выбора профиля стоит перезагрузить страницу с зажатым чекбоксом "Disable cache", чтобы исключить влияние закэшированных ресурсов на результат теста.

## На что смотрит интервьюер

- Знание точного расположения функции (Network tab, выпадающий список throttling), а не общих слов "где-то в настройках".
- Понимание, что throttling можно настроить кастомно (не только готовые пресеты), и что есть отдельный CPU throttling.
- Умение объяснить, зачем это нужно с точки зрения тестирования UX (индикаторы загрузки, обработка ошибок сети, skeleton screens).
- Follow-up: "Чем текущий throttling отличается от реального тестирования на устройстве в плохой сети?", "Как проверить работу приложения в офлайн-режиме?".
- Красный флаг — если кандидат не знает про эту функцию вообще или путает её с ограничением локальной пропускной способности через сторонние программы (что тоже валидный подход, но не то, о чём спрашивается).
