# Какие инструменты **DevTools** вы используете чаще всего?

**Коротко:** QA-инженер чаще всего использует вкладки Elements (инспекция DOM/CSS), Console (ошибки JS), Network (анализ запросов и ответов) и Application (куки, storage), а также режим устройств для проверки адаптивности.

## Развёрнутый ответ

Chrome/Firefox DevTools — основной инструмент ручного тестировщика для анализа поведения веб-приложения без необходимости лезть в код. Ключевые вкладки, которые используются в повседневной работе:

- **Elements (Inspector)** — просмотр и изменение DOM-дерева и CSS "на лету". Используется для проверки вёрстки, поиска несоответствий макету (Figma), проверки атрибутов доступности (aria-*), временного редактирования HTML для проверки гипотез бага.
- **Console** — вывод логов, ошибок и предупреждений JavaScript. Первое место, куда смотрит QA при "странном" поведении страницы — часто там видна необработанная JS-ошибка, которая объясняет баг. Также используется для выполнения произвольного JS-кода (например, `localStorage.clear()`).
- **Network** — все сетевые запросы страницы: статус-коды, время ответа, заголовки, тело запроса/ответа. Ключевой инструмент для проверки API-интеграции, поиска 4xx/5xx ошибок, проверки правильности передаваемых параметров, анализа кэширования.
- **Application (Storage)** — просмотр и редактирование куки, LocalStorage, SessionStorage, IndexedDB, Cache Storage, а также проверка Service Worker и манифеста PWA.
- **Sources** — просмотр исходного кода, установка breakpoints в JS для дебага сложных сценариев (совместно с разработчиками).
- **Performance** и **Lighthouse** — используются реже, но важны для проверки производительности и метрик Core Web Vitals.
- **Device Toolbar (Ctrl+Shift+M)** — эмуляция мобильных устройств и разных разрешений экрана для проверки адаптивной вёрстки.

## Пример / когда применяется

Типичный рабочий процесс QA при расследовании бага "кнопка не работает":
1. Открыть **Console** — проверить, нет ли JS-ошибки, блокирующей обработчик клика.
2. Открыть **Network** — проверить, ушёл ли запрос при клике, какой статус-код вернулся, что в теле ответа.
3. Открыть **Elements** — проверить, действительно ли элемент кликабелен (нет ли невидимого элемента поверх него, `pointer-events: none`, `z-index` конфликтов).
4. Открыть **Application** — проверить, не истекла ли сессия (кука) или не испорчено ли состояние в LocalStorage.

## На что смотрит интервьюер

- Умение назвать не одну, а несколько вкладок и объяснить, для какой задачи используется каждая — это показывает реальный опыт, а не заученный ответ.
- Способность привести конкретный пример бага, который был найден именно через DevTools.
- Follow-up: "Как посмотреть, какие запросы упали с ошибкой 500?", "Как найти элемент на странице по клику мышью?" (иконка "select element" / Ctrl+Shift+C).
- Красный флаг — если кандидат называет только Console и не может объяснить назначение Network или Application.
