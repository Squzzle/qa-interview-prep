# Какие заголовки запроса вы знаете (`Host`, `User-Agent`, `Authorization`)?

**Коротко:** заголовки запроса передают серверу метаданные о клиенте и параметрах запроса — например, целевой домен (`Host`), информацию о клиентском приложении (`User-Agent`) и данные для аутентификации (`Authorization`).

## Развёрнутый ответ

Заголовки запроса (request headers) отправляются клиентом серверу и описывают контекст запроса. Разберём основные:

- **`Host`** — обязательный заголовок в HTTP/1.1, указывающий доменное имя (и опционально порт) целевого сервера. Необходим для виртуального хостинга, когда на одном IP-адресе крутится несколько сайтов, — по значению `Host` веб-сервер понимает, какой из них обслуживать.
- **`User-Agent`** — строка, идентифицирующая клиентское приложение: браузер и его версию, операционную систему, иногда движок рендеринга. Используется для аналитики, адаптации контента под устройство и иногда (небезопасно) для проверок безопасности.
- **`Authorization`** — передаёт учётные данные для аутентификации, например `Authorization: Bearer <token>` для токенов или `Authorization: Basic <base64>` для базовой аутентификации по логину/паролю.

Другие важные заголовки запроса:

- **`Accept`** — какие MIME-типы контента клиент готов принять (`application/json`, `text/html`).
- **`Accept-Language`** — предпочитаемый язык ответа.
- **`Content-Type`** — формат тела запроса (важен для `POST`/`PUT` с телом).
- **`Content-Length`** — размер тела запроса в байтах.
- **`Cookie`** — передаёт cookie, ранее установленные сервером через `Set-Cookie`.
- **`Referer`** — URL страницы, с которой пришёл запрос.
- **`If-None-Match`** / **`If-Modified-Since`** — условные заголовки для кеширования, позволяют серверу вернуть `304 Not Modified`, если ресурс не изменился.

## Пример / когда применяется

```text
GET /api/orders/55 HTTP/1.1
Host: api.example.com
User-Agent: Mozilla/5.0 (Macintosh; Intel Mac OS X 10_15)
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
Accept: application/json
Accept-Language: ru-RU
```

При тестировании API заголовки запроса проверяют в первую очередь для негативных сценариев: запрос без `Authorization`, с некорректным `Content-Type`, с отсутствующим `Host` — сервер должен реагировать предсказуемо и отдавать соответствующий код ошибки (`400`/`401`).

## На что смотрит интервьюер

Проверяют, может ли кандидат назвать заголовки помимо трёх упомянутых в вопросе, и понимает ли разницу между заголовками запроса и ответа. Частая ошибка — путать `Authorization` (передача учётных данных клиентом) с `WWW-Authenticate` (запрос сервера на аутентификацию). Хороший ответ включает понимание условных заголовков (`If-None-Match`) как основы HTTP-кеширования.
