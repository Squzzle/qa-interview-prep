# Какова **структура HTTP-ответа**?

**Коротко:** HTTP-ответ состоит из статусной строки (версия протокола, код статуса, поясняющая фраза), заголовков и опционального тела с данными.

## Развёрнутый ответ

Структура HTTP-ответа зеркальна структуре запроса и включает:

1. **Статусная строка (status line)** — содержит:
   - версию протокола, например `HTTP/1.1`;
   - код статуса (status code) — трёхзначное число, например `200`, `404`, `500`;
   - поясняющую фразу (reason phrase), например `OK`, `Not Found`, `Internal Server Error`.

2. **Заголовки ответа (headers)** — метаданные о самом ответе:
   - `Content-Type` — тип содержимого тела (например, `application/json`, `text/html`);
   - `Content-Length` — размер тела в байтах;
   - `Set-Cookie` — установка cookie на стороне клиента;
   - `Cache-Control`, `ETag`, `Expires` — управление кэшированием;
   - `Location` — используется при редиректах (коды 3xx), указывает новый URL;
   - `Access-Control-Allow-Origin` — заголовок CORS, разрешающий кросс-доменные запросы.

3. **Пустая строка**, отделяющая заголовки от тела.

4. **Тело ответа (body)** — собственно данные: HTML-страница, JSON-объект, файл, изображение и т.д. Тело может отсутствовать (например, при статусе 204 No Content или в ответ на HEAD-запрос).

Коды статуса группируются по первой цифре:

- **1xx** — информационные (Continue, Switching Protocols);
- **2xx** — успех (200 OK, 201 Created, 204 No Content);
- **3xx** — редиректы (301 Moved Permanently, 302 Found, 304 Not Modified);
- **4xx** — ошибка клиента (400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity);
- **5xx** — ошибка сервера (500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout).

## Пример / когда применяется

Пример типичного HTTP-ответа при успешном создании ресурса:

```text
HTTP/1.1 201 Created
Content-Type: application/json
Content-Length: 58
Location: /api/users/42

{"id": 42, "name": "Ivan", "email": "ivan@example.com"}
```

При тестировании API QA-инженер проверяет: соответствие кода статуса ожидаемому поведению (например, не 200, а 201 при создании ресурса), корректность заголовков (`Content-Type` должен совпадать с реальным форматом тела), а также структуру и содержимое JSON-тела ответа.

## На что смотрит интервьюер

- Знание основных групп кодов статуса и умение привести примеры из каждой группы.
- Понимание разницы между 401 (не аутентифицирован) и 403 (аутентифицирован, но нет прав) — частый вопрос-ловушка.
- Знание, что тело может отсутствовать (204, HEAD-ответы).
- Follow-up: «Чем отличается 301 от 302?», «Что означает 304 Not Modified и когда он возникает?», «Почему сервер может вернуть 500 вместо конкретной ошибки?».
