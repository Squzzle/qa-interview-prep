# Опишите процесс **SSL/TLS Handshake**.

**Коротко:** TLS Handshake — это процесс согласования параметров шифрования и обмена ключами между клиентом и сервером, в результате которого стороны получают общий сессионный ключ для дальнейшего симметричного шифрования трафика.

## Развёрнутый ответ

Handshake происходит после установления TCP-соединения (three-way handshake) и до начала передачи прикладных данных. Классический процесс (TLS 1.2) выглядит так:

1. **Client Hello** — клиент отправляет серверу поддерживаемые версии TLS, список алгоритмов шифрования (cipher suites), случайное число (client random) и опционально SNI (Server Name Indication — имя домена, к которому обращается клиент).
2. **Server Hello** — сервер выбирает версию протокола и cipher suite из предложенных клиентом, отправляет своё случайное число (server random).
3. **Сертификат сервера** — сервер отправляет свой SSL/TLS-сертификат, содержащий публичный ключ и подпись центра сертификации (CA). Клиент проверяет сертификат: срок действия, цепочку доверия до корневого CA, соответствие домену.
4. **Обмен ключами** — в зависимости от алгоритма (RSA или, чаще сейчас, Diffie-Hellman/ECDHE) стороны вычисляют общий секрет (pre-master secret), из которого затем деривируется мастер-секрет и сессионные ключи.
5. **Client/Server Finished** — обе стороны отправляют зашифрованное сообщение о завершении handshake, подтверждая, что ключи совпадают и целостность процесса не нарушена.
6. Далее всё общение идёт по **симметричному шифрованию** с использованием согласованного сессионного ключа — это быстрее, чем асимметричное шифрование, которое использовалось только на этапе обмена ключами.

В TLS 1.3 процесс упрощён и ускорен: handshake требует всего один round-trip (1-RTT), а при повторном подключении к уже известному серверу возможен 0-RTT (session resumption), что снижает задержку.

## Пример / когда применяется

При открытии HTTPS-сайта в браузере можно увидеть весь процесс handshake в DevTools (вкладка Network → Timing → SSL). QA-инженер может использовать `openssl s_client -connect example.com:443` для ручной проверки handshake, версии TLS и параметров сертификата:

```bash
openssl s_client -connect example.com:443 -tls1_2
```

Такой запрос покажет цепочку сертификатов, выбранный cipher suite и результат проверки (Verify return code: 0 (ok)).

## На что смотрит интервьюер

- Понимание порядка этапов handshake и роли асимметричного/симметричного шифрования на разных стадиях.
- Знание, что асимметричные ключи используются только для обмена/согласования, а не для шифрования всего трафика (иначе это было бы слишком медленно).
- Осведомлённость об SNI и его роли при виртуальном хостинге нескольких доменов на одном IP.
- Follow-up: «Чем отличается handshake в TLS 1.3 от TLS 1.2?», «Что такое mutual TLS?», «Что произойдёт, если сертификат не подписан доверенным CA?».
