# Что делает команда `ping`?

**Коротко:** `ping` проверяет доступность узла в сети, отправляя ему ICMP-запросы (Echo Request) и измеряя время отклика (Echo Reply), что позволяет быстро определить, доступен ли сервер и насколько стабильна сеть до него.

## Развёрнутый ответ

`ping` — кроссплатформенная утилита (доступна и в Windows, и в Linux/macOS), использующая протокол ICMP (Internet Control Message Protocol) для проверки связи с удалённым узлом. Принцип работы: команда отправляет на целевой хост несколько пакетов Echo Request, а хост, если доступен и не блокирует ICMP, отвечает пакетами Echo Reply.

Что можно узнать из вывода `ping`:

- **Доступен ли хост** — если ответы приходят, узел в сети достижим; если приходит "Request timed out" / "100% packet loss" — узел недоступен или блокирует ICMP.
- **Время отклика (latency)** — round-trip time (RTT) в миллисекундах, показывает задержку сети.
- **Потерю пакетов (packet loss)** — процент пакетов, на которые не пришёл ответ; высокий процент говорит о нестабильности сети.
- **TTL (Time To Live)** — количество "прыжков" (хопов), которое осталось у пакета; помогает косвенно оценить количество сетевых узлов на пути.

Типичные ключи:

- Windows: `ping -t` (непрерывный пинг до Ctrl+C), `ping -n <число>` (количество запросов), `ping -l <размер>` (размер пакета).
- Linux/macOS: `ping -c <число>` (количество запросов), `ping -i <интервал>` (интервал между запросами).

Важный нюанс для тестировщика: отсутствие ответа на `ping` не всегда означает, что сервер недоступен — многие серверы и файрволы намеренно блокируют ICMP-трафик из соображений безопасности, при этом сам сервис (например, HTTP на 80 порту) может работать нормально. Поэтому `ping` — это только первый, грубый шаг диагностики; для точной проверки сервиса используют `telnet`, `curl`, `nc -zv` (проверка конкретного порта).

## Пример / когда применяется

Тестировщик проверяет, почему тестовый стенд недоступен:

```bash
ping test-server.company.com
```

Вывод:

```text
Reply from 10.0.0.5: bytes=32 time=15ms TTL=64
Reply from 10.0.0.5: bytes=32 time=14ms TTL=64
Ping statistics for 10.0.0.5:
    Packets: Sent = 4, Received = 4, Lost = 0 (0% loss)
```

Если сервер не отвечает, следующий шаг — проверить доступность конкретного порта:

```bash
curl -v telnet://test-server.company.com:8080
```

## На что смотрит интервьюер

- Понимание, что `ping` работает через ICMP, а не через TCP/HTTP, и поэтому не гарантирует доступность конкретного сервиса или порта.
- Знание, что отсутствие ответа может быть вызвано блокировкой ICMP файрволом, а не реальной недоступностью сервера — это частый вопрос-ловушка.
- Умение интерпретировать метрики: время отклика, потерю пакетов, TTL.
- Follow-up вопрос: "Сервер не пингуется, но сайт открывается в браузере — в чём может быть причина?" (ответ: ICMP заблокирован на файрволе, а HTTP/HTTPS порты открыты).
