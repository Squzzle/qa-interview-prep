# Как с помощью `curl` отправить POST-запрос с JSON-телом?

**Коротко:** нужно указать метод `-X POST`, заголовок `Content-Type: application/json` через `-H` и передать JSON-тело через `-d` (или `--data`).

## Развёрнутый ответ

Для отправки POST-запроса с телом в формате JSON в `curl` требуется указать три вещи:

1. HTTP-метод: флаг `-X POST` (иногда его можно опустить, если используется `-d`, так как `curl` автоматически переключается на POST при наличии тела, но явное указание метода — хорошая практика, повышающая читаемость команды).
2. Заголовок с типом контента: `-H "Content-Type: application/json"`. Без этого заголовка сервер может интерпретировать тело как `application/x-www-form-urlencoded` и не суметь его распарсить как JSON.
3. Само тело запроса: `-d '{"key": "value"}'` или `--data '{"key": "value"}'`.

Дополнительные нюансы:

- Тело JSON можно передавать как строку в кавычках прямо в команде, либо загружать из файла: `-d @body.json` — это удобнее для больших или сложных payload.
- По умолчанию `curl` не следует редиректам (3xx), для этого нужен флаг `-L`.
- Флаг `-d` по умолчанию использует POST, если метод не указан явно, но лучше не полагаться на это и всегда указывать `-X POST` для явности и предсказуемости.
- Для аутентифицированных POST-запросов заголовок авторизации добавляется отдельным `-H`, например `-H "Authorization: Bearer <token>"`.
- Можно комбинировать несколько заголовков, добавляя `-H` столько раз, сколько нужно.

## Пример / когда применяется

```bash
# POST-запрос с JSON-телом, переданным прямо в команде
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name": "Ivan", "email": "ivan@example.com"}'

# POST-запрос с телом из файла и выводом статус-кода
curl -X POST https://api.example.com/orders \
  -H "Content-Type: application/json" \
  -d @order.json \
  -w "\nHTTP status: %{http_code}\n"

# Пример содержимого файла order.json
# {
#   "productId": 101,
#   "quantity": 3
# }
```

Такой запрос применяется при ручном тестировании REST API: проверке валидации входных данных, кодов ответа (`201 Created`, `400 Bad Request`), а также при написании автоматизированных smoke-тестов в CI-пайплайне.

## На что смотрит интервьюер

- Не забывает ли кандидат указать заголовок `Content-Type: application/json` — без него сервер может некорректно обработать тело запроса.
- Знает ли синтаксис передачи тела как строки (`-d '...'`) и из файла (`-d @file.json`).
- Понимает ли, что нужно экранировать или правильно оформлять кавычки при передаче JSON прямо в shell (особенно на разных ОС, где кавычки обрабатываются по-разному, например в Windows cmd vs bash).
- Может ли объяснить разницу между `-d`, `--data-raw` и `--data-binary` — тонкие различия в обработке специальных символов, таких как `&`.
- Красный флаг: если кандидат не знает, зачем нужен заголовок `Content-Type`, и пытается просто передать JSON без него, надеясь, что сервер сам разберётся.
