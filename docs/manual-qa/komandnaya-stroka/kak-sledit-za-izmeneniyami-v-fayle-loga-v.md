# Как следить за изменениями в файле лога в реальном времени (`tail -f`)?

**Коротко:** для слежения за логом в реальном времени используется команда `tail -f <файл>`, которая выводит новые строки по мере их добавления в файл, не завершая работу, — это стандартный способ наблюдать за поведением приложения во время воспроизведения бага.

## Развёрнутый ответ

Команда `tail` по умолчанию выводит последние строки файла и завершает работу. Ключ `-f` (follow) переводит её в режим "слежения": процесс не завершается, а продолжает читать файл и выводить новые строки по мере их появления — это удобно для мониторинга активно пишущихся логов (например, логов веб-сервера или приложения).

Ключевые варианты использования:

- `tail -f app.log` — следить за одним файлом.
- `tail -f -n 100 app.log` — начать со последних 100 строк, а затем следить за новыми.
- `tail -F app.log` (заглавная F) — то же самое, но команда переоткрывает файл, если он был пересоздан (например, при ротации логов через logrotate) — более надёжный вариант для продакшн-логов.
- `tail -f file1.log file2.log` — одновременное слежение за несколькими файлами, с выводом заголовка перед строками каждого файла.
- `tail -f app.log | grep ERROR` — комбинация с `grep` для слежения только за строками, содержащими определённый шаблон (например, только ошибки).
- `multitail` — сторонняя утилита для удобного параллельного слежения за несколькими логами в разделённом экране.

В Windows аналогичного поведения можно достичь через PowerShell командой:

```powershell
Get-Content app.log -Wait
```

Для тестировщика `tail -f` особенно полезен при ручном тестировании: пока тестировщик выполняет действия в приложении (например, нажимает кнопку "Оплатить"), в соседнем терминале открыт `tail -f`, показывающий, какие запросы и ошибки появляются в логе сервера в реальном времени — это позволяет мгновенно увидеть причину сбоя без необходимости перезапускать сценарий и заново собирать логи постфактум.

## Пример / когда применяется

Тестировщик воспроизводит баг с оплатой и одновременно следит за логом бэкенда:

```bash
tail -f -n 50 /var/log/payment-service/app.log
```

Если нужно видеть только ошибки во время слежения:

```bash
tail -f /var/log/payment-service/app.log | grep -i "error\|exception"
```

Если сервис использует ротацию логов (logrotate), безопаснее использовать:

```bash
tail -F /var/log/payment-service/app.log
```

## На что смотрит интервьюер

- Понимание разницы между `tail -f` и `tail -F` (переоткрытие файла при ротации логов) — частый нюанс, который отделяет опытного пользователя от новичка.
- Умение комбинировать `tail -f` с `grep` для фильтрации нужных сообщений в реальном времени.
- Практическое понимание сценария: слежение за логом во время воспроизведения бага — это ключевой навык ручного тестировщика при локализации дефектов.
- Follow-up: "Как остановить `tail -f`?" (ответ: `Ctrl+C`), "Что будет, если файл лога был удалён и пересоздан во время работы `tail -f` без ключа `-F`?" (ответ: команда продолжит следить за старым, уже удалённым файловым дескриптором и не увидит новых записей).
