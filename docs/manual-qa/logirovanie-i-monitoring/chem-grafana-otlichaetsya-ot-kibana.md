# Чем Grafana отличается от Kibana?

**Коротко:** Kibana изначально заточена под работу с Elasticsearch и логами (текстовый поиск, полнотекстовые запросы), а Grafana изначально заточена под метрики и мониторинг временных рядов из множества разных источников данных (Prometheus, Elasticsearch, InfluxDB, Loki, MySQL и др.).

## Развёрнутый ответ

Основные различия между Grafana и Kibana:

- **Источники данных.** Kibana работает практически исключительно с Elasticsearch (это её "родная" база данных). Grafana изначально универсальна и поддерживает десятки источников данных через плагины: Prometheus, InfluxDB, Elasticsearch, Loki, MySQL/PostgreSQL, CloudWatch и т.д. Это делает Grafana предпочтительным выбором, когда нужно свести на одном дашборде метрики из разных систем.
- **Основной сценарий использования.** Kibana исторически ориентирована на анализ логов и полнотекстовый поиск — сильна там, где нужно "покопаться" в сырых данных и найти конкретные записи. Grafana исторически ориентирована на мониторинг метрик и построение дашбордов реального времени (CPU, память, RPS, latency) — сильна в визуализации временных рядов и алертинге по числовым порогам.
- **Модель данных.** Elasticsearch/Kibana оперируют документами (JSON) и полнотекстовым поиском с релевантностью. Grafana в связке с Prometheus/InfluxDB оперирует временными рядами (time series) — числовыми значениями метрик с метками (labels) во времени.
- **Язык запросов.** В Kibana используется KQL/Lucene syntax для поиска по документам. В Grafana язык запросов зависит от источника данных: PromQL для Prometheus, LogQL для Loki, InfluxQL/Flux для InfluxDB и т.д.
- **Алертинг.** Обе системы поддерживают алерты, но Grafana исторически считается более гибкой и мощной именно для алертинга по метрикам (пороговые значения, аномалии, множественные условия по разным источникам данных).
- **Экосистема логов у Grafana.** Со временем Grafana добавила собственный стек для логов — Loki (аналог Elasticsearch, но более лёгкий, индексирующий только метаданные, а не полный текст), что позволяет использовать Grafana и для логов, размывая границу с Kibana.

На практике грань стирается: можно подключить Elasticsearch как источник данных к Grafana и строить дашборды по логам в Grafana; также в Kibana есть возможности для мониторинга метрик (например, через Elastic APM и Metricbeat). Но по умолчанию и по духу: Kibana = логи + Elasticsearch, Grafana = метрики + множество источников.

## Пример / когда применяется

| Критерий | Kibana | Grafana |
|---|---|---|
| Источник данных | Только Elasticsearch | Prometheus, InfluxDB, Elasticsearch, Loki, SQL и др. |
| Основной сценарий | Поиск и анализ логов | Мониторинг метрик, дашборды |
| Язык запросов | KQL / Lucene | PromQL, LogQL, зависит от источника |
| Типичный стек | ELK (Elasticsearch, Logstash, Kibana) | Grafana + Prometheus (+ Loki для логов) |

Пример использования на практике: DevOps-команда мониторит нагрузку на серверы (CPU, память, RPS) через Prometheus + Grafana, а для расследования конкретных ошибок в логах приложения использует ELK + Kibana.
