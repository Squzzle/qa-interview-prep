# Что можно делать в Kibana?

**Коротко:** Kibana — это веб-интерфейс визуализации и анализа данных, хранящихся в Elasticsearch: поиск и фильтрация логов, построение графиков и дашбордов, настройка алертов и мониторинг состояния кластера.

## Развёрнутый ответ

Kibana сама по себе не хранит данные — она обращается к Elasticsearch и предоставляет удобный интерфейс для работы с ними. Основные возможности Kibana:

- **Discover** — интерактивный поиск и просмотр "сырых" логов в табличном виде с возможностью фильтрации по времени, полям, использованием языка запросов (KQL или Lucene syntax). Это основной инструмент для быстрого расследования: найти все записи по конкретному пользователю, request id, уровню логирования и т.д.
- **Visualize / Lens** — построение визуализаций: линейные и столбчатые графики, круговые диаграммы, тепловые карты, таблицы с агрегированными данными (например, количество ошибок по часам, топ-10 самых частых сообщений об ошибках).
- **Dashboard** — объединение нескольких визуализаций в единую панель мониторинга, которую можно шарить с командой и обновлять в реальном времени.
- **Alerting** — настройка правил, которые оповещают (email, Slack, PagerDuty и т.д.), если, например, количество ошибок за 5 минут превысило пороговое значение.
- **Maps** — визуализация геоданных (например, откуда идут запросы по IP-адресам).
- **Dev Tools / Console** — консоль для прямого выполнения запросов Query DSL к Elasticsearch, полезна для отладки самих запросов.
- **Machine Learning / Anomaly Detection** (в платной подписке) — автоматическое обнаружение аномалий в метриках и логах.
- **Управление индексами и мониторинг кластера** — просмотр состояния индексов Elasticsearch, использования диска, здоровья кластера (Stack Monitoring).

Kibana особенно полезна QA-инженеру, потому что позволяет без обращения к разработчикам самостоятельно находить нужные логи по фильтрам и временным диапазонам, вместо того чтобы просить логи вручную или искать их grep-ом на сервере.

## Пример / когда применяется

Пример использования: QA-инженер тестирует функциональность оформления заказа, баг воспроизводится нестабильно. Вместо повторного воспроизведения он открывает Kibana → Discover, фильтрует по `service: order-service` и `level: ERROR` за последние 30 минут, находит конкретную запись с trace, содержащую стек ошибки и параметры запроса, и прикладывает её к баг-репорту.

```text
service: "order-service" and level: "ERROR" and @timestamp >= now-30m
```

## На что смотрит интервьюер

Интервьюер оценивает, знает ли кандидат реальные разделы Kibana (Discover, Visualize, Dashboard, Alerting), а не только абстрактное "визуализация данных". Хороший ответ показывает практический опыт использования Kibana для расследования багов. Красный флаг — путаница Kibana с Elasticsearch (например, утверждение, что Kibana хранит данные). Follow-up: "как в Kibana настроить оповещение при всплеске ошибок?", "как искать логи по конкретному временному диапазону?".
