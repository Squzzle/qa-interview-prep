# Что такое ELK Stack? Из каких компонентов он состоит?

**Коротко:** ELK Stack — это набор из трёх open-source инструментов (Elasticsearch, Logstash, Kibana) для сбора, обработки, хранения и визуализации логов; часто дополняется Beats (тогда набор называют Elastic Stack).

## Развёрнутый ответ

ELK — аббревиатура по первым буквам трёх основных компонентов:

- **E — Elasticsearch** — распределённая поисковая и аналитическая система (построена на Apache Lucene), которая хранит логи в виде документов и обеспечивает быстрый полнотекстовый поиск, фильтрацию и агрегацию по большим объёмам данных.
- **L — Logstash** — конвейер обработки данных (data pipeline), который собирает логи из разных источников, парсит и трансформирует их (фильтрует, обогащает, приводит к структурированному виду) и отправляет в Elasticsearch или другое хранилище.
- **K — Kibana** — веб-интерфейс для визуализации данных из Elasticsearch: поиск по логам, построение дашбордов, графиков, таблиц, настройка алертов.

Позже в стек добавили **Beats** — легковесные агенты для сбора данных (Filebeat — для логов из файлов, Metricbeat — для метрик, Packetbeat — для сетевого трафика и т.д.), которые устанавливаются непосредственно на источник данных и передают их в Logstash или напрямую в Elasticsearch. С добавлением Beats весь набор часто называют **Elastic Stack** вместо ELK.

Типичный поток данных: приложение пишет логи в файл → Filebeat читает файл и отправляет в Logstash → Logstash парсит (например, разбирает строку регулярным выражением или grok-паттерном), обогащает и передаёт в Elasticsearch → Elasticsearch индексирует и хранит документы → Kibana обращается к Elasticsearch и строит поиск/дашборды/алерты для пользователя.

ELK Stack решает проблему разрозненных логов в распределённых системах (микросервисах): вместо того чтобы заходить на каждый сервер и смотреть локальные файлы логов, все логи централизованно собираются, индексируются и доступны для поиска и анализа в едином интерфейсе.

## Пример / когда применяется

Пример: компания с 20 микросервисами хочет искать все ошибки, связанные с конкретным `requestId`, по всем сервисам сразу. Без ELK пришлось бы вручную grep-ать логи на каждом сервере. С ELK достаточно ввести в Kibana:

```text
requestId: "a1b2c3d4" and level: "ERROR"
```

и увидеть все связанные записи из всех сервисов в одном месте, отсортированные по времени.

## На что смотрит интервьюер

Интервьюер проверяет знание роли каждого компонента и общей архитектуры потока данных, а не просто расшифровку аббревиатуры. Красный флаг — если кандидат путает роли (например, говорит, что Logstash хранит данные, а не Elasticsearch). Частые follow-up: "чем Filebeat отличается от Logstash?", "как называется весь набор с добавлением Beats?", "какие есть альтернативы ELK?" (Grafana Loki, Splunk, Datadog, Graylog).
