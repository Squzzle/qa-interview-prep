# Для чего используется Logstash?

**Коротко:** Logstash — это конвейер обработки данных в составе ELK/Elastic Stack, который собирает логи из разных источников, парсит и трансформирует их, а затем отправляет в Elasticsearch или другое хранилище.

## Развёрнутый ответ

Logstash решает задачу ETL (Extract-Transform-Load) применительно к логам и событиям. Его конвейер состоит из трёх последовательных стадий, каждая из которых настраивается через плагины:

- **Input (сбор данных).** Logstash умеет читать данные из множества источников: файлы логов, syslog, очереди сообщений (Kafka, RabbitMQ), Beats-агенты (Filebeat), TCP/UDP-сокеты, базы данных и др.
- **Filter (обработка/трансформация).** Это ключевая часть Logstash: сырые, часто неструктурированные строки логов преобразуются в структурированные документы. Самый известный инструмент — **grok** — фильтр, использующий шаблоны на основе регулярных выражений для разбора текстовых строк на именованные поля. Также доступны фильтры для парсинга JSON, удаления/переименования полей, приведения типов данных, обогащения данными из внешних источников (например, geoip — определение геолокации по IP), маскирования чувствительных данных.
- **Output (вывод).** Обработанные данные отправляются в одно или несколько хранилищ назначения — чаще всего в Elasticsearch, но также может быть файл, очередь сообщений, внешний API и т.д.

Logstash полезен именно тогда, когда логи приходят в разном, часто неструктурированном виде (например, обычная строка лога Apache или приложения на Java с трассировкой стека), и их нужно привести к единому структурированному формату перед индексацией в Elasticsearch. Недостаток Logstash — он относительно ресурсоёмкий (написан на JRuby/Java, требует JVM), поэтому для лёгкого сбора данных на источнике часто используют Filebeat, который передаёт данные в Logstash (или напрямую в Elasticsearch) для более тяжёлой обработки.

## Пример / когда применяется

Пример конфигурации Logstash, которая парсит строку лога с помощью grok-паттерна и отправляет результат в Elasticsearch:

```text
input {
  beats {
    port => 5044
  }
}

filter {
  grok {
    match => { "message" => "%{TIMESTAMP_ISO8601:timestamp} %{LOGLEVEL:level} %{GREEDYDATA:msg}" }
  }
  date {
    match => ["timestamp", "ISO8601"]
  }
}

output {
  elasticsearch {
    hosts => ["http://localhost:9200"]
    index => "logs-%{+YYYY.MM.dd}"
  }
}
```

Такая конфигурация превращает сырую строку `2026-07-09 10:15:32 ERROR Payment failed` в структурированный документ с отдельными полями `timestamp`, `level`, `msg`.

## На что смотрит интервьюер

Интервьюер проверяет понимание того, что Logstash — это про обработку/трансформацию данных, а не хранение или визуализацию, и знание концепции input-filter-output. Плюсом будет упоминание grok-паттернов и разницы между Logstash и Filebeat (лёгкий агент сбора vs тяжёлый конвейер обработки). Красный флаг — если кандидат путает Logstash с Elasticsearch или Kibana.
