# Какие уровни логирования вы знаете (DEBUG, INFO, WARN, ERROR)?

**Коротко:** уровни логирования — это градация важности сообщений (обычно TRACE, DEBUG, INFO, WARN, ERROR, FATAL), позволяющая фильтровать логи по критичности и управлять объёмом записываемой информации.

## Развёрнутый ответ

Большинство систем логирования (Log4j, Logback, SLF4J, Python logging, Winston и др.) используют иерархию уровней от наименее важного к наиболее критичному:

- **TRACE** — самый детальный уровень, используется для очень подробной отладочной информации (например, вход/выход из каждого метода). На проде почти никогда не включён.
- **DEBUG** — отладочная информация, полезная разработчикам при поиске проблем: значения переменных, состояние объектов, шаги алгоритма. Включается только в dev/test окружениях или временно на проде для диагностики.
- **INFO** — информационные сообщения о нормальном ходе работы приложения: запуск сервиса, успешное завершение операции, старт/остановка компонентов. Это "деловой" уровень, показывающий, что система работает штатно.
- **WARN (WARNING)** — предупреждение о потенциально проблемной ситуации, которая не приводит к сбою прямо сейчас, но может привести в будущем: например, использование deprecated API, повторная попытка подключения, приближение к лимиту ресурсов.
- **ERROR** — ошибка, которая нарушила выполнение конкретной операции, но не всё приложение: например, не удалось обработать один запрос, но сервис продолжает работать.
- **FATAL / CRITICAL** — критическая ошибка, приводящая к неработоспособности всего приложения или его части (например, невозможность подключиться к базе данных при старте).

Уровни образуют иерархию: включение уровня INFO обычно означает, что будут также записываться WARN, ERROR и FATAL, но не DEBUG и TRACE. Это позволяет гибко управлять объёмом логов на разных окружениях: на проде обычно включают INFO+, в тестовой среде — DEBUG+ для диагностики.

## Пример / когда применяется

| Уровень | Пример сообщения |
|---|---|
| DEBUG | "Получен ответ от кэша: &#123;key=user_123, hit=true&#125;" |
| INFO | "Сервис авторизации успешно запущен на порту 8080" |
| WARN | "Время ответа внешнего API превысило 2 секунды" |
| ERROR | "Не удалось сохранить заказ #4521: превышен таймаут БД" |
| FATAL | "Невозможно подключиться к базе данных при старте приложения" |

Тестировщик, воспроизводя баг, может попросить разработчика временно поднять уровень логирования до DEBUG на тестовом стенде, чтобы увидеть больше деталей.

## На что смотрит интервьюер

Интервьюер проверяет знание иерархии уровней и умение объяснить разницу между WARN и ERROR, между ERROR и FATAL. Частая ошибка кандидатов — путать уровни местами или не знать, что уровни иерархичны (фильтрация "от уровня и выше"). Follow-up вопросы: "на каком уровне логировать неудачную повторную попытку с retry?", "что будет, если на проде включить DEBUG?" (ответ: рост объёма логов, нагрузка на диск/сеть, возможное падение производительности).
