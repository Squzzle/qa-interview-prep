# Чем **нативное** приложение отличается от **гибридного** и **веб-приложения**?

**Коротко:** нативное приложение написано под конкретную платформу и даёт максимум производительности и доступа к железу, веб-приложение — это сайт в браузере без установки, а гибридное — веб-код, упакованный в нативную оболочку (WebView).

## Развёрнутый ответ

**Нативное приложение** пишется на языке, который поддерживает конкретная ОС: Kotlin/Java с Android SDK, Swift с iOS SDK. Устанавливается из магазина приложений, компилируется в машинный код под конкретную платформу. Плюсы: максимальная производительность, полный доступ ко всем API устройства (камера, датчики, Bluetooth, NFC, биометрия), нативный UI и жесты, работа офлайн "из коробки". Минусы: отдельная кодовая база под каждую платформу — дороже разработка и поддержка.

**Веб-приложение** — это адаптивный сайт, открывающийся в браузере устройства. Не требует установки, доступен по URL, работает на любой ОС с браузером. Плюсы: один код для всех платформ, мгновенное обновление без публикации в сторе. Минусы: ограниченный доступ к аппаратным возможностям, зависимость от качества соединения и браузера, менее отзывчивый UI, невозможность работы полностью офлайн без PWA-технологий.

**Гибридное приложение** — это веб-приложение (HTML/CSS/JS), обёрнутое в нативный контейнер через WebView с помощью фреймворков (Cordova, Ionic, Capacitor). Устанавливается как обычное приложение, но UI рендерится браузерным движком внутри приложения. Доступ к нативным API реализуется через мост (bridge) — прослойку плагинов, которая транслирует вызовы JS в нативный код. Плюсы: единая кодовая база, возможность установки из стора. Минусы: производительность ниже нативной, задержки и визуальные артефакты WebView, зависимость от корректности работы моста.

Ключевое отличие для тестировщика: у нативного приложения тестируется взаимодействие с ОС напрямую (permissions, lifecycle, push), у веб — кроссбраузерность и responsive-вёрстка, у гибридного — дополнительно тестируется слой WebView и корректность передачи данных между JS и нативным кодом (например, поведение при потере контекста WebView, memory leaks в WebView, задержки при вызове нативных плагинов).

## Пример / когда применяется

Банковское приложение с биометрической аутентификацией и офлайн-доступом к балансу почти всегда делается нативным — критична безопасность и доступ к Secure Enclave/Keystore. Простой каталог товаров без сложной логики можно реализовать как мобильную веб-версию. MVP-стартап с ограниченным бюджетом часто выбирает гибридный подход (Ionic/Cordova) или кросс-платформенный (Flutter/React Native), чтобы быстрее выйти на обе платформы одной командой.

## На что смотрит интервьюер

- Понимание, что "гибридное" не равно "кросс-платформенное" (React Native/Flutter компилируются в нативные виджеты, а не рендерятся в WebView).
- Способность связать архитектурные различия с конкретными рисками тестирования (WebView-баги, отсутствие офлайн-режима у веб).
- Follow-up: "Как бы вы протестировали переход приложения из фонового режима в активный для гибридного приложения?" (проверка сохранения состояния WebView).
- Красный флаг — поверхностный ответ "нативное быстрее, веб медленнее" без объяснения причин.
