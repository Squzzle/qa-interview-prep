# Что нужно проверить при тестировании **установки и обновления** приложения?

**Коротко:** Нужно проверить корректность самого процесса установки/обновления (успех, отмена, сбои), сохранность пользовательских данных при обновлении, совместимость с разными версиями ОС и устройств, а также поведение приложения сразу после первого запуска.

## Развёрнутый ответ

Тестирование установки (installation testing) и обновления (upgrade testing) — важная часть мобильного тестирования, так как это первая точка контакта пользователя с приложением, а ошибки на этом этапе (краш при установке, потеря данных при апдейте) критично влияют на рейтинг приложения в сторе.

При тестировании **установки** проверяют:
- Успешную установку из разных источников: Google Play, App Store, тестовые каналы (TestFlight, внутреннее тестирование, APK напрямую).
- Корректность размера приложения, скорости скачивания и установки.
- Установку на разные версии ОС (минимальную поддерживаемую и последнюю), разные архитектуры процессора (ARM/x86) и разное количество свободной памяти (в т.ч. при нехватке места).
- Прерывание установки: потеря сети, отключение экрана, низкий заряд батареи, принудительное закрытие процесса установки.
- Корректность запрошенных разрешений и системных диалогов при первом запуске.
- Первый запуск (first launch / onboarding): отображение приветственных экранов, отсутствие "мусорных" данных от предыдущих тестов.
- Установку при уже установленном конфликтующем приложении (одинаковый package name/bundle id при side-loading).
- Проверку подписи и сертификата (для Android — совпадение подписей при обновлении, иначе система откажет в установке).

При тестировании **обновления** проверяют:
- Обновление с предыдущей версии (N-1) и с более старых версий (N-2, N-3) на актуальную — так называемое "upgrade path" тестирование.
- Сохранность пользовательских данных, настроек, локальной базы данных, кэша и авторизации после обновления.
- Корректность миграции схемы базы данных (если менялась структура локального хранилища).
- Работу автообновления через сторы и принудительного обновления (forced update), если в приложении реализована такая логика.
- Прерывание процесса обновления (потеря сети во время скачивания, недостаток места).
- Совместимость новых серверных API со старой версией приложения (обратная совместимость, если обновление не мгновенное у всех пользователей).
- Откат обновления (если поддерживается) и поведение при удалении/повторной установке.
- Изменение разрешений между версиями — новые разрешения должны запрашиваться корректно.

## Пример / когда применяется

Пример матрицы для регрессии установки/обновления:

| Сценарий | Что проверяем |
|---|---|
| Чистая установка на минимальную версию ОС | Приложение запускается, нет крашей |
| Установка при нехватке памяти на устройстве | Понятная ошибка, нет краша системы |
| Обновление с версии N-2 на текущую | Данные пользователя и настройки сохранены |
| Обновление с изменением схемы БД | Миграция данных прошла без потерь |
| Прерывание загрузки обновления (потеря сети) | Возможность докачать/повторить, нет "битой" установки |
| Принудительное обновление (forced update) | Старая версия блокируется с понятным сообщением |

## На что смотрит интервьюер

Интервьюер ожидает, что кандидат вспомнит не только "установилось — работает", но и сценарии с прерыванием, миграцией данных и upgrade path с нескольких версий назад. Частая ошибка — забыть про сохранность данных при обновлении и про несовпадение подписей APK. Плюсом считается упоминание тестирования на реальных сторах (staged rollout, closed/open testing) и forced update логики.
