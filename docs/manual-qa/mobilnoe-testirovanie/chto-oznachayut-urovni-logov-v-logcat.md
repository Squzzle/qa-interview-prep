# Что означают уровни логов в LogCat?

**Коротко:** LogCat в Android различает шесть уровней важности сообщений — Verbose, Debug, Info, Warning, Error, Assert (иногда обозначается как Fatal/WTF) — от самых подробных до критических сбоев.

## Развёрнутый ответ

LogCat — это встроенная система логирования Android, куда пишут сообщения как система, так и само приложение (через класс `Log`). Каждое сообщение имеет тег, текст и уровень (приоритет), что позволяет фильтровать вывод.

Уровни логов по возрастанию критичности:

- **V (Verbose)** — самый подробный уровень, используется для отладочной информации любого рода, включается только вручную, не должен попадать в релизные сборки.
- **D (Debug)** — отладочные сообщения, полезные при разработке, обычно отключаются в продакшене.
- **I (Info)** — информационные сообщения о нормальной работе приложения (например, успешное выполнение операции, важные события жизненного цикла).
- **W (Warning)** — предупреждения о потенциальных проблемах, которые не приводят к падению, но требуют внимания (например, устаревший API, повторная попытка запроса).
- **E (Error)** — ошибки, которые повлияли на работу функции или компонента, но не обязательно уронили всё приложение (например, исключение при парсинге, неудачный сетевой запрос).
- **A (Assert / WTF — "What a Terrible Failure")** — критическая ошибка, сигнализирующая о состоянии, которое теоретически невозможно; фактически указывает на серьёзный баг в логике.

Каждому уровню в коде соответствует свой метод: `Log.v()`, `Log.d()`, `Log.i()`, `Log.w()`, `Log.e()`, `Log.wtf()`. Для тестировщика важно уметь фильтровать логи по уровню и тегу, чтобы быстро находить релевантную информацию среди огромного потока системных сообщений — особенно при поиске крашей, ANR (Application Not Responding) и исключений (`FATAL EXCEPTION` в стектрейсе обычно относится к уровню Error).

## Пример / когда применяется

Типичный сценарий: тестировщик воспроизводит краш приложения и снимает логи через ADB, затем фильтрует их по тегу приложения и уровню Error/Assert, чтобы найти стектрейс исключения и приложить его к баг-репорту.

```bash
adb logcat *:E
```

Эта команда покажет только сообщения уровня Error и выше (Error, Assert) от всех тегов — удобно для быстрого поиска критичных проблем.

```bash
adb logcat MyAppTag:D *:S
```

Здесь показываются логи с тегом `MyAppTag` начиная с уровня Debug, а всё остальное (`*:S` — Silent) подавляется.

## На что смотрит интервьюер

Интервьюер проверяет, знает ли кандидат порядок уровней и их практическое назначение, а не просто помнит буквы. Частая ошибка — перепутать порядок важности (например, считать Debug более критичным, чем Warning) или не знать про уровень Assert/WTF. Хороший ответ включает упоминание фильтрации логов по тегам и уровням как рабочего навыка. Часто задают follow-up: "как отфильтровать логи только своего приложения?", "чем ANR отличается от краша в логах?", "как найти стектрейс исключения в потоке логов?".
