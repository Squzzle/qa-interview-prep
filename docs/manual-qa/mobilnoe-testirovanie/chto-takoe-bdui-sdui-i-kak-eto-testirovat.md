# Что такое **BDUI / SDUI** и как это тестировать?

**Коротко:** BDUI (Backend-Driven UI) и SDUI (Server-Driven UI) — подходы, при которых структура и содержимое экрана описываются сервером в виде JSON/конфига, а клиент лишь рендерит полученную разметку, что требует тестирования не только клиента, но и серверной части, отдающей UI-конфигурацию.

## Развёрнутый ответ

При традиционном подходе разметка экрана (набор компонентов, их расположение, поведение) зашита в коде мобильного приложения и меняется только через выпуск новой версии в сторе. При BDUI/SDUI сервер отдаёт клиенту декларативное описание экрана — обычно JSON со списком компонентов (заголовок, кнопка, список карточек, баннер), их свойствами (текст, цвет, ссылка, видимость) и правилами поведения, а клиентское приложение содержит только "движок рендеринга" этих компонентов.

Ключевые преимущества такого подхода: возможность менять контент и даже структуру экрана без обновления приложения в сторе, проведение A/B-тестов и персонализации на уровне сервера, единая логика для разных платформ (Android/iOS/Web получают одинаковый конфиг).

Особенности тестирования BDUI/SDUI:

- **Тестирование клиента (движка рендеринга)**: проверка корректности отображения каждого типа компонента, устойчивости к некорректным/неполным данным от сервера (fallback-поведение, если пришёл незнакомый тип компонента или отсутствует обязательное поле), обработка ошибок сети при получении конфигурации, кэширование последней успешной конфигурации на случай отсутствия связи.
- **Тестирование серверной части**: валидация корректности отдаваемого JSON (соответствие контракту/схеме, например JSON Schema), проверка логики персонализации и таргетинга (разным сегментам пользователей должен приходить разный конфиг), обратная совместимость конфигов со старыми версиями клиента (старое приложение не должно падать, получив новый тип компонента).
- **Контрактное тестирование**: так как клиент и сервер разрабатываются часто разными командами и независимо, важно контрактное тестирование (например, Pact) или строгая схема JSON, чтобы изменения на бэкенде не ломали рендеринг на клиенте.
- **Тестирование деградации**: что произойдёт, если сервер недоступен, отдаёт пустой конфиг, отдаёт конфиг с ошибкой валидации, отдаёт неизвестную клиенту версию схемы — приложение не должно крашиться, должен быть предусмотрен fallback UI или кэш.
- **Кросс-версионность**: важно проверять совместимость нового серверного конфига со старыми версиями приложения, которые ещё не обновились (пользователи могут месяцами не обновлять приложение), — это специфичный риск именно для SDUI, отсутствующий при классическом подходе.
- **Визуальная регрессия**: поскольку контент экрана может динамически меняться сервером, полезны автоматизированные скриншот-тесты для отслеживания непреднамеренных визуальных изменений.

## Пример / когда применяется

Пример практического кейса: главный экран мобильного приложения (лента баннеров, блоки рекомендаций, промо) полностью описывается конфигом с сервера. Тестировщик должен проверить: что произойдёт при получении конфига с неизвестным типом компонента (приложение должно скрыть/пропустить его, а не упасть), что покажется при отсутствии сети (кэшированная версия экрана), корректно ли применяется таргетинг (пользователю из тестового сегмента показывается экспериментальный баннер, остальным — нет), и что старая версия приложения (например, годичной давности) корректно отрисовывает новый конфиг без критичных элементов, специфичных для новых версий.

Пример упрощённого фрагмента конфига, который могла бы отдавать серверная часть:

```json
{
  "screen": "home",
  "components": [
    { "type": "banner", "title": "Скидка 20%", "imageUrl": "https://example.com/banner.png" },
    { "type": "productList", "categoryId": "electronics" },
    { "type": "unknownWidget", "data": {} }
  ]
}
```

Задача теста — убедиться, что компонент `unknownWidget` (неизвестный клиенту тип) корректно игнорируется без падения приложения.

## На что смотрит интервьюер

Интервьюер оценивает, понимает ли кандидат, что тестирование BDUI/SDUI выходит за рамки чисто клиентского QA и требует внимания к серверному контракту, версионной совместимости и деградации при ошибках. Частая ошибка — рассматривать SDUI просто как "ещё один экран", не упоминая риски рассинхронизации версий клиента и сервера. Хороший ответ обязательно затрагивает fallback-поведение и обратную совместимость. Возможные follow-up вопросы: "как тестировать совместимость нового конфига со старой версией приложения в сторах?", "какие типы автотестов применимы к SDUI (контрактные, визуальные, e2e)?", "как обеспечить, что баг в конфиге на сервере не уронит приложение всем пользователям одновременно?".
