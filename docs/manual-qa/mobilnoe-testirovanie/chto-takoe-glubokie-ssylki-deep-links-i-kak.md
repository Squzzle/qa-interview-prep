# Что такое **глубокие ссылки (Deep Links)** и как их тестировать?

**Коротко:** Deep Link — это ссылка, которая открывает не главный экран приложения, а конкретный экран/контент внутри него, и её тестирование включает проверку корректности навигации, обработки невалидных ссылок и поведения при неустановленном приложении.

## Развёрнутый ответ

Глубокая ссылка (Deep Link) — это URI-ссылка специального формата, при переходе по которой операционная система открывает не браузер или главный экран приложения, а конкретный экран/раздел/контент внутри мобильного приложения. Различают несколько видов:

- **Обычные (Traditional/Custom scheme) Deep Links** — используют собственную схему URI приложения, например `myapp://product/123`. Работают только если приложение уже установлено, иначе — ошибка.
- **Deferred Deep Links (отложенные)** — если приложение не установлено, пользователь сначала перенаправляется в магазин приложений для установки, а после первого запуска приложение "помнит", куда изначально хотел перейти пользователь, и открывает нужный экран.
- **Universal Links (iOS) / App Links (Android)** — используют обычные HTTP/HTTPS-ссылки (например, `https://example.com/product/123`), которые операционная система распознаёт как принадлежащие приложению (через файл верификации на сервере — apple-app-site-association для iOS, assetlinks.json для Android) и открывает приложение вместо браузера, если оно установлено; иначе открывается сайт в браузере.

Что нужно тестировать:

1. **Корректность навигации** — переход по ссылке должен открывать именно нужный экран с нужными параметрами (например, конкретный товар, конкретный заказ), а не главный экран.
2. **Обработка при закрытом/открытом приложении** — deep link должен корректно работать как при холодном старте (cold start), так и когда приложение уже запущено в фоне (warm start).
3. **Невалидные/некорректные ссылки** — например, `myapp://product/несуществующий_id` — приложение должно показывать понятную ошибку, а не падать.
4. **Deferred deep link** — сценарий "нет приложения → переход в стор → установка → первый запуск → открытие нужного экрана".
5. **Верификация домена (App Links/Universal Links)** — проверка, что файл ассоциации (assetlinks.json/apple-app-site-association) корректно настроен и ссылка не открывается в браузере вместо приложения.
6. **Передача параметров и трекинг** — если deep link используется маркетингом (UTM-метки, реферальные коды), проверяется корректная передача и сохранение этих параметров.
7. **Конфликт нескольких приложений с одной схемой** — на Android несколько приложений могут регистрировать одну и ту же custom scheme, что может привести к неоднозначности при открытии ссылки.
8. **Безопасность** — deep link не должен позволять обходить авторизацию (например, открытие защищённого контента без входа в аккаунт).

## Пример / когда применяется

Пример: push-уведомление о скидке на товар содержит deep link `https://shop.example.com/product/456?utm_source=push`. При тапе по уведомлению должно открыться приложение (если установлено) сразу на странице товара 456 с примененной UTM-меткой для аналитики; если приложение не установлено — открывается страница товара в браузере или сторе с последующим deferred deep link после установки.

```bash
# Тестирование Deep Link через adb на Android
adb shell am start -W -a android.intent.action.VIEW -d "myapp://product/456" com.example.app
```
