# Что такое **кросс-платформенная разработка**? Как это влияет на тестирование?

**Коротко:** Кросс-платформенная разработка — это создание одного кодовой базы, работающей одновременно на iOS и Android (например, на React Native, Flutter, Kotlin Multiplatform, Xamarin), что ускоряет разработку, но требует от тестировщика проверки платформенно-специфичного поведения и мест, где общий код "просвечивает" нативные отличия ОС.

## Развёрнутый ответ

Кросс-платформенная разработка противопоставляется нативной разработке (отдельно Swift/Objective-C для iOS и Kotlin/Java для Android). Основные технологии: **React Native**, **Flutter**, **Kotlin Multiplatform (KMP)**, **Xamarin/.NET MAUI**, **Ionic/Cordova** (гибридные веб-приложения в нативной оболочке). Цель — писать бизнес-логику и часто UI один раз и переиспользовать на обеих платформах, сокращая время и стоимость разработки.

Как это влияет на тестирование:

1. **Экономия усилий на дублирующихся тест-кейсах.** Если бизнес-логика едина, баги в ней обычно воспроизводятся на обеих платформах одинаково — можно один раз детально протестировать логику и не дублировать это тестирование полностью на каждой платформе.
2. **Необходимость платформенно-специфичного тестирования UI/UX.** Кросс-платформенные фреймворки (особенно Flutter, рисующий свой UI поверх Skia/Impeller) могут отрисовывать элементы не так, как нативные компоненты платформы — нужно отдельно проверять соответствие HIG на iOS и Material Design на Android, поведение системных жестов, шрифты, анимации.
3. **Разное поведение платформенных API через мосты (bridge/plugin).** React Native и Flutter обращаются к нативным возможностям (камера, геолокация, уведомления) через плагины/мосты — баг может проявляться только в мосте на одной из платформ, тестировщику нужно явно проверять паритет функциональности между iOS и Android.
4. **Производительность.** Кросс-платформенные приложения исторически могут уступать нативным по производительности (особенно анимации, сложный рендеринг списков) — нужно отдельно тестировать плавность интерфейса и потребление памяти на обеих платформах.
5. **Версионная фрагментация фреймворка.** Проблемы совместимости самого кросс-платформенного фреймворка с новыми версиями ОС (например, обновление iOS/Android ломает старую версию React Native/Flutter) — нужно тестировать после обновления ОС и после обновления самого фреймворка/плагинов.
6. **Разные размеры и структура сборок.** IPA/APK, собранные из общего кода, всё равно содержат нативную обёртку — нужно проверять установку, размер, время старта на обеих платформах отдельно.
7. **Локализация и системные диалоги.** Общий код может некорректно обрабатывать платформенные различия в форматах даты, разрешениях, системных всплывающих окнах — это частый источник платформенно-специфичных багов.

Таким образом, кросс-платформенность не отменяет необходимость тестировать на обеих ОС — она смещает фокус: меньше дублирования логики, но больше внимания к местам "стыка" с нативной платформой.

## Пример / когда применяется

Пример: приложение на Flutter реализует push-уведомления через общий плагин. На Android уведомления приходят и отображаются корректно даже при закрытом приложении, а на iOS из-за особенностей APNs и настройки capabilities в Xcode-проекте уведомления не приходят в фоне. Баг обнаруживается только при отдельном тестировании каждой платформы, несмотря на единый код.

## На что смотрит интервьюер

Интервьюер ожидает понимания, что кросс-платформенность не устраняет необходимость тестирования на обеих ОС, а смещает фокус на нативные мосты, платформенно-специфичный UI и паритет функциональности. Плюсом будет знание конкретных фреймворков (React Native, Flutter, KMP) и типичных проблем (жесты, шрифты, отличия в рендеринге). Красный флаг — утверждение, что "раз код общий, достаточно протестировать один раз на одной платформе".
