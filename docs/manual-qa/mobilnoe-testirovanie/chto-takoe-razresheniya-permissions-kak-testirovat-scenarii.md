# Что такое **разрешения (Permissions)**? Как тестировать сценарии с разрешениями?

**Коротко:** Разрешения — это механизм, при котором приложение запрашивает у пользователя доступ к защищённым ресурсам системы (камера, геолокация, контакты, микрофон, память и т.д.), а пользователь может разрешить или отклонить доступ; тестирование разрешений проверяет корректность запроса, поведение при отказе и работу с этим доступом.

## Развёрнутый ответ

Разрешения (permissions) — это часть модели безопасности мобильных ОС (Android runtime permissions, iOS privacy permissions), которая ограничивает доступ приложений к чувствительным данным и функциям устройства: камере, микрофону, геолокации, контактам, фото/файлам, Bluetooth, уведомлениям, календарю и т.п. Начиная с Android 6.0 (API 23) и во всех современных версиях iOS большинство таких разрешений запрашиваются не при установке, а в момент первого обращения к соответствующей функции (runtime permissions), и пользователь может дать согласие однократно, всегда, только во время использования приложения либо отказать.

Разрешения делятся на обычные (normal, автоматически выдаются без диалога — например, доступ к интернету) и опасные (dangerous, требуют явного согласия пользователя — камера, геолокация, микрофон, контакты). На iOS похожая градация: часть возможностей доступна без запроса, часть требует явного подтверждения через системный алерт, описание цели указывается разработчиком в Info.plist (usage description strings).

Тестирование сценариев с разрешениями включает:
- Проверку того, что разрешение запрашивается именно в момент, когда оно реально нужно (а не сразу при запуске без объяснения причины).
- Проверку текста и корректности объяснения (rationale), почему приложению нужен доступ.
- Проверку поведения приложения при предоставлении разрешения — функция должна начать работать.
- Проверку поведения при отказе — приложение не должно падать, должно показывать понятное сообщение и предлагать альтернативу (например, ручной ввод адреса вместо геолокации) либо ссылку на настройки.
- Проверку сценария "не спрашивать больше" (Android) и повторного запроса после отказа.
- Проверку отзыва разрешения через системные настройки во время работы приложения (runtime revoke) — приложение должно корректно обработать потерю доступа.
- Проверку версии ОС, где логика permissions отличается (например, доступ к местоположению "только во время использования" появился в iOS 11+, "приблизительная геолокация" — в iOS 14+, разрешения на уведомления и медиафайлы менялись в Android 13).
- Проверку многократных запросов подряд (не должно быть спама диалогами).
- Для чувствительных разрешений — проверку, что данные действительно используются только для заявленной цели (актуально при security- и privacy-тестировании).

## Пример / когда применяется

Пример чек-листа для функции "прикрепить фото из галереи":

| Сценарий | Ожидаемый результат |
|---|---|
| Первый вызов функции | Показывается системный диалог запроса доступа к фото/медиа |
| Пользователь разрешил | Открывается галерея, можно выбрать фото |
| Пользователь отклонил | Показывается понятное сообщение с предложением перейти в настройки |
| Повторный вызов после отказа "не спрашивать больше" | Диалог не показывается повторно, сразу — сообщение с переходом в настройки |
| Разрешение отозвано в настройках во время использования приложения | Приложение на следующем обращении к функции корректно обнаруживает отсутствие доступа |
| Частичный доступ (например, iOS "выбрать фото") | Приложению доступны только выбранные фото, работа с остальными недоступна |

## На что смотрит интервьюер

Интервьюер проверяет, знает ли кандидат разницу между install-time и runtime permissions, помнит ли про поведение "не спрашивать больше" и про возможность отозвать разрешение в любой момент через настройки (частая ошибка — тестировать только "счастливый путь", забывая про отказ и отзыв доступа). Плюсом будет упоминание конкретных версий ОС, где менялась модель разрешений, и понимание, что отсутствие разрешения не должно приводить к крашу приложения.
