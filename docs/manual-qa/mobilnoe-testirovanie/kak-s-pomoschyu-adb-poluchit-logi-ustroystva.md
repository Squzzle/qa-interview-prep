# Как с помощью ADB получить логи устройства?

**Коротко:** логи снимаются командой `adb logcat`, которую можно фильтровать по тегам, уровням важности и сохранять в файл для последующего анализа и приложения к баг-репорту.

## Развёрнутый ответ

`adb logcat` выводит поток системных и прикладных сообщений в реальном времени. Без параметров команда покажет всё, что пишут в лог все процессы устройства, что обычно избыточно — поэтому важно уметь фильтровать вывод.

Основные приёмы работы с logcat:

- Фильтрация по тегу и уровню: `adb logcat ТЕГ:УРОВЕНЬ *:S` — покажет сообщения с указанным тегом от заданного уровня и подавит (Silent) всё остальное.
- Фильтрация по PID процесса конкретного приложения: `adb logcat --pid=$(adb shell pidof -s com.example.app)`.
- Очистка буфера перед воспроизведением бага, чтобы не было "мусора" от предыдущих действий: `adb logcat -c`.
- Сохранение логов в файл для приложения к баг-репорту: `adb logcat > device_log.txt` (либо `adb logcat -d > device_log.txt` для однократного дампа без ожидания новых строк).
- Получение логов краша конкретно через buffer crash: `adb logcat -b crash`.
- Получение всей диагностической информации об устройстве разом (более полный набор, включая logcat, dumpsys, версии, состояние батареи): `adb bugreport`.

Логи имеют формат `дата время PID-TID уровень/тег: сообщение`, что позволяет сопоставлять события во времени с шагами воспроизведения бага.

## Пример / когда применяется

Тестировщик воспроизводит краш и хочет получить чистый лог именно этого случая для баг-репорта:

```bash
adb logcat -c
# выполнить шаги воспроизведения бага в приложении
adb logcat -d -b crash > crash_log.txt
```

Сначала буфер очищается, затем воспроизводится баг, после чего снимается дамп именно из буфера crash — там гарантированно будет стектрейс `FATAL EXCEPTION`, если приложение упало.

## На что смотрит интервьюер

Интервьюер смотрит, умеет ли кандидат фильтровать логи целенаправленно, а не просто запускать `adb logcat` вслепую, а также знает ли про очистку буфера перед воспроизведением и про отдельный буфер crash. Частая ошибка — прикладывать к баг-репорту весь необработанный лог без фильтрации, что затрудняет анализ разработчиком.
