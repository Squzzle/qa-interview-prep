# Как тестировать поведение приложения при **прерываниях** (звонок, смс)?

**Коротко:** нужно проверить, что приложение корректно переходит в фоновый режим при прерывании (звонок, SMS, уведомление, будильник), сохраняет состояние и данные, а после возврата в приложение — восстанавливает их без потерь.

## Развёрнутый ответ

Прерывания (interruptions) — это внешние события, которые заставляют приложение временно уйти в фон или временно потерять фокус. Основные типы прерываний для тестирования:

- **Входящий/исходящий звонок** — приложение уходит в background, экран блокируется системным UI звонка.
- **SMS/push-уведомление** — может показываться поверх приложения (баннер) без полного ухода в фон.
- **Будильник, календарное напоминание** — системный диалог поверх приложения.
- **Разрядка батареи / низкий заряд** — системное предупреждение.
- **Подключение/отключение зарядки, наушников, Bluetooth-устройства.**
- **Блокировка экрана (Lock screen).**
- **Переключение на другое приложение (Home button, App Switcher) и возврат обратно.**
- **Системные обновления или установка других приложений в фоне.**

Что именно нужно проверять для каждого прерывания:

1. **Сохранение состояния (State preservation).** Данные, введённые пользователем (например, незавершённая форма, текст в поле ввода, прогресс видео), должны сохраняться при возврате в приложение.
2. **Корректная приостановка процессов.** Видео/аудио должны ставиться на паузу при звонке; таймеры, анимации — приостанавливаться.
3. **Возобновление сетевых операций.** Если во время прерывания шла загрузка данных, приложение должно либо продолжить, либо корректно обработать прерванный запрос при возврате.
4. **Отсутствие крашей.** Приложение не должно падать при возврате из фона, особенно если система в это время выгрузила процесс из памяти (low memory kill) — тогда приложение должно "холодно" перезапуститься, но восстановить последнее состояние (state restoration).
5. **Синхронизация UI.** После возврата экран должен отображать актуальное состояние (например, обновлённые данные, если они изменились за время прерывания).

Технически такие тесты выполняются вручную (реальный звонок на тестовое устройство) или через эмуляцию: adb-команды для симуляции входящего звонка на эмуляторе Android, инструменты Xcode Simulator для симуляции системных событий на iOS, а также через специальные тест-фреймворки, которые могут программно отправлять приложение в фон и возвращать обратно.

## Пример / когда применяется

Пример теста: "Пользователь заполняет форму оплаты (номер карты, сумма) → поступает входящий звонок → пользователь отвечает и завершает разговор → возвращается в приложение → все введённые данные формы оплаты должны остаться на месте, приложение не должно требовать повторного ввода".

```bash
# Симуляция входящего звонка на Android-эмуляторе через adb
adb emu gsm call 1234567890
# Завершение звонка
adb emu gsm cancel 1234567890
```

## На что смотрит интервьюер

- Знание конкретных инструментов симуляции прерываний (adb emu gsm call, Xcode Simulator hardware menu).
- Понимание разницы между "приложение просто в фоне" и "система выгрузила процесс приложения из памяти" — и что второй случай требует восстановления состояния "с нуля" через сохранённые данные (savedInstanceState на Android, state restoration на iOS).
- Follow-up: "Как вы протестируете сценарий, когда система убивает процесс приложения в фоне из-за нехватки памяти?"
- Красный флаг — если кандидат говорит только про "проверить, что приложение не падает" без упоминания сохранения состояния данных.
