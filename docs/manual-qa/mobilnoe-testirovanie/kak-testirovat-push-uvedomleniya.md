# Как тестировать **Push-уведомления**?

**Коротко:** тестирование push-уведомлений включает проверку доставки в разных состояниях приложения (foreground/background/killed), корректность содержимого и действий по нажатию, поведение при отключённых разрешениях и работу с большим объёмом/частотой уведомлений.

## Развёрнутый ответ

Push-уведомления — асинхронные сообщения, доставляемые на устройство через сервисы push-нотификаций (Firebase Cloud Messaging для Android, Apple Push Notification service для iOS) даже когда приложение не активно. Тестирование охватывает несколько уровней:

1. **Состояния приложения при получении push:**
   - **Foreground** (приложение открыто и активно) — уведомление обычно нужно обработать программно (показать in-app баннер или обновить данные), т.к. системный баннер по умолчанию может не показываться.
   - **Background** (приложение свёрнуто) — системное уведомление должно появиться в шторке уведомлений с корректным текстом/иконкой.
   - **Killed/Terminated** (приложение полностью закрыто) — push всё равно должен быть доставлен и, при тапе, приложение должно запуститься и открыть нужный экран (часто через deep link).

2. **Содержимое уведомления:**
   - Корректность заголовка, текста, иконки, изображения (rich push), звука/вибрации.
   - Локализация текста уведомления в зависимости от языка устройства.
   - Ограничение длины текста (обрезка на разных платформах/лаунчерах).

3. **Действия по уведомлению:**
   - Тап по уведомлению открывает нужный экран приложения (deep link) с правильными данными.
   - Action buttons (кнопки прямо в уведомлении — например, "Ответить", "Отметить как прочитанное") должны выполнять соответствующее действие без обязательного открытия приложения.
   - Swipe для удаления уведомления не должен вызывать побочных эффектов.

4. **Разрешения (Permissions).**
   - Сценарий первого запроса разрешения на уведомления (особенно важно на iOS и Android 13+, где разрешение явно запрашивается).
   - Поведение приложения при отклонённом разрешении — уведомления не должны приходить, приложение не должно падать при попытке их отправить.
   - Повторный запрос разрешения после отказа (через настройки системы).

5. **Группировка и множественные уведомления.**
   - Несколько push подряд должны корректно группироваться (stacking) и не "заваливать" пользователя.
   - Проверка счётчика непрочитанных (badge count) на иконке приложения.

6. **Технические аспекты доставки:**
   - Доставка при разных состояниях сети (push должен приходить даже при плохом соединении, за счёт retry на стороне push-сервиса).
   - Обработка токена устройства (device token/registration token) — обновление при переустановке приложения, отвязка токена при логауте.
   - Time-sensitive/приоритетные уведомления — не должны блокироваться режимом энергосбережения устройства.

7. **Тихие (silent/data-only) push** — используются для фоновой синхронизации данных без показа уведомления пользователю — тестируется отдельно, так как они не видны пользователю напрямую.

## Пример / когда применяется

Пример тест-кейса: интернет-магазин отправляет push "Ваш заказ доставлен" → 
1. Приложение закрыто → push приходит в шторку → тап открывает экран деталей заказа.
2. Приложение открыто (foreground) → показывается in-app баннер с тем же текстом.
3. Разрешение на уведомления отклонено → push не показывается, но данные заказа всё равно доступны при открытии приложения вручную.

```text
Чек-лист тестирования push-уведомлений:
1. Доставка в foreground / background / killed состоянии
2. Корректность текста, изображения, локализации
3. Тап по push открывает правильный экран (deep link)
4. Action buttons выполняют нужное действие
5. Поведение при отклонённом разрешении на уведомления
6. Группировка нескольких уведомлений и badge count
7. Обновление device token при переустановке/логауте
```

## На что смотрит интервьюер

- Знание разницы между foreground/background/killed обработкой push и того, что foreground требует программной обработки.
- Понимание работы токенов устройства и того, что происходит с ними при логауте/переустановке.
- Follow-up: "Как вы протестируете, что при логауте пользователю больше не приходят push, предназначенные для его аккаунта?"
- Красный флаг — сведение ответа только к "уведомление пришло — тест пройден" без учёта разных состояний приложения и разрешений.
