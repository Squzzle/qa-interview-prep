# Как тестировать работу приложения при **потере сети** или ее переключении?

**Коротко:** нужно проверять поведение приложения при полном отсутствии сети, при переключении между Wi-Fi/мобильной сетью, при медленном/нестабильном соединении, следя за корректной обработкой ошибок, сохранностью данных и плавным восстановлением после возврата сети.

## Развёрнутый ответ

Тестирование сетевых условий (network conditions testing) — важная часть мобильного тестирования, так как в отличие от десктопа мобильные устройства часто меняют тип соединения и качество сети. Основные сценарии:

1. **Полное отсутствие сети (Airplane mode / отключенный Wi-Fi и мобильные данные).**
   - Приложение должно показывать понятное сообщение об ошибке ("Нет подключения к интернету"), а не падать или зависать.
   - Данные, которые уже были загружены (кэш), должны оставаться доступны, если это предусмотрено (офлайн-режим).
   - Действия пользователя (например, отправка формы) должны либо ставиться в очередь (offline queue) для отправки после восстановления сети, либо явно блокироваться с уведомлением.

2. **Переключение сети (Wi-Fi &lt;-> мобильные данные, 4G &lt;-> 5G).**
   - Активные соединения (загрузка файла, видеозвонок, стриминг) не должны обрываться безвозвратно — приложение должно переподключаться прозрачно для пользователя.
   - Проверяется корректность повторной аутентификации сессии, если сервер требует новый токен при смене IP.

3. **Медленное/нестабильное соединение (throttling, packet loss, high latency).**
   - Проверяется наличие таймаутов запросов и корректная обработка "зависших" запросов (loading spinner не должен крутиться бесконечно).
   - Проверяется деградация UX — например, показ низкокачественных изображений при медленном интернете (lazy loading, progressive image loading).

4. **Кратковременный обрыв связи (flaky network).**
   - Retry-механизмы: приложение должно повторять неудачные запросы автоматически (с ограничением числа попыток), а не требовать от пользователя вручную перезапускать экран.

5. **Восстановление сети после длительного отсутствия.**
   - Синхронизация отложенных действий (offline-first архитектура), обновление устаревших данных, отсутствие дублирования отправленных ранее действий (idempotency).

Инструменты для симуляции сетевых условий: встроенные настройки эмулятора/симулятора (Network Link Conditioner на iOS, Android Emulator Extended Controls → Cellular), прокси-инструменты (Charles Proxy, mitmproxy) для эмуляции задержек и потери пакетов, а также физическое отключение Wi-Fi/переход в Airplane mode на реальном устройстве.

## Пример / когда применяется

Пример теста для приложения-мессенджера: пользователь пишет сообщение → выключается Wi-Fi → сообщение должно остаться в статусе "отправляется"/"не отправлено", а не пропасть → включается сеть → сообщение автоматически отправляется без дублирования и без потери порядка сообщений.

```text
Матрица сетевых сценариев для тестирования:
- No network (Airplane mode) → корректная ошибка, нет краша
- Wi-Fi → Mobile data (переключение на ходу) → сессия не рвётся
- Slow 2G / High latency (throttling) → таймауты и retry работают
- Network drop mid-request (обрыв во время запроса) → корректный retry, нет дублей
- Network restored after long offline → синхронизация без потери данных
```

## На что смотрит интервьюер

- Знание конкретных инструментов эмуляции сети (Network Link Conditioner, Charles Proxy, настройки эмулятора).
- Понимание концепций offline-first, retry, idempotency запросов.
- Follow-up: "Как вы протестируете, что повторная отправка запроса при восстановлении сети не создаёт дубликат заказа?"
- Красный флаг — ответ ограничивается фразой "просто выключить Wi-Fi и посмотреть на ошибку" без понимания глубины сценариев (retry, sync, идемпотентность).
