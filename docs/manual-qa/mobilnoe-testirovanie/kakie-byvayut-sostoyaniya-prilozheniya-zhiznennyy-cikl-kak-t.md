# Какие бывают состояния приложения (жизненный цикл)? Как тестировать переходы?

**Коротко:** мобильное приложение проходит через состояния Not Running, Foreground (Active), Background, Suspended и Terminated, и тестирование должно проверять корректность сохранения и восстановления данных при каждом переходе между этими состояниями.

## Развёрнутый ответ

Жизненный цикл приложения (App Lifecycle) описывает набор состояний, через которые проходит приложение с момента запуска до полного завершения. Хотя терминология немного отличается между Android и iOS, общая модель схожа:

- **Not Running** — приложение не запущено, полностью выгружено из памяти.
- **Foreground / Active (на iOS) / Resumed (на Android)** — приложение открыто и находится на переднем плане, пользователь активно с ним взаимодействует.
- **Inactive (iOS)** — короткое переходное состояние, когда приложение видно, но не принимает события (например, во время входящего звонка или системного алерта поверх приложения).
- **Background** — приложение свёрнуто (пользователь нажал Home или переключился на другое приложение), но процесс ещё жив в памяти и может выполнять ограниченные фоновые задачи (загрузка, воспроизведение аудио, геолокация).
- **Suspended (iOS) / Paused-Stopped (Android)** — приложение в фоне, но система приостановила выполнение кода — приложение "заморожено" в памяти, но не выполняет никаких операций.
- **Terminated/Killed** — приложение полностью выгружено из памяти системой (обычно для освобождения ресурсов) или пользователем вручную через диспетчер задач.

На Android соответствующие колбэки жизненного цикла Activity: `onCreate → onStart → onResume → onPause → onStop → onDestroy` (плюс `onRestart` при повторном открытии). На iOS: `application(_:didFinishLaunchingWithOptions:) → applicationDidBecomeActive → applicationWillResignActive → applicationDidEnterBackground → applicationWillEnterForeground → applicationWillTerminate`.

Что нужно тестировать при переходах между состояниями:

1. **Foreground → Background → Foreground.** Данные, введённые пользователем (незавершённая форма, позиция скролла, состояние выбранных фильтров), должны сохраняться при возврате.
2. **Background → Terminated (killed системой из-за нехватки памяти) → повторный запуск.** Приложение должно восстановить последнее состояние экрана (state restoration), а не всегда открывать стартовый экран заново — либо явно предупредить пользователя о сбросе, если это ожидаемое поведение.
3. **Принудительное закрытие пользователем через диспетчер задач.** После повторного запуска приложение должно корректно инициализироваться "с нуля" (cold start), не показывая устаревшие/битые данные из кэша.
4. **Смена ориентации экрана** во время активного состояния — часто провоцирует пересоздание Activity на Android (`onDestroy`/`onCreate`), важно проверить, что данные не теряются.
5. **Фоновые задачи** — воспроизведение музыки, загрузка файла, определение геолокации — должны либо продолжаться в background в рамках ограничений ОС, либо корректно приостанавливаться и возобновляться.
6. **Push-уведомление, приводящее к открытию приложения из killed-состояния** — должен сработать полноценный cold start с последующей навигацией на нужный экран.
7. **Быстрое переключение между несколькими приложениями (App Switcher)** — проверка отсутствия утечек памяти и деградации производительности при многократных переходах.

## Пример / когда применяется

Пример тест-кейса: пользователь заполняет длинную анкету → сворачивает приложение (Background) → открывает другое приложение и работает там 10 минут → система из-за нехватки памяти закрывает процесс анкеты (Terminated) → пользователь возвращается через App Switcher → приложение делает cold start → ожидаемый результат: приложение восстанавливает введённые данные анкеты (если реализовано сохранение черновика) либо явно предупреждает о необходимости начать заново, но не падает и не показывает пустой/битый экран.

```text
Матрица переходов для тестирования:
- Foreground -> Background -> Foreground (быстрый Home + возврат)
- Foreground -> Background -> Terminated (killed OS) -> Cold start
- Активный экран -> поворот экрана -> проверка сохранности данных
- Push-уведомление -> запуск из Terminated -> навигация на нужный экран
- Быстрое многократное сворачивание/разворачивание -> проверка memory leaks
```

## На что смотрит интервьюер

- Знание конкретных колбэков жизненного цикла (onPause/onStop на Android, applicationDidEnterBackground на iOS) и понимание разницы между "просто в фоне" и "убито системой".
- Умение объяснить, зачем нужен state restoration и как его тестировать.
- Follow-up: "Как вы отличите на тестировании обычный возврат из фона от восстановления после полного убийства процесса системой?" (можно принудительно убить процесс через adb/Xcode debug menu и сравнить поведение).
- Красный флаг — если кандидат не различает background и terminated состояния или не упоминает необходимость тестировать оба сценария отдельно.
