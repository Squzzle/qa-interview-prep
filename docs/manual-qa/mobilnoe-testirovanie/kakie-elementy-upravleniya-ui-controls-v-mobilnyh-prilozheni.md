# Какие **элементы управления (UI controls)** в мобильных приложениях вы знаете?

**Коротко:** в мобильных приложениях используются стандартные элементы управления платформы — кнопки, поля ввода, переключатели, списки, слайдеры, вкладки, навигационные панели, которые нужно тестировать не только на функциональность, но и на удобство взаимодействия пальцем.

## Развёрнутый ответ

Основные категории UI-элементов управления в мобильных приложениях:

- **Кнопки (Buttons)** — обычные, иконочные, floating action button (FAB на Android). Проверяется область нажатия (tap target), состояния (disabled/enabled/pressed).
- **Поля ввода (Text Fields/Input)** — с валидацией, масками ввода, автозаполнением, разными типами клавиатур (числовая, email).
- **Переключатели** — Switch (тумблер), Checkbox, Radio Button — для выбора одного/нескольких вариантов.
- **Списки и коллекции** — ListView/RecyclerView (Android), TableView/CollectionView (iOS) — со скроллом, pull-to-refresh, ленивой подгрузкой (pagination/infinite scroll).
- **Слайдеры и Seek Bar** — для выбора значения из диапазона (громкость, прогресс видео).
- **Вкладки (Tabs) и нижняя навигация (Bottom Navigation)** — переключение между разделами приложения.
- **Модальные окна, Bottom Sheet, Alert/Dialog** — всплывающие элементы для подтверждений и выбора.
- **Picker/Date-Time Picker** — колесо выбора даты, времени, значений.
- **Progress Bar / Spinner / Skeleton loader** — индикаторы загрузки.
- **Swipe-элементы** — карточки Tinder-like, swipe-to-delete в списках.
- **Segmented Control** (iOS) — аналог радиокнопок в виде сегментов.

Специфика тестирования UI controls на мобильных устройствах: важно проверять размер зоны нажатия (рекомендуется не менее 44x44pt на iOS / 48x48dp на Android — иначе пользователю сложно попасть пальцем), поведение при быстром множественном тапе (debounce), корректность работы с виртуальной клавиатурой (перекрытие полей ввода), адаптацию под разные размеры экрана и плотность пикселей (DPI), доступность (accessibility) — работу с TalkBack/VoiceOver, достаточный цветовой контраст.

## Пример / когда применяется

Пример UI-controls чек-листа для формы регистрации:

| Элемент | Что проверять |
|---|---|
| Text Field "Email" | Валидация формата, показ клавиатуры с "@", очистка при ошибке |
| Switch "Согласие с условиями" | Кнопка "Продолжить" неактивна, пока Switch выключен |
| Button "Зарегистрироваться" | Область нажатия достаточна, disabled-состояние до валидации |
| Date Picker "Дата рождения" | Ограничение диапазона (нельзя выбрать будущую дату) |

## На что смотрит интервьюер

- Способность назвать не абстрактные "кнопки и поля", а специфичные для мобильных платформ элементы (Bottom Sheet, FAB, Segmented Control).
- Понимание, что тестирование UI-контролов на мобильных отличается от веба зоной нажатия и жестами.
- Follow-up: "Как тестировать элемент, перекрытый виртуальной клавиатурой?"
- Красный флаг — если кандидат перечисляет только веб-элементы без учёта мобильной специфики (тач, размер экрана).
