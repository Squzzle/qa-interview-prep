# Какие **виды мобильных приложений** существуют?

**Коротко:** существует три основных вида мобильных приложений — нативные (native), веб-приложения (mobile web) и гибридные (hybrid), а также промежуточный вариант — кросс-платформенные приложения на общем коде и PWA.

## Развёрнутый ответ

Мобильные приложения принято классифицировать по способу разработки и запуска:

- **Нативные (Native)** — разрабатываются под конкретную платформу с использованием её родного языка и SDK: Kotlin/Java с Android SDK для Android, Swift/Objective-C с UIKit/SwiftUI для iOS. Устанавливаются через магазины приложений (Google Play, App Store), имеют полный доступ к аппаратным возможностям устройства (камера, GPS, датчики, Bluetooth) и максимальную производительность.
- **Веб-приложения (Mobile Web / Web Apps)** — обычные сайты, адаптированные под мобильный экран (responsive design), открываются через браузер, не требуют установки. Работают на любой платформе с браузером, но ограничены в доступе к аппаратным функциям устройства и зависят от подключения к сети.
- **Гибридные (Hybrid)** — веб-приложение, обёрнутое в нативный контейнер (WebView) с помощью фреймворков типа Cordova/PhoneGap, Ionic. Устанавливаются как обычное приложение из магазина, но интерфейс рендерится через HTML/CSS/JS внутри WebView.
- **Кросс-платформенные (Cross-platform)** — компилируются в нативный код из единой кодовой базы: React Native, Flutter, Xamarin/.NET MAUI. Занимают промежуточное положение между нативными и гибридными — код общий, но итоговое приложение ближе к нативному по производительности и внешнему виду.
- **PWA (Progressive Web App)** — веб-приложение с расширенными возможностями: работа офлайн, push-уведомления, установка на домашний экран, использование части нативных API через Service Worker.

Различие между видами напрямую влияет на стратегию тестирования: для нативных приложений критичны разрешения ОС, аппаратные датчики и жизненный цикл (lifecycle) конкретной платформы; для веб — кроссбраузерность и адаптивность; для гибридных — производительность WebView и корректность передачи данных между JS-слоем и нативным мостом (bridge); для кросс-платформенных — единообразие поведения на разных ОС при общем коде.

## Пример / когда применяется

| Тип | Пример | Особенность тестирования |
|---|---|---|
| Native | Instagram (Kotlin/Swift) | Тест доступа к камере, разрешений ОС |
| Web | Mobile-версия сайта интернет-магазина | Тест адаптивности, кроссбраузерность |
| Hybrid | Приложение на Ionic/Cordova | Тест производительности WebView, JS-bridge |
| Cross-platform | Приложение на Flutter/React Native | Тест единообразия UI на Android/iOS |
| PWA | Twitter Lite | Тест офлайн-режима, установки на экран |

## На что смотрит интервьюер

- Знает ли кандидат разницу не только по названию, но и по последствиям для тест-стратегии (доступ к железу, офлайн-работа, установка).
- Частая ошибка — путать гибридные и кросс-платформенные приложения, считая их одним и тем же.
- Follow-up: "Какие риски специфичны для гибридных приложений?" (WebView-баги, устаревшие версии рендерера на старых Android).
- Красный флаг — если кандидат не может привести примеры реальных приложений каждого типа.
