# Назовите **критерии выбора устройства для тестирования**.

**Коротко:** Устройства для тестирования выбирают на основе аналитики реальной аудитории (доли рынка по моделям, версиям ОС, размерам экрана), учитывая производительность, разрешение экрана, версию ОС, наличие специфичных аппаратных особенностей и требования бизнеса.

## Развёрнутый ответ

Так как охватить все существующие модели устройств невозможно (особенно в Android-фрагментации), выбор тестового набора устройств строится на основе приоритизации по реальным данным и рискам. Основные критерии:

1. **Аналитика аудитории продукта.** Данные из систем аналитики (Google Analytics, Firebase, App Store Connect Analytics) о том, какими устройствами и версиями ОС реально пользуются пользователи приложения — устройства с наибольшей долей должны быть в приоритете.
2. **Версия операционной системы.** Минимальная поддерживаемая версия (min SDK / minimum iOS version) и последняя актуальная версия ОС — обязательно тестируются обе границы, плюс 1-2 промежуточные популярные версии.
3. **Производительность устройства (low-end / mid-end / high-end).** Слабые устройства с малым объёмом RAM и старым процессором нужны для проверки производительности, отсутствия лагов, корректной работы при ограниченных ресурсах.
4. **Размер и разрешение экрана, соотношение сторон.** Разные диагонали, DPI, наличие "чёлки"/выреза камеры, safe area, планшеты и складные устройства — для проверки адаптивности вёрстки.
5. **Производитель и кастомизация ОС (для Android).** Разные вендоры (Samsung, Xiaomi, Huawei и др.) модифицируют стандартный Android своими оболочками (One UI, MIUI и т.д.), что может влиять на поведение уведомлений, фоновых процессов, разрешений.
6. **Аппаратные особенности.** Наличие NFC, датчика отпечатка пальца/Face ID, конкретной камеры, поддержки 5G, GPS-модуля — если функциональность приложения их использует.
7. **Региональная специфика.** Локаль, язык, форматы даты/валюты, а также устройства, популярные в целевом регионе (например, определённые бюджетные модели в конкретной стране).
8. **Требования бизнеса и SLA.** Список поддерживаемых устройств может быть явно закреплён в требованиях продукта или в контракте с заказчиком.
9. **История багов.** Устройства/версии ОС, на которых ранее находили специфичные дефекты, стоит держать в регрессионном наборе.
10. **Наличие сетевых условий.** Устройства с поддержкой разных типов подключения (Wi-Fi, 4G/5G, отсутствие SIM) для проверки сетевых сценариев.

На практике используется комбинация реальных устройств (для ключевых моделей топ аудитории) и эмуляторов/симуляторов или облачных ферм устройств (BrowserStack, Firebase Test Lab, Sauce Labs) — для расширения покрытия без покупки физических устройств.

## Пример / когда применяется

Пример построения матрицы устройств для мобильного банковского приложения:

| Критерий | Выбор |
|---|---|
| Топ-3 модели по аналитике | Samsung Galaxy A54, iPhone 13, Xiaomi Redmi Note 12 |
| Минимальная версия ОС | Android 9, iOS 15 |
| Последняя версия ОС | Android 14, iOS 17 |
| Low-end устройство | Устройство с 2 ГБ RAM для проверки производительности |
| Особенности | Устройство с датчиком отпечатка пальца и Face ID для проверки биометрической авторизации |
| Экран | Устройство с "чёлкой"/вырезом камеры для проверки safe area |

## На что смотрит интервьюер

Интервьюер ожидает, что кандидат упомянет данные аналитики как основной источник приоритизации, а не выбор "наугад" или "какие есть в шкафу". Плюс — если кандидат знает про облачные фермы устройств как способ расширить покрытие. Красный флаг — выбор только топовых флагманов без учёта low-end устройств и версий ОС на границах поддержки.
