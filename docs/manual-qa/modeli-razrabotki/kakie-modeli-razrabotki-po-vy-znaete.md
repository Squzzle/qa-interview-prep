# Какие модели разработки ПО вы знаете?

**Коротко:** Основные модели — Waterfall (водопадная), итеративная модель, спиральная модель, V-модель, Agile (Scrum, Kanban), а также инкрементальная модель и подход DevOps.

## Развёрнутый ответ

Модель разработки ПО — это подход к организации процесса создания продукта, определяющий порядок и характер выполнения фаз SDLC. Наиболее часто упоминаемые модели:

- **Waterfall (водопадная)** — строго последовательное прохождение стадий, каждая следующая начинается только после завершения предыдущей.
- **Итеративная модель** — продукт разрабатывается небольшими циклами (итерациями), каждая из которых добавляет функциональность.
- **Инкрементальная модель** — похожа на итеративную, но система собирается по частям (инкрементам), каждый из которых представляет собой законченный работающий кусок функциональности.
- **Спиральная модель (Spiral)** — сочетает итеративность с явным акцентом на анализ и снижение рисков на каждом витке спирали.
- **V-модель (Verification and Validation model)** — расширение Waterfall, где каждой стадии разработки соответствует своя стадия тестирования.
- **Agile-модели** — семейство гибких методологий (Scrum, Kanban, XP, Lean), основанных на коротких итерациях, постоянной обратной связи от заказчика и адаптивности к изменениям.
- **RAD (Rapid Application Development)** — акцент на быстром прототипировании и вовлечении пользователей.
- **DevOps** — не модель разработки в классическом смысле, а культура/практика, объединяющая разработку и эксплуатацию, часто накладывается поверх Agile для ускорения релизов через автоматизацию (CI/CD).

Выбор модели зависит от многих факторов: стабильности требований, размера и критичности проекта, вовлечённости заказчика, сроков и бюджета. Для проектов с чётко зафиксированными требованиями (например, госконтракты) чаще применяют Waterfall или V-модель, а для продуктов с изменчивыми требованиями и необходимостью быстрого вывода на рынок — Agile.

## Пример / когда применяется

Например, разработка бортового ПО для медицинского оборудования, где требования жёстко регламентированы и цена ошибки высока, часто ведётся по V-модели с полным набором формальных верификаций. А разработка мобильного приложения стартапа, где фичи часто меняются по фидбеку пользователей, скорее будет вестись по Scrum с двухнедельными спринтами.

| Модель | Когда применяется |
|---|---|
| Waterfall | Стабильные требования, регуляторные проекты |
| Итеративная/инкрементальная | Постепенное наращивание функциональности |
| Спиральная | Крупные проекты с высокими рисками |
| V-модель | Критичные системы, где важна верификация на каждом шаге |
| Agile (Scrum/Kanban) | Изменчивые требования, быстрый вывод продукта |

## На что смотрит интервьюер

Важно, чтобы кандидат не просто перечислил названия, а понимал ключевое отличие каждой модели и мог привести пример подходящего проекта. Красный флаг — путаница между моделью разработки и методологией управления проектом (например, Scrum — это фреймворк управления, применяемый в рамках Agile-модели). Частые follow-up вопросы: "В каком проекте вы бы выбрали Waterfall, а в каком Agile и почему?", "Чем отличается итеративная модель от инкрементальной?".
