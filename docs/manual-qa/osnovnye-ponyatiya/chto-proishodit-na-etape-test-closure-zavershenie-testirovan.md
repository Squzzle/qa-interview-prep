# Что происходит на этапе **Test Closure** (завершение тестирования)?

**Коротко:** Test Closure — финальный этап STLC, на котором подводятся итоги цикла тестирования: проверяется выполнение критериев выхода, собирается статистика по дефектам и покрытию, формируется итоговый отчёт, архивируются тестовые артефакты и фиксируются извлечённые уроки для будущих проектов.

## Развёрнутый ответ

Test Closure наступает после того, как основное выполнение тестов (Test Execution) завершено, и включает следующие ключевые активности:

1. **Проверка критериев выхода (exit criteria).** Команда сверяет фактический результат с заранее согласованными критериями завершения тестирования: достигнут ли требуемый процент покрытия тест-кейсами, все ли критичные и блокирующие дефекты закрыты, приемлемо ли количество открытых дефектов низкого приоритета.
2. **Анализ незакрытых дефектов.** Оставшиеся открытые баги классифицируются: одни переносятся в следующий релиз (deferred), для других принимается решение "won't fix", по критичным дефектам эскалируется вопрос о переносе даты релиза.
3. **Подготовка итогового отчёта о тестировании (Test Summary Report).** В отчёте фиксируются: объём выполненного тестирования, количество выполненных/пройденных/упавших тест-кейсов, статистика по найденным дефектам (по severity/priority), риски, оставшиеся на момент релиза, рекомендации команде и стейкхолдерам (например, "рекомендуем отложить релиз из-за незакрытого критичного дефекта X").
4. **Архивирование тестовых артефактов.** Тест-кейсы, тестовые данные, скрипты автоматизации, отчёты о дефектах сохраняются в системе управления тестами (TMS) или репозитории для использования в будущих релизах и регрессионных наборах.
5. **Освобождение ресурсов тестового окружения.** Тестовые стенды, временные доступы, тестовые данные могут быть очищены или переданы другой команде, если они больше не нужны.
6. **Ретроспектива и извлечённые уроки (Lessons Learned).** Команда обсуждает, что прошло хорошо, а что можно улучшить в процессе тестирования: были ли проблемы с окружением, качеством требований, коммуникацией с разработкой, точностью оценки сроков. Эти выводы используются для улучшения процесса в следующих циклах.

Test Closure важен не только как формальность — это точка, где команда объективно оценивает качество продукта перед релизом и передаёт эту информацию заинтересованным сторонам (менеджменту, продакту) для принятия решения "готовы ли мы релизить".

## Пример / когда применяется

Пример из практики: в конце регрессионного цикла перед релизом QA-лид готовит Test Summary Report, где указано: выполнено 320 из 325 тест-кейсов (98.5%), найдено 45 дефектов, из них 40 закрыто, 3 отложены на следующий релиз как некритичные, 2 критичных дефекта требуют исправления до релиза. На основании этого отчёта релиз-менеджер принимает решение отложить дату релиза на два дня до исправления критичных дефектов.

Пример структуры отчёта:

| Метрика                          | Значение |
|-----------------------------------|----------|
| Всего тест-кейсов                  | 325      |
| Выполнено                          | 320      |
| Пройдено успешно                   | 300      |
| Найдено дефектов                   | 45       |
| Критичных дефектов открыто          | 2        |
| Рекомендация                       | Отложить релиз |

## На что смотрит интервьюер

- Понимание, что Test Closure — не просто "тестирование закончилось", а формальный анализ результатов с конкретными артефактами (отчёт, метрики, решение по открытым дефектам).
- Знание, что на этом этапе принимается go/no-go решение относительно готовности к релизу на основе критериев выхода.
- Способность объяснить ценность архивирования тест-кейсов и ретроспективы для будущих проектов.
- Красный флаг — если кандидат сводит Test Closure только к "написанию отчёта" и не упоминает анализ открытых дефектов, критерии выхода или lessons learned.
