# Какие виды тестовых сред вы знаете (DEV, QA, STAGING, PROD)?

**Коротко:** типовой конвейер сред — DEV (разработка), QA/TEST (тестирование), STAGING (предпродакшн, максимально похож на PROD) и PROD (боевая среда для реальных пользователей); между ними код проходит поэтапную проверку перед релизом.

## Развёрнутый ответ

Среда (environment) — это отдельная инфраструктура (сервер, база данных, конфигурация, набор сервисов), на которой разворачивается версия приложения для определённых целей. Основная идея разделения сред — не допустить, чтобы непроверенный код повлиял на реальных пользователей, и дать каждой роли (разработчику, тестировщику, заказчику) изолированное пространство для работы.

Типичный набор сред:

- **DEV (Development)** — среда разработки. Здесь разработчики пишут и отлаживают код, разворачивают самые свежие и нестабильные сборки. Данные обычно тестовые, синтетические, среда может часто падать и пересобираться. Тестировщики иногда заходят сюда для раннего смоук-тестирования новых фич.
- **QA / TEST** — среда для тестировщиков. Сюда попадают более стабильные сборки (обычно после прохождения CI и базовых проверок), на которых проводится функциональное, регрессионное, интеграционное тестирование. Может быть несколько QA-сред (например, под разные ветки или команды).
- **STAGING (предпродакшн, pre-prod, UAT)** — среда, максимально приближенная по конфигурации, инфраструктуре и объёму данных к продакшну. Используется для финальной проверки перед релизом: приёмочного тестирования (UAT), нагрузочного тестирования, проверки миграций и деплой-скриптов.
- **PROD (Production)** — боевая среда, с которой работают реальные пользователи. Любые изменения сюда попадают только после прохождения всех предыдущих этапов, изменения максимально контролируются (feature flags, canary-релизы, мониторинг).

Дополнительно на проектах встречаются: **SIT** (System Integration Testing — среда для проверки интеграции между системами), **PERF/LOAD** (отдельная среда для нагрузочного тестирования, чтобы не мешать другим тестировщикам), **DEMO** (для показа заказчику), **Sandbox** (изолированная песочница для экспериментов, часто при интеграции с внешними API, например платёжными шлюзами).

## Пример / когда применяется

| Среда   | Кто использует         | Стабильность | Данные                 |
|---------|-------------------------|--------------|-------------------------|
| DEV     | Разработчики             | Низкая       | Синтетические, моки      |
| QA      | Тестировщики             | Средняя      | Тестовые наборы данных   |
| STAGING | QA, продакт, заказчик    | Высокая      | Близкие к боевым (обезличенные) |
| PROD    | Реальные пользователи    | Максимальная | Реальные                |

Пример потока фичи: разработчик закоммитил код → автосборка развёрнута на DEV → после прохождения unit- и smoke-тестов сборка попадает на QA → тестировщики проводят регрессию → успешная сборка деплоится на STAGING для UAT и нагрузочных тестов → после одобрения релиз выкатывается на PROD, часто поэтапно (canary/blue-green).

## На что смотрит интервьюер

- Понимает ли кандидат, зачем нужно разделение сред, а не просто перечисляет названия.
- Знает ли про дополнительные среды (SIT, Perf, Sandbox) и умеет объяснить их назначение.
- Может ли объяснить, чем данные и конфигурация на STAGING отличаются от PROD и почему это важно (например, обезличивание персональных данных).
- Частая ошибка — путать STAGING и PROD или считать, что STAGING не нужен, если есть QA.
- Follow-up: "Что делать, если баг воспроизводится только на STAGING/PROD, но не на QA?" — ожидается разговор про различие конфигураций, объёма данных, внешних интеграций.
