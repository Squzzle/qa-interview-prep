# Какие **виды валидации на форме** вы знаете (мгновенная, ограничивающая, отложенная)?

**Коротко:** На формах различают мгновенную валидацию (inline, срабатывает сразу при вводе или потере фокуса), ограничивающую валидацию (preventive, не даёт ввести некорректные данные физически) и отложенную валидацию (проверка выполняется после отправки формы, например при клике на кнопку "Отправить").

## Развёрнутый ответ

**Мгновенная (inline / real-time) валидация** — проверка данных происходит непосредственно во время ввода или сразу после того, как пользователь покинул поле (событие blur). Пользователь получает обратную связь мгновенно, не дожидаясь отправки формы. Пример: при вводе email система сразу подсвечивает поле красным, если формат некорректен, ещё до перехода к следующему полю.

**Ограничивающая (restrictive / preventive) валидация** — система физически не позволяет ввести недопустимые данные. Например, поле для номера телефона принимает только цифры, попытка ввести букву просто игнорируется полем ввода. Сюда же относится ограничение максимальной длины ввода через атрибут maxlength или использование выпадающих списков вместо свободного ввода текста, что структурно исключает часть невалидных значений.

**Отложенная (deferred / on submit) валидация** — проверка данных происходит только после того, как пользователь попытался отправить форму (нажал кнопку "Отправить" / Submit). Все ошибки показываются одновременно после этого действия. Это более традиционный подход, менее удобный с точки зрения UX, так как пользователь узнаёт об ошибках только в конце заполнения формы.

Часто на практике используется комбинация подходов: ограничивающая валидация применяется на уровне ввода символов (например, только цифры в поле телефона), мгновенная — для полей, которые можно проверить сразу (email, совпадение паролей), а отложенная (в том числе серверная валидация) — как финальная проверка перед сохранением данных, включая проверки, требующие обращения к серверу (например, уникальность логина).

## Пример / когда применяется

Пример формы регистрации:
- Поле "Логин" — при вводе символов, не разрешённых в логине (например, пробел), символ просто не появляется в поле (ограничивающая валидация).
- Поле "Email" — после того как пользователь кликнул за пределы поля, появляется сообщение "Некорректный формат email" (мгновенная валидация).
- При нажатии кнопки "Зарегистрироваться" форма проверяет, что логин уникален (запрос к серверу), и если нет — показывает ошибку под полем (отложенная валидация).

## На что смотрит интервьюер

Интервьюер проверяет, понимает ли кандидат разницу между этими видами с точки зрения UX и технической реализации, и может ли он на реальном примере (форма логина, чекаут в интернет-магазине) объяснить, где какой вид валидации уместен. Частый follow-up — вопрос о клиентской и серверной валидации и о том, почему нельзя полагаться только на клиентскую. Красный флаг — если кандидат не может привести конкретные примеры полей и типов ошибок для каждого вида.
