# Приведите примеры методов верификации и валидации.

**Коротко:** Методы верификации — это в основном статические техники (ревью, инспекции, статический анализ, аудит требований), а методы валидации — это динамические техники, где продукт реально запускается и проверяется (функциональное, приёмочное, юзабилити- и бета-тестирование).

## Развёрнутый ответ

Методы верификации (статические, без выполнения кода):
- Ревью требований и дизайн-документов (requirements review, design review) — коллеги или заказчик просматривают документацию на предмет полноты и корректности.
- Инспекции (inspection) — формализованный процесс проверки документов или кода по чек-листу с назначенными ролями (модератор, автор, ревьюеры).
- Технический просмотр / walkthrough — автор проводит коллег по документу или коду, объясняя логику, чтобы выявить несоответствия.
- Статический анализ кода (static analysis) — использование инструментов (линтеров, анализаторов) для проверки кода без его выполнения на предмет соответствия стандартам кодирования.
- Аудит трассируемости требований (traceability audit) — проверка, что каждое требование покрыто соответствующим элементом дизайна и кода.
- Peer review / код-ревью — проверка кода коллегами перед мержем.

Методы валидации (динамические, с реальным выполнением продукта):
- Функциональное тестирование — проверка, что функции системы работают в соответствии с ожиданиями пользователя.
- Приёмочное тестирование (User Acceptance Testing, UAT) — заказчик или конечные пользователи проверяют, что система решает их реальные задачи.
- Бета-тестирование — ограниченная группа реальных пользователей использует продукт до релиза в реальных условиях.
- Юзабилити-тестирование (usability testing) — проверка удобства использования продукта реальными пользователями.
- Альфа-тестирование — тестирование внутри компании-разработчика перед выходом к внешним пользователям.
- Эксплуатационное тестирование в реальных или приближенных к реальным условиях (operational acceptance testing).

## Пример / когда применяется

Пример применения: перед релизом мобильного приложения команда сначала проводит верификацию — ревью технического задания и дизайн-макетов, статический анализ кода на CI. Затем, когда сборка готова, проводится валидация — бета-версия выкладывается ограниченной группе пользователей, которые дают обратную связь по удобству использования (юзабилити) и соответствию их ожиданиям.

| Этап | Метод верификации | Метод валидации |
|---|---|---|
| Требования | Ревью требований | — |
| Дизайн | Инспекция дизайн-документа | — |
| Код | Статический анализ, код-ревью | — |
| Готовый продукт | Аудит трассируемости | UAT, бета-тест, юзабилити-тест |

## На что смотрит интервьюер

Интервьюер хочет услышать конкретные названия техник, а не общие фразы. Важно, чтобы кандидат правильно распределил методы между верификацией (статические) и валидацией (динамические), не перепутав их местами. Частая ошибка — назвать юнит-тестирование методом верификации только потому, что оно проводится рано; на самом деле юнит-тесты — динамический метод (код выполняется), хотя и относится к ранним стадиям. Хороший кандидат уточнит этот нюанс.
