# Какие критерии включает в себя Definition of Done (DoD)?

**Коротко:** Definition of Done (DoD) — это согласованный командой набор критериев, при выполнении которых задача считается полностью завершённой; обычно включает написание и прохождение кода, тестов, ревью, документации и развёртывание в нужном окружении.

## Развёрнутый ответ

DoD — это чек-лист, определяющий, что означает «готово» для любой задачи, пользовательской истории или релиза в команде. В отличие от DoR (который проверяет готовность задачи к началу работы), DoD применяется в конце — для подтверждения, что работа действительно завершена, а не просто «код написан».

Типичные критерии, которые может включать DoD:

- **Код написан и соответствует стандартам кодирования** команды (code style, naming conventions).
- **Код прошёл ревью (code review)** минимум одним другим разработчиком.
- **Написаны и проходят unit-тесты** с определённым уровнем покрытия кода (например, не менее 80%).
- **Пройдено функциональное тестирование** — QA-инженер проверил историю на соответствие критериям приёмки (acceptance criteria).
- **Пройдено регрессионное тестирование** — убедились, что новая функциональность не сломала существующую.
- **Устранены все критичные и блокирующие дефекты**, найденные при тестировании.
- **Обновлена документация** — техническая документация, API-документация, пользовательские инструкции при необходимости.
- **Проверены нефункциональные требования**, если применимо (производительность, доступность, безопасность).
- **Код смёржен в основную ветку** и прошёл сборку в CI/CD без ошибок.
- **Функциональность развёрнута в нужном окружении** (staging или production) и доступна для демонстрации.
- **Продакт-оунер принял результат** и подтвердил соответствие бизнес-требованиям.

DoD может формироваться на разных уровнях: для отдельной задачи (story-level DoD), для спринта (sprint-level DoD) и для релиза (release-level DoD) — с разной степенью строгости критериев.

Ключевая цель DoD — обеспечить единое понимание «готовности» во всей команде, избежать ситуаций, когда разработчик считает задачу выполненной («код написан»), а тестировщик или продакт-оунер — нет («не протестировано», «не соответствует требованиям»). Это снижает технический долг и скрытую недоделанную работу, которая накапливается от спринта к спринту.

## Пример / когда применяется

Пример DoD для команды веб-разработки:

| Критерий | Обязателен |
|---|---|
| Код прошёл ревью | Да |
| Unit-тесты написаны и проходят (покрытие ≥ 80%) | Да |
| QA протестировал по критериям приёмки | Да |
| Нет открытых критичных багов | Да |
| Обновлена документация API | Да, если менялся API |
| Проведено нагрузочное тестирование | Да, для критичных модулей |
| Задеплоено на staging | Да |

Если задача «Добавить фильтр по цене в каталоге» реализована, но unit-тесты не написаны или QA не успел протестировать, задача не может считаться выполненной (Done) и остаётся в статусе «In Progress» на следующий спринт.

## На что смотрит интервьюер

Интервьюер проверяет, различает ли кандидат DoD и DoR, а также понимает ли он роль тестирования в критериях DoD (не просто «код написан», а «протестировано и соответствует требованиям»). Частая ошибка — сводить DoD только к «написан код и смёржен».

Красный флаг — если кандидат не упоминает тестирование как обязательный критерий DoD или не понимает, зачем вообще нужен формализованный DoD.

Возможный follow-up: «Что делать, если задача не проходит DoD, а спринт заканчивается?» — ожидается ответ про перенос задачи в следующий спринт незавершённой (не засчитывать как «Done» ради красивых цифр в отчётности), поскольку нарушение DoD ведёт к скрытому техническому долгу.
