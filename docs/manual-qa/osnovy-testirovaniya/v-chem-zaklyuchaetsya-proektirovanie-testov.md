# В чем заключается проектирование тестов?

**Коротко:** Проектирование тестов (Test Design) — это преобразование тестовых условий, выявленных на этапе анализа, в конкретные тест-кейсы: с точными шагами, входными данными и ожидаемым результатом, с применением техник тест-дизайна.

## Развёрнутый ответ

Проектирование тестов — третья активность фундаментального процесса тестирования (после планирования и анализа). Если на этапе анализа определялось "что тестировать" (test conditions), то на проектировании отвечают на вопрос "как именно это тестировать".

Что входит в проектирование тестов:

- **Разработка тест-кейсов** на основе тестовых условий — каждый test condition может породить один или несколько тест-кейсов (позитивные, негативные, граничные сценарии).

- **Определение конкретных тестовых данных** — точные значения для проверки (например, для граничного анализа поля "возраст от 18 до 65" — значения 17, 18, 65, 66).

- **Применение техник тест-дизайна**:
  - тест-дизайн по спецификации (black-box): эквивалентное разбиение (equivalence partitioning), анализ граничных значений (boundary value analysis), таблицы решений (decision tables), диаграммы состояний и переходов (state transition testing), попарное тестирование (pairwise), сценарии использования (use case testing);
  - структурный тест-дизайн (white-box): покрытие операторов, ветвей, условий (statement/branch/condition coverage);
  - опыт-ориентированные техники (experience-based): исследовательское тестирование, атака по чек-листам, предположение об ошибке (error guessing).

- **Определение ожидаемого результата** для каждого тест-кейса — что должно произойти, если система работает корректно.

- **Проектирование тестовой среды** на верхнем уровне — какое окружение, конфигурация, интеграции понадобятся.

- **Оценка необходимости автоматизации** — определение, какие тест-кейсы стоит проектировать сразу с прицелом на автоматизацию.

- **Приоритизация тест-кейсов** — какие сценарии критичны и должны выполняться в первую очередь (smoke, критичный путь) и какие можно отложить.

Результат этапа — набор задокументированных тест-кейсов (в виде чек-листов, таблиц, BDD-сценариев Given/When/Then или тест-скриптов), готовых к следующему этапу — реализации тестов (Test Implementation), где эти кейсы будут собраны в тестовые наборы (suites) и подготовлены к выполнению.

## Пример / когда применяется

Тестовое условие из этапа анализа: "Валидация поля 'Возраст' в форме регистрации (допустимо от 18 до 65 лет)".

На этапе проектирования тестировщик, применяя технику граничных значений, создаёт тест-кейсы:

| ID | Входные данные | Ожидаемый результат |
|---|---|---|
| TC-01 | Возраст = 17 | Ошибка "Недопустимый возраст" |
| TC-02 | Возраст = 18 | Регистрация успешна |
| TC-03 | Возраст = 65 | Регистрация успешна |
| TC-04 | Возраст = 66 | Ошибка "Недопустимый возраст" |
| TC-05 | Возраст = "abc" (нечисловое значение) | Ошибка валидации формата |
| TC-06 | Поле пустое | Ошибка "Поле обязательно" |

## На что смотрит интервьюер

Интервьюер оценивает, знает ли кандидат конкретные техники тест-дизайна и умеет ли их применить на практике (а не просто назвать термин "тест-дизайн"). Плюсом будет упоминание разницы между test condition (из анализа) и test case (результат проектирования). Красный флаг — если кандидат описывает проектирование тестов как "просто написание шагов", не упоминая техники black-box/white-box/experience-based. Возможные follow-up вопросы: "Какие техники тест-дизайна вы использовали на практике?", "Как выбрать, какую технику применить для конкретного требования?", "Чем эквивалентное разбиение отличается от анализа граничных значений?".
