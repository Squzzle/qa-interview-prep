# Что такое сниффер (анализатор трафика)? Приведите примеры (Wireshark, Fiddler).

**Коротко:** Сниффер — это инструмент, который перехватывает и анализирует сетевой трафик, проходящий через сетевой интерфейс, позволяя увидеть содержимое передаваемых пакетов, заголовков, запросов и ответов; примеры — Wireshark, Fiddler, Charles Proxy, tcpdump.

## Развёрнутый ответ

Сниффер (packet sniffer, анализатор трафика) — это программа или устройство, которое перехватывает данные, передаваемые по сети, и позволяет их просматривать, фильтровать и анализировать. Принцип работы основан на переводе сетевого интерфейса в режим "прослушивания" (promiscuous mode), при котором интерфейс принимает не только адресованные ему пакеты, но и весь трафик, проходящий через сегмент сети (в случае с хабами) либо трафик, направленный через прокси или локальный интерфейс (в случае с большинством современных инструментов на коммутируемых сетях).

Основные типы и инструменты:

- **Wireshark** — универсальный анализатор сетевого трафика с графическим интерфейсом. Перехватывает пакеты на разных уровнях модели OSI (Ethernet, IP, TCP/UDP, HTTP и др.), позволяет фильтровать трафик по протоколу, IP-адресу, порту, содержимому, строить статистику и визуализировать взаимодействие узлов.
- **tcpdump** — консольный аналог Wireshark для Linux/Unix, часто используется для захвата трафика на сервере с последующим анализом захваченного файла (`.pcap`) в Wireshark.
- **Fiddler** — HTTP/HTTPS-прокси-дебаггер, ориентированный на анализ веб-трафика (запросы и ответы браузера или приложения), удобен для отладки REST API, кэширования, редиректов, cookies.
- **Charles Proxy** — аналог Fiddler, кроссплатформенный, также используется для анализа HTTP(S)-трафика, в том числе мобильных приложений.

Снифферы работают на разных уровнях: Wireshark и tcpdump могут анализировать трафик от канального уровня (Ethernet-кадры) до прикладного уровня (HTTP, DNS и т.д.), в то время как Fiddler и Charles специализируются именно на HTTP(S)-трафике, работая как локальный прокси-сервер, через который проходят запросы приложения.

Важный нюанс — для перехвата HTTPS-трафика нужен MITM-подход (man-in-the-middle): инструмент подменяет сертификат сервера своим собственным (self-signed), который нужно установить как доверенный на клиенте, иначе браузер/приложение будет выдавать ошибку сертификата.

## Пример / когда применяется

Пример использования Wireshark для диагностики: тестировщик подозревает, что клиент не получает ответ от сервера. Запускается захват трафика на нужном интерфейсе с фильтром:

```text
ip.addr == 192.168.1.100 && tcp.port == 443
```

Это позволяет увидеть, отправляется ли запрос, доходит ли он до сервера, приходит ли ответ, есть ли ретрансмиссии (TCP retransmission) или сброс соединения (RST).

Пример использования Fiddler: тестировщик отлаживает мобильное приложение, которое обращается к REST API. В Fiddler видно тело запроса и ответа в формате JSON, заголовки (включая токены авторизации), коды статуса, что позволяет проверить корректность API-вызовов без доступа к серверным логам.

## На что смотрит интервьюер

Частые ошибки: путают сниффер общего назначения (Wireshark) с HTTP-прокси (Fiddler), не понимают разницу в уровнях анализа; не знают про необходимость установки корневого сертификата для перехвата HTTPS; путают promiscuous mode с обычным режимом сетевой карты.
