# Для чего нужен обратный прокси (Reverse Proxy)?

**Коротко:** обратный прокси принимает запросы клиентов от имени backend-серверов и решает задачи балансировки нагрузки, безопасности, кэширования и терминации SSL.

## Развёрнутый ответ

Обратный прокси располагается перед одним или несколькими backend-серверами и является для внешнего клиента единой точкой входа. Клиент обращается к обратному прокси, не зная и не заботясь о том, сколько реальных серверов находится за ним и на какой из них будет направлен запрос.

Основные задачи, которые решает обратный прокси:

- Балансировка нагрузки (load balancing): распределяет входящие запросы между несколькими экземплярами backend-сервера по алгоритмам (round-robin, least connections, по хэшу IP и т.д.), что повышает отказоустойчивость и масштабируемость системы.
- Терминация SSL/TLS: обратный прокси принимает на себя шифрованное соединение с клиентом (HTTPS) и общается с backend-серверами по обычному HTTP внутри защищённой сети, снимая нагрузку по шифрованию с backend-серверов и централизуя управление сертификатами.
- Кэширование ответов: часто запрашиваемый статичный контент (изображения, CSS, JS) может кэшироваться на уровне прокси, снижая нагрузку на backend и ускоряя ответ клиенту.
- Защита инфраструктуры: скрывает реальные IP-адреса, версии ПО и внутреннюю структуру серверов от внешнего мира, снижая поверхность атаки; может фильтровать вредоносные запросы, ограничивать частоту запросов (rate limiting), защищать от части DDoS-атак.
- Единая точка для маршрутизации (API Gateway-подобная функциональность): направление запросов на разные backend-сервисы по URL-пути или домену (например, `/api` на один сервис, `/static` — на другой).
- Сжатие ответов (gzip/brotli) и добавление security-заголовков без изменения кода приложения.

## Пример / когда применяется

Типовая архитектура высоконагруженного веб-приложения:
```text
Клиент (HTTPS) → Nginx (reverse proxy, SSL termination, балансировка)
                    ├─ App Server 1
                    ├─ App Server 2
                    └─ App Server 3
```
Клиент видит единый домен и сертификат, а Nginx распределяет запросы между тремя одинаковыми экземплярами приложения. Если один из серверов упадёт, прокси перенаправит трафик на оставшиеся, не прерывая работу для пользователя.

Другой пример — использование Nginx как обратного прокси перед несколькими микросервисами, где маршрутизация идёт по пути: `/orders` → сервис заказов, `/users` → сервис пользователей.

## На что смотрит интервьюер

Важно перечислить конкретные практические задачи (балансировка, SSL termination, кэширование, безопасность), а не ограничиваться общей фразой "прячет сервер". Частый follow-up: "чем reverse proxy отличается от load balancer" (по сути load balancer — частный случай reverse proxy), "какие инструменты используются" (Nginx, HAProxy, Traefik, Envoy), "как обратный прокси связан с API Gateway".
