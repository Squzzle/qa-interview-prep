# Что такое **фаззинг (Fuzz Testing)**?

**Коротко:** фаззинг — техника тестирования, при которой на вход системы подаются случайные, некорректные или неожиданные данные (в большом объеме, часто автоматически) с целью обнаружить сбои, падения, утечки памяти и уязвимости безопасности.

## Развёрнутый ответ

Fuzz Testing (фаззинг) — техника, при которой специальный инструмент (fuzzer) генерирует большое количество случайных, искаженных или граничных входных данных и подает их системе, отслеживая аварийные завершения (crash), зависания, утечки памяти, необработанные исключения и нарушения инвариантов.

Виды фаззинга:
- **Random/mutation-based fuzzing** — берется валидный входной пример и случайным образом мутируется (изменяются байты, добавляются/удаляются символы);
- **Generation-based fuzzing** — данные генерируются с нуля на основе знания формата/грамматики протокола или файла;
- **Coverage-guided fuzzing** (например, AFL, libFuzzer) — фаззер отслеживает покрытие кода и приоритизирует мутации, которые открывают новые пути выполнения — наиболее эффективный современный подход.

Фаззинг обычно применяется на низких уровнях (unit/API/protocol), чаще инженерами по автоматизации, безопасности или разработчиками, а не мануальным QA через UI, хотя понимание техники важно для полноты картины тест-дизайна. Он особенно эффективен для:
- парсеров файлов и форматов (JSON, XML, изображения, видео);
- сетевых протоколов и API;
- полей ввода, где важна защита от инъекций и переполнения буфера;
- поиска уязвимостей безопасности (security testing).

Ограничения фаззинга: он находит крэши/исключения, но обычно не проверяет бизнес-корректность результата (то есть не скажет, "правильный" ли ответ с точки зрения бизнес-логики) — для этого поверх фаззинга добавляют oracle-функции (проверки инвариантов).

## Пример / когда применяется

Пример: тестируется REST API endpoint, принимающий JSON с полем `"amount": number`. Фаззер автоматически подает миллионы вариантов запроса:

```json
{"amount": -99999999999999}
{"amount": "1' OR '1'='1"}
{"amount": null}
{"amount": 1e400}
{"amount": "😀😀😀😀😀😀😀😀😀😀"}
{"amount": {"nested": "object"}}
```

Цель — увидеть, не упадет ли сервис с 500-й ошибкой, не произойдет ли утечка памяти, не откроется ли SQL-инъекция или уязвимость десериализации.

Применяется на этапах тестирования безопасности и надежности (security/robustness testing), особенно для внешних API, парсеров загружаемых файлов, форм с полями, принимающими произвольный текст.

## На что смотрит интервьюер

- Понимание, что фаззинг — это про поиск крэшей и уязвимостей за счет большого объема случайных/невалидных данных, а не просто "ввод рандомных значений вручную".
- Знание базовых видов фаззинга (mutation-based, generation-based, coverage-guided) хотя бы на уровне терминов.
- Понимание границ применимости — фаззинг находит технические сбои, но не заменяет проверку бизнес-логики.
- Красный флаг — если кандидат путает фаззинг с обычным негативным тестированием (подстановка пары некорректных значений вручную) — фаззинг подразумевает автоматизацию и большой объем данных.
