# Что такое **исследовательское тестирование (Exploratory Testing)**?

**Коротко:** Exploratory Testing — подход, при котором проектирование тестов, их выполнение и обучение системе происходят одновременно, без заранее написанных подробных тест-кейсов, на основе опыта и интуиции тестировщика.

## Развёрнутый ответ

Исследовательское тестирование (Exploratory Testing) — техника, основанная на опыте (experience-based), при которой тестировщик одновременно изучает приложение, проектирует тесты и выполняет их, используя полученные результаты для корректировки следующих шагов. Формула, часто используемая для описания подхода: "simultaneous learning, test design, and test execution" (одновременное обучение, проектирование и выполнение тестов).

Ключевые характеристики:
- отсутствие детальных заранее написанных тест-кейсов — вместо них используют тестовые чартеры (test charters) с указанием цели, области и ограничений по времени (time-box);
- тестировщик активно принимает решения "на лету" — что тестировать дальше, основываясь на увиденном;
- сильно зависит от квалификации, опыта и креативности тестировщика;
- часто применяется в формате сессионного тестирования (Session-Based Test Management, SBTM) — тестирование ведется сессиями фиксированной длины (например, 60-90 минут) с фиксацией результатов в отчете сессии (session report/session sheet).

Exploratory Testing не заменяет формализованный тест-дизайн полностью, а дополняет его:
- эффективен для поиска неочевидных, "нестандартных" дефектов, которые не покрыты формальными тест-кейсами;
- полезен при недостатке документации или сжатых сроках;
- часто используется для регресс-тестирования новых фич "по-быстрому" или для оценки общего качества сборки (smoke/sanity в исследовательском формате).

Важно отличать exploratory testing от ad-hoc testing (бессистемного тестирования "как получится") — исследовательское тестирование, в отличие от чистого ad-hoc, структурировано через чартеры, тайм-боксинг и обязательную фиксацию результатов.

## Пример / когда применяется

Чартер (test charter) для сессии исследовательского тестирования:

```text
Цель: исследовать обработку файлов при загрузке аватара пользователя
Область: страница профиля, форма загрузки изображения
Ресурсы: тестовые изображения разных форматов и размеров, поврежденные файлы
Время: 60 минут
Риски: некорректная обработка больших/поврежденных файлов, XSS через имя файла
```

Во время сессии тестировщик пробует: загрузить файл нестандартного формата, файл с очень длинным именем, поврежденный файл, файл огромного размера, отменить загрузку на середине — и фиксирует найденные дефекты и заметки в отчете сессии.

Применяется на этапе приемки новой фичи, при тестировании нестабильных/часто меняющихся частей продукта, при ограниченном времени на тестирование, для дополнения формального тест-дизайна.

## На что смотрит интервьюер

- Понимание, что exploratory testing — не оправдание отсутствия документации, а осознанная техника со своей методологией (чартеры, тайм-боксинг, session-based test management).
- Умение отличить exploratory testing от бессистемного ad-hoc тестирования.
- Знание, что результаты исследовательских сессий должны фиксироваться (иначе теряется воспроизводимость и метрика покрытия).
- Красный флаг — если кандидат говорит "просто кликаю по приложению без плана" как единственное определение — это ad-hoc, а не exploratory testing в полном смысле.
