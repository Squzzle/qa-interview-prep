# Для чего нужен стенд со **стабами (Stab)**?

**Коротко:** стенд со стабами используется для тестирования системы в изоляции от реальных внешних зависимостей — стаб имитирует поведение внешней системы (сервиса, API, базы данных), возвращая заранее заданные ответы, что позволяет тестировать интеграционные сценарии, недоступные или нестабильные в реальной среде, без риска для реальных систем.

## Развёрнутый ответ

Стаб (Stub, в некоторых компаниях используется термин "стаб"/"Stab" как локальный сленг) — это упрощенная имитация реального компонента системы (внешнего сервиса, API, базы данных, аппаратного устройства), которая возвращает заранее определенные, фиксированные ответы на запросы, не выполняя реальной бизнес-логики.

Стенд со стабами создается для замены реальных внешних зависимостей их имитациями в следующих случаях:

- **Внешняя система недоступна для тестирования** — например, партнерский API платежной системы не предоставляет тестовый контур, или доступ к нему ограничен (лимиты запросов, договорные ограничения).
- **Внешняя система нестабильна** — если реальная система часто падает или имеет непредсказуемые задержки, тестирование через неё делает тесты недетерминированными (flaky).
- **Необходимо протестировать редкие/аварийные сценарии** — например, ответ внешней системы с ошибкой 500, таймаут, некорректный формат данных — такие сценарии сложно или невозможно воспроизвести на реальной системе намеренно, а стаб можно настроить вернуть любой нужный ответ.
- **Изоляция тестируемого компонента** — при интеграционном или компонентном тестировании важно проверить именно логику тестируемой системы, а не поведение внешней зависимости; стаб убирает "шум" и делает тест сфокусированным и воспроизводимым.
- **Ускорение и удешевление тестирования** — реальные вызовы к внешним системам могут быть медленными или платными (например, SMS-шлюз, платежный процессинг с реальными комиссиями); стаб исключает эти издержки.
- **Разработка и тестирование до готовности зависимости** — если внешний сервис еще не разработан или не интегрирован, команда может договориться о контракте API заранее и работать со стабом, который эмулирует этот контракт, не дожидаясь готовности реальной системы.

Важно отличать стаб от мока (mock): стаб просто возвращает заданные данные без проверки того, как и сколько раз он был вызван, тогда как мок дополнительно проверяет корректность вызовов (какие параметры были переданы, сколько раз был вызван метод) и используется в первую очередь в модульном тестировании для верификации взаимодействий.

## Пример / когда применяется

Пример: сервис "Оформление кредита" интегрируется с внешним бюро кредитных историй (БКИ). Реальное БКИ недоступно для нагрузочного тестирования и не позволяет намеренно возвращать сценарий "клиент имеет плохую кредитную историю" для тестов.

На стенде со стабом настраивается имитация БКИ:

```text
Запрос: POST /credit-history/check {"client_id": "12345"}
Стаб-ответ: {"status": "BAD_HISTORY", "score": 320}
```

Это позволяет тестировщику проверить, что система "Оформление кредита" корректно обрабатывает случай отказа, не дожидаясь реального клиента с плохой кредитной историей и не завися от доступности реального БКИ. Аналогично можно настроить стаб для имитации таймаута внешней системы и проверить, что приложение корректно обрабатывает эту ситуацию (например, показывает пользователю понятное сообщение об ошибке вместо зависания).

Стенды со стабами широко применяются в микросервисной архитектуре, где сервис зависит от десятков других сервисов, и поднять реальное окружение всех зависимостей для одного теста нецелесообразно.

## На что смотрит интервьюер

- Понимает ли кандидат разницу между стабом и моком, а также когда использовать каждый из них.
- Может ли объяснить, почему стенд со стабами не заменяет полностью интеграционное тестирование на реальном стенде (стаб не гарантирует, что реальная система ведет себя так же).
- Красный флаг: кандидат считает, что тестирование на стабах полностью эквивалентно тестированию с реальной интеграцией — важно, чтобы кандидат понимал ограничения такого подхода и необходимость дополнительной проверки на реальной интеграции (например, на стенде ИФТ).
- Follow-up вопросы: "Какие риски несет тестирование только на стабах?", "Кто обычно поддерживает актуальность стабов при изменении контракта реального API?", "Приведите пример инструмента для создания стабов (например, WireMock)."
