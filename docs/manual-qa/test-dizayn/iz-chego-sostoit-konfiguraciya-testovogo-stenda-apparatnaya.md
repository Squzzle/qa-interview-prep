# Из чего состоит **конфигурация тестового стенда** (аппаратная/программная часть)?

**Коротко:** конфигурация тестового стенда включает аппаратную часть (серверы, сеть, устройства, объем ресурсов CPU/RAM/диск) и программную часть (операционная система, версии ПО, база данных, middleware, конфигурационные файлы, тестовые данные, права доступа) — вся совокупность, которая должна быть документирована и воспроизводима для повторяемости результатов тестирования.

## Развёрнутый ответ

Конфигурация тестового стенда — это полное описание всех компонентов среды, в которой выполняется тестирование, необходимое для того, чтобы результаты тестирования были воспроизводимыми, а сама среда — управляемой и контролируемой. Обычно конфигурацию делят на два больших блока.

**Аппаратная часть (Hardware configuration):**
- Серверы (физические или виртуальные машины, контейнеры) — их количество, характеристики (CPU, оперативная память, объем и тип диска — SSD/HDD).
- Сетевая инфраструктура — топология сети, пропускная способность, настройки firewall, VPN, балансировщики нагрузки.
- Специализированные устройства — для мобильного тестирования это реальные устройства или эмуляторы/симуляторы разных моделей и версий ОС; для IoT-проектов — конкретные аппаратные модули, сенсоры, платы.
- Резервирование и отказоустойчивость стенда — наличие кластеров, реплик, механизмов failover (актуально для стендов, приближенных к продакшену).

**Программная часть (Software configuration):**
- Операционная система и её версия на серверах и клиентских машинах.
- Версии используемого программного обеспечения: версия тестируемого приложения (билд), версии зависимостей, библиотек, фреймворков.
- Сервер базы данных, её версия, схема, объем и состав тестовых данных.
- Middleware и вспомогательные сервисы — очереди сообщений (Kafka, RabbitMQ), кэш (Redis), веб-серверы (Nginx, Apache), контейнеризация (Docker, Kubernetes) и их версии.
- Конфигурационные файлы приложения — настройки окружения (environment variables), фичефлаги, тайминги, лимиты, интеграционные endpoint-ы (в том числе URL смежных систем или стабов).
- Учетные записи и права доступа — тестовые пользователи с разными ролями и уровнями доступа.
- Инструменты тестирования и мониторинга — установленные на стенде агенты логирования, системы сбора метрик, инструменты автоматизации.
- Сертификаты, ключи шифрования, настройки безопасности (особенно критично для финтех- и медицинских проектов).

Полная и актуальная документация конфигурации стенда важна по нескольким причинам: она позволяет воспроизвести баг в других окружениях, обеспечивает сопоставимость результатов тестирования между разными циклами тестирования, ускоряет расследование причин, когда баг воспроизводится на одном стенде и не воспроизводится на другом (часто причина кроется именно в различии конфигураций).

## Пример / когда применяется

Пример карточки конфигурации тестового стенда для веб-приложения:

| Параметр | Значение |
|---|---|
| Аппаратная часть | 2 виртуальные машины: 4 vCPU, 16 GB RAM, 100 GB SSD каждая |
| Операционная система | Ubuntu 22.04 LTS |
| Версия приложения | build 2.14.3 (commit abc123) |
| База данных | PostgreSQL 15, объем тестовых данных ~50 GB (обезличенная копия продакшена) |
| Middleware | Nginx 1.24, Redis 7, RabbitMQ 3.12 |
| Контейнеризация | Docker 24, Kubernetes 1.28 (namespace test-env) |
| Интеграции | Стаб платежного шлюза (WireMock), реальный тестовый контур СМС-провайдера |
| Тестовые учетные записи | admin_test, user_test (роль "клиент"), user_manager_test (роль "менеджер") |

Такая конфигурация фиксируется в документе (например, в Confluence или в файле `stand-config.yaml` в репозитории инфраструктуры) и используется при постановке бага — тестировщик указывает версию билда и конфигурацию стенда, чтобы разработчик мог однозначно воспроизвести проблему.

## На что смотрит интервьюер

- Может ли кандидат разделить аппаратную и программную часть конфигурации и назвать конкретные компоненты каждой.
- Понимает ли кандидат важность документирования конфигурации для воспроизводимости багов.
- Красный флаг: кандидат ограничивается общими словами вроде "сервер и база данных" без деталей (версии, объем ресурсов, тестовые данные, права доступа).
- Follow-up вопросы: "Что вы будете делать, если баг воспроизводится на вашем стенде, но не воспроизводится у разработчика?", "Как вы поддерживаете актуальность тестовых данных на стенде?", "Как версионируется конфигурация стенда (Infrastructure as Code)?"
