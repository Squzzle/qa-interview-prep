# Какие техники относят к тестированию **"серого ящика"**?

**Коротко:** тестирование серого ящика сочетает элементы black-box (тестирование по требованиям, без глубокого знания кода) и white-box (частичное знание внутреннего устройства — архитектуры, БД, API, логов) — используются техники анализа логов, тестирование API, проверка данных в БД, анализ путей выполнения на уровне интеграций.

## Развёрнутый ответ

Тестирование серого ящика (Gray Box Testing) занимает промежуточное положение между тестированием черного ящика (тестировщик не знает внутреннего устройства системы) и тестированием белого ящика (полное знание кода и структуры). Тестировщик обладает частичными знаниями о внутренней реализации — знает архитектуру, схему базы данных, взаимодействие компонентов, API-контракты, — но не тестирует непосредственно исходный код через unit-тесты.

К техникам и практикам, которые относят к тестированию серого ящика, обычно причисляют:

- **Тестирование на основе матрицы (Matrix Testing)** — анализ бизнес- и технических переменных системы для определения покрытия.
- **Регрессионное тестирование** с пониманием того, какие модули были изменены (по данным из системы контроля версий или архитектурной документации).
- **Тестирование состояний и переходов (State Transition Testing)** с опорой на знание внутренней модели состояний системы.
- **Тестирование через API** — вызовы методов API напрямую, минуя UI, с проверкой корректности ответов, кодов состояния, структуры данных.
- **Анализ и проверка данных в базе данных** — сверка того, что показывает интерфейс, с реальным состоянием записей в БД (data integrity testing).
- **Анализ логов приложения и сервера** — поиск ошибок, исключений, аномального поведения, которые не видны через UI.
- **Тестирование паттернов проектирования и интеграций** — проверка взаимодействия между модулями/микросервисами с частичным знанием их контрактов.
- **Orthogonal Array Testing (OAT)** — иногда также относят сюда, так как требует понимания внутренних параметров системы.

Ключевая идея — тестировщик использует внутренние знания системы (структуру данных, архитектуру, точки интеграции) для более точного проектирования тест-кейсов, чем это было бы возможно при чистом black-box подходе, но не пишет и не анализирует построчно исходный код, как это делают при white-box тестировании (покрытие кода, юнит-тесты).

## Пример / когда применяется

Пример: тестировщик проверяет функциональность "перевод денег между счетами" в банковском приложении. Он не читает исходный код сервиса перевода, но знает, что операция записывается в таблицу `transactions`, а баланс обновляется в таблице `accounts`. После выполнения перевода через UI тестировщик:

1. Проверяет корректность отображения в интерфейсе (black-box подход).
2. Делает запрос к базе данных, чтобы убедиться, что записи созданы верно:

```sql
SELECT id, from_account, to_account, amount, status
FROM transactions
WHERE from_account = 12345
ORDER BY created_at DESC
LIMIT 1;
```

3. Проверяет логи сервиса на наличие ошибок или предупреждений.
4. При необходимости вызывает внутренний API напрямую (например, `POST /api/v1/transfer`) для проверки граничных случаев без прохождения через UI.

Такой подход применяется в интеграционном тестировании, тестировании безопасности (проверка SQL-инъекций с пониманием структуры запросов), тестировании веб-приложений с известной архитектурой (клиент-сервер, REST API).

## На что смотрит интервьюер

- Понимает ли кандидат разницу между black-box, white-box и gray-box тестированием, а не путает ли их.
- Приводит ли конкретные примеры техник (работа с БД, API, логами), а не общие фразы.
- Красный флаг: кандидат утверждает, что gray-box — это "немного white-box, немного black-box" без конкретики, какие именно активности выполняются.
