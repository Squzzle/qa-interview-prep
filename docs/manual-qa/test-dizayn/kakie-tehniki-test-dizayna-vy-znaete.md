# Какие **техники тест-дизайна** вы знаете?

**Коротко:** техники тест-дизайна делятся на три группы — черноящичные (based on специфике поведения), белоящичные (based on структуре кода) и экспертные/основанные на опыте.

## Развёрнутый ответ

По классификации ISTQB техники тест-дизайна делятся на три категории.

**1. Black-box (специфицированные, based on поведении):**
- эквивалентное разделение (Equivalence Partitioning);
- анализ граничных значений (Boundary Value Analysis);
- таблицы принятия решений (Decision Tables);
- тестирование переходов состояний (State Transition Testing);
- попарное тестирование (Pairwise Testing) и комбинаторные техники (Combinatorial Testing, Orthogonal Arrays);
- тестирование на основе use case (Use Case Testing);
- классификационное дерево (Classification Tree Method).

**2. White-box (структурные, based on коде):**
- покрытие операторов (statement coverage);
- покрытие решений/ветвей (decision/branch coverage);
- покрытие условий (condition coverage);
- покрытие путей (path coverage).

**3. Experience-based (основанные на опыте):**
- исследовательское тестирование (Exploratory Testing);
- тестирование на основе догадки об ошибках (Error Guessing);
- чек-листы (Checklist-based testing).

Отдельно стоит упомянуть смежные подходы, которые часто относят к тест-дизайну на практике: фаззинг (Fuzz Testing), риск-ориентированное тестирование (Risk-Based Testing), сценарное тестирование (Scenario Testing).

Выбор техники зависит от типа системы, критичности функциональности, доступности спецификаций и уровня тестирования (модульное, интеграционное, системное).

## Пример / когда применяется

| Техника | Когда применять |
|---|---|
| Эквивалентное разделение | Большой диапазон входных данных, нужно сократить количество тестов |
| Граничные значения | Числовые диапазоны, лимиты, ошибки off-by-one |
| Таблицы решений | Сложная бизнес-логика с множеством условий и комбинаций |
| Переходы состояний | Системы с явными состояниями (заказ, документ, сессия) |
| Попарное тестирование | Много независимых параметров конфигурации |
| Исследовательское тестирование | Мало документации, сжатые сроки, поиск неочевидных багов |

## На что смотрит интервьюер

- Способность структурировать ответ по группам (black-box / white-box / experience-based), а не просто перечислить названия вперемешку.
- Понимание, что White-box техники обычно применяет разработчик или тестировщик, пишущий unit-тесты, а не мануальный QA на уровне UI.
- Может ли кандидат для каждой техники привести пример ситуации применения.
- Красный флаг — если кандидат называет только 1-2 техники или путает тест-дизайн с уровнями тестирования (unit, integration, system).
