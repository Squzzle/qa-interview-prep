# Опишите технику **тестирования переходов состояний (State Transition Testing)**.

**Коротко:** State Transition Testing проверяет, что система корректно переходит из одного состояния в другое под действием событий, и не допускает недопустимых переходов — техника особенно полезна для систем с явным жизненным циклом объекта.

## Развёрнутый ответ

State Transition Testing (тестирование переходов состояний) — черноящичная техника, применяемая к системам, поведение которых зависит не только от текущего входа, но и от истории — то есть от текущего состояния объекта. Основа техники — модель конечного автомата (Finite State Machine), которая описывает:

- **состояния (states)** — все возможные состояния объекта/системы;
- **события/переходы (events/transitions)** — что вызывает переход из одного состояния в другое;
- **действия (actions)** — что происходит при переходе;
- **недопустимые переходы** — какие переходы невозможны из данного состояния.

Модель обычно представляют в виде диаграммы состояний (state diagram) или таблицы переходов (state table), где строки — текущие состояния, столбцы — события, а ячейки — результирующее состояние (или отметка о недопустимости перехода).

На основе модели строят тесты, покрывающие:
1. **0-switch coverage** — все состояния хотя бы раз посещены;
2. **1-switch coverage** (более строгий) — все допустимые переходы (пары состояние→состояние) протестированы хотя бы раз;
3. **N-switch coverage** — цепочки из N последовательных переходов;
4. тесты на недопустимые/невозможные переходы — проверка, что система корректно отклоняет попытку перехода, которого не должно быть.

Технику применяют для систем с жизненным циклом: заказы, документы, банковские счета, сессии пользователя, workflow согласования, состояние заявки в тикет-системе и т.п.

## Пример / когда применяется

Жизненный цикл заказа в интернет-магазине: `Новый → Оплачен → Собран → Отправлен → Доставлен`, а также `Новый/Оплачен → Отменен`.

Таблица переходов:

| Текущее состояние | Событие | Новое состояние |
|---|---|---|
| Новый | Оплата прошла | Оплачен |
| Новый | Отмена | Отменен |
| Оплачен | Заказ собран | Собран |
| Оплачен | Отмена | Отменен |
| Собран | Отгружен | Отправлен |
| Отправлен | Получен клиентом | Доставлен |
| Доставлен | Отмена | **недопустимо** (ошибка/блокировка) |

Тест-кейсы: проверка каждого валидного перехода (например, "Оплачен → Собран") и хотя бы одного недопустимого перехода (например, попытка отменить уже доставленный заказ должна быть заблокирована системой с понятным сообщением об ошибке).

## На что смотрит интервьюер

- Понимание разницы между состоянием и просто значением поля — состояние подразумевает, что поведение системы зависит от истории/текущей фазы жизненного цикла.
- Знание уровней покрытия (0-switch, 1-switch, N-switch) хотя бы на базовом уровне.
- Обязательно должен упомянуть тестирование недопустимых переходов, а не только "happy path" переходов — это частый пробел у слабых кандидатов.
- Follow-up: "Как визуализировать модель?" — диаграмма состояний или таблица переходов.
- Красный флаг — если кандидат путает эту технику с таблицами принятия решений (там про комбинации условий, здесь — про последовательность состояний во времени).
