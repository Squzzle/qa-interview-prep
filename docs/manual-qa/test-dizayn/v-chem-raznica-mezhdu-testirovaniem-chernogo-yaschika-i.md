# В чем разница между тестированием **"черного ящика"** и **"белого ящика"**?

**Коротко:** тестирование черного ящика проверяет систему по её внешнему поведению без знания внутренней реализации, а тестирование белого ящика основано на знании и анализе внутренней структуры кода.

## Развёрнутый ответ

**Black-box testing (тестирование "черного ящика")**:
- тестировщик рассматривает систему как "черный ящик" — известны только входы и ожидаемые выходы, внутренняя реализация не имеет значения и обычно неизвестна;
- тесты строятся на основе требований, спецификаций, пользовательских сценариев;
- типичные техники: эквивалентное разделение, анализ граничных значений, таблицы принятия решений, тестирование переходов состояний, попарное тестирование;
- обычно выполняется мануальными QA-инженерами и тестировщиками автоматизации на уровне UI/API, не требует знания языка программирования, на котором написана система.

**White-box testing (тестирование "белого ящика")**:
- тестировщик имеет доступ к исходному коду и строит тесты на основе его внутренней структуры (ветвлений, циклов, путей выполнения);
- цель — покрыть определенные структурные элементы кода: операторы (statement coverage), ветви/решения (branch/decision coverage), условия (condition coverage), пути (path coverage);
- типичные техники: покрытие операторов, покрытие решений, покрытие условий, покрытие путей, юнит-тестирование, мутационное тестирование (mutation testing);
- выполняется обычно разработчиками или SDET (Software Development Engineer in Test), требует навыков программирования.

Есть также промежуточный подход — **Grey-box testing** (тестирование "серого ящика"): тестировщик частично знаком с внутренним устройством системы (например, знает структуру базы данных или архитектуру API), но тестирует преимущественно через внешний интерфейс — распространено в тестировании API и интеграционном тестировании.

Важно: black-box и white-box — это классификация по уровню знания о внутренней структуре системы, а не по уровню тестирования (unit/integration/system) — оба подхода могут применяться на разных уровнях, хотя white-box чаще ассоциируется с unit-тестами, а black-box — с системным и приемочным тестированием.

## Пример / когда применяется

| Критерий | Black-box | White-box |
|---|---|---|
| Знание кода | Не требуется | Требуется |
| Кто выполняет | QA-инженер, тестировщик автоматизации | Разработчик, SDET |
| Основа тестов | Требования, спецификации | Исходный код |
| Метрика покрытия | Покрытие требований/сценариев | Покрытие кода (statement/branch/path) |
| Пример техники | Equivalence Partitioning, BVA | Branch coverage, mutation testing |
| Уровень | Чаще system/acceptance | Чаще unit/component |

Пример black-box: тестировщик проверяет форму логина, вводя валидный и невалидный пароль, не заглядывая в код аутентификации.

Пример white-box: разработчик пишет unit-тест, который явно покрывает обе ветки условия `if (password.length < 8) throw Error` — и valid, и invalid ветку, глядя непосредственно в код функции.

## На что смотрит интервьюер

- Кандидат должен четко разделить критерий классификации — доступ к внутренней структуре системы, а не "кто именно" тестирует или "на каком уровне".
- Знание про промежуточный grey-box подход — плюс к ответу.
- Частая ошибка — путать black-box/white-box с уровнями тестирования (unit/integration/system) — это независимые классификации.
- Follow-up: "Какие метрики покрытия используются в white-box тестировании?" — statement, branch/decision, condition, path coverage.
