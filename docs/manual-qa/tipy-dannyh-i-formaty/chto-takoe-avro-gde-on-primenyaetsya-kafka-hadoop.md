# Что такое **Avro**? Где он применяется (Kafka, Hadoop)?

**Коротко:** Apache Avro — бинарный формат сериализации данных со схемой в формате JSON, оптимизированный для потоковой обработки больших данных, широко применяется в экосистемах Kafka и Hadoop.

## Развёрнутый ответ

Avro — проект Apache, предоставляющий компактную и быструю бинарную сериализацию данных с обязательной схемой. В отличие от Protobuf, где схема компилируется в код заранее, схема Avro сама записывается в формате JSON и может передаваться вместе с данными (или храниться отдельно в реестре схем), что позволяет читать данные динамически без предварительной кодогенерации.

Ключевые особенности:

- **Схема в JSON** — описывает поля, их типы и структуру записи (`record`).
- **Самоописываемость файлов** — файл контейнера Avro (`.avro`) хранит схему внутри себя, что упрощает чтение данных без внешних зависимостей.
- **Компактность** — бинарный формат, так как в потоке данных не хранятся имена полей (только значения в порядке, заданном схемой), что делает его компактнее даже Protobuf в некоторых сценариях.
- **Поддержка эволюции схемы** — Avro имеет чёткие правила совместимости (backward, forward, full compatibility) при изменении схемы со временем.
- **Отсутствие кодогенерации как обязательного шага** — можно работать с данными динамически через `GenericRecord`, хотя кодогенерация тоже поддерживается для типизированного доступа.
- **Splittable-формат** — Avro-файлы можно эффективно разбивать для параллельной обработки в распределённых системах (важно для Hadoop MapReduce/Spark).

**Где применяется:**

- **Apache Kafka** — Avro один из самых распространённых форматов сообщений, особенно вместе со **Schema Registry** (Confluent Schema Registry), который централизованно хранит версии схем и обеспечивает совместимость между продюсерами и консьюмерами.
- **Apache Hadoop** — формат хранения данных в HDFS, поддерживается Hive, Spark, MapReduce как эффективный splittable-формат для батчевой обработки больших данных.
- **Системы обмена данными между сервисами**, где схема данных часто меняется и важна гибкая эволюция контракта.

## Пример / когда применяется

Пример схемы Avro:

```json
{
  "type": "record",
  "name": "User",
  "fields": [
    { "name": "id", "type": "int" },
    { "name": "name", "type": "string" },
    { "name": "isActive", "type": "boolean", "default": true }
  ]
}
```

В Kafka продюсер сериализует сообщение по этой схеме и регистрирует её в Schema Registry, консьюмер получает схему по идентификатору и десериализует сообщение, даже если версия схемы у него отличается (при соблюдении правил совместимости).

## На что смотрит интервьюер

Интервьюер проверяет понимание разницы между Avro и Protobuf (схема в JSON и передаётся с данными или через реестр против компилируемой схемы в отдельном файле), а также практическое знание Schema Registry в контексте Kafka. Часто спрашивают про типы совместимости схем (backward/forward/full) и про то, как тестировать сообщения в Kafka с Avro-сериализацией (например, через Kafka Avro Console Consumer или тестовые контейнеры). Красный флаг — путать Avro с Protobuf или не знать, зачем нужен Schema Registry.
