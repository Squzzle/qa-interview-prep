# Что такое **XML**? Чем он отличается от JSON?

**Коротко:** XML (eXtensible Markup Language) — язык разметки для описания структурированных данных с помощью тегов, более многословный, чем JSON, но поддерживающий схемы, атрибуты, пространства имён и комментарии.

## Развёрнутый ответ

XML — текстовый формат представления данных, где структура задаётся вложенными парными тегами (`<tag>значение</tag>`), которые могут иметь атрибуты. XML был стандартом для веб-сервисов (SOAP) и до сих пор широко используется в enterprise-системах, банковской сфере, конфигурационных файлах (например, Maven `pom.xml`, Android-разметка).

Ключевые особенности XML:

- **Теги** — открывающий и закрывающий, обязательное закрытие (или самозакрывающийся тег `<tag/>`).
- **Атрибуты** — дополнительные данные внутри открывающего тега (`<user id="1">`).
- **Пространства имён (namespaces)** — механизм избежания конфликтов имён тегов.
- **DTD/XSD** — схемы для валидации структуры документа.
- **Поддержка комментариев** — `<!-- комментарий -->`.
- **XPath/XSLT** — языки для навигации и трансформации XML-документов.

**Отличия XML от JSON:**

| Критерий | XML | JSON |
|---|---|---|
| Синтаксис | Теги, атрибуты | Ключ-значение |
| Размер | Более многословный | Компактнее |
| Читаемость | Ниже из-за тегов | Выше |
| Комментарии | Поддерживаются | Не поддерживаются |
| Схема валидации | XSD, DTD | JSON Schema |
| Атрибуты у элементов | Есть | Нет (только вложенные объекты) |
| Типы данных | Всё текст, типизация через схему | string, number, boolean, null, object, array |
| Пространства имён | Есть | Нет |
| Порядок элементов | Значим | Не значим для объектов |
| Использование | SOAP, конфиги enterprise, документы | REST API, конфиги, NoSQL |

## Пример / когда применяется

Один и тот же объект в XML и JSON:

```xml
<user id="1">
  <name>Иван</name>
  <isActive>true</isActive>
</user>
```

```json
{
  "user": {
    "id": 1,
    "name": "Иван",
    "isActive": true
  }
}
```

XML применяется там, где важна строгая валидация по схеме (XSD), поддержка смешанного контента (текст + вложенные теги) и обратная совместимость с legacy SOAP-сервисами. JSON выбирают для современных REST API за компактность и простоту.

## На что смотрит интервьюер

Ожидается чёткое сравнение по нескольким критериям, а не общее "XML сложнее". Хороший ответ включает упоминание атрибутов, пространств имён и того, что XML поддерживает смешанный контент, чего нет в JSON. Могут спросить про SOAP vs REST, про XSD/DTD, про XPath. Красный флаг — незнание, что в XML данные могут храниться и в тегах, и в атрибутах.
