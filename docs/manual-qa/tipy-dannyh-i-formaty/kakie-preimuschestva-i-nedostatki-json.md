# Какие преимущества и недостатки JSON?

**Коротко:** JSON прост, компактен и легко читается человеком и машиной, но не имеет строгой схемы "из коробки", не поддерживает комментарии и типы вроде дат, и менее эффективен по размеру, чем бинарные форматы.

## Развёрнутый ответ

**Преимущества JSON:**

- **Простота и читаемость** — минималистичный синтаксис, легко понимается человеком без специальной подготовки.
- **Компактность** — по сравнению с XML занимает меньше места за счёт отсутствия закрывающих тегов.
- **Широкая поддержка** — практически все языки программирования и фреймворки имеют встроенные или сторонние библиотеки для парсинга JSON.
- **Нативная совместимость с JavaScript** — не требует дополнительного парсинга в браузере (`JSON.parse`/`JSON.stringify`).
- **Хорошо подходит для REST API** — стал де-факто стандартом для веб-сервисов.
- **Поддержка вложенных структур** — объекты и массивы можно вкладывать друг в друга произвольно.

**Недостатки JSON:**

- **Отсутствие комментариев** — нельзя документировать данные прямо внутри файла.
- **Ограниченный набор типов данных** — нет типов "дата", "бинарные данные" (нужно кодировать в строку, например Base64), нет различия int/float.
- **Нет схемы по умолчанию** — для валидации структуры нужен отдельный инструмент (JSON Schema).
- **Меньшая производительность по сравнению с бинарными форматами** — Protobuf или Avro компактнее и быстрее парсятся при больших объёмах данных.
- **Проблемы с большими числами** — числа, превышающие пределы точности `double` (например, 64-битные ID), могут терять точность при парсинге в некоторых языках.
- **Нет поддержки циклических ссылок** — в отличие от некоторых бинарных форматов сериализации.

## Пример / когда применяется

Пример проблемы с большими числами:

```json
{
  "id": 9007199254740993
}
```

При парсинге в JavaScript это число может округлиться, так как превышает `Number.MAX_SAFE_INTEGER` — на практике такие ID часто передают строкой.

JSON выбирают для веб-API и конфигураций, где важна читаемость и простота интеграции; для высоконагруженных систем с миллионами сообщений в секунду (Kafka, gRPC) чаще выбирают Protobuf или Avro.

## На что смотрит интервьюер

Интервьюер ждёт, что кандидат назовёт конкретные недостатки, а не общие фразы. Хорошим сигналом является упоминание проблемы точности чисел, отсутствия комментариев и необходимости JSON Schema для валидации. Красный флаг — если кандидат не может привести ни одного недостатка или путает JSON с XML/YAML по возможностям.
