# Что проверяется на **интеграционном тестировании**?

**Коротко:** проверяется корректность взаимодействия между двумя и более уже протестированными по отдельности модулями/компонентами — передача данных, вызовы интерфейсов, обработка ошибок на стыках.

## Развёрнутый ответ

Интеграционное тестирование фокусируется не на внутренней логике отдельных модулей (это уже проверено на уровне unit-тестов), а на **корректности взаимодействия** между ними. Основные объекты проверки:

- **Передача данных между компонентами** — правильно ли данные сериализуются, передаются и десериализуются между модулями (например, между сервисом заказов и сервисом оплаты).
- **Корректность вызовов API/интерфейсов** — соответствуют ли форматы запросов и ответов согласованному контракту (API-контракты, форматы JSON/XML, коды ответов).
- **Обработка ошибок на границах модулей** — что происходит, если один компонент недоступен, вернул ошибку или таймаут.
- **Совместная работа с базой данных, файловой системой, внешними сервисами** — корректность транзакций, блокировок, синхронизации.
- **Соответствие требованиям к производительности взаимодействия** — задержки при обмене данными между компонентами.

Интеграционное тестирование может проводиться на разных уровнях: между модулями внутри одного приложения, между приложением и базой данных, между микросервисами, между системой и внешними API (платёжные шлюзы, почтовые сервисы и т.д.).

## Пример / когда применяется

Пример: есть сервис "Корзина" и сервис "Скидки". Интеграционный тест проверяет:

```text
Сценарий: пользователь добавил товар с промокодом в корзину
1. Сервис "Корзина" отправляет запрос в сервис "Скидки" с ID товара и промокодом
2. Сервис "Скидки" возвращает корректную сумму скидки
3. Сервис "Корзина" правильно применяет скидку к итоговой сумме
4. При недоступности сервиса "Скидки" корзина показывает корректное сообщение об ошибке, а не падает
```

## На что смотрит интервьюер

- Понимание границы между unit- и интеграционным тестированием: интеграционные тесты не должны дублировать проверку внутренней логики модулей.
- Знание типичных проблем интеграции: несовпадение форматов данных, версии API, тайминги, обработка отказов.
- Умение привести конкретный пример стыка двух реальных компонентов.
- Follow-up: "Какие методы/стратегии интеграции вы знаете?" (Big Bang, Top-Down, Bottom-Up) — часто задаётся сразу после этого вопроса.
