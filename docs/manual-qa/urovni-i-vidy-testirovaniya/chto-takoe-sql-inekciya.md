# Что такое **SQL-инъекция**?

**Коротко:** SQL-инъекция — это уязвимость, при которой злоумышленник внедряет произвольный SQL-код через входные данные приложения (поля форм, параметры URL и т.д.), заставляя базу данных выполнить непредусмотренные разработчиком команды.

## Развёрнутый ответ

SQL-инъекция возникает, когда пользовательский ввод конкатенируется напрямую в текст SQL-запроса без экранирования и без использования параметризованных запросов. В результате данные, введённые пользователем, интерпретируются сервером базы данных как часть кода запроса, а не как обычное значение.

Типичные последствия успешной SQL-инъекции:

- обход аутентификации (вход в систему без знания пароля);
- чтение конфиденциальных данных из базы (номера карт, пароли, персональные данные);
- изменение или удаление данных;
- в некоторых случаях — выполнение команд на уровне операционной системы или получение полного контроля над сервером БД.

Основные виды SQL-инъекций:

- **Классическая (in-band)** — результат виден сразу в ответе приложения (например, через ошибку или изменение вывода).
- **Слепая (blind SQL injection)** — прямого вывода данных нет, но по поведению приложения (true/false, задержка ответа) можно восстановить данные по битам.
- **Time-based blind** — используется задержка выполнения (`SLEEP()`, `WAITFOR DELAY`) для подтверждения условия.
- **Second-order** — вредоносные данные сохраняются в базе и выполняются позже, при другом запросе.

Методы защиты:

- параметризованные запросы (prepared statements) и ORM;
- валидация и экранирование пользовательского ввода;
- принцип наименьших привилегий для учётной записи приложения в БД;
- использование WAF (Web Application Firewall);
- регулярные security-сканирования и код-ревью.

## Пример / когда применяется

Классический уязвимый код на стороне сервера может формировать запрос так:

```sql
SELECT * FROM users WHERE login = 'admin' AND password = '' OR '1'='1'
```

Если поле пароля не экранируется, а пользователь вводит значение `' OR '1'='1`, итоговый запрос всегда возвращает true и позволяет войти без знания пароля.

Безопасный вариант — параметризованный запрос (псевдокод):

```sql
SELECT * FROM users WHERE login = ? AND password = ?
```

Значения передаются отдельно от текста запроса, поэтому ввод не может изменить структуру SQL-выражения.

QA-инженер проверяет SQL-инъекции вручную (вводя спецсимволы `'`, `--`, `;`, `OR 1=1` в поля форм) или с помощью автоматизированных сканеров, например SQLMap, Burp Suite.

## На что смотрит интервьюер

- Понимает ли кандидат разницу между тестированием функциональности и security-тестированием.
- Может ли привести конкретный пример уязвимого запроса и объяснить, почему он опасен.
- Знает ли способы защиты (параметризованные запросы, валидация ввода).
- Частая ошибка — путать SQL-инъекцию с XSS или считать, что достаточно "просто фильтровать кавычки".
- Follow-up вопросы: чем отличается блайнд-инъекция от обычной, как найти SQL-инъекцию вручную, какие инструменты используются для автоматизации.
