# Что такое **структурное тестирование (white-box)**?

**Коротко:** структурное (белый ящик) тестирование строится на основе анализа внутренней структуры кода — потока управления и данных — и оценивается через метрики покрытия кода, в отличие от функционального (чёрный ящик) тестирования, которое строится на требованиях.

## Развёрнутый ответ

Структурное тестирование (Structure-based / White-box testing) — подход, при котором тестовые сценарии проектируются на основе анализа внутренней структуры и реализации программы: исходного кода, архитектуры, потока управления (control flow) и потока данных (data flow). В отличие от black-box тестирования, тестировщик (обычно разработчик) видит и понимает код тестируемого компонента.

Основная цель — обеспечить достаточное покрытие кода тестами, чтобы минимизировать риск необнаруженных дефектов в непротестированных участках. Основные метрики/техники покрытия:

- **Purpose коэффициента (Statement coverage)** — доля строк/операторов кода, выполненных хотя бы один раз в ходе тестов.
- **Branch/Decision coverage (покрытие ветвлений/решений)** — доля выполненных истинных и ложных исходов каждого условного оператора (if/else, switch).
- **Condition coverage (покрытие условий)** — проверка каждого отдельного логического условия внутри сложного выражения на true и false.
- **Path coverage (покрытие путей)** — проверка всех возможных путей выполнения через код (самый полный, но и самый затратный вид покрытия).

Структурное тестирование чаще всего применяется на уровне модульного (unit) тестирования, но принципы white-box могут использоваться и при интеграционном тестировании (например, анализ потоков данных между модулями).

Важный нюанс: 100% покрытие кода не гарантирует отсутствие дефектов — оно лишь говорит о том, что код был выполнен, но не проверяет корректность бизнес-логики или соответствие требованиям. Поэтому white-box тестирование обычно дополняется black-box (функциональным) тестированием.

## Пример / когда применяется

Пример функции:

```javascript
function getDiscount(price, isVip) {
  if (isVip) {
    return price * 0.8;
  } else {
    return price;
  }
}
```

Для 100% branch coverage нужно как минимум два тест-кейса:

```text
Тест 1: getDiscount(100, true) -> ожидается 80 (покрыта ветка true)
Тест 2: getDiscount(100, false) -> ожидается 100 (покрыта ветка false)
```

Если протестировать только `getDiscount(100, true)`, statement coverage может быть высоким, но branch coverage — только 50%, так как ветка `else` не выполнялась.

## На что смотрит интервьюер

- Чёткое понимание разницы между white-box (структурное) и black-box (функциональное) тестированием.
- Знание конкретных метрик покрытия (statement, branch, condition, path) и умение объяснить разницу между ними на примере.
- Понимание ограничения: высокое покрытие кода не означает отсутствие багов.
- Follow-up: "Какое покрытие кода достаточно?" или "Чем branch coverage отличается от condition coverage?" — часто задаётся как проверочный вопрос на глубину знаний.
