# Для чего нужно **тестирование стабильности (Stability/Soak testing)**?

**Коротко:** Soak testing проверяет поведение системы при работе под средней (обычной) нагрузкой в течение длительного времени, чтобы выявить проблемы, которые проявляются только со временем: утечки памяти, деградацию производительности, накопление ошибок.

## Развёрнутый ответ

Тестирование стабильности (Stability testing, чаще называют Soak testing или endurance testing) — это разновидность нефункционального тестирования, при котором система подвергается устойчивой, обычно средней или пиковой, нагрузке на протяжении длительного периода — часы, дни, иногда недели. В отличие от load/stress testing, где важна величина нагрузки, в soak testing ключевой фактор — время.

Зачем это нужно:

- Обнаружить утечки памяти (memory leaks), которые незаметны при коротком тесте, но приводят к падению приложения через сутки работы.
- Выявить деградацию производительности со временем: замедление ответов, рост времени сборки мусора (GC), фрагментацию памяти или диска.
- Проверить корректность работы с ресурсами: закрываются ли соединения с БД, файловые дескрипторы, сетевые сокеты, не накапливаются ли "зависшие" процессы.
- Проверить стабильность работы фоновых задач, планировщиков, очередей сообщений при длительной непрерывной работе.
- Оценить надёжность системы в реальных условиях эксплуатации (24/7 сервисы, IoT-устройства, embedded-системы).

Типичные проблемы, которые находит soak testing: постепенный рост потребления RAM, замедление ответа API после N часов работы, переполнение логов и дискового пространства, накопление "мёртвых" сессий.

## Пример / когда применяется

Пример: веб-сервис запускают под постоянной средней нагрузкой (например, 500 запросов в минуту) на 72 часа. Мониторят потребление памяти каждые 15 минут. Если график памяти монотонно растёт без выхода на плато — это признак утечки памяти, которую нужно исправить до релиза.

Применяется перед выпуском систем, которые должны работать непрерывно длительное время: серверные приложения, микросервисы с высокой доступностью, встраиваемое ПО, биллинговые системы.

## На что смотрит интервьюер

- Понимает ли кандидат, что ключевой фактор soak testing — длительность, а не величина нагрузки.
- Может ли назвать конкретные проблемы, которые выявляет этот вид тестирования (memory leak, деградация производительности).
- Знает ли инструменты мониторинга (профилировщики памяти, APM-системы) для отслеживания метрик в ходе теста.
- Красный флаг: путает soak testing со stress testing, фокусируясь только на нагрузке, а не на времени.
