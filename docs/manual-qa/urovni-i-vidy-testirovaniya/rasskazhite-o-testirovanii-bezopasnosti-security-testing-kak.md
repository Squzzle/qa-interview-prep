# Расскажите о **тестировании безопасности (Security testing)**. Какие виды угроз ищут?

**Коротко:** тестирование безопасности проверяет систему на защищённость от несанкционированного доступа, утечек данных и атак; ищут уязвимости, связанные с конфиденциальностью, целостностью, доступностью, аутентификацией и авторизацией.

## Развёрнутый ответ

Тестирование безопасности (Security testing) — это вид нефункционального тестирования, цель которого — выявить уязвимости и недостатки в механизмах защиты системы, которые могут привести к потере данных, нарушению работы системы или получению несанкционированного доступа. Оценивается по ключевым атрибутам информационной безопасности, часто описываемым моделью **CIA (Confidentiality, Integrity, Availability)**:

- **Конфиденциальность (Confidentiality)** — данные доступны только авторизованным пользователям.
- **Целостность (Integrity)** — данные не могут быть изменены незаметно и без прав.
- **Доступность (Availability)** — система доступна легитимным пользователям, когда это необходимо.

Основные виды угроз/уязвимостей, которые ищут при тестировании безопасности:

- **SQL-инъекции (SQL injection)** — возможность внедрить вредоносный SQL-код через поля ввода и получить несанкционированный доступ к данным.
- **XSS (Cross-Site Scripting)** — внедрение вредоносного скрипта в страницу, который выполняется в браузере другого пользователя.
- **CSRF (Cross-Site Request Forgery)** — выполнение нежелательных действий от имени авторизованного пользователя без его ведома.
- **Проблемы аутентификации и авторизации** — слабые пароли, отсутствие ограничения попыток входа (brute force), некорректная проверка прав доступа (например, доступ к чужим данным через изменение ID в URL — IDOR).
- **Утечка конфиденциальных данных (Sensitive data exposure)** — передача паролей, токенов, персональных данных в открытом виде, без шифрования.
- **Небезопасная конфигурация (Security misconfiguration)** — открытые административные панели, дефолтные пароли, избыточные права доступа.
- **DoS/DDoS-атаки (Denial of Service)** — перегрузка системы запросами с целью сделать её недоступной.
- **Проблемы с сессиями (Session management)** — незащищённые или предсказуемые сессионные токены, отсутствие таймаута сессии.

Тестирование безопасности включает такие практики как **penetration testing** (симуляция реальной атаки специалистом по безопасности), **vulnerability scanning** (автоматическое сканирование известных уязвимостей), **security code review**, а также проверку по чек-листам известных стандартов, например **OWASP Top 10**.

## Пример / когда применяется

Примеры проверок безопасности для формы логина интернет-магазина:

```text
1. Ввод в поле логина: ' OR '1'='1
   -> система должна отклонить ввод, а не аутентифицировать пользователя (проверка SQL-инъекции)

2. Ввод в поле поиска: <script>alert(1)</script>
   -> скрипт не должен выполниться на странице (проверка XSS)

3. 10 неудачных попыток входа подряд с одного IP
   -> аккаунт должен временно блокироваться или требовать капчу (проверка brute force)

4. Изменение параметра order_id=1001 на order_id=1002 в URL чужого заказа
   -> система должна вернуть ошибку доступа, а не показать чужой заказ (проверка IDOR)
```

## На что смотрит интервьюер

- Знание конкретных типов уязвимостей (SQL injection, XSS, CSRF, IDOR), а не общие фразы про "проверку безопасности".
- Понимание модели CIA (конфиденциальность, целостность, доступность) как основы для классификации угроз.
- Знание OWASP Top 10 как индустриального стандарта — упоминание этого списка обычно воспринимается как сильный ответ.
- Follow-up: "Как бы вы протестировали конкретную форму на XSS/SQL-инъекцию вручную?" или "Какие инструменты для security-тестирования вы знаете (Burp Suite, OWASP ZAP)?"
