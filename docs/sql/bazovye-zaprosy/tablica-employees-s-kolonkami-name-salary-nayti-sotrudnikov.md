# Таблица Employees с колонками name, salary. Найти сотрудников с зарплатой больше 5000.

**Коротко:** используется оператор строгого сравнения `WHERE salary > 5000`, важно чётко разграничивать "больше" (`>`) и "больше или равно" (`>=`).

## Развёрнутый ответ

Запрос строится на числовом сравнении: оператор `>` возвращает строки, где значение колонки `salary` строго больше 5000. Если бы формулировка была "от 5000", правильным оператором был бы `>=`.

Технические нюансы, которые стоит проговорить на собеседовании:

- Если колонка `salary` допускает `NULL` (сотрудник без указанной зарплаты), такие строки автоматически исключаются из результата, так как сравнение `NULL > 5000` даёт `UNKNOWN`, а не `TRUE`. Это часто упускают из виду.
- Тип данных колонки имеет значение: если `salary` хранится как строка (`VARCHAR`), сравнение `'500' > '5000'` будет лексикографическим, а не числовым, и даст неверный результат. Корректная схема должна использовать числовой тип (`NUMERIC`, `DECIMAL`, `INT`).
- Для регулярных запросов с диапазонной фильтрацией по зарплате полезен индекс на `salary`, особенно в сочетании с `ORDER BY salary`.
- Стоит уточнить у интервьюера единицы измерения (не критично для запроса, но показывает внимательность к контексту задачи).

## Пример / когда применяется

Такой запрос применяется, например, для выявления сотрудников, попадающих под повышенную ставку налога или бонусную программу.

```sql
SELECT name, salary
FROM Employees
WHERE salary > 5000;
```

Пример результата:

| name   | salary |
|--------|--------|
| Иванов | 6200   |
| Козлов | 8000   |

## На что смотрит интервьюер

- Различает ли кандидат `>` и `>=` и внимательно ли читает формулировку задачи ("больше" vs "от").
- Помнит ли про поведение `NULL` при сравнениях.
- Замечает ли потенциальную проблему хранения числовых данных в виде строк.
- Follow-up: "а как отсортировать по убыванию зарплаты и вывести топ-3?" — ожидается `ORDER BY salary DESC LIMIT 3`.
