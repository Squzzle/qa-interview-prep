# Добавить ограничение внешнего ключа, чтобы customer_id в Orders ссылался на Customers.

**Коротко:** ограничение внешнего ключа добавляется командой `ALTER TABLE Orders ADD CONSTRAINT ... FOREIGN KEY (customer_id) REFERENCES Customers (id)`, что обеспечивает ссылочную целостность между таблицами.

## Развёрнутый ответ

Внешний ключ (`FOREIGN KEY`) — это ограничение ссылочной целостности, гарантирующее, что значение в колонке дочерней таблицы (Orders.customer_id) обязательно существует как значение первичного (или уникального) ключа в родительской таблице (Customers.id). Это предотвращает появление "осиротевших" записей — заказов, ссылающихся на несуществующего клиента.

Ключевые моменты синтаксиса:
- Добавление ограничения к уже существующей таблице выполняется через `ALTER TABLE ... ADD CONSTRAINT имя FOREIGN KEY (колонка) REFERENCES родительская_таблица (колонка)`.
- Явное указание имени ограничения (`CONSTRAINT fk_orders_customer`) — хорошая практика, упрощающая последующее изменение или удаление внешнего ключа.
- Колонка, на которую ссылаются (обычно `id` в Customers), должна быть первичным ключом или иметь уникальный индекс.
- Типы данных customer_id в Orders и id в Customers должны совпадать (или быть совместимыми).

Дополнительные возможности при определении FK:
- `ON DELETE CASCADE` — при удалении клиента автоматически удаляются все его заказы.
- `ON DELETE SET NULL` — при удалении клиента customer_id в заказах станет NULL (требует, чтобы колонка допускала NULL).
- `ON DELETE RESTRICT` / `NO ACTION` (поведение по умолчанию в большинстве СУБД) — запрещает удаление клиента, если у него есть связанные заказы.
- `ON UPDATE CASCADE` — аналогичное поведение при обновлении значения первичного ключа родителя.

Важный нюанс: перед добавлением FK к существующей таблице с данными СУБД проверит все текущие строки на соответствие ограничению — если в Orders уже есть customer_id, отсутствующие в Customers, операция завершится ошибкой, и потребуется предварительно очистить или исправить данные.

## Пример / когда применяется

Классическая связь "один ко многим": один клиент может иметь много заказов.

```sql
ALTER TABLE Orders
ADD CONSTRAINT fk_orders_customer
FOREIGN KEY (customer_id)
REFERENCES Customers (id)
ON DELETE RESTRICT
ON UPDATE CASCADE;
```

Тот же результат при создании таблицы с нуля:

```sql
CREATE TABLE Orders (
    id SERIAL PRIMARY KEY,
    customer_id INT NOT NULL,
    order_date DATE NOT NULL DEFAULT CURRENT_DATE,
    CONSTRAINT fk_orders_customer FOREIGN KEY (customer_id) REFERENCES Customers (id)
);
```

## На что смотрит интервьюер

- Знает ли кандидат, что ссылаемая колонка должна быть PRIMARY KEY или иметь UNIQUE-ограничение.
- Понимает ли разницу между `ON DELETE CASCADE`, `SET NULL` и `RESTRICT`/`NO ACTION`, и может привести бизнес-пример для каждого варианта.
- Упоминает ли, что при добавлении FK к таблице с существующими данными возможна ошибка валидации, если есть несогласованные строки.
- Красный флаг — предложение обеспечивать целостность только на уровне приложения без ограничения в БД (это создаёт риск рассинхронизации данных).
- Follow-up: что такое индекс на внешнем ключе и зачем он нужен для производительности JOIN и каскадных операций; чем внешний ключ отличается от обычного индекса.
