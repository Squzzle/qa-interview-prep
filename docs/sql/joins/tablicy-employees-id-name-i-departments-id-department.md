# Таблицы Employees (id, name) и Departments (id, department_name). Написать запрос с INNER JOIN, выводящий имя и отдел.

**Коротко:** нужно связать таблицы по внешнему ключу (обычно `Employees.department_id = Departments.id`) через `INNER JOIN`, который вернёт только те строки, где совпадение найдено в обеих таблицах.

## Развёрнутый ответ

`INNER JOIN` — это соединение двух таблиц, при котором в результат попадают только строки, для которых выполняется условие связи (`ON`) в обеих таблицах одновременно. Если у сотрудника нет отдела (значение внешнего ключа `NULL` или не найдено совпадение в `Departments`), такая строка в результат `INNER JOIN` не попадёт — в отличие от `LEFT JOIN`, который сохранил бы все строки из левой таблицы.

Для решения задачи предполагается, что в таблице `Employees` есть внешний ключ на `Departments`, например `department_id`. В условии задачи столбец не назван явно, поэтому на собеседовании важно проговорить это допущение: обычно подразумевается, что в `Employees` есть колонка `department_id`, ссылающаяся на `Departments.id`.

Ключевые моменты, которые стоит отметить:
- порядок таблиц в `INNER JOIN` не важен для результата (в отличие от `LEFT`/`RIGHT JOIN`);
- условие соединения указывается в `ON`, а не в `WHERE` (хотя для `INNER JOIN` результат формально совпадает, стилистически это разные вещи и для `OUTER JOIN` разница критична);
- если нужно явно выбрать конкретные столбцы, а не `*`, стоит квалифицировать их именем таблицы или алиасом во избежание неоднозначности (`name` может быть и в `Employees`, и потенциально пересекаться с другими таблицами).

## Пример / когда применяется

Допустим, схема такая: `Employees(id, name, department_id)`, `Departments(id, department_name)`.

```sql
SELECT
    e.name AS employee_name,
    d.department_name
FROM Employees e
INNER JOIN Departments d
    ON e.department_id = d.id;
```

Такой запрос типичен для отчётов, где нужно показать сотрудника вместе с расшифровкой связанного справочника (отдел, должность, категория и т.п.), и при этом строки без связи неинтересны или считаются некорректными данными.

## На что смотрит интервьюер

- Понимает ли кандидат разницу между `INNER JOIN` и `OUTER JOIN` (что INNER отбрасывает несовпадающие строки).
- Использует ли алиасы таблиц и квалифицирует ли столбцы, чтобы избежать неоднозначности имён.
- Задаёт ли уточняющий вопрос про поле связи, если оно явно не указано в условии, вместо того чтобы придумывать несуществующий столбец без пояснений.
- Частая ошибка — писать условие соединения в `WHERE` вместо `ON`, что не критично для `INNER JOIN`, но показывает непонимание синтаксиса для более сложных случаев.
- Возможный follow-up: "а что если добавить фильтр по отделу — куда его лучше поставить, в `ON` или в `WHERE`, и как это изменит результат при `LEFT JOIN`?"
