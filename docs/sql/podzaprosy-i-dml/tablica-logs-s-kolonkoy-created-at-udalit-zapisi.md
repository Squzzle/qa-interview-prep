# Таблица Logs с колонкой created_at. Удалить записи старше 2023 года (DELETE).

**Коротко:** используется `DELETE FROM Logs WHERE created_at < '2023-01-01'` — удаляются все строки, у которых дата создания раньше начала 2023 года; важно правильно выбрать граничное условие и по возможности выполнять удаление батчами.

## Развёрнутый ответ

Оператор `DELETE` удаляет строки, удовлетворяющие условию `WHERE`. "Старше 2023 года" означает `created_at < '2023-01-01'` — все записи, созданные до 1 января 2023 года. Частая ошибка — использовать `<= '2023-12-31'`, что тоже сработает, но менее надёжно при наличии временной составляющей (`TIMESTAMP`), так как `'2023-12-31'` без времени интерпретируется как полночь и может случайно не захватить записи за последний день года; более безопасная и явная формулировка через нижнюю границу следующего периода — `created_at < '2024-01-01'`.

Важные нюансы:

- Если `created_at` имеет тип `TIMESTAMP`/`DATETIME`, сравнение со строкой `'2023-01-01'` обычно неявно приводится к timestamp `2023-01-01 00:00:00`, что и нужно.
- На больших таблицах массовый `DELETE` может заблокировать таблицу надолго, создать большую нагрузку на журнал транзакций (WAL/redo log) и раздуть его. Поэтому на практике удаление часто выполняют **батчами** (например, по 10 000 строк за раз в цикле), чтобы не держать долгую блокировку и не переполнять журнал транзакций.
- Перед `DELETE` стоит выполнить эквивалентный `SELECT COUNT(*)` с тем же условием, чтобы оценить объём удаляемых данных и убедиться в правильности условия.
- Если данные не нужно удалять физически сразу, а нужно архивировать, распространённый паттерн — сначала скопировать удаляемые строки в архивную таблицу (`INSERT INTO Logs_archive SELECT * FROM Logs WHERE created_at < '2024-01-01'`), а затем удалить их из основной таблицы.
- В некоторых СУБД для таблиц с секционированием по дате (partitioning) вместо `DELETE` эффективнее просто удалить/отсоединить целую партицию (`DROP PARTITION`), что происходит почти мгновенно, в отличие от построчного `DELETE`.
- Операцию стоит оборачивать в транзакцию, чтобы иметь возможность отката при ошибке.

## Пример / когда применяется

Простой вариант:

```sql
DELETE FROM Logs
WHERE created_at < '2024-01-01';
```

Безопасный сценарий с проверкой и транзакцией:

```sql
-- 1. Оценить объём удаляемых данных
SELECT COUNT(*) FROM Logs WHERE created_at < '2024-01-01';

-- 2. Выполнить удаление в транзакции
BEGIN;
DELETE FROM Logs
WHERE created_at < '2024-01-01';
COMMIT;
```

Батчевое удаление (псевдо-цикл для больших таблиц, синтаксис MySQL/PostgreSQL с LIMIT через подзапрос):

```sql
DELETE FROM Logs
WHERE id IN (
    SELECT id FROM Logs
    WHERE created_at < '2024-01-01'
    LIMIT 10000
);
-- повторять, пока не будет удалено 0 строк
```

## На что смотрит интервьюер

- Правильность граничного условия (`< '2024-01-01'` вместо ненадёжного `<= '2023-12-31'`), особенно с учётом времени в `TIMESTAMP`.
- Понимание рисков массового `DELETE` на больших таблицах: блокировки, рост журнала транзакций, влияние на реплики.
- Знание паттерна батчевого удаления как способа снизить нагрузку.
- Упоминание архивирования как альтернативы физическому удалению логов.
- Готовность проверить условие через `SELECT`/`COUNT(*)` перед реальным удалением — признак аккуратного инженера.
