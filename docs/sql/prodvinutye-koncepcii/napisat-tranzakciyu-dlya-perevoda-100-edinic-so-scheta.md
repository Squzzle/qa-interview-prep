# Написать транзакцию для перевода 100 единиц со счёта 1 на счёт 2 с откатом при ошибке.

**Коротко:** транзакция открывается через `BEGIN`, внутри выполняются два `UPDATE` (списание со счёта 1 и зачисление на счёт 2), а завершается либо `COMMIT` при успехе, либо `ROLLBACK` при ошибке (например, если баланс ушёл в отрицательное значение или произошёл сбой) — это обеспечивает атомарность перевода.

## Развёрнутый ответ

Транзакция — это последовательность операций, которая выполняется как единое неделимое целое: либо применяются все изменения, либо ни одно. Это обеспечивается свойствами ACID:

- **Atomicity (атомарность)** — все операции внутри транзакции выполняются полностью или не выполняются вовсе.
- **Consistency (согласованность)** — транзакция переводит базу из одного согласованного состояния в другое, соблюдая все ограничения (constraints).
- **Isolation (изолированность)** — параллельные транзакции не видят промежуточные, незафиксированные изменения друг друга (степень зависит от уровня изоляции).
- **Durability (долговечность)** — после `COMMIT` изменения сохраняются даже при сбое системы.

Перевод денег — хрестоматийный пример, почему нужны транзакции: если списание со счёта 1 прошло, а зачисление на счёт 2 не удалось (например, обрыв соединения, нарушение constraint, деление на ноль в триггере), без транзакции деньги "исчезнут". Обернув обе операции в одну транзакцию, при любой ошибке можно откатить (`ROLLBACK`) все изменения целиком.

На практике проверка на достаточность средств (`balance >= 100`) должна быть частью логики — либо через явную проверку перед `UPDATE` с последующим `ROLLBACK`, либо через `CHECK` constraint на колонке `balance`, который автоматически прервёт транзакцию, если баланс станет отрицательным.

В большинстве клиентов (приложениях, скриптах) обработка ошибок реализуется через конструкцию try/catch на уровне кода, где `ROLLBACK` вызывается в блоке catch. Но чистый SQL-скрипт для интервью обычно демонстрируется с явными командами `BEGIN`/`COMMIT`/`ROLLBACK`, а в PL/pgSQL или T-SQL — с блоком `EXCEPTION`/`TRY...CATCH`.

## Пример / когда применяется

Универсальный вариант (стандартный SQL, психологически ожидаемый на собеседовании):

```sql
BEGIN;

UPDATE accounts
SET balance = balance - 100
WHERE account_id = 1 AND balance >= 100;

UPDATE accounts
SET balance = balance + 100
WHERE account_id = 2;

-- Если что-то пошло не так (например, первая строка не была обновлена
-- из-за недостатка средств, или возникла ошибка) — откатываем:
-- ROLLBACK;

COMMIT;
```

Более надёжный вариант с явной проверкой количества затронутых строк (псевдокод логики приложения, оборачивающей транзакцию):

```sql
BEGIN;

UPDATE accounts SET balance = balance - 100
WHERE account_id = 1 AND balance >= 100;

-- если UPDATE не затронул ни одной строки (недостаточно средств) —
-- приложение вызывает ROLLBACK и прерывает операцию

UPDATE accounts SET balance = balance + 100
WHERE account_id = 2;

COMMIT;
```

Вариант на PostgreSQL с PL/pgSQL и блоком исключений:

```sql
DO $$
BEGIN
    UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;
    IF NOT FOUND OR (SELECT balance FROM accounts WHERE account_id = 1) < 0 THEN
        RAISE EXCEPTION 'Недостаточно средств на счёте 1';
    END IF;

    UPDATE accounts SET balance = balance + 100 WHERE account_id = 2;
EXCEPTION
    WHEN OTHERS THEN
        ROLLBACK;
        RAISE;
END $$;
```

## На что смотрит интервьюер

Проверяется знание принципа атомарности и того, что `COMMIT` должен вызываться только после успешного выполнения всех операций, а `ROLLBACK` — при любой ошибке. Частая ошибка — забыть проверку достаточности средств, из-за чего баланс может уйти в минус. Также интервьюер может спросить про уровни изоляции транзакций (`READ COMMITTED`, `REPEATABLE READ`, `SERIALIZABLE`) и про то, как избежать deadlock при параллельных переводах между одними и теми же счетами (например, всегда блокировать счета в одном и том же порядке — по возрастанию `account_id`).
