# Создать представление HighEarners для сотрудников с зарплатой выше 8000.

**Коротко:** представление создаётся командой `CREATE VIEW HighEarners AS SELECT ... FROM employees WHERE salary > 8000` — это сохранённый именованный запрос, который выполняется заново при каждом обращении к представлению и не хранит данные физически.

## Развёрнутый ответ

Представление (view) — это виртуальная таблица, основанная на результате хранимого SQL-запроса. Представление не хранит данные физически (в отличие от материализованного представления), а при каждом обращении заново выполняет исходный запрос над актуальными данными базовой таблицы.

Преимущества использования представлений:

- **Инкапсуляция сложности** — сложный запрос с джойнами, фильтрами и агрегатами скрывается за простым именем, к которому можно обращаться как к обычной таблице.
- **Безопасность** — можно дать пользователю доступ только к представлению, скрыв часть столбцов или строк базовой таблицы (например, скрыть зарплаты ниже определённого порога от посторонних, показывать только нужные колонки).
- **Переиспользование логики** — часто используемый фильтр или расчёт не нужно дублировать в разных запросах.
- **Логическая независимость** — если структура базовой таблицы изменится, можно адаптировать определение представления, не меняя запросы клиентского кода, которые к нему обращаются.

Важные нюансы:

- Обычное представление (`VIEW`) не даёт прироста производительности — оно просто разворачивается в исходный запрос при выполнении. Для кэширования результата с ускорением чтения используется `MATERIALIZED VIEW`, но его нужно обновлять вручную или по расписанию (`REFRESH MATERIALIZED VIEW`), и данные могут быть неактуальны.
- Представление можно использовать не только для `SELECT`, но в ряде случаев и для `INSERT`/`UPDATE`/`DELETE`, если оно "обновляемое" (основано на одной таблице без агрегатов, `DISTINCT`, `GROUP BY` и т.п.).
- Если нужно изменить определение представления, используется `CREATE OR REPLACE VIEW` (в большинстве СУБД можно добавлять столбцы в конец списка, но не менять типы существующих без пересоздания).
- Для удаления — `DROP VIEW HighEarners;`.

## Пример / когда применяется

Таблица `employees(employee_id, name, department_id, salary)`.

```sql
CREATE VIEW HighEarners AS
SELECT employee_id, name, department_id, salary
FROM employees
WHERE salary > 8000;
```

Использование представления как обычной таблицы:

```sql
SELECT *
FROM HighEarners
WHERE department_id = 3
ORDER BY salary DESC;
```

Обновление определения представления:

```sql
CREATE OR REPLACE VIEW HighEarners AS
SELECT employee_id, name, department_id, salary, hire_date
FROM employees
WHERE salary > 8000;
```

## На что смотрит интервьюер

Проверяется понимание того, что представление — это не физическая копия данных, а сохранённый запрос, выполняемый "на лету". Частая ошибка — путать `VIEW` с `MATERIALIZED VIEW` и считать, что представление ускоряет выполнение запроса. Также могут спросить, можно ли делать `INSERT`/`UPDATE` через представление (да, но с ограничениями — только для "простых", обновляемых представлений) и как обеспечить безопасность данных через представления (скрытие столбцов, ограничение строк, выдача прав только на view, а не на базовую таблицу).
