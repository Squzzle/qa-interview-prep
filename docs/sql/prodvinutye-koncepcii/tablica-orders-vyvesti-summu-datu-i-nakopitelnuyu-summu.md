# Таблица Orders. Вывести сумму, дату и накопительную сумму по клиентам (оконная функция SUM() OVER).

**Коротко:** накопительный итог (running total) считается оконной функцией `SUM(amount) OVER (PARTITION BY customer_id ORDER BY order_date)`, которая суммирует значения текущей и всех предыдущих строк в пределах партиции клиента, упорядоченных по дате.

## Развёрнутый ответ

Накопительная сумма — классический пример использования оконной функции с так называемым "оконным кадром" (window frame). По умолчанию, когда в `OVER` указан `ORDER BY` без явного указания рамки, применяется неявная рамка `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` — то есть суммируются все строки от начала партиции до текущей строки включительно.

`PARTITION BY customer_id` разбивает данные так, чтобы накопление шло отдельно для каждого клиента, а не по всей таблице целиком. `ORDER BY order_date` задаёт хронологический порядок, в котором строки "накапливаются".

Важные нюансы:

- Если у нескольких заказов одного клиента совпадает дата, `RANGE` (используемый по умолчанию) просуммирует их как одну "пиковую точку" — все строки с одинаковым значением `order_date` получат одинаковую накопленную сумму, равную сумме до конца этой группы дат. Чтобы сумма считалась строго построчно (в порядке физических строк, а не значений), нужно явно указать `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`.
- Можно ограничить окно, например, только последними 3 заказами (`ROWS BETWEEN 2 PRECEDING AND CURRENT ROW`) — это уже будет не накопительная, а скользящая сумма.
- Оконные функции не убирают дублирующиеся строки — если нужен результат "один заказ = одна строка с накопленной суммой", исходные данные не должны содержать дублей на уровне заказа.

## Пример / когда применяется

Таблица `orders(order_id, customer_id, order_date, amount)`.

```sql
SELECT
    customer_id,
    order_date,
    amount,
    SUM(amount) OVER (
        PARTITION BY customer_id
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total
FROM orders
ORDER BY customer_id, order_date;
```

Такой запрос типичен для отчётов "динамика трат клиента нарастающим итогом", дашбордов и финансовой аналитики.

## На что смотрит интервьюер

Проверяется понимание оконных кадров (`ROWS` vs `RANGE`), а не только сам факт "использовать SUM() OVER". Частая ошибка кандидатов — не указывать явно `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` и не осознавать, что поведение по умолчанию (`RANGE`) может дать неожиданный результат при дублирующихся значениях `ORDER BY`. Также могут спросить, как получить скользящее среднее или сумму за последние N записей — это модификация рамки окна.
