# Дизайн: сайт для подготовки к собеседованиям по тестированию (QA)

**Дата:** 2026-07-09
**Статус:** утверждён (ожидает финального ревью спеки)
**Источник вопросов:** https://habr.com/ru/articles/1007736/

## 1. Цель

Собрать сайт-тренажёр для подготовки к QA-собеседованиям (2025–2026):

1. **Ресурс вопросов** — структурированный каталог: **один вопрос = одна страница**, с исчерпывающим ответом, который «удовлетворит любого интервьюера».
2. **Блок проверки знаний** — курируемая база тестовых вопросов (5 вариантов ответа, правильных от 1 до 5); кнопка «Пройти тест» → тест из 20 случайных вопросов; при неверном/неполном ответе — ссылка на страницу разбора темы.
3. **Инфраструктура** — `Dockerfile` для сборки образа и Helm-chart для деплоя в k8s (minikube, namespace `interview-question`).

## 2. Стек

- TypeScript
- Docusaurus (docs-mode, статический сайт)
- React (компонент теста)
- Docker (multi-stage: node build → nginx)
- Helm + minikube

## 3. Объём контента (scope)

### Включаем

- **Manual QA — теория (18 блоков):** Основы тестирования; Документы и отчёты; Основные понятия; Модели разработки; Agile и Scrum; Уровни и виды тестирования; Баги и дефекты; Тест-дизайн; Эстимация и метрики; API; Сети и протоколы; HTTP; Архитектура ПО; Базы данных и SQL; Мобильное тестирование; Логирование и метрики; Типы данных и форматы; Командная строка.
- **Automation QA (12 блоков)** — как концептуальные вопросы по автоматизации (без разбора Java-кода): Основы автоматизации; Git; CI/CD; Тест-раннеры; Системы сборки; Локаторы; Инициализация браузера и Selenium; Playwright; API-тестирование; Архитектура фреймворка; Логгеры и репортеры; Автоматизация мобильных приложений.
- **Лайвкодинг SQL (~30 задач).**
- **HR / логические задачи / вопросы для интервьюера.**

### Исключаем

- Раздел **«Вопросы по Java»** (12 блоков теории Java).
- **«Лайвкодинг Java»** (~43 задачи).

> Раздел Automation QA остаётся, но подаётся как вопросы про инструменты/подходы автоматизации, а не как разбор Java-синтаксиса.

## 4. Хранение данных

Базы данных нет — принцип YAGNI (нет аккаунтов, истории прохождений, админки).

| Данные | Хранилище |
|---|---|
| Ответы на вопросы (теория) | Markdown-файлы `docs/**/*.md` |
| База тестовых вопросов | JSON-файлы `data/quiz/*.json` |
| Результаты прохождения теста | не хранятся (считаются в браузере) |

Итог: сайт полностью статический, под в k8s stateless.

## 6. Шаблон страницы-ответа

Каждая страница вопроса самодостаточна и «интервью-готова»:

```markdown
---
id: <slug>
title: <короткий заголовок>
sidebar_position: <n>
tags: [<категория>, ...]
---

# <Полный текст вопроса>

**Коротко:** 1–2 фразы (TL;DR).

## Развёрнутый ответ
Суть, определения, детали, важные нюансы.

## Пример / когда применяется
Практический пример или контекст применения.

## На что смотрит интервьюер
Частые ошибки, follow-up вопросы, «красные флаги».
```

## 7. Блок теста (часть 2)

### 7.1 Модель данных тестового вопроса

```json
{
  "id": "mq-osnovy-001",
  "category": "Manual QA / Основы тестирования",
  "question": "Что из перечисленного относится к тестированию?",
  "options": ["Вариант A", "Вариант B", "Вариант C", "Вариант D", "Вариант E"],
  "correct": [0, 2],
  "topicUrl": "/docs/manual-qa/osnovy-testirovaniya/chto-takoe-testirovanie"
}
```

- Ровно 5 `options`.
- `correct` — массив индексов правильных вариантов, длина от 1 до 5.
- `topicUrl` — ссылка на страницу разбора темы.

### 7.2 База

Курируемая база, покрывающая все включённые темы: ~5–10 тестовых вопросов на блок. Каждый тестовый вопрос ссылается на существующую страницу-разбор.

### 7.3 Поведение

- `src/pages/test.tsx` — лендинг с кнопкой **«Пройти тест»** и кратким описанием.
- По нажатию `<Quiz>` берёт **20 случайных** вопросов из базы (без повторов), рисует чекбоксы (множественный выбор).
- Кнопка «Проверить» → оценка.
- **Экран результатов:** счёт `X/20`; по каждому неверному/неполному вопросу — текст вопроса и ссылка на `topicUrl` («Разобрать тему»).
- Кнопка «Пройти заново» перезапускает с новой случайной выборкой.

### 7.4 Логика оценки (чистая функция, TDD)

`grade(selected: number[], correct: number[]) => boolean`

- Ответ засчитывается **только при точном совпадении множеств** выбранного и правильного (сортировка/сравнение множеств).
- Неполный выбор (выбрано меньше правильных) — **неверно**.
- Лишний выбор (есть неправильный вариант) — **неверно**.
- Покрывается модульными тестами до реализации компонента.

## 8. Docker

Multi-stage `Dockerfile`:

1. **builder** (`node:22-alpine`): `npm ci` → `npm run build` (Docusaurus → `build/`).
2. **runtime** (`nginx:alpine`): копия `build/` в `/usr/share/nginx/html`, свой `nginx.conf` (gzip, `try_files`, кэш статики), `EXPOSE 80`.

Результат — лёгкий stateless-образ.

## 9. Helm-chart

`helm/interview-question/`:

- **Chart.yaml** — метаданные чарта.
- **values.yaml** — `image.repository`, `image.tag`, `image.pullPolicy: IfNotPresent`, `replicaCount: 1`, `service.type: ClusterIP`, `service.port: 80`, `resources` (requests/limits), настройки проб.
- **templates/deployment.yaml** — 1 реплика, HTTP liveness/readiness пробы на `/` (порт 80), ресурсы из values.
- **templates/service.yaml** — `ClusterIP`, порт 80.
- **templates/_helpers.tpl** — стандартные хелперы имён/лейблов.
- **templates/NOTES.txt** — печатает готовую команду `kubectl port-forward` для доступа к сайту.

**Доступ:** `Service ClusterIP` + `kubectl port-forward` (без ingress-addon).

**Образ:** собирается прямо в docker-демоне minikube (`eval $(minikube docker-env)` + `docker build`), поэтому `pull` не требуется (`IfNotPresent`).

**Namespace:** `interview-question` (создаётся перед деплоем).

## 10. Тестирование и верификация

- **TDD** на функцию оценки теста (`grading.test.ts`, Vitest/Jest).
- `npm run build` проходит без битых ссылок (Docusaurus `onBrokenLinks: throw`).
- Сборка Docker-образа успешна; контейнер локально отдаёт сайт (curl).
- Helm-деплой в namespace `interview-question` в minikube; под в статусе `Running`; `port-forward` + проверка ответа сайта.

## 11. Подход к реализации (ultracode)

Из-за объёма контента реализация раскладывается на workflow-этапы с параллельными сабагентами:

1. **Каркас:** инициализация Docusaurus + TypeScript, конфиг, `<Quiz>` с TDD-логикой оценки, `Dockerfile`, `nginx.conf`, Helm-chart.
2. **Извлечение вопросов** из статьи по каждой включённой категории (fan-out, дословный текст вопросов).
3. **Генерация** страниц-ответов (по шаблону из §6) + записей тестовой базы (§7.1) пакетами по категориям (pipeline).
4. **Сборка и верификация:** `npm run build` → Docker build → helm install в minikube → port-forward → проверка.

## 12. Вне объёма (out of scope)

- Аккаунты пользователей, авторизация.
- Сохранение истории прохождений, аналитика.
- Админ-панель редактирования вопросов.
- Разделы Java-теории и лайвкодинга Java.
- CI/CD-пайплайн (только Dockerfile + Helm как артефакты).
