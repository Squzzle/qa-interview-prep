# Чем различаются ключевые слова interface и type в TypeScript?

**Коротко:** `interface` описывает форму объекта и поддерживает декларативное слияние (declaration merging) и расширение через `extends`, а `type` — это псевдоним для любого типа (объекта, union, tuple, примитива, условного типа) и не поддерживает слияние объявлений; в большинстве практических случаев для объектных форм они взаимозаменяемы.

## Развёрнутый ответ

Основные различия:

1. **Область применимости.** `type` — это псевдоним (alias) для абсолютно любого типа: объекта, примитива, union (`"a" | "b"`), tuple (`[string, number]`), функционального типа, условного типа, mapped-типа. `interface` предназначен только для описания формы объектов (включая функции и конструкторы как вызываемые/конструируемые сигнатуры), но не может напрямую описать union или примитив.
2. **Декларативное слияние (declaration merging).** Если объявить два `interface` с одинаковым именем в одной области видимости, TypeScript автоматически объединит их поля в один интерфейс. Это активно используется для расширения типов сторонних библиотек (например, добавление полей в `Express.Request`). `type` с одинаковым именем в той же области видимости объявить нельзя — это вызовет ошибку дублирования идентификатора.
3. **Расширение.** `interface` расширяется через `extends` (может наследовать сразу от нескольких интерфейсов). `type` комбинируется через пересечение типов (`&`). Семантически результат похож, но механизм и сообщения об ошибках отличаются: при конфликте типов полей `interface extends` даст ошибку сразу при объявлении, а пересечение (`&`) в `type` может тихо создать тип `never` для конфликтующего поля.
4. **Производительность компилятора.** На больших кодовых базах `interface`, по данным команды TypeScript, обычно быстрее проверяется компилятором, особенно при частом расширении, так как интерфейсы кешируются по имени, тогда как сложные пересечения типов пересчитываются каждый раз заново.
5. **Вычисляемые/условные типы.** Только `type` может выражать условные типы (`T extends U ? X : Y`), mapped-типы (`{ [K in keyof T]: ... }`) и рекурсивные union-конструкции — `interface` этого не умеет.

На практике общепринятая рекомендация (в том числе в TypeScript Handbook): использовать `interface` для описания публичных форм объектов и API (особенно когда нужна возможность расширения библиотечных типов), а `type` — для всего остального: union, tuple, вычисляемых и составных типов.

## Пример / когда применяется

```ts
// interface: декларативное слияние
interface Window {
  myGlobal: string;
}
interface Window {
  anotherGlobal: number;
}
// итоговый Window содержит оба поля — myGlobal и anotherGlobal

// type: нельзя объявить дважды с одним именем — ошибка компиляции
type Config = { debug: boolean };
// type Config = { verbose: boolean }; // Error: Duplicate identifier 'Config'
```

```ts
// type умеет то, что не умеет interface
type Status = "pending" | "success" | "error"; // union
type Pair = [string, number]; // tuple
type Nullable<T> = T | null; // generic-псевдоним
type ReadonlyKeys<T> = {
  readonly [K in keyof T]: T[K];
}; // mapped-тип

interface User {
  id: number;
  name: string;
}
interface Admin extends User {
  permissions: string[];
}
```

## На что смотрит интервьюер

- Знает ли кандидат про декларативное слияние `interface` и умеет привести реальный пример его применения (расширение типов из `node_modules`, augmentation Express/React).
- Понимает ли, что `type` может выразить union/tuple/условные/mapped-типы, а `interface` — нет.
- Может ли объяснить разницу между `extends` у интерфейсов и пересечением (`&`) у типов при конфликте полей.
- Частая ошибка — утверждение, что "interface и type абсолютно одинаковы" без оговорок, или наоборот, незнание, что для простых объектных форм они действительно взаимозаменяемы в 90% случаев.
