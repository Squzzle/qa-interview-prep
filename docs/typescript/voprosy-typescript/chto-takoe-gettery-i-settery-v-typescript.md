# Что такое геттеры и сеттеры в TypeScript?

**Коротко:** Геттеры (`get`) и сеттеры (`set`) — это методы доступа, позволяющие управлять чтением и записью свойства объекта или класса так, как будто это обычное поле, добавляя при этом валидацию, вычисление значения или инкапсуляцию внутреннего состояния.

## Развёрнутый ответ

Геттеры и сеттеры — это стандартная возможность JavaScript (ES5+), унаследованная TypeScript и дополненная строгой типизацией. Они объявляются с ключевыми словами `get` и `set` перед именем метода, но используются как свойства — без круглых скобок при вызове.

Зачем они нужны:

- **Инкапсуляция.** Внутреннее состояние объекта хранится в приватном поле (например, `private _value`), а доступ к нему извне контролируется через геттер/сеттер, что позволяет менять внутреннюю реализацию, не затрагивая внешний API.
- **Валидация при записи.** Сеттер может проверять входящее значение перед сохранением и бросать ошибку или игнорировать некорректные данные.
- **Вычисляемые свойства.** Геттер может возвращать значение, вычисленное на основе других полей, не храня его физически (например, `get fullName()` из `firstName` и `lastName`).
- **Свойства только для чтения.** Если объявить только `get` без `set`, свойство становится доступным лишь для чтения снаружи — попытка присваивания вызовет ошибку компиляции.
- **Побочные эффекты при записи.** Сеттер может, помимо сохранения значения, выполнять дополнительную логику — логирование, эмит событий, инвалидацию кэша.

Технические детали и нюансы:

- Геттер и сеттер с одинаковым именем должны иметь совместимые типы: тип, возвращаемый геттером, должен быть присваиваем типу, ожидаемому сеттером (начиная с TypeScript 4.3 они могут иметь разные типы).
- В transpiled JS геттеры/сеттеры реализуются через `Object.defineProperty` с дескрипторами `get`/`set`, а не как обычные поля.
- Внутри класса TypeScript геттеры/сеттеры можно комбинировать с модификаторами доступа (`private set`, `public get`) начиная с TS 4.3, что позволяет сделать свойство публично читаемым, но приватно изменяемым.
- Использование геттеров/сеттеров имеет небольшие накладные расходы по сравнению с прямым доступом к полю, поэтому в высокопроизводительных участках кода стоит оценивать целесообразность.

## Пример / когда применяется

```ts
class BankAccount {
  private _balance: number;

  constructor(initialBalance: number) {
    this._balance = initialBalance;
  }

  get balance(): number {
    return this._balance;
  }

  set balance(amount: number) {
    if (amount < 0) {
      throw new Error("Баланс не может быть отрицательным");
    }
    this._balance = amount;
  }
}

const account = new BankAccount(100);
console.log(account.balance); // 100 — вызов геттера без скобок
account.balance = 200; // вызов сеттера
// account.balance = -50; // бросит Error во время выполнения

class Person {
  constructor(private firstName: string, private lastName: string) {}

  get fullName(): string {
    return `${this.firstName} ${this.lastName}`;
  }
}
```

## На что смотрит интервьюер

- Понимание, что геттеры/сеттеры вызываются без скобок, как обращение к свойству, а не как обычный метод.
- Знание, что это работает и в объектных литералах, и в классах.
- Умение объяснить разницу между свойством «только для чтения» через `readonly` (compile-time запрет присваивания) и через геттер без сеттера (может дополнительно включать рантайм-логику).
- Частая ошибка — забыть, что несовместимость типов геттера и сеттера (до TS 4.3) приведёт к ошибке компиляции.
- Follow-up: `Object.defineProperty`, decorators для валидации свойств, влияние геттеров/сеттеров на производительность и сериализацию (`JSON.stringify` не вызывает геттеры автоматически, если они не enumerable через обычное объявление в классе — на деле accessor-свойства класса enumerable).
