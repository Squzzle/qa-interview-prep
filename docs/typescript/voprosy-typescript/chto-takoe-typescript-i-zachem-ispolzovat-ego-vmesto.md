# Что такое TypeScript и зачем использовать его вместо JavaScript?

**Коротко:** TypeScript — это надмножество JavaScript, разработанное Microsoft, которое добавляет статическую типизацию, компилируется (транспилируется) в обычный JavaScript и позволяет находить ошибки на этапе разработки, а не в рантайме.

## Развёрнутый ответ

TypeScript (TS) — язык программирования с открытым исходным кодом, являющийся строгим синтаксическим надмножеством JavaScript. Это значит, что любой валидный JavaScript-код является одновременно валидным TypeScript-кодом. TS добавляет к языку статическую (опциональную) систему типов, интерфейсы, generics, enum, namespace, декораторы и другие возможности, которых нет в чистом JS. Код на TS не выполняется браузером или Node.js напрямую — он компилируется компилятором `tsc` в обычный JavaScript нужной версии (ES5, ES2015 и т.д.), после чего типы стираются (erasure) и не влияют на рантайм.

Основные причины использовать TypeScript вместо JavaScript:

- **Статическая проверка типов.** Ошибки вида «обращение к несуществующему свойству», «передача аргумента неверного типа», «использование `undefined` там, где ожидается объект» отлавливаются на этапе компиляции, а не после деплоя.
- **Улучшенный developer experience.** IDE (VS Code, WebStorm) получают точную автодополнение, навигацию по коду, безопасный рефакторинг (переименование символов), подсказки по сигнатурам функций — всё это работает благодаря системе типов.
- **Самодокументируемый код.** Типы, интерфейсы и generics описывают контракт функций и структур данных, что снижает необходимость в дополнительной документации и облегчает онбординг новых разработчиков.
- **Масштабируемость.** На больших кодовых базах и в командах из многих разработчиков статическая типизация снижает количество runtime-ошибок, связанных с несоответствием форматов данных между модулями.
- **Поддержка современных и будущих возможностей JS.** TypeScript часто поддерживает синтаксис новых стандартов ECMAScript раньше, чем он становится доступен во всех средах выполнения, транспилируя его в совместимый код.
- **Постепенное внедрение.** TypeScript можно подключать в существующий JS-проект инкрементально благодаря `allowJs` и `checkJs`, а также флагу `strict`, который можно включать не сразу, а по мере готовности кода.

Важный нюанс: типизация в TypeScript — это структурная (duck typing) система, а не номинативная, как, например, в Java или C#. Совместимость типов определяется формой (набором свойств и методов), а не именем типа или явным объявлением реализации интерфейса.

## Пример / когда применяется

```ts
// JavaScript — ошибка обнаружится только в рантайме
function sum(a, b) {
  return a + b;
}
sum(2, "3"); // "23" — не бросит ошибку, но результат неожиданный

// TypeScript — ошибка находится сразу в IDE и при компиляции
function sumTs(a: number, b: number): number {
  return a + b;
}
sumTs(2, "3"); // Error: Argument of type 'string' is not assignable to parameter of type 'number'
```

TypeScript особенно оправдан в больших фронтенд- и бэкенд-приложениях (React, Angular, NestJS), в командных проектах с долгим жизненным циклом, в библиотеках с публичным API, а также там, где важна надёжность рефакторинга.

## На что смотрит интервьюер

- Понимает ли кандидат, что TypeScript — это именно надмножество, а не отдельный язык, и что типы полностью стираются при компиляции (не дают рантайм-гарантий, например, не защищают от неверных данных, пришедших из `JSON.parse` без валидации).
- Может ли назвать конкретные практические преимущества, а не только «типы — это хорошо».
- Частая ошибка — утверждение, что TypeScript «делает JS безопасным в рантайме»: важно подчеркнуть, что это только статическая проверка на этапе компиляции.
- Follow-up: чем структурная типизация отличается от номинативной; что такое `tsc` и `strict` режим; как TS работает с `any` и `unknown`.
