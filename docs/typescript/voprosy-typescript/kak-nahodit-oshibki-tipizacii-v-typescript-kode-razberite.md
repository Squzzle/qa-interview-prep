# Как находить ошибки типизации в TypeScript-коде? Разберите типичные ошибки на примере.

**Коротко:** Ошибки типизации находятся через компилятор `tsc` (сборка или `--noEmit`), встроенную проверку IDE в реальном времени, ESLint с `@typescript-eslint`, а также включение строгого режима (`strict`) и CI-проверок; типичные ошибки — неявный `any`, некорректная работа с `null`/`undefined`, несоответствие сигнатур и утечка типов через `as`.

## Развёрнутый ответ

Инструменты для поиска ошибок типизации:

- **Компилятор `tsc`.** Основной источник истины: запуск `tsc` или `tsc --noEmit` (без генерации файлов) в терминале или в CI-пайплайне выводит полный список ошибок типов по всему проекту.
- **IDE / Language Service.** VS Code и другие редакторы подсвечивают ошибки типов в реальном времени по мере набора кода, используя тот же движок, что и `tsc`.
- **ESLint + `@typescript-eslint`.** Линтер с TypeScript-плагином ловит не только синтаксические, но и типовые антипаттерны: `no-explicit-any`, `no-unsafe-assignment`, `no-floating-promises`, `strict-boolean-expressions` и т.д. Это дополняет `tsc`, так как компилятор не запрещает многие небезопасные, но валидные с точки зрения типов конструкции.
- **Строгий режим (`strict: true`).** Включает пакет флагов: `strictNullChecks`, `noImplicitAny`, `strictFunctionTypes`, `strictBindCallApply`, `strictPropertyInitialization`, `noImplicitThis`, `alwaysStrict`. Без этих флагов огромный класс ошибок (например, обращение к `null`) вообще не диагностируется.
- **CI/pre-commit hooks.** Запуск проверки типов как отдельного шага сборки (например, в GitHub Actions) до деплоя, чтобы не пропустить регрессии типов.

Типичные ошибки типизации и способы их находить/исправлять:

1. **Неявный `any` (implicit any).** Параметр функции без указанного типа и без возможности вывода получает тип `any`, что отключает проверку. Флаг `noImplicitAny` превращает это в ошибку компиляции.
2. **Null/undefined ошибки.** Обращение к свойству потенциально `null`/`undefined` значения без проверки — ловится флагом `strictNullChecks`.
3. **Несовместимость сигнатур функций.** Передача callback с более широким набором параметров туда, где ожидается более узкий (или наоборот), нарушение контравариантности параметров — ловится `strictFunctionTypes`.
4. **Злоупотребление приведением типов (`as`, `<Type>`).** `as` отключает проверку компилятора в конкретном месте, «обманывая» его; при неправильном приведении ошибка проявится только в рантайме.
5. **Несоответствие форм объектов (excess property checks).** При передаче объектного литерала напрямую в функцию/переменную с ожидаемым типом TypeScript проверяет лишние свойства, но эта проверка обходится, если сначала присвоить литерал в промежуточную переменную.
6. **Ошибки в generic-ограничениях.** Указание неверного типа, не удовлетворяющего `extends`-ограничению обобщённого параметра.
7. **`strictPropertyInitialization`.** Поле класса объявлено с типом, не допускающим `undefined`, но не инициализировано ни при объявлении, ни в конструкторе.

## Пример / когда применяется

```ts
// 1. Implicit any — ловится noImplicitAny
function greet(name) { // Error: Parameter 'name' implicitly has an 'any' type
  return "Hello, " + name;
}

// 2. Null/undefined — ловится strictNullChecks
function getLength(s: string | null): number {
  return s.length; // Error: Object is possibly 'null'
  // Исправление:
  // return s?.length ?? 0;
}

// 3. Excess property check — обходится через промежуточную переменную
interface Point {
  x: number;
  y: number;
}
function draw(p: Point) {}
draw({ x: 1, y: 2, z: 3 }); // Error: Object literal may only specify known properties
const p = { x: 1, y: 2, z: 3 };
draw(p); // Ошибка НЕ выявляется — excess property check не сработал через переменную

// 4. Опасное приведение типов
const value: unknown = "42";
const num = value as number; // Компилируется без ошибки, но это неверно
console.log(num + 1); // Рантайм-баг: "421" вместо 43, если бы value была строкой

// 5. strictPropertyInitialization
class User {
  name: string; // Error: Property 'name' has no initializer and is not definitely assigned
  constructor() {}
}
```

Для проверки всего проекта на такие ошибки используется команда:

```bash
tsc --noEmit --strict
npx eslint . --ext .ts,.tsx
```

## На что смотрит интервьюер

- Знание конкретных флагов `strict`-режима и понимание, какую категорию ошибок каждый из них ловит — это ключевой маркер практического опыта.
- Способность привести реальный пример из практики, а не абстрактное «TypeScript ловит ошибки».
- Понимание ограничений компилятора: приведение типов (`as`), excess property check через промежуточную переменную — классические «дыры», которые компилятор не всегда закрывает.
- Частая ошибка — путать линтинг (ESLint) и проверку типов (`tsc`); это разные, взаимодополняющие инструменты.
- Follow-up: как настроить CI-проверку типов, что такое `// @ts-expect-error` и `// @ts-ignore` и чем они отличаются, как писать безопасные type guards вместо `as`.
