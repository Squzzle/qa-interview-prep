# Как сделать так, чтобы классы, объявленные в модуле, были доступны за его пределами?

**Коротко:** Нужно явно экспортировать класс из модуля с помощью ключевого слова `export` (именованный экспорт) или `export default` (экспорт по умолчанию), а также, при работе со старыми `namespace`/`module`-конструкциями TypeScript, использовать `export` внутри пространства имён.

## Развёрнутый ответ

В TypeScript (как и в ES-модулях, на которых он основан) каждый файл с хотя бы одним `import`/`export` считается модулем — всё, что объявлено внутри него, по умолчанию **приватно** и невидимо снаружи, пока не будет явно экспортировано. Есть несколько вариантов сделать класс доступным за пределами модуля:

1. **Именованный экспорт** — `export class User { ... }`. Импортируется через `import { User } from './user'`. Такой подход предпочтителен, так как позволяет иметь несколько экспортируемых сущностей в одном файле и даёт более явные, статически проверяемые имена при импорте.
2. **Экспорт по умолчанию** — `export default class User { ... }`. Импортируется через `import User from './user'` (имя при импорте может быть любым). Такой подход используют реже в TS-проектах, так как он хуже поддерживает автодополнение и рефакторинг, и Airbnb/большинство style guide рекомендуют именованные экспорты.
3. **Ре-экспорт через barrel-файлы** — `export { User } from './user';` в `index.ts`, чтобы дать единую точку входа для модуля.
4. **Экспорт из namespace (legacy)** — если класс объявлен внутри `namespace App { export class User {} }`, доступ снаружи пространства имён идёт через `App.User`, и для этого сам класс тоже должен быть помечен `export` внутри namespace. Это устаревший подход (появился до ES-модулей) и используется в основном в legacy-коде или при работе с global-скриптами без модульной системы.
5. **Экспорт из внешних (ambient) модулей** — для `.d.ts` файлов, описывающих сторонние библиотеки, класс объявляется как `export class` внутри `declare module 'lib-name' { ... }`.

Важный нюанс: если класс не экспортирован явно, TypeScript даже не позволит на него сослаться из другого файла — попытка `import` вызовет ошибку компиляции, потому что имя просто не будет частью экспортируемого API модуля. Также стоит различать `export class Foo` (значение + тип одновременно — класс формирует и то, и другое) и `export type { Foo }` — при использовании `isolatedModules` может понадобиться отдельно управлять экспортом только типов через `export type`.

## Пример / когда применяется

```ts
// user.ts
export class User {
  constructor(public name: string) {}
}

export default class Admin extends User {
  constructor(name: string, public permissions: string[]) {
    super(name);
  }
}
```

```ts
// main.ts
import Admin, { User } from './user';

const user = new User('Alice');
const admin = new Admin('Bob', ['read', 'write']);
```

```ts
// legacy-стиль через namespace
namespace App {
  export class Logger {
    log(message: string) {
      console.log(message);
    }
  }
}

const logger = new App.Logger();
```

## На что смотрит интервьюер

- Понимает ли кандидат разницу между именованным и дефолтным экспортом и может обосновать, какой предпочтительнее в командной разработке.
- Знает ли, что без `export` класс недоступен даже при использовании `declare` (не путать `declare` с `export`).
- Может ли объяснить разницу между модульным `export` и legacy `namespace`/`module` подходом, и почему второй сейчас не рекомендуется вне описания глобальных скриптов или `.d.ts`.
- Частая ошибка — путать CommonJS (`module.exports`) и ES-модули (`export`) при настройке `tsconfig.json` (`module`, `esModuleInterop`).
