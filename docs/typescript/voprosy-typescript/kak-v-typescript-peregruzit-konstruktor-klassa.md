# Как в TypeScript перегрузить конструктор класса?

**Коротко:** Так же, как обычные функции — объявляется несколько сигнатур `constructor(...)` без тела, а затем одна реализующая сигнатура с телом, которая через union-типы и проверки внутри (`typeof`, `Array.isArray`, количество аргументов) обрабатывает все допустимые варианты вызова.

## Развёрнутый ответ

Класс в TypeScript может иметь только один фактический конструктор (в отличие от языков с истинной перегрузкой методов, таких как Java или C#), но, как и для обычных функций, можно объявить несколько **сигнатур перегрузки конструктора**, после которых следует одна **реализующая сигнатура** с телом. Компилятор при вызове `new ClassName(...)` проверяет аргументы по объявленным сигнатурам сверху вниз и подсказывает разработчику допустимые варианты вызова, скрывая при этом реализующую сигнатуру от внешнего использования (снаружи нельзя вызвать конструктор с параметрами, которые не совпадают ни с одной перегрузкой).

Ключевые моменты:
- Все параметры перегрузок должны быть совместимы с параметрами реализующей сигнатуры — обычно её делают широкой (через union-тип или optional-параметры), а внутри тела уже точно определяют, какой вариант вызова произошёл.
- Такой же подход применим не только к обычным классам, но и к производным классам, вызывающим `super(...)` — важно смотреть, чтобы перегрузки базового класса корректно поддерживали нужные сценарии наследования.
- Альтернатива перегрузке — статические фабричные методы (`static create(...)`), которые часто более гибки и читаемы, чем набор перегруженных конструкторов, особенно если вариантов создания объекта много или они сильно различаются по смыслу.
- Ещё один паттерн — единственный параметр в виде объекта с необязательными полями (options object), который заменяет необходимость перегрузки, когда вариантов инициализации много.

Как и с обычными функциями, перегрузка конструктора существует только на уровне типов: после компиляции в JavaScript остаётся один конструктор, вся логика выбора поведения в зависимости от переданных аргументов реализуется вручную внутри тела.

## Пример / когда применяется

```ts
class Point {
  x: number;
  y: number;

  constructor(x: number, y: number);
  constructor(pair: [number, number]);
  constructor(json: { x: number; y: number });
  constructor(a: number | [number, number] | { x: number; y: number }, b?: number) {
    if (typeof a === "number") {
      this.x = a;
      this.y = b!;
    } else if (Array.isArray(a)) {
      [this.x, this.y] = a;
    } else {
      this.x = a.x;
      this.y = a.y;
    }
  }
}

const p1 = new Point(10, 20);
const p2 = new Point([10, 20]);
const p3 = new Point({ x: 10, y: 20 });
```

```ts
// Альтернатива — фабричные статические методы вместо перегрузки
class Point2 {
  private constructor(public x: number, public y: number) {}

  static fromCoords(x: number, y: number): Point2 {
    return new Point2(x, y);
  }

  static fromArray(pair: [number, number]): Point2 {
    return new Point2(pair[0], pair[1]);
  }
}
```

## На что смотрит интервьюер

- Знает ли кандидат, что у класса физически может быть только один реализующий конструктор, а "перегрузка" — это набор сигнатур поверх него.
- Умеет ли выбрать между перегрузкой конструктора и альтернативами (фабричные методы, options-объект) в зависимости от сложности сценариев инициализации.
- Понимает ли, как работать с наследованием и `super()` при перегруженном конструкторе базового класса.
- Частая ошибка — попытка дать разным перегрузкам конструктора несовместимые с реализующей сигнатурой параметры, что вызывает ошибку компиляции.
