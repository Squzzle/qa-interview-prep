# Как в TypeScript реализовать свойства класса, являющиеся константами?

**Коротко:** Для констант экземпляра используется модификатор `readonly` у поля класса, для констант, общих для всех экземпляров, — комбинация `static readonly`, а для вычисляемых на лету значений — геттер без сеттера.

## Развёрнутый ответ

TypeScript предоставляет несколько инструментов для объявления неизменяемых свойств класса, каждый решает свою задачу:

- **`readonly` для свойства экземпляра.** Поле, помеченное `readonly`, можно присвоить только при объявлении или внутри конструктора. Любая последующая попытка изменить его вне конструктора приведёт к ошибке компиляции. Это проверка только на этапе компиляции — в скомпилированном JS обычное присваивание технически возможно, если обойти типизацию (например, через `as any`).
- **`static readonly` для констант класса.** Если значение общее для всех экземпляров и не зависит от конкретного объекта, используется статическое неизменяемое поле: `static readonly MAX_RETRIES = 3;`. Обращение идёт через имя класса: `MyClass.MAX_RETRIES`.
- **Геттер без сеттера.** Если константа должна вычисляться на основе других полей объекта, применяется `get`-аксессор без соответствующего `set` — извне свойство доступно только для чтения, попытка присвоения вызовет ошибку компиляции (а при использовании `"use strict"` в рантайме — `TypeError`).
- **`Readonly<T>` и `as const`.** Utility type `Readonly<T>` делает все свойства объектного типа доступными только для чтения (полезно для параметров функций и DTO). Литерал `as const` замораживает литеральные значения и типы, полезен для констант вне класса или для инициализации значений по умолчанию.
- **`Object.freeze()`** — рантайм-механизм иммутабельности объекта (в отличие от `readonly`, который существует только на этапе компиляции). Совместное использование `readonly` (compile-time) и `Object.freeze` (runtime) даёт двойную защиту.

Важно чётко разграничивать: `readonly` — это гарантия только для TypeScript-компилятора и разработчиков, использующих типизированный код; она не защищает объект в рантайме от прямой мутации через нетипизированный JS-код или приведение типов.

## Пример / когда применяется

```ts
class Circle {
  readonly radius: number; // можно задать только в конструкторе
  static readonly PI = 3.14159; // общая константа для всех экземпляров

  constructor(radius: number) {
    this.radius = radius;
  }

  // Геттер без сеттера — вычисляемая "константа" на основе других полей
  get area(): number {
    return Circle.PI * this.radius ** 2;
  }
}

const c = new Circle(5);
console.log(c.area); // 78.53975
// c.radius = 10; // Error: Cannot assign to 'radius' because it is a read-only property.
// Circle.PI = 3; // Error: Cannot assign to 'PI' because it is a read-only property.

interface Config {
  readonly apiUrl: string;
}

function setup(config: Readonly<Config>) {
  // config.apiUrl = "changed"; // Error
}
```

## На что смотрит интервьюер

- Понимание, что `readonly` — проверка только на этапе компиляции, а не рантайм-иммутабельность (в отличие от `Object.freeze`).
- Умение отличить `readonly` поле экземпляра от `static readonly` константы класса.
- Знание, что `readonly` не даёт глубокой (deep) неизменяемости: `readonly items: string[]` запрещает переприсвоение массива, но не запрещает `items.push(...)`.
- Follow-up: как сделать глубокую иммутабельность (рекурсивный `Readonly`, библиотеки типа Immutable.js), разница между `const` и `readonly`.
