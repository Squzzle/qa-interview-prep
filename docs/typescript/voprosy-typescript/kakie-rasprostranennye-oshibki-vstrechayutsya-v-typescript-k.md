# Какие распространённые ошибки встречаются в TypeScript-коде и как их избежать?

**Коротко:** Чаще всего проблемы возникают из-за злоупотребления `any`, неверных приведений типов (`as`), игнорирования `strict`-режима, неправильной работы с `null`/`undefined`, а также из-за путаницы между типами и структурами времени выполнения; избежать их помогают включённый `strict`, линтеры (`@typescript-eslint`) и дисциплина в написании типов.

## Развёрнутый ответ

Ниже — типичный набор ошибок, которые встречаются в реальных TypeScript-проектах:

1. **Злоупотребление `any`** — использование `any` "чтобы компилятор не ругался" полностью отключает проверку типов для этого значения и всего, что из него выведено, фактически превращая типизированный код в обычный JS с ложным чувством безопасности. Решение: использовать `unknown` вместо `any` там, где тип действительно неизвестен, и сужать его через type guard-ы перед использованием.
2. **Небезопасные приведения типов через `as`** — `as` не проверяет соответствие типов в рантайме, а просто "убеждает" компилятор; двойное приведение `as unknown as T` вообще отключает всякую проверку. Решение: использовать приведение только когда есть уверенность в структуре данных (например, после валидации через `zod`/`io-ts`), либо писать явные type guard-функции (`function isUser(x: unknown): x is User`).
3. **Отключённый или частично включённый `strict`-режим** — без `strictNullChecks` `null` и `undefined` совместимы с любым типом, что убирает главное преимущество TS — защиту от `null`-ошибок в рантайме. Решение: всегда включать `strict: true` в новых проектах и постепенно мигрировать legacy-код.
4. **Игнорирование ошибок компилятора через `@ts-ignore`/`@ts-expect-error` без причины** — маскирует реальные проблемы вместо их решения. `@ts-expect-error` предпочтительнее `@ts-ignore`, так как выдаёт ошибку, если проблема уже исправлена (что помогает не забывать убирать такие подавления).
5. **Мутация полей, помеченных `readonly`, через приведение типов или `Object.assign`** — `readonly` — это только компайл-тайм защита, в рантайме её можно обойти, что даёт ложное чувство иммутабельности.
6. **Путаница между структурной и номинальной типизацией** — ожидание, что два разных класса с одинаковой структурой полей будут несовместимы (как в номинально типизированных языках), тогда как TypeScript использует структурную типизацию, и совместимость определяется формой объекта, а не именем класса.
7. **Широкие типы вместо литеральных** — объявление `let status = "active"` даёт тип `string` вместо литерального `"active"`, что теряет точность в union-типах; решение — использовать `as const` или явную аннотацию.
8. **Отсутствие типизации границ приложения (API, внешние данные)** — типы `interface Response { ... }` описывают "ожидаемую" форму данных, но TypeScript не проверяет её в рантайме; без runtime-валидации (zod, io-ts) неверные данные от бэкенда пройдут молча и упадут глубже в коде.
9. **Циклические импорты типов и раздутые union-типы**, которые сильно замедляют компиляцию — решение: использовать `import type`, разбивать типы на более мелкие модули.

Общая рекомендация: включать `strict` полностью, использовать `@typescript-eslint` с правилами `no-explicit-any`, `no-unsafe-assignment` и т.д., и относиться к типам как к контракту, а не как к "подсказке", которую можно обойти при первой неудобности.

## Пример / когда применяется

```ts
// Плохо: any скрывает реальную проблему
function getUserName(user: any): string {
  return user.profile.name; // упадёт в рантайме, если profile отсутствует
}

// Лучше: unknown + type guard
function isUser(value: unknown): value is { profile: { name: string } } {
  return (
    typeof value === "object" &&
    value !== null &&
    "profile" in value
  );
}

function getUserNameSafe(user: unknown): string {
  if (!isUser(user)) {
    throw new Error("Некорректная форма пользователя");
  }
  return user.profile.name;
}
```

```ts
// Плохо: широкий тип вместо литерала
let role = "admin"; // тип string

// Хорошо: точный литеральный тип
let role2 = "admin" as const; // тип "admin"
const roles = ["admin", "editor", "viewer"] as const;
type Role = (typeof roles)[number]; // "admin" | "editor" | "viewer"
```

## На что смотрит интервьюер

- Может ли кандидат привести конкретные примеры ошибок из практики, а не только теоретический список.
- Понимает ли разницу между `any` и `unknown` и знает, почему `unknown` безопаснее.
- Знает ли, что `readonly` и приведения типов (`as`) — компайл-тайм механизмы, не защищающие в рантайме.
- Красный флаг — если кандидат считает, что TypeScript сам по себе гарантирует отсутствие ошибок в рантайме без runtime-валидации внешних данных.
