# Можно ли использовать TypeScript в серверной разработке, и если да — то как?

**Коротко:** Да, TypeScript широко используется на сервере поверх Node.js (и других рантаймов вроде Deno и Bun) — код компилируется в JavaScript (через `tsc`, `esbuild`, `swc` или напрямую исполняется через `ts-node`/`tsx`), а фреймворки вроде NestJS, Express с типами и Fastify обеспечивают типобезопасную серверную разработку.

## Развёрнутый ответ

TypeScript не привязан к браузеру — это язык общего назначения, который транспилируется в стандартный JavaScript, а значит может выполняться везде, где есть JS-движок: Node.js, Deno, Bun, Cloudflare Workers и т.д. Серверное использование TypeScript — одна из самых распространённых практик в современной индустрии.

Основные подходы к запуску TypeScript на сервере:

- **Компиляция + запуск скомпилированного JS.** Классический вариант: `tsc` компилирует `.ts` в `.js` в директорию `dist`, затем `node dist/index.js` запускает обычный JavaScript. Это стандартный подход для production-сборок — компиляция отделена от запуска, что даёт максимальную производительность в рантайме.
- **`ts-node`.** Инструмент, который компилирует и исполняет TypeScript «на лету» без промежуточного шага сборки — удобен для разработки и скриптов, но добавляет накладные расходы на старте.
- **`tsx` / `swc-node` / `esbuild-register`.** Более быстрые альтернативы `ts-node`, использующие транспиляторы на Rust/Go (esbuild, SWC) вместо полного тайпчекера — существенно ускоряют старт для разработки (при этом типы не проверяются, только транспиляция синтаксиса).
- **Прямая поддержка рантаймами.** Deno и Bun умеют исполнять `.ts`-файлы «из коробки», без отдельной настройки компиляции.

Типичный серверный стек с TypeScript:

- **NestJS** — фреймворк, спроектированный вокруг TypeScript с самого начала: декораторы, DI-контейнер, модули — всё построено на типах и метаданных TypeScript.
- **Express/Koa/Fastify + `@types/...`** — типизация через отдельные `@types`-пакеты (DefinitelyTyped) для нетипизированных изначально библиотек.
- **ORM с типобезопасностью** — Prisma, TypeORM, Drizzle генерируют или используют TypeScript-типы для моделей БД, обеспечивая типобезопасные запросы.
- **Валидация схем** — Zod, io-ts, class-validator позволяют одновременно валидировать данные в рантайме (например, тело HTTP-запроса) и выводить (`infer`) TypeScript-типы из схемы, решая проблему «типы TS не проверяются в рантайме».

Важный нюанс, который стоит подчеркнуть: типы TypeScript полностью стираются при компиляции, поэтому они не защищают сервер от некорректных данных, пришедших извне (тело запроса, ответ стороннего API, данные из очереди). Для рантайм-валидации внешних данных необходимо использовать отдельные библиотеки схем (Zod, Yup, class-validator и т.п.), а не полагаться на TypeScript-типы.

## Пример / когда применяется

```ts
// package.json (фрагмент) — сборка и запуск
// "build": "tsc",
// "start": "node dist/index.js",
// "dev": "tsx watch src/index.ts"

// src/index.ts — минимальный сервер на Express с типами
import express, { Request, Response } from "express";

interface CreateUserBody {
  name: string;
  email: string;
}

const app = express();
app.use(express.json());

app.post("/users", (req: Request<{}, {}, CreateUserBody>, res: Response) => {
  const { name, email } = req.body;
  res.status(201).json({ id: 1, name, email });
});

app.listen(3000, () => console.log("Server started on port 3000"));
```

```json
// tsconfig.json (фрагмент для Node.js сервера)
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "commonjs",
    "outDir": "dist",
    "strict": true,
    "esModuleInterop": true
  }
}
```

## На что смотрит интервьюер

- Понимание, что типы TypeScript — только compile-time и не защищают сервер от невалидных данных из внешнего мира (нужна рантайм-валидация).
- Знание конкретных инструментов запуска (`ts-node`, `tsx`, компиляция + `node`) и разницы между ними по скорости/надёжности.
- Знакомство хотя бы с одним типобезопасным серверным стеком (NestJS, Prisma/TypeORM, Zod).
- Частая ошибка — путать типы TypeScript с валидацией схем; путать `@types/express` (типы) с самим фреймворком.
