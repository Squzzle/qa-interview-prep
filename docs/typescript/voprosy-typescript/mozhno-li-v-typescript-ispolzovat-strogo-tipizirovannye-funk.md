# Можно ли в TypeScript использовать строго типизированные функции в качестве параметров?

**Коротко:** Да — TypeScript позволяет описывать тип параметра-функции полной сигнатурой (аргументы и возвращаемое значение), используя синтаксис функционального типа, интерфейс с вызываемой сигнатурой или generic-типы, что даёт полную проверку типов при передаче callback-ов.

## Развёрнутый ответ

В TypeScript функции — полноценные значения первого класса, и их тип можно описать так же строго, как тип любой другой переменной. Есть несколько способов задать тип параметра-функции:

1. **Стрелочная сигнатура** — `(a: number, b: number) => number`. Это самый распространённый способ описать callback напрямую в списке параметров функции.
2. **Именованный тип или интерфейс** — можно вынести сигнатуру в `type Comparator = (a: number, b: number) => number;` или в интерфейс с вызываемой сигнатурой `interface Comparator { (a: number, b: number): number; }`. Это удобно для переиспользования и читаемости.
3. **Generic-функции как параметры** — можно типизировать параметр-функцию как обобщённую, чтобы сохранить связь между входным и выходным типом вызывающей функции, например `<T, R>(fn: (item: T) => R) => R[]`.
4. **Перегрузки в типе callback-а** — при необходимости тип функции-параметра может содержать несколько сигнатур вызова (через пересечение типов с вызываемыми сигнатурами).

Компилятор проверяет совместимость сигнатур по правилам **структурной типизации** (structural typing) и **бивариантности параметров функций** (для методов) — при передаче функции в качестве аргумента TypeScript проверяет, что параметры принимаемой функции хотя бы так же общие или более общие, чем ожидаемые, а возвращаемый тип — совместим (обычно ковариантен). Это называется совместимостью функциональных типов (function type compatibility) и основано на принципе подстановки Лисков в применении к функциям: функцию с менее строгими требованиями к аргументам и с более специфичным возвращаемым значением можно передать туда, где ожидается более общая функция.

Также важно различать `strictFunctionTypes` — при включённом флаге (часть `strict`) параметры обычных функций (не методов) проверяются контравариантно, что даёт более безопасную, но и более строгую проверку в некоторых сценариях с callback-ами.

## Пример / когда применяется

```ts
type Comparator<T> = (a: T, b: T) => number;

function sortBy<T>(items: T[], compare: Comparator<T>): T[] {
  return [...items].sort(compare);
}

const numbers = [5, 3, 8, 1];
const sorted = sortBy(numbers, (a, b) => a - b); // строго типизировано: a, b: number

interface Formatter {
  (value: number): string;
}

function printPrice(value: number, formatter: Formatter): void {
  console.log(formatter(value));
}

printPrice(1500, (v) => `${v} ₽`);
```

```ts
// Generic-функция как параметр, сохраняющая связь типов
function mapArray<T, R>(items: T[], transform: (item: T, index: number) => R): R[] {
  return items.map(transform);
}

const lengths = mapArray(["a", "bb", "ccc"], (s) => s.length); // R выведен как number
```

## На что смотрит интервьюер

- Знает ли кандидат про структурную типизацию функций и совместимость сигнатур (можно ли передать функцию с менее специфичными параметрами вместо более специфичной).
- Понимает ли разницу между `strictFunctionTypes` включённым и выключенным, и как это влияет на проверку callback-ов.
- Может ли отличить типизацию через `type`/`interface` с вызываемой сигнатурой от инлайновой аннотации параметра.
- Частая ошибка — путать перегрузку функции (multiple call signatures) с несколькими параметрами разных типов; также забывают, что возвращаемый тип `void` в типе callback-а позволяет передавать функции, которые реально что-то возвращают (TypeScript просто игнорирует результат).
