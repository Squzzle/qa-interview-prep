# Поддерживает ли TypeScript перегрузку функций?

**Коротко:** Да, но не в рантайме, а только на уровне типов — TypeScript позволяет описать несколько сигнатур (overload signatures) для одной функции, компилятор выбирает подходящую сигнатуру при проверке типов, а в JavaScript в итоге компилируется только одна реальная реализация функции.

## Развёрнутый ответ

В отличие от языков вроде Java или C#, где перегрузка методов — рантайм-механизм диспетчеризации по типам аргументов, в TypeScript перегрузка — это чисто **компайл-тайм** конструкция. Механизм работает так:

1. Объявляется несколько **сигнатур перегрузки** (overload signatures) — заголовков функции без тела.
2. После них идёт одна **реализующая сигнатура** (implementation signature) с телом функции — она не видна снаружи при вызове и должна быть совместима со всеми объявленными перегрузками (обычно её параметры делают через union-типы или `any`/`unknown`, чтобы охватить все варианты).
3. При вызове функции компилятор проверяет аргументы по порядку перегрузок сверху вниз и использует первую подходящую — поэтому порядок объявления перегрузок важен (более специфичные сигнатуры должны идти раньше более общих).
4. После компиляции в JS все сигнатуры перегрузки исчезают — остаётся только одна функция с одной реализацией, всю логику выбора поведения в зависимости от типа аргумента реализует сам разработчик внутри тела функции (через `typeof`, `instanceof`, проверку количества аргументов и т.д.).

Такой же механизм применим к методам классов и к конструкторам (сигнатуры `constructor` без тела + одна реализация). Перегрузка полезна, когда одна функция должна возвращать разный тип в зависимости от типа входного аргумента, и обычный union-тип параметра/результата хуже описывает связь между входом и выходом, чем набор конкретных перегрузок.

Важно отличать перегрузку от **generic-функций**: generic — это один тип, параметризованный переменной типа, тогда как перегрузка — набор разных, потенциально несвязанных сигнатур. Иногда правильнее использовать generic с условными типами вместо перегрузок — это даёт более гибкую типизацию без дублирования сигнатур.

## Пример / когда применяется

```ts
// Сигнатуры перегрузки
function createElement(tag: "a"): HTMLAnchorElement;
function createElement(tag: "img"): HTMLImageElement;
function createElement(tag: string): HTMLElement;
// Реализующая сигнатура (не видна извне)
function createElement(tag: string): HTMLElement {
  return document.createElement(tag);
}

const link = createElement("a");   // тип: HTMLAnchorElement
const img = createElement("img");  // тип: HTMLImageElement
const div = createElement("div");  // тип: HTMLElement
```

```ts
class Vector {
  constructor(x: number, y: number);
  constructor(pair: [number, number]);
  constructor(xOrPair: number | [number, number], y?: number) {
    if (Array.isArray(xOrPair)) {
      this.x = xOrPair[0];
      this.y = xOrPair[1];
    } else {
      this.x = xOrPair;
      this.y = y!;
    }
  }
  x: number;
  y: number;
}
```

## На что смотрит интервьюер

- Понимает ли кандидат, что перегрузка в TS — это только про типы, и в скомпилированном JS остаётся одна функция.
- Знает ли, что порядок перегрузок влияет на выбор сигнатуры компилятором (от более специфичной к более общей).
- Может ли объяснить, чем перегрузка отличается от параметра с union-типом и когда стоит выбрать одно, а не другое.
- Частая ошибка — попытка вызвать функцию с аргументами реализующей сигнатуры напрямую, не понимая, что снаружи видны только объявленные перегрузки, а не сигнатура реализации.
- Follow-up: "Как перегрузка сочетается с generic-типами и условными типами?", "Можно ли перегрузить оператор в TypeScript?" (нет, в отличие от C++/C#, TypeScript не поддерживает перегрузку операторов).
