# Поддерживает ли TypeScript все принципы объектно-ориентированного программирования?

**Коротко:** Да, TypeScript поддерживает все четыре классических принципа ООП — инкапсуляцию, наследование, полиморфизм и абстракцию, добавляя к прототипному ООП JavaScript классический синтаксис классов, модификаторы доступа, интерфейсы и абстрактные классы.

## Развёрнутый ответ

JavaScript изначально построен на прототипном наследовании, а классы, появившиеся в ES2015, являются синтаксическим сахаром над прототипами. TypeScript расширяет этот синтаксис, добавляя строгую типизацию и конструкции, характерные для классических (номинативных) ООП-языков вроде Java или C#.

- **Инкапсуляция.** TypeScript вводит модификаторы доступа `public`, `private`, `protected`, а также `readonly` для полей. Это позволяет скрывать внутреннее состояние класса и предоставлять доступ к нему только через методы или геттеры/сеттеры. Важно понимать: модификаторы `private`/`protected` — это проверка только на этапе компиляции, в скомпилированном JS (если не используется `target` с поддержкой `#` приватных полей) они отсутствуют и доступ технически возможен через `obj["field"]`. Реальную приватность на уровне рантайма дают нативные приватные поля JS (`#field`), которые TypeScript также поддерживает.
- **Наследование.** Ключевое слово `extends` позволяет классу наследовать поля и методы другого класса, `super()` — вызывать конструктор и методы родителя. TypeScript поддерживает одиночное наследование классов (множественное наследование классов не поддерживается — вместо него используются интерфейсы и миксины).
- **Полиморфизм.** Переопределение методов в дочерних классах (method overriding), а также структурная типизация позволяют работать с объектами разных классов через общий интерфейс или базовый тип. TypeScript поддерживает и параметрический полиморфизм через generics.
- **Абстракция.** Ключевое слово `abstract` позволяет объявлять абстрактные классы и абстрактные методы, которые не имеют реализации в базовом классе и обязаны быть реализованы в наследниках. Также абстракция достигается через `interface`, которые описывают контракт без реализации.

Отдельные особенности, которые стоит упомянуть:

- TypeScript поддерживает **интерфейсы** (`interface`), которых нет в JavaScript, для описания контрактов (`implements`).
- Классы в TypeScript могут реализовывать несколько интерфейсов одновременно (`class Foo implements A, B`), что частично компенсирует отсутствие множественного наследования.
- Есть поддержка **статических членов** (`static`), **абстрактных классов**, **декораторов** (экспериментальная/stage 3 фича, используемая, например, в Angular и NestJS).
- TypeScript не добавляет ООП-возможностей в рантайм — все проверки типов и модификаторов доступа существуют только на этапе компиляции и исчезают в итоговом JS-коде (кроме нативных приватных полей `#field`).

## Пример / когда применяется

```ts
abstract class Animal {
  protected constructor(protected name: string) {}

  abstract makeSound(): string;

  introduce(): string {
    return `${this.name} says ${this.makeSound()}`;
  }
}

class Dog extends Animal {
  constructor(name: string, private breed: string) {
    super(name);
  }

  makeSound(): string {
    return "Woof!";
  }
}

interface Trainable {
  train(): void;
}

class ServiceDog extends Dog implements Trainable {
  train(): void {
    console.log(`${this.name} is training`);
  }
}

const dog: Animal = new ServiceDog("Rex", "Labrador");
console.log(dog.introduce()); // полиморфизм: вызовется переопределённый makeSound
```

## На что смотрит интервьюер

- Понимание того, что модификаторы доступа TypeScript — это compile-time конструкция, а не рантайм-защита (в отличие от нативных `#private` полей).
- Знание, что множественное наследование классов не поддерживается, и умение объяснить альтернативы (интерфейсы, композиция, миксины).
- Способность привести пример абстрактного класса и объяснить отличие `abstract class` от `interface`.
- Частая ошибка — путать структурную типизацию TypeScript с номинативной типизацией классических ООП-языков, где реализация интерфейса проверяется по имени, а не по форме.
- Follow-up: миксины в TypeScript, декораторы, разница между `interface` и `type` при описании контрактов классов.
