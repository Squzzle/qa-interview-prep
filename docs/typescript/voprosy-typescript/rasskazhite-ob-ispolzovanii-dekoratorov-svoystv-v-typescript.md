# Расскажите об использовании декораторов свойств в TypeScript.

**Коротко:** Декоратор свойства — это функция, которая применяется к полю класса на этапе его объявления и позволяет перехватывать или изменять поведение свойства (например, добавлять логирование, валидацию, реактивность) без изменения самого класса; для работы требуется включить `experimentalDecorators` (legacy) либо использовать декораторы по новому стандарту ECMAScript (TS 5+).

## Развёрнутый ответ

Декораторы — это специальный синтаксис `@expression`, который применяется к классам, методам, аксессорам, свойствам или параметрам. Декоратор свойства — это функция, которая вызывается компилятором в момент объявления класса (не при создании экземпляра) и получает информацию о цели декорирования. Исторически (legacy-декораторы, `experimentalDecorators: true` в `tsconfig.json`) декоратор свойства имеет сигнатуру `(target: any, propertyKey: string | symbol) => void`, где `target` — это прототип класса (для статических свойств — сам конструктор), а `propertyKey` — имя декорируемого свойства. В отличие от декораторов методов, декоратор свойства не получает `PropertyDescriptor` и не может напрямую вернуть новое значение дескриптора — legacy-версия спецификации в первую очередь предназначена для регистрации метаданных (например, через `reflect-metadata`), а не для замены самого поля.

Начиная с TypeScript 5.0 поддерживается новый стандарт декораторов (Stage 3 ECMAScript decorators), который не требует `experimentalDecorators`. В нём декоратор свойства получает `ClassFieldDecoratorContext` и может возвращать функцию-инициализатор, которая принимает исходное значение и возвращает новое — это уже полноценный механизм трансформации значения поля. Именно этот вариант используют современные библиотеки (например, MobX 6, некоторые ORM) для реализации реактивности и валидации.

Практическое применение декораторов свойств:
- **Метаданные для DI и валидации** — Angular и NestJS используют декораторы свойств (`@Input()`, `@IsString()` из `class-validator`) для описания метаданных, которые затем читаются через `Reflect.getMetadata`.
- **Реактивность** — MobX использует `@observable` для превращения обычного поля в реактивное.
- **Валидация и трансформация** — оборачивание сеттера, чтобы проверять или приводить значение при присваивании.
- **Логирование и readonly-подобное поведение** — можно перехватить доступ к свойству через `Object.defineProperty` внутри декоратора.

Важный нюанс: для legacy-декораторов TypeScript сам по себе не меняет рантайм-поведение свойства — вся логика реализуется вручную внутри функции-декоратора через `Object.defineProperty`. Также стоит помнить, что декораторы — это фича стадии предложения (proposal), синтаксис и семантика менялись между версиями TypeScript, поэтому при работе в команде важно зафиксировать, какой вариант декораторов используется (legacy или новый стандарт), и не смешивать их в одном проекте.

## Пример / когда применяется

```ts
// tsconfig.json: "experimentalDecorators": true, "emitDecoratorMetadata": true

function LogChanges(target: any, propertyKey: string) {
  let value: unknown;

  const getter = () => value;
  const setter = (newValue: unknown) => {
    console.log(`${propertyKey} изменено на`, newValue);
    value = newValue;
  };

  Object.defineProperty(target, propertyKey, {
    get: getter,
    set: setter,
    enumerable: true,
    configurable: true,
  });
}

class User {
  @LogChanges
  name: string = "";
}

const u = new User();
u.name = "Alice"; // выведет в консоль: "name изменено на Alice"
```

```ts
// Новый стандарт декораторов (TS 5+, без experimentalDecorators)
function Trim(value: undefined, context: ClassFieldDecoratorContext) {
  return function (this: unknown, initialValue: string) {
    return initialValue.trim();
  };
}

class Form {
  @Trim
  title: string = "  Заголовок  ";
}
```

## На что смотрит интервьюер

- Понимает ли кандидат разницу между legacy-декораторами (`experimentalDecorators`) и новым стандартом ECMAScript-декораторов в TS 5+.
- Знает ли, что декоратор свойства выполняется один раз при объявлении класса, а не при каждом создании экземпляра — частая ошибка думать, что это хук на каждый инстанс.
- Может ли объяснить, зачем нужен `reflect-metadata` и как декораторы связаны с DI-контейнерами (Angular, NestJS).
- Красный флаг — если кандидат путает декораторы свойств с декораторами методов или не понимает, что для legacy-декораторов свойств нет доступа к `PropertyDescriptor`, в отличие от декораторов методов.
