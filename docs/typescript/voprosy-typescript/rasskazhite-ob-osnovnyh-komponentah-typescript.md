# Расскажите об основных компонентах TypeScript.

**Коротко:** TypeScript как экосистема состоит из языка (синтаксис и система типов), компилятора `tsc` (транспиляция и проверка типов), Language Service (поддержка IDE) и конфигурационного файла `tsconfig.json`, определяющего поведение компиляции.

## Развёрнутый ответ

Полезно разбирать TypeScript не как один монолитный инструмент, а как набор взаимосвязанных компонентов:

- **Язык (Language).** Синтаксис и семантика, надстроенные над JavaScript: аннотации типов, интерфейсы, enum, generics, namespace/modules, декораторы, условные и mapped-типы. Это то, что видит и пишет разработчик.
- **Компилятор (Compiler, `tsc`).** Основной инструмент, выполняющий две связанные, но раздельные задачи:
  - **Type checking** — статический анализ кода на соответствие типам, вывод ошибок несоответствия.
  - **Transpilation (emit)** — преобразование TS-синтаксиса (а также нового синтаксиса JS) в JavaScript целевой версии (`target`), совместимый с указанной средой выполнения.
  Важно: эти два процесса можно разделять — например, `tsc --noEmit` только проверяет типы без генерации файлов, а инструменты типа Babel/esbuild/SWC делают только транспиляцию без проверки типов (что быстрее, но не ловит ошибки типов).
- **Language Service.** Отдельный API компилятора, который используют IDE (VS Code, WebStorm) для автодополнения, подсказок по типам при наведении, поиска использований, безопасного переименования, быстрых исправлений (quick fixes) и подсветки ошибок в реальном времени, без полной перекомпиляции проекта.
- **Декларативные файлы (`.d.ts`).** Файлы, содержащие только объявления типов без реализации — используются для описания типов сторонних JS-библиотек (пакеты `@types/*` из DefinitelyTyped) или для публикации типов вместе со скомпилированной библиотекой.
- **`tsconfig.json`.** Конфигурационный файл, определяющий, какие файлы включены в проект и как их компилировать: `target` (версия ECMAScript на выходе), `module` (система модулей — CommonJS/ESNext), `strict` (набор строгих проверок), `outDir`/`rootDir`, `lib`, `paths` и т.д. Также поддерживает наследование через `extends` и составные проекты через `references` (project references) для монорепозиториев.
- **Компилятор API (Compiler API/Program).** Программный интерфейс, позволяющий инструментам (линтерам, тестовым фреймворкам, генераторам кода, IDE) программно анализировать AST TypeScript-кода, использовать TypeChecker для получения информации о типах.
- **Система типов (Type System).** Отдельно стоит выделить структурную систему типов — совместимость типов определяется формой объекта, а не явным объявлением реализации, что принципиально отличает TS от номинативных систем Java/C#.

## Пример / когда применяется

```json
// tsconfig.json — типичная конфигурация, объединяющая компоненты
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "declaration": true,
    "outDir": "./dist",
    "lib": ["ES2020", "DOM"]
  },
  "include": ["src/**/*.ts"]
}
```

```bash
# Только проверка типов, без генерации файлов
tsc --noEmit

# Полная компиляция с генерацией .js и .d.ts
tsc
```

```ts
// example.d.ts — файл объявления типов для JS-библиотеки без встроенных типов
declare module "some-untyped-lib" {
  export function doSomething(value: string): number;
}
```

## На что смотрит интервьюер

- Понимание, что type checking и emit (генерация JS) — раздельные процессы, и умение объяснить, зачем нужен `tsc --noEmit` в CI при использовании Babel/esbuild для сборки.
- Знание роли `.d.ts`-файлов и пакетов `@types/*`.
- Понимание, что Language Service — отдельный слой, обеспечивающий работу IDE, а не то же самое, что компилятор командной строки.
- Частая ошибка — считать, что `tsconfig.json` влияет на рантайм-поведение JS напрямую, хотя на деле он только управляет тем, как исходный код проверяется и транспилируется.
- Follow-up: project references для монорепо, `isolatedModules`, разница между `tsc` и Babel-транспиляцией TypeScript.
